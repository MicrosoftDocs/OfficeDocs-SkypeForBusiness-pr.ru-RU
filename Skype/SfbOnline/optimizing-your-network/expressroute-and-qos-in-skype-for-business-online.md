# <a name="expressroute-and-qos-in-skype-for-business-online"></a>ExpressRoute и качество обслуживания в Skype для бизнеса Online

[!INCLUDE [sfbo-retirement](../../Hub/includes/sfbo-retirement.md)]

Подключение Microsoft 365 или Office 365 через выделенное сетевое подключение с помощью Azure ExpressRoute для Microsoft 365 или Office 365 и Skype для бизнеса Online. Выделенное подключение для приложений Skype для бизнеса позволит надежно и предсказуемо работать, а также конфиденциальность, не влияя на общедоступный Интернет. Теперь вы можете приобрести более надежное сетевое подключение к Microsoft 365 или Office 365 и Skype для бизнеса Online, что повышает предсказуемость, надежность бизнес-класса и поставляется с простоем SLA.
  
> [!NOTE]
> Доступна новая версия калькулятора пропускной способности: [Skype для бизнеса, Калькулятор пропускной способности](https://go.microsoft.com/fwlink/?LinkId=715766). Однако в этом документе используются калькулятор пропускной способности Lync 2010 и 2013. 
  
## <a name="skype-for-business-online-and-expressroute"></a>Skype для бизнеса Online и ExpressRoute

Работая с партнером Майкрософт по ExpressRoute, вы можете подключать различные приложения Microsoft 365 и Office 365, включая Skype для бизнеса Online в облаке, по выделенного подключению. Однако для голосовой и видеосвязи в режиме реального времени для Skype для бизнеса требуются сетевые службы, специально настроенные для поддержки таких Microsoft 365 или Office 365 рабочих нагрузок в режиме реального времени. К ним относится сеть с достаточной пропускной способностью для выполнения необходимого объема трафика и поддержки качества обслуживания (QoS) для обслуживания пользователей бизнес-класса.
  
Этот документ поможет вам, администраторам и сетевым дизайнерам понять, какие особые проблемы необходимы для связи в режиме реального времени, средства, предоставляемые корпорацией Майкрософт, чтобы помочь в разработке сети, поддерживающей эти требования, и помогут вам спроектировать процесс проектирования с помощью анализа. 
  
В первой части этого документа содержится пример, который поможет вам с проектированием сети с помощью калькулятора полосы пропускания [Lync 2010 и 2013](https://go.microsoft.com/fwlink/?LinkId=690282) оценить требования к сети для развертывания ExpressRoute с большим Skype для бизнеса с несколькими сайтами. Вторая часть этого документа содержит основные понятия качества обслуживания (QoS), подробно изумляет технические сведения о Skype для бизнеса связи в режиме реального времени и необходимых типах сетевых служб.
  
Вся информация в этой области позволит получить технические сведения и сведения о QoS и ExpressRoute, понять, с чем вы столкнулись, а также получить знания о средствах и методах, которые позволят успешно развернуть ExpressRoute в сети Skype для бизнеса.

### <a name="setting-up-the-network"></a>Настройка сети

Чтобы обеспечить согласованную и высококачественную службу в режиме реального времени для компании Dewey Law LLC., необходимо выполнить несколько базовых требований.
  
- Они хотят предоставлять голосовые службы во время сбоя питания, чтобы сетевые коммутаторы распространения и маршрутизаторы могли обеспечить питание по Ethernet (PoE) IEEE 802.3af или 802.3at.
    
- Сетевые коммутаторы и маршрутизаторы также должны использовать бесперебойные источники питания (UPS), чтобы они могли работать во время сбоя питания.
    
    У них есть Wi-Fi к локальному офису, поэтому мы настоятельно рекомендуем им использовать сертифицированного партнера по инфраструктуре Skype для бизнеса Wi-Fi от [Skype для бизнеса Solutions.](https://go.microsoft.com/fwlink/?LinkId=690281)
    
    > [!TIP]
    >  Рекомендуется использовать беспроводные точки доступа 802.11n и 802.11ac.
  
- И, что самое важное, все сети локальной сети во всех офисах должны быть настроены для обеспечения качества обслуживания (QoS). К ним относятся компьютеры, ноутбуки и сетевое оборудование, например коммутаторы и маршрутизаторы.
    
Теперь, когда у вас есть все необходимое для предоставления голосовых служб бизнес-класса для компании Dewey Law LLC., мы рекомендуем использовать ip-службу MPLS от партнера по сетевой службе, который подключается к службе Azure ExpressRoute. MPLS предоставляет ip-службу с гарантией производительности при задержке, дрожания и потере пакетов. Однако если MPLS не доступен, можно использовать Ethernet, подключенный к одному из наших партнеров по обмену данными ExpressRoute.
  
Поставщики MPLS предлагают несколько уровней обслуживания, но для их определения используются разные термины. Вам необходимо тесно сотрудничать со своим поставщиком, чтобы он понял, какие данные вы вводим в [Lync 2010 и 2013 Bandwidth Calculator,](https://go.microsoft.com/fwlink/?LinkID=690282) а также доступные параметры, рекомендуемые для разных приложений Microsoft 365 и Office 365 рабочей нагрузки в режиме реального времени.
  
Существует два варианта, как данные из Skype для бизнеса приложений можно соедеть с соответствующими классами обслуживания MPLS:
  
- Пометка трафика конечными точками с помощью контрольных точек DiffServ (DSCP)
    
- Список управления сетевым доступом (ACL) на основе
    
Чтобы реализовать маркировку конечной точки, необходимо настроить все компьютеры, Windows домена, для компании Dewey Law LLC. чтобы пометить каждый пакет с помощью соответствующей маркировки DiffServ Control Point (DSCP), а затем внедрить QoS на всех сетевых коммутаторах и маршрутизаторах на всех сайтах офисов, чтобы убедиться, что маркировка QoS сохранена и не удалена. Маркировка DSCP на сетевых пакетах сообщает поставщику услуг, как этот пакет имеет приоритет. **Дополнительные сведения о DSCP можно получить в разделе QoS части 2.**
  
При назначении на основе ACL сети маркировки приоритетов DSCP внедряются на маршрутизаторе с последующим потоком и основаны на порте источника UDP. Рекомендуемые диапазоны портов для каждого приложения перечислены в разделе 2.6.1.1 статьи Планирование сети, мониторинг и устранение неполадок с [сервером Lync Server.](https://go.microsoft.com/fwlink/?LinkId=690286) Важно, чтобы вы координировали эту работу с общей реализацией и проектированием QoS компании Dewey Law LLC, а также знаете о различных политиках QoS и о потенциальных несоответствиях маркировки пакетов.
  
У каждого поставщика сетевых служб ExpressRoute будет класс обслуживания (QoS), который подходит для голосовой и видеосвязи в режиме реального времени. Это COS называется "Ускоренная переадваровка" (EF) для голосовой связи и "Assured Forwarding" (AF) для видео. Следует внимательно относиться к размеру пропускной способности, приобретаемой для голосового трафика EF. Причина в том, что класс обслуживания голосовой почты очень неумолим в случае, если вы отправляете больше голосового трафика, чем класс обслуживания предусмотрен.
  
> [!TIP]
>  Любой трафик, который отправляется в класс обслуживания голосовой связи, превышающий обязательства поставщика услуг, немедленно удаляется, что непосредственно влияет на качество голосовой связи.
  
При взгляде на общий дизайн dewey Law LLC. очень важно точно определить пропускную способность сети, необходимую для поддержки голосового трафика в сети и пометки каждого голосового пакета (и только голосовых пакетов) параметром DSCP для голосовой связи (то есть DSCP EF 46).
  
Чтобы внедрить QoS в корпоративной сети организации, конечные точки или маршрутизаторы должны пометить каждый пакет соответствующим индикатором приоритета уровня 3 (то есть DSCP). На всем сетевом пути для каждого переключателя и маршрутизатора должен быть включен параметр QoS. Если даже один сетевой переключатель или маршрутизатор, не включив QoS, маркировка QoS на голосовых или видеополях, проходящих через этот переключатель или маршрутизатор, может быть отключена. Это фактически отключает QoS во всех низих коммутаторах и маршрутизаторах, что снижает значение expressRoute.
  
Это также требует, чтобы на каждой точке определялась связь приоритетов QoS уровня 3 и 2. Механизмы приоритета уровня 2 определены в IEEE 802.1p для проводных сетей и 802.11e/WMM для Wi-Fi сетей. Что еще более важно, сетевой маршрутизатор для сети MPLS поставщика сетевых услуг должен поддерживать параметры DSCP на всех исходячих пакетах, чтобы они сохраняли соответствующий класс обслуживания MPLS. 
  
> [!TIP]
>  Подробные сведения о настройках QoS можно найти в разделе 2.6 Сетевое планирование, мониторинг и устранение неполадок с [сервером Lync Server.]( https://go.microsoft.com/fwlink/?LinkId=760669) Дополнительные требования к планированию сети см. в Skype для бизнеса [2015.](../../SfbServer/plan-your-deployment/network-requirements/network-requirements.md)
  
### <a name="ordering-network-access-services"></a>Заказ сетевых службы Access

После создания необходимых условий и механизмов QoS для поддержки ExpressRoute необходимо разместить заказ для служб сетевого доступа ExpressRoute. При заказе услуг доступа ExpressRoute для компании Dewey Law LLC у партнера-поставщика сетевых служб Майкрософт необходимо предоставить две вещи:
  
- Общая пропускная способность, необходимая для подключения каждого сайта к ExpressRoute и Microsoft 365 или Office 365.
    
- Общая пропускная способность, необходимая для каждого класса обслуживания, необходимая для поддержки Skype для бизнеса приложений, используемых на dewey Law LLC. Требования к пропускной способности класса обслуживания обусловлены объемом трафика, который ожидается от каждого из приложений Skype для бизнеса, таких как голосовая и видеосвязь, обмен мгновенными сообщениями, присутствие и общий доступ к экрану.

### <a name="the-real-time-problem"></a>Проблема в режиме реального времени

Качество услуг голосовой и видеосвязи требует особых требований к IP-сети. Для передачи данных в режиме реального времени используется протокол RTP, который осуществляется с помощью протокола UDP. В отличие от протокола TCP, который проверяет каждое сообщение на ошибки и включает другие механизмы обнаружения и повторной передачи потерянных или ошибок, UDP не обеспечивает надежность этого типа. Если сообщения повреждены по ошибке или теряются из-за переполнения буфера, они теряются. UDP был выбран для использования с RTP, так как характер трафика в режиме реального времени в том, что даже если потерянные сообщения были повторно отправить, они будут поступать слишком позднее время, чтобы положительно повлиять на поток голосовых сообщений.
  
Зная о последствиях потери голосовых пакетов, разработчики разработали два подхода к повышению производительности голосовой и видеосвязи по IP-адресу:
  
- Сделайте кодирование и декодировку голосовой кодировки более отказоустойчивым в случае потери пакетов. Для этого можно использовать исправление ошибки вперед (FEC) для исправления процентов найденных ошибок , которые находятся в Microsoft 365 или Office 365 real time Transport, или путем разработки систем декодирования голоса, которые пытаются скрыть последствия потери пакетов , что является характеристикой кодеков Майкрософт. 
    
- Используйте транспортные службы, которые используют механизмы качества обслуживания для обеспечения производительности сети в отношении задержки, потери пакетов и дрожания, а также изменения задержки между пакетами.
    
Отказоустойчивый голосовой код позволяет решить проблему потери пакетов, поэтому важно, чтобы в сети, используемой для передачи голосовой и видеосвязи в режиме реального времени, были механизмы, позволяющие уменьшить задержку и дрожание. Даже при отказоустойчивом кодировании в случае потери слишком потери пакетов на принимающих станциях недостаточно информации для восстановления распознаемой версии голосового сигнала. Процент потери пакетов, что может привести к заметному снижению качества голосовой связи, которое зависит от используемого метода кодиирования голоса. Однако в большинстве случаев потерять строки последовательных пакетов очень проблематично.
  
Важно свести к минимуму задержку, так как она может повлиять на поток беседы и привести к помехам для динамиков. Мы сообщите нам о том, что 150 миллисекунд (так называемая задержка в уст-ухе) должна храниться ниже 150 миллисекунд (msecs). задержка в одну сторону, а не "круговая поездка". Конечно, задержка увеличивается при более длинных ссылках передачи, таких как по океану, с учетом задержки распространения или времени, необходимого для физического перемещения сигнала по кабелю.
  
Если задержка превышает 150 мсек. в одну сторону он оказывает странный эффект на динамик. К сожалению, в мозге докладчика отключается часы, из-за чего они могут подумать, что получатель не слышен их и что они повторяют последнее. При этом отложенный ответ будет отложен с самого конца. Если вы когда-нибудь общались над спутниковым каналом, вы узнаете этот эффект. Задержка в одну сторону через спутниковый канал составляет примерно 250 мс, что значительно больше допустимой задержки.

ExpressRoute предоставляет выделенное подключение через поставщика сетевых услуг (NSP) или поставщика Exchange (EXP) одним из трех вариантов подключения: 
  
- Облачное Exchange Colocation
    
- Подключение Ethernet "точка-точка"
    
- Подключение "любой-к-любому" (IPVPN)
    
Это обеспечивает высокую доступность (SLA 99,9 % безотвредных операций) и надежную маршрутизацию, которая защищена (без интернет-транзита), не влияет на различия в интернет-трафике и соблюдает стандарты качества обслуживания для приоритета трафика (QoS объясняется ниже). ExpressRoute вместе с хорошо спланированной сетью WAN обеспечивает голосовую сеть бизнес-класса.
  
ExpressRoute можно использовать для передачи данных из офисов или центра обработки данных (если это гибридная топология), подключенных к каналу. Данные для пользователей вне офиса (например, из домашних офисов, путешествий и т. д.) не будут использовать канал ExpressRoute, если пользователи не подключены к VPN и не должны быть включены в оценку пропускной способности для оценки ширины канала ExpressRoute. Если вы клиент из нескольких стран, вы можете приобрести канал ExpressRoute в каждом регионе и использовать теги сообщества BGP для информирования правил маршрутирования, чтобы трафик направлялся в предпочитаемый канал ExpressRoute (как правило, это ближайший канал для каждого сайта), а другие схемы обеспечивают избыточность в случае простоя, которое влияет на один канал. 
  
### <a name="if-expressroute-isnt-an-option"></a>Если expressRoute не является параметром

Подключение всех сайтов к ExpressRoute может быть нецелесообразным из-за затрат, невозможности выполнить предварительные условия ExpressRoute или ограничений текущего NSP. Если вы не можете использовать ExpressRoute, по-прежнему рекомендуется следовать указаниям ниже для пометки QoS в сети и планирования договоров со своим NSP для обеспечения достаточной пропускной способности и поддержки приоритетов трафика на основе QoS.
  
Кроме того, если у вас есть офисы в разных регионах, но у вас нет каналов ExpressRoute во всех регионах, следует использовать теги сообщества BGP при настройке маршрутиза трафика в/из спутниковых офисов, чтобы избежать ненужных длинных попутных переходов. Например, рассмотрим компанию с организацией Skype для бизнеса Online, которая работает в США, но с филиалами в Европе, а в Этой компании есть только один канал ExpressRoute в Кремниевой долине. Основная часть трафика Skype для бизнеса в Интернете будет перенаться в центр обработки данных, в котором находится организация (например, конференции с другими пользователями в организации), для большинства трафика предпочтительнее использовать канал ExpressRoute. Однако если пользователь из Европы присоединяется к конференции, размещенной в другой компании, размещенной в Европе, то местом назначения для мультимедиа в этом звонке будет европейский центр обработки данных, в котором находится вторая компания. Маршруты трафика через канал ExpressRoute в Кремниевой долине будут менее прямыми, чем через Интернет. В таком случае может потребоваться настроить маршрутизаторы в своей сети (например, в европейских офисах) для проверки тегов сообщества при создании правил маршрутики и маршрутики через Интернет, а не через канал ExpressRoute в Microsoft Valley для трафика с тегами европейских регионов.
  
### <a name="basic-concepts-of-quality-of-service-qosclass-of-service-cos"></a>Основные понятия качества обслуживания (QoS)/Класс обслуживания (CoS)

В IP-адресе качество обслуживания (QoS) описывает любой механизм, используемый для обработки приоритетов для одних пакетов. Согласно определению Международного телекоммуникационного союза (ITU), QoS включает все аспекты качества подключения, включая задержку, потерю, соотношение сигнала и шума, перекрестный сигнал, эхо, прерывания, частоту отклика, уровень громкости и т. д. То, что мы называем QoS в сетях пакетов, правильнее называют классом обслуживания (CoS), который акцентирует внимание на повышении производительности для задержки, дрожания и потери пакетов, но мы продолжим использовать термин QoS по мере его использования.
  
Обеспечение QoS в IP-сети вызывает два основных компонента:
  
- резервирование определенной ширины полосы пропускания для каждой связи для трафика в режиме реального времени; если эта пропускная способность не нужна для трафика в режиме реального времени, ее можно использовать для другого трафика. Общие рекомендации: для голосового трафика должно быть назначено не более 30 % мощности любой ссылки.
    
- Пометка пакетов индикатором приоритета в заголовке, который сообщает переключателям и маршрутизаторам на пути к приоритету пакета, который должен быть назначен.
    
При получении пакета на коммутаторе или маршрутизаторе он перемещается в выходную очередь для следующего отгона или перехода. Существуют разные выходные очереди для разных уровней приоритета. Коммутатор или маршрутизатор использует алгоритм, который чаще всего используется для обслуживания очереди с высоким приоритетом, чем очереди с более низким приоритетом.
  
Проблема в том, что на уровнях 2 (уровень Ethernet или Wi-Fi) и 3 (уровень IP) реализованы различные методы QoS. Эти различные решения QoS могут быть настроены на каждом коммутаторе и маршрутизаторе сети, а также на интерфейсе между вашей сетью и сетью поставщика сетевых услуг.
  
Существует два варианта того, как данные из разных приложений Skype для бизнеса можно соедеть с соответствующими классами обслуживания:
  
- Пометка конечной точки для трафика с помощью контрольной точки DSCP 
    
- Список управления сетевым доступом (ACL)
    
### <a name="end-point-traffic-marking--differentiated-services-control-point-dscp"></a>Контрольная точка маркированного трафика конечных точек — контрольная точка для дифференцированных служб (DSCP)

Маршрутизаторы и другие устройства, реализуют функции уровня 3, определяют приоритет пакета с помощью точки управления DiffServ (DSCP). ### <a name="layer-2-qos-ieee-8021pwi-fi-multi-media-ieee-80211e"></a>QoS уровня 2: IEEE 802.1p/Wi-Fi Multi-Media (IEEE 802.11e)

Хотя DSCP является стандартным механизмом внедрения QoS на уровне 3, существуют различные механизмы QoS уровня 2 для проводных (например, Ethernet) и беспроводных сетей (например, Wi-Fi сетей). Механизм QoS для проводных сетей определен в стандарте IEEE 802.1p. Механизм QoS WLAN определен в IEEE 802.11e, который компания Wi-Fi Alliance определяет как "Wi-Fi Multi-Media Certified" (WMM Certified).
  
IEEE 802.1p использует 3-битную кодовую точку приоритета (PCP) для определения приоритета сообщения; PCP является частью 32-битного поля в загоне Ethernet, которое также содержит идентификатор VLAN. Определения значений PCP приведены ниже.

Внедрение IEEE 802.1p во значительной степени отличается от DSCP с трафиком, отсорт большим количеством очередей приоритетов для каждого уровня приоритета, но общий характер сети WLANs требует другого подхода. Хотя точка доступа и клиент будут поддерживать отдельные очереди выходных данных для разных уровней приоритета, существуют и различия в способе раздатки фреймов по радиоканалу.
  
В сети Wi-Fi все клиенты, связанные с точкой доступа, имеют доступ к одному каналу с полудуплексным каналом (то есть за один раз может отправляться только одна клиентская станция или точка доступа). Чтобы свести к минимуму риск переполохов по радиоканалу, перед отправкой рамки станция будет простаивать в течение определенного периода времени, который называется "Межкадровое интервал", если канал занят при отправке станции, она отключается от случайного периода времени. Если после отправки фрейма отправитель не получает от получателя подтверждение, предполагается, что произошла неудача или другой сбой, а перед попыткой повторной отправки доступа к радиоканалу происходит случайный интервал. Интервал обратного отключения случайный, чтобы снизить вероятность того, что эти две части снова повратятся.
  
Для определения приоритета доступа к радиоканалу IEEE 802.11e/WMM определяет различные интервалы ожидания перед передачей, называемые AFIS (Inter-Frame AFIS), и разные диапазоны открепки для различных классов трафика; определены четыре уровня приоритета под названием "Категории Доступа".
  
Приоритет задается путем назначения более коротких значений AFIS кадрам с более высоким приоритетом. Таким образом, если одна станция ожидает отправки голосовой рамки, а другая ожидает отправки кадра данных, голосовая рамка всегда будет отправляться в первую очередь. Технически для видео- и видеокадров назначено одинаковое значение AFIS, но диапазон отрезков для видеокадров выше. Таким образом, голосовые и видеокадры могут с первого попытки соеденяться, но голосовая рамка всегда будет перенастраивана раньше. Корреляция между IEEE 802.1p и IEEE 802.11e показана ниже.

### <a name="implementing-qos-using-network-access-control-list-acl"></a>Реализация QoS с помощью списка контроля доступа к сети (ACL)

Альтернативный способ внедрения QoS в конфигурации ExpressRoute — использовать список управления сетевым доступом (ACL). В этом подходе вместо того, чтобы конечные точки вставлять соответствующую маркировку DSCP в заглавную часть каждого пакета, маркировку можно делать с помощью потока маршрутизатора, основанного на порте источника UDP. Для сохранения параметров DSCP все переключатели и маршрутизаторы все еще должны быть настроены для поддержки QoS. Что еще более важно, маршрутизатор, подключенный к сети поставщика услуг, должен поддерживать DSCP в заглавной части каждого пакета, так как этот параметр DSCP фактически является инструкцией для поставщика сетевых услуг о том, как следует обрабатывать этот пакет.
  
Рекомендуемые диапазоны портов для каждого приложения Skype для бизнеса перечислены в разделе 2.6.1.1 руководства по планированию, мониторингу и устранению неполадок [с Lync Server.](https://go.microsoft.com/fwlink/?LinkId=690286) Важно, чтобы этот подход координировался с общим подходом организации к QoS, и вы должны быть подытожены на различные политики QoS и возможные несоответствия примечания пакетов.
  
Хотя основной причиной использования QoS и сетевых служб MPLS является обеспечение удобной работы пользователей с голосовой и видеосвязью в режиме реального времени, эти же возможности можно применять и к приложениям для работы с данными. Вместо того чтобы обрабатывать все приложения одинаково, сети MPLS позволяют организациям придать приоритет некоторым приложениям данных по сравнению с другими. При использовании MPLS приложения в режиме реального времени, такие как операции с кредитными картами или общий доступ к экрану, могут иметь приоритет над менее конфиденциальным трафиком, например электронной почтой.
  
### <a name="understanding-the-types-of-ip-network-services--basic-ip-and-mpls"></a>Типы служб IP Network Services — основные IP-адреса и MPLS

Исходная переад управление IP-пакетами выполнялось по принципу "наилучшего усилий". Это означает, что маршрутизаторы, перенаправленные этими IP-пакетами, будут делать все возможное для их доставки в места назначения, но никаких гарантий относительно того, когда и когда они будут доставляться в эти места, не было. Именно так сегодня работают базовые интернет-службы, включая домашнее подключение к Интернету. Идея в том, что если для определенного приложения была необходима надежность, она будет предоставляться на более высоком уровне в стопке протоколов. Надежный механизм доставки — протокол TCP. Протокол user Datagram Protocol (UDP), который используется для передачи голосовой и видеосвязи в режиме реального времени, является ненадежным (то есть "наилучшим образом") механизмом доставки. 
  
Переключение меток с несколькими протоколами (MPLS) было разработано для того, чтобы поставщики сетевых услуг предлагают IP-службы с гарантией производительности в случае задержки, дрожания и потери пакетов. Для обеспечения этих гарантий MPLS обеспечивает непредсказуемость традиционных IP-адресов. Во-первых, MPLS перенаправление всех пакетов по "виртуальному каналу" с фиксированным маршрутом, который называется "Путь с переключателем меток" (LSP), вместо того чтобы каждый пакет находит путь к точке назначения (в результате чего каждый пакет проходит по другому маршруту). Если одна из ссылок на этом пути не справилась, все LSP, использующие ее, быстро перенаправляются.
  
При отправлении пакета в сеть MPLS гранный маршрутизатор поставщика сетевых услуг добавит к пакету дополнительный загон с меткой, которая используется для его перенаправки по соответствующему LSP. На другом конце сети MPLS границы маршрутизатора обрезают метку.
  
Помимо упрощения процесса перенаправления, еще одно преимущество MPLS заключается в том, что система управления сетью будет знать, какие подключения нужно использовать по каждой ссылке в сети. Контроль направления трафика по сети обеспечивается оператором QoS, который обеспечивает каждый из путей. В отличие от производительности традиционных или базовых IP-адресов, операторы MPLS обеспечивают предсказуемую производительность IP-службы. Этот LSP также обеспечивает более безопасную защиту MPLS по своей сути, чем традиционные интернет-службы. Поэтому при использовании базовой службы IP-адресов мы можем надеемся, что сеть будет работать достаточно хорошо, чтобы обеспечить хорошее качество голосовой связи и использовать такие методы, как FEC, и более отказоустойчивый кодирование голосовой связи для повышения шансов, но с помощью MPLS мы можем быть уверены в этом.
  
Поставщики MPLS предлагают несколько классов градиентов обслуживания, которые, к сожалению, используют разные термины для их определения. Вам придется тесно сотрудничать со своим поставщиком, чтобы убедиться, что он понимает выходные данные из [Lync 2010 и 2013 Bandwidth Calculator](https://go.microsoft.com/fwlink/?LinkID=690282) и рекомендуемые параметры для различных приложений Microsoft 365 или Office 365 рабочих нагрузок в режиме реального времени.
  
## <a name="conclusion"></a>Заключение

Skype для бизнеса улучшает взаимодействие в бизнесе. Вместо телефона, подключенного к УАТС, отдельной системы видеоконференций, отдельной платформы для электронной почты, внешней службы для аудиоконференций и некоторого автомобиля для мгновенных сообщений и присутствия, Skype для бизнеса может использовать все эти возможности в едином пользовательском интерфейсе.
  
Чтобы обеспечить постоянное предоставление голосовых и видеослужб бизнес-класса в режиме реального времени, требуется end-to-end network infrastructure, которая поддерживает QoS. Без правильного размера и настройки сетевой инфраструктуры вы рискуете иметь развертывание ExpressRoute Skype бизнеса, которое не будет соответствовать ожиданиям пользователей по качеству и согласованности.
  
Эффективные бизнес-инструменты должны надежно и согласованно работать и обеспечить пользовательский интерфейс, который способствует внедрению пользователей. Планирование, проектирование, внедрение и обслуживание этой инфраструктуры не всегда является простой задачей. Оборудование, инструменты и сетевые службы, доступные на сегодняшний день, но ИТ-Pro должны убедиться, что они разработаны, внедрены и поддерживаются таким образом, чтобы пользователи могли получить набор служб связи и совместной работы, которые позволят им эффективно работать и эффективно работать, а организация сможет извлечь все преимущества этой технологии.
