# <a name="expressroute-and-qos-in-skype-for-business-online"></a>ExpressRoute и качество обслуживания в Skype для бизнеса Online

Подключайтесь к Microsoft 365 или Office 365 по выделенной сетевой сети с помощью Azure ExpressRoute для Microsoft 365 или Office 365 и Skype для бизнеса Online. Выделенное соединение для приложений Skype для бизнеса позволит вам надежную и предсказуемую производительность, а также конфиденциальность не из интернета. Теперь вы можете приобрести лучшее сетевое подключение к Microsoft 365, Office 365 и Skype для бизнеса Online, которое делает его более предсказуемым, надежности бизнес-класса и поставляется с новым SLA.
  
> [!NOTE]
> Доступна новая версия калькулятора пропускной способности: [Skype для бизнеса, Bandwidth Calculator.](https://go.microsoft.com/fwlink/?LinkId=715766) Однако в инструкциях этого документа используется калькулятор пропускной способности для Lync 2010 и 2013. 
  
## <a name="skype-for-business-online-and-expressroute"></a>Skype для бизнеса Online и ExpressRoute

Работая с партнером Microsoft ExpressRoute, вы можете подключать различные приложения Microsoft 365 и Office 365, включая Skype для бизнеса Online в облаке, по выделенному подключению. Однако для голосовой и видеосвязи в режиме реального времени в Skype для бизнеса требуются сетевые службы, специально настроенные для поддержки этих рабочих нагрузок Microsoft 365 или Office 365 в режиме реального времени. Это сеть с достаточной пропускной способностью для выполнения необходимого объема трафика и поддержкой качества обслуживания (QoS) для бизнес-класса пользователей.
  
Этот документ поможет вам, администраторам и дизайнерам сети понять, какие особые проблемы необходимы для связи в режиме реального времени, инструменты, предоставляемые корпорацией Майкрософт для помощи в разработке сети, поддерживающей эти требования, и помогут вам в проектировании с учетом исследования. 
  
В первой части этого документа содержится пример проектирования сети с помощью калькулятора полосы пропускания [Lync 2010 и 2013](https://go.microsoft.com/fwlink/?LinkId=690282) для оценки сетевых требований для развертывания Крупной многоэтапной системы Skype для бизнеса ExpressRoute. Вторая часть этого документа содержит основные понятия качества обслуживания (QoS), подробные технические сведения о поддержке связи в режиме реального времени в Skype для бизнеса и необходимые типы сетевых служб.
  
Вся информация в этой области позволит получить технические сведения и понимание для QoS и ExpressRoute, общее представление о конкретных проблемах, с которыми вы столкнулись, а также предоставить знания о средствах и методах, которые позволят успешно развернуть ExpressRoute в сети Skype для бизнеса.

### <a name="setting-up-the-network"></a>Настройка сети

Для предоставления однообразных и высококачественных услуг в режиме реального времени для компании Dewey Law LLC., необходимо выполнить ряд базовых требований.
  
- Они хотят предоставлять голосовые службы во время сбоя питания, чтобы сетевые коммутаторы и маршрутизаторы сети могли обеспечить питание по сети Ethernet (PoE) IEEE 802.3af или 802.3at.
    
- Сетевые коммутаторы и маршрутизаторы также должны использовать бесперебойные источники питания (UPS), чтобы они могли работать во время сбоя питания.
    
    У них есть подключение Wi-Fi к своим локальной сети, поэтому мы настоятельно рекомендуем им использовать сертифицированного партнера по инфраструктуре Skype для Wi-Fi от [Skype для бизнеса Solutions.](https://go.microsoft.com/fwlink/?LinkId=690281)
    
    > [!TIP]
    >  Рекомендуется использовать беспроводные точки доступа 802.11n и 802.11ac.
  
- И, что самое важное, все сети локальной сети во всех офисах должны быть настроены для обеспечения качества обслуживания (QoS). К ним относятся компьютеры, ноутбуки и любое сетевое оборудование, например коммутаторы и маршрутизаторы.
    
Теперь, когда у вас есть все необходимое для предоставления голосовых служб бизнес-класса для Dewey Law LLC., мы рекомендуем использовать IP-службу MPLS от партнера, который подключается к службе Azure ExpressRoute. MPLS предоставляет IP-службу с гарантией производительности в связи с задержкой, дрожанием и потерей пакетов. Однако если MPLS не доступен, можно использовать Ethernet, подключенный к одному из наших партнеров по обмену данными ExpressRoute.
  
Поставщики MPLS предлагают несколько уровней обслуживания, но для их определения используются разные термины. Чтобы убедиться, что данные, вводимые в калькулятор пропускной способности [Lync 2010 и 2013,](https://go.microsoft.com/fwlink/?LinkID=690282) а также доступные параметры, рекомендуется для различных приложений Microsoft 365 и Office 365 с рабочей нагрузкой в режиме реального времени, вам необходимо тесно с ним поработать.
  
Существует два варианта для того, как данные из приложений Skype для бизнеса можно сопосоставить с соответствующими классами обслуживания MPLS:
  
- Маркировка трафика конечными точками с помощью контрольных точек DiffServ (DSCP)
    
- На основе списка контроля доступа к сети (Network Access Control List, ACL)
    
Чтобы реализовать маркировку конечной точки, необходимо настроить все компьютеры, которые присоединились к домену, для компании Dewey Law LLC. чтобы пометить каждый пакет соответствующей маркировкой DiffServ Control Point (DSCP), а затем внедрить QoS на всех сетевых коммутаторах и маршрутизаторах на всех сайтах офисов компании, чтобы убедиться, что маркировки QoS сохраняются и не удаляются. Маркировки DSCP на сетевых пакетах поймеют поставщика услуг о том, как он имеет приоритет. **Дополнительные сведения о DSCP можно получить в разделе "QoS" части 2.**
  
При назначении через ACL сети маркировки приоритетов DSCP внедряются на маршрутизаторе с последующим выполнением и основаны на порте источника UDP. Рекомендуемые диапазоны портов для каждого приложения перечислены в разделе 2.6.1.1 раздела "Планирование сети", "Мониторинг и устранение неполадок с [Lync Server".](https://go.microsoft.com/fwlink/?LinkId=690286) Очень важно координировать эту работу с общей реализацией и проектированием QoS компании Dewey Law LLC, а также знать о различных политиках QoS и потенциальных несоответствиях маркировки пакетов.
  
У каждого поставщика сетевых служб ExpressRoute будет свой класс обслуживания (QoS), который подходит для голосовой и видеосвязи в режиме реального времени. Это COS называется "Expedited Forwarding" (EF) для голосовой связи и "Assured Forwarding" (AF) для видео. Необходимо внимательно подбирать объем пропускной способности, приобретаемой для трафика голосовой связи EF. Причина состоит в том, что класс обслуживания голосовой почты очень неforgiving в случае, если вы отправляете больше голосового трафика, чем класс обслуживания предусмотрен.
  
> [!TIP]
>  Любой трафик, превышающий обязательства поставщика услуг голосовой связи, немедленно удаляется, что непосредственно влияет на качество голосовой связи.
  
При общем проектировании dewey Law LLC. Очень важно точно определить ширину полосы пропускания, необходимую для поддержки голосового трафика в сети компании и маркировки каждого голосового пакета (и только голосовых пакетов) параметром DSCP для голосовой связи (то есть DSCP EF 46).
  
Чтобы внедрить QoS в корпоративную сеть организации, конечные точки или маршрутизаторы должны пометить каждый пакет соответствующим индикатором приоритета уровня 3 (то есть DSCP). На всех коммутаторах и маршрутизаторах должны быть включены QoS-параметры. Если даже один сетевой коммутатор или маршрутизатор без QoS не включен, маркировка QoS на голосовых и видео пакетах, проходящих через этот коммутатор или маршрутизатор, может быть отключена. Это фактически отключает QoS на всех пониженных коммутаторах и маршрутизаторах, что снижает ценность expressRoute.
  
Кроме того, необходимо определить связь приоритетов QoS уровней 3 и 2 на каждой точке. Механизмы приоритета уровня 2 определены в IEEE 802.1p для проводных сетей и 802.11e/WMM для Wi-Fi сетей. Что еще более важно, сетевой маршрутизатор, находящийся в сети MPLS поставщика сетевых услуг, должен поддерживать параметры DSCP на всех исходячих пакетах, чтобы сохранить соответствующий класс обслуживания MPLS. 
  
> [!TIP]
>  Подробные сведения о настройках QoS можно найти в разделе 2.6 "Сетевое планирование, мониторинг и устранение неполадок с [Lync Server".]( https://go.microsoft.com/fwlink/?LinkId=760669) Дополнительные требования к планированию сети см. в требованиях к планированию сети для Skype для бизнеса [2015.](https://go.microsoft.com/fwlink/?LinkId=690287)
  
### <a name="ordering-network-access-services"></a>Ordering Network службы Access

После создания предварительных условий и механизмов QoS для поддержки ExpressRoute необходимо наказать службы сетевого доступа ExpressRoute. При заказе служб доступа ExpressRoute для компании Dewey Law LLC у партнера-поставщика сетевых служб Майкрософт необходимо предоставить две вещи:
  
- Общая пропускная способность, необходимая для подключения каждого сайта к ExpressRoute и Microsoft 365 или Office 365.
    
- Общая пропускная способность, необходимая для каждого класса обслуживания для поддержки приложений Skype для бизнеса, которые используются в Dewey Law LLC. Требования к полосе пропускания для класса обслуживания определяется объемом трафика, который вы ожидаете от каждого из приложений Skype для бизнеса, таких как голосовая и видеосвязь, обмен мгновенными сообщениями, присутствие и общий доступ к экрану.
    
### <a name="determining-bandwidth-requirements-for-skype-for-business-applications"></a>Определение требований к пропускной способности для приложений Skype для бизнеса

Чтобы определить эти требования на каждой из компания Dewey Law LLC. будут использовать калькулятор пропускной [способности для Lync 2010 и 2013.](https://go.microsoft.com/fwlink/?LinkID=690282) Этот калькулятор — это инструмент на базе Excel, с помощью котором можно задать ожидаемое использование различных приложений Skype для бизнеса, включая голосовую и видеосвязь, а также общий доступ к экрану. Калькулятор автоматически формирует оценку пропускной способности и требований CoS для каждого сайта в сети компании. При загрузке калькулятора полосы пропускания для Lync 2010 и 2013 также скачивается руководство пользователя, в котором вы сможете подробнее его использовать. 
  
Чтобы помочь вам с электронными таблицами, ячейки в ней имеют цветовую кодивную кодию:
  
- **Зеленый** Это области ввода общих данных.
    
- **Желтый** Это расширенные области ввода данных. Вы можете изменить их, но будьте внимательны.
    
- **Красный** Области, вводимые только для чтения, являются заблокированными и не могут быть изменены.
    
- **Серый** Это области, которые будут отображаться только для отображения. Это результаты или данные из общих областей ввода.

Определяя количество пользователей для каждого человека, расположенного на каждом сайте, калькулятор может вычислять общую пропускную способность, необходимую для каждого расположения.
  
Вы также можете указать используемые аудио- и видео кодеки, указать, будет ли использоваться исправление ошибок вперед, а также другие системные параметры, влияющие на требования к пропускной способности. Вы можете использовать стандартные параметры в калькуляторе полосы пропускания Lync 2010 и 2013 или выбрать другие кодеки и другие системные параметры. Для проекта сети Dewey Law LLC можно использовать параметры по умолчанию. Однако для изменения любого из стандартных параметров существует меню со всеми доступными вариантами. Пропускная способность, используемая для каждого выбора, включается в таблицу "Кодеки". ### <a name="the-real-time-problem"></a>Проблема в режиме реального времени

Качество голосовой и видеосвязи требует особых требований к IP-сети. Для передачи данных в режиме реального времени используется протокол RTP (Real-time Transport Protocol, RTP), который передается по протоколу UDP (User Datagram Protocol, UDP). В отличие от протокола TCP, который проверяет каждое сообщение на ошибки и использует другие механизмы обнаружения и повторной передачи потерянных или ошибок сообщений, протокол UDP не обеспечивает надежность этого типа. Если сообщения повреждены ошибками или теряются из-за переполнения буфера, они теряются. UDP была выбрана для использования с RTP, так как трафик в режиме реального времени имеет тот характер, что даже если потерянные сообщения будут отправляться повторно, они будут поступать слишком поздно, чтобы положительно повлиять на поток голосовых сообщений.
  
Зная о последствиях потери голосовых пакетов, разработчики разработали два подхода к повышению производительности голосовой и видеосвязи по IP-адресу:
  
- Сделайте кодирование и декодировку голоса более отказоустойчивым в случае потери пакетов. Для этого можно использовать исправление ошибок (FEC) вперед, чтобы исправить процент найденных ошибок (эта возможность есть в Microsoft 365 или Office 365 Real Time Transport), или с помощью разработки систем декодирования голоса, которые пытаются скрыть последствия потери пакетов (это одна из особенностей работы кодеков Майкрософт). 
    
- Используйте транспортные службы, которые используют механизмы качества обслуживания, чтобы гарантировать производительность сети в связи с задержкой, потерей пакетов и дрожанием, а также изменением задержки пакетов.
    
Отказоустойчивый кодирование голоса решает только проблему потери пакетов, поэтому важно, чтобы в сети, используемой для передачи голосовой и видеосвязи в режиме реального времени, были механизмы, которые минимизируют задержки и дрожание. Даже при отказоустойчивом кодировании в случае потери слишком потери пакетов на приемной станции недостаточно информации для восстановления распознаемой версии голосового сигнала. Процент потери пакетов, что может привести к существенному снижению качества голосовой связи, которое зависит от используемого метода кодиирования голоса. Однако в большинстве случаев потеря цепочки последовательных пакетов будет проблемой.
  
Свести к минимуму задержку очень важно, так как лишние задержки могут повлиять на поток беседы и привести к невыговорам для динамиков. Мы емкие методы, которые говорят о том, что междуговая задержка для голосовой связи (так называемая "удочка") должна храниться не более 150 миллисекунд (МСЕК). однонабоковая, а не "круглая" задержка. Конечно, с учетом задержки распространения или физического перемещения сигнала по кабеля увеличивается задержка передачи данных по таким ссылкам, как в другие океаны.
  
Если задержка превышает 150 мсек. в одну сторону это странное воздействие на динамик. К сожалению, в мозге докладчика отключается часы, из-за чего они думают, что получатель их не услышал, и повторяют последнее. При этом отложенный ответ будет отложен с самого конца. Если вы когда-нибудь проговорили спутниковый канал, вы сможете распознать этот эффект. Задержка в одну сторону примерно через спутниковый канал составляет примерно 250 мсек, что значительно больше допустимой задержки.

### <a name="expressroute-as-part-of-a-business-grade-voice-network"></a>ExpressRoute в составе голосовой сети бизнес-класса

ExpressRoute предоставляет выделенное подключение через поставщика сетевых услуг (NSP) или поставщика Exchange (EXP) в одном из 3 параметров подключения: 
  
- Облачное совместное местонахождение Exchange
    
- Ethernet-подключение "точка-точка"
    
- Подключение "любой к любому" (IPVPN)
    
Это обеспечивает высокую доступность (SLA, 99,9 % в режиме простоя), а также надежную маршрутизацию без интернет-транзита, не зависят от колебаний интернет-трафика и соблюдение стандартов качества обслуживания при расценке трафика (QoS) приведено ниже. ExpressRoute вместе с хорошо спланированной сетью WAN обеспечивает голосовую сеть бизнес-класса.
  
ExpressRoute можно использовать для передачи данных из офисов или центра обработки данных (если гибридная топология), подключенных к каналу. Данные о пользователях вне офиса (например, домашних офисах, в поездках и т. д.) не будут использовать канал ExpressRoute, если только пользователи не подключены к VPN и их не нужно включать в оценку пропускной способности для его оценки. Если вы — клиент с несколькими национальными клиентами, вы можете приобрести схемы ExpressRoute в каждом регионе и использовать теги сообщества BGP для информирования правил маршрутирования, чтобы трафик направлялся в предпочитаемый канал ExpressRoute (обычно это ближайший канал для каждого сайта), а остальные каналы обеспечивают избыточность в случае скайпа, который влияет на один канал. 
  
### <a name="if-expressroute-isnt-an-option"></a>Если ExpressRoute не является параметром

Подключение всех сайтов к ExpressRoute может оказаться нецелесообразным из-за затрат, невозможности выполнить предварительные условия ExpressRoute или ограничений текущего NSP. Если вы не можете использовать ExpressRoute, вам по-прежнему рекомендуется следовать указаниям ниже для пометки QoS в сети и для планирования договоров со своим NSP для обеспечения достаточной пропускной способности и поддержки расстояния приоритетов трафика на основе QoS.
  
Кроме того, если у вас есть офисы в разных регионах, но нет каналов ExpressRoute во всех регионах, при настройке маршрутиза трафика в спутниковые офисы и из них следует использовать теги сообщества BGP, чтобы избежать ненужных длинных маршрутов общественного транспорта. Например, рассмотрим организацию Skype для бизнеса Online, которая работает в США, но имеет филиалы в Европе, а в Кремниевой долине имеется только один канал ExpressRoute. Основная часть трафика Skype для бизнеса Online будет маршрутизации в центр обработки данных, в котором находится организация (например, конференции с другими пользователями в организации), для большинства трафика предпочтительнее использовать канал ExpressRoute. Однако если пользователь из Европы присоединится к конференции, размещенной другой компанией, размещенной в Европе, то местом назначения для мультимедиа в этом звонке будет европейский центр обработки данных, в котором находится вторая компания. Маршрутия трафика через канал ExpressRoute в Кремниевой долине будет менее прямой, чем это возможно через Интернет. В этом случае вам может потребоваться настроить маршрутизаторы в своей сети (например, в европейских офисах) для проверки тегов сообщества при создании правил маршрутики, а также маршрутов через Интернет, а не через канал ExpressRoute в Кремниевой долине для трафика, который имеет теги европейских регионов.
  
### <a name="basic-concepts-of-quality-of-service-qosclass-of-service-cos"></a>Основные понятия качества обслуживания (QoS) и класса обслуживания (CoS)

В IP-адресе качество обслуживания (QoS) описывает механизм, используемый для обработки пакетов по приоритету перед другими. Согласно определению, определению Международного телекоммуникационного союза (ITU), качество соединения зависит от задержки, потерь, коэффициента сигнала к шуму, перекрестного сигнала, эха, прерывания, частотного отклика, уровней громкости и т. д. В пакетных сетях qoS более правильно называют классом обслуживания (CoS), который акцентирует внимание на повышении производительности в условиях задержки, дрожания и потери пакетов, но мы продолжим использовать термин QoS как наиболее распространенный.
  
Обеспечение QoS в IP-сети вызывает два основных компонента:
  
- резервирование определенной ширины полосы пропускания для каждой связи для трафика в режиме реального времени; если эта пропускная способность не нужна для трафика в режиме реального времени, ее можно использовать для другого трафика. Общие рекомендации в том, что для голосового трафика должно быть назначено не более 30 % мощности любой связи.
    
- Пометка пакетов индикатором приоритета в заголовке, который указывает коммутаторам и маршрутизаторам в пути на приоритет пакета, который нужно наставить.
    
При получении пакета на коммутаторе или маршрутизаторе он перемещается в выходную очередь для следующей разгона или перехода. Существуют разные выходные очереди для разных уровней приоритета. Коммутатор или маршрутизатор использует алгоритм, который чаще всего используется для обслуживания очереди с более высоким приоритетом, чем очереди с более низким приоритетом.
  
Проблема заключается в том, что на уровнях 2 (уровень сети Ethernet или Wi-Fi) и 3 (уровень IP-сети) реализованы разные методы QoS. Эти разные системы QoS могут быть настроены на каждом коммутаторе и маршрутизаторе сети, а также на интерфейсе между вашей сетью и сетью поставщика сетевых услуг.
  
Существует два варианта для того, как данные из различных приложений Skype для бизнеса можно сопосоставить с соответствующими классами обслуживания:
  
- Маркированный трафик с помощью контрольных точек дифференцированного обслуживания (Differentiated Services Control Point, DSCP) 
    
- Список контроля доступа к сети (Network Access Control List, ACL)
    
### <a name="end-point-traffic-marking--differentiated-services-control-point-dscp"></a>Маркированный трафик на конечных точках — контрольные точки дифференцированного обслуживания (DSCP)

Маршрутизаторы и другие устройства, реализуют функции уровня 3, определяют приоритет пакета с помощью контрольных точек DiffServ (DSCP). QoS реализуется путем вставки 6-битного значения DSCP в поле Differentiated Services (ранее — "Тип услуги") в IP-закадров. Для 6-битных номеров можно использовать 64 уровня приоритета. ### <a name="layer-2-qos-ieee-8021pwi-fi-multi-media-ieee-80211e"></a>QoS уровня 2: IEEE 802.1p/Wi-Fi Multi-Media (IEEE 802.11e)

Хотя DSCP является стандартным механизмом реализации QoS на уровне 3, существуют разные механизмы QoS уровня 2 для проводных (например, Ethernet) и беспроводных сетей (например, Wi-Fi сетей). Механизм QoS для проводных сетей определен в стандарте IEEE 802.1p. Механизм QoS WLAN определен в IEEE 802.11e, который компания Wi-Fi Alliance определяет как "Wi-Fi Multi-Media Certified" (WMM Certified).
  
IEEE 802.1p использует 3-битовую кодовую точку приоритета (PCP) для определения приоритета сообщения. PCP является частью 32-битного поля в загоне Ethernet, которое также содержит идентификатор VLAN. Определения значений PCP приведены ниже.

|7  <br/> |7  <br/> |NC  <br/> |Сетевой контроль  <br/> |
|6  <br/> |6  <br/> |IC  <br/> |Internetwork Control  <br/> |

В то время как реализация IEEE 802.1p во значительной степени отличается от DSCP с трафиком, отсортным в разные очереди для каждого уровня приоритета, а общий доступ к сети мультимедиа требует другого подхода. Несмотря на то, что точка доступа и клиент поддерживают отдельные очереди для разных уровней приоритета, существуют отличия в том, как кадры отправляются по радиоканалу.
  
В сети Wi-Fi все клиенты, связанные с точкой доступа, имеют один канал полудуплексной связи (то есть за один раз можно отправлять только один клиентский канал или точку доступа). Чтобы свести к минимуму вероятность переключежений на радиоканал, перед отправкой рамки станция ждет, что канал будет неающим в течение определенного периода времени, который называется "Межкадровое интервал", если канал занят при отправке станции, он отключает случайный период времени. Если отправитель не получает сообщение с подтверждением от получателя, то при отправке кадра предполагается, что произошла неудача или произошла другая ошибка, а перед попыткой доступа к радиоканалу для повторной отправки происходит случайный интервал. Промежуток времени отрезка случайный, чтобы снизить вероятность того, что две пунктов снова повернуются.
  
Для определения приоритета доступа к радиоканалу в IEEE 802.11e/WMM определены различные интервалы ожидания передачи данных, называемые "Третейские Inter-Frame Интервалы" (AFIS), и разные диапазоны отстояния для различных классов трафика; Определены четыре уровня приоритета, называемые "категориями Access".
  
Приоритет задается путем назначения более коротких значений AFIS кадрам с более высоким приоритетом. Таким образом, если одна станция ожидает отправки кадра голосовой почты, а другая — кадр данных, голосовая рамка всегда будет отправлена в первую очередь. Технически голосовым и видеокадрам назначено одинаковое значение AFIS, но диапазон отрезков выше для видеокадров. Таким образом, голосовые и видеокадры могут с первого попытки соеденяться, но голосовая рамка всегда будет раньше перенастраивана. Корреляция между IEEE 802.1p и IEEE 802.11e показана ниже:

### <a name="implementing-qos-using-network-access-control-list-acl"></a>Реализация QoS с помощью списка контроля доступа к сети (Network Access Control List, ACL)

Альтернативный способ внедрения QoS в конфигурации ExpressRoute — использовать список контроля доступа к сети (Network Access Control List, ACL). Таким образом, вместо того чтобы вставлять соответствующую маркировку DSCP в заглавную часть каждого пакета, маркировку можно проходить с помощью потока маршрутизатора на основе порта источника UDP. Для сохранения параметров DSCP все коммутаторы и маршрутизаторы все еще должны быть настроены для поддержки QoS. Что еще более важно, маршрутизатор, подключенный к сети поставщика услуг, должен поддерживать DSCP в header каждого пакета, так как этот параметр DSCP — это, по сути, инструкция для поставщика сетевых услуг о том, как следует обрабатывать этот пакет.
  
Рекомендуемые диапазоны портов для каждого приложения Skype для бизнеса перечислены в разделе 2.6.1.1 руководства по планированию, мониторингу сети и устранению неполадок [с Lync Server.](https://go.microsoft.com/fwlink/?LinkId=690286) Важно, чтобы эта политика была согласована с общим подходом организации к QoS, и вам следует искать различные политики QoS и возможные несоответствия примечания пакетов.
  
Хотя основной причиной использования QoS и сетевых служб MPLS является обеспечение удобной работы пользователей с голосовой и видеосвязью в режиме реального времени, эти же возможности можно применять и к приложениям для работы с данными. Вместо того чтобы обрабатывать все приложения одинаково, сети MPLS позволяют организациям придать приоритет приложениям данных по сравнению с другими. При использовании MPLS приложения в режиме реального времени, такие как операции с кредитными картами или общий доступ к экрану, могут иметь приоритет над менее конфиденциальным трафиком, например электронной почтой.
  
### <a name="understanding-the-types-of-ip-network-services--basic-ip-and-mpls"></a>Типы служб IP Network Services ( Основные IP-адреса и MPLS)

Исходная переад управление IP-пакетами выполнялось по принципу "наилучшего усилий". Это означало, что маршрутизаторы, перенаправленные IP-пакеты, будут делать все возможное для их доставки в назначения, но совершенно не было никаких гарантий того, когда и когда они будут доставляться в эти страны. Именно так сейчас работают базовые интернет-службы, включая домашнее подключение к Интернету. Идея была в том, что если для определенного приложения была необходима надежность, оно будет предоставляться на более высоком уровне в стеке протоколов. Механизм надежной доставки — протокол TCP. Протокол передачи данных пользователя (UDP), который используется для передачи голосовой и видеосвязи в режиме реального времени, является ненадежным (то есть "наилучшим образом") механизмом доставки данных. 
  
С помощью MPLS поставщики сетевых услуг могут предлагать IP-службы с гарантией производительности в случае задержки, дрожания и потери пакетов. Для обеспечения таких гарантий MPLS обеспечивает непредсказуемость традиционных IP-адресов. Во-первых, mPLS перенаправление всех пакетов по "виртуальному каналу" с фиксированным маршрутом, который называется путем перенаправления меток (LSP). Если одна из ссылок на этом пути не справилась, все LSP, использующие ее, быстро перенаправляются.
  
При отправлении пакета в сеть MPLS гранный маршрутизатор сети поставщика услуг добавит к пакету дополнительный заглавный за header, который включает метку, которая используется для его перенаправки по соответствующему LSP. Гранный маршрутизатор на другом конце сети MPLS обрезает метку.
  
Еще одно преимущество MPLS состоит в том, что система управления сетью не только упрощает этот процесс, но и знает, какие подключения нужно использовать для каждой связи в сети. Контролируя способ перенаправки трафика по сети, оператор может гарантировать, что QoS будет предоставлять каждый из путей. В отличие от производительности традиционных или базовых IP-адресов, операторы MPLS обеспечивают предсказуемую производительность IP-службы. Он также делает MPLS более безопасным по своей природе, чем обычные интернет-службы. Так что с помощью базовой службы IP-адресов мы можем надеемся, что сеть будет работать достаточно хорошо, чтобы обеспечить высокое качество голосовой связи и использовать такие методы, как FEC и более отказоустойчивый кодирование голосовой связи, чтобы повысить вероятность этого, но с помощью MPLS мы можем быть уверены в этом.
  
Поставщики MPLS предлагают несколько классов градиентов обслуживания, которые, к сожалению, используют разные термины для их определения. Вам необходимо тесно сотрудничать с поставщиком, чтобы убедиться, что он понимает выходные данные с помощью калькулятора пропускной способности [Lync 2010 и 2013,](https://go.microsoft.com/fwlink/?LinkID=690282) а также рекомендуемых параметров для различных приложений Microsoft 365 или Office 365 Real Time Workloads.
  
## <a name="conclusion"></a>Заключение

Skype для бизнеса улучшает деловое взаимодействие. Вместо телефона, подключенного к УАТС, отдельной системы видеоконференций, отдельной платформы для электронной почты, внешней службы для аудиоконференций и некоторых автомобилей для мгновенных сообщений и присутствия, Skype для бизнеса объединяет все эти возможности в едином пользовательском интерфейсе.
  
Постоянное предоставление голосовых и видеослужб бизнес-класса в режиме реального времени требует от вас инфраструктуры сети, обеспечиваемой QoS. Это относится как к службам локальной сети, так и к службам WAN. Корпорация Майкрософт предоставляет такие инструменты, как калькулятор полосы пропускания [для Lync 2010 и 2013,](https://go.microsoft.com/fwlink/?LinkID=690282) для оценки пропускной способности сети, необходимой для различных служб. Кроме того, в программе IT Pro Tools Skype для бизнеса Solutions есть партнеры: it [Pro Tools,](https://go.microsoft.com/fwlink/?LinkID=690307) которые предлагают инструменты для предварительной оценки инфраструктуры сети и поддержки мониторинга, отчетности и устранения неполадок. Если у вас нет правильной и настроенной сетевой инфраструктуры, вы рискуете, что развертывание ExpressRoute Skype для бизнеса не будет соответствовать ожиданиям пользователей по качеству и согласованности.
  
Эффективные бизнес-инструменты должны работать надежно, согласованно и обеспечить пользовательский интерфейс, который способствует внедрению пользователей. С точки зрения сети это означает наличие инфраструктуры сети (локальной и широкой, фиксированной и мобильной), которая позволяет это сделать. Планирование, проектирование, внедрение и обслуживание этой инфраструктуры не всегда легко доработать. Оборудование, инструменты и сетевые службы, доступные на сегодняшний день, но ИТ-профессионалы должны видеть, что они разработаны, реализованы и реализованы таким образом, что пользователи получат набор служб для связи и совместной работы, которые позволят им работать эффективно и эффективно и что организация сможет воспользоваться полной выгодой от того, что может предложить эта технология.
