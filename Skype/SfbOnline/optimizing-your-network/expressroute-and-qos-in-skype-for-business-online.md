### <a name="setting-up-the-network"></a>Настройка сети

Для обеспечения совместимости и высококачественных служб в реальном времени для Девэй закона LLC, необходимо соблюдать следующие основные требования:
  
- Они хотят предоставлять услуги голосовой связи в случае сбоя электросети, чтобы сетевые коммутаторы и маршрутизаторы могли поддерживать Power over Ethernet (PoE) IEEE 802.3 AF или 802.3 at.
    
- Сетевые коммутаторы и маршрутизаторы также должны использовать непрерывные источники питания (ИБП), чтобы они могли продолжать работу во время сбоя питания.
    
    У них есть подключения Wi-Fi к своим офисам локальных сетей, поэтому мы настоятельно рекомендуем использовать сертифицированный партнер по инфраструктуре Wi-Fi в Skype для бизнеса [для решений Skype для бизнеса](https://go.microsoft.com/fwlink/?LinkId=690281).
    
    > [!TIP]
    >  рекомендуются точки доступа к беспроводной сети 802.11 n и 802.11 AC.
  
- И, что важнее всего, все сетевые сети во всех офисах должны быть настроены для обеспечения качества обслуживания (QoS). Сюда входят компьютеры, Ноутбуки и сетевое оборудование, например коммутаторы и маршрутизаторы.
    
Теперь, когда у вас есть основы, для доставки служб голосовой связи по Девэй закону LLC. Мы рекомендуем использовать IP-службу коммутации наклеек с несколькими протоколами (MPLS) от сетевого партнера, который будет подключаться к службе Azure ExpressRoute. MPLS предоставляет IP-службу с гарантированной производительностью для задержек, колебаний и потери пакетов. Тем не менее, если MPLS недоступен, возможно также использование Ethernet, подключенного к одному из наших партнеров обмена данными ExpressRoute.
  
Поставщики MPLS предлагают различные уровни обслуживания, но каждый из них использует разные термины для их идентификации. Вам придется тесно работать с поставщиком, чтобы убедиться в том, что данные введены в [калькуляторе Lync 2010 и 2013](https://go.microsoft.com/fwlink/?LinkID=690282) с пропускной способностью и доступные параметры, и рекомендуются для разных рабочих нагрузок Office 365 в реальном времени. применения.
  
Существует два способа сопоставления данных из приложений Skype для бизнеса с подходящими классами служб MPLS:
  
- Пометка трафика конечной точки с помощью контрольной точки DiffServ (DSCP)
    
- На основе списка управления доступом к сети (ACL)
    
Для реализации пометки конечной точки необходимо настроить все домены, присоединенные к Девэй закону LLC. чтобы пометить каждый пакет с помощью соответствующей отметкой контрольной точки DiffServ (DSCP) и затем внедрить его на все сетевые коммутаторы и маршрутизаторы на всех сайтах Office, убедитесь в том, что метки QoS поддерживаются и не удаляются. Маркировка DSCP в сетевых пакетах указывает поставщику услуг, каким образом приоритет этого сетевого пакета. **Дополнительные сведения о DSCP можно получить в разделе QoS, часть 2.**
  
Для назначения на основе ACL в сети на вышестоящем маршрутизаторе реализуются обозначения приоритета DSCP и они основываются на UDP-порте источника. Рекомендуемые диапазоны портов для каждого приложения перечислены в разделе 2.6.1.1 [сетевого планирования, мониторинга и устранения неполадок с помощью Lync Server](https://go.microsoft.com/fwlink/?LinkId=690286). Важно, чтобы вы координировать это с помощью Девэй закона LLC общей реализации и разработки QoS и осведомленности о различных политиках качества обслуживания и возможностях пометки несовпадений пакетов.
  
Каждый поставщик сетевых услуг ExpressRoute будет иметь класс обслуживания (QoS), подходящий для голосовой связи и видео в режиме реального времени. Эта COS называется "упрощенная переадресация" (EF) для голоса и "гарантированная переадресация" (AF) для видео. Вы должны быть особенно внимательны при изменении размера пропускной способности, которую вы приобрели для передачи данных по каналу связи. Причина в том, что класс голосовой связи в событии очень унфоргивинг в том случае, если вы отправляете больше всего потокового трафика по сравнению с классом службы.
  
> [!TIP]
>  Весь трафик, который отправляется на голосовую почту службы сверх обязательства поставщика услуг, немедленно удаляется, что позволяет напрямую отменять качество голоса.
  
При рассмотрении общего дизайна Девэй закона LLC. очень важно точно определить объем пропускной способности сети, необходимый для поддержки голосового трафика в своей сети, и помечать каждый из них (и только голосовые пакеты) с помощью параметра DSCP для Voice (например, DSCP EF 46).
  
Чтобы реализовать QoS в корпоративной сети, конечные точки или маршрутизаторы должны помечать каждый из пакетов соответствующим индикатором приоритета уровня 3 (например, DSCP). На каждом переключателе и на маршрутизаторе должен быть включен параметр QoS для всего сетевого пути. Если у вас есть только один сетевой коммутатор или маршрутизатор, на котором не включена служба QoS, Маркировка QoS для голосовых и видеофайлов, передаваемых через этот коммутатор или маршрутизатор, может быть отключена. Это позволяет отключить QoS во всех нижестоящих коммутаторах и маршрутизаторах, которые применяют значение ExpressRoute.
  
Для этого также необходимо, чтобы в каждой точке была определена связь уровней приоритетов QoS и слоев 3 и 2. Механизмы приоритета второго уровня определяются в стандартах IEEE 802.1 p для проводных сетей и 802.11 e/WMM для сетей Wi-Fi. Что важнее, сетевой маршрутизатор, развернутый в сети поставщика сетевых услуг, должен поддерживать параметры DSCP для всех исходящих пакетов, чтобы они поддерживали соответствующий класс обслуживания MPLS. 
  
> [!TIP]
>  Конкретные сведения о настройке QoS можно найти в разделе [планирование сети 2,6, наблюдение и устранение неполадок с помощью Lync Server]( https://go.microsoft.com/fwlink/?LinkId=760669). Вы также можете просмотреть [требования к сети для Skype для бизнеса 2015](https://go.microsoft.com/fwlink/?LinkId=690287) , чтобы найти требования к сетевому планированию.
  
### <a name="ordering-network-access-services"></a>Упорядочение служб сетевого доступа

После того как у вас есть требования к сети QoS и механизмы поддержки ExpressRoute, следующий шаг — поместить заказ для служб доступа к сети ExpressRoute. При заказе служб Access ExpressRoute для Девэй закона LLC от партнера Microsoft Network Services Provider вам потребуется выполнить два действия:
  
- Общий объем пропускной способности, необходимый для подключения каждого сайта к ExpressRoute и Office 365.
    
- Общая пропускная способность требуется каждому классу услуг, необходимых для поддержки приложений Skype для бизнеса, которые используются на Девэй закону LLC. Классическое требование к пропускной способности службы определяется объемом трафика, который ожидается в разных приложениях Skype для бизнеса, таких как голосовая связь, видео, мгновенные сообщения, присутствие и демонстрация экрана.

Для определения требований на каждом из Девэй закона LLC. сайты, вы будете использовать [Калькулятор Lync 2010 и 2013](https://go.microsoft.com/fwlink/?LinkID=690282)для пропускной способности. При загрузке калькулятора пропускной способности Lync 2010 и 2013 Вы также сможете загрузить руководство пользователя, в котором вы сможете получить подробные сведения об использовании этого приложения.

- **Yellow (желтый** ) Это дополнительные области ввода данных. Вы можете изменить эти настройки, но это нужно осторожно.

- **Серый цвет** Эти области доступны только для отображения. Они являются результатами или данными из общих областей ввода.

Определив количество пользователей для каждого из сайтов, калькулятор может вычислять общую пропускную способность, необходимую для каждого местоположения.
  
Вы также можете указать используемые звуковые и видеокодеки, а также использовать ли исправление ошибок переадресации и другие системные параметры, влияющие на требования к пропускной способности. Вы можете использовать параметры по умолчанию в калькуляторе Lync 2010 и 2013 с пропускной способностью либо выбрать другие кодеки и другие системные параметры. Для дизайна сайта Девэй LLC можно использовать параметры по умолчанию. Однако для выбора любого из параметров по умолчанию существует раскрывающееся меню со всеми доступными параметрами. Пропускная способность, используемая для каждого выбора, включается на лист кодека. При изменении настроек пропускной способности и набора служб (CoS) на каждом сайте обновляется. ### <a name="the-real-time-problem"></a>Проблема в реальном времени

Предоставление услуг голосовой и видеосвязи по деловым отношениям обеспечивает специальные требования в IP-сети. Трафик в реальном времени использует транспортный протокол (RTP), который переносится по протоколу UDP. В отличие от протокола TCP, в котором номера и тестирование всех сообщений на наличие ошибок и другие механизмы обнаружения и повторной передачи потерянных или ошибочных сообщений, UDP не обеспечивает для этого типа надежность. Если сообщения повреждаются с ошибками или теряются из-за переполнения буфера, они теряются. UDP был выбран для использования с протоколом RTP, так как характер трафика в реальном времени состоит в том, что даже если потерянные сообщения были повторно отправлены, они приступили к уже слишком поздно, чтобы не повлиять на поток голосовой почты.
  
Учитывая влияние потерянных голосовых пакетов, разработчики построили два подхода к повышению эффективности голосовой и видеосвязи по протоколу IP:
  
- Обеспечьте более устойчивое кодирование и декодирование речи, когда пакеты будут потеряны. Это можно сделать с помощью функции пересылки ошибки переадресации (FEC), чтобы исправить процент обнаруженных ошибок, которые могут быть обнаружены в приложении Office 365 в реальном времени, или разрабатывать системы для раскодирования голосовых данных, которые пытаются маскировать эффект потерянных пакетов, которые — Это характеристика кодеков Microsoft. 
    
- Используйте транспортные службы, которые используют механизмы качества обслуживания для обеспечения производительности сети с точки зрения задержки, потери и колебания пакетов и варианта задержки между пакетами.
    
Устойчивая Голосовая кодировка решает проблему потери пакетов, поэтому важно, чтобы в сети, используемой для передачи голоса и видео в реальном времени, были механизмы, уменьшающие задержку и колебанию. Даже при использовании отказоустойчивого кода, если потеряно слишком много пакетов, принимающая станция не сможет получить достаточно сведений для восстановления распознаваемой версии голосового сигнала. Процент потерянных пакетов, которые приводят к заметному ухудшению качества звука, в зависимости от используемого метода кодирования голосовой связи. Однако в некоторых случаях потеряны строки последовательных пакетов очень проблематично.
  
Минимизация задержки очень важна, так как чрезмерная задержка может повлиять на поток разговора и создать аннойанце для динамиков. Рекомендации, которые указывают на то, что обратная задержка для голоса (то, что мы будем использовать в качестве задержки), должна быть меньше 150 миллисекунд (МС). односторонняя, а не "круговой" задержкой. Разумеется, задержка будет увеличиваться на более длинных ссылках, например, для тех, которые переходят за пределы океана, с учетом задержки распространения или времени, затрачиваемого на физическое перемещение сигнала по кабелю.
  
Если задержка выходит более чем на 150 мс. одностороннее, на динамике есть странный эффект. Говоря логически, часы в определенном состоянии выводятся в мозге динамика, и они считают, что получатель не слышал их, и они повторяют Последнее произнесенное ими. Это конфликтует с отложенным ответом, поступающей от дальнего конца. Если вы когда-либо находились по спутниковому каналу, вы будете знать этот эффект. По каналу спутниковой связи, односторонняя задержка составляет примерно 250 мс., что значительно выходит за пределы допустимой задержки.

### <a name="expressroute-as-part-of-a-business-grade-voice-network"></a>ExpressRoute в составе голосовой сети Business TIGI2U

ExpressRoute — это выделенное подключение с помощью поставщика сетевых услуг (НСП) или поставщика Exchange (EXP) в одном из трех вариантов подключения. 
  
- Сорасположение в облаке Exchange
    
- Ethernet-соединение «точка за точку»
    
- Любое подключение (IPVPN)
    
Это обеспечивает преимущества высокого уровня доступности (соглашения об уровне обслуживания 99,9%), а также надежной маршрутизации, которая является безопасной (без Интернет-транзита), не зависит от вариаций трафика в Интернете и учитывает качество обслуживания для определения приоритета трафика (качество обслуживания рассматривается ниже). . ExpressRoute, а также с помощью высокозапланированной глобальной сети может предоставить вам услуги голосовой связи для бизнеса.
  
Вы можете использовать ExpressRoute для передачи данных из офисов или центров обработки данные (если используется гибридная топология), подключенных к каналу. Данные для внешних пользователей (например, из домашних и прошедших) не будут использовать цепь ExpressRoute, если только пользователи не подключены к сети, и их не нужно включать в оценку пропускной способности для изменения размера канала ExpressRoute. Если вы используете многонациональной клиент, вы можете приобрести цепи ExpressRoute в каждом регионе и использовать теги сообщества BGP для информирования правил маршрутизации, чтобы трафик направлялся на предпочтительную цепь ExpressRoute (как правило, это ближайший канал для каждого сайта), а другой цепь предлагают избыточность в случае сбоя, влияющего на единую цепь. 
  
### <a name="if-expressroute-isnt-an-option"></a>Если не удается выбрать параметр ExpressRoute

Возможно, вам не удастся подключить все сайты к ExpressRoute из-за расходов, невозможности выполнить необходимые требования к ExpressRoute или ограничения текущего НСП. Если вы не можете использовать ExpressRoute, вы по-прежнему рекомендуете следовать рекомендациям ниже, чтобы пометить качество обслуживания в сети, а также спланировать контракты с помощью НСП, чтобы обеспечить достаточную пропускную способность и поддержку приоритизации трафика на основе QoS.
  
Кроме того, если у вас есть офисы в нескольких регионах, но у вас нет каналов ExpressRoute в разных регионах, следует использовать теги сообщества BGP в области для настройки маршрутизации для трафика между вспомогательными офисами, чтобы избежать нежелательной продолжительности передачи. Например, предположим, что компания, в которой есть организация Skype для бизнеса Online, размещена в США, но с филиалами в Европе, а у компании есть только одна цепь ExpressRoute в Silicon впадины. Большая часть трафика Skype для бизнеса Online будет перенаправляться в центр обработки данных, на котором размещается организация (например, конференции с другими пользователями в компании), и для большинства трафиков лучше использовать цепь ExpressRoute. Тем не менее, если пользователь в Европе присоединяется к Конференции, размещенной в другой компании, организация которой расположена в Европе, конечный файл для мультимедиа в этом звонке — это Европейский центр обработки данных, на котором расположена вторая компания. Маршрутизация трафика через цепь ExpressRoute в Silicon впадин является менее прямым маршрутом, чем это возможно через Интернет. В таком случае вы можете настроить маршрутизаторы в сети (например, в европейских офисах) для проверки тегов сообщества при создании правил и маршрутизации через Интернет, а не с канала Silicon впадин ExpressRoute для трафика с Теги Европейского региона.
  
### <a name="basic-concepts-of-quality-of-service-qosclass-of-service-cos"></a>Основные принципы обслуживания качества обслуживания (QoS)/класс (CoS)

В IP качество обслуживания (QoS) описывает любой механизм, который используется для предоставления приоритетов для некоторых пакетов по сравнению с другими. В соответствии с определением международной связи между коммуникациями (ITU) качество обслуживания включает все аспекты качества подключения, включая задержку, потерю, пропорции сигнала и шума, кроссталк, эхо, прерываний, частоту отклика, уровень громкости и т. д. То, что мы используем как QoS в сетях пакетов, более правильно обозначает класс Service (CoS), который фокусируется на улучшении производительности для задержки, колебаний и потери пакетов, но мы будем использовать термин QoS, так как он используется чаще всего.
  
Предоставление QoS в IP-сети вызывает два основных компонента:
  
- Резервирование определенного количества пропускной способности на каждой ссылке для передачи данных в реальном времени; Если эта пропускная способность не требуется в реальном времени, она может быть использована для другого трафика. Общее руководство: не более 30% мощности ссылки должны быть назначены для голосового трафика.
    
- Помечайте пакеты с индикатором приоритета в заголовке, который сообщает переключателям и маршрутизаторам в пути, какой приоритет должен быть назначен пакет.
    
При получении пакета на коммутаторе или маршрутизаторе он перемещается в очередь вывода для следующего места или прыжка. Существуют разные очереди вывода для разных уровней приоритета. Коммутатор или маршрутизатор использует алгоритм, который служба очереди с высоким приоритетом работает чаще, чем очереди с более низким приоритетом.
  
Проблема в том, что существуют различные методы QoS, реализованные на уровне 2 (например, Ethernet или Wi-Fi Layer) и уровня 3 (например, уровень IP). Эти различные реализации QoS могут быть настроены для каждого коммутатора и маршрутизатора в сети, а также интерфейса между вашей сетью и сетью поставщика сетевых услуг.
  
Существует два способа сопоставления данных из различных приложений Skype для бизнеса с соответствующими классами обслуживания.
  
- Пометка трафика на конечных точках с помощью дифференцированной контрольной точки служб (DSCP) 
    
- На основе списка управления доступом к сети (ACL)

### <a name="layer-2-qos-ieee-8021pwi-fi-multi-media-ieee-80211e"></a>QoS уровня 2: IEEE 802.1 p/Wi-Fi Multi-Media (IEEE 802.11 e)

Несмотря на то, что DSCP является стандартным механизмом реализации QoS на уровне 3, для проводной связи (например, Ethernet) и беспроводного доступа (например, сети Wi-Fi) есть различные механизмы QoS уровня 2. Механизм QoS для проводных сетей определен в стандарте IEEE 802.1 p; механизм QoS для беспроводных сетей определен в стандарте IEEE 802.11 e, а также о том, как Wi-Fi Alliance идентифицирует "Wi-Fi Multi-Media" (заверенный сертификат).

Пери Переключател. После отправки кадра, если отправитель не получает от получателя сообщение о подтверждении, предполагается, что возникла проблема или другой сбой, и она перейдет за случайный интервал времени, прежде чем пытаться получить доступ к радиоканалу для повторной отправки. ### <a name="implementing-qos-using-network-access-control-list-acl"></a>Реализация QoS с помощью списка управления доступом к сети (ACL)

Альтернативным способом реализации QoS в конфигурации ExpressRoute является использование списка управления доступом к сети (ACL). В таком подходе вместо того, чтобы конечные точки вставляют соответствующие метки DSCP в заголовок каждого пакета, пометка может выполняться вышестоящим маршрутизатором на основе порта источника UDP. Все переключатели и маршрутизаторы по-прежнему должны поддерживать QoS для обеспечения обслуживания параметров DSCP. Что еще важнее, маршрутизатор, подключенный к сети поставщика услуг, должен поддерживать DSCP в заголовке каждого пакета, так как этот параметр DSCP является инструкцией для поставщика сетевых услуг о том, как следует обрабатывать этот пакет.
  
Рекомендуемые диапазоны портов для каждого приложения Skype для бизнеса указаны в разделе 2.6.1.1 [сетевого планирования, мониторинга и устранения неполадок с помощью руководства по Lync Server](https://go.microsoft.com/fwlink/?LinkId=690286) . Важно, чтобы эта возможность была согласована с общим подходом к QoS для Организации, и вы должны находиться в Лукаут для разных политик качества обслуживания и возможностях пометки несовпадений пакетов.
  
Несмотря на то, что используются основные сетевые службы QoS и MPLS, чтобы обеспечить оптимальное взаимодействие пользователей с голосовыми и видеофайлами в режиме реального времени, эти же возможности также можно применять к приложениям для работы с данными. Вместо того чтобы рассматривать все приложения одинаково, MPLS сети могут позволить организациям придать приоритет некоторым приложениям для обработки данных другим пользователям. С помощью MPLS приложения в реальном времени, такие как транзакции кредитной карты или общий доступ к экрану, могут иметь приоритет по сравнению с меньшим временем, таким как электронная почта.
  
### <a name="understanding-the-types-of-ip-network-services--basic-ip-and-mpls"></a>Общие сведения о типах служб IP-сети — Basic IP и MPLS

Первоначальная переадресация IP-пакетов, направленная на принцип "лучшие усилия". Это означало, что маршрутизаторы, пересылаемые по IP-адресам, будут лучше доставлять их в конечные точки, но вы не уверены в том, что и когда они будут доставлены по их адресатам. Вот как базовые Интернет-службы, в том числе домашнюю Интернет-подключение, работают уже сегодня. Идея в том, что если для определенного приложения требовалась надежность, она будет предоставлена на более высоком уровне в стеке протоколов. Надежным механизмом доставки является протокол TCP. Протокол UDP, используемый для голосовых и видеофайлов в реальном времени, является ненадежным механизмом доставки (например, "лучшие усилия"). 
  
Многопротокольное переключение этикеток (MPLS) было разработано как средство для поставщиков сетевых услуг, предлагающее услуги IP с гарантированной производительностью для задержек, колебаний и потери пакетов. Для того чтобы дополнить эти гарантии на производительность, MPLS берет на себя некоторые из непредсказуемости из традиционных IP-адресов. Во-первых, вместо того чтобы каждый пакет мог найти путь к месту назначения маршрутизатора (это может привести к тому, что каждый пакет получает другой маршрут от источника к целевому адресу), MPLS маршрутизирует все пакеты в соединении виртуального канала с помощью Фиксированный маршрут, именуемый путем переключения метки (LSP). Если одна из ссылок в этом пути не проходит, все Лспс с помощью этой ссылки быстро пересылаются.
  
Когда пакет отправляется в сеть MPLS, граничный маршрутизатор поставщика сетевых услуг добавляет дополнительный заголовок к пакету, который содержит метку, которая используется для перенаправления ее на соответствующий BSP. Метка вырезается пограничным маршрутизатором на другом конце сети MPLS.
  
Помимо упрощения процесса переадресации, другие преимущества MPLS — это то, что система сетевого управления будет знать, какие подключения будут передаваться на каждую ссылку в сети. Контролируя способ маршрутизации трафика через сеть, оператор может гарантировать качество обслуживания каждого пути. Поэтому, в отличие от наилучшего быстродействия традиционных или основных IP-протоколов, MPLS операторы могут предоставлять IP-службы с предсказуемой производительностью. Этот LSP также делает MPLS более безопасным, чем традиционные Интернет-услуги. Итак, благодаря базовой службе IP мы можем надеяться, что сеть будет работать достаточно хорошо, чтобы обеспечить качественную голосовую почту и использовать такие приемы, как FEC и более устойчивая Голосовая кодировка, чтобы улучшить вероятность, но с помощью MPLS, мы можем быть уверенным в этом.
  
Поставщики MPLS предлагают несколько классических градиентов обслуживания, которые в некоторых из них используют разные термины для их идентификации. Для того чтобы убедиться в том, что у вас есть результаты из [калькулятора Lync 2010 и 2013](https://go.microsoft.com/fwlink/?LinkID=690282) и рекомендуемые параметры для разных приложений Office 365 для рабочих нагрузок в реальном времени, вам придется тесно работать с вашим поставщиком.
