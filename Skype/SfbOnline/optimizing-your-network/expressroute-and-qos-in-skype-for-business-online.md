### <a name="setting-up-the-network"></a>Настройка сети

Обеспечить согласованную и высокого качества в режиме реального времени службы для законодательство Dewey LLC. существует несколько требований основные сведения, которые должны быть выполнены.
  
- Они хотят предоставления услуг голосовой связи во время сбоя, поэтому их рассылки сетевые коммутаторы и маршрутизаторы подачи питания через Ethernet (PoE) IEEE 802.3af или 802.3at.
    
- Сетевые коммутаторы и маршрутизаторы также необходимо использовать источники бесперебойного питания (UPS), поэтому они могут продолжать работу во время сбоя питания.
    
    У них подключений Wi-Fi в своих офисах по локальной сети, поэтому мы настоятельно рекомендуем, как они используют сертифицированного Скайп для партнеров инфраструктуры Business Wi-Fi из [Скайп для бизнес-решений](https://go.microsoft.com/fwlink/?LinkId=690281).
    
    > [!TIP]
    >  Рекомендуется использовать 802.11n и 802.11ac точки беспроводного доступа.
  
- И что самое важное, всех Локальных сетей во всех офисах необходимо настроить для обеспечения качества обслуживания (QoS). Это включает в себя ПК, ноутбуки и любого сетевого оборудования, например, коммутаторы и маршрутизаторы.
    
Теперь, когда рассматриваются основные сведения для предоставления бизнес-марки голосовых служб для закон Dewey LLC. рекомендуется использовать службы переключение многопротокольной метки (MPLS) IP-адресов с партнером сетевой службы, которые будут подключаться к службе Azure ExpressRoute. MPLS предоставляет службу IP-адресов с гарантией производительности для задержкой, дрожанием и потерей пакетов. Тем не менее если MPLS не поддерживается, Ethernet подключены к одному из наших ExpressRoute данных exchange партнеры также можно использовать.
  
Поставщики MPLS предлагают несколько классов уровни обслуживания, но каждый условий использования для их идентификации. Необходимо работать в тесном сотрудничестве с поставщиком, чтобы убедиться, что они понять данные, вовлекаются в [Lync 2010 и 2013 калькулятор пропускной способности](https://go.microsoft.com/fwlink/?LinkID=690282) и параметры, доступные и рекомендуется для разных рабочей нагрузки в режиме реального времени, Office 365 приложения.
  
Существует два варианта сопоставление данных из Скайп для бизнес-приложений на соответствующие классы MPLS службы:
  
- Конечная точка маркировку трафика с помощью элемента управления DiffServ Point (DSCP)
    
- Сеть управления списка управления Доступом на основе
    
Для реализации пометки конечной точки, необходимо настроить все домену машины Windows для Dewey LLC законодательство. для пометки каждого пакета с пометкой соответствующие DiffServ управления Point (DSCP) и затем реализации качества обслуживания на все сетевые коммутаторы и маршрутизаторы во всех своих сайтов office убедитесь, что QoS маркировка поддерживаются и не удаляются. Маркировку DSCP на сетевых пакетов сообщить поставщика услуг, как приоритеты пакета сети. **Есть Дополнительные сведения о DSCP в разделе качества обслуживания в части 2.**
  
Для сети на основе списка управления Доступом назначения отметками приоритета DSCP реализовано на вышестоящие маршрутизатора и основаны на UDP-порт источника. Диапазоны Рекомендуемый порт для каждого приложения, перечислены в разделе 2.6.1.1 [Планирование сети, мониторинг и устранение неполадок с помощью Lync Server](https://go.microsoft.com/fwlink/?LinkId=690286). Важно скоординировать это Dewey законодательство LLC общую QoS реализации и разработки и необходимо учитывать различные политики QoS и возможности для пометки несоответствий пакетов.
  
Каждый поставщик услуг сети ExpressRoute будут иметь класс обслуживания (QoS), подходящую для голосовой связи в режиме реального времени и видео. В этом COS называется «Значительно ускорить переадресации» (EF) для голосовой связи и «Обеспечиваемым переадресации» (AF) для видео. Необходимо быть очень осторожным при оценке пропускной способности покупки голосовая связь EF трафика. Причина является очень не реагирует на класс голосовой связи службы в том случае, если отправить несколько передачи голосовых данных не подготовлена класс службы для.
  
> [!TIP]
>  Немедленно удаляется любой трафик, отправляемый в классе голосовой связи службы за пределами обязательства поставщика услуг которого будет направлять влияние качества голосовой связи.
  
При просмотре общая схема для Dewey LLC законодательство. очень важно точно определить пропускной способности сети, необходимые для поддержки трафика голосовой связи между своей сети и пометки каждого пакета голосовой связи (и только пакетов голосовой связи) с помощью параметра DSCP для голосовой связи (то есть 46 EF DSCP).
  
Для реализации качества обслуживания в своей корпоративной сети, конечные точки или маршрутизаторы необходимо отметить каждый пакет с индикатором приоритет соответствующий уровень 3 (то есть DSCP). Пути всей сети каждый переключатель и маршрутизатора должны иметь включен параметр качества обслуживания. Необходимости даже только один сетевой коммутатор или маршрутизатор, для которого не включен качества обслуживания, QoS, может быть маркировку на звука или видео пакетов, проходящих через этот переключатель или маршрутизатор вырезает. Это позволяет отключить QoS во всех нижестоящие коммутаторы и маршрутизаторы которых снижает преимущества ExpressRoute.
  
Для этого также необходимо задать связь приоритетов третьего уровня и уровня 2 QoS на каждом этапе. Механизмы приоритет уровень 2 определены в IEEE 802. 1 p для проводной сети и 802.11e / WMM для сетей Wi-Fi. Еще важнее маршрутизатора сети, направленными сети поставщика услуг сети MPLS должен поддерживать DSCP параметры для всех исходящих пакетов, чтобы они будут поддерживать соответствующий класс MPLS службы.

### <a name="the-real-time-problem"></a>Проблема режиме реального времени

Предоставление услуг голосовой и видеосвязи качества business помещает особые требования на IP-сеть. В режиме реального времени трафик использует протокол транспорта в режиме реального времени (RTP) с помощью протокола датаграмм пользователя (UDP). В отличие от протокол TCP (Transmission Control), номера и тестируются все сообщения на ошибки и включает в себя другие механизмы обнаружения и повторно передавать потерянных или поврежденные сообщения, UDP-ПОРТ не предоставляются для этого типа надежности. Если не сохраняются из-за переполнения буфера сообщений поврежден, ошибки, они не сохраняются. UDP-ПОРТ был выбран для использования с RTP, так как характер трафика в режиме реального времени, что даже в том случае, если были повторно потерянных сообщений, будет прибытия гораздо позднее иметь любое положительное влияние на поток голосовое сообщение.
  
Знание воздействия пакеты потеряны голосовой связи, дизайнеры пришли к двух подходов для повышения производительности голосовой и видеосвязи по протоколу IP:
  
- Сделать голосовой связи, кодирования и декодирования больше устойчивое при пакетов не сохраняются. Это можно сделать, либо с помощью метода прямой коррекции ошибок (FEC), чтобы исправить процент возникают ошибки, это возможность найти в режиме реального времени транспорта Office 365 или по голосовой связи проектирование декодирования систем, которые пытаются скрытие влияние потерянных пакетов которого — это характеристика кодеки Майкрософт. 
    
- Использование служб транспорта, использующие качества службы механизма для обеспечения производительности сети по отношению к задержки, потеря пакетов и дрожание и вариантов в задержки между пакетов.
    
Только написания кода устойчивое голосовой связи решает проблему потери пакетов, поэтому важно, что сеть используется для выполнения в режиме реального времени голосовой связи и видео механизмов, Минимизация задержки и дрожание. Даже при использовании устойчивое написания кода, если слишком много пакетов не сохраняются, получающей станции не располагает достаточной информацией для восстановления распознавание версия сигнала голосовой связи. Процент потерянных пакетов, которые может привести к снижению, заметно в качество голосовой связи, которое может отличаться в зависимости от голосовой связи, метод используется кодировка. Во всех случаях, потери данных строк последовательных пакетов уже очень инспектором.
  
Минимизация задержки важна, так как чрезмерное задержка может повлиять на поток беседы и создание досадной для динамики. Рекомендации по обеспечению сообщите нам, задержку начала до конца для голосовой связи (что мы называть задержка «рот Очистить») должен храниться ниже 150 миллисекунд (мс). Односторонняя, не «кругового пути» задержки. ### <a name="expressroute-as-part-of-a-business-grade-voice-network"></a>ExpressRoute в рамках корпоративной сети марки голосовой связи

ExpressRoute предлагают выделенного подключения через поставщика службы сети (NSP) или Exchange поставщика (EXP) в одном из 3 параметры подключения. 
  
- Colocation Exchange в облако
    
- Точка-точка Ethernet
    
- Любой к любой подключения (IPVPN)
    
Это позволяет использовать преимущества высокой доступности (99,9% времени работоспособности. соглашения об уровне ОБСЛУЖИВАНИЯ) и надежность маршрутизации, обеспечения безопасности (не время передачи в Интернет), не оказывали вариантов в Интернет-трафик и отношениях маркировку качества обслуживания по назначению приоритетов трафика (Далее приведено QoS) . В корпоративную сеть голосовой связи марки можно предоставить ExpressRoute, а также хорошо плановых глобальной сети.
  
Можно использовать для передачи данных от офисы или центров обработки данных ExpressRoute (если гибридной топологии), подключенные к цепи. Данных для внешних пользователей (например, из дома, или поездок, и т.д.) не будут использовать канал ExpressRoute, если только пользователей, подключенных через VPN и не должны будут добавлены в оценки пропускной способности для изменения размера ExpressRoute цепи. Глобальной экономике клиента, могут приобрести цепи ExpressRoute в каждой области и использование протокола BGP сообщества тегов для оповещения правила маршрутизации, чтобы трафик перенаправляется предпочтительный канал ExpressRoute (обычно ближайшего один для каждого сайта), то время как другие цепи обеспечивают избыточность в случае сбоя влияния на одном канала.

### <a name="basic-concepts-of-quality-of-service-qosclass-of-service-cos"></a>Основные понятия качества обслуживания (QoS) / класс службы (CoS)

В IP-адресов качество обслуживания (QoS) описывает любой механизм, используемый для обеспечения приоритет обработки некоторых пакетов по другим пользователям. В соответствии с определением Международный союз телекоммуникаций (ITU) QoS включает все аспекты качества подключения, включая задержку, потери, соотношение сигнал шума, перекрестных помех, эхо, прерывания, Частотная характеристика, уровни громкости и т. д. Что мы ссылаться на качество обслуживания в сетях пакетов является более правильно определить термином класс службы (CoS), которая фокусируется на повышение производительности для задержкой, дрожанием и потерей пакетов, но будет продолжаться использовать QoS терминов, как обычно используется.
  
Предоставление QoS на IP-адрес сети вызывает два основных компонента:
  
- Резервирование определенный объем пропускной способности на каждой ссылки для трафика в режиме реального времени; Если для трафика в режиме реального времени в любое время не требуется, пропускной способности, его можно использовать для других трафика. Общее руководство является присвоение не более 30% емкости любыми ссылками для передачи голосовых данных.
    
- Пометка пакеты с индикатор приоритет в заголовке, сообщающее, коммутаторы и маршрутизаторы в поле путь приоритет пакета, которая должна быть назначена.
    
При получении пакета переключатель или маршрутизатора перемещается в очередь выходных данных для следующего ветвь или перехода. Существует очередей различных выходных данных для различных уровней приоритета. Переключатель или маршрутизатор использует алгоритм, который чаще, чем меньше приоритет очереди службы очереди наивысший приоритет.
  
Проблема в том, что существуют разные способы качества обслуживания, реализованных в второго уровня (то есть уровень Ethernet или Wi-Fi) и уровень 3 (то есть уровень IP). Эти различных реализаций качества обслуживания может потребоваться настроить в каждом переключатель и маршрутизатора в сети, а также взаимодействие между вашей сети и сети поставщика услуг сети.
  
Существует два варианта сопоставление данных из различных Скайп для бизнес-приложений для соответствующих классов службы:
  
- Конечная точка маркировки трафика с помощью Differentiated Services управления выберите команду (DSCP) 
    
- Сеть список управления доступом (ACL)-на основе

### <a name="implementing-qos-using-network-access-control-list-acl"></a>Внедрение QoS с помощью сетевого списка управления доступом (ACL)

Альтернативный способ реализации качества обслуживания в конфигурации ExpressRoute является использование сети списка управления доступом (ACL). В том, что подход, устраняя конечные точки вставки в заголовок каждого пакета соответствующие разметки DSCP, можно выполнить маркировку вышестоящие маршрутизатором, на основании UDP-порт источника.
