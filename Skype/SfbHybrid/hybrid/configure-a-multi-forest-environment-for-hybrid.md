## <a name="topology-requirements"></a>Требования к топологии

Поддерживаются несколько лесов пользователей. Учитывайте следующее: 
    
- Для поддерживаемых версий Lync Server и Skype для бизнеса Server гибридной конфигурации см. требования к версии [Сервера](plan-hybrid-connectivity.md#server-version-requirements) в плане гибридного подключения между Skype для бизнеса Server и [Microsoft 365 или Office 365](plan-hybrid-connectivity.md).
    
- Exchange Server могут быть развернуты в одном или более лесах, которые могут включать или не включать лес, содержащий Skype для бизнеса Server. Убедитесь, что вы применили последнее накопительное обновление.
    
- Сведения о совместном существовании с Exchange Server, включая критерии поддержки и ограничения в различных сочетаниях [](../../sfbserver/plan-your-deployment/integrate-with-exchange/integrate-with-exchange.md#feature_support) локального и онлайн-бизнеса, см. в материале Поддержка функций в плане интеграции Skype для бизнеса [и Exchange](../../sfbserver/plan-your-deployment/integrate-with-exchange/integrate-with-exchange.md).
    
  
## <a name="user-homing-considerations"></a>Учет учетных записей пользователей

Skype для бизнеса, которые могут быть Exchange в помещениях или в Интернете. Skype для бизнеса Интернет-пользователи должны использовать Exchange Online для оптимального использования; однако это не требуется. Exchange не требуется для реализации Skype для бизнеса в любом случае.
  
## <a name="configure-forest-trusts"></a>Настройка лесных трастов

В топологии лесных ресурсов леса ресурсов, Skype для бизнеса Server должны доверять каждому лесу учетных записей, которые содержат учетные записи пользователей, которые будут получать к нему доступ. Если у вас несколько лесов пользователей, чтобы включить межлесную проверку подлинности, важно включить маршрутизация Имен Суффикса для каждого из этих лесных трастов. Инструкции см. в [инструкции "Управление лесными трастами".](/previous-versions/windows/it-pro/windows-server-2008-R2-and-2008/cc772440(v=ws.11)) Если вы Exchange Server в другом лесу и он предоставляет функциональность для Skype для бизнеса пользователей, Exchange должны доверять лесохозяйничанию Skype для бизнеса Server. Например, если Exchange были развернуты в лесу учетных записей, это фактически означает двуединое доверие между учетной записью и Skype для бизнеса лесами в этой конфигурации.
  
## <a name="synchronize-accounts-into-the-forest-hosting-skype-for-business"></a>Синхронизация учетных записей в лесной Skype для бизнеса

Если Skype для бизнеса Server развертывается в одном лесу (лесу ресурсов), но предоставляет пользователям функции в одном или более других лесах (леса учетных записей), пользователи в других лесах должны быть представлены в качестве объектов отключенных пользователей в лесу, где Skype для бизнеса Server развернута. Продукт управления удостоверениями, например Microsoft Identity Manager, должен быть развернут и настроен для обеспечения и синхронизации пользователей из лесов учетных записей в лес, где Skype для бизнеса Server развертывается. Пользователи должны быть синхронизированы в лесу, Skype для бизнеса Server как отключенные объекты пользователей. Пользователи не могут синхронизироваться с контактными объектами Active Directory, так как Azure Active Directory Подключение не будут должным образом синхронизировать контакты в Azure AD для использования с Skype.
  
Независимо от конфигурации с несколькими лесами, Skype для бизнеса Server может также предоставлять функциональные возможности для всех включенных пользователей, которые существуют в одном лесу.
  
Чтобы получить правильную синхронизацию удостоверений, необходимо синхронизировать следующие атрибуты: 
  
|**Леса пользователей**|**Леса ресурсов**|
|:-----|:-----|
|выбранный атрибут ссылки учетной записи  <br/> |выбранный атрибут ссылки учетной записи  <br/> |
|почта;  <br/> |почта;  <br/> |
|ProxyAddresses  <br/> |ProxyAddresses  <br/> |
|ObjectSID  <br/> |msRTCSIP-OriginatorSID  <br/> |
   
Выбранный [атрибут ссылки учетной](/azure/active-directory/hybrid/plan-connect-design-concepts) записи будет использоваться в качестве источника якоря. Если у вас есть другой и неоменяемый атрибут, который вы предпочитаете использовать, вы можете сделать это; просто убедитесь, что изменить правило утверждений AD FS и выбрать атрибут во время конфигурации AAD Подключение.
  
Не синхронизируйте upNs между лесами. Во время тестирования мы обнаружили, что для каждого леса пользователей необходимо использовать уникальный upN, так как нельзя использовать один и тот же upN в нескольких лесах. В результате нам были представлены две возможности, чтобы синхронизировать upN или не синхронизировать. 
  
- Если уникальный upN из каждого пользовательского леса не синхронизирован с связанным отключенным объектом в лесу ресурсов, один вход (SSO) будет нарушен по крайней мере для начальной попытки регистрации (при условии, что пользователь выбрал параметр для сохранения пароля). В клиенте Skype для бизнеса мы предполагаем, что значения SIP/UPN одинаковы. Так как SIP-адрес в этом сценарии user@company.com, но upN включенного объекта в лесу пользователей на самом деле user@contoso.company.com, первоначальная попытка входа будет неудачной, и пользователю будет предложено ввести учетные данные. После ввода правильного и фактического upN запрос на проверку подлинности будет выполнен в отношении контроллеров домена в лесу пользователей, и вход будет успешным.
    
- Если уникальный upN из каждого пользовательского леса синхронизирован с связанным отключенным объектом в лесу ресурсов, проверка подлинности AD FS не будет. В этом правиле будет находиться upN на объекте в лесу ресурсов, который отключен и не может использоваться для проверки подлинности.

## <a name="configure-active-directory-federation-services"></a>Настройка служб Федерации Active Directory

После того как у вас есть клиент, вам потребуется настроить службы Федерации Active Directory (AD FS) в каждом из лесов пользователей. Это предполагает, что у вас есть уникальный адрес SIP и SMTP и имя пользователя (UPN) для каждого леса. AD FS необязательный и используется здесь для получения единого входного знака (SSO). DirSync с синхронизацией паролей также поддерживается и может также использоваться на месте AD FS. 
  
Тестировали только развертывания с соответствием SIP/SMTP и UPNs. Несопополнение SIP/SMTP/UPNs может привести к снижению функциональных возможностей, например проблем с Exchange интеграцией и SSO. 
  
Если вы не используете уникальный SIP/SMTP/UPN для пользователей из каждого леса, вы все равно можете возникнуть проблемы с SSO, независимо от того, где развертывается AD FS: 
  
- One-way или two-way trusts between resource/user forests with AD FS farm deployed in each user forest, all users share common SIP/SMTP domain but unique UPN for each user forest. 
    
- Двухходовые доверительные отношения между лесами ресурсов и пользователей с фермой AD FS, развернутой только в лесу ресурсов, все пользователи имеют общий домен SIP/SMTP, но уникальный upN для каждого леса пользователей. 
    
Размещая ферму AD FS в каждом лесу пользователей и используя уникальный SIP/SMTP/UPN для каждого леса, мы решим обе проблемы. При попытках проверки подлинности будут искаться и соответствовать только учетные записи в этом конкретном лесу пользователей. Это поможет обеспечить более плавный процесс проверки подлинности. 
  
Это будет стандартное развертывание Windows Server 2012 R2 AD FS и должно работать перед продолжением. Инструкции см. в инструкции Как установить [AD FS 2012 R2 для](https://blogs.technet.com/b/rmilne/archive/2014/04/28/how-to-install-adfs-2012-r2-for-office-365.aspx)Microsoft 365 или Office 365. 
  
После развертывания необходимо изменить правило утверждений, чтобы соответствовать выбранному ранее источнику якоря. В MMC AD FS в соответствии с доверчивыми партийным доверием щелкните правой кнопкой мыши Microsoft 365 платформы удостоверений или **Microsoft Office 365** платформы удостоверений, а затем выберите правила редактирования **утверждений.**  Измените первое правило и измените ObjectSID на **employeeNumber.**

## <a name="configure-aad-connect"></a>Настройка AAD-Подключение

В topologies леса ресурсов необходимо синхронизировать атрибуты пользователей из леса ресурсов и любых лесов учетных записей (s) в Azure AD. Самый простой и рекомендуемый способ это сделать, чтобы Azure AD Подключение синхронизировать и объединить идентификаторы пользователей из всех лесов, которые включили учетные записи пользователей и лес, содержащий Skype для бизнеса.  Сведения см. в [материале Configure Azure AD Подключение для Skype для бизнеса и Teams.](configure-azure-ad-connect.md)

Обратите внимание, Подключение AAD не обеспечивает синхронизацию в помещениях между учетной записью и лесами ресурсов. Это должно быть отдельно настроено с помощью Microsoft Identity Manager или аналогичного продукта, как описано выше.
  
После завершения и объединения Подключение AAD, если вы посмотрите на объект в метавселенной, вы должны увидеть что-то похожее на это: 
  
![Многолесный экран метавселенной объектной области](../../sfbserver/media/16379880-2de3-4c43-b219-1551f5dec5f6.png)
  
Зеленые выделенные атрибуты были объединены из Microsoft 365 или Office 365, желтые — из пользовательского леса, а синие — из леса ресурсов. 
  
Это тестовый пользователь, и вы можете видеть, что AAD Подключение идентифицировал sourceAnchor и cloudSourceAnchor от пользователя и объекты леса ресурсов из Microsoft 365 или Office 365, в нашем случае 1101, который является сотрудником, выбранным ранее. Затем он смог объединить этот объект в то, что вы видите выше. 
  
Дополнительные сведения см. в [ссылке Интеграция](/azure/active-directory/hybrid/whatis-hybrid-identity)локального каталога с Azure Active Directory. 
  
AAD Подключение по умолчанию, за исключением следующих: 
  
1. Один вход - с AD FS уже развернуты и работают: Выберите **Не настраивать**.
    
2. Подключение каталогов: Добавьте все домены.
    
3. Определение пользователей в локальном каталоге: идентификаторы select **User** существуют в нескольких каталогах и выберите атрибуты **ObjectSID** и **msExchangeMasterAccountSID.**
    
4. Определите пользователей в Azure AD: исходный якорь: Выберите атрибут, выбранный после прочтения Выбора хорошего атрибута [sourceAnchor,](/azure/active-directory/hybrid/plan-connect-design-concepts#selecting-a-good-sourceanchor-attribute)основного имени пользователя — **userPrincipalName**.
    
5.  Необязательные функции. Выберите, Exchange ли Exchange гибрид.
    
    > [!NOTE]
    >  Если у вас есть Exchange Online, может быть проблема с ошибками OAuth во время автооткрытия из-за перенаправления CNAME. Чтобы исправить это, необходимо установить URL-Exchange автообнаружаемого, задав следующий кодлет из Skype для бизнеса Server Management Shell:
    >
    > ```powershell
    > Set-CsOAuthConfiguration -ExchangeAutoDiscoverURL https://autodiscover-s.outlook.com/autodiscover/autodiscover.svc 
    > ```
    
6.  Ферма AD FS: выберите Windows Server 2012 **ферму R2 AD FS** и введите имя сервера AD FS.
    
7.  Завершите мастер и выполните необходимые проверки.
