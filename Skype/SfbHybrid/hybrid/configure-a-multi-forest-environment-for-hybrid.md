# <a name="deploy-a-resource-forest-topology"></a>Развертывание топологии леса ресурсов
 
В следующих разделах приводится руководство по настройке среды с несколькими лесами в модели лесов ресурсов или пользователей для предоставления функциональных возможностей Skype для бизнеса в гибридном сценарии. 
  
![Среда с несколькими лесами для гибридной среды](../../sfbserver/media/5f079435-b252-4a6a-9638-3577d55b2873.png)
  
## <a name="topology-requirements"></a>Требования к топологии

Поддерживается несколько лесов пользователей. Учитывайте следующее: 
    
- Поддерживаемые версии Lync Server и Skype для бизнеса [](plan-hybrid-connectivity.md#server-version-requirements) Server в гибридной конфигурации см. в требованиях к версии сервера в плане гибридного подключения между Skype для бизнеса Server и [Microsoft 365 или Office 365.](plan-hybrid-connectivity.md)
    
- Exchange Server могут быть развернуты в одном или более лесах, которые могут включать или не включать лес, содержащий Skype для бизнеса Server. Убедитесь, что вы применили последнее накопительное обновление.
    
- Подробные сведения о сосуществовании с Exchange Server, в том числе критерии поддержки и ограничения в [](../../sfbserver/plan-your-deployment/integrate-with-exchange/integrate-with-exchange.md#feature_support) различных сочетаниях локальной и сетевой функций, см. в поддержке функций в планировании интеграции Skype для бизнеса и [Exchange.](../../sfbserver/plan-your-deployment/integrate-with-exchange/integrate-with-exchange.md)
    
  
## <a name="user-homing-considerations"></a>Вопросы, которые следует учитывать при отлове пользователей

Пользователи Skype для бизнеса, которые были в локальной сети, могут иметь exchange в локальной или сетевой сети. Пользователям Skype для бизнеса Online следует использовать Exchange Online для оптимального использования; однако это не является обязательной. Для реализации Skype для бизнеса в любом случае не требуется локальное приложение Exchange.
  
## <a name="configure-forest-trusts"></a>Настройка доверия леса

В топологии леса ресурсов леса ресурсов, в которых размещен Skype для бизнеса Server, должны доверять каждому лесу учетных записей, который содержит учетные записи пользователей, которые будут получать к нему доступ. Если имеется несколько лесов пользователей, для включения проверки подлинности между лесами важно, чтобы для каждого из этих доверия леса была включена маршрутизация суффиксов имен. Инструкции см. в [под управлением отношениями доверия леса.](https://technet.microsoft.com/library/cc772440.aspx) Если вы развернули Exchange Server другом лесу и он предоставляет пользователям Skype для бизнеса функциональные возможности, лес, в который размещен Exchange, должен доверять лесу, в который размещен Skype для бизнеса Server. Например, если бы Exchange был развернут в лесу учетных записей, это фактически означает, что в этой конфигурации требуется двуназначить доверие между учетной записью и лесами Skype для бизнеса.
  
## <a name="synchronize-accounts-into-the-forest-hosting-skype-for-business"></a>Синхронизация учетных записей в лесу, где размещен Skype для бизнеса

Если Skype для бизнеса Server развернут в одном лесу (лесу ресурсов), но предоставляет функциональные возможности пользователям в одном или более других лесах (лесах учетных записей), пользователи в других лесах должны быть представлены как отключенные объекты пользователей в лесу, в котором развернут Skype для бизнеса Server. Необходимо развернуть и настроить продукт управления удостоверениями, например Microsoft Identity Manager, для предоставления и синхронизации пользователей из лесов учетных записей в лесу, в котором развернут Skype для бизнеса Server. Пользователи должны быть синхронизированы в лесу, где размещен Skype для бизнеса Server, как отключенные объекты пользователей. Пользователи не могут быть синхронизированы как контактные объекты Active Directory, так как Azure Active Directory Connect не будет должным образом синхронизировать контакты с Azure AD для использования со Skype.
  
Независимо от конфигурации с несколькими лесами лес, в лесу, где размещен Skype для бизнеса Server, также могут быть функциональные возможности для всех включенных пользователей, которые существуют в одном лесу.
  
Для правильной синхронизации удостоверений необходимо синхронизировать следующие атрибуты: 
  
|**Леса пользователей**|**Леса ресурсов**|
|:-----|:-----|
|выбранный атрибут ссылки учетной записи  <br/> |выбранный атрибут ссылки учетной записи  <br/> |
|mail  <br/> |mail  <br/> |
|ProxyAddresses  <br/> |ProxyAddresses  <br/> |
|ObjectSID  <br/> |msRTCSIP-OriginatorSID  <br/> |
   
Выбранный [атрибут ссылки учетной](https://docs.microsoft.com/azure/active-directory/hybrid/plan-connect-design-concepts) записи будет использоваться в качестве якоря источника. Если у вас другой и не изменяемый атрибут, который вы предпочитаете использовать, это можно сделать; Просто отредактировать правило утверждений AD FS и выбрать атрибут во время настройки AAD Connect.
  
Не синхронизируйте upNs между лесами. В ходе тестирования было обнаружено, что необходимо использовать уникальное имя-пользователя для каждого леса пользователей, так как одно и то же имя пользователя нельзя использовать в нескольких лесах. В результате мы получили две возможности: синхронизировать upN или не синхронизироваться. 
  
- Если уникальное имя пользователя из каждого леса пользователей не было синхронизировано со связанным отключенным объектом в лесу ресурсов, единый вход будет нарушен по крайней мере при первой попытке (предполагается, что пользователь выбрал параметр для сохранения пароля). В клиенте Skype для бизнеса предполагается, что значения SIP/UPN одинаковы. Так как SIP-адрес в этом сценарии имеет user@company.com, а имя пользователя включенного объекта в лесу пользователей на самом деле user@contoso.company.com, первая попытка входа не удалась, и пользователю будет предложено ввести учетные данные. После ввода правильного или фактического имени upN запрос на проверку подлинности будет выполнен для контроллеров домена в лесу пользователей, и вход будет выполнен успешно.
    
- Если уникальное имя пользователя из каждого леса пользователей синхронизировано со связанным отключенным объектом в лесу ресурсов, проверка подлинности AD FS не будет. Правило поиска совпадений находит upN объекта в лесу ресурсов, который был отключен и не может использоваться для проверки подлинности.

## <a name="configure-active-directory-federation-services"></a>Настройка служб федерации Active Directory

После того как у вас есть клиент, вам потребуется настроить службы федерации Active Directory (AD FS) в каждом из лесов пользователей. Предполагается, что у вас есть уникальный адрес SIP и SMTP и имя субъекта-пользователя (UPN) для каждого леса. AD FS является необязательным и используется здесь для получения единого вход (SSO). DirSync с синхронизацией паролей также поддерживается и может использоваться в качестве места AD FS. 
  
Проверялись только развертывания с совпадающих SIP/SMTP и UPNS. Несоотвляющие SIP/SMTP/UPNs могут привести к снижению функциональности, например проблемам с интеграцией Exchange и SSO. 
  
Если для пользователей из каждого леса не используется уникальное SIP/SMTP/UPN, могут возникнуть проблемы с использованием SSO независимо от того, где развернуты AD FS: 
  
- Отношения доверия между лесами ресурсов и пользователей с фермой AD FS, развернутой в каждом лесу пользователей, имеют общий домен SIP/SMTP, но уникальное имя субъекта-пользователя для каждого леса пользователей. 
    
- Двухнапорные отношения доверия между лесами ресурсов и пользователей с фермой AD FS, развернутой только в лесу ресурсов, все пользователи имеют общий домен SIP/SMTP, но уникальное имя субъекта-пользователя для каждого леса пользователей. 
    
Размещая ферму AD FS в каждом лесу пользователей и используя уникальное имя SIP/SMTP/UPN для каждого леса, мы устраняем обе проблемы. При попытках проверки подлинности поиск и поиск будет искаться только в учетных записях в этом лесу пользователей. Это поможет обеспечить более плавный процесс проверки подлинности. 
  
Это стандартное развертывание служб AD FS Windows Server 2012 R2, которые должны работать перед продолжением. Инструкции см. в инструкциях по установке [AD FS 2012 R2 для Microsoft 365 или Office 365.](https://blogs.technet.com/b/rmilne/archive/2014/04/28/how-to-install-adfs-2012-r2-for-office-365.aspx) 
  
После развертывания необходимо изменить правило утверждений в соответствие с ранее выбранной привязкой источника. В MMC AD FS, в области отношения доверия с отношениями, щелкните правой кнопкой мыши платформу удостоверений **Microsoft 365** или **платформу удостоверений Microsoft Office 365,** а затем выберите "Изменить правила утверждений".  Измените первое правило и измените ObjectSID на **employeeNumber.** 
  
![Экран редактирования правил с несколькими лесами](../../sfbserver/media/f5d485bd-52cc-437f-ba71-217f8902056c.png)
  
## <a name="configure-aad-connect"></a>Настройка AAD Connect

В topologies леса ресурсов необходимо, чтобы атрибуты пользователя из леса ресурсов и всех лесов учетных записей синхронизировались с Azure AD. Самый простой и рекомендуемый способ сделать это — синхронизировать Azure  AD Connect и объединить удостоверения пользователей из всех лесов, включив учетные записи пользователей, и леса, который содержит Skype для бизнеса. For details see, [Configure Azure AD Connect for Skype for Business and Teams.](configure-azure-ad-connect.md)

Обратите внимание, что AAD Connect не обеспечивает локальной синхронизации между учетной записью и лесами ресурсов. Это необходимо отдельно настроить с помощью Microsoft Identity Manager или аналогичного продукта, как описано выше.
  
После завершения объединения aAD Connect, если вы посмотрите на объект в метавселенной, вы увидите что-то вроде этого: 
  
![Экран объекта метавселенной с несколькими лесами](../../sfbserver/media/16379880-2de3-4c43-b219-1551f5dec5f6.png)
  
Атрибуты, выделенные зеленым цветом, были объединены из Microsoft 365 или Office 365, желтый — из леса пользователей, а синий — из леса ресурсов. 
  
Это тестовый пользователь, и вы можете увидеть, что AAD Connect определил sourceAnchor и cloudSourceAnchor из объектов пользователя и леса ресурсов из Microsoft 365 или Office 365, в нашем случае это 1101, который является employeeNumber, выбранный ранее. Затем ему удалось объединить этот объект с тем, что вы видите выше. 
  
Дополнительные сведения см. в теме "Интеграция локального [каталога с Azure Active Directory".](https://azure.microsoft.com/documentation/articles/active-directory-aadconnect/) 
  
AAD Connect следует установить по умолчанию, за исключением следующих случаев: 
  
1. Единый вход — если AD FS уже развернута и работает: выберите **"Не настраивать".**
    
2. Подключите каталоги: добавьте все домены.
    
3. Определите пользователей в локальном каталоге: выберите удостоверения пользователей в нескольких каталогах **и** выберите атрибуты **ObjectSID** и **msExchangeMasterAccountSID.**
    
4. Определите пользователей в Azure AD: привязка источника: выберите атрибут, выбранный после прочтения параметра [Selecting a good sourceAnchor attribute](https://docs.microsoft.com/azure/active-directory/hybrid/plan-connect-design-concepts#selecting-a-good-sourceanchor-attribute), User Principal Name - **userPrincipalName**.
    
5.  Необязательные функции: выберите, развернута ли гибридная система Exchange.
    
    > [!NOTE]
    >  Если у вас есть только Exchange Online, может быть проблема с ошибками OAuth во время автообнаружения из-за перенаправления CNAME. Чтобы устранить эту ошибку, необходимо настроить URL-адрес автообнаружия Exchange с помощью следующего команды в оболочке управления Skype для бизнеса Server:
    >
    > ```powershell
    > Set-CsOAuthConfiguration -ExchangeAutoDiscoverURL https://autodiscover-s.outlook.com/autodiscover/autodiscover.svc 
    > ```
    
6.  Ферма AD FS: выберите использование существующей фермы **AD FS Windows Server 2012 R2** и введите имя сервера AD FS.
    
7.  Завершите работу мастера и выполните необходимые проверки.
