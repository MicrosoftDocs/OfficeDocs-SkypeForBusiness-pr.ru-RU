# <a name="cloud-consolidation-for-teams-and-skype-for-business"></a>Консолидация облачных хранилищ для Teams и Skype для бизнеса

[!INCLUDE [sfbo-retirement](../../Hub/includes/sfbo-retirement.md)]


Многие крупные организации имеют более одного локального леса службы AD, а в некоторых случаях — несколько развертываний Exchange и/или Skype для бизнеса Server (или сервера Lync Server). Кроме того, даже организации, у которых есть только один локальный лес, могут оказаться в аналогичной ситуации за счет бизнес-слияния или поглощения. По мере перемещения этих клиентов в облако они хотят консолидировать несколько экземпляров локальной рабочей нагрузки в облако в одну Microsoft 365 организацию. В этой статье описывается, как добиться этой консолидации для организаций с несколькими локальной развертываниями Skype для бизнеса (или Lync), которые хотят полностью переместить свою организацию в Microsoft Teams (все пользователи только Teams).

Традиционно рекомендация для клиентов в такой ситуации заключалась в том, чтобы сначала объединить локальные развертывания, а затем перейти в облако. В этой статье по-прежнему можно использовать решение, которое позволяет организациям с несколькими Skype для бизнеса развертываниями перенести одно развертывание одновременно в одну Microsoft 365 организацию, не делая локальной консолидации. Обратите внимание, Microsoft Teams не поддерживает несколько лесов Skype для бизнеса или Lync Server в гибридном режиме с одной Microsoft 365 организацией. 

> [!Important]
> Прежде чем использовать это руководство для настройки, обязательно просмотрите и поймете [ограничения,](#limitations)так как они могут повлиять на организацию.

## <a name="overview-of-cloud-consolidation"></a>Обзор облачной консолидации

Консолидация всех пользователей из локального облака в облако в одной Microsoft 365 организации может быть достигнута для любой организации с несколькими Skype для бизнеса развертываниями при условии, что будут выполнены следующие ключевые требования:

- Должна быть задействована как минимум одна Microsoft 365 организация. Консолидация в сценариях с более чем одной организацией не поддерживается.
- В любой момент времени только один локальный лес Skype для бизнеса может находиться в гибридном режиме (общее адресное пространство SIP). Все другие локальные леса Skype для бизнеса должны оставаться в локальном режиме (и предположительно быть федеративными по отношению друг к другу). Обратите внимание, что эти  другие локальной организации могут синхронизироваться с Azure AD при желании при [отключке доменов SIP в Интернете.](/powershell/module/skype/disable-csonlinesipdomain)

Клиенты с развертыванием Skype для бизнеса в нескольких лесах должны полностью перенести всех пользователей одного гибридного леса Skype для бизнеса по отдельности в организацию Microsoft 365 с помощью функции общего пространства адресов SIP, а затем отключить гибрид с этим локальное развертывание, прежде чем перейти к миграции следующего локального Skype для бизнеса развертывания. До миграции в облако локальные пользователи остаются в федеративном состоянии с любыми пользователями, не представленными в этом же локальном каталоге пользователя.

Ниже приведены основные действия, необходимые для получения от исходного состояния до нужного состояния.  Обратите внимание, что некоторые организации могут обнаружить, что их отправная точка находится где-то в середине этих действий. Другие [отправные точки](#other-starting-points)см. в этой статье. Наконец, в некоторых случаях порядок может быть скорректирован в зависимости от необходимости. [Основные ограничения и ограничения](#limitations) описаны позже.

1.  Получите организацию Microsoft 365, если она еще не существует.
2.  Убедитесь, что все соответствующие домены SIP в обоих локальном развертывании проверяются Microsoft 365 доменами.
3.  Выберите один Skype для бизнеса, который будет гибридным с Microsoft 365. В этом примере мы будем использовать OriginalCompany. <span> com.
4.  [Включить AAD Подключение для леса,](configure-azure-ad-connect.md) который сначала станет гибридным (OriginalCompany. <span> com). 
5.  Установите политику [teamsUpgradePolicy](/powershell/module/skype/grant-csteamsupgradepolicy) для всей клиентской группы в SfBWithTeamsCollab или одном из других режимов SfBOnly или SfBWithTeamsCollabAndMeetings. Это важно для обеспечения маршрутивки звонков и чатов от пользователей, переехавших в Teams только для пользователей, которые остаются на месте.
6.  Рекомендуется на данном этапе (но пока не требуется до шага 11) включить [AAD Подключение](cloud-consolidation-aad-connect.md) для другого леса (AcquiredCompany). <span> com). Если предположить, Подключение AAD включен в обоих лесах, эта оргия выглядит как **[рис.](#figure-a)** A , которая может быть общей отправной точкой для некоторых орготов. 
7.  Для всех доменов SIP, которые размещены в других локальном развертывании (в данном случае AcquiredCompany. <span> com), [отключите эти](/powershell/module/skype/disable-csonlinesipdomain) домены SIP в Microsoft 365 организации с помощью `Disable-CsOnlineSipDomain` Teams PowerShell Module. 
8.  [Настройка Skype для бизнеса гибрида](configure-federation-with-skype-for-business-online.md) для OriginalCompany. <span> com (одно развертывание, которое по-прежнему включило домены SIP в Интернете).
9.  В гибридном развертывании (OriginalCompany. <span> com), [начните перемещение](move-users-between-on-premises-and-cloud.md) пользователей из Skype для бизнеса в облако (независимо от того, Teams или нет), чтобы пользователь Teams только. Теперь организация выглядит как **[рис. B](#figure-b)**. Основные изменения из рисунка A:
    - Пользователи из обоих локального каталога теперь находятся в AAD.
    - AcquiredCompany. <span> com — это отключенный домен SIP в Интернете.
    - Некоторые пользователи были перемещены в Teams Только. (См. фиолетовый пользователь A.)
10. После того как все пользователи будут перемещены в облако, отключать гибрид с помощью локального развертывания [Skype для бизнеса](cloud-consolidation-disabling-hybrid.md) для OriginalCompany. <span> com из Microsoft 365:  
    - Отключить раздельный домен в Microsoft 365 организации.
    - Отключение возможности связи с Microsoft 365 в OriginalCompany. <span> com на локальной основе.
    - Обновление записей DNS для OriginalCompany. <span> com, чтобы указать на Microsoft 365.
11. Если еще не сделано, в Подключение [AAD](cloud-consolidation-aad-connect.md) для следующего леса, который будет гибридным (AcquiredCompany. <span> com). На данный момент организация выглядит как **[рис.](#figure-c)** Это может быть еще одной общей отправной точкой для некоторых организаций. 
12. В Teams PowerShell включить [домены SIP](/powershell/module/skype/enable-csonlinesipdomain) для следующего локального развертывания, которое будет гибридным, AcquiredCompany. <span> com. Это делается с `Enable-CsOnlineSipDomain` использованием новых функциональных возможностей, доступных с декабря 2018 г.
13. Если вы используете закрытую федерацию, необходимо добавить все домены SIP (за исключением .microsoftonline.com) чистого онлайн-клиента в качестве разрешенных доменов в \* одном Microsoft 365.  Обратите внимание, что это может занять некоторое время, прежде чем изменение вступает в силу, и нет никакого вреда в этом рано, поэтому мы предлагаем сделать это заблаговременно перед переходом на шаг 14.
14. Обновим локальное окружение, чтобы принять все домены SIP от онлайн-клиента, чтобы они совпадали.
    - [Обновив](cloud-consolidation-edge-certificates.md) САН во всех краях сертификатов, чтобы иметь то же значение, что и раньше, плюс значения для любых существующих доменов SIP в Интернете (за исключением *.microsoftonline.com), в данном случае Sip.OriginalCompany. <span> com.
    - Убедитесь, что OriginalCompany. <span> com — это [разрешенный домен](/powershell/module/skype/new-csalloweddomain) в локальном развертывании AcquiredCompany. Добавление разрешенных доменов.
15. [Включить Skype для бизнеса между](configure-federation-with-skype-for-business-online.md) локальной acquiredCompany. <span> com и облако.
16. При желании [переадмить](move-users-between-on-premises-and-cloud.md) пользователей из локального в облако, чтобы стать [пользователями TeamsOnly.](/microsoftteams/teams-and-skypeforbusiness-coexistence-and-interoperability) Во время этого состояния организация выглядит как **[рисунок D](#figure-d)**.

## <a name="other-starting-points"></a>Другие отправные точки

Действия в каноническом примере выше предполагают, что организация начинается с двух федератированных локального развертывания без Microsoft 365 присутствия. Однако некоторые организации могут иметь существующий Microsoft 365, и в последовательность выше могут быть разные точки входа. Существует четыре типичных конфигурации:

- Несколько федератированных локальной организации без Microsoft 365 организации. В этом случае начните с шага 1.
- Несколько федерарно-локальной организации, которые уже синхронизируются с несколькими Skype для бизнеса лесом в один клиент Azure AD. Такая организация похожа на гипотетическую организацию на рисунке A, которая выполнила шаги 1-6 и должна начаться с 7-го шага.
- Гибридная организация, которая федератирует с 1 или более другими чистыми локальной организацией, ни одна из которых не синхронизируется с AAD. Такая организация будет напоминать гипотетическую организацию на **рисунке E,** показанном ниже.
    - Эта организация похожа на рис. B, который выполнил шаги 1-9, за исключением:
        - Его не гибридные Skype для бизнеса еще *не* синхронизируются с Azure AD.
        -  Домены SIP в Интернете еще не отключены. 
    - Эти организации должны либо:
        - Полная миграция существующей гибридной организации и введите вышеуказанную последовательность на шаге 10.  OR,
        - Если перед завершением миграции гибридной организации необходимо синхронизировать все другие леса Skype для бизнеса в AAD, организация должна выполнить шаг 7 (отключить все домены SIP в любых других локальном развертывании Skype для бизнеса, синхронизируются в AAD), а затем включить AAD Подключение, а затем продолжить с шагом 10 (отключать исходное гибридное развертывание).       
                **Рис. E**<br>
                ![Диаграмма E.](../media/cloudconsolidationfige.png)
- Чистая Teams только организация, которая федератирует с отдельной локальной Skype для бизнеса организацией. Как только эта организация отключает домен SIP для локальной организации и включает AAD Подключение для локальной Skype для бизнеса организации, она напоминает гипотетическую организацию, показанную на рисунке **[C,](#figure-c)** завершив шаги 1-11.

## <a name="limitations"></a>Ограничения

- Должна быть задействована как минимум одна Microsoft 365 организация. Консолидация в сценариях с более чем одной организацией не поддерживается.
- Только одно локальное Skype для бизнеса может быть в гибридном режиме (общее адресное пространство SIP) одновременно. Все другие локально Skype для бизнеса леса должны оставаться чисто локальной и должны быть федерадированы друг с другом и Microsoft 365 организацией.
- Перед переносом в облако существует асимметричный опыт для пользователей в этом развертывании, так как не все пользователи в Интернете представлены локально:
    - Этот опыт можно подвести следующим образом:
        - Любой пользователь, который находится в сети, будет взаимодействовать с локальной пользователями в гибридной среде, как если бы пользователь был гибридным.
        - Локально пользователи в гибридном развертывании будут взаимодействовать с интернет-пользователями, представленными в локальном каталоге, как если бы они были гибридными. 
        - Локально пользователи в гибридном развертывании будут взаимодействовать с сетевыми пользователями, которые не представлены в локальной AD в федераированном режиме.
    - На **[рисунке D](#figure-d)** выше пользователь E находится на локальной основе в AcquiredCompany. <span> com.  Пользователь E будет взаимодействовать с пользователем D (домашний онлайн) с помощью стандартного гибридного интерфейса, но пользователь E будет иметь федеративную возможность взаимодействия с пользователями A, B и C, так как они не представлены в локальном каталоге. Тем не менее, пользователи A, B и C будут взаимодействовать с пользователем E, как если бы пользователь был в гибридной.
    - Последствия гибридного взаимодействия и федерации:
        - Присутсвие автоматически не подписывается для федератированных пользователей, если только пользователь не помечен как контакт.
        - Переададка вызовов не работает между федерадными доменами.
        - Сценарии переноса вызовов являются более ограниченными.
        - Регулирование может применяться к федеративу трафика.
- Учитывая этот асимметричный опыт, официальная поддержка функций вызова в межучрещенных сценариях между локального пользователя и пользователем облака, который не находится в локальном каталоге, ограничивается одноранговой поддержкой. 
    - Переададка вызовов, передача, очереди вызовов и т. д. Между этими пользователями не поддерживается.
    - Эти не поддерживаемые сценарии вызовов по-прежнему будут отображаться включенной, но во многих случаях они не будут непредсказуемыми способами. 
    - На **[рисунке D](#figure-d)** выше пользователь E является локальной, и звонки с пользователями A, B или C будут поддерживаться только в одноранговом качестве. (Вызовы с пользователем D не будут иметь ограничений поддержки.)  Однако после перемещений локального пользователя E в облако это ограничение больше не применяется.
- Если у вас есть несколько развертывание Skype для бизнеса Server 2019 в вашей среде, только 1 из этих развертывание может быть настроено на использование организационных автосекретарь, так как эта функция требует Skype для бизнеса Server гибридной конфигурации. 
- Порядок некоторых предыдущих действий может быть скорректирован. Ключевое требование, которое должно быть выполнены, это то, что если все эти требования верны:
    - Несколько локального Skype для бизнеса синхронизируются с одним клиентом Microsoft 365 Azure AD.
    - Разделенный домен включен в одном локальном лесу
    - По крайней мере один пользователь гибридной организации был перенесен в облако<br>   Затем необходимо *отключить* все другие домены SIP в Интернете из любого другого локального Skype для бизнеса леса. В противном случае федерация между пользователями сети в гибридной организации и локальной пользователями в других организациях будет нарушаться в одном направлении.

## <a name="implications"></a>Implications

- Так как существуют ограничения в поддержке расширенных функций вызовов, как описано выше, организации должны рассматривать эти асимметричные состояния как временные как часть миграции, а не преследовать их как **стабильное состояние.**  
- Организации с несколькими локальной Skype для бизнеса развертыванием обычно должны начинаться с развертывания, которое можно полностью перенести в облако, чтобы консолидация продолжалась. Понятно, что в некоторых случаях будут задерживаться определенные группы пользователей, для которых пока не целесообразно перейти в Teams. Если это рассматривается в сценариях, связанных с несколькими Skype для бизнеса лесами, приступить к миграции с другим лесом, который не имеет этих ограничений, если это вообще возможно.
- При переходе из локального в облако пользователи, которые имеют отношения делегирования и/или обычно участвуют в сценариях переададации вызовов, должны быть перемещены вместе в качестве единицы.

## <a name="considerations-for-moving-to-teamsonly-mode"></a>Соображения для перехода в режим TeamsOnly

При переходе пользователей из локального в облако в гибридной среде эти пользователи становятся Teams только пользователями.

- При назначении режима TeamsOnly пользователю все чаты и вызовы любого другого пользователя будут Teams клиента. 
- Если пользователи с Skype для бизнеса в основном используют Skype для бизнеса клиента, а не Teams, подумайте о настройке TeamsUpgradePolicy таким образом, чтобы маршрутизация для локального пользователя всегда была Skype для бизнеса, а не Teams. Чтобы обеспечить надлежащую маршрутику чатов и звонков между пользователями, которые являются TeamsOnly, и пользователями, которые по-прежнему используют Skype для бизнеса на локальной основе, локальное пользователи должны иметь эффективное значение TeamsUpgradePolicy с одним из режимов SfB, а не островами (по умолчанию). 
    - Для этого необходимо сначала установить глобальный экземпляр *TeamsUpgradePolicy* клиента для одного из этих значений:
        - SfBWithTeamsCollab (рекомендуется)
        - SfBWithTeamsCollabAndMeetings
        - SfBOnly
    - Вы можете предоставить политику для клиента с помощью этой команды:<br>`Grant-CsTeamsUpgradePolicy -PolicyName SfBWithTeamsCollab -Global`
    - Примечание. Это необходимо сделать на уровне клиента, так как политика не может быть назначена отдельным пользователям, у которых нет SIP-адреса в каталоге в Интернете. Несмотря на отключение доменов SIP в Интернете для вашего локального развертывания (s), пользователи в этих доменах не будут иметь SIP-адреса в каталоге онлайн по дизайну. Таким образом, единственный способ применить политику к локальному пользователю — назначить ее на уровне клиента. В отличие от этого, в гибридном развертывании пользователи будут иметь SIP-адрес в каталоге онлайн, чтобы им можно было явно задать политику, если требуется, чтобы у них было другое значение, чем глобальная политика клиента.
