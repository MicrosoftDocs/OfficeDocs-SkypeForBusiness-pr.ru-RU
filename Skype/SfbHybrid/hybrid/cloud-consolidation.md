# <a name="cloud-consolidation-for-teams-and-skype-for-business"></a>Консолидация облачных хранилищ для Teams и Skype для бизнеса

Многие крупные организации имеют более одного локального леса службы AD, а в некоторых случаях — несколько развертываний Exchange и/или Skype для бизнеса Server (или сервера Lync Server). Кроме того, даже организации, у которых есть только один локальный лес, могут оказаться в аналогичной ситуации за счет бизнес-слияния или поглощения. По мере перехода этих клиентов в облачную среду у них возникает необходимость в объединении нескольких экземпляров определенной локальной рабочей нагрузки в одном облаке и в едином клиенте Office 365. В этой статье описывается, как добиться такой консолидации для организаций с несколькими локальными развертываниями Skype для бизнеса (или Lync), которые хотят перенести свою рабочую нагрузку в Microsoft Cloud, например Microsoft Teams и/или Skype для бизнеса Online.

Традиционно рекомендация для клиентов в такой ситуации заключалась в том, чтобы сначала объединить локальные развертывания, а затем перейти в облако. Несмотря на то, что в этой статье по-прежнему есть вариант, в этой статье описывается решение, основанное на новых функциях, которые позволяют организациям с несколькими развертываниями Skype для бизнеса выполнять миграцию одного развертывания за раз в один клиент Office 365 без проведения локального развертывания. Консол. Обратите внимание, что даже с помощью этой новой функции Skype для бизнеса Online и Microsoft Teams не поддерживают несколько лесов Skype для бизнеса и Lync в гибридном режиме с одним клиентом Office 365. 

> [!Important]
> Прежде чем использовать это руководство для настройки, обязательно ознакомьтесь с [ограничениями](#limitations), которые могут повлиять на вашу организацию.

## <a name="overview-of-cloud-consolidation"></a>Обзор облачной консолидации

Консолидацию всех пользователей из локальной среды в облако в одном клиенте Office 365 можно выполнить для любой организации, в которой есть несколько развертываний Skype для бизнеса, при условии соблюдения указанных ниже требований.

- Необходимо использовать не более одного клиента Office 365. Консолидация в сценариях с несколькими клиентами Office 365 не поддерживается.
- В любой момент времени только один локальный лес Skype для бизнеса может находиться в гибридном режиме (общее адресное пространство SIP). Все другие локальные леса Skype для бизнеса должны оставаться в локальном режиме (и предположительно быть федеративными по отношению друг к другу). Обратите внимание, что эти другие локальные организации *могут* синхронизироваться с AAD, если вы хотите [Добавить новые функции, чтобы отключить домены SIP в Интернете](https://docs.microsoft.com/powershell/module/skype/disable-csonlinesipdomain?view=skype-ps) , доступные на 2018 декабря.

Для клиентов, использующих Skype для бизнеса в нескольких лесах, необходимо полностью перенести всех пользователей из единого гибридного леса Skype для бизнеса в клиент Office 365, используя функцию общего адресного пространство SIP, а затем отключить гибридную среду с этим локальным развертыванием, прежде чем переходить на следующее локальное развертывание Skype для бизнеса. До миграции в облако локальные пользователи остаются в федеративном состоянии с любыми пользователями, не представленными в этом же локальном каталоге пользователя.

## <a name="other-starting-points"></a>Другие начальные точки

- Несколько федеративных локальных организаций без клиента Office 365. В этом случае начните с шага 1.

## <a name="limitations"></a>Ограничения

- Необходимо использовать не более одного клиента Office 365. Консолидация в сценариях с несколькими клиентами Office 365 не поддерживается.
- Только один локальный лес Skype для бизнеса может находиться в гибридном режиме (общее адресное пространство SIP) за раз. Все остальные локальные леса Skype для бизнеса должны храниться исключительно в локальной среде и должны быть объединены друг с другом и клиентом Office 365.
- Перед переносом в облако существует асимметричная работа пользователей в этом развертывании, так как не все пользователи в сети представлены в локальной среде:
    - Возможности можно суммировать следующим образом:
        - Любой пользователь, подключенный к сети, будет взаимодействовать с локальными пользователями в гибридной среде, как если бы пользователь был гибридным.
        - Локальные пользователи в гибридном развертывании будут взаимодействовать с интерактивными пользователями, представленными в локальном каталоге, как если бы они были гибридными. 
        - Локальные пользователи в гибридном развертывании будут взаимодействовать с пользователями сети, которые не представлены в локальной службе AD как Федеративные.
    - На **[рисунке D](#figure-d)** выше пользователь E является локальным в аккуиредкомпани. <span>com.  Пользователь E будет взаимодействовать с пользователем D (размещен в сети), используя стандартный гибридный интерфейс, но пользователь E будет иметь Федеративные взаимодействия с пользователями A, B и C, так как они не представлены в локальном каталоге. Однако пользователи A, B и C будут взаимодействовать с пользователем E так, как если бы пользователь был в гибридной конфигурации.
    - Последствия взаимодействия гибридной среды и Федерации:
        - Для федеративных пользователей присутствие не подписывается автоматически, если пользователь не помечен как контакт.
        - Переадресация звонков между федеративными доменами не работает.
        - Сценарии передачи вызовов более ограничены.
        - Регулирование можно применять к федеративным трафику.
- С учетом этого асимметричного интерфейса официальная поддержка вызовов в межлокальных сценариях между локальным пользователем и облачным пользователем, не находящейся в локальном каталоге, ограничена только одноранговой связью. 
    - Переадресация звонков, передача, очереди вызовов и т. д. между этими пользователями не поддерживаются.
    - Эти неподдерживаемые сценарии связи по-прежнему будут включены, но во многих случаях они могут работать непредсказуемым образом. 
    - На приведенном выше **[рисунке D](#figure-d)** пользователь E является локальным, а звонки с пользователями A, B и C будут поддерживаться только как одноранговые. (Звонки с пользователем D не будут иметь ограничений поддержки.)  Однако после перемещения локального пользователя е в облако это ограничение больше не применяется.
- Если в вашей среде имеется несколько развертываний Skype для бизнеса Server 2019, то для использования автосекретаря в Организации может быть настроено только одно из этих развертываний, так как для этого компонента требуется гибридная конфигурация Skype для бизнеса Server. 
- Порядок выполнения некоторых предыдущих действий можно изменить. Ключевое требование, которое должно быть удовлетворено, состоит в том, что если выполняются все указанные ниже условия.
    - Несколько локальных лесов Skype для бизнеса, синхронизированных с одним клиентом AAD
    - Разделенный домен включен с одним локальным лесом
    - По крайней мере один пользователь в гибридной организации перенесен в облако<br>   После этого *необходимо* отключить все остальные домены SIP в сети, используя любой другой локальный лес Skype для бизнеса. В противном случае Федерация между интерактивными пользователями в гибридной Организации и локальными пользователями в других организациях будет прерываться в одном направлении.

## <a name="implications"></a>Следствий

- Из-за ограничений в поддержке расширенных функций звонков, как описано выше, **Организации должны рассматривать эти ассиметричные состояния как транзитные в рамках миграции, а не помещаются в состояние стабильной**работы.  
- Организациям с несколькими локальными развертываниями Skype для бизнеса обычно следует начать с развертывания, которое может быть полностью перенесено в облако, чтобы можно было продолжить консолидацию. Это означает, что в некоторых случаях будут холдаутс определенные группы пользователей, для которых он пока недоступен для перемещения в Teams. Когда это рассматривается в сценариях с участием нескольких лесов Skype для бизнеса, начните перенос с другого леса, не имеющего этих ограничений, если это возможно.
- При переходе с локальной организации на облако пользователи с отношениями делегирования и/или, как правило, участвуют в сценариях переадресации вызовов, должны переноситься как единое целое.

## <a name="considerations-for-moving-to-teamsonly-mode"></a>Рекомендации по переходу в режим Теамсонли

При перемещении локальных пользователей в облако в гибридной среде их можно перенести в режим "только Skype для бизнеса" или "Теамсонли". *Если вы планируете переместить пользователей в режим Теамсонли, не забудьте сначала ознакомиться с этим разделом.*

- При назначении пользователю режима Теамсонли все сеансы разговора и вызовы от любого другого пользователя будут полагаться на клиента Teams этого пользователя. 
- Если пользователи, у которых есть Skype для бизнеса, изначально используют клиент Skype для бизнеса и не Teams, рассмотрите возможность установки Теамсупградеполици таким образом, чтобы маршрутизация на локальные пользователи всегда намерены в Skype для бизнеса, а не в Teams. Чтобы обеспечить правильную маршрутизацию чатов и звонков между пользователями Теамсонли и пользователями, которые по-прежнему работают со Skype для бизнеса локально, у локальных пользователей должно быть действительное значение Теамсупградеполици с одним из режимов SfB, а не на острова ( по умолчанию). 
    - Для этого *необходимо сначала задать для глобального экземпляра теамсупградеполици одно из следующих значений*:
        - Сфбвистеамсколлаб (рекомендуется)
        - SfBWithTeamsCollabAndMeetings
        - сфбонли
    - Политику на уровне клиента можно предоставить с помощью следующей команды:<br>`Grant-CsTeamsUpgradePolicy -PolicyName SfBWithTeamsCollab -Global`
    - Примечание: это необходимо сделать на уровне клиента, так как политика не может быть назначена отдельным пользователям, у которых нет SIP-адреса в сетевом каталоге. Несмотря на то что вы отключили доступ к сетевым доменам SIP для чистых локальных развертываний, пользователи в этих доменах не смогут использовать SIP-адреса в каталоге. Следовательно, единственным способом применения политики к этим локальным пользователям является назначение на уровне клиента. В отличие от этого, в случае, если пользователи гибридного развертывания имеют SIP-адрес в сетевом каталоге, им можно явно назначить политику, если она имеет значение, отличное от глобальной политики клиента.
- У клиента Teams еще не поддерживаются режимы SfB для Теамсупградеполици. Например, в этих режимах в настоящее время запускаются звонки в Teams и чата, хотя в будущем это не так. Это может привести к путанице среди пользователей, поскольку ответы иногда могут наземныйся в Teams и иногда в Skype для бизнеса в зависимости от обстоятельств. Рекомендуется отключать звонки и обмениваться сообщениями с помощью Теамсмессагингполици и Теамскаллингполици для пользователей, которые по-прежнему являются локальными.
