# <a name="cloud-consolidation-for-teams-and-skype-for-business"></a>Консолидация облачных хранилищ для Teams и Skype для бизнеса

Многие крупные организации имеют более одного локального леса службы AD, а в некоторых случаях — несколько развертываний Exchange и/или Skype для бизнеса Server (или сервера Lync Server). Кроме того, даже организации, у которых есть только один локальный лес, могут оказаться в аналогичной ситуации за счет бизнес-слияния или поглощения. По мере перемещения этих клиентов в облако они хотят консолидировать несколько экземпляров локальной рабочей нагрузки в облако в одну Microsoft 365 или Office 365 организацию. В этой статье описывается, как добиться этой консолидации для организаций с несколькими локальной развертываниями Skype для бизнеса (или Lync), которые хотят перенести нагрузку UC в облако Microsoft, например, Microsoft Teams и/или Skype для бизнеса Online.

[!INCLUDE [sfbo-retirement-skype](../../Hub/includes/sfbo-retirement.md)]

Традиционно рекомендация для клиентов в такой ситуации заключалась в том, чтобы сначала объединить локальные развертывания, а затем перейти в облако. В этой статье описывается решение, основанное на новых функциональных возможностях, которое позволяет организациям с несколькими развертываниями Skype для бизнеса одновременно перенести одно развертывание в одну организацию Microsoft 365 или Office 365, не делая локальной консолидации. Обратите внимание, что даже с помощью этой новой функции Skype для бизнеса Online и Microsoft Teams не поддерживают несколько лесов Skype для бизнеса/Lync в гибридном режиме с помощью одной Microsoft 365 или Office 365 организации. 

> [!Important]
> Прежде чем использовать это руководство для настройки, обязательно просмотрите и поймете [ограничения,](#limitations)так как они могут повлиять на организацию.

## <a name="overview-of-cloud-consolidation"></a>Обзор облачной консолидации

Консолидация всех пользователей из локального облака в облако в одной организации Microsoft 365 или Office 365 может быть достигнута для любой организации с несколькими Skype для бизнеса развертываниями при условии, что будут выполнены следующие ключевые требования:

- Должна быть задействована как минимум одна Microsoft 365 или Office 365 организации. Консолидация в сценариях с более чем одной организацией не поддерживается.
- В любой момент времени только один локальный лес Skype для бизнеса может находиться в гибридном режиме (общее адресное пространство SIP). Все другие локальные леса Skype для бизнеса должны оставаться в локальном режиме (и предположительно быть федеративными по отношению друг к другу). Обратите внимание, что эти  другие локальной организации могут синхронизироваться с AAD при желании с новыми функциями для отключения доменов [SIP](/powershell/module/skype/disable-csonlinesipdomain) в Интернете, доступных с декабря 2018 г.

Клиенты с развертыванием Skype для бизнеса в нескольких лесах должны полностью перенести всех пользователей одного гибридного леса Skype для бизнеса по отдельности в организацию Microsoft 365 или Office 365 с помощью функции общего пространства адресов SIP, а затем отключить гибрид с этим локальное развертывание, прежде чем перейти к миграции следующего локального Skype для бизнеса развертывания. До миграции в облако локальные пользователи остаются в федеративном состоянии с любыми пользователями, не представленными в этом же локальном каталоге пользователя.

## <a name="other-starting-points"></a>Другие отправные точки

Действия в каноническом примере выше предполагают, что организация начинается с двух федератированных локального развертывания без присутствия Microsoft или Office 365. Однако некоторые организации могут иметь существующий Microsoft 365 или Office 365, и в последовательность выше могут быть разные точки входа. Существует четыре типичных конфигурации:

- Несколько федератированных организаций, не Microsoft 365 или Office 365 организации. В этом случае начните с шага 1.
- Несколько федерарно-локальной организации, которые уже синхронизируются с несколькими Skype для бизнеса лесом в один клиент Azure AD. Такая организация похожа на гипотетическую организацию на рисунке A, которая выполнила шаги 1-6 и должна начаться с 7-го шага.
- Гибридная организация, которая федератирует с 1 или более другими чистыми локальной организацией, ни одна из которых не синхронизируется с AAD. Такая организация будет напоминать гипотетическую организацию на **рисунке E,** показанном ниже.
    - Эта организация похожа на рис. B, который выполнил шаги 1-9, за исключением:
        - Его не гибридные Skype для бизнеса еще *не* синхронизируются с Azure AD.
        -  Домены SIP в Интернете еще не отключены. 
    - Эти организации должны либо:
        - Полная миграция существующей гибридной организации и введите вышеуказанную последовательность на шаге 10.  OR,
        - Если перед завершением миграции гибридной организации необходимо синхронизировать все другие леса Skype для бизнеса в AAD, организация должна выполнить шаг 7 (отключить все домены SIP в любых других локальном развертывании Skype для бизнеса, синхронизируются в AAD), а затем включить AAD Подключение, а затем продолжить с шагом 10 (отключать исходное гибридное развертывание).       
                **Рис. E**<br>
                ![Диаграмма E](../media/cloudconsolidationfige.png)
- Чистая организация Skype для бизнеса Online (которая может или не использует Teams), которая Skype для бизнеса отдельной Skype для бизнеса организации. Как только эта организация отключает домен SIP для локальной организации и включает AAD Подключение для локальной Skype для бизнеса организации, она напоминает гипотетическую организацию, показанную на рисунке **[C,](#figure-c)** завершив шаги 1-11.

## <a name="limitations"></a>Ограничения

- Должна быть задействована как минимум одна Microsoft 365 или Office 365 организации. Консолидация в сценариях с более чем одной организацией не поддерживается.
- Только одно локальное Skype для бизнеса может быть в гибридном режиме (общее адресное пространство SIP) одновременно. Все другие Skype для бизнеса леса должны оставаться чисто локальной и должны быть федерадированы друг с другом и Microsoft 365 или Office 365 организацией.
- Перед переносом в облако существует асимметричный опыт для пользователей в этом развертывании, так как не все пользователи в Интернете представлены локально:
    - Этот опыт можно подвести следующим образом:
        - Любой пользователь, который находится в сети, будет взаимодействовать с локальной пользователями в гибридной среде, как если бы пользователь был гибридным.
        - Локально пользователи в гибридном развертывании будут взаимодействовать с интернет-пользователями, представленными в локальном каталоге, как если бы они были гибридными. 
        - Локально пользователи в гибридном развертывании будут взаимодействовать с сетевыми пользователями, которые не представлены в локальной AD в федераированном режиме.
    - На **[рисунке D](#figure-d)** выше пользователь E находится на локальной основе в AcquiredCompany. <span> com.  Пользователь E будет взаимодействовать с пользователем D (домашний онлайн) с помощью стандартного гибридного интерфейса, но пользователь E будет иметь федеративную возможность взаимодействия с пользователями A, B и C, так как они не представлены в локальном каталоге. Тем не менее, пользователи A, B и C будут взаимодействовать с пользователем E, как если бы пользователь был в гибридной.
    - Последствия гибридного взаимодействия и федерации:
        - Присутсвие автоматически не подписывается для федератированных пользователей, если только пользователь не помечен как контакт.
        - Переададка вызовов не работает между федерадными доменами.
        - Сценарии переноса вызовов являются более ограниченными.
        - Регулирование может применяться к федеративу трафика.
- Учитывая этот асимметричный опыт, официальная поддержка функций вызова в межучрещенных сценариях между локального пользователя и пользователем облака, который не находится в локальном каталоге, ограничивается одноранговой поддержкой. 
    - Переададка вызовов, передача, очереди вызовов и т. д. Между этими пользователями не поддерживается.
    - Эти не поддерживаемые сценарии вызовов по-прежнему будут отображаться включенной, но во многих случаях они не будут непредсказуемыми способами. 
    - На **[рисунке D](#figure-d)** выше пользователь E является локальной, и звонки с пользователями A, B или C будут поддерживаться только в одноранговом качестве. (Вызовы с пользователем D не будут иметь ограничений поддержки.)  Однако после перемещений локального пользователя E в облако это ограничение больше не применяется.
- Если у вас есть несколько развертывание Skype для бизнеса Server 2019 в вашей среде, только 1 из этих развертывание может быть настроено на использование организационных автосекретарь, так как эта функция требует Skype для бизнеса Server гибридной конфигурации. 
- Порядок некоторых предыдущих действий может быть скорректирован. Ключевое требование, которое должно быть выполнены, это то, что если все эти требования верны:
    - Несколько локального Skype для бизнеса синхронизируются с одним клиентом AAD
    - Разделенный домен включен в одном локальном лесу
    - По крайней мере один пользователь гибридной организации был перенесен в облако<br>   Затем необходимо *отключить* все другие домены SIP в Интернете из любого другого локального Skype для бизнеса леса. В противном случае федерация между пользователями сети в гибридной организации и локальной пользователями в других организациях будет нарушаться в одном направлении.

## <a name="implications"></a>Implications

- Так как существуют ограничения в поддержке расширенных функций вызовов, как описано выше, организации должны рассматривать эти асимметричные состояния как временные как часть миграции, а не преследовать их как **стабильное состояние.**  
- Организации с несколькими локальной Skype для бизнеса развертыванием обычно должны начинаться с развертывания, которое можно полностью перенести в облако, чтобы консолидация продолжалась. Понятно, что в некоторых случаях будут задерживаться определенные группы пользователей, для которых пока не целесообразно перейти в Teams. Если это рассматривается в сценариях, связанных с несколькими Skype для бизнеса лесами, приступить к миграции с другим лесом, который не имеет этих ограничений, если это вообще возможно.
- При переходе из локального в облако пользователи, которые имеют отношения делегирования и/или обычно участвуют в сценариях переададации вызовов, должны быть перемещены вместе в качестве единицы.

## <a name="considerations-for-moving-to-teamsonly-mode"></a>Соображения для перехода в режим TeamsOnly

При переходе пользователей из локального в облако в гибридной среде их можно переместить в режим Skype для бизнеса Only или TeamsOnly. *Если планируется переместить пользователей в режим TeamsOnly, сначала ознакомьтесь с этим разделом.*

- При назначении режима TeamsOnly пользователю все чаты и вызовы любого другого пользователя будут Teams клиента. 
- Если пользователи с Skype для бизнеса в основном используют Skype для бизнеса клиента, а не Teams, подумайте о настройке TeamsUpgradePolicy таким образом, чтобы маршрутизация для локального пользователя всегда была Skype для бизнеса, а не Teams. Чтобы обеспечить надлежащую маршрутику чатов и звонков между пользователями, которые являются TeamsOnly, и пользователями, которые по-прежнему используют Skype для бизнеса на локальной основе, локальное пользователи должны иметь эффективное значение TeamsUpgradePolicy с одним из режимов SfB, а не островами (по умолчанию). 
    - Для этого необходимо сначала установить глобальный экземпляр *TeamsUpgradePolicy* клиента для одного из этих значений:
        - SfBWithTeamsCollab (рекомендуется)
        - SfBWithTeamsCollabAndMeetings
        - SfBOnly
    - Вы можете предоставить политику для клиента с помощью этой команды:<br>`Grant-CsTeamsUpgradePolicy -PolicyName SfBWithTeamsCollab -Global`
    - Примечание. Это необходимо сделать на уровне клиента, так как политика не может быть назначена отдельным пользователям, у которых нет SIP-адреса в каталоге в Интернете. Несмотря на отключение доменов SIP в Интернете для вашего локального развертывания (s), пользователи в этих доменах не будут иметь SIP-адреса в каталоге онлайн по дизайну. Таким образом, единственный способ применить политику к локальному пользователю — назначить ее на уровне клиента. В отличие от этого, в гибридном развертывании пользователи будут иметь SIP-адрес в каталоге онлайн, чтобы им можно было явно задать политику, если требуется, чтобы у них было другое значение, чем глобальная политика клиента.
- Клиентская Teams еще не почитает режимы SfB TeamsUpgradePolicy. Например, если в этих режимах в настоящее время возможно Teams инициация вызовов и чатов, то в будущем этого не будет. Это может вызвать путаницу среди пользователей, так как ответы иногда могут Teams, а иногда Skype для бизнеса, в зависимости от обстоятельств. Рекомендуется отдельно отключить вызовы и чат через TeamsMessagingPolicy и TeamsCallingPolicy для пользователей, которые все еще находятся на месте.
