# <a name="cloud-consolidation-for-teams-and-skype-for-business"></a>Консолидация облачных хранилищ для Teams и Skype для бизнеса

Многие крупные организации имеют более одного локального леса службы AD, а в некоторых случаях — несколько развертываний Exchange и/или Skype для бизнеса Server (или сервера Lync Server). Кроме того, даже организации, у которых есть только один локальный лес, могут оказаться в аналогичной ситуации за счет бизнес-слияния или поглощения. По мере перемещения клиентов в облако они хотят объединить несколько экземпляров данной локальной рабочей нагрузки в облако в одну организацию Microsoft 365 или Office 365. В этой статье описывается, как добиться этой консолидации для организаций с несколькими локально развертыванием Skype для бизнеса (или Lync), которые хотят перенести свою нагрузку UC в облако Майкрософт, например Microsoft Teams и/или Skype для бизнеса Online.

Традиционно рекомендация для клиентов в такой ситуации заключалась в том, чтобы сначала объединить локальные развертывания, а затем перейти в облако. Хотя этот вариант по-прежнему существует, в этой статье описывается решение, основанное на новых функциях, позволяющих организациям с несколькими развертываниями Skype для бизнеса одновременно перенести одно развертывание в одну организацию Microsoft 365 или Office 365 без локальной консолидации. Обратите внимание, что даже с этой новой функциональностью Skype для бизнеса Online и Microsoft Teams не поддерживают несколько лесов Skype для бизнеса и Lync в гибридном режиме с одной организацией Microsoft 365 или Office 365. 

> [!Important]
> Прежде чем использовать это руководство для настройки, обязательно просмотрите и проявите [ограничения,](#limitations)так как они могут повлиять на вашу организацию.

## <a name="overview-of-cloud-consolidation"></a>Обзор облачной консолидации

Консолидацию всех пользователей из локальной организации в облако в одной организации Microsoft 365 или Office 365 можно достичь для любой организации с несколькими развертываниями Skype для бизнеса при условии, что выполнены следующие ключевые требования:

- Должна быть задействована не более одной организации Microsoft 365 или Office 365. Консолидация в сценариях с более чем одной организацией не поддерживается.
- В любой момент времени только один локальный лес Skype для бизнеса может находиться в гибридном режиме (общее адресное пространство SIP). Все другие локальные леса Skype для бизнеса должны оставаться в локальном режиме (и предположительно быть федеративными по отношению друг к другу). Обратите внимание, что эти  другие локальной организации могут синхронизироваться с AAD при наличии новых функций для отключения сетевых [доменов SIP,](https://docs.microsoft.com/powershell/module/skype/disable-csonlinesipdomain) доступных в декабре 2018 г.

Клиенты с развертываниями Skype для бизнеса в нескольких лесах должны полностью перенести всех пользователей одного гибридного леса Skype для бизнеса по отдельности в организацию Microsoft 365 или Office 365 с помощью функции общего адресного пространства SIP, а затем отключить гибридную функциональность в этом локальном развертывании, прежде чем перейти к миграции следующего локального развертывания Skype для бизнеса. До миграции в облако локальные пользователи остаются в федеративном состоянии с любыми пользователями, не представленными в этом же локальном каталоге пользователя.

## <a name="other-starting-points"></a>Другие начальные точки

В каноническом примере выше предполагается, что организация начинается с двух федераированных локального развертывания без присутствия Майкрософт или Office 365. Тем не менее в некоторых организациях может существовать место в Microsoft 365 или Office 365, и в этой последовательности могут существовать разные точки входа. Существует четыре типичных конфигурации:

- Несколько федераированных локальной организации без Microsoft 365 или Office 365. В этом случае начните с шага 1.
- Несколько федераированных локальной организации, которые уже синхронизировали несколько лесов Skype для бизнеса с одним клиентом Azure AD. Такая организация похожа на гипотетическую организацию на рисунке А, которая выполнила шаги 1–6 и должна начаться с шага 7.
- Гибридная организация, которая федерирует с 1 или более другими чисто локальной организацией, ни одна из которых не синхронизируется с AAD. Такая организация будет напоминать гипотетическую организацию на **рисунке E,** показанном ниже.
    - Эта организация похожа на рис. Б, где выполнены действия 1–9, за исключением:
        - Его не гибридные развертывания Skype для бизнеса пока *НЕ* синхронизируются с Azure AD.
        -  Сетевые домены SIP еще не отключены. 
    - Эти организации должны:
        - Завершите миграцию существующей гибридной организации и введите вышеперечисленную последовательность на шаге 10.  ИЛИ,
        - Если необходимо синхронизировать любые другие леса Skype для бизнеса с AAD до завершения миграции гибридной организации, организация должна выполнить шаг 7 (отключить все сетевые домены SIP в любом другом локальном развертывании Skype для бизнеса, которое будет синхронизироваться с AAD), а затем включить AAD Connect и только после этого продолжить выполнение шага 10 (списание исходного гибридного развертывания).

## <a name="limitations"></a>Ограничения

- Должна быть задействована не более одной организации Microsoft 365 или Office 365. Консолидация в сценариях с более чем одной организацией не поддерживается.
- Только один локального леса Skype для бизнеса может быть в гибридном режиме (общее адресное пространство SIP) одновременно. Все остальные локального леса Skype для бизнеса должны оставаться исключительно в локальной сети и должны быть федераированы друг с другом и организацией Microsoft 365 или Office 365.
- До миграции в облако для пользователей в этом развертывании существовала асимметричная емкость, так как не все пользователи в Сети представлены локально:
    - Этот опыт можно подвести к итогу следующим образом:
        - Любой пользователь, который находится в сети, будет взаимодействовать с пользователями локальной среды в гибридной среде, как если бы пользователь был гибридным.
        - Пользователи локальной сети в гибридном развертывании будут взаимодействовать с сетевыми пользователями, представленными в их локальном каталоге, как если бы они были гибридными. 
        - Пользователи локальной сети в гибридном развертывании будут взаимодействовать с сетевыми пользователями, которые не представлены в локальной службе AD как федераированные.
    - На **[рисунке D](#figure-d)** выше пользователь E находится локально в AcquiredCompany. <span> com.  Пользователь E будет взаимодействовать с пользователем D (в сети), используя стандартный гибридный интерфейс, но пользователь E будет иметь федеративную интерфейс с пользователями A, B и C, так как они не представлены в локальном каталоге. Однако пользователи A, B и C будут взаимодействовать с пользователем E, как если бы пользователь был в гибридной сети.
    - Последствия гибридного взаимодействия и федерации:
        - Для федераированных пользователей подписка на присутствие не подписывается автоматически, если пользователь не помечен как контакт.
        - Переададант вызовов не работает между федератными доменами.
        - Сценарии передачи вызовов ограничены.
        - Регулирование может применяться к федеративным трафикам.
- Учитывая асимметричную функциональность, официальные функции звонков в межлокаментных сценариях между локальной и облачной пользователями, которые не находится в локальном каталоге, ограничены одноранговой связью. 
    - Переададант вызовов, передача вызовов, очереди вызовов и т. д. между этими пользователями не поддерживается.
    - Эти не поддерживаемые сценарии звонков по-прежнему будут отображаться включенными, но во многих случаях они будут неудались непредсказуемым образом. 
    - На **[рисунке D](#figure-d)** выше пользователь E находится локально, а звонки с пользователями A, B или C будут поддерживаться только как одноранговая. (Вызовы с пользователем D не будут иметь ограничений поддержки.)  Однако после того как локального пользователя E перемещают в облако, это ограничение больше не применяется.
- Если в вашей среде развернуто несколько skype для бизнеса Server 2019, только для 1 из них можно настроить использование организационных автосекретарь, так как для этой функции требуется гибридная конфигурация Skype для бизнеса Server. 
- Порядок некоторых из предыдущих действий можно скорректировать. Ключевое требование, которое должно быть совме в том, что если все эти условия выполнены:
    - Синхронизация более одного локального леса Skype для бизнеса с одним клиентом AAD
    - Разделенный домен включен в одном локальном лесу
    - По крайней мере один пользователь в гибридной организации был перенесен в облако<br>   Затем необходимо *отключить все* остальные сетевые домены SIP из любого локального леса Skype для бизнеса. В противном случае федерация между сетевыми пользователями в гибридной организации и локальной организацией в других организациях будет разорвана в одном направлении.

## <a name="implications"></a>Последствия

- Поскольку существуют ограничения в поддержке расширенных функций звонков, как описано выше, организациям следует рассматривать эти асимметричные состояния как временные в рамках миграции и не рассматривать их как устойчивое **состояние.**  
- Организации с несколькими развертываниями Skype для бизнеса, как правило, должны начинаться с развертывания, которое можно полностью перенести в облако, чтобы консолидация продолжалась. Понятно, что в некоторых случаях существуют удержания определенных групп пользователей, для которых пока невозможно перейти на Teams. When this is a consideration in scenarios involving multiple Skype for Business forests, start migrating with another forest that does not have these limitations, if at all possible.
- При переходе от локальной к облаку пользователи с отношениями делегирования и/или, как правило, участвуют в сценариях переададации вызовов, должны быть перемещены вместе в качестве единицы.

## <a name="considerations-for-moving-to-teamsonly-mode"></a>Вопросы перехода на режим TeamsOnly

При переходе пользователей из локальной среды в облако в гибридной среде их можно переместить в режим "Только Skype для бизнеса" или "Только Teams". *Если вы планируете переместить пользователей в режим TeamsOnly, сначала ознакомьтесь с этим разделом.*

- При назначении пользователю режима "Только Teams" все чаты и звонки от любого другого пользователя будут переназначаться в клиент Teams этого пользователя. 
- Если пользователи с локальной skype для бизнеса в основном используют клиент Skype для бизнеса, а не Teams, рассмотрите возможность настройки TeamsUpgradePolicy таким образом, чтобы маршрутизация на этих пользователей всегда переналась в Skype для бизнеса, а не в Teams. Чтобы обеспечить правильную маршрутику чатов и звонков между пользователями, которые являются пользователями TeamsOnly, и пользователями, которые по-прежнему используют Skype для бизнеса локально, у пользователей локальной сети должно быть эффективное значение TeamsUpgradePolicy в одном из режимов SfB, а не "Острова" (по умолчанию). 
    - Для этого необходимо сначала установить для глобального экземпляра *TeamsUpgradePolicy* вашего клиента одно из указанных значений:
        - SfBWithTeamsCollab (рекомендуется)
        - SfBWithTeamsCollabAndMeetings
        - SfBOnly
    - Вы можете предоставить политику на все клиенты с помощью этой команды:<br>`Grant-CsTeamsUpgradePolicy -PolicyName SfBWithTeamsCollab -Global`
    - Примечание. Это необходимо сделать на уровне клиента, так как политика не может быть назначена отдельным пользователям, у которых нет SIP-адреса в сетевом каталоге. Хотя вы отключили сетевые домены SIP для чисто локального развертывания, пользователи в этих доменах не будут иметь SIP-адреса в сетевом каталоге по дизайну. Поэтому единственный способ применить политику к пользователям в локальной сети — назначить ее на уровне клиента. В отличие от этого, в гибридном развертывании пользователи будут иметь SIP-адрес в сетевом каталоге, чтобы им могли явно на назначенную политику при желании иметь значение, которое отличается от глобальной политики клиента.
- Клиентский режим Teams пока не работает с режимами SfB TeamsUpgradePolicy. Например, если в этих режимах в настоящее время возможно инициация зовите и чат в Teams, но в будущем это не так. Это может привести к путанице среди пользователей, так как ответы иногда могут быть в Teams, а иногда в Skype для бизнеса в зависимости от обстоятельств. Рекомендуется отдельно отключить вызовы и чат через TeamsMessagingPolicy и TeamsCallingPolicy для пользователей, которые по-прежнему находятся в локальной сети.
