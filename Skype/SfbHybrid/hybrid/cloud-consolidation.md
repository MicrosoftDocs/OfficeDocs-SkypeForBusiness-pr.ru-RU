# <a name="cloud-consolidation-for-teams-and-skype-for-business"></a>Консолидация облачных хранилищ для Teams и Skype для бизнеса

Многие крупные организации имеют более одного локального леса службы AD, а в некоторых случаях — несколько развертываний Exchange и/или Skype для бизнеса Server (или сервера Lync Server). Кроме того, даже организации, у которых есть только один локальный лес, могут оказаться в аналогичной ситуации за счет бизнес-слияния или поглощения. По мере перемещения этих клиентов в облако они хотят консолидировать несколько экземпляров локальной рабочей нагрузки в облако в одну организацию Microsoft 365 или Office 365. В этой статье описывается, как добиться этой консолидации для организаций с несколькими локальной развертываниями Skype для бизнеса (или Lync), которые хотят перенести нагрузку UC в облако Microsoft, например, Microsoft Teams и/или Skype для бизнеса Online.

Традиционно рекомендация для клиентов в такой ситуации заключалась в том, чтобы сначала объединить локальные развертывания, а затем перейти в облако. В этой статье описывается решение, основанное на новых функциональных возможностях, которое позволяет организациям с несколькими развертываниями Skype для бизнеса перенести одно развертывание одновременно в одну организацию Microsoft 365 или Office 365 без локальной консолидации. Обратите внимание, что даже с помощью этой новой функции Skype для бизнеса Online и Microsoft Teams не поддерживают несколько лесов Skype для бизнеса и Lync в гибридном режиме с одной организацией Microsoft 365 или Office 365. 

> [!Important]
> Прежде чем использовать это руководство для настройки, обязательно просмотрите и поймете [ограничения,](#limitations)так как они могут повлиять на организацию.

## <a name="overview-of-cloud-consolidation"></a>Обзор облачной консолидации

Консолидация всех пользователей из локальной сети в облако в одной организации Microsoft 365 или Office 365 может быть достигнута для любой организации с несколькими развертываниями Skype для бизнеса при условии, что будут выполнены следующие ключевые требования:

- Должна быть как минимум одна организация Microsoft 365 или Office 365. Консолидация в сценариях с более чем одной организацией не поддерживается.
- В любой момент времени только один локальный лес Skype для бизнеса может находиться в гибридном режиме (общее адресное пространство SIP). Все другие локальные леса Skype для бизнеса должны оставаться в локальном режиме (и предположительно быть федеративными по отношению друг к другу). Обратите внимание, что эти  другие локальной организации могут синхронизироваться с AAD при желании с новыми функциями для отключения доменов [SIP](/powershell/module/skype/disable-csonlinesipdomain) в Интернете, доступных с декабря 2018 г.

Клиенты с развертыванием Skype для бизнеса в нескольких лесах должны полностью перенести всех пользователей единого гибридного леса Skype для бизнеса по отдельности в организацию Microsoft 365 или Office 365 с использованием функций Общего пространства адресов SIP, а затем отключить гибрид с этим локальное развертывание, прежде чем перейти к миграции следующего локального развертывания Skype для бизнеса. До миграции в облако локальные пользователи остаются в федеративном состоянии с любыми пользователями, не представленными в этом же локальном каталоге пользователя.

Ниже приведены основные действия, необходимые для получения от исходного состояния до нужного состояния.  Обратите внимание, что некоторые организации могут обнаружить, что их отправная точка находится где-то в середине этих действий. Другие [отправные точки](#other-starting-points)см. в этой статье. Наконец, в некоторых случаях порядок может быть скорректирован в зависимости от необходимости. [Основные ограничения и ограничения](#limitations) описаны позже.

1.  Получите организацию Microsoft 365 или Office 365, если она еще не существует.
2.  Убедитесь, что все соответствующие домены SIP в обоих локальном развертывании проверены доменами Microsoft 365 или Office 365.
3.  Выберите одно развертывание Skype для бизнеса, которое будет гибридным с Microsoft 365 или Office 365. В этом примере мы будем использовать OriginalCompany. <span> com.
4.  [Включении AAD Connect для леса,](configure-azure-ad-connect.md) который сначала станет гибридным (OriginalCompany. <span> com). 
5.  Если вы введете Teams в свою организацию, установите политику [teamsUpgradePolicy](/powershell/module/skype/grant-csteamsupgradepolicy) для всей клиентской группы в SfBWithTeamsCollab или одном из других режимов SfBOnly или SfBWithTeamsCollabAndMeetings. Это необходимо для обеспечения маршрутивки звонков и чатов от пользователей, переехавших в Teams Only для пользователей, которые остаются на месте.
6.  На данном этапе рекомендуется (но пока не требуется до 11-го шага) включить [AAD Connect](cloud-consolidation-aad-connect.md) для другого леса (AcquiredCompany). <span> com). Если предположить, что подключение AAD включено в обоих лесах, эта оргия выглядит как **[рисунок A,](#figure-a)** который может быть общей отправной точкой для некоторых орготов. 
7.  Для всех доменов SIP, которые размещены в других локальном развертывании (в данном случае AcquiredCompany. <span> com), [отключить эти домены SIP](/powershell/module/skype/disable-csonlinesipdomain) в Skype для бизнеса Online с помощью `Disable-CsOnlineSipDomain` PowerShell. (Это новые функции с декабря 2018 г.).
8.  [Настройте гибрид Skype для бизнеса](configure-federation-with-skype-for-business-online.md) для OriginalCompany. <span> com (одно развертывание, которое по-прежнему включило домены SIP в Интернете).
9.  В гибридном развертывании (OriginalCompany. <span> com), [начните перемещение](move-users-between-on-premises-and-cloud.md) пользователей из Skype для бизнеса в облако (только для teams или нет), чтобы учетная запись была домашней в Skype для бизнеса Online. Теперь организация выглядит как **[рис. B](#figure-b)**. Основные изменения из рисунка A:
    - Пользователи из обоих локального каталога теперь находятся в AAD.
    - AcquiredCompany. <span> com — это отключенный домен SIP в Интернете.
    - Некоторые пользователи были перемещены через Интернет в Skype для бизнеса Online или Teams. (См. фиолетовый пользователь A.)
10. После перемещений всех пользователей в облако отключать гибрид с локальной развертыванием [Skype для](cloud-consolidation-disabling-hybrid.md) бизнеса для OriginalCompany. <span> com из Office 365:  
    - Отключить раздельный домен в организации Microsoft 365 или Office 365.
    - Отключить возможность связи с Microsoft 365 или Office 365 в OriginalCompany. <span> com на локальной основе.
    - Обновление записей DNS для OriginalCompany. <span> com, чтобы указать на Microsoft 365 или Office 365.
11. Если еще не сделано, [включении AAD Connect для следующего](cloud-consolidation-aad-connect.md) леса, который будет идти гибрид (AcquiredCompany. <span> com). На данный момент организация выглядит как **[рис.](#figure-c)** Это может быть еще одной общей отправной точкой для некоторых организаций. 
12. В PowerShell [включить домены SIP](/powershell/module/skype/enable-csonlinesipdomain) для следующего локального развертывания, которое будет гибридным, AcquiredCompany. <span> com. Это делается с `Enable-CsOnlineSipDomain` использованием новых функциональных возможностей, доступных с декабря 2018 г.
13. Если вы используете закрытую федерацию, необходимо добавить все домены SIP (за исключением .microsoftonline.com) чистого онлайн-клиента в качестве разрешенных доменов в том же \* Microsoft 365 или Office 365.  Обратите внимание, что это может занять некоторое время, прежде чем изменение вступает в силу, и нет никакого вреда в этом рано, поэтому мы предлагаем сделать это заблаговременно перед переходом на шаг 14.
14. Обновим локальное окружение, чтобы принять все домены SIP от онлайн-клиента, чтобы они совпадали.
    - [Обновив](cloud-consolidation-edge-certificates.md) САН во всех краях сертификатов, чтобы иметь то же значение, что и раньше, плюс значения для любых существующих доменов SIP в Интернете (за исключением *.microsoftonline.com), в данном случае Sip.OriginalCompany. <span> com.
    - Убедитесь, что OriginalCompany. <span> com — это [разрешенный домен](/powershell/module/skype/new-csalloweddomain) в локальном развертывании AcquiredCompany. Добавление разрешенных доменов.
15. [Включить гибрид Skype для бизнеса](configure-federation-with-skype-for-business-online.md) между локальной acquiredCompany. <span> com и облако.
16. При желании [перенаселять](move-users-between-on-premises-and-cloud.md)пользователей из локального в облако. Вы можете перенести пользователей непосредственно в [режим TeamsOnly](/microsoftteams/teams-and-skypeforbusiness-coexistence-and-interoperability) или перенести их сначала в Skype для бизнеса Online. Во время этого состояния организация выглядит как **[рисунок D](#figure-d)**.
17. После переноса всех [](cloud-consolidation-disabling-hybrid.md) пользователей отключать гибрид с локальной средой, чтобы сделать организацию *чистой облаком!*

## <a name="other-starting-points"></a>Другие отправные точки

Действия в каноническом примере выше предполагают, что организация начинается с двух федератированных локального развертывания без присутствия Microsoft или Office 365. Однако некоторые организации могут иметь существующий след Microsoft 365 или Office 365, и в последовательность выше могут быть разные точки входа. Существует четыре типичных конфигурации:

- Несколько федератированных локальной организации без Microsoft 365 или Office 365. В этом случае начните с шага 1.
- Несколько федерарно-локальной организации, которые уже синхронизируются с несколькими лесами Skype для бизнеса в один клиент Azure AD. Такая организация похожа на гипотетическую организацию на рисунке A, которая выполнила шаги 1-6 и должна начаться с 7-го шага.
- Гибридная организация, которая федератирует с 1 или более другими чистыми локальной организацией, ни одна из которых не синхронизируется с AAD. Такая организация будет напоминать гипотетическую организацию на **рисунке E,** показанном ниже.
    - Эта организация похожа на рис. B, который выполнил шаги 1-9, за исключением:
        - Его не гибридное развертывание Skype для бизнеса еще *не* синхронизируется с Azure AD.
        -  Домены SIP в Интернете еще не отключены. 
    - Эти организации должны либо:
        - Полная миграция существующей гибридной организации и введите вышеуказанную последовательность на шаге 10.  OR,
        - Если требуется синхронизировать любые другие леса Skype для бизнеса в AAD до завершения миграции гибридной организации, организация должна выполнить шаг 7 (отключить все домены SIP в любых других локальном развертывании Skype для бизнеса, которое синхронизируется с AAD), а затем включить AAD Connect и только затем продолжить с шагом 10 (вывести из эксплуатации исходное гибридное развертывание).       
                **Рис. E**<br>
                ![Диаграмма E](../media/cloudconsolidationfige.png)
- Чистая организация Skype для бизнеса Online (которая может или не может использовать Teams), которая федерация с отдельной локальной организацией Skype для бизнеса. После отключения домена SIP для локальной организации и подключения AAD для локальной организации Skype для бизнеса эта организация напоминает гипотетическую организацию, показанную на рисунке **[C,](#figure-c)** завершив действия 1-11.

## <a name="limitations"></a>Ограничения

- Должна быть как минимум одна организация Microsoft 365 или Office 365. Консолидация в сценариях с более чем одной организацией не поддерживается.
- Только один локально лес Skype для бизнеса может быть в гибридном режиме (общее пространство адресов SIP) одновременно. Все остальные локального леса Skype для бизнеса должны оставаться чисто локальной и должны быть федерадированы друг с другом и организацией Microsoft 365 или Office 365.
- Перед переносом в облако существует асимметричный опыт для пользователей в этом развертывании, так как не все пользователи в Интернете представлены локально:
    - Этот опыт можно подвести следующим образом:
        - Любой пользователь, который находится в сети, будет взаимодействовать с локальной пользователями в гибридной среде, как если бы пользователь был гибридным.
        - Локально пользователи в гибридном развертывании будут взаимодействовать с интернет-пользователями, представленными в локальном каталоге, как если бы они были гибридными. 
        - Локально пользователи в гибридном развертывании будут взаимодействовать с сетевыми пользователями, которые не представлены в локальной AD в федераированном режиме.
    - На **[рисунке D](#figure-d)** выше пользователь E находится на локальной основе в AcquiredCompany. <span> com.  Пользователь E будет взаимодействовать с пользователем D (домашний онлайн) с помощью стандартного гибридного интерфейса, но пользователь E будет иметь федеративную возможность взаимодействия с пользователями A, B и C, так как они не представлены в локальном каталоге. Тем не менее, пользователи A, B и C будут взаимодействовать с пользователем E, как если бы пользователь был в гибридной.
    - Последствия гибридного взаимодействия и федерации:
        - Присутсвие автоматически не подписывается для федератированных пользователей, если только пользователь не помечен как контакт.
        - Переададка вызовов не работает между федерадными доменами.
        - Сценарии переноса вызовов являются более ограниченными.
        - Регулирование может применяться к федеративу трафика.
- Учитывая этот асимметричный опыт, официальная поддержка функций вызова в межучрещенных сценариях между локального пользователя и пользователем облака, который не находится в локальном каталоге, ограничивается одноранговой поддержкой. 
    - Переададка вызовов, передача, очереди вызовов и т. д. Между этими пользователями не поддерживается.
    - Эти не поддерживаемые сценарии вызовов по-прежнему будут отображаться включенной, но во многих случаях они не будут непредсказуемыми способами. 
    - На **[рисунке D](#figure-d)** выше пользователь E является локальной, и звонки с пользователями A, B или C будут поддерживаться только в одноранговом качестве. (Вызовы с пользователем D не будут иметь ограничений поддержки.)  Однако после перемещений локального пользователя E в облако это ограничение больше не применяется.
- Если у вас есть несколько развертывание Skype для бизнеса Server 2019 в вашей среде, только 1 из этих развертывание может быть настроено на использование организационных автосекретарь, так как эта функция требует Skype для бизнес-сервера гибридной конфигурации. 
- Порядок некоторых предыдущих действий может быть скорректирован. Ключевое требование, которое должно быть выполнены, это то, что если все эти требования верны:
    - Несколько локального леса Skype для бизнеса синхронизируются с одним клиентом AAD
    - Разделенный домен включен в одном локальном лесу
    - По крайней мере один пользователь гибридной организации был перенесен в облако<br>   Затем необходимо *отключить* все другие домены SIP в Интернете из любого другого локального леса Skype для бизнеса. В противном случае федерация между пользователями сети в гибридной организации и локальной пользователями в других организациях будет нарушаться в одном направлении.

## <a name="implications"></a>Implications

- Так как существуют ограничения в поддержке расширенных функций вызовов, как описано выше, организации должны рассматривать эти асимметричные состояния как временные как часть миграции, а не преследовать их как **стабильное состояние.**  
- Организации с несколькими локальной развертываниями Skype для бизнеса обычно должны начинать с развертывания, которое можно полностью перенести в облако, чтобы консолидация продолжалась. Понятно, что в некоторых случаях будут задерживаться определенные группы пользователей, для которых пока не целесообразно переходить в Teams. Если это рассматривается в сценариях, связанных с несколькими лесами Skype для бизнеса, приступить к миграции с другим лесом, который не имеет этих ограничений, если это вообще возможно.
- При переходе из локального в облако пользователи, которые имеют отношения делегирования и/или обычно участвуют в сценариях переададации вызовов, должны быть перемещены вместе в качестве единицы.

## <a name="considerations-for-moving-to-teamsonly-mode"></a>Соображения для перехода в режим TeamsOnly

При переходе пользователей из локального в облако в гибридной среде их можно переместить в режим Skype для бизнеса или TeamsOnly. *Если планируется переместить пользователей в режим TeamsOnly, сначала ознакомьтесь с этим разделом.*

- При назначении пользователю режима TeamsOnly все чаты и вызовы любого другого пользователя будут высадлены в клиенте Teams этого пользователя. 
- Если пользователи с локальной командой Skype для бизнеса в основном используют клиент Skype для бизнеса, а не Teams, подумайте о настройке TeamsUpgradePolicy, чтобы маршрутизация этих локального пользователя всегда была в Skype для бизнеса, а не в Teams. Чтобы обеспечить надлежащую маршрутику чатов и звонков между пользователями, которые являются TeamsOnly, и пользователями, которые по-прежнему используют Skype для бизнеса на локальной основе, локально пользователи должны иметь эффективное значение TeamsUpgradePolicy с одним из режимов SfB, а не островами (по умолчанию). 
    - Для этого необходимо сначала установить глобальный экземпляр *TeamsUpgradePolicy* клиента для одного из этих значений:
        - SfBWithTeamsCollab (рекомендуется)
        - SfBWithTeamsCollabAndMeetings
        - SfBOnly
    - Вы можете предоставить политику для клиента с помощью этой команды:<br>`Grant-CsTeamsUpgradePolicy -PolicyName SfBWithTeamsCollab -Global`
    - Примечание. Это необходимо сделать на уровне клиента, так как политика не может быть назначена отдельным пользователям, у которых нет SIP-адреса в каталоге в Интернете. Несмотря на отключение доменов SIP в Интернете для вашего локального развертывания (s), пользователи в этих доменах не будут иметь SIP-адреса в каталоге онлайн по дизайну. Таким образом, единственный способ применить политику к локальному пользователю — назначить ее на уровне клиента. В отличие от этого, в гибридном развертывании пользователи будут иметь SIP-адрес в каталоге онлайн, чтобы им можно было явно задать политику, если требуется, чтобы у них было другое значение, чем глобальная политика клиента.
- Клиентская UX Teams пока не чтим режимы SfB TeamsUpgradePolicy. Например, если в этих режимах в настоящее время возможно инициирование вызовов и чатов в Teams, то в будущем этого не будет. Это может вызвать путаницу среди пользователей, так как ответы иногда могут приземлиться в Teams, а иногда Skype для бизнеса, в зависимости от обстоятельств. Рекомендуется отдельно отключить вызовы и чат через TeamsMessagingPolicy и TeamsCallingPolicy для пользователей, которые все еще находятся на месте.
