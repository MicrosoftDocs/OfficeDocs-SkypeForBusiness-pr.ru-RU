# <a name="move-users-between-on-premises-and-cloud"></a>Перемещение пользователей между локальной средой и облаком

В локальном развертывании сервера Skype для бизнеса Server, для которого включена гибридная среда, можно перемещать пользователей между локальной средой и облаком (будь то Microsoft Teams или Skype для бизнеса Online). Размещается ли пользователь локально или в облаке, это местоположение называется пользовательским домом Skype для бизнеса.

- Пользователи, которые находятся в помещении, взаимодействуют с локальной skype для бизнес-серверов.
- Пользователи, размещенные в Интернете, взаимодействуют со службой Skype для бизнеса Online.

*Пользователи teams по своей сути имеют домашний Skype для бизнеса, независимо от того, используют они Skype для бизнеса или нет.* Если у вас есть локальное приложение Skype для бизнеса, которое также использует Teams (бок о бок), эти пользователи находятся в помещении. Пользователи teams с Skype для бизнеса на локальной основе не имеют возможности взаимодействовать с пользователями Skype для бизнеса из своего клиента Teams, а также не могут общаться из Teams с пользователями федерадной организации. Такие функции доступны только после того, как пользователь перемещается из Skype для бизнеса в интернете. При перемещении пользователя в сетевой режим вы можете разрешить использовать Skype для бизнеса Online (и, при необходимости, Teams) или перевести их в режим Teams Only. Если в вашей организации уже используется Teams, настоятельно рекомендуем перевести пользователей в режим "Только Teams", который обеспечит маршрутизацию всех входящих чатов и звонков в клиент Teams. Дополнительные сведения см. в материале [Teams coexistence with Skype for Business and](/microsoftteams/coexistence-chat-calls-presence) Migration and [interoperability guidance for organisations using Teams together with Skype for Business.](/microsoftteams/migration-interop-guidance-for-teams-with-skype)

## <a name="prerequisites"></a>Необходимые компоненты

Необходимые условия для перемещения пользователя в облако (в режиме Skype только для бизнеса или в режиме Teams Only):

- Организация должна правильно настроить Azure AD Connect и синхронизировать все соответствующие атрибуты для пользователя, как описано в [Configure Azure AD Connect.](configure-azure-ad-connect.md)
- Гибрид Skype для бизнеса необходимо настроить, как описано в [настройках гибрида Skype для бизнеса.](configure-federation-with-skype-for-business-online.md)
- Пользователю должна быть назначена лицензия на Skype для бизнеса Online (план 2), и если будет использоваться режим Teams, то также потребуется лицензия Teams.  Дополнительно:
    - Если пользователь включен для телефонных конференций в помещениях, по умолчанию пользователь должен также иметь лицензию аудиоконференций, назначенную в Microsoft 365 или Office 365, прежде чем запустить перемещение пользователя в Интернете. После миграции в облако пользователь будет готов к проведению звуковой конференц-связи в облаке. Если по каким-либо причинам вы хотите переместить пользователя в облако, но не использовать функции аудиоконференций, вы можете переопрестить эту проверку, указав параметр `BypassAudioConferencingCheck` `Move-CsUser` в .
    - Если пользователь включен для Корпоративная голосовая связь на месте, по умолчанию он должен иметь лицензию системы телефонов, назначенную в Microsoft 365 или Office 365, прежде чем переместить пользователя в Интернете. После миграции в облако пользователь будет готов к корпоративной голосовой связи в облаке. Если по какой-то причине вы хотите переместить пользователя в облако, но не использовать функции телефонной системы, вы можете переопрестить эту проверку, указав `BypassEnterpriseVoiceCheck` параметр `Move-CsUser` в .


## <a name="moving-users"></a>Перемещение пользователей

При перемещении пользователя из локальной среды в облако:

- пользователь начинает использовать службы Skype для бизнеса Online для всех функций Skype для бизнеса;
- пользователям Teams разрешено взаимодействовать с пользователями Skype для бизнеса, и они также могут объединяться с другими организациями;
- Контакты из локального помещения перемещаются в облако (Skype для бизнеса или Teams).
- Существующие собрания, организованные ими в будущем, переносят в интернет: если пользователи перемещаются непосредственно в TeamsOnly (см. ниже), собрания преобразуются в собрания Teams, в противном случае собрания остаются Skype для бизнеса, но будут перенесены, чтобы они были организованы в Интернете, а не на месте.  Миграция собраний происходит асинхронно и запускается примерно через 90 минут после переноса пользователя.  Чтобы определить состояние миграции собрания, можно использовать командлет [Get-csMeetingMigrationStatus](../../SfbOnline/audio-conferencing-in-office-365/setting-up-the-meeting-migration-service-mms.md#managing-mms). Обратите внимание, что все содержимое, которое было передано ранее, не переносится.

Чтобы перемещать пользователей между локальной и облачной (будь то Teams или Skype для бизнеса Online), используйте командлет Move-CsUser или панель управления Skype для бизнеса администратора, которые являются локальной программой. Эти инструменты поддерживают три разных пути перехода.

- [От Skype для бизнеса Server (на месте)](move-users-from-on-premises-to-skype-for-business-online.md)до Skype для бизнеса Online .
- [От Skype для бизнеса Server (на месте)](move-users-from-on-premises-to-teams.md) непосредственно к Teams Only (которая также перемещает их в Skype для бизнеса Online).  Возможность непосредственного перемещения из локального помещения в Teams Only доступна в Skype для бизнеса Server 2019, а также в накопительном обновлении 8 для Skype для бизнеса Server 2015. Организации, использующие более ранние версии приложения Skype для бизнеса Server, могут переносить пользователей в группы "Только Teams", сначала переместив их в Skype для бизнеса Online, а затем применяя режим "Только Teams" к таким пользователям после их подключения к сети.
- [От online (независимо от того, только teams или нет) до локального.](move-users-from-the-cloud-to-on-premises.md)

## <a name="required-administrative-credentials"></a>Необходимые учетные данные администратора

Чтобы перемещать пользователей между локальной и облачной, необходимо использовать учетную запись с достаточными привилегиями как в локальной среде Skype для бизнеса Server, так и в организации Microsoft 365 или Office 365. Вы можете использовать одну учетную запись, которая имеет все необходимые привилегии, или использовать две учетные записи, в этом случае вы сможете получить доступ к локальному инструменту с помощью учетных данных локального доступа, а затем в этих средствах вы сможете предоставить дополнительные учетные данные для административной учетной записи Microsoft 365 или Office 365.  

- В локальной среде пользователю, выполняющему перемещение, должна быть назначена рольCSServerAdminstrator в службе Skype для бизнеса Server.
- В Microsoft 365 и Office 365 пользователь, исполняющий этот шаг, должен быть глобальным администратором или иметь роли администратора Skype для бизнеса и администратора пользователей.  

    > [!Important]
    > - Если вы используете панель управления Skype для бизнеса, вам будет предложено предоставить учетные данные для учетной записи Microsoft 365 или Office 365 с соответствующими ролями, как отмечалось выше. Необходимо предоставить учетную запись, которая заканчивается в .onmicrosoft.com. Если это невозможно, используйте Move-CsUser.
    >- Если вы используете Move-CsUser в PowerShell, вы можете использовать учетную запись, которая заканчивается в .onmicrosoft.com, или вы можете использовать любую учетную запись локальной, синхронизируются в Azure AD, при условии, что вы также укажите параметр HostedMigrationOverrideUrl в cmdlet. Значение URL-адреса переопределения переноса является вариантом следующего URL-адреса: https://adminXX.online.lync.com/HostedMigration/hostedmigrationService.svc<br>В этом URL-адресе замените XX двумя или тремя символами, которые определяются следующим образом:
    >   - В сеансе Skype для бизнеса Online PowerShell запустите следующий cmdlet:<br>`Get-CsTenant|ft identity`
    >    - Итоговая величина будет в следующем формате:<br>`OU=<guid>,OU=OCS Tenants,DC=lyncXX001,DC=local`
    >    - Двух- или трехзначный код — это XX, содержащийся в разделе DC=lyncXX001. Если это двухзначный код, это будет цифра, за которой следует номер (например, 0a). Если это трехзначный код, то это будут две буквы, за которыми следует цифра (например, jp1). Во всех случаях вы увидите 001 сразу после кода XX.


## <a name="voice-configuration-requirements"></a>Требования к конфигурации голосовой системы

Если пользователи настроены на корпоративный голос в локальном режиме, вам потребуется согласовать обновление конфигурации голосовой связи при переносе их в Интернет, или же вы можете перенести их без возможности телефонии. Доступные параметры зависят от того, будет ли пользователь использовать клиент Teams или Skype для бизнеса после их работы в Интернете:

- Вы можете обновить поставщика телефонии пользователя, чтобы использовать [Microsoft Calling Plan.](/microsoftteams/calling-plans-for-office-365) Это вариант, будут ли пользователи использовать клиенты Teams или Skype для бизнеса.
- Вы можете продолжать использовать локального поставщика PSTN:
  - Пользователи голосовой связи, которые будут использовать Teams, должны быть настроены для [прямой маршрутивки.](/microsoftteams/direct-routing-plan) Прямая маршрутная маршрутивка доступна только после того, как пользователь перемещается из локального помещения в интернет.
  - Пользователи голосовой системы, которые будут использовать клиент Skype для бизнеса после их перемещений в Интернете, должны быть настроены для функции гибридного голосового голоса Skype для бизнеса.

Дополнительные сведения о параметрах телефонии в гибридных средах, а также матрице поддержки см. в материале [User accounts in a hybrid environment with PSTN connectivity.](/microsoftteams/direct-routing-user-accounts-in-a-hybrid-environment)

## <a name="other-considerations"></a>Другие особенности

Политики (например, для управления обменом сообщениями, собраниями и звонками) в локальных и сетевых средах независимы друг от друга. Может потребоваться настроить любые политики в среде и назначить их пользователю перед перемещением этого пользователя из локального помещения в облако, чтобы у него была правильная конфигурация, как только они будут перенесены в интернет.
