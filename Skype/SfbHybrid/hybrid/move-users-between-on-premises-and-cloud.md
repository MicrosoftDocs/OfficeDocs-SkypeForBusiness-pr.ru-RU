# <a name="move-users-between-on-premises-and-cloud"></a>Перемещение пользователей между локальной средой и облаком

В локальном развертывании сервера Skype для бизнеса Server, для которого включена гибридная среда, можно перемещать пользователей между локальной средой и облаком (будь то Microsoft Teams или Skype для бизнеса Online). Размещается ли пользователь локально или в облаке, это местоположение называется пользовательским домом Skype для бизнеса.

- Пользователи, которые находятся локально, взаимодействуют с локальной skype для бизнеса servers.
- Пользователи, размещенные в Интернете, взаимодействуют со службой Skype для бизнеса Online.

*Пользователи Teams изначально имеют домашняя учетная запись Skype для бизнеса независимо от того, используют ли они Skype для бизнеса.* Если у вас есть пользователи локальной сети Skype для бизнеса, которые также используют Teams (рядом), эти пользователи находятся локально. Пользователи Teams с локальной skype для бизнеса не могут взаимодействовать с пользователями Skype для бизнеса из клиента Teams и не могут общаться из Teams с пользователями в федераированной организации. Такие функции доступны только после того, как пользователь будет перемещен из локальной службы Skype для бизнеса в сеть. При перемещении пользователя в сетевой режим вы можете разрешить использовать Skype для бизнеса Online (и, при необходимости, Teams) или перевести их в режим Teams Only. Если в вашей организации уже используется Teams, настоятельно рекомендуем перевести пользователей в режим "Только Teams", который обеспечит маршрутизацию всех входящих чатов и звонков в клиент Teams. Дополнительные сведения см. в руководстве по сосуществованию Teams со [Skype](/microsoftteams/coexistence-chat-calls-presence) для бизнеса, а также по миграции, а также по вопросам обеспечения связи для организаций, использующих Teams вместе со [Skype для бизнеса.](/microsoftteams/migration-interop-guidance-for-teams-with-skype)

## <a name="prerequisites"></a>Предварительные условия

Необходимые условия для перемещения пользователя в облако (в режиме "Только Skype для бизнеса" или "Только Teams"):

- Организация должна правильно настроить Azure AD Connect и синхронизировать все соответствующие атрибуты для пользователя, как описано в описании настройки [Azure AD Connect.](configure-azure-ad-connect.md)
- Гибридная конфигурация Skype для бизнеса должна быть настроена, как описано в описании настройки гибридной конфигурации [Skype для бизнеса.](configure-federation-with-skype-for-business-online.md)
- Пользователю должна быть назначена лицензия на Skype для бизнеса Online (план 2), и если будет использоваться режим Teams, то также потребуется лицензия Teams.  Дополнительно:
    - Если для пользователя включена возможность локальной аудиоконференции, то по умолчанию пользователю также должна быть назначена лицензия на аудиоконференцию в Microsoft 365 или Office 365 перед запуском перемещения пользователя в Интернет. После миграции в облако пользователь будет готов к проведению звуковой конференц-связи в облаке. Если по какой-либо причине вы хотите переместить пользователя в облако, но не использовать функции аудиоконференций, эту проверку можно переопречить, указав параметр `BypassAudioConferencingCheck` в `Move-CsUser` .
    - Если для пользователя включена возможность Корпоративная голосовая связь локально, по умолчанию пользователю должна быть назначена лицензия на телефонную систему в Microsoft 365 или Office 365, прежде чем вы переместите пользователя в сеть. После миграции в облако пользователь будет готов к корпоративной голосовой связи в облаке. Если по какой-либо причине вы хотите переместить пользователя в облако, но не использовать функции телефонной системы, эту проверку можно переопредить, указав `BypassEnterpriseVoiceCheck` параметр `Move-CsUser` в .


## <a name="moving-users"></a>Перемещение пользователей

При перемещении пользователя из локальной среды в облако:

- пользователь начинает использовать службы Skype для бизнеса Online для всех функций Skype для бизнеса;
- пользователям Teams разрешено взаимодействовать с пользователями Skype для бизнеса, и они также могут объединяться с другими организациями;
- Контакты из локальной сети перемещаются в облако (Skype для бизнеса или Teams).
- Существующие собрания, которые они организовывали в будущем, переносят в Интернет: если пользователи перемещаются непосредственно в TeamsOnly (см. ниже), собрания преобразуются в собрания Teams, в противном случае собрания остаются в Skype для бизнеса, но будут перенесены, поэтому они будут перенесены в Интернет, а не локально.  Миграция собраний происходит асинхронно и запускается примерно через 90 минут после переноса пользователя.  Чтобы определить состояние миграции собрания, можно использовать командлет [Get-csMeetingMigrationStatus](../../SfbOnline/audio-conferencing-in-office-365/setting-up-the-meeting-migration-service-mms.md#managing-mms). Обратите внимание, что все содержимое, которое было передано ранее, не переносится.

Чтобы переместить пользователей между локальной и облачной (в Teams или Skype для бизнеса Online), используйте командлет Move-CsUser или панель управления администрирования Skype для бизнеса, оба из которых являются локальной программой. Эти инструменты поддерживают три разных пути перехода.

- [От Skype для бизнеса Server (локально) до Skype для бизнеса Online.](move-users-from-on-premises-to-skype-for-business-online.md)
- [Из Skype для бизнеса Server (локально)](move-users-from-on-premises-to-teams.md) непосредственно в Teams Only (что также перемещает их в Skype для бизнеса Online).  Возможность непосредственного перемещения из локальной сети в Teams доступна только в Skype для бизнеса Server 2019, а также накопительном обновлении 8 для Skype для бизнеса Server 2015. Организации, использующие более ранние версии приложения Skype для бизнеса Server, могут переносить пользователей в группы "Только Teams", сначала переместив их в Skype для бизнеса Online, а затем применяя режим "Только Teams" к таким пользователям после их подключения к сети.
- [Из Интернета (только в Teams или нет) в локальное решение.](move-users-from-the-cloud-to-on-premises.md)

## <a name="required-administrative-credentials"></a>Необходимые учетные данные администратора

Для перемещения пользователей между локальной средой и облаком необходимо использовать учетную запись с достаточными привилегиями как в локальной среде Skype для бизнеса Server, так и в организации Microsoft 365 или Office 365. Можно использовать либо одну учетную запись со всеми необходимыми привилегиями, либо две учетные записи. В этом случае вы будете получать доступ к локальному средству с помощью учетных данных локальной службы, а затем в этих средствах у вас будут дополнительные учетные данные для учетной записи администратора Microsoft 365 или Office 365.  

- В локальной среде пользователю, выполняющему перемещение, должна быть назначена рольCSServerAdminstrator в службе Skype для бизнеса Server.
- В Microsoft 365 и Office 365 пользователь, который выполняет перемещение, должен быть глобальным администратором или иметь роли администратора Skype для бизнеса и администратора пользователя.  

    > [!Important]
    > - Если вы используете панель управления "Администратор Skype для бизнеса", вам будет предложено в качестве учетной записи Microsoft 365 или Office 365 в качестве учетной записи у вас должны быть соответствующие роли, как было отмечено выше. Необходимо предоставить учетную запись, заканчиваемую на .onmicrosoft.com. Если это невозможно, используйте Move-CsUser управления.
    >- Если вы используете Move-CsUser в PowerShell, можно использовать учетную запись, которая заканчивается на .onmicrosoft.com, или любую из локальной учетной записи, синхронизируются с Azure AD при условии, что вы также указываете параметр HostedMigrationOverrideUrl в этом параметре. Значение URL-адреса переопределения переноса является вариантом следующего URL-адреса: https://adminXX.online.lync.com/HostedMigration/hostedmigrationService.svc<br>В url-адресе выше замените XX двумя или тремя символами, которые определены следующим образом:
    >   - В сеансе PowerShell Skype для бизнеса Online запустите следующий cmdlet:<br>`Get-CsTenant|ft identity`
    >    - Итоговые значения будут иметь следующий формат:<br>`OU=<guid>,OU=OCS Tenants,DC=lyncXX001,DC=local`
    >    - Двух- или трехзначный код — это XX-код, содержащийся в разделе, DC=lyncXX001. Если это двухзначный код, это будет цифра, за которой следует число (например, 0a). Если это трехзначный код, за ним следуют две буквы, за которыми следуют цифры (например, jp1). Во всех случаях вы увидите код 001 сразу после кода XX.


## <a name="voice-configuration-requirements"></a>Требования к конфигурации голосовой почты

Если пользователи настроены для корпоративной голосовой связи в локальной организации, вам потребуется координировать обновление конфигурации голосовой связи при их переносе в интернет или же перенести их без возможности телефонии. Доступные варианты зависят от того, будет ли пользователь использовать клиент Teams или Skype для бизнеса после работы в сети:

- Вы можете обновить поставщика телефонии пользователя, чтобы использовать план [звонков Майкрософт.](/microsoftteams/calling-plans-for-office-365) Этот параметр позволяет пользователям использовать клиенты Teams или Skype для бизнеса.
- Вы можете продолжать использовать локального поставщика STN:
  - Для пользователей голосовой связи, которые будут использовать Teams, необходимо настроить [прямую маршрутику.](/microsoftteams/direct-routing-plan) Прямая маршрутка доступна только после того, как пользователь перемещается из локальной сети в сеть.
  - Для пользователей голосовой почты, которые будут использовать клиент Skype для бизнеса после их перемещений в сеть, необходимо настроить функции гибридной голосовой почты Skype для бизнеса.

Дополнительные сведения о параметрах телефонии в гибридных средах, а также о матрице поддержки см. в сведениях об учетных записях пользователей в гибридной среде с возможностью подключения к [STN.](/microsoftteams/direct-routing-user-accounts-in-a-hybrid-environment)

## <a name="other-considerations"></a>Другие особенности

Политики (например, для управления обменом сообщениями, собраниями и звонками) в локальных и сетевых средах независимы друг от друга. Может потребоваться настроить любые политики в среде и назначить их пользователю, прежде чем перемещать этого пользователя из локальной среды в облако, чтобы у него была правильная конфигурация сразу после переноса в сеть.
