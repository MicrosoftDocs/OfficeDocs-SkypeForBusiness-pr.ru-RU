# <a name="plan-for-media-bypass-in-cloud-connector-edition"></a>Планирование обхода сервера мультимедиа в Cloud Connector Edition
 
В этом разделе приведены рекомендации по планированию для реализации обхода сервера-посредника с помощью Cloud Connector Edition версии 2,0 и более поздних версий. Сведения о развертывании обхода сервера мультимедиа можно найти [в статье Deploy Media обход в Cloud Connector Edition](deploy-media-bypass-in-cloud-connector.md).
  
Обход сервера-посредника позволяет клиенту отправлять мультимедиа напрямую на следующий транзитный номер телефонной сети общего пользования (PSTN), шлюз или пограничный контроллер сеансов (SBC), а также удалять компонент Cloud Connector Edition из пути к носителю.
  
Обход сервера мультимедиа может способствовать повышению качества голосовой связи за счет уменьшения задержки, возможности потери пакетов и количества точек возможного сбоя. Отключение обработки мультимедиа для обходных вызовов сокращает нагрузку на Cloud Connector, которая обеспечивает более высокое количество одновременных вызовов и может повысить масштабируемость. 
  
 Освобождение облачного соединителя из задач обработки мультимедиа может уменьшить количество устройств Cloud Connector, необходимых инфраструктуре, поэтому следует по возможности включить обход сервера мультимедиа.
  
## <a name="how-media-bypass-affects-media-and-signaling-pathways"></a>Влияние обхода сервера мультимедиа на каналы передачи сигналов

Хотя передача сигналов имеет такой же путь с обходом сервера или без него, поток мультимедиа будет отличаться. На следующих схемах показаны пути мультимедиа и передачи сигналов в топологиях с обходом сервера мультимедиа и без него. 
  
Например, в следующей топологии, в которой не используется обход сервера-посредника, клиент Skype для бизнеса помещает Звонок на внешний номер, сигналы SIP передаются в Office 365, а Office 365 направляет трафик сигналов в соответствии с политикой голосовых вызовов конечных пользователей. Для пользователей Cloud Connector политика голосовой связи направляет трафик на пограничный сервер Cloud Connector, который затем направляет трафик сигналов на пограничный контроллер сеансов PSTN (SBC) или шлюз с помощью сервера-посредника Cloud Connector. Поток мультимедиа из клиента Skype для бизнеса на сервер-посредник Cloud Connector, а затем на SBC или шлюз, как показано на следующей схеме:
  
**Пути мультимедиа и сигналов без обхода сервера мультимедиа**

![передача сигналов без обхода сервера мультимедиа](../../media/5cd7e3bf-2565-4bd9-ad5a-f03e13c01060.png)
  
Входящий звонок от PSTN использует один и тот же контур сигналов в обратном направлении. Для внутренних пользователей мультимедиа по-прежнему будет по-прежнему передаваться между клиентом Skype для бизнеса и сервером-посредником Cloud Connector, а затем SBC или шлюзом.
  
В следующей топологии, в которой применяется обход сервера-посредника, передача сигналов осуществляется по одному и тому же пути, но поток мультимедиа напрямую передается между клиентом Skype для бизнеса и SBC или шлюзом, как показано на следующей схеме:
  
**Пути мультимедиа и сигналов с обходом сервера мультимедиа**

![передача сигналов с обходом сервера мультимедиа](../../media/60400c38-4921-4964-89f2-5e53b68fb497.png)
  
## <a name="multi-site-scenario-and-media-bypass"></a>Сценарий с несколькими сайтами и обход сервера-посредника

Обход сервера-посредника также полезен, если вы хотите предоставить услуги телефонии нескольким сайтам с помощью одного устройства Cloud Connector. Так как Cloud Connector не может маршрутизировать вызовы на основе исходных или конечных номеров, большинство предприятий развертывают SBC или шлюз за Cloud Connector для принятия решений по маршрутизации. Обход сервера мультимедиа в этом сценарии исключает переход между клиентом и центральным SBC или шлюзом, как показано на следующей схеме:
  
**Приложение с несколькими сайтами**

![Пример многосайтового сайта Cloud Connector](../../media/ace8dc3c-1082-46a2-b8b4-98cbf678620e.png)
  
1. Трафик SIP направляется от пользователя в Цюрихе к Office 365.
    
2. Затем трафик перенаправляется на устройство Cloud Connector в Амстердам, как указано в политике маршрутизации голосовых вызовов пользователей.
    
3. Устройство Cloud Connector в Амстердам отправляет трафик SIP центральному шлюзу в Амстердам.
    
4. Центральный шлюз в Амстердам выполняет соответствующие решения по маршрутизации, а затем отправляет трафик на SBC или шлюз в Цюрихе, а поток мультимедиа напрямую между клиентом Skype для бизнеса и SBC или шлюза в Амстердам.
    
   Такой подход позволяет обслуживать больше пользователей в одном развертывании Cloud Connector, где централизованно используется Cloud Connector. Несмотря на то, что Cloud Connector исключается из пути к носителю, в централизованном многоязычном носителе с несколькими сайтами может по-прежнему передаваться через глобальную сеть, так как это необходимо для последовательного соединения или шлюза.
  
Если клиент находится вне корпоративной сети, в которой размещается исходящий вызов, трафик мультимедиа проходит через серверы пограничного сервера и сервера-посредника в Cloud Connector и канал глобальной сети между Цюрихе и Амстердам, как показано на следующей схеме:
  
![Пример многосайтового сайта Cloud Connector 2](../../media/ef95839c-4552-440e-9698-7615707a1b50.png)
  
## <a name="supported-clients-for-media-bypass"></a>Поддерживаемые клиенты для обхода сервера мультимедиа

При первом выпуске обхода сервера-посредника единственным поддерживаемым клиентом является клиент Windows Skype для бизнеса 2016, который входит в состав приложений Microsoft 365 для предприятий, версия 16.0.7870.2020 или более поздняя. Клиенты могут использовать любой канал: Current, отложенный или первый выпуск, отложенный. 
  
> [!NOTE]
> Если вы используете клиентское VPN-решение в сочетании с клиентом Skype для бизнеса, обход сервера-посредника поддерживается только в конфигурации с разделением VPN. 
  
Для получения дополнительных сведений о каналах выпуска, ознакомьтесь со статьей [Обзор каналов обновления для приложений Microsoft 365 для предприятий](https://support.office.com/article/Overview-of-update-channels-for-Office-365-ProPlus-9ccf0f13-28ff-4975-9bd2-7e4ea2fefef4?ui=en-US&amp;rs=en-US&amp;ad=US).
  
Для текущей версии клиентов в разных каналах ознакомьтесь [со сведениями о выпуске обновлений для приложений Microsoft 365 для предприятий](https://docs.microsoft.com/officeupdates/release-notes-office365-proplus). 
  
## <a name="cloud-connector-capacity-considerations-with-media-bypass"></a>Рекомендации по емкости Cloud Connector с обходом сервера мультимедиа

Без обхода сервера-посредника — и в зависимости от аппаратного обеспечения — устройство Cloud Connector может обрабатывать от 50 до 500 одновременных вызовов, требующих передачи данных через сервер-посредник. Более подробную информацию можно найти в статье [Plan for Skype for Business Cloud Connector Edition](https://technet.microsoft.com/library/mt605227.aspx). 
  
Если включен режим обхода сервера-посредника, внутренние клиенты в поддерживаемой версии не используют сервер-посредник, поэтому число внутренних клиентов может значительно увеличиться. 
  
Как отмечалось выше, внешние клиенты или неподдерживаемые клиенты будут использовать пограничные и серверы-посредники Cloud Connector для мультимедиа. При расчете количества устройств Cloud Connector, которые должны быть размещены на сайте, необходимо учитывать трафик от внешних пользователей и пользователей на неподдерживаемых клиентах.
  
## <a name="cloud-connector-supports-always-bypass-mode"></a>Cloud Connector поддерживает режим "всегда обходить"

Cloud Connector поддерживает только режим "всегда обходить". В локальных средах существует два варианта: всегда обходится и используйте сведения о сайтах и регионах.
  
Всегда обходить означает, что обход сервера мультимедиа будет выполняться для всех звонков по сети PSTN с внутренними клиентами в качестве источника или конечной точки. Чтобы определить, является ли клиент внутренним или внешним, используется веб-сайт на виртуальной машине сервера-посредника. Если клиент может подключиться к сайту, он считается внутренним и используется обход сервера-посредника. Если клиент не может связаться с сайтом (например, клиент находится в домашней сети), обход сервера-посредника не используется. 
  
Режим "всегда обходить" требует наличия неограниченного подключения между пользователями и шлюзами PSTN на сайте PSTN. 
  
Более подробную информацию можно найти в статье [Plan for Skype for Business Cloud Connector Edition](https://technet.microsoft.com/library/mt605227.aspx). 
  
Например, на приведенной ниже схеме пользователи Европы должны быть хорошо подключены к трем пограничным контроллерам сеансов (SBCs) в Амстердам, тогда как пользователи из США должны быть хорошо подключены к двум SBCs в Сиэтле. Надежное подключение означает, что они расположены на тех же сетевых сайтах, что и SBCs или шлюзы, или по каналам глобальной сети, имеющим достаточную пропускную способность.
  
![Емкость Cloud Connector](../../media/efb2269b-d44f-474e-aea8-c5158e729cfe.png)
  
> [!NOTE]
> Если пользователь из Цюрихе перемещается в офис Seattle и вы хотите использовать внутреннюю сеть для доставки мультимедийного трафика между проходящими пользователями и шлюзами в Европе (а не в Интернете), необходимо убедиться, что Office в Сиэтле и в приложении для европейских SBCs и шлюзов надежно подключены. 
  
## <a name="codecs-used-in-media-bypass"></a>Кодеки, используемые в обход сервера мультимедиа

При включенном обходе сервера мультимедийного трафика между клиентом и SBC или шлюзом используется кодек G. 711.
