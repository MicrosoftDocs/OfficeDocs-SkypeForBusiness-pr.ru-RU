# <a name="plan-for-media-bypass-in-cloud-connector-edition"></a>План обхода сервера-посредника в Cloud Connector Edition
 
Ознакомьтесь с этой статьей, чтобы изучить сведения по планированию для реализации обхода сервера-посредника с помощью Cloud Connector Edition версии 2.0 и более поздних версий. Сведения о развертывании мультимедиа обходить, см [Развертывание сервера-посредника в облаке соединителя Edition](deploy-media-bypass-in-cloud-connector.md).
  
Обход сервера-посредника позволяет клиентским отправить мультимедиа следующего прыжка общедоступных переключения телефонной сети общего пользования (PSTN) — шлюза или пограничный контроллер сеансов (SBC) — и ликвидировать компонент Edition соединителя облака из пути.
  
Обход сервера-посредника может улучшить качество звука, поскольку уменьшается задержка, вероятность потери пакетов и число точек вероятного сбоя. Исключения с мультимедиа для вызовов, обходящих снижает нагрузку на облако соединителя, который позволяет чем больше число одновременных звонков, и повышает масштабируемость. 
  
 Освобождение облачных соединителя из задачи обработки мультимедиа может сократить число appliances соединителя облачных необходимую инфраструктуру, следует включить обход сервера-посредника по возможности.
  
## <a name="how-media-bypass-affects-media-and-signaling-pathways"></a>Влияние обхода сервера-посредника на пути передачи медиаданных и сигналов

В то время как передача сигналов происходит по одному и тому же каналу с обходом сервера-посредника или без него, поток медиаданных будет отличаться. На следующих схемах показаны основные пути мультимедиа и передачи сигналов в топологиях с обходом сервера-посредника и без такого обхода.  
  
Например, в следующей топологией — которого does не прием на работу сервера-посредника — Скайп для клиента Business звонит ТСОП на внешний номер, сигналы SIP переходит к Office 365 и Office 365 затем направляет передачи трафика в соответствии с голосовой связи конечных пользователей политика. Для соединителя облачных пользователей к политике голосовой связи направляет сигналы трафик на пограничном сервере облачных соединитель, который затем направляет передачи трафика для шлюза через сервер-посредник облачных соединителя или пограничный контроллер PSTN сеансов (SBC). Мультимедиа потоков Скайп для клиента Business на облако соединителя промежуточный сервер, а затем SBC или шлюз, как показано на рисунке:
  
**Пути передачи мультимедийных данных и сигналов при отключенном режиме обхода сервера-посредника**

![передача сигналов без обхода сервера-посредника](../../media/5cd7e3bf-2565-4bd9-ad5a-f03e13c01060.png)
  
Для входящего звонка по ТСОП в обратном направлении используется тот же канал сигнала. Для внутренних пользователей мультимедиа будет по-прежнему в конечном счете передаются между Скайп для бизнеса клиента и сервера-посредника соединителя облачных и SBC или шлюза.
  
В следующем топологии — которого does прием на работу сервера-посредника — сигналы использует тот же путь, но потоков мультимедиа напрямую между Скайп для бизнеса и SBC или шлюз, как показано на рисунке:

## <a name="multi-site-scenario-and-media-bypass"></a>Сценарий с несколькими сайтами и обход сервера-посредника 

Обход сервера-посредника также полезна, если необходимо использовать для предоставления услуг телефонии для нескольких узлов с помощью одно устройство соединителя облака. Так как соединитель облако не может маршрутизировать вызовы на основании числа источника и назначения, большинство предприятий развертывания SBC или шлюза за облачных соединителя для принятия решений маршрутизации. Обход сервера-посредника в этом сценарии исключает прыжок между клиентом и центральным контроллером SBC или шлюзом, как показано на следующей схеме.
  
**Многосайтовое приложение**

![Пример многосайтового развертывания Cloud Connector](../../media/ace8dc3c-1082-46a2-b8b4-98cbf678620e.png)
  
1. Трафик SIP потоков пользователя в Цюрихе в Office 365.
    
2. Трафик затем направляет appliance соединителя облака в Амстердам, указанный в политику маршрутизации голосовой связи пользователя.
    
3. Соединитель облачных устройства для обеспечения связи в Амстердам отправляет трафик SIP центра шлюза в Амстердам.
    
4. Центра шлюза в Амстердам делает соответствующую решений о маршрутизации и отправляет трафик SBC или шлюз в Цюрихе, при потоков мультимедиа напрямую между Скайп для бизнеса и SBC или шлюз в Амстердам.
    
 Такой подход позволяет обслуживания нескольких пользователей на один соединитель облачных развертывания которых осуществляется централизованно соединителя облака. Несмотря на то, что соединитель облачных исключается из пути, в сценарии централизованного нескольких сайтах мультимедиа может по-прежнему проходят через глобальную сеть дважды по мере необходимости проходить централизованного SBC или шлюза.
  
Если клиент находится за пределами корпоративной сети, поместив исходящего звонка, трафика мультимедиа потоки через серверы пограничных серверов и посредника глобальной сети и облачных соединителя связи между Цюрихе и Амстердам, как показано на рисунке:

## <a name="cloud-connector-capacity-considerations-with-media-bypass"></a>Сведения о пропускной способности Cloud Connector с обходом сервера-посредника

Без сервера-посредника — и в зависимости от оборудования, appliance облачных соединителя может обрабатывать от 50 для 500 одновременных звонков, которые требуют мультимедиа в поездке через сервер-посредник. Для получения дополнительных сведений см [Скайп облаке соединитель для выпуска для бизнеса](https://technet.microsoft.com/en-us/library/mt605227.aspx). 
  
При разрешенном обходе сервера-посредника внутренние клиенты на поддерживаемой версии не используют сервер-посредник, поэтому число внутренних клиентов может значительно увеличиться.  
  
Как указано выше, внешних клиентов или клиентов, неподдерживаемые будет использоваться серверов Edge соединителя облачных и посредника для мультимедиа. При расчете, сколько устройств для соединителя облаке необходимо разместить на сайте, необходимо учесть трафик от внешних пользователей и пользователей на клиентах не поддерживается.
  
## <a name="cloud-connector-supports-always-bypass-mode"></a>Cloud Connector поддерживает режим "Всегда обходить"

Соединитель облачных поддерживает только в режиме всегда обходить. Для локальных сред возможны два варианта: "Всегда обходить" и Use Site and Region Information (Использовать информацию о сайте и регионе).
  
"Всегда обходить" означает, что для всех звонков по ТСОП с внутренними клиентами в качестве исходной точки или точки назначения будет предприниматься попытка обхода сервера-посредника. Чтобы определить, является клиент внутренним или внешним, используется веб-сайт на виртуальной машине сервера-посредника. Если клиент может связаться с сайтом, он считается внутренним и используется обход сервера-посредника. Если клиент не может связаться с сайтом (например, клиент находится в домашней сети), обход сервера-посредника не используется.  
  
Для режима "Всегда обходить" требуется бесперебойное подключение между пользователями и шлюзами ТСОП на узле ТСОП.  
  
Для получения дополнительных сведений см [Скайп облаке соединитель для выпуска для бизнеса](https://technet.microsoft.com/en-us/library/mt605227.aspx). 
  
Например в приведенной ниже диаграмме Европа пользователей должно быть хорошее подключение для трех пограничных контроллеров сеансов (SBC) в Амстердам Запад "мне Нравится" пользователи должны быть хорошее подключение для двух SBC в Сиэтл. Стабильное подключение означает, что пользователи либо расположены на тех же сетевых сайтах, что и контроллеры SBC или шлюзы, либо подключаться по каналам глобальной сети, имеющим соответствующую пропускную способность.
  
![Емкость Cloud Connector](../../media/efb2269b-d44f-474e-aea8-c5158e729cfe.png)

## <a name="codecs-used-in-media-bypass"></a>Кодеки, используемые при обходе сервера-посредника

При разрешенном обходе сервера-посредника для трафика медиаданных между клиентом и контроллером SBC или шлюзом используется кодек G.711.
