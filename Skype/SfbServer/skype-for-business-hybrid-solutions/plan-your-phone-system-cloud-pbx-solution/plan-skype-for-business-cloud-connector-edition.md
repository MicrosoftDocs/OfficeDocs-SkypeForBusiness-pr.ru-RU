# <a name="plan-for-skype-for-business-cloud-connector-edition"></a>Планирование Skype для бизнеса Cloud Connector Edition

> [!Important]
> Cloud Connector Edition выйдет 31 июля 2021 вместе со Skype для бизнеса Online. После обновления вашей организации до Teams Узнайте, как подключить локальную телефонную сеть к Teams с помощью [прямой маршрутизации](https://docs.microsoft.com/MicrosoftTeams/direct-routing-landing-page).

Сведения о Skype для бизнеса Cloud Connector Edition, наборе упакованных виртуальных машин (ВМ), которые реализуют локальную сеть PSTN с телефонной системой (облачная УАТС).

Cloud Connector Edition может быть правильным решением для вашей организации, если у вас еще нет существующего развертывания Lync Server или Skype для бизнеса Server. Если вы по-прежнему изучаете, какое решение телефонной системы подходит для вашего бизнеса, ознакомьтесь с [разрешениями Microsoft для телефонии](https://docs.microsoft.com/SkypeForBusiness/hybrid/msft-telephony-solutions).

В этом документе описываются требования к Cloud Connector Edition и поддерживаемые топологии, а также планирование развертывания Cloud Connector Edition. Обязательно прочтите этот раздел перед настройкой среды Cloud Connector. Когда вы будете готовы развернуть и настроить Cloud Connector Edition, ознакомьтесь со статьей [Настройка и управление Skype для бизнеса Cloud Connector Edition](configure-skype-for-business-cloud-connector-edition.md).

Cloud Connector Edition 2,1 теперь доступно. Если вы еще не выполнили обновление до 2,1, ознакомьтесь со статьей " [обновление до новой версии Cloud Connector](upgrade-to-a-new-version-of-cloud-connector.md)". Файл установки можно найти по адресу [https://aka.ms/CloudConnectorInstaller](https://aka.ms/CloudConnectorInstaller) .

> [!NOTE]
> Корпорация Майкрософт поддерживает предыдущую версию Cloud Connector Edition в течение 60 дней после выпуска новой версии. Корпорация Майкрософт будет поддерживать версию 2.0.1 в течение 60 дней после выпуска 2,1, чтобы разрешить время на обновление. Все версии, предшествующие версии 2.0.1, больше не поддерживаются.

Cloud Connector Edition — это гибридное предложение, состоящее из набора упакованных виртуальных машин (ВМ), которые реализуют локальную сеть PSTN с телефонной системой. При развертывании минимальной топологии Skype для бизнеса Server в виртуализированной среде пользователи в вашей организации, размещенные в облаке, могут получать службы УАТС из Microsoft Cloud, но подключение PSTN обеспечивается с помощью существующей локальной инфраструктуры голосовой связи.

![Схема топологии, на которой показан шлюз облачной УАТС, соединяющий облачную УАТС с локальным развертыванием Skype для бизнеса.](../../media/bd898e69-6458-4276-aebe-1854f28ed6fa.png)

Так как Cloud Connector позволяет интегрировать службы телефонной системы с существующей телефонной средой (например, УАТС, аналоговые устройства и центры обработки звонков), вы можете реализовать поэтапную миграцию из существующего решения телефонии в телефонную систему.

Например, предположим, что у вашей компании есть сложный центр обработки звонков с определенными функциями, которые не поддерживаются телефонной системой. Вы можете оставить пользователей центра обработки звонков с существующим решением, но переместить других пользователей в телефонную систему.

Cloud Connector обеспечит маршрутизацию между пользователями, размещенными локально и в сети, и вы можете выбрать использование собственного поставщика PSTN с помощью телефонной системы.

При планировании развертывания Cloud Connector Edition учитывайте следующие рекомендации.

- Чтобы использовать Cloud Connector для использования облачных решений для голосовой связи, вам потребуется зарегистрироваться в организации Microsoft 365 или Office 365, которая включает телефонную систему. Если у вас еще нет организации Microsoft 365 или Office 365, вы можете узнать, как зарегистрироваться здесь: [microsoft 365 для бизнеса](https://products.office.com/business/office). Обратите внимание на то, что вам потребуется зарегистрировать план, включающий Skype для бизнеса Online.

Для версий Cloud Connector, предшествующих 2,0, требуется выделенная учетная запись Microsoft 365 или Office 365 с правами глобального администратора клиента.

## <a name="cloud-connector-edition-components"></a>Компоненты Cloud Connector Edition
<a name="BKMK_Components"> </a>

В Cloud Connector Edition разворачивается набор упакованных виртуальных машин, содержащих минимальную топологию Skype для бизнеса Server, состоящую из пограничного компонента, компонента-посредника и роли центрального хранилища управления (CMS). Кроме того, вы также устанавливаете контроллер домена, необходимый для внутреннего функционирования Cloud Connector. Эти службы настроены для гибридной среды с вашей организацией Microsoft 365 или Office 365, которая включает в себя службы Skype для бизнеса Online.

![Компоненты Cloud Connector Edition](../../media/f2d4b8a7-c2f4-4cfc-8137-f187399c1298.png)

Компоненты Cloud Connector предоставляют следующие функциональные возможности:

- **Пограничный компонент** — обмен данными между локальной топологией и веб-службами осуществляется через пограничный компонент, включающий в себя следующие компоненты:

  - **Пограничный сервер доступа** — обеспечивает маршрутизацию SIP между локальным развертыванием и Skype для бизнеса Online.

  - **Ретрансляция мультимедиа** — обеспечивает маршрутизацию мультимедиа между компонентом-посредником и другими конечными точками мультимедиа.

  - **Проверка подлинности Media Relay/MRAS** — создает маркеры для доступа к ретрансляции мультимедиа.

- **Маршрутизация исходящей** почты — обеспечивает балансировку нагрузки голосового трафика между шлюзами или SBCs, подключенными к устройству Cloud Connector. Звонки будут равномерно распределены между всеми шлюзами или SBCs, подключенными к устройству Cloud Connector.

    Обеспечивает маршрутизацию на шлюзы в соответствии с политиками. Поддерживаются только глобальные политики, основанные на целевых (исходящих) номерах PSTN.

- **Роль центрального хранилища управления (CMS)** . включает хранилище конфигурации для компонентов топологии, включая передачу CMS-файлов.

- **Реплика центрального хранилища управления (CMS)** синхронизирует сведения о конфигурации из глобальной базы данных CMS на сервере ролей CMS.

- **Контроллер домена** — доменные службы Active Directory облачного соединителя для хранения всех глобальных параметров и групп, необходимых для развертывания компонентов Cloud Connector. Для каждого устройства Cloud Connector будет создан один лес. Контроллер домена не должен иметь подключений к рабочей среде Active Directory. Службы Active Directory включают:

  - Доменные службы Active Directory

  - Службы сертификатов Active Directory для выпуска внутренних сертификатов

- **Компонент-посредник** — РЕАЛИЗУЕТ протокол SIP и протокол сопоставления шлюза мультимедиа между Skype для бизнеса и шлюзами PSTN. Включает реплику CMS, которая синхронизирует конфигурацию из глобальной базы данных CMS.

## <a name="cloud-connector-edition-topologies"></a>Топологии Cloud Connector Edition
<a name="BKMK_Topologies"> </a>

В целях этого обсуждения мы будем ссылаться на сайты PSTN. Сайт PSTN представляет собой комбинацию устройств Cloud Connector, развернутых в том же месте и с которыми подключены распространенные шлюзы PSTN. Сайты PSTN позволяют:

- Обеспечьте подключение к шлюзам, ближайшим к вашим пользователям.

- Обеспечение масштабируемости за счет развертывания нескольких устройств Cloud Connector в пределах одного или нескольких сайтов PSTN.

- Обеспечение высокой доступности за счет развертывания нескольких устройств Cloud Connector на одном сайте PSTN.

В этом разделе описываются сайты PSTN. Дополнительные сведения о планировании сайтов PSTN приведены в статье [Plan for Cloud Connector Edition PSTN](plan-for-cloud-connector-edition-pstn-sites.md).

Вы можете развернуть следующие топологии Cloud Connector:

- Одно устройство Cloud Connector Edition для каждого сайта PSTN. Эта топология рекомендуется только для целей оценки, так как она не обеспечивает высокую доступность.

- Несколько устройств Cloud Connector Edition на сайте PSTN для обеспечения высокой доступности.

- Несколько сайтов PSTN с несколькими устройствами Cloud Connector Edition для обеспечения масштабируемости и высокой доступности. Можно развернуть до 200 сайтов.

При планировании топологии учитывайте следующие возможности.

- В Cloud Connector 2,0 и более поздних версиях один сайт PSTN может иметь до 16 устройств Cloud Connector. Предыдущие версии поддерживают до 4 устройств для каждого сайта.

- Существует два типа аппаратных конфигураций, проверяемых с помощью Cloud Connector:

  - Более крупная версия может обрабатывать большие объемы одновременных вызовов и поддерживается во всех типах рабочих сред.

  - Эта уменьшенная версия предназначена для работы с низкими аппаратными средствами и может использоваться в целях оценки или для сайтов с низкими объемами звонков. При развертывании уменьшенной версии Cloud Connector по-прежнему необходимо осторожным требования к оборудованию производственного класса (например, двойные источники питания).

- Если у вас есть Cloud Connector версии 2,0 или более поздней и вы разворачиваете максимальную конфигурацию 16 устройств (с большим оборудованием), ваш сайт PSTN может обрабатывать до 8 000 одновременных вызовов. При развертывании меньшей версии поддерживаемого ограничения составляет 800.

    Кроме того, необходимо выделить некоторые устройства для обеспечения высокой доступности. Минимальная рекомендация состоит в том, что одно устройство должно быть зарезервировано для обеспечения высокой доступности.

  - В версии 2 при развертывании конфигурации 15 + 1 сайт PSTN может обрабатывать до 7 500 одновременных вызовов.

  - Если вы используете более раннюю версию и развертываете максимальную конфигурацию 3 + 1 (с большим оборудованием), ваш сайт PSTN может обрабатывать до 1500 одновременных вызовов. При развертывании меньшей версии поддерживаемого ограничения составляет 150.

-  Если требуется больше звонков на сайт PSTN, можно масштабировать их, развертывая дополнительные сайты PSTN в одном и том же расположении.

> [!NOTE]
> Если не указано иное, в схемах и примерах ниже предполагается использование более крупной версии Cloud Connector.

### <a name="single-cloud-connector-appliance-within-a-single-pstn-site"></a>Одно устройство Cloud Connector на одном сайте PSTN

На следующей схеме показано одно устройство Cloud Connector Edition на одном сайте PSTN. Обратите внимание, что Cloud Connector состоит из четырех виртуальных машин, установленных на одном физическом хост-компьютере, который находится в сети периметра в целях обеспечения безопасности.

![Один Cloud Connector с одним сайтом PSTN](../../media/7ffe6953-8c66-4323-940e-cd2e6c3c2a66.png)

### <a name="multiple-cloud-connector-appliances-within-a-single-pstn-site"></a>Несколько устройств Cloud Connector на одном сайте PSTN

 Для обеспечения масштабируемости и высокой доступности можно выбрать несколько выпусков Cloud Connector на одном сайте PSTN, как показано на следующей схеме. Примите во внимание следующее:

- Звонки распределяются в случайном порядке между облачными соединителями в одном пуле.

- При планировании мощности необходимо учитывать возможность обработки нагрузки, если один или несколько облачных соединителей переходят в автономный режим на основе следующих вычислений:

  - **N + 1 ящиков.** Для более крупной версии Cloud Connector поля N + 1 поддерживают 500 \* N одновременных вызовов с 99,8% доступности.

    Для уменьшенной версии Cloud Connector поля N + 1 поддерживают 50 \* N одновременных вызовов с 99,8% доступности.

  - **N + 2 ящиков.** Для более крупной версии Cloud Connector поля N + 2 поддерживают 500 \* N одновременных вызовов с 99,9% доступности.

    Для уменьшенной версии Cloud Connector поля N + 2 поддерживают 50 \* N одновременных вызовов с 99,9% доступности.

![Два облачных соединителя в пределах одного сайта PSTN](../../media/fc0dc47f-5595-42cb-9432-9c8ff3e134e9.png)

### <a name="multiple-pstn-sites-with-one-or-more-cloud-connectors-per-site"></a>Несколько сайтов PSTN с одним или несколькими облачными соединителями на сайт

Вы также можете использовать несколько сайтов PSTN с одним или несколькими выпусками Cloud Connector на каждом сайте. Если сайт PSTN достигает предельного числа одновременных звонков, вы можете добавить еще один сайт PSTN для обработки нагрузки.

Несколько сайтов PSTN также позволяют предоставлять подключения к шлюзам, ближайшим к вашим пользователям. Например, предположим, что у вас есть шлюзы PSTN в Сиэтле и Амстердам. Вы можете развернуть два сайта PSTN — один в Сиэтле, один в Амстердам, и назначить пользователям использование сайта PSTN, который является ближайшим к ним. ## <a name="requirements-for-deployment"></a>Требования к развертыванию
<a name="BKMK_Requirements"> </a>

Прежде чем развертывать Cloud Connector Edition, убедитесь, что у вас есть следующие условия для вашей среды:

- **Для ведущего компьютера —** Виртуальные машины Cloud Connector необходимо развертывать на выделенном оборудовании под управлением Windows Server 2012 R2 Datacenter Edition (английский) с включенной ролью Hyper-V.

    В версии 2,0 и более поздней версии сетевой карты главного компьютера, привязанной к коммутатору в сети Skype для бизнеса, должен быть IP-адрес, настроенный в той же подсети, что и корпоративные корпоративные компьютеры с Cloud Connector.

- Для версии 2,1 и более поздней версии ведущее устройство должно иметь .NET Framework 4.6.1 или более поздней версии.

- **Для виртуальных машин —** Изображение на языке Windows Server 2012 R2 ISO (ISO). ISO-файл будет преобразован в виртуальные диски VHD для виртуальных машин, на которых будет выполняться Skype для бизнеса Cloud Connector Edition.

- Необходимое оборудование для поддержки установки 4 виртуальных машин для каждого выпуска Cloud Connector Edition в развертывании. Рекомендуется использовать следующие конфигурации:

  - 64-разрядный двухъядерный процессор, шесть ядер (12 реальных ядер), 2,50 ГГц (ГГц) или выше

  - 64 ГБ ОЗУ ECC (ГБ)

  - 4 600 ГБ (или более) 10000 об/мин 128M кэш SAS 6Gbps диски, настроенные в конфигурации RAID 5

  - Три сетевых адаптера с высокой пропускной способностью 1 Гбит/с

- Если вы решили развернуть уменьшенную версию Cloud Connector Edition, поддерживающую до 50 одновременных вызовов, вам потребуется следующее оборудование:

  - Intel i7 4790 Quad Core с графической платой Intel 4600 (не требуется высокая подсистема)

  - 32 ГБ DDR3 – 1600 без ECC

  - 2:1 ТБАЙТ/с SATA III (6 Гбит/с) в RAID 0

  - 2:1 Гбит/с, Ethernet (RJ45)

В противном случае подключение управления завершится с ошибками и не позволит развернуть и автоматически восстановить Cloud Connector. Ниже приведен пример команды настройки WinHTTP: netsh WinHTTP Set proxy "10.10.10.175:8080" обход-List = " \* . local; 1. \* ; 172,20. \* ; 192.168.218. \* ' \<local\> ".

### <a name="ports-and-protocols"></a>Порты и протоколы
<a name="BKMB_Ports"> </a>

При определении диапазонов портов мультимедиа учитывайте следующее:

- Клиенты всегда используют диапазон портов 50000 — 50019 для трафика мультимедиа — этот диапазон предопределен в Skype для бизнеса Online и не может быть изменен.

- По умолчанию компонент-посредник будет использовать диапазон портов 49 152 — 57 500 для трафика мультимедиа. Однако подключение устанавливается через внутренний брандмауэр, и в целях безопасности вы можете ограничить этот диапазон портов в топологии. Для каждого вызова вам потребуется до 4 портов. Если необходимо ограничить количество портов между компонентом-посредником и шлюзом PSTN, необходимо также настроить соответствующий диапазон портов на шлюзе.

- Необходимо развернуть Cloud Connector в сети периметра. Это означает, что у вас будет два брандмауэра:

  - Первый брандмауэр является внешним между Интернетом и сетью периметра.

  - Второй брандмауэр является внутренним между сетью периметра и внутренней сетью.

    Клиенты могут находиться в Интернете или во внутренней сети:

  - Клиенты в Интернете будут подключаться к сети PSTN через внешний брандмауэр через пограничный компонент.

  - Клиенты во внутренней сети будут подключаться через внутренний брандмауэр к компоненту-посреднику в сети периметра, который будет соединять трафик с шлюзом SBC или PSTN.

### <a name="dns-name-resolution-for-the-edge-component"></a>Разрешение DNS-имен для пограничного компонента
<a name="BKMB_Ports"> </a>

Пограничный компонент должен разрешить внешние имена служб Microsoft 365 или Office 365, а также внутренние имена других компонентов Cloud Connector.

Каждый пограничный компонент является многосетевым компьютером с внешним и внутренним интерфейсом. Cloud Connector развертывает DNS-серверы в компоненте контроллера домена в сети периметра. Пограничный сервер можно указать на DNS-сервере в пределах периметра для всех разрешений имен, но необходимо включить DNS-сервер Cloud Connector для разрешения внешних имен, задав зону DNS, содержащую одну или несколько записей DNS A для внешних запросов, которые ссылаются на поиск имен на другие общедоступные DNS-серверы.

Если в ini-файле задано полное доменное имя для шлюзов из того же пространства домена, что и ваш домен SIP, в DNS-сервере в пределах периметра будет создана удостоверяющая зона для этого домена SIP. Если пограничный сервер указывает на этот DNS-сервер для разрешения имен, пограничный сервер не будет разрешать _sipfederationtls.\<yourdomain\> Запись DNS, которая является обязательной для направления звонка. В этом случае корпорация Майкрософт рекомендует указать DNS-сервер на пограничном внешнем интерфейсе для разрешения поиска имен в Интернете, а каждый пограничный компонент должен использовать файл узла для сопоставления имен компонентов Cloud Connector с IP-адресами.

> [!NOTE]
> В целях безопасности не рекомендуется направлять DNS-сервер Cloud Connector на внутренние серверы в рабочем домене для разрешения имен.

### <a name="certificate-requirements"></a>Требования к сертификатам
<a name="BKMK_Certs"> </a>

Для каждого пограничного компонента требуется сертификат из общедоступного центра сертификации. Для копирования между пограничными компонентами сертификаты должны иметь экспортируемый закрытый ключ. Чтобы удовлетворить требования к сертификатам, необходимо выбрать один из следующих вариантов и указать имя субъекта (SN) и альтернативное имя субъекта (SAN) для сертификата.

 **Если у вас есть один домен SIP:**

- **Вариант 1.** Имя субъекта должно содержать имя пула, назначенное компонентам пограничного сервера. Обратите внимание, что имя субъекта не может быть sip.sipdomain.com, так как это имя зарезервировано для компонента Skype для бизнеса в Интернете. Сеть хранения данных должна содержать sip.sipdomain.com и имя пограничного пула доступа:

  ```console
  SN = accessedgepoolnameforsite1.sipdomain.com, SAN = sip.sipdomain.com,
  acessedgepoolnameforsite1.sipdomain.com
  ```

- **Вариант 2.** Если вы хотите использовать один сертификат с подстановочными знаками на всех развертываемых серверах пограничных пулов, вы можете использовать подстановочную запись в формате \* sipdomain.com вместо имени пограничного пула в сертификате. Имя субъекта может быть именем пограничного пула доступа для всех развернутых пограничных пулов:

  ```console
  SN = accessedgepoolnameforsite1.sipdomain.com, SAN = sip.sipdomain.com, SAN = *.sipdomain.com
  ```

> [!NOTE]
> Не следует создавать внешнюю запись DNS для SIP. \<sipdomain\> . com, так как это имя относится к развертыванию Microsoft 365 или Office 365.

> [!NOTE]
> Если вы хотите использовать один сертификат для всех пулов пограничных серверов, развернутых в Организации, и не может использовать сертификат с подстановочными знаками, как определено в варианте 2, необходимо включить полное доменное имя для всех развернутых пограничных пулов в имени сети хранения данных в сертификате.

 **Если у вас несколько доменов SIP:**

Вам потребуется добавить sip.sipdomain.com для каждого домена SIP и имя пулов пограничных серверов доступа на домен (это может быть один физический пул, но с разными именами). Ниже приведен пример записей SN и SAN в сценарии с несколькими SIP-доменами:

- **Вариант 1.** Имя субъекта должно содержать имя пула, назначенное для пограничных компонентов. Обратите внимание, что имя субъекта не может быть sip.sipdomain.com, так как это имя зарезервировано для компонента Skype для бизнеса в Интернете. Сеть хранения данных должна содержать sip.sipdomain.com и имя пограничного пула доступа:

  ```console

  ```

- <strong>Вариант 2.</strong> Если вы хотите использовать один сертификат с подстановочными знаками на всех развертываемых серверах пограничных пулов, вы можете использовать подстановочную запись в формате \* sipdomain.com вместо имени пограничного пула в сертификате. Имя субъекта может быть именем пограничного пула доступа для всех развернутых пограничных пулов:

  ```console

  SAN = *.sipdomain1.com
  ```

## <a name="high-availability-considerations"></a>Рекомендации по обеспечению высокого уровня доступности
<a name="BKMK_HA"> </a>

При развертывании Cloud Connector Edition для обеспечения высокой доступности необходимо развернуть по крайней мере два устройства, которые выполняют роль резервного копирования. Каждое устройство состоит из четырех компонентов: пограничный, пограничный, центрального хранилища управления (CMS) и контроллера домена.

В общем случае, если один компонент устройства отключается, Cloud Connector Edition может продолжать обработку вызовов, но необходимо учитывать следующее:

- **Рекомендации по компонентам контроллеров-посредников, CMS и контроллеров домена**

    Предположим, что сервер CMS или компонент контроллера домена в одном устройстве отключен. Устройство по-прежнему может обрабатывать входящие и исходящие вызовы, но если перезапустить компонент-посредник, если контроллер домена или компонент CMS недостижимы, то это не поможет. То же самое относится к перезапуску компонента CMS при отключенном контроллере домена.

    **Рекомендация:** Прежде чем перезапускать компоненты, проверьте доступность других компонентов в устройстве.

- **Рекомендации по пограничным компонентам**

    Если пограничный компонент на одном устройстве в недоступном, поведение для входящих и исходящих вызовов будет отличаться следующим образом:

  - **Исходящий звонок**— звонок от пользователя в Интернете в сеть PSTN.

    Механизм распространения вызовов в облаке определит, что один пограничный компонент не работает, и направит все вызовы на другое устройство, чтобы исходящий вызов был успешным.

  - **Входящий звонок**— звонок от сети PSTN пользователю, который находится либо в локальной сети, либо в Интернете.

     Если пограничный компонент устройства, получившего вызов, не работает, входящие вызовы этого устройства не будут успешными, так как компонент-посредник не может перенаправить вызов на пограничный компонент другого устройства.

    **Рекомендация:** Система мониторинга установлена на месте. Определив неисправность пограничного компонента, закройте все компоненты устройства, где пограничный компонент недоступен.

## <a name="cloud-connector-media-flow"></a>Поток мультимедиа для Cloud Connector
<a name="BKMK_MediaFlow"> </a>

На следующих схемах приводится последовательность исходящего и входящего вызовов с помощью Cloud Connector Edition. Эта информация полезна для понимания того, как устанавливается подключение.

На первой схеме внутренний пользователь помещает исходящий звонок следующим образом:

1. Дениса — пользователь, размещенный в Интернете, но теперь во внутренней сети, вызывает внешнего пользователя PSTN.

2. Трафик SIP перенаправляется в Skype для бизнеса Online.

3. В Skype для бизнеса Online выполняется обратный просмотр номера. Неудачный Просмотр обратного номера из-за того, что этот номер не принадлежит ни одному из пользователей в Организации Skype для бизнеса.

4. Вызов перенаправляется на пограничный компонент (поток SIP и поток мультимедиа сначала через интерактивный пограничный сервер; Мультимедиа перейдет к компоненту-посреднику через внутренний брандмауэр).

5. Если маршрут существует, пограничный компонент передает трафик компоненту-посреднику в сети периметра.

6. Компонент-Посредник отправляет трафик шлюзу PSTN.

![Исходящий поток мультимедиа для Cloud Connector](../../media/c495a2bb-305c-46ef-b16d-b8f9f2b937a8.png)

На следующей схеме внутренний пользователь получает входящий звонок, как показано ниже.

1. Шлюз PSTN получает вызов для пользователя Дениса, размещенный в Интернете, но теперь находится во внутренней сети.

2. Трафик SIP направляется компоненту-посреднику.

3. Компонент-Посредник отправляет трафик SIP на пограничный компонент, а затем отправляет в Skype для бизнеса Online.

4. В Skype для бизнеса Online выполняется обратный поиск номера и обнаруживается, что это пользователь Дениса.

5. Сигнализация SIP передается всем точкам присутствия Дениса.

6. Трафик мультимедиа будет устанавливаться между шлюзом и компонентом-посредником, а также между компонентом-посредником и конечной точкой.

## <a name="monitoring-and-troubleshooting"></a>Мониторинг и устранение неполадок
<a name="BKMK_Monitor"> </a>

Механизм мониторинга и устранения неполадок устанавливается автоматически вместе с каждым устройством Cloud Connector. Механизм обнаруживает следующие события:

- Одна или несколько виртуальных машин устройства Cloud Connector не подключены к внутреннему виртуальному коммутатору или виртуальному коммутатору в Интернете.

- Одна или несколько виртуальных машин устройства Cloud Connector находятся в состоянии "сохранено" или "остановлен".

- Незапущенные службы.

  Если обнаружено одно из следующих событий, все устройство Cloud Connector очищается и помечается как отключенное, чтобы предотвратить попытки установить вызовы в неисправное устройство. В дальнейшем функции автоматического восстановления Cloud Connector будут восстанавливать службы и помечать устройство как подключенное к сети. Если по какой либо причине не удается выполнить автоматическое восстановление, обратитесь к разделу [Устранение неполадок развертывания Cloud Connector](troubleshoot-your-cloud-connector-deployment.md).

  - На виртуальной машине центрального хранилища управления:

     - Агент главной репликации Skype для бизнеса

     - Агент репликации реплики Skype для бизнеса

  - На виртуальной машине сервера-посредника:

     - Агент репликации реплики Skype для бизнеса

     - Устранение неполадок Skype для бизнеса Server

  - На виртуальной машине пограничного сервера

     - Агент репликации реплики Skype для бизнеса

     -  Пограничный сервер доступа Skype для бизнеса Server

     - Skype для бизнеса Server Audio/видео Edge

     - Проверка подлинности с помощью аудио-и видеоданных Skype для бизнеса Server

     - Пограничный сервер веб-конференций Skype для бизнеса Server

- Правило входящего трафика брандмауэра Windows для "CS RTCSRV" на пограничном сервере, "CS РТКМЕДСРВ" на сервере-посреднике отключено.

Cloud Connector 2,1 и более поздних версий поддерживает мониторинг Cloud Connector с помощью Operations Management Suite (OMS). Дополнительные сведения см в статье [Monitor Cloud Connector с помощью Operations Management Suite (OMS)](monitor-cloud-connector-using-operations-management-suite-oms.md)
