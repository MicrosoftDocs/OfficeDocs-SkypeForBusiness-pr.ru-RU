# <a name="plan-for-skype-for-business-cloud-connector-edition"></a>Планирование Skype для бизнеса Cloud Connector Edition

> [!Important]
> Выпуск Cloud Connector Edition завершится 31 июля 2021 г. вместе со Skype для бизнеса Online. После обновления вашей организации до Teams узнайте, как подключить свою локальной телефонной сети к Teams с помощью [прямой маршрутизации.](https://docs.microsoft.com/MicrosoftTeams/direct-routing-landing-page)

Find information on Skype for Business Cloud Connector Edition, a set of packaged Virtual Machines (VMs) that implement on-premises PSTN connectivity with Phone System (Cloud PBX).

Cloud Connector Edition может быть правильным решением для вашей организации, если у вас еще нет существующего развертывания Lync Server или Skype для бизнеса Server. Если вы все еще изучаете, какое решение телефонной системы правильно для вашего бизнеса, см. решения [Майкрософт для телефонии.](https://docs.microsoft.com/SkypeForBusiness/hybrid/msft-telephony-solutions)

В этом документе описываются требования Cloud Connector Edition и поддерживаемые topologies, а также помогают планировать развертывание Cloud Connector Edition. Обязательно ознакомьтесь с этим разделом перед настройкой среды Cloud Connector. When you are ready to deploy and configure Cloud Connector Edition, see [Configure and manage Skype for Business Cloud Connector Edition.](configure-skype-for-business-cloud-connector-edition.md)

Теперь доступен выпуск Cloud Connector Edition 2.1. Если вы еще не обновили версию 2.1, см. обновление до новой версии [Cloud Connector.](upgrade-to-a-new-version-of-cloud-connector.md) Файл установки можно найти по этой [https://aka.ms/CloudConnectorInstaller](https://aka.ms/CloudConnectorInstaller) папке.

> [!NOTE]
> Корпорация Майкрософт поддерживает предыдущую версию Cloud Connector Edition в течение 60 дней после выпуска новой версии. Корпорация Майкрософт будет поддерживать версию 2.0.1 в течение 60 дней после выпуска 2.1, чтобы у вас было время на обновление. Все версии, предшествующие версии 2.0.1, больше не поддерживаются.

Cloud Connector Edition — это гибридное предложение, состоящее из набора упакованных виртуальных машин (ВМ), реализующих локальное подключение к STN с телефонной системой. Развертывая минимальную топологию Skype для бизнеса Server в виртуализированной среде, пользователи в вашей организации, которые размещены в облаке, могут получать службы УАЦ из облака Майкрософт, но подключение по STN обеспечивается через существующую локализованную инфраструктуру голосовой связи.

![Схема топологии, на которой показан шлюз облачной УАКС, соединяющий облачную УАП с локальной развертыванием Skype для бизнеса.](../../media/bd898e69-6458-4276-aebe-1854f28ed6fa.png)

Так как Cloud Connector позволяет интегрировать службы телефонной системы с существующей телефонной средой, например УАПС, аналоговыми устройствами и центрами обработки вызовов, можно реализовать поэтапную миграцию из существующего телефонного решения в телефонную систему.

Например, предположим, что ваша компания имеет сложный центр обработки вызовов с определенными функциональными возможностями, которые не предоставляет телефонная система. Вы можете оставить пользователям центра обработки вызовов существующее решение, но переместить других пользователей в телефонную систему.

Cloud Connector обеспечит маршрутику между пользователями, которые были в локальной и сетевой сети, и вы можете использовать собственного поставщика STN с телефонной системой.

При планировании развертывания Cloud Connector Edition учитывайте следующее:

- Чтобы использовать Cloud Connector для использования облачных решений голосовой связи, необходимо зарегистрироваться в организации Microsoft 365 или Office 365, которая включает телефонную систему. Если у вас еще нет организации Microsoft 365 или Office 365, вы можете узнать, как зарегистрироваться здесь: [Microsoft 365 для бизнеса.](https://products.office.com/business/office) Обратите внимание, что вам потребуется зарегистрироваться в плане, включаемом Skype для бизнеса Online.

- Для регистрации устройств Cloud Connector в службе Skype для бизнеса Online и запуска различных cmdlets Для Cloud Connector 2.0 и более поздних требуется выделенная учетная запись Microsoft 365 или Office 365 с правами администратора клиента Skype для бизнеса. Для версий Cloud Connector, предшествующих версии 2.0, требуется выделенная учетная запись Microsoft 365 или Office 365 с правами глобального администратора клиента.

- Для Cloud Connector не требуется полное локальное развертывание Skype для бизнеса Server.

    В настоящее время Cloud Connector не может сосуществовать с локальной серверами Lync или Skype для бизнеса. Если вы хотите перенести существующих пользователей Lync или Skype для бизнеса в Microsoft 365 и продолжать предоставлять пользователям поддержку локальной телефонии, рассмотрите возможность телефонной системы с локальной связью с помощью существующего развертывания Skype для бизнеса Server. For more information, see [Plan your Phone System (Cloud PBX) solution](plan-your-phone-system-cloud-pbx-solution.md) and Plan Phone System with [on-premises PSTN connectivity in Skype for Business Server.](plan-phone-system-with-on-premises-pstn-connectivity.md)

- Если вы ранее развертывали Skype для бизнеса или Lync Server и расширили схему, вам не нужно очищать схему для развертывания Cloud Connector, если вы удалили из среды все компоненты Skype для бизнеса или Lync Server.

- Ваши пользователи находятся в Интернете.

- Если в организации настроена синхронизация службы каталогов (DirSync), сначала необходимо создать в локальном развертывании все учетные записи пользователей, планируемых для гибридной голосовой почты, а затем синхронизировать их с облаком.

- При необходимости вы можете сохранить текущего оператора STN.

- Если вы хотите предоставить доступ к аудиоконференциям для пользователей, которые имеются в Cloud Connector, вы можете приобрести лицензию на аудиоконференцию STN или оплатить ее по мере получения предложения аудиоконференции от корпорации Майкрософт.

- Лицензия на аудиоконференцию (или платная по мере предложения) также необходима для эскалации вызовов. Если пользователь Skype для бизнеса получает звонок от внешнего пользователя STN и хочет добавить к этому вызову еще одного участника (перенаправить вызов на конференцию), эскалация будет выполняться через службу аудиоконференции Майкрософт.

- Cloud Connector 2.0 и более поздние службы теперь поддерживают обход мультимедиа. Обход мультимедиа позволяет клиенту отправлять мультимедиа напрямую на следующий прыжок PSTN (шлюз или пограничный контроллер сеансов) и исключить компонент Cloud Connector Edition из пути мультимедиа. Дополнительные сведения см. в [сведениях о планировании обхода мультимедиа в Cloud Connector Edition.](plan-for-media-bypass-in-cloud-connector-edition.md)

- Cloud Connector 2.1 и более поздних поддерживает мониторинг Cloud Connector с помощью Operations Management Suite (OMS). Дополнительные сведения см. в руководстве [Monitor Cloud Connector с помощью Operations Management Suite (OMS)](monitor-cloud-connector-using-operations-management-suite-oms.md)

- Cloud Connector доступен во всех странах, где доступен Office 365 корпоративный E5.

## <a name="cloud-connector-edition-components"></a>Компоненты Cloud Connector Edition
<a name="BKMK_Components"> </a>

В Cloud Connector Edition развертывается набор упакованных ВМ, содержащих минимальную топологию Skype для бизнеса Server, состоящую из компонента edge, компонента-посредника и роли центрального банка управления (CMS). Вы также установите контроллер домена, необходимый для внутреннего функционирования Cloud Connector. Эти службы настроены для гибридной работы с вашей организацией Microsoft 365 или Office 365, которая включает службы Skype для бизнеса Online.

![Компоненты Cloud Connector Edition](../../media/f2d4b8a7-c2f4-4cfc-8137-f187399c1298.png)

Компоненты Cloud Connector предоставляют следующие функции:

- **Edge component** - Communication between the on-premises topology and the online services goes through the Edge component, which includes the following components:

  - **Access Edge** — обеспечивает маршрутику SIP между локальной развертыванием и Skype для бизнеса Online.

  - **Ретрансляция** мультимедиа — обеспечивает маршрутику мультимедиа между компонентом-посредником и другими конечными точками мультимедиа.

  - **Проверка подлинности при ретрансляции мультимедиа / MRAS** — создает маркеры для доступа к ретрансляции мультимедиа.

- **Исходящие маршруты** — обеспечивает балансировку нагрузки голосового трафика между шлюзами или SBC, подключенными к устройству Cloud Connector. Вызовы будут разделены между всеми шлюзами или SBC, подключенными к устройству Cloud Connector.

    Обеспечивает маршрутику на шлюзы на основе политик. Поддерживаются только глобальные политики, основанные на номерах PSTN назначения (исходящие).

- **Роль центрального банка управления (CMS)** — включает хранилище конфигурации для компонентов топологии, включая передачу файлов CMS.

- Реплика центрального банка управления **(CMS)** синхронизирует сведения о конфигурации из глобальной DB CMS на сервере роли CMS.

- **Контроллер домена** — доменные службы Active Directory Cloud Connector для хранения всех глобальных параметров и групп, необходимых для развертывания компонентов Cloud Connector. Для каждого устройства Cloud Connector будет создан один лес. Контроллер домена не должен иметь подключений к производственной службе Active Directory. Службы Active Directory включают:

  - Доменные службы Active Directory

  - Службы сертификатов Active Directory для выдачи внутренних сертификатов

- **Компонент-посредник** — реализует протокол сопоставления SIP и медиа шлюзов между Skype для бизнеса и шлюзами STN. Включает реплику CMS, которая синхронизирует конфигурацию из глобальной базы данных CMS.

## <a name="cloud-connector-edition-topologies"></a>Topologies Cloud Connector Edition
<a name="BKMK_Topologies"> </a>

В рамках этого обсуждения мы будем ссылаться на сайты STN. Сайт STN — это сочетание устройств Cloud Connector, развернутых в одном расположении и с общими шлюзами STN, подключенными к ним. Сайты STN позволяют:

- Обеспечение подключения к шлюзам, которые ближе всего к вашим пользователям.

- Чтобы обеспечить масштабируемость, развернем несколько устройств Cloud Connector на одном или нескольких сайтах STN.

- Для высокой доступности можно развернуть несколько устройств Cloud Connector на одном сайте STN.

В этом разделе данная тема представляет сайты PSTN. Дополнительные сведения о планировании сайтов STN см. в сведениях [о планировании сайтов STN Cloud Connector Edition.](plan-for-cloud-connector-edition-pstn-sites.md)

Вы можете развернуть следующие облачные соединители:

- Одно устройство Cloud Connector Edition для каждого сайта STN. Эту топологию рекомендуется использовать только в целях оценки, поскольку она не обеспечивает высокую доступность.

- Несколько устройств Cloud Connector Edition на сайт STN для обеспечения высокой доступности.

- Несколько сайтов STN с несколькими устройствами Cloud Connector Edition для обеспечения масштабируемости и высокой доступности. Можно развернуть до 200 сайтов.

При планировании топологии учитывайте следующее:

- В Cloud Connector 2.0 и более поздних, один сайт STN может иметь до 16 устройств Cloud Connector. Предыдущие версии поддерживают до 4 устройств на сайт.

- Существует два типа конфигураций оборудования, тестовых с помощью Cloud Connector:

  - Более крупная версия может обрабатывать большие объемы одновременных вызовов и поддерживается во всех типах производственных сред.

  - Более небольшая версия предназначена для запуска на оборудовании нижнего уровня и может использоваться для оценки или для сайтов с небольшим объемом вызовов. При развертывании более маленькой версии Cloud Connector необходимо помнить о требованиях к оборудованию производственного класса (например, к двойному источнику питания).

- Если у вас есть Cloud Connector версии 2.0 или более поздней версии и развернута максимальная конфигурация из 16 устройств (с большим оборудованием), сайт STN может обрабатывать до 8000 одновременных вызовов. При развертывании меньшей версии поддерживаемый предел составляет 800.

    Вам также необходимо выделенные некоторые устройства для высокой доступности. Минимальной рекомендацией является то, что одно устройство должно быть зарезервировано для высокой доступности.

  - В версии 2 при развертывании конфигурации 15+1 сайт STN может обрабатывать до 7500 одновременных вызовов.

  - Если у вас более ранная версия и развернута максимальная конфигурация 3 + 1 (с большим оборудованием), сайт STN может обрабатывать до 1500 одновременных вызовов. При развертывании более маленькой версии поддерживаемый предел составляет 150.

-  Если требуется больше вызовов на сайт STN, можно развернуть дополнительные сайты STN в том же расположении.

> [!NOTE]
> Если не отмечено, на схемах и примерах ниже предполагается использование более крупной версии Cloud Connector.

### <a name="single-cloud-connector-appliance-within-a-single-pstn-site"></a>Одно устройство Cloud Connector на одном сайте STN

На следующей схеме показано одно устройство Cloud Connector Edition на одном сайте STN. Обратите внимание, что Cloud Connector состоит из четырех ВМ, установленных на одном физическом хост-компьютере, который находится в сети периметра в целях безопасности.

![Один cloud Connector с одним сайтом STN](../../media/7ffe6953-8c66-4323-940e-cd2e6c3c2a66.png)

### <a name="multiple-cloud-connector-appliances-within-a-single-pstn-site"></a>Несколько устройств Cloud Connector на одном сайте STN

 В целях масштабируемости и высокой доступности можно выбрать наличие нескольких выпусков Cloud Connector Edition на одном сайте STN, как показано на следующей схеме. Примите во внимание следующее:

- Вызовы распределяются в случайном порядке между cloud Connectors в одном пуле.

- В целях планирования емкости необходимо рассмотреть возможность обработки нагрузки, если один или несколько cloud Connectors от сети отключены на основе следующих вычислений:

  - **N+1 полей.** Для более крупной версии Cloud Connector N+1 поля поддерживают 500 N одновременно звонков с \* доступностью 99,8 %.

    Для небольших версий Cloud Connector N+1 поля поддерживают 50 N одновременно звонков с \* доступностью 99,8 %.

  - **N+2 поля.** Для более крупной версии Cloud Connector N+2 поля поддерживают 500 N одновременно звонков с \* доступностью 99,9 %.

    Для небольших версий Cloud Connector N+2 поля поддерживают 50 N одновременно звонков с \* доступностью 99,9 %.

![Два облачных соединитела на 1 сайте STN](../../media/fc0dc47f-5595-42cb-9432-9c8ff3e134e9.png)

### <a name="multiple-pstn-sites-with-one-or-more-cloud-connectors-per-site"></a>Несколько сайтов STN с одним или несколькими облачными соединительми на сайт

Вы также можете выбрать несколько сайтов STN с одним или несколькими выпусками Cloud Connector на каждом сайте. Если на сайте STN достигается ограничение одновременных вызовов, можно добавить другой сайт STN для обработки нагрузки.

Несколько сайтов STN также позволяют обеспечить подключение к шлюзам, которые ближе всего к вашим пользователям. Например, предположим, что у вас есть шлюзы STN в Сиэтле и Амстердаме. Вы можете развернуть два узла STN — один в Сиэтле, один в Амстердаме — и назначить пользователям использовать ближайший к ним сайт PSTN. Пользователи из Сиэтла будут перенаться на сайт и шлюзы STN в Сиэтле, а пользователи в Амстердаме — на сайт и шлюзы STN в Амстердаме:

![Cloud Connector Edition в пределах 2 сайтов STN](../../media/16ead6d3-67da-4e71-b4d5-d895b4c9384e.png)

## <a name="requirements-for-deployment"></a>Требования к развертыванию
<a name="BKMK_Requirements"> </a>

Перед развертыванием Cloud Connector Edition убедитесь, что в вашей среде есть следующее:

- **Для хост-компьютера —** VMs Cloud Connector must be deployed on dedicated hardware running Windows Server 2012 R2 Datacenter edition (English) with the Hyper-V role enabled.

    Для версии 2.0 и более поздних версий для сетевой карты хост-компьютера, привязанной к коммутатору Skype для бизнеса Corpnet, должен быть настроен IP-адрес в той же подсети, что и компьютеры корпоративной сети Cloud Connector.

- Для версий 2.1 и более поздних версий на хост-устройстве должна быть установлена .NET Framework 4.6.1 или более поздней версии.

- **Для виртуальных машин —** Iso-образ ISO -образа Windows Server 2012 R2 (английский). IsO будет преобразован в виртуальныеHD для виртуальных машин, на которые будет работать Skype для бизнеса Cloud Connector Edition.

### <a name="ports-and-protocols"></a>Порты и протоколы
<a name="BKMB_Ports"> </a>

При определении диапазонов портов мультимедиа необходимо помнить о следующих действиях:

- Клиенты всегда используют диапазон портов от 50000 до 50019 для трафика мультимедиа — этот диапазон предварительно заранее заданной в Skype для бизнеса Online и не может быть изменен.

- Компонент-посредник по умолчанию использует диапазон портов от 49 152 до 57 500 для трафика мультимедиа. Однако подключение устанавливается через внутренний брандмауэр, и в целях безопасности этот диапазон портов можно ограничить в топологии. Для каждого вызова потребуется до 4 портов. Если необходимо ограничить число портов между компонентом-посредником и шлюзом ТСТС, необходимо также настроить соответствующий диапазон портов на шлюзе.

- Необходимо развернуть Cloud Connector в сети периметра. Это означает, что у вас будет два брандмауэра:

  - Первый брандмауэр является внешним между Интернетом и сетью периметра.

  - Второй брандмауэр является внутренним между сетью периметра и внутренней сетью.

    Ваши клиенты могут быть в Интернете или во внутренней сети:

  - Клиенты в Интернете будут подключаться к PSTN через внешний брандмауэр через компонент Edge.

  - Клиенты во внутренней сети будут подключаться через внутренний брандмауэр к компоненту-посреднику в сети периметра, который будет подключать трафик к SBC или шлюзу STN.

### <a name="dns-name-resolution-for-the-edge-component"></a>Разрешение имен DNS для компонента Edge
<a name="BKMB_Ports"> </a>

Компонент Edge должен разрешать внешние имена служб Microsoft 365 или Office 365 и внутренние имена других компонентов Cloud Connector.

Каждый компонент Edge — это компьютер с несколькими домами с внешними и внутренними интерфейсами. Cloud Connector развертывает DNS-серверы в компоненте контроллера домена в сети периметра. Можно указать на DNS-сервер в периметре для всех разрешений имен, но необходимо включить DNS-сервер Cloud Connector для разрешения внешних имен, задав зону DNS, содержащую одну или несколько записей A DNS для внешних запросов, которые ссылаются на запросы на другие общедоступные DNS-серверы.

Если в INI-файле задайте имя FQDN для шлюзов из того же доменного пространства, что и ваш домен SIP, в DNS-сервере в демилитарии будет создана дополная зона для этого домена SIP. Если для разрешения имен на этот DNS-сервер указывает edge Server, он никогда не разрешит _sipfederationtls.\<yourdomain\> DNS-запись, необходимая для потока вызовов. В этом случае корпорация Майкрософт рекомендует предоставить DNS-сервер на внешнем интерфейсе Edge для разрешения поисков по именам в Интернете, а каждый из этих компонентов должен использовать файл HOST для разрешения имен других компонентов Cloud Connector в IP-адреса.

> [!NOTE]
> Из соображений безопасности не рекомендуется указать DNS-сервер Cloud Connector внутренним серверам в производственном домене для разрешения имен.

### <a name="certificate-requirements"></a>Требования к сертификатам
<a name="BKMK_Certs"> </a>

Каждому компоненту edge требуется сертификат от публичного органа сертификации. Сертификаты должны иметь экспортируемый закрытый ключ для копирования между компонентами edge. Для удовлетворения требований к сертификату необходимо выбрать один из следующих вариантов и предоставить для сертификата имя субъекта (SN) и альтернативное имя субъекта (SAN).

 **Если у вас есть один домен SIP:**

- **Вариант 1.** Имя субъекта должно содержать имя пула, назначенное компонентам edge. Обратите внимание, что имя субъекта не может быть sip.sipdomain.com поскольку это имя зарезервировано для веб-компонента Skype для бизнеса Edge. Имя SAN должно содержать sip.sipdomain.com имя пула доступа:

  ```console
  SN = accessedgepoolnameforsite1.sipdomain.com, SAN = sip.sipdomain.com,
  acessedgepoolnameforsite1.sipdomain.com
  ```

- **Вариант 2.** Если вы хотите использовать один сертификат с подданными на всех развернутых серверах пула, можно использовать запись SAN с подданными .sipdomain.com вместо имени пула edge в \* сертификате. Имя субъекта может быть именем пула доступа для любого из развернутых пулов.

  ```console
  SN = accessedgepoolnameforsite1.sipdomain.com, SAN = sip.sipdomain.com, SAN = *.sipdomain.com
  ```

> [!NOTE]
> Не следует создавать внешнюю запись DNS для \<sipdomain\> SIP. com, так как это имя относится к развертыванию Microsoft 365 или Office 365.

> [!NOTE]
> Если вы хотите использовать один сертификат для всех пулов, развернутых в организации, и не можете использовать сертификат с подданными, как определено в варианте 2, необходимо включить в сертификат имя SAN для всех развернутых пулов edge.

 **Если у вас несколько доменов SIP:**

Вам потребуется добавить sip.sipdomain.com для каждого домена SIP и имя пулов доступа для каждого домена (это может быть один физический пул с разными именами). Ниже приведен пример записей SN и SAN в сценарии с несколькими доменами SIP:

- **Вариант 1.** Имя субъекта должно содержать имя пула, назначенное для компонентов Edge. Обратите внимание, что имя субъекта не может быть sip.sipdomain.com поскольку это имя зарезервировано для веб-компонента Skype для бизнеса Edge. Имя SAN должно содержать sip.sipdomain.com имя пула доступа:

  ```console
  SN = accessedgepoolnameforsite1.sipdomain1.com, SAN = sip.sipdomain1.com, sip.sipdomain2.com,
  acessedgepoolnameforsite1.sipdomain1.com
  ```

- <strong>Вариант 2.</strong> Если вы хотите использовать один сертификат с подданными на всех развернутых серверах пула, вы можете использовать запись SAN с подданными .sipdomain.com вместо имени пула в \* сертификате. Имя субъекта может быть именем пула доступа для любого из развернутых пулов.

  ```console
  SN = accessedgepoolnameforsite1.sipdomain.com, SAN = sip.sipdomain1.com, sip.sipdomain2.com,
  SAN = *.sipdomain1.com
  ```

> [!NOTE]
> Не следует создавать внешнюю запись DNS для \<sipdomain\> SIP. com, так как это имя относится к развертыванию Microsoft 365 или Office 365.

В целях развертывания можно использовать следующую таблицу:

|**Вариант**|**Описание**|**Примечания**|
|:-----|:-----|:-----|
|Какой вариант будет применяться для развертывания?  <br/> |Вариант 1 или 2  <br/> ||
|SN  <br/> |Предоставление SN для сертификата  <br/> ||
|SAN  <br/> |Предоставляет san для сертификата  <br/> ||

Если между шлюзом и сервером-посредником используется TLS, необходимо получить корневой сертификат или полную цепочку сертификатов для сертификата, назначенного шлюзу.

## <a name="high-availability-considerations"></a>Вопросы высокой доступности
<a name="BKMK_HA"> </a>

При развертывании Cloud Connector Edition для высокой доступности развертываются по крайней мере два устройства, которые выступают в качестве резервного копирования друг для друга. Каждое устройство состоит из четырех компонентов: Edge, Mediation, Central Management Store (CMS) и контроллера домена.

Как правило, если один компонент устройства выходит из системы, Cloud Connector Edition может продолжать обрабатывать вызовы, но необходимо учитывать следующее:

- **Аспекты компонента посредника, CMS и контроллера домена**

    Предположим, что компонент CMS или контроллера домена на одном устройстве выходит из системы. Устройство по-прежнему может обрабатывать входящие и исходящие вызовы, но если перезапустить компонент-посредник, когда контроллер домена или компонент CMS недоступны, посредник не будет работать. Это же относится и к перезапуску компонента CMS, когда контроллер домена не используется.

    **Рекомендация:** Перед перезапуском компонентов проверьте доступность других компонентов на устройстве.

- **Вопросы, которые следует учитывать для компонента edge**

    Если компонент Edge в одном устройстве не доступен, поведение входящие и исходящие вызовы будет отличаться следующим образом:

  - **Исходящие вызовы**— вызов пользователя в Интернете в сеть STN.

    Механизм распределения вызовов в облаке определяет, что один из компонентов edge не работает, и будет перенапределять все вызовы на другое устройство, чтобы исходящие вызовы были успешными.

  - **Входящий вызов**— звонок из сети STN пользователю, который находится в локальной сети или в Интернете.

     Если пограничный компонент устройства, которое принял вызов, не работает, входящие вызовы на это устройство не будут успешными, так как компонент-посредник не может перенаправить вызов на пограничный компонент на другом устройстве.

    **Рекомендация:** На месте должна быть система мониторинга. После определения неисправности компонента Edge выключим все компоненты на устройстве, где этот компонент не доступен.

## <a name="cloud-connector-media-flow"></a>Поток мультимедиа Cloud Connector
<a name="BKMK_MediaFlow"> </a>

На следующих схемах описывается поток исходящие и входящие вызовы через Cloud Connector Edition. Это полезная информация для понимания того, как устанавливается подключение.

На первой схеме внутренний пользователь помещает исходящие вызовы следующим образом:

1. Пользователь, который находится в сети, но теперь находится во внутренней сети, звонит внешнему пользователю STN.

2. Маршруты трафика SIP в Skype для бизнеса Online.

3. Skype для бизнеса Online выполняет обратный просмотр номера. Не удается найти обратный номер, так как этот номер не принадлежит кому-либо в организации Skype для бизнеса.

4. Вызов перенапрался в компонент Edge (сначала через интернет-поймила SIP и поток мультимедиа; Мультимедиа будут перенаходить к компоненту-посреднику через внутренний брандмауэр.

5. Если маршрут существует, по периметру компонент передает трафик компоненту-посреднику в сети периметра.

6. Компонент-посредник отправляет трафик на шлюз STN.

![Исходящие потоки мультимедиа для Cloud Connector](../../media/c495a2bb-305c-46ef-b16d-b8f9f2b937a8.png)

На следующей схеме внутренний пользователь получает входящий вызов следующим образом:

1. Шлюз STN получает вызов пользователя Юли, который находится в сети, но находится во внутренней сети.

2. Трафик SIP маршрутит на компонент-посредник.

3. Компонент-посредник отправляет трафик SIP в компонент edge, а затем передается в Skype для бизнеса Online.

4. Skype для бизнеса Online выполняет обратный поиск номера и обнаруживает, что это пользователь Ю.

5. Сигнал SIP проходит ко всем точкам присутствия Сергея.

6. Трафик мультимедиа будет установлен между шлюзом и компонентом-посредником, а также между компонентом-посредником и конечным пунктом.

![Входящий поток мультимедиа для Cloud Connector](../../media/ba5da6f6-e357-43c6-9e8f-4bfdde97c176.png)

## <a name="monitoring-and-troubleshooting"></a>Мониторинг и устранение неполадок
<a name="BKMK_Monitor"> </a>

Механизм мониторинга и устранения неполадок устанавливается автоматически с каждым устройством Cloud Connector. Механизм обнаруживает следующие события:

- Одна или несколько виртуальных машин устройства Cloud Connector не подключены к внутреннему или интернет-виртуальному коммутатору.

- Одна или несколько виртуальных машин устройства Cloud Connector находятся в состоянии "сохранено" или "остановлено".

- Не запущенные службы.

  Если обнаружено одно из следующих событий, все устройство Cloud Connector будет разрядилось и помечено как автономное, чтобы предотвратить попытку установить вызовы на неисправное устройство. Функции автоматического восстановления Cloud Connector впоследствии восстановят службы и пометят устройство как подключенное. Если по какой-либо причине автоматическое восстановление не удается, см. ["Устранение неполадок развертывания Cloud Connector".](troubleshoot-your-cloud-connector-deployment.md)

  - На виртуальной машине центрального магазина управления:

     - Агент репликатора хозяина Skype для бизнеса

     - Агент репликатора репликации Skype для бизнеса

  - На виртуальной машине сервера-посредника:

     - Агент репликатора репликации Skype для бизнеса

     - Skype для бизнеса Server Mediation

  - На виртуальной машине сервера

     - Агент репликатора репликации Skype для бизнеса

     -  Skype для бизнеса Server Access Edge

     - Skype для бизнеса Server Audio/Video Edge

     - Проверка подлинности аудио- и видеосвязи Skype для бизнеса Server

     - Skype для бизнеса Server Web Conferencing Edge

- Правило брандмауэра Windows для "CS RTCSRV" в Edge, "CS RTCMEDSRV" на сервере-посреднике отключено.

Cloud Connector 2.1 и более поздних поддерживает мониторинг Cloud Connector с помощью Operations Management Suite (OMS). Дополнительные сведения см. в руководстве [Monitor Cloud Connector с помощью Operations Management Suite (OMS)](monitor-cloud-connector-using-operations-management-suite-oms.md)
