# <a name="plan-for-skype-for-business-cloud-connector-edition"></a>Планирование Skype для бизнеса Cloud Connector Edition

Сведения о Skype для бизнеса Cloud Connector Edition, который представляет собой набор упакованных виртуальных машин, реализующий локальную связь через ТСОП с помощью телефонной системы Office 365 (облачная УАТС).

Если у вас еще нет существующего сервера Lync Server или развертывания Skype для бизнеса Server, в вашей организации может быть правильно установлено приложение Cloud Connector Edition. Если вы еще не изучаете, какая телефонная система в решении Office 365 является наиболее подходящего для вашего бизнеса, ознакомьтесь с разрешениями [телефонной связи Microsoft](https://docs.microsoft.com/en-us/SkypeForBusiness/hybrid/msft-telephony-solutions).

В этом документе описаны требования и поддерживаемые топологии в облаке облачных соединителей, а затем вы можете спланировать развертывание Cloud Connector Edition. Обязательно прочтите этот раздел перед настройкой среды облачного соединителя. Если вы готовы к развертыванию и настройке Cloud Connector Edition, просмотрите раздел [Настройка и управление Skype для бизнеса Cloud Connector Edition](configure-skype-for-business-cloud-connector-edition.md).

Теперь вы можете получить доступ к облачному соединителю выпуска 2,1. Если вы еще не выполнили обновление до версии 2.1, см. статью [Upgrade to a new version of Cloud Connector](upgrade-to-a-new-version-of-cloud-connector.md). Файл установки можно найти по адресу [https://aka.ms/CloudConnectorInstaller](https://aka.ms/CloudConnectorInstaller).

> [!NOTE]
> Корпорация Майкрософт поддерживает предыдущую версию облачного соединителя Edition для 60 дней после выпуска новой версии. Майкрософт будет поддерживать версию 2.0.1 в течение 60 дней с момента выпуска 2.1, чтобы дать вам время на обновление. Поддержка всех версий, предшествующих версии 2.0.1, прекращена.

Cloud Connector Edition — это гибридное предложение, содержащее набор упакованных виртуальных машин (ВМ), которые реализуют локальную сеть PSTN с телефонной системой в Office 365. При развертывании минимальной топологии сервера Skype для бизнеса в виртуализованной среде пользователи в вашей организации, которые находятся в облаке, могут получать службы УАТС из Microsoft Cloud, но подключение по сети PSTN обеспечивается с помощью существующего локального голоса. инфраструктура.

![Топологическая схема шлюза облачной УАТС, соединяющего ее с локальным развертыванием Skype для бизнеса.](../../media/bd898e69-6458-4276-aebe-1854f28ed6fa.png)

Так как Cloud Connector обеспечивает интеграцию служб телефонной системы в Office 365 с существующей средой телефонной связи (например, УАТС, аналоговые устройства или центры обработки вызовов), вы можете реализовать ступенчатый переход от существующего решения телефонной связи к телефонной системе в Office 365.

Например, в вашей компании используется сложный центр обработки вызовов с определенными функциями, которые не предоставляет телефонная система в Office 365. Можно оставить пользователям центра обработки вызовов существующее решение, а остальных пользователей перевести на телефонную систему в Office 365.

Cloud Connector обеспечит маршрутизацию между пользователями, находящимися локально и в сети, и можно будет выбрать использование собственного поставщика ТСОП с телефонной системой в Office 365.

Если вы планируете развертывание облачного соединителя, учитывайте следующее:

- Чтобы использовать облачный соединитель для использования облачных решений для голосовой связи, вам потребуется зарегистрироваться в клиенте Office 365, который включает телефонную систему в Office 365. Если у вас еще нет клиента Office 365, вы можете узнать, как подписаться на этой странице: [Office 365 для бизнеса](https://products.office.com/en-us/business/office). Обратите внимание, что вам потребуется зарегистрировать план, включающий Skype для бизнеса Online.

- Чтобы зарегистрировать устройства облачного соединителя с помощью службы Skype для бизнеса Online, а также для выполнения различных командлетов, в облачном соединителе 2,0 и более поздних версий требуется специальная учетная запись Office 365 с правами администратора клиента Skype для бизнеса. Для версий, предшествующих Cloud Connector 2.0, требуется специальная учетная запись Office 365 с правами глобального администратора.

- Для облачного соединителя не требуется полное развертывание в локальной среде Skype для бизнеса Server.

    В настоящее время облачный соединитель не может находиться на локальных серверах Lync и Skype для бизнеса. Если вы хотите переместить существующих пользователей Lync или Skype для бизнеса в Office 365 и поддерживать локальную телефонную связь с пользователями, рассматривайте телефонную систему в Office 365 с использованием локальных подключений с помощью существующего развертывания Skype для бизнеса Server. Дополнительные сведения можно найти в разделе [планирование телефонной системы в office 365 (облачная УАТС)](plan-your-phone-system-cloud-pbx-solution.md) и [планирование телефонной системы в Office 365 с помощью локальной сети PSTN в Skype для бизнеса Server](plan-phone-system-with-on-premises-pstn-connectivity.md).

- Если вы выполнили предыдущую развертывание Skype для бизнеса или Lync Server и вы расширили схему, вам не нужно очищать схему развертывания облачного соединителя, пока вы удалили все компоненты Skype для бизнеса или Lync Server из вашей среды.

- Ваши пользователи размещены в Интернете.

- Если в вашей организации настроена синхронизация каталогов (DirSync), необходимо сначала создать в локальном развертывании учетные записи всех пользователей, для которых планируется использование гибридной службы голосовой связи, после чего выполнить их синхронизацию с облаком.

- При необходимости можно сохранить текущего оператора ТСОП.

- Если вы хотите предоставить пользователям Конференц-связь с телефонным подключением для пользователей, размещенных в облачном соединителе, вы можете приобрести лицензию на Конференции по сети PSTN или заплатить, как если бы вы ни набрались от Microsoft.

- Лицензия на голосовую конференцию (или плата за то, что вы предлагаете вам) также требуется для эскалации звонков. Если пользователь Skype для бизнеса получает звонок от внешнего пользователя PSTN и хочет добавить еще одного участника на этот звонок (эскалировать Звонок на конференцию), эскалация будет проводиться с помощью службы голосовой связи Microsoft.

- В Cloud Connector 2.0 и более поздних версиях теперь поддерживается обход сервера-посредника. Обход мультимедиа позволяет клиенту отправлять носители прямо в телефонную сеть с открытым коммутируемым подключением (КТСОП) следующего прыжка (PSTN), например, для связи между шлюзом или сеансом (SBC), и удаления компонента Cloud Connector Edition из пути к носителю. Дополнительные сведения можно найти [в разделе Планирование обхода мультимедиа в Cloud Connector Edition](plan-for-media-bypass-in-cloud-connector-edition.md).

- Cloud Connector 2.1 и более поздние версии поддерживают мониторинг Cloud Connector с помощью Operations Management Suite (OMS). Дополнительные сведения: [Monitor Cloud Connector using Operations Management Suite (OMS)](monitor-cloud-connector-using-operations-management-suite-oms.md).

- Облачный соединитель доступен во всех странах, в которых доступна Office 365 Enterprise ".

## <a name="cloud-connector-edition-components"></a>Компоненты Cloud Connector Edition
<a name="BKMK_Components"> </a>

С помощью Cloud Connector Edition вы развертываете набор упакованных виртуальных машин, который содержит минимальную топологию сервера Skype для бизнеса, состоящую из компонента EDGE, компонента для устранения проблем и роли хранилища центрального управления (CMS). Кроме того, вы можете установить контроллер домена, необходимый для внутренней функционирования облачного соединителя. Эти службы настроены для гибридного развертывания с помощью клиента Office 365, включающего службы Skype для бизнеса Online.

![Компоненты Cloud Connector Edition](../../media/f2d4b8a7-c2f4-4cfc-8137-f187399c1298.png)

Компоненты облачного соединителя предоставляют следующие функциональные возможности:

- **Edge-компонент** — связь между локальной топологией и веб-службами проходят через компонент EDGE, который включает следующие компоненты:

  - **Access Edge** — обеспечивает маршрутизацию SIP между локальным развертыванием и Skype для бизнеса Online.

  - **Ретрансляция мультимедиа** — обеспечивает маршрутизацию носителей между компонентом "исправления" и другими конечными точками мультимедиа.

  - **Проверка подлинности Media Relay/мрас** — создает маркеры для доступа к ретрансляции мультимедиа.

- **Маршрутизация исходящих** — обеспечивает балансировку нагрузки для передачи голоса между шлюзами или SBCs, подключенными к устройству облачного соединителя. Звонки будут равномерно распределены между всеми шлюзами или SBCs, подключенными к устройству облачного соединителя.

    Обеспечивается маршрутизация на шлюзы на основе политик. Поддерживаются только глобальные политики, основанные на номерах ТСОП места назначения (внешних).

- **Роль хранилища центрального управления (CMS)** — включает хранилище конфигураций для компонентов топологии, включая передачу CMS-файлов.

- **Реплика центрального хранилища (CMS)** — синхронизирует сведения о конфигурации из глобальной базы данных CMS на сервере роли CMS.

- **Контроллер домена** — доменные службы Active Directory в облаке, чтобы сохранить все глобальные параметры и группы, необходимые для развертывания компонентов облачного соединителя. Для каждого управляющего устройства облачного соединителя будет создан один лес. Контроллер домена не должен иметь подключений к рабочей службе Active Directory. Службы Active Directory включают:

  - Active Directory

  - Службы сертификатов Active Directory для выпуска внутренних сертификатов

- **Компонент "исправления** " — протокол сопоставления SIP и шлюза мультимедиа между шлюзами Skype для бизнеса и PSTN. Включает в себя реплику CMS, которая синхронизирует конфигурацию из глобальной базы данных CMS.

## <a name="cloud-connector-edition-topologies"></a>Топологии Cloud Connector Edition
<a name="BKMK_Topologies"> </a>

В рамках этого обсуждения рассматриваются сайты ТСОП. КОММУТИРУЕМый сайт — это сочетание управляющих устройств облака, развернутых в том же месте и с которыми связаны стандартные шлюзы PSTN. Сайты ТСОП позволяют:

- обеспечить подключение к шлюзам, ближайшим к вашим пользователям;

- Разрешите масштабируемость, разворачивая несколько управляющих устройств с облачным соединителем на одном или нескольких сайтах PSTN.

- Возможность высокой доступности путем развертывания нескольких управляющих устройств с облачным соединителем на одном сайте PSTN.

В этом разделе приводятся общие сведения об узлах ТСОП. Дополнительные сведения о планировании узлов ТСОП см. в разделе [Plan for Cloud Connector Edition PSTN sites](plan-for-cloud-connector-edition-pstn-sites.md).

Вы можете развернуть следующие топологии облачного соединителя:

- Одно устройство Cloud Connector Edition на сайте PSTN. Такую топологию рекомендуется использовать для оценки, так как в ней не реализуется среда высокой доступности.

- Несколько управляющих приложений для облачных соединителей на сайте PSTN для обеспечения высокой доступности.

- Несколько сайтов PSTN с несколькими управляющими устройствами с облачным соединителем для обеспечения масштабируемости с высокой степенью доступности. Можно развернуть до 200 сайтов.

При планировании своей топологии следует учитывать следующее:

- С помощью облака Connector 2,0 и более поздних версий один сайт PSTN может иметь до 16 устройств с облачным соединителем. В предыдущих версиях поддерживалось до 4 устройств на узел.

- С помощью облачного соединителя проверяются два типа аппаратных конфигураций:

  - Наибольшая конфигурация способна обрабатывать большие объемы одновременных вызовов и поддерживается во всех типах рабочих сред.

  - Наименьшая конфигурация предназначена для бюджетного оборудования и может использоваться для оценки или для узлов с небольшим объемом вызовов. При развертывании наименьшей конфигурации Cloud Connector вам по-прежнему следует исходить из требований к оборудованию производственного класса (например, необходимы два источника питания).

- Если у вас есть облачный соединитель версии 2,0 или более поздней и вы разворачиваете максимальную конфигурацию 16 устройств (с большим оборудованием), ваш сайт PSTN может обрабатывать до 8 000 одновременных звонков. При развертывании конфигурации меньшего размера поддерживаемое ограничение составляет 800 вызовов.

    Кроме того, необходимо предоставить несколько устройств для обеспечения высокой доступности. Рекомендуется зарезервировать как минимум одно устройство для обеспечения высокой доступности.

  - Если вы развертываете 15 + 1 конфигурацию, сайт PSTN может поддерживать до 7 500 одновременных звонков.

  - В предыдущей версии при развертывании максимальной конфигурации 3 + 1 узел ТСОП может обрабатывать до 1500 одновременных вызовов. При развертывании наименьшей конфигурации поддерживаемое ограничение составляет 150.

-  Если на каждом узле ТСОП необходимо обрабатывать большее количество вызовов, в том же месте можно развернуть дополнительные узлы ТСОП.

> [!NOTE]
> Если не указано иное, в схемах и примерах ниже используется более крупная версия облачного соединителя.

### <a name="single-cloud-connector-appliance-within-a-single-pstn-site"></a>Одно устройство Cloud Connector в рамках одного сайта ТСОП

На приведенной ниже схеме показана отдельная версия одного устройства облачного соединителя на одном сайте PSTN. Обратите внимание, что облачный соединитель состоит из четырех виртуальных машин, которые установлены на одном физическом хост-компьютере, который находится в сети периметра в целях обеспечения безопасности.

![Один комплект Cloud Connector на одном узле PSTN](../../media/7ffe6953-8c66-4323-940e-cd2e6c3c2a66.png)

### <a name="multiple-cloud-connector-appliances-within-a-single-pstn-site"></a>Несколько устройств Cloud Connector в рамках одного сайта ТСОП

 Для обеспечения масштабируемости и высокой доступности вы можете выбрать несколько выпусков облачного соединителя на одном сайте PSTN, как показано на следующей схеме. Рассмотрим следующее:

- Вызовы в случайном порядке распределяются между экземплярами Cloud Connector в одном пуле.

- В целях планирования мощности необходимо учесть способность обрабатывать нагрузку при отключении одного или нескольких экземпляров Cloud Connector. Это делается на основе следующих вычислений:

  - **N+1 блоков.** Для более крупной версии облачного соединителя N + 1 полей поддерживают одновременные звонки 500\*n с 99,8ом доступности.

    Для уменьшенной версии облачного соединителя N + 1 полей поддерживают одновременные звонки 50\*n с 99,8ом доступности.

  - **N+2 блоков.** В более крупных версиях облачных соединителей N + 2 поддерживают одновременные звонки 500\*n с 99,9% доступности.

    Для небольших версий облачного соединителя N + 2 поля поддерживают одновременные\*звонки 50 n с 99,9ом доступности.

![Два комплекта Cloud Connector на 1 узле PSTN](../../media/fc0dc47f-5595-42cb-9432-9c8ff3e134e9.png)

### <a name="multiple-pstn-sites-with-one-or-more-cloud-connectors-per-site"></a>Несколько узлов ТСОП с одним или несколькими экземплярами Cloud Connector на каждом узле

Также можно развернуть несколько узлов ТСОП с одним или несколькими экземплярами Cloud Connector Edition на каждом узле. Если на узле ТСОП достигается предельное количество параллельных вызовов, для обработки этой нагрузки можно добавить еще один узел ТСОП.

Несколько узлов ТСОП также позволяют предоставить подключение к ближайшим к вашим пользователям шлюзам. Например, предположим, что шлюзы ТСОП находятся в Сиэтле и Амстердаме. Можно развернуть два узла ТСОП — один в Сиэтле, а другой в Амстердаме — и настроить использование пользователями ближайшего узла ТСОП. В результате пользователи из Сиэтла перенаправляются на узел и шлюзы ТСОП в Сиэтле, а пользователи в Амстердаме — на узел и шлюзы ТСОП в Амстердаме:

## <a name="requirements-for-deployment"></a>Требования для развертывания
<a name="BKMK_Requirements"> </a>

Перед развертыванием Cloud Connector Edition убедитесь в том, что для вашей среды есть следующее:

- **Для ведущего компьютера —** Виртуальные облачные соединители должны быть развернуты на специальном оборудовании под управлением Windows Server 2012 R2 Datacenter Edition (на английском языке) с включенной ролью Hyper-V.

    Для версии 2.0 и более поздних версий IP-адрес сетевой карты хост-компьютера, связанной с коммутатором Skype для бизнеса Corpnet, должен быть настроен в той же подсети, что и компьютеры корпоративной сети Cloud Connector.

- Для версии 2,1 и более поздних версий на устройстве приложения должно быть установлено приложение .NET Framework 4.6.1 или более поздняя версия.

- **Для виртуальных машин:** ISO-образ Windows Server 2012 R2 (ISO). ISO-образ будет преобразован в виртуальные жесткие диски виртуальных машин, которые будут запускать Skype для бизнеса Cloud Connector Edition.

- Необходимое оборудование для поддержки установки 4 виртуальных компьютеров для каждого выпуска облачного соединителя в развертывании. Рекомендованы следующие конфигурации:

  - 64-разрядный двухъядерный процессор, шесть ядер (12 реальных ядер), 2,50 ГГц (ГГц) или выше.

  - 64 ГБ ОЗУ с ECC

  - Четыре SAS-диска с емкостью 600 ГБ (или больше), скоростью вращения 10 тыс. об/мин, 128 МБ кэш-памяти и пропускной способностью 6 Гбит/с в конфигурации RAID 5

  - Три сетевых адаптера с разъемом RJ45 и пропускной способностью 1 Гбит/с

- Если вы решили развернуть более маленькую версию Cloud Connector Edition, поддерживающую до 50 одновременных звонков, вам понадобится следующее оборудование:

  - Четырехъядерный процессор Intel i7 4790 с набором микросхем Intel 4600 Graphics (не требуются высококлассные графические карты)

  - Память DDR3-1600 32 ГБ, без ECC

  - 2: 7200RPM SATA III 1 ТБ (6 Гбит/с) в конфигурации RAID 0

  - 2: Ethernet (RJ45) 1 Гбит/с

- Если компьютер узла использует прокси-сервер для выхода в Интернет, измените конфигурацию следующим образом.

  - Чтобы обойти прокси-сервер, укажите параметры прокси WinHTTP, заданные прокси-сервером, и список обхода, включая "192.168.213. \*"сеть, используемая вашими облачными службами управления соединителями и подсетью для подсети Skype для бизнеса, определенная в файле клаудконнектор. ini. В противном случае не удастся создать подключение для управления, а также выполнить развертывание и автовосстановление Cloud Connector. Ниже приведен пример команды конфигурации WinHTTP: netsh WinHTTP Set proxy "10.10.10.175:8080"-List = "\*. local; 1. \*; 172,20. \*; 192.168.218. \*"\<местная\>".

Чтобы указать параметры прокси-сервера на уровне компьютера, внесите изменения в реестр или измените параметры групповой политики следующим образом.

- Убедитесь, что у вас есть оборудование, необходимое для поддержки четырех виртуальных компьютеров, которые вы будете устанавливать для каждого облачного соединителя.

Для каждого сайта PSTN, который планируется развернуть, необходимо выполнить следующие действия:

- Создавайте имена всех компонентов на каждом управляющем устройстве-соединителе (см. раздел [Определение параметров развертывания](plan-skype-for-business-cloud-connector-edition.md#BKMK_SiteParams)).

- определить диапазоны портов (см. раздел [Порты и протоколы](plan-skype-for-business-cloud-connector-edition.md#BKMB_Ports)); 

- создать внешние записи DNS для пограничного компонента (см. раздел [Требования для развертывания](plan-skype-for-business-cloud-connector-edition.md#BKMK_Requirements));

- определить требования к сертификатам для пограничного компонента (см. раздел [Требования к сертификатам](plan-skype-for-business-cloud-connector-edition.md#BKMK_Certs)).

### <a name="ports-and-protocols"></a>Порты и протоколы
<a name="BKMB_Ports"> </a>

При определении диапазонов портов медиаданных следует помнить о следующем.

- Клиенты всегда используют диапазон портов 50000 – 50019 для передачи данных мультимедиа — этот диапазон предопределен в Skype для бизнеса Online и не может быть изменен.

- Для трафика медиаданных компонент-посредник по умолчанию использует диапазон портов 49 152–57 500. Однако подключение устанавливается через внутренний брандмауэр, и в целях обеспечения безопасности вы можете ограничить этот диапазон портов в вашей топологии. Для каждого вызова требуется до 4 портов. Чтобы ограничить число открытых портов между компонентом-посредником и шлюзом ТСОП, также требуется настроить соответствующий диапазон портов в шлюзе.

- Необходимо развернуть облачный соединитель в сети периметра. Это означает, что потребуются два брандмауэра.

  - Первый брандмауэр — внешний брандмауэр между Интернетом и сетью периметра.

  - Второй межсетевой экран является внутренним между демилитаризованной зоной и внутренней сетью.

    Ваши клиенты могут находиться в Интернете или внутренней сети:

  - Клиенты в Интернете подключаются к вашей ТСОП через внешний брандмауэр и пограничный компонент.

  - Клиенты во внутренней сети будут подключаться через внутренний брандмауэр к компоненту "исправления" в демилитаризованной зоне, который будет подключаться к трафику для шлюза SBC или КТСОП.

    Это означает, что необходимо открыть порты на обоих брандмауэрах.

### <a name="dns-name-resolution-for-the-edge-component"></a>Разрешение имен DNS для пограничного компонента
<a name="BKMB_Ports"> </a>

Компоненту Edge требуется разрешение внешних имен служб Office 365 и внутренних имен других компонентов облачного соединителя.

Каждый пограничный компонент представляет собой многосетевой компьютер, имеющий внешние и внутренние интерфейсы. Облачный соединитель развертывает DNS-серверы в компоненте контроллера домена в демилитаризованной зоне. Вы можете указать пограничный сервер на DNS-сервер в пределах периметра для всех разрешений имен, но необходимо включить DNS-сервер облачного соединителя для разрешения внешних имен, указав зону DNS с одной или несколькими DNS-записями A для внешних запросов, которые ссылаются на имя подстановки на другие общедоступные DNS-серверы.

Если в INI-файле настроить полное доменное имя для шлюзов из того же доменного пространства, к которому относится домен SIP, полномочная зона этого домена SIP будет создана на сервере DNS в периметре. Если сервер граничного сервера указывает на этот DNS-сервер для разрешения имен, EDGE не будет обрабатывать _сипфедератионтлс. \<йоурдомаин\> DNS-запись, которая требуется для потока звонков. В этом случае корпорация Майкрософт рекомендует предоставлять DNS-сервер на внешнем интерфейсе Edge для разрешения поиска имен в Интернете, а каждый компонент Edge должен использовать файл HOST для разрешения имен других компонентов облачного соединителя в IP-адреса.

> [!NOTE]
> По соображениям безопасности мы рекомендуем не указывать DNS-сервер облачного соединителя на внутренние серверы в домене рабочей области для разрешения имен.

### <a name="certificate-requirements"></a>Требования к сертификатам
<a name="BKMK_Certs"> </a>

Каждому компоненту Edge требуется сертификат в общедоступном центре сертификации. Для копирования между компонентами пограничного сервера сертификаты должны иметь экспортируемые закрытые ключи. Для соблюдения требований к сертификатам вам потребуется выбрать следующие параметры и указать для сертификата имя субъекта (SN) и альтернативное имя темы (SAN).

 **Если у вас есть один домен SIP:**

- **Вариант 1.** Имя субъекта должно содержать имя пула, которое вы назначили компонентам Edge. Обратите внимание, что имя субъекта не может быть sip.sipdomain.com, так как это имя зарезервировано для компонента Skype для бизнеса Edge в Интернете. Сеть хранения данных должна содержать sip.sipdomain.com и имя пула Edge Access:

  ```
  SN = accessedgepoolnameforsite1.sipdomain.com, SAN = sip.sipdomain.com,
  acessedgepoolnameforsite1.sipdomain.com
  ```

- **Вариант 2.** Если вы хотите использовать один сертификат с подстановочными знаками на всех серверах пула пограничного сервера, вы можете использовать подстановочный элемент \*в sipdomain.com, а не имя пула пограничных данных в сертификате. Имя субъекта может быть именем пула пограничного доступа для всех развернутых пулов Edge.

  ```
  SN = accessedgepoolnameforsite1.sipdomain.com, SAN = sip.sipdomain.com, SAN = *.sipdomain.com
  ```

> [!NOTE]
> Для SIP не нужно создавать внешнюю запись DNS. \<сипдомаин\>. com, так как это имя относится к развертыванию Office 365.

> [!NOTE]
> Если вы хотите использовать один сертификат для всех пулов пограничных серверов, развернутых в вашей организации, и не можете использовать сертификат с подстановочными знаками, заданные в варианте 2, необходимо включить полное доменное имя для всех развернутых пулов пограничных областей в сертификате.

 **Если у вас несколько доменов SIP:**

Вам потребуется добавить sip.sipdomain.com для каждого домена SIP и имя пула пограничного доступа для каждого домена (это может быть один из физических пулов, но с разными именами). Ниже приведен пример записей СН и SAN в сценарии для нескольких доменов SIP:

- **Вариант 1.** Имя субъекта должно содержать имя пула, которое вы назначили для компонентов Edge. Обратите внимание, что имя субъекта не может быть sip.sipdomain.com, так как это имя зарезервировано для компонента Skype для бизнеса Edge в Интернете.   ```
  SN = accessedgepoolnameforsite1.sipdomain1.com, SAN = sip.sipdomain1.com, sip.sipdomain2.com,

## <a name="high-availability-considerations"></a>Рекомендации по обеспечению высокой доступности
<a name="BKMK_HA"> </a>

При развертывании Cloud Connector Edition для обеспечения высокой доступности вы развертываете по крайней мере два устройства, которые играют роль резервного копирования. Каждое устройство состоит из четырех компонентов: EDGE, устранение, хранилище Центрального управления (CMS) и контроллер домена.

Как правило, если один компонент в управляющем устройстве отключается, облачный соединительный разъем может продолжать обрабатывать звонки, но вы должны иметь в виду следующее:

- **Сведения о компонентах «исправления», «CMS» и «контроллеры домена»**

    Предположим, что сервер CMS или контроллер домена в одном устройстве отключен. Управляющее устройство по-прежнему может обрабатывать входящие и исходящие звонки, но если вы перезапустите компонент "исправления", когда контроллер домена или CMS-компонент недоступен, это не повлияет на работу. Это относится и к перезапуску компонента CMS, когда контроллер домена не работает.

    **Рекомендация.** Прежде чем перезапускать компоненты, проверьте доступность других компонентов на устройстве.

- **Общие сведения о компонентах Edge**

    Если компонент EDGE в одном устройстве недоступен, поведение для входящих и исходящих вызовов будет отличаться от приведенных ниже.

  - **Исходящий звонок**– звонок от пользователя через Интернет в сеть PSTN.

    Механизм распространения звонков в облаке будет определять, что один компонент EDGE не работает, и будет маршрутизировать все звонки на другое устройство, чтобы исходящий вызов успешно завершился.

  - **Входящий звонок**— звонок от сети PSTN пользователю, который находится в локальной сети или в Интернете.

     Если компонент Edge устройства, получившего звонок, не работает, входящие вызовы этого устройства не будут успешными, так как компонент "исправление" не может перенаправить звонок на Граничный компонент другого устройства.

    **Рекомендация.** Система мониторинга на своем компьютере. Определив неисправность компонента EDGE, закройте все компоненты устройства, на котором компонент Edge недоступен.

<a name="BKMK_MediaFlow"> </a>

На приведенных ниже схемах описаны потоки исходящего и входящего звонков с помощью облачного соединителя Edition. Это полезные сведения для понимания того, как установлена связь.

На первой схеме внутренний пользователь поместит исходящий звонок, как описано ниже.

1. Дэйв, пользователь, который подключен к Интернету, но теперь во внутренней сети, позвонит на внешний пользователя PSTN.

2. Трафик SIP в Skype для бизнеса Online.

3. В Skype для бизнеса Online выполняется обратный просмотр номера. Обратный просмотр номера не работает, поскольку этот номер не принадлежит ни одному из пользователей в Организации Skype для бизнеса.

4. Этот звонок пересылается на компонент EDGE (поток SIP и потоки мультимедиа в режиме онлайн-Edge. Мультимедиа перейдет в компонент "исправления" через внутренний брандмауэр.

5. Если маршрут существует, компонент Edge передает трафик в компонент "исправление" в демилитаризованной зоне.

6. Компонент "исправление" отправляет трафик шлюзу PSTN.

![Исходящий поток мультимедиа для Cloud Connector](../../media/c495a2bb-305c-46ef-b16d-b8f9f2b937a8.png)

На следующей схеме внутренний пользователь получает входящий звонок, как описано ниже.

1. Шлюз PSTN получает вызов для пользователей Дэйв, которые подключены к сети, но теперь находятся во внутренней сети.

2. Трафик SIP перенаправляется в компонент "исправление".

3. Компонент "исправление" отправляет трафик SIP в компонент EDGE, а затем переходит в Skype для бизнеса Online.

4. В Skype для бизнеса Online выполняется обратный просмотр номера и обнаруживается, что это Дэйв пользователя.

5. Сигнал SIP поступает на все точки присутствия Дэйв.

6. Трафик мультимедиа будет устанавливаться между шлюзом и компонентом исправления, а также между компонентом «исправление» и конечной точкой.

## <a name="monitoring-and-troubleshooting"></a>Мониторинг и устранение неполадок
<a name="BKMK_Monitor"> </a>

Механизм мониторинга и устранения неполадок устанавливается автоматически при каждом управляющем соединителе облака. Механизм обнаруживает следующие события:

- Одна или несколько виртуальных машин устройства облачного соединителя не подключены к внутреннему или виртуальному коммутатору Интернета.

- Одна или несколько виртуальных машин устройства облачного соединителя находятся в состоянии "сохранено" или "остановлено".

- Незапущенные службы.

  Если обнаружено одно из указанных ниже событий, все управляющее устройство облачного соединителя будет стоко и помечено как не в сети, чтобы предотвратить попытку установить звонки на неработающее устройство. После этого функции автоматического восстановления в облачном соединителе будут восстанавливать службы и помечать устройство как подключенное к сети. Если не удается выполнить автоматическое восстановление по какой – либо причине, ознакомьтесь с разделом [Устранение неполадок развертывания облачного соединителя](troubleshoot-your-cloud-connector-deployment.md).

  - На виртуальной машине в магазине Central Management:

     - Агент главной репликации Skype для бизнеса

     - Агент репликации реплики в Skype для бизнеса

  - На виртуальной машине сервера исправлений:

     - Агент репликации реплики в Skype для бизнеса

     - Устранение неполадок в Skype для бизнеса Server

  - На виртуальной машине пограничного сервера

     - Агент репликации реплики в Skype для бизнеса

     -  Microsoft Skype для бизнеса Server — Edge

     - Аудио-и видеоedge в Skype для бизнеса Server

     - Проверка подлинности звука и видео в Skype для бизнеса Server

     - Веб-конференции в Skype для бизнеса Server

- Правило входящего трафика брандмауэра Windows для "CS РТКСРВ" в EDGE, "CS РТКМЕДСРВ" на сервере исправной сети отключено.

Cloud Connector 2.1 и более поздние версии поддерживают мониторинг Cloud Connector с помощью Operations Management Suite (OMS). Дополнительные сведения: [Monitor Cloud Connector using Operations Management Suite (OMS)](monitor-cloud-connector-using-operations-management-suite-oms.md).
