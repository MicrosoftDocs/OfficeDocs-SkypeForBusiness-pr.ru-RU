# <a name="dns-requirements-for-simple-urls-in-skype-for-business-server"></a>Требования DNS для простых URL-адресов в Skype для бизнеса Server

**Сводка:** Прежде чем внедрять записи DNS для Skype для бизнеса Server, рассмотрите соображения по простому URL-адресу, рассмотренные в этом разделе.

Простые URL-адреса упрощают присоединение пользователей к собраниям и упрощают доступ к средствам администрирования Skype для бизнеса Server для администраторов. Простые URL-адреса используют собственный домен, который не должен совпадать ни с одним из определенных вами доменов SIP. 

Skype для бизнеса Server поддерживает следующие три простых URL-адреса: meet, Dial-In и Admin. Вам необходимо настроить простые URL-адреса для meet и Dial-In, а простой URL-адрес администратора является необязательным. Записи службы доменных имен (DNS), необходимые для поддержки простых URL-адресов, зависят от того, как именно вы определили эти простые URL-адреса, а также необходима ли поддержка аварийного восстановления для простых URL-адресов. 

## <a name="simple-url-scope"></a>Область простого URL-адреса

Вы можете настроить для простых URL-адресов глобальную область действия или задать разные простые URL-адреса для каждого центрального узла в организации. Если заданы как глобальный простой URL-адрес, так и простой URL-адрес на уровне узла, последний имеет приоритет. 

В большинстве случаев рекомендуется устанавливать простые URL-адреса только на глобальном уровне, чтобы простой URL-адрес "Meet" пользователя не менялся при переходе с одного сайта на другой. Исключением являются организации, в которых требуются различные номера телефонного подключения для пользователей в разных узлах. Обратите внимание на то, что если вы зададите один простой URL-адрес (например, простой URL-адрес телефонного подключения) в узле как простой URL-адрес на уровне узла, вам также нужно задать другие простые URL-адреса как адреса на уровне узла.

Глобальные простые URL-адреса можно настроить в построителье топологий. Чтобы установить простой URL-адрес на уровне сайта, используйте Set-CsSimpleURLConfiguration.

Определение простого URL-адреса также потребует настройки записи A и/или AAAA в конфигурации DNS.

## <a name="simple-url-naming-and-validation-rules"></a>Правила именования и проверки простых URL-адресов
<a name="BK_Valid"> </a>

Построитель топологий и команды управления Skype для бизнеса Server принудительно применяют несколько правил проверки для простых URL-адресов. Вы должны задать простые URL-адреса для собраний и телефонного подключения, однако задавать URL-адрес администратора необязательно. Для каждого домена SIP требуется отдельный простой URL-адрес собраний, однако для всей организации нужен только один простой URL-адрес телефонного подключения и URL-адрес администратора.

Каждый простой URL-адрес в организации должен иметь уникальное имя и не может быть префиксом другого простого URL-адреса (например, вы не можете установить SfB2015.contoso.com/Meet в качестве простого URL-адреса meet, а SfB2015.contoso.com/Meet/Dialin в качестве простого URL-адреса Dialin). Имена простых URL-адресов не могут содержать имя FQDN пулов или сведения о портах (например, https://FQDN:88/meet не разрешено). Простые URL-адреса должны начинаться с префикса https://. 

Простые URL-адреса могут содержать только буквы и цифры (a–z, A–Z, 0–9), а также точки (.). При использовании других символов простые URL-адреса могут работать неправильно.

## <a name="changing-simple-urls-after-deployment"></a>Изменение простых URL-адресов после развертывания
<a name="BK_Valid"> </a>

Если вы изменяете простой URL-адрес после развертывания, вам следует учесть, как это изменение повлияет на записи DNS и сертификаты для простых URL-адресов. Если изменяется базовый адрес простого URL-адреса, то вам следует изменить соответствующим образом записи DNS и сертификаты. Например, измените базовый URL-адрес с SfB2015.contoso.com на meet.contoso.com, чтобы изменить записи DNS и сертификаты для ссылки на https://SfB2015.contoso.com/Meet https://meet.contoso.com meet.contoso.com. Если вы изменили простой URL-адрес с на , базовый URL-адрес SfB2015.contoso.com остается таким же, поэтому DNS или сертификат изменений https://SfB2015.contoso.com/Meet https://SfB2015.contoso.com/Meetings не требуется.

Однако при каждом изменении имени простого URL-адреса для регистрации изменения необходимо запустить **enable-CsComputer** на каждом директоре и сервере переднего сервера.

## <a name="naming-examples-for-simple-urls"></a>Примеры именования простых URL-адресов
<a name="BK_Valid"> </a>

Существует три рекомендованных способа задания имен простых URL-адресов. Выбираемый способ косвенно зависит от способа настройки записей A DNS и сертификатов для поддержки простых URL-адресов. В любом случае вам нужно настроить один простой URL-адрес собраний для каждого домена SIP в организации. 

В любом случае в организации требуется только один простой URL-адрес для телефонного подключения и один URL-адрес администратора вне зависимости от количества доменов SIP.

В первом случае вам нужно создать имя домена SIP для каждого простого URL-адреса.

В этом случае для каждого простого URL-адреса требуется отдельная запись A DNS, а каждый простой URL-адрес собрания должен быть поименован в сертификатах

**Вариант именования простых URL-адресов 1**


| **Простой URL-адрес** <br/> | **Пример** <br/>                                                                                                    |

При варианте 2 простые URL-адреса основаны на имени SfB2015.contoso.com. Поэтому вам нужна только одна запись A DNS, которая позволяет использовать все три типа простых URL-адресов. Эта запись DNS A ссылается на SfB2015.contoso.com. Кроме того, вам требуются отдельные записи A DNS для других доменов SIP в организации.

## <a name="disaster-recovery-option-for-simple-urls"></a>Параметр аварийного восстановления для простых URL-адресов
<a name="BK_Valid"> </a>

Если имеется несколько сайтов, содержащих пулы переднего уровня, а ваш поставщик DNS поддерживает GeoDNS, можно настроить записи DNS для простых URL-адресов для поддержки аварийного восстановления, чтобы функции простых URL-адресов продолжались даже в случае сработки всего пула переднего уровня. Этот компонент аварийного восстановления поддерживает простые URL-адреса Meet и Dial-In.

Для выполнения этой настройки создайте два адреса GeoDNS. Каждый адрес содержит две записи DNS A или CNAME, которые разрешаются в два пула, спаренных в целях аварийного восстановления. Один адрес GeoDNS используется для внутреннего доступа и разрешается в полное доменное имя внутренней веб-службы или IP-адрес балансировщика нагрузки для двух пулов. Другой адрес GeoDNS используется для внешнего доступа и разрешается в полное доменное имя внешней веб-службы или IP-адрес балансировщика нагрузки для двух пулов. Ниже приведен пример простого URL-адреса Meet для полных имен доменов этих пулов. 

```console
Meet-int.geolb.contoso.com
     Pool1InternalWebFQDN.contoso.com
     Pool2InternalWebFQDN.contoso.com
```

```console
Meet-ext.geolb.contoso.com
     Pool1ExternalWebFQDN.contoso.com
     Pool2ExternalWebFQDN.contoso.com
```

Затем создайте записи CNAME, которые разрешают простой URL-адрес meet (например, meet.contoso.com) в два адреса GeoDNS.

> [!NOTE]
> Если сеть поддерживает разворот пакетов (маршрутизация всего трафика простых URL-адресов через внешнюю ссылку, включая трафик из организации), можно просто настроить внешний адрес GeoDNS и разрешать простой URL-адрес Meet только во внешний адрес.

При использовании этого метода можно настроить каждый адрес GeoDNS для использования метода циклического перебора для распределения запросов между двумя пулами или для подключения в основном к одному пулу (например, к пулу, который географически ближе расположен) и использования другого пула только в случае сбоя подключения. 

Можно задать ту же конфигурацию для простого URL-адреса Dial-In. Для этого создайте дополнительные записи, как в предыдущем примере, заменив их на  `dialin` `meet` записи DNS. Для простого URL-адреса Admin используйте один из трех вариантов, приведенных выше в этом разделе.

После настройки этой конфигурации необходимо использовать приложение мониторинга для настройки мониторинга HTTP на предмет сбоев. Для внешнего доступа необходимо следить за тем, чтобы httpS GET lyncdiscover.<sipdomain> Запросы к внешнему веб-FQDN или IP-адресу балансировки нагрузки для двух пулов успешны. Например, следующие запросы не должны содержать какие-либо заголовки **ACCEPT** и должны возвращать **200 OK**.

```console
HTTPS GET Pool1ExternalWebFQDN.contoso.com/autodiscover/autodiscoverservice.svc/root
HTTPS GET Pool2ExternalWebFQDN.contoso.com/autodiscover/autodiscoverservice.svc/root
```

Для внутреннего доступа следует отслеживать порт 5061 на полном доменном имени внутреннего интерфейса или IP-адресе балансировщика нагрузки применительно к двум пулам. При обнаружении каких-либо сбоев подключения VIP-порт для этих пулов должен закрыть порты 80, 443 и 4443.
