# <a name="deployment-guidelines-for-mediation-server-in-skype-for-business-server"></a>Рекомендации по развертыванию для сервера-посредника в Скайп для Business Server
 
В этом разделе описываются рекомендации по планированию для развертывания сервера-посредника.
  
## <a name="collocated-or-stand-alone-mediation-server"></a>Сервер-посредник выровненный или изолированный?

Сервер-посредник по умолчанию, совмещенного на сервере Standard Edition или сервера переднего плана в пуле переднего плана в центральных сайтов. Число вызовов телефонной сети (общего пользования PSTN), которые могут быть обработаны и количество компьютеров, необходимых в пуле зависит от:
  
- Число шлюзов, которыми управляет пула серверов-посредников.
    
- от трафика через эти шлюзы в периоды высокой нагрузки;
    
- Процент вызовов, которые обходят сервер-посредник.
    
При планировании, необходимо учитывать требования для вызовов ТСОП и обработки мультимедиа аудио- и видеоконференций, которые не поддерживают мультимедиа обхода и обработки, необходимые для обработки сигналы взаимодействия для количество вызовов доступности час, которые должны быть поддерживается. Если у вас нет достаточного количества ЦП, то необходимо развертывание изолированного пула серверов-посредников. Кроме того шлюзы ТСОП, IP-УАТС и их изготовителей потребуется разбить на подмножества, управляются выровненных серверов-посредников в одном пуле и изолированных серверов посредников в один или несколько пулов изолированный.
  
Если вы развернули шлюзов PSTN, IP-УАТС или пограничных контроллеров сеансов (SBC), который не имеют возможность взаимодействовать с пулом серверов-посредников, их необходимо должен быть связан с автономного пула, состоящий из одного сервера-посредника. Некоторые действия шлюзов PSTN, IP-PBX или SBC потребуется включают:
  
- Выполнять сетевого уровня доменных имен (DNS) балансировки нагрузки между серверов-посредников в пуле (или другим способом равномерно маршрутизировать трафик на все серверы-посредники в пуле).
    
- Примите трафик от любого сервера-посредника в пуле.
    
Скайп для средство планирования бизнеса можно использовать для оценки ли collocating сервер-посредник с интерфейсным пулом может справиться с нагрузкой. Если вашей среде не может соответствовать следующим требованиям, то необходимо для развертывания изолированного пула серверов-посредников.
  
## <a name="central-site-and-branch-site-considerations"></a>Рекомендации для центрального сайта и сайта филиала

 Серверов-посредников на центральном сайте можно использовать для маршрутизации вызовов для IP-УАТС или шлюзы ТСОП на сайтах филиалов. Если развертывание магистралях SIP, тем не менее, необходимо развернуть сервером-посредником на сайте, где завершает каждой магистрали. С сервером-посредником на центральном сайте вызывает маршрута для IP-УАТС или шлюз ТСОП на сайте филиала не требует использования носителя обхода, но рекомендуется сервера-посредника. Это связано с тем, что при включении сервера-посредника его будет уменьшения задержки пути мультимедиа и, следовательно, привести в качестве улучшенные мультимедиа из-за пути передачи данных не требуется для выполнения сигналы путь. Обход сервера-посредника также будет уменьшить нагрузку в пуле.
  
> [!NOTE]
> Функция обхода сервера-посредника взаимодействует не со всеми шлюзами ТСОП, IP-PBX и пограничным контроллером SBC. Корпорация Microsoft протестировала набор шлюзов ТСОП и контроллеров SBC с сертифицированными партнерами и провела некоторые тесты для Cisco IP-PBX. Обход сервера-посредника поддерживается только с помощью продуктов и версий перечисленные в объединенных коммуникаций программы открытого взаимодействия - Lync Server в [Проводник проверен устройства, инфраструктуры и средства, которые поддерживают и расширить вашей Скайп для бизнеса программой](http://partnersolutions.skypeforbusiness.com/solutionscatalog). 
  
Если требуется устойчивости сайта филиала, для обеспечения связи в филиалах или сочетание сервера переднего плана, на сервер-посредник и шлюз должен быть развернут на сайте филиала. (С устойчивости сайта филиала предполагается, что сведения о присутствии и конференц-связи не устойчивое на узле.) Руководство на сайте филиала, планирование голосовой связи см нашей документации [планирования для обеспечения устойчивости голосовой связи сайта филиала](https://technet.microsoft.com/en-us/library/gg398477%28v=ocs.15%29.aspx) , как оно должно быть по-прежнему относится к Скайп для Business Server 2015.
  
Для взаимодействия с IP-УАТС Если IP-PBX не поддерживает правильно первых мультимедийные взаимодействия с помощью нескольких первых диалоговые окна и взаимодействия RFC 3960 может быть Вырезка первый несколько слов приветствие для входящие звонки от IP-УАТС к конечным точкам Lync. Это может быть более сложных, если сервером-посредником на сайте центра маршрутизации вызовов для IP-УАТС где завершает маршрут в филиале, так как требуется больше времени для передачи сигналов для выполнения. При возникновении этой проблемы развертывание сервера-посредника на сайте филиала является единственным способом для уменьшения Вырезка первый несколько слов.
  
И, наконец если центральный узел имеет TDM УАТС или IP-PBX не отказаться от шлюза ТСОП, необходимо развернуть шлюз в подключении сервером-посредником и УАТС маршрут вызова.
  
> [!NOTE]
> Чтобы улучшить производительность автономного сервера-посредника, следует включить масштабирование на принимающей стороне (RSS) в сетевых адаптерах на этих серверах. RSS поддерживает параллельную обработку входящих пакетов несколькими процессорами сервера. Дополнительные сведения см «[Со стороны получения масштабирование усовершенствования в Windows Server](https://go.microsoft.com/fwlink/p/?LinkId=268731)». Дополнительные сведения о том, как включить RSS, см. в документации вашего сетевого адаптера.
