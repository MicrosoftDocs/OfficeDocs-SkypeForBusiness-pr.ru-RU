# <a name="deployment-guidelines-for-mediation-server-in-skype-for-business-server"></a>Рекомендации по развертыванию сервера обновлений в Skype для бизнеса Server
 
В этой статье описаны рекомендации по планированию развертывания сервера исправлений.
  
## <a name="collocated-or-stand-alone-mediation-server"></a>Размещенный или изолированный сервер-посредник?

Сервер-посредник по умолчанию размещен на сервере Standard Edition или на сервере переднего плана в пуле переднего плана на центральных сайтах. The number of public switched telephone network (PSTN) calls that can be handled and the number of machines required in the pool will depend on:
  
- Количество одноранговых узлов шлюза, которые находятся в элементе управления "пул сервера исправлений".
    
- от трафика через эти шлюзы в периоды высокой нагрузки;
    
- Процент звонков, в которых мультимедиа обходит сервер исправлений.
    
When you're planning, be sure to take into account the media processing requirements for PSTN calls and A/V conferences that don't support media bypass, as well as the processing needed to handle signaling interactions for the number of busy-hour calls that need to be supported. Если у вас недостаточно ресурсов ЦП, вам потребуется развернуть изолированный пул серверов-посредников. Кроме того, шлюзы PSTN, IP-АТС и SBCs должны быть разделены на подмножества, управляемые объединенными серверами в одном и том же пуле, а также в одном или нескольких изолированных пулах.
  
Если вы развернули шлюзы PSTN, IP-АТС или контроллеры границ сеансов (SBCs), которые не могут взаимодействовать с пулом серверов-исправлений, они должны быть связаны с отдельным пулом, состоящим из одного сервера-посредника. Some of the things your PSTN gateways, IP-PBXs or SBCs would need to do include:
  
- Выполняйте балансировку нагрузки DNS-данных в сетевом уровне для серверов обновлений в пуле (или иным образом трафик по маршруту для всех серверов-посредников в пуле).
    
- Пронимайте трафик с любого сервера-посредника в пуле.
    
Вы можете использовать средство планирования Skype для бизнеса, чтобы определить, может ли выгрузить Организатор сервера-посредника с интерфейсом переднего плана. Если среда не отвечает этим требованиям, вам потребуется развернуть пул серверов изолированных исправлений.
  
## <a name="central-site-and-branch-site-considerations"></a>Рекомендации для центрального сайта и сайта филиала

 Серверы исправлений на центральном веб-сайте могут использоваться для маршрутизации звонков для шлюзов IP-АТС или PSTN на сайтах филиалов. Однако при развертывании магистральных каналов SIP вы должны развернуть сервер-посредник на сайте, на котором завершается Каждая магистральная линия. Если у вас есть сервер-посредник на маршруте на центральном веб-сайте или шлюзе PSTN на сайте филиала, использовать обход мультимедиа не требуется, но рекомендуется обойтись файлами мультимедиа. That's because, if you can enable media bypass, it'll reduce media path latency and, consequently, result in improved media quality because the media path isn't required to follow the signaling path. Media bypass will also decrease the processing load on the pool.
  
> [!NOTE]
> Функция обхода сервера-посредника взаимодействует не со всеми шлюзами ТСОП, IP-PBX и пограничным контроллером SBC. Корпорация Microsoft протестировала набор шлюзов ТСОП и контроллеров SBC с сертифицированными партнерами и провела некоторые тесты для Cisco IP-PBX. Обход мультимедиа поддерживается только в том случае, если у вас есть продукты и версии, указанные в едином приложении для взаимодействия с открытым интерфейсом, на Lync Server на сайте [изучение протестированных устройств, инфраструктуры и средств, которые поддерживают и расширяют возможности Skype для бизнеса](http://partnersolutions.skypeforbusiness.com/solutionscatalog). 
  
Если требуется устойчивость сайта филиала, то можно разворачивать работающее устройство филиала или сочетание сервера переднего плана, сервера-посредника и шлюза на сайте филиала. (Допущение с учетом устойчивости сайтов филиалов состоит в том, что присутствие и конференции на сайте не устойчивы.) Рекомендации по планированию веб-сайтов для голосовой связи можно найти [в статье Планирование устойчивости корпоративной голосовой связи в Skype для бизнеса Server](../enterprise-voice-solution/enterprise-voice-resiliency.md).
  
При взаимодействии с IP-УАТС, если IP-УАТС неправильно поддерживает раннее взаимодействие с мультимедиа в нескольких ранних диалоговых окнах и взаимодействиях RFC 3960, может быть выполнена обрезка первых нескольких слов приветствия для входящих звонков из IP-УАТС в конечные точки Lync. Такое поведение может быть более серьезным, если сервер-посредник на центральном сайте является маршрутной маршрутизацией для IP-УАТС, в которой маршрут завершается на сайте филиала, так как для завершения передачи сигналов требуется больше времени. Если вы попытаетесь устранить эту проблему, развертывание сервера обновлений на сайте филиала является единственным способом сокращения количества первых слов.
  
Наконец, если у вашего центрального сайта есть УАТС ТДМ или если ваш IP-УАТС не устраняет необходимость в шлюзе PSTN, необходимо развернуть шлюз на сервере-отсоединении и АТС, соединяющем маршрут звонков.
  
> [!NOTE]
> Чтобы улучшить производительность автономного сервера-посредника, следует включить масштабирование на принимающей стороне (RSS) в сетевых адаптерах на этих серверах. RSS поддерживает параллельную обработку входящих пакетов несколькими процессорами сервера. Дополнительные сведения можно найти[в разделе улучшенные возможности масштабирования на стороне приема в Windows Server](https://go.microsoft.com/fwlink/p/?LinkId=268731). Сведения о том, как включить RSS, см. в документации сетевого адаптера.
