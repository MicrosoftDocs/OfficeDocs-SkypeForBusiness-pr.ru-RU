# <a name="front-end-pool-disaster-recovery-in-skype-for-business-server"></a>Аварийное восстановление пула переднего сервера в Skype для бизнеса Server
 
Для аварийного восстановления Skype для бизнеса Server предлагает сопряжение пула с отбойным процессом на случай сбой одного пула.
  
Для наиболее надежных вариантов аварийного восстановления в Skype для бизнеса Server развернем пары пулов переднего сервера на двух географически распределенных сайтах. Каждый сайт имеет пул переднего входа, который сопряжен с соответствующим пулом переднего сайта на другом сайте. Оба сайта активны, а служба резервного копирования обеспечивает репликацию данных в режиме реального времени для синхронизации пулов. См. ["Развертывание сопряженных](../../deploy/deploy-high-availability-and-disaster-recovery/front-end-pools-for-disaster-recovery.md) пулов переднего сервера для аварийного восстановления в Skype для бизнеса Server", если вы хотите реализовать сопряжение пулов переднего сервера.
  
![Отображает пулы переднего уровня на двух разных сайтах, сопряженных друг с другом](../../media/f74533c0-a10e-4f18-85a8-b9a008497573.jpg)
  
В случае сбой пула на одном сайте можно перенаправить пользователей из этого пула в пул на другом сайте, который затем обслуживает всех пользователей в обоих пулах. При планировании емкости следует разработать каждый пул, чтобы он мог обрабатывать нагрузку всех пользователей в обоих пулах в случае аварии.
  
Два центра обработки данных, которые включают пулы переднего входа, сопряженные друг с другом, могут быть на любом расстоянии друг от друга. Рекомендуется использовать два центра обработки данных в одном регионе мира с высокоскоростными связями между ними. 
  
Наличие двух центров обработки данных в разных регионах мира возможно, но может привести к большей потере данных в случае аварии из-за задержки репликации данных.
  
При планировании парных пулов следует помнить, что поддерживаются только следующие пары:
  
- Пулы Enterprise Edition могут быть сопряжены только с другими пулами Enterprise Edition. Аналогично, пулы Standard Edition могут быть сопряжены только с другими пулами Standard Edition.
    
- Физические пулы могут быть сопряжены только с другими физическими пулами. Аналогичным образом виртуальные пулы могут быть сопряжены только с другими виртуальными пулами.
    
- В сопряженных пулах должна быть запущена одна и та же базовая операционная система.
    
Ни построитель топологий, ни проверка топологии не запрещают сопряжение двух пулов способом, который не следует этим рекомендациям. Например, построитель топологий позволяет сопряжение пула Enterprise Edition с пулом Standard Edition. Однако эти типы сопряжений не поддерживаются.
  
## <a name="backup-registrar-relationships-and-survivable-branch-appliances"></a>Связи резервного регистратора и устройства для связи в филиалах

Помимо обеспечения возможности аварийного восстановления, два сопряженных пула служат как резервные регистраторы друг для друга. Каждый пул может быть резервным только для одного другого пула переднего план.
  
Хотя отношения резервного копирования между двумя пулами переднего планка должны быть симметричными и 1:1, каждый пул переднего плановых пулов также может быть резервным регистратором для любого количества устройств для survivable Branch Appliance.
  
Обратите внимание, что Skype для бизнеса не расширяет поддержку аварийного восстановления для пользователей, которые были в филиалах. При отказе пула переднего планирования, который служит резервной копией для устройства для обеспечения поддержки в филиалах, пользователи, во время входа в устройство для обеспечения поддержки в филиалах, перенаправляются в режим устойчивости, даже если пользователи, которые находятся в пуле переднего планирования, перенаправляются в резервный пул переднего планирования.
  
## <a name="recovery-time-for-pool-failover-and-pool-failback"></a>Время восстановления для отката пула и отката пула

Для отладки пула и восстановления пула целевое время восстановления (RTO) составляет 15-20 минут. Это время, необходимое для от сбойного процесса, после того как администраторы определили, что произошла авария, и начали процедуры от сбойов. Оно не учитывает время, необходимое администраторам для оценки ситуации и принятия решения, а также не учитывает время, необходимое пользователям для повторного входа после завершения отработки отказа.
  
Для отката пула и восстановления пула целевой точкой восстановления (RPO) является 5 минут. Оно представляет временную меру данных, которые могут быть потеряны вследствие аварии из-за задержки репликации службы резервного копирования. Например, если пул выходит из пула в 10:00, а RPO составляет 5 минут, данные, записанные в пул в период между 9:55 A.M. и 10:00 может не реплицироваться в резервный пул и будет потеряно.
  
Все значения RTO и RPO в этом документе предполагают, что два центра обработки данных размещены в одном мировом регионе с высокоскоростным транспортом без задержек между двумя узлами. Эти значения измеряются для пула с 40 000 одновременно активных пользователей и 200 000 пользователей, включенных в Skype для бизнеса, в отношении предварительно определенной пользовательской модели, в которой нет неисполненности репликации данных. Они могут изменяться после тестирования производительности и проверки.
  
## <a name="central-management-store-failover"></a>Отбой центрального банка управления

Центральное хранилище управления содержит данные конфигурации о серверах и службах в развертывании. Каждое развертывание Skype для бизнеса Server включает одно центральное хранилище управления, которое размещено на тылевом сервере одного пула переднего сервера.
  
При сопряжении пула, в котором размещено центральное хранилище управления, в резервном пуле будет настроена резервная база данных центрального банка управления. В любой момент одна из двух баз данных центрального банка управления активна, а другая — в режиме ожидания. Контент реплицируется службой резервного копирования из активной базы данных в резервную.
  
![Показывает два пула переднего копирования: один с активным хранилищем CMS, а другой с пассивным резервным хранилищем CMS](../../media/aa479398-eb56-4854-8d50-1eff39c58a56.jpg)
  
Во время отсеченного пула, в который входит пул, в который входит центральное хранилище управления, необходимо перенаправить центральное хранилище управления перед отбойом пула переднего входа.
  
После устранения аварии восстановление центрального банка управления не требуется. Центральное хранилище управления может оставаться в пуле, в который он был перенаправлен.
  
Проектные цели для отладки центрального банка управления : 5 минут для целевого времени восстановления (RTO) и 5 минут для целевой точки восстановления (RPO).
  
## <a name="front-end-pool-pairing-data-security"></a>Безопасность данных сопряжения пула переднего конца

Служба резервного копирования непрерывно передает пользовательские данные и содержимое конференций между двумя сопряженными интерфейсными пулами. Данные пользователя содержат пользовательские URIS SIP, а также расписания конференций, списки контактов и параметры. Содержимое конференций включает отправки Microsoft PowerPoint, а также доски, используемые в конференциях.
  
Из пула источника эти данные экспортируются из локального хранилища, замещены, а затем передаются в целевой пул, где они разрезаются и импортируются в локальное хранилище. Служба резервного копирования предполагает, что канал связи между двумя центрами обработки данных находится в пределах корпоративной сети, защищенной от Интернета. Он не шифрует переданные данные между двумя центрами обработки данных и не инкапсулируется в безопасном протоколе, например HTTPS. Следовательно, атака "человек в середине" из внутреннего персонала в корпоративной сети возможна.
  
Любое предприятие, которое развертывает Skype для бизнеса Server в нескольких центрах обработки данных и использует функцию аварийного восстановления, должно обеспечить защиту трафика между центрами обработки данных с помощью корпоративной интрасети. Предприятия, для которых важно обеспечение внутренней защиты от атак, должны обеспечить безопасность каналов связи между центрами обработки данных. Это стандартное требование, которое также помогает поддерживать многие другие типы корпоративных конфиденциальных данных, которые передаются между центрами обработки данных.
  
Хотя риск атаки с посредником внутри корпоративной сети существует, он относительно небольшой по сравнению с предоставлением доступа к трафику из Интернета. В частности, пользовательские данные, доступные службой резервного копирования (например, SIP URIS), обычно доступны всем сотрудникам компании с помощью других средств, таких как глобальная адресная книга или другое программное обеспечение каталогов. Поэтому основное внимание должно быть уделялось обеспечению безопасности WAN между двумя центрами обработки данных, когда служба резервного копирования используется для копирования данных между двумя сопряженными пулами.
  
### <a name="mitigating-security-risks"></a>Меры по смягчению рисков безопасности

Имеется множество способов повышения защиты трафика службы резервного копирования. Это может быть от ограничения доступа к центрам обработки данных до защиты транспорта WAN между двумя центрами обработки данных. В большинстве случаев предприятия, развертывающие Skype для бизнеса Server, могут уже иметь необходимую инфраструктуру безопасности. Для предприятий, которые ищут руководство, корпорация Майкрософт предоставляет решение в качестве примера создания безопасной ИТ-инфраструктуры. Подробные сведения [https://go.microsoft.com/fwlink/p/?LinkId=268544](https://go.microsoft.com/fwlink/p/?LinkId=268544) см. в . 
  
Мы не подразумеваем, что это единственное решение, и не подразумеваем, что это предпочтительное решение для Skype для бизнеса Server. Мы рекомендуем корпоративным клиентам выбирать решение, которое соответствует их конкретным потребностям, в зависимости от инфраструктуры ИТ-безопасности и требований. В примере решения Майкрософт применяются IPSec и групповая политика для изоляции сервера и домена.
  
Другим возможным решением является использование IPSec только для защиты данных, отправленных самой службой резервного копирования. При выборе этого метода необходимо настроить правила IPSec для протокола SMB для следующих серверов, где пул A и пул Б являются двумя сопряженными пулами переднего сервера.
  
- Служба SMB (TCP/445) от каждого сервера переднего сервера в пуле A до файловой папки, используемой пулом B.
    
- Служба SMB (TCP/445) с каждого сервера переднего сервера в пуле B в хранилище файлов, используемом пулом A.
    
> [!CAUTION]
>  IPsec не предназначен для замены безопасности на уровне приложения, например SSL/TLS. Одно из преимуществ использования IPsec заключается в том, что он может обеспечить безопасность сетевого трафика для существующих приложений без необходимости их изменения. Предприятиям, которые хотят просто защитить транспорт между двумя центрами обработки данных, следует проконсультироваться со своими поставщиками сетевого оборудования о способах настроить безопасные подключения WAN с помощью оборудования поставщика.
