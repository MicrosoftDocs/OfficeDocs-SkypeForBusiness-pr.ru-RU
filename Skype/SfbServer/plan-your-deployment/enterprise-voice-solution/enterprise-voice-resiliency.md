# <a name="plan-for-enterprise-voice-resiliency-in-skype-for-business-server"></a>Планирование устойчивости Корпоративная голосовая связь в Skype для бизнеса Server

Узнайте, как поддерживать устойчивость голосовой связи в Skype для бизнеса Server Корпоративная голосовая связь, как на центральных сайтах, так и на сайтах филиалов. Варианты сайта филиала включают развертывание устройств для обеспечения связи в филиалах или серверов филиалов.

Отказоустойчивость голосовой связи означает возможность пользователей продолжать совершать и получать звонки, если центральный сайт, на котором Skype для бизнеса Server, становится недоступным независимо от того, из-за сбоя глобальной сети (WAN) или другой причины. При сбое центрального узла служба Enterprise Voice должна без прерываний перейти на резервный узел. При сбое глобальной сети вызовы узла филиала должны перенаправляться на локальный шлюз ТСОП. В данном разделе описывается планирование устойчивости голосовой связи при сбое центрального узла или глобальной сети.

## <a name="central-site-resiliency"></a>Устойчивость центрального сайта

Все чаще у предприятий появляется много сайтов, распределенных по всему миру. Обслуживание экстренных служб, доступ к службе технической поддержки и возможность выполнять критически важные бизнес-задачи, когда центральный сайт не обслуживается, крайне важно для любого решения Корпоративная голосовая связь обеспечения устойчивости. Когда центральный сайт выходит из строя, должны выполняться следующие условия.

- Должна обеспечиваться отработка отказа голосовой связи.

- Пользователи, которые обычно регистрируются в пуле переднего плана на центральном сайте, должны иметь возможность зарегистрироваться в альтернативном пуле переднего плана. Это можно сделать, создав несколько записей SRV DNS, каждая из которых разрешается в пул директоров или пул переднего плана на каждом центральном сайте. Вы можете настроить приоритет и вес записей SRV, чтобы пользователи, обслуживаемые этим центральным сайтом, опережали соответствующий директор и пул переднего плана в других записях SRV.

- Входящие и исходящие вызовы пользователей, расположенных на других сайтах, должны перенаправляться в ТСОП.

В этой статье описывается рекомендуемое решение для защиты устойчивости центрального сайта голосовой связи.

### <a name="architecture-and-topology"></a>Архитектура и топология

Для планирования устойчивости голосовой связи на центральном сайте требуется базовое представление о центральной роли, Skype для бизнеса Server регистратора при включении голосовой отработки отказа. Регистратор Skype для бизнеса Server — это служба, которая позволяет выполнять регистрацию и проверку подлинности клиента, а также предоставляет службы маршрутизации. Он выполняется на выпуск Standard сервере, сервере переднего плана, директоре или устройстве филиала для обеспечения связи. Пул регистраторов состоит из служб регистратора, работающих в пуле переднего плана и размещенных на одном сайте. Клиент Skype для бизнеса обнаруживает пул переднего плана с помощью следующего механизма обнаружения:

1. DNS-запись SRV

2. Веб-служба автообнаружения

3. Параметр DHCP 120

После Skype для бизнеса клиент подключается к пулу переднего плана, он направляется подсистемой балансировки нагрузки на один из серверов переднего плана в пуле. Этот интерфейсный сервер, в свою очередь, перенаправляет клиента к предпочтительному регистратору в пуле.

Каждый пользователь, Корпоративная голосовая связь, назначается определенному пулу регистраторов, который становится основным пулом регистраторов этого пользователя. На конкретном сайте сотни или тысячи пользователей обычно совместно используют один основной пул регистратора. Учитывая потребление ресурсов центрального сайта всеми пользователями сайтов филиалов, функции присутствия, конференц-связи или отработки отказа которых зависят от центрального сайта, мы рекомендуем рассматривать каждого пользователя сайта филиала, как если бы он был зарегистрирован на центральном сайте. В настоящее время нет ограничений на количество пользователей сайта филиала, включая пользователей, зарегистрированных на устройстве для обеспечения связи филиалов.

Для обеспечения устойчивости голосовой связи в случае сбоя центрального сайта основной пул регистратора должен иметь один назначенный резервный пул регистратора, расположенный на другом сайте. Резервное копирование можно настроить с помощью построитель топологий параметров устойчивости. Если существует устойчивая связь WAN между этими двумя сайтами, то пользователи, чей основной пул регистратора стал недоступен, автоматически перенаправляются в резервный пул регистратора.

Следующие действия описывают процесс обнаружения и регистрации клиента.

1. Клиент обнаруживает Skype для бизнеса Server записи DNS SRV. В Skype для бизнеса Server записи SRV DNS можно настроить для возврата нескольких полных доменных имен в SRV-запрос DNS. Например, если предприятие Contoso имеет три центральных сайта (в Северной Америке, в Европе и в Азиатско-Тихоокеанском регионе), и пул Директор существует на каждом центральном сайте, то записи DNS SRV могут указывать на полное доменное имя пула Директор в каждом из этих трех расположений. Пока доступен пул директоров в одном из расположений, клиент может подключиться к первому прыжку Skype для бизнеса Server.

    > [!NOTE]
    > Использование пула директоров является необязательным. Вместо этого можно использовать пул переднего плана.

2. Пул директоров сообщает клиенту Skype для бизнеса о основном пуле регистратора пользователя и пуле регистраторов резервного копирования.

3. Клиент Skype для бизнеса сначала пытается подключиться к основному пулу регистратора пользователя. Если основной пул регистратора доступен, то регистратор принимает регистрацию. Если основной пул регистратора недоступен, Skype для бизнеса пытается подключиться к резервному пулу регистратора. Если пул регистратора резервных копий доступен и определил, что основной пул регистратора пользователя недоступен (путем обнаружения отсутствия пульса для указанного интервала отработки отказа), пул регистратора резервного копирования принимает регистрацию пользователя. Когда регистратор резервного копирования обнаружит, что основной регистратор снова доступен, пул регистратора резервного копирования перенаправит клиенты отработки отказа в основной пул.

### <a name="requirements-and-recommendations"></a>Требования и рекомендации

Следующие требования и рекомендации для реализации устойчивости голосовой связи на центральном сайте подходят для большинства организаций.

- Сайты, на которых размещаются основной и резервный пулы регистратора, должны быть связаны устойчивой связью WAN.

- На каждом центральном сайте должен быть пул регистратора, состоящий из одного или нескольких регистраторов.

- Каждый пул регистратора должен быть сбалансированный с помощью балансировки нагрузки DNS, аппаратной балансировки нагрузки или и того, и другого. Подробные сведения о планировании конфигурации балансировки нагрузки см. в разделе о требованиях к балансировке нагрузки [для Skype для бизнеса](../../plan-your-deployment/network-requirements/load-balancing.md).

- Каждый пользователь должен быть назначен основному пулу регистраторов с помощью командлета **Skype для бизнеса Server Management Shell set-CsUser** или Skype для бизнеса Server панель управления.

- Основной пул регистратора должен иметь один резервный пул регистратора, расположенный на другом центральном сайте.

- Основной пул регистратора должен быть настроен для отработки отказа в резервный пул регистратора. По умолчанию задана отработка отказа основного пула регистратора в резервный пул через 300 секунд. Этот интервал можно изменить с помощью Skype для бизнеса Server построитель топологий.

- Настройте маршрут отработки отказа. При настройке маршрута укажите шлюз, расположенный на другом сайте, чем шлюз, указанный в основном маршруте.

- Если центральный сайт содержал основной сервер управления и сайт, скорее всего, будет отключен в течение длительного периода времени, вам потребуется переустановить средства управления на сайте резервного копирования. в противном случае вы не сможете изменить параметры управления.

Skype для бизнеса Server зависит от следующих компонентов инфраструктуры и программного обеспечения для обеспечения устойчивости голосовой связи:

|**Компонент** <br/> |**Функциональные** <br/> |
|:-----|:-----|
|DNS  <br/> |Разрешение записей SRV и A для сервера и для возможности подключения клиента к серверу  <br/> |

|Единая система обмена сообщениями Exchange и веб-службы Exchange  <br/> |Журналы вызовов, список голосовой почты, голосовая почта  <br/> |
|Параметры DHCP 120  <br/> |Если запись DNS SRV недоступна, то для обнаружения регистратора клиент будет пытаться использовать параметр DHCP 120. Для этого необходимо настроить DHCP-сервер или включить Skype для бизнеса Server DHCP.  <br/> |

### <a name="survivable-voice-features"></a>Функции обеспечения голосовой связи

Если предыдущие требования и рекомендации реализованы, то резервным пулом регистратора будут предоставляться следующие функции голосовой связи.

- Исходящие вызовы ТСОП.

- Входящие вызовы ТСОП, если поставщик услуг телефонной связи поддерживает возможность отработки отказа на резервный сайт.

- Корпоративные вызовы между пользователями как одного сайта, так и разных сайтов.

- Базовая обработка вызова, включая удержание, возобновление и переключение вызовов.

- Двусторонний обмен мгновенными сообщениями и совместное использование аудио и видео пользователями на одном сайте.

- Услуги переадресации вызовов, одновременных звонков в нескольких конечных точках, делегирования вызовов и групповых звонков, но только если обе стороны делегирования вызова или все члены группы настроены на одном и том же сайте.

- Продолжение функционирования существующих телефонов и клиентов.

- Регистрация вызовов (CDR).

- Проверка подлинности и авторизация.

Когда основной центральный сайт выходит из строя, могут также работать следующие функции голосовой связи (в зависимости от того, как они были настроены).

- Размещение и извлечение голосовых сообщений.

    Если необходимо, чтобы при выходе из строя основного центрального сайта оставалась доступной единая система обмена сообщениями Exchange, то необходимо выполнить одно из следующих действий.

  - Изменить запись DNS SRV таким образом, чтобы серверы единой системы обмена сообщениями Exchange на центральном сайте указывали на резервные серверы единой системы обмена сообщениями Exchange на другом сайте.

  - Настройте абонентской Exchange единой системы обмена сообщениями для каждого пользователя, чтобы включить серверы единой системы обмена сообщениями Exchange как на центральном сайте, так и на сайте резервного копирования, но назначьте резервные копии Exchange единой системы обмена сообщениями как отключенные. Если первичный сайт становится недоступным, Exchange должен пометить серверы единой системы обмена Exchange на сайте резервного копирования как включенные.

    Если ни одно из описанных выше решений не возможно, Exchange единой системы обмена сообщениями не будет доступен в случае, если центральный сайт станет недоступным.

- Конференц-связь всех типов.

    Пользователь, который в результате отработки отказа оказался на резервном сайте, может присоединяться к конференции, созданной или размещенной тем организатором, чей пул доступен, но не может создавать или размещать конференцию в своем основном пуле, который стал недоступен. Аналогичным образом другие пользователи не могут присоединяться к конференциям, размещенным в основном пуле затронутого пользователя.

Следующие функции голосовой связи не будут работать в случае выхода из строя основного центрального сайта.

- Автосекретарь конференции.

- Функция присутствия и маршрутизация на основе DND.

- Обновление параметров переадресации вызовов.

- Служба "Группа ответа" и парковка вызовов.

- Подготовка новых телефонов и клиентов.

- Веб-поиск в адресной книге

## <a name="branch-site-resiliency"></a>Устойчивость сайта филиала

Если вы хотите обеспечить устойчивость сайта филиала, т. е. высокодоступную Корпоративная голосовая связь службу, у вас есть три варианта:

- Устройство для обеспечения связи в филиалах

- сервер для обеспечения связи в филиалах

- Полное развертывание Skype для бизнеса Server на сайте филиала

Это руководство поможет вам определить, какое решение обеспечения устойчивости лучше всего подходит для вашей организации, и в зависимости от принятого решения выбрать вариант ТСОП-связи, подходящий для использования. Руководство также поможет вам лучше подготовиться к развертыванию выбранного решения с учетом предварительных требований и других факторов, которые необходимо принять во внимание при планировании.

### <a name="branch-site-resiliency-features"></a>Функции устойчивости сайта филиала

Если вы обеспечиваете устойчивость сайта филиала, если подключение глобальной сети сайта филиала к центральному сайту завершается сбоем или центральный сайт недоступен, по-прежнему должны быть доступны следующие функции голосовой связи:

- Входящие и исходящие вызовы по телефонной сети общего пользования (ТСОП).

- Корпоративные вызовы между пользователями как одного сайта, так и разных сайтов.

- Базовая обработка вызова, включая удержание, возобновление и переключение вызовов.

- Двусторонний обмен мгновенными сообщениями.

- Переадресация вызовов, одновременный вызов конечных точек, делегирование вызовов и службы командных вызовов, но только в том случае, если делегатор и делегат (например, руководитель и администратор руководителя) или все участники команды настроены на одном сайте.

- Записи регистрации вызовов (CDR).

- Конференц-связь с телефонным подключением из ТСОП с автосекретарем конференц-связи.

- Возможности голосовой почты, если настроены параметры маршрутизации голосовой почты.

- Проверка подлинности и авторизация пользователей.

Следующие функции будут доступны, только если решением для обеспечения устойчивости является полномасштабное развертывание Skype для бизнеса Server на сайте филиала:

- Обмен мгновенными сообщениями, веб-конференции, а также аудио- и видеоконференции.

- Маршрутизация на основании сведений о присутствии и статуса "не беспокоить" (когда вызовы не осуществляются для расширений со статусом "не беспокоить").

- Обновление параметров переадресации вызовов.

- Приложение группы ответа и приложение "Парковка вызовов"

- Подготовка новых телефонов и клиентов, но только доменные службы Active Directory на сайте филиала.

- Enhanced 9-1-1 (E9-1-1).

    Если развернут E9-1-1, а магистраль SIP на центральном сайте недоступна из-за того, что связь с глобальной сетью не работает, устройство обеспечения связи с филиалами направит вызовы E9-1-1 в шлюз локальной ветви. Чтобы включить эту функцию, политики голосовой связи пользователей сайта филиала должны направлять вызовы к локальному шлюзу в случае сбоя глобальной сети.

> [!NOTE]
> SBA (филиал для обеспечения связи) не поддерживается для XMPP. Пользователи, которые размещены в конфигурациях SBA, не смогут отправлять мгновенные сообщения или просматривать сведения о присутствии с контактами XMPP.

### <a name="branch-site-resiliency-solutions"></a>Решения для обеспечения устойчивости сайтов филиалов

Существуют очевидные преимущества обеспечения устойчивости сайта филиала для вашей организации. В частности, если вы потеряете подключение к центральному сайту, пользователи сайта филиала будут продолжать использовать Корпоративная голосовая связь и голосовую почту (если вы настроите параметры перенаправления голосовой почты). Однако для сайтов с менее чем 25 пользователями решение для обеспечения устойчивости может не обеспечить достаточную рентабельность инвестиций.

#### <a name="survivable-branch-appliance-details"></a>Описание механизма обеспечения связи в филиалах

Устройство Skype для бизнеса Server для обеспечения связи филиалов включает в себя следующие компоненты:

- Регистратор для проверки подлинности пользователей, регистрации и маршрутизации вызовов

- Сервер-посредник для обработки сигналов между регистратором и ТСОП-шлюзом

- ТСОП-шлюз для маршрутизации вызовов в ТСОП в случае отказа глобальной сети

- SQL Server Express для локального хранения пользовательских данных

Устройство для обеспечения связи в филиалах также включает магистрали ТСОП, аналоговые порты и адаптер Ethernet.

Если подключение глобальной сети сайта филиала к центральному сайту становится недоступным, пользователи внутренних филиалов по-прежнему регистрируются в регистраторе устройств для обеспечения связи филиалов и получают бесперебойную голосовую службу с помощью подключения к ТСОП. Пользователи сайта филиала, подключающиеся из дома и из других удаленных расположений смогут зарегистрироваться в регистраторе центрального сайта в случае  отсутствия подключения к сайту филиала посредством глобальной сети. Эти пользователи имеют доступ ко всем возможностям единой системы обмена данными за одним исключением — входящие вызовы на сайт филиала будут переадресовываться на голосовую почту. После восстановления подключения по глобальной сети пользователям сайта будут снова предоставлены все функциональные возможности. Для отработки отказа на устройство для обеспечения связи филиалов и восстановления службы не требуется наличие ИТ-администратора.

Skype для бизнеса Server поддерживает до двух устройств для обеспечения связи филиалов на сайте филиала.

### <a name="branch-site-resiliency-requirements"></a>Требования к устойчивости сайта филиала

В этом разделе приведены инструкции по подготовке пользователей к использованию устойчивости сайта филиала и обеспечения связи для голосовой почты, а также приведены требования к оборудованию и программному обеспечению.

#### <a name="preparing-branch-users-for-branch-site-resiliency"></a>Подготовка пользователей филиала для устойчивости сайта филиала

Подготовьте пользователей к устойчивости сайта филиала, настроив для них пул регистраторов в качестве устройства для обеспечения связи филиалов (SBA) или сервера ветвей связи.

#### <a name="registrar-assignments-for-branch-users"></a>Назначения Регистратора для пользователей филиала

Независимо от выбранного решения для обеспечения устойчивости сайта филиала каждому пользователю необходимо назначить основного регистратора. Пользователи сайта филиала всегда должны регистрироваться в регистраторе на сайте филиала независимо от того, находится ли этот регистратор на устройстве для обеспечения связи филиалов, сервере ветвей связи или автономном сервере Skype для бизнеса Server Standard или выпуск Enterprise server. Чтобы клиент мог обнаружить пул Регистратора, требуется DNS-запись SRV. Если устройство для обеспечения связи филиала становится недоступным, клиенты сайта филиала автоматически обнаруживают регистратор резервного копирования.

Если на сайте филиала нет DNS-сервера, существует два альтернативных способа настройки обнаружения устройства для обеспечения связи в филиалах или сервера филиала для обеспечения связи.

- Настройте параметр DHCP 120 на DHCP-сервере DHCP на сайте филиала, чтобы указать полное доменное имя (FQDN) устройства обеспечения связи ветвей или сервера ветвей связи.

- Настройте устройство для обеспечения связи ветвей или сервер ветвей для реагирования на запросы DHCP 120.

#### <a name="voice-routing-for-branch-users"></a>Маршрутизация голосовых вызовов для пользователей филиала

Мы рекомендуем создать отдельную политику голосовой связи на уровне пользователя по протоколу VoIP для пользователей на сайте филиала. Эта политика должна включать основной маршрут, использующий устройство для обеспечения связи филиалов или шлюз сервера филиалов, а также один или несколько маршрутов резервного копирования, использующих магистраль со шлюзом телефонной сети общего пользования (ТСОП) на центральном сайте. Если основной маршрут недоступен, вместо него используется маршрут резервного копирования, использующий один или несколько центральных шлюзов сайта. Таким образом, независимо от того, где зарегистрирован пользователь — в регистраторе сайта филиала или в резервном пуле регистраторов на центральном сайте — политика VoIP пользователя действует всегда. Это важное соображение для сценариев отработки отказа. Например, если необходимо переименовать устройство для обеспечения связи в филиалах или перенастроить устройство для обеспечения связи филиалов для подключения к резервному пулу регистраторов на центральном сайте, необходимо переместить пользователей сайта филиала на центральный сайт на период времени. (Дополнительные сведения о переименовании или перенастройки устройства для обеспечения связи в филиалах см. в приложении [Б.](/previous-versions/office/lync-server-2013/lync-server-2013-appendix-b-managing-a-survivable-branch-appliance) Управление устройством для обеспечения связи в филиалах в документации по развертыванию.) Если у этих пользователей нет политик VoIP на уровне пользователя или абонентской группы на уровне пользователя, то при перемещении пользователей на другой сайт политики VoIP уровня сайта и абонентские группы на уровне сайта центрального сайта применяются к пользователям по умолчанию, а не к политикам VoIP и абонентным планам на уровне сайта филиала. В этом сценарии, если политики VoIP уровня сайта и абонентские группы на уровне сайта, используемые пулом регистратора резервных копий, не могут также применяться к пользователям сайта филиала, их вызовы завершались сбоем. Например, если пользователи с сайта филиала, расположенного в Японии, перемещаются на центральный сайт в Редмонде, то абонентская группа с правилами нормализации, которые добавляют +1425 ко всем 7-значным звонкам, вряд ли будет правильно переводить вызовы для этих пользователей.

> [!IMPORTANT]
> При создании резервного маршрута для филиала мы рекомендуем добавить в пользовательскую политику филиала два режима работы с ТСОП и назначить каждому режиму отдельные маршруты. Первый или основной маршрут будет направлять вызовы к шлюзу, связанному с устройством обеспечения связи филиалов (SBA) или сервером филиала. Второй маршрут (или резервное копирование) будет направлять вызовы к шлюзу на центральном сайте. Перед использованием второго режима работы с ТСОП устройство для обеспечения связи в филиалах или сервер филиала попытаются использовать все маршруты, назначенные первому режиму работы с ТСОП.

Чтобы обеспечить недоступность входящих вызовов для пользователей сайта филиала, когда шлюз филиала или компонент Windows сайта устройства для обеспечения связи филиала недоступен (например, если устройство филиала или шлюз филиала были отключены для обслуживания), создайте маршрут отработки отказа на шлюзе (или обратитесь к поставщику direct Inward Dialing (DID) для перенаправления входящих вызовов в пул регистратора резервных копий на центральном сайте. Звонки будут перенаправляться пользователям филиала через глобальную сеть. Убедитесь, что маршрут переводит номера в соответствии со шлюзом ТСОП или принятыми форматами номеров телефонов другого однорангового узла магистрали. Дополнительные сведения о создании маршрута отработки отказа см. в разделе [Configuring a Failover Route](/previous-versions/office/lync-server-2013/lync-server-2013-configuring-a-failover-route). Кроме того, для применения правил нормализации к входящим звонкам создайте абонентские группы на уровне службы для магистрали, связанной со шлюзом на сайте филиала. Если на сайте филиала есть два устройства для обеспечения связи филиалов, можно создать абонентский план уровня сайта для обоих устройств, если для каждого из них не требуется отдельный план уровня обслуживания.

> [!NOTE]
> Для учета ресурсов центрального сайта, потребляемых пользователями сайта филиала, которые используют центральный сайт для получения сведений о присутствии, конференц-связи или отработки отказа, мы рекомендуем учитывать каждого пользователя сайта филиала как пользователя, зарегистрированного на центральном сайте. В настоящее время нет ограничений на количество пользователей сайта филиала, включая пользователей, зарегистрированных на устройстве для обеспечения связи филиалов.

Кроме того, мы рекомендуем создать абонентскую группу и политику голосовой связи на уровне пользователя и затем назначить их пользователям сайта филиала. Дополнительные сведения см[](../../deploy/deploy-enterprise-voice/dial-plans.md). в разделах "Создание или изменение абонентской группы" Skype для бизнеса Server и "Создание политики маршрутизации [VoIP](/previous-versions/office/lync-server-2013/lync-server-2013-create-the-voip-routing-policy-for-branch-users) для пользователей филиалов" в документации по развертыванию.

#### <a name="routing-extension-numbers"></a>Маршрутизация добавочных номеров

При подготовке абонентской группы и политик голосовой связи для пользователей сайта филиала обязательно включите правила нормализации и правила преобразования, соответствующие строкам и формату номеров, используемым в атрибуте msRTCSIP-line (или URI строки), чтобы Skype для бизнеса звонки, включенные между пользователями сайта филиала и пользователями центрального сайта, перенаправлялись правильно, особенно если вызовы должны перенаправляться по ТСОП, так как ссылка на глобальную сеть недоступна. В отличие от обычных номеров для номеров с добавочными номерами существуют особые требования.

Правила нормализации и преобразования, которые соответствуют Line URI, содержащим добавочный номер (отдельный номер или в дополнении к полному номеру телефона E.164), имеют дополнительные требования. В этом разделе описываются примеры маршрутизации звонков для Line URI с добавочными номерами.

Если в вашей организации не настроен прямой набор внутренних номеров для отдельных пользователей и в качестве URI линии для всех пользователей указан только добавочный номер, внутренние пользователи могут звонить друг другу, используя только добавочный номер. Однако вам потребуется настроить правила нормализации, которые можно применить при звонках пользователя сайта филиала пользователю центрального сайта и которые соответствуют добавочным номерам.

Если канал глобальной сети между сайтом филиала и центральным сайтом доступен, то для звонков пользователей сайта филиала пользователям центрального сайта не требуется правило нормализации для преобразования номеров, поскольку звонки не маршрутизируются через ТСОП. Пример:

|**Название правила**|**Описание**|**Шаблон номера**|**Преобразование**|**Пример**|
|:-----|:-----|:-----|:-----|:-----|
|5digitExtensions  <br/> |Не преобразует номера, состоящие из 5 знаков  <br/> |^(\d{5})$  <br/> |$1  <br/> |10001 не преобразуется  <br/> |

Вам также необходимо адаптировать добавочные номера для случаев, когда канал глобальной сети между сайтом филиала и центральным сайтом недоступен, а звонок из сайта филиала необходимо перенаправить через ТСОП. Во время сбоя глобальной сети, если пользователь сайта филиала вызывает центрального пользователя сайта только путем набора расширения пользователя центрального сайта, необходимо иметь правило преобразования исходящего трафика, которое добавляет полный номер телефона центрального пользователя сайта. Если URI строки пользователя содержит полный номер телефона вашей организации и уникальный номер расширения вместо полного номера телефона, уникального для пользователя, необходимо иметь правило преобразования исходящего трафика, которое добавляет полный номер телефона вашей организации. Например:

|**Описание**|**Шаблон сопоставления**|**Преобразование**|**Пример**|
|:-----|:-----|:-----|:-----|
|Перевод 5-значных номеров в номер телефона и расширение пользователя  <br/> |^(\d{5})$  <br/> |+14255550123;ext=$1  <br/> |10001 преобразуется в +14255550123;доб.=10001  <br/> |
|Перевод 5-значных номеров в номер телефона вашей организации и расширение пользователя  <br/> |^(\d{5})$  <br/> |+14255550100;ext=$1  <br/> |10001 преобразуется в +14255550100;доб.=10001  <br/> |

В этом сценарии, если узел магистрали, который обрабатывает изменение маршрута в ТСОП, не поддерживает добавочные номера, то правило преобразования исходящих звонков также должно удалять добавочные номера. Пример:

|**Описание**|**Шаблон сопоставления**|**Преобразование**|**Пример**|
|:-----|:-----|:-----|:-----|
|Удаляет добавочный номер из номеров телефонов, содержащих добавочные номера  <br/> |^\+(\d\*); ext=(\d\*)$  <br/> |+$1  <br/> |+14255550123;доб.=10001 преобразуется в +14255550123  <br/> |

Независимо от того, доступна ли ссылка глобальной сети, если в вашей организации не настроены номера DID для отдельных пользователей и URI строки для пользователя содержит номер телефона вашей организации и уникальный номер расширения пользователя, необходимо настроить URI строки номера телефона организации с номером, доступным для узла магистрали или шлюза ТСОП на сайте филиала. Кроме того, необходимо настроить универсальный код ресурса (URI) строки номера телефона организации, чтобы включить собственное уникальное расширение для маршрутизации звонков на этот номер.

#### <a name="preparing-for-voice-mail-survivability"></a>Подготовка к обеспечению связи для голосовой почты

Exchange единой системы обмена сообщениями (UM) обычно устанавливается только на центральном сайте, а не на сайтах филиалов. Даже если канал глобальной сети между сайтом филиала и центральным сайтом недоступен, у вызывающего абонента должна быть возможность отправки сообщения голосовой почты. В результате настройка URI строки для номера телефона автосекретаря единой системы обмена сообщениями Exchange, который предоставляет голосовую почту для пользователей сайта филиала, требует особого рассмотрения, помимо политики голосовой связи, абонентской группы и правил нормализации, применимых к номеру голосовой почты.

Устройства для обеспечения связи в филиалах (SBAs) и серверы филиалов для обеспечения связи голосовой почты для пользователей филиалов во время сбоя глобальной сети. В частности, если вы используете устройство для обеспечения связи ветвей или сервер ветвей связи и глобальная сеть становится недоступной, сервер SBA или сервер ветвей связи перенаправит неотвеченные вызовы через ТСОП в Exchange единой системы обмена сообщениями на центральном сайте. С помощью SBA или сервера филиала для обеспечения связи пользователи также могут получать сообщения голосовой почты через ТСОП во время сбоя глобальной сети. Наконец, во время сбоя глобальной сети устройство для обеспечения связи филиала или сервер филиалов очереди уведомлений о пропущенных вызовах, а затем отправляет их на сервер единой системы обмена сообщениями Exchange при восстановлении глобальной сети. Чтобы обеспечить отказоустойчивое перенаправление голосовой почты, добавьте запись для полного доменного имени центрального пула сайтов и запись для полного доменного имени пограничного сервера в файл hosts на сервере survivable Branch Server. В противном случае при отсутствии DNS-сервера на сайте филиала разрешение DNS-имен не будет работать.

При настройке обеспечения связи для голосовой почты для пользователей сайта филиала необходимо учитывать следующие рекомендации.

- Администратор Exchange Майкрософт должен настроить Exchange единой системы обмена сообщениями (AA) для принятия только сообщений. Эта конфигурация отключает все остальные функциональные возможности, например передачу пользователю, передачу оператору, и позволяет автосекретарю только принимать сообщения. В качестве альтернативного варианта администратор Exchange может использовать общий автосекретарь или настроенный секретарь для маршрутизации звонков оператору.

- Администратор Skype для бизнеса Server должен взять номер телефона AA и использовать его в качестве номера автосекретаря **обмена** сообщениями в параметрах перенаправления голосовой почты для устройства обеспечения связи филиала или сервера филиала.

- Администратор Skype для бизнеса Server должен получить номер телефона для доступа к подписчику единой системы обмена сообщениями Exchange и использовать его в качестве  номера доступа подписчика в параметрах перенаправления голосовой почты для устройства обеспечения связи филиала или сервера филиала.

- Администратор Skype для бизнеса Server должен настроить Exchange единой системы обмена сообщениями, чтобы со всеми пользователями филиала, которым требуется доступ к голосовой почте во время сбоя глобальной сети, была связана только одна абонентской группы.

- Если ссылка глобальной сети недоступна, вызовы пользователей сайта филиала могут быть перенаправлены в почтовый ящик единой системы обмена сообщениями Exchange пользователя, но только в том случае, если политика голосовой связи, примененная к вызову, указывает уникальный номер телефона голосовой почты и не включает номер расширения.

#### <a name="hardware-and-software-requirements-for-branch-site-resiliency"></a>Требования к оборудованию и программному обеспечению для устойчивости сайта филиала

Требования к оборудованию и программному обеспечению зависят от решения, используемого для обеспечения устойчивости.

#### <a name="requirements-for-survivable-branch-appliances"></a>Требования к устройствам для обеспечения связи в филиалах

Необходимое оборудование и программное обеспечение встроены в устройство для обеспечения связи в филиалах. Однако мы также рекомендуем каждому сайту филиала развернуть DHCP-сервер для получения IP-адресов клиента. в противном случае по истечении срока аренды DHCP у клиентов не будет IP-подключения.

Если корпоративные DNS-серверы находятся только на центральных сайтах, пользователи сайта филиала не смогут подключиться к ним во время сбоя глобальной сети, поэтому обнаружение Skype для бизнеса Server, использующее запись ресурса DNS SRV (служба (SRV), завершится сбоем. Для маршрутизации в условиях отсутствия доступа к глобальной сети требуется кэшировать DNS-записи на сайте филиала. Если маршрутизатор филиала поддерживает кэширование DNS, включите его. В противном случае вы можете развернуть DNS-сервер в филиале. Это может быть автономный сервер или версия устройства для обеспечения связи ветвей, поддерживающих возможности DNS. Чтобы просмотреть дополнительные сведения, обратитесь к поставщику устройства для обеспечения связи филиалов.

> [!NOTE]
> Контроллер домена на сайте филиала не требуется. Устройство обеспечения связи ветвей выполняет проверку подлинности клиентов с помощью специального сертификата, который отправляет клиенту в ответ на запрос сертификата клиента при входе.

Skype для бизнеса клиенты могут обнаружить Skype для бизнеса Server с помощью параметра DHCP 120 (параметр регистратора SIP). Этот запрос можно настроить следующими способами:

- Настройте DHCP-сервер на сайте филиала для ответа на запросы DHCP 120, которые возвращают полное доменное имя регистратора на устройстве для обеспечения связи филиалов или на сервере ветвей связи.

- Включите Skype для бизнеса Server DHCP. Если этот параметр включен, Skype для бизнеса Server регистратор отвечает на запросы DHCP параметра 120. Обратите внимание, что Регистратор не отвечает на все DHCP-запросы, кроме "DHCP Options 120".

Кроме того, для крупных сайтов филиалов, имеющих несколько подсетей, нужно включить агенты DHCP-ретрансляции для переадресации запросов "DHCP Option 120" на DHCP-сервер (конфигурация 1) или Регистратор (конфигурация 2).

Наконец, пользователи сайта филиала должны быть настроены для Корпоративная голосовая связь и подготовлены с помощью соответствующей унифицированной конечной точки связи.

 В следующем примере показано, как администратор может определить маршрут отработки отказа на тот случай, если шлюз Dallas-GW1 отключается для проведения технического обслуживания или недоступен по иным причинам. В таблице ниже показаны изменения, которые нужно внести в конфигурацию.

|Политика звонков по умолчанию  <br/> |Локальный  <br/> GlobalPSTNHopoff  <br/> |
|Локальная политика в Редмонде  <br/> |RedmondLocal  <br/> |
|Политика звонков в Далласе  <br/> |DallasUsers  <br/> GlobalPSTNHopoff  <br/> |
