# <a name="plan-for-enterprise-voice-resiliency-in-skype-for-business-server"></a>Планирование устойчивости Корпоративная голосовая связь в Skype для бизнеса Server

Узнайте, как поддерживать устойчивость голосовой почты в Skype для бизнеса Server Корпоративная голосовая связь как на центральных сайтах, так и на сайтах филиалов. Варианты сайта филиала включают развертывание устройств для survivable Branch Appliances или серверов для survivable Branch Server.

Устойчивость голосовой связи означает возможность пользователей продолжать выполнять и принимать вызовы, если центральный сайт, на котором размещен Skype для бизнеса Server, становится недоступным при сбое в сети WAN или по другой причине. При сбое центрального узла служба Enterprise Voice должна без прерываний перейти на резервный узел. При сбое глобальной сети вызовы узла филиала должны перенаправляться на локальный шлюз ТСОП. В данном разделе описывается планирование устойчивости голосовой связи при сбое центрального узла или глобальной сети.

## <a name="central-site-resiliency"></a>Устойчивость центрального сайта

Все чаще у предприятий появляется много сайтов, распределенных по всему миру. Обслуживание экстренных служб, доступ к службе поддержки и возможность выполнения важных бизнес-задач в случае, когда центральный сайт выходит из службы, крайне важна для любого решения Корпоративная голосовая связь устойчивости. Когда центральный сайт выходит из строя, должны выполняться следующие условия.

- Должна обеспечиваться отработка отказа голосовой связи.

- Пользователи, которые обычно регистрируются в пуле переднего входа на центральном сайте, должны иметь возможность зарегистрироваться в альтернативном пуле переднего входа. Для этого можно создать несколько DNS-записей SRV, каждая из которых разрешатся в пул директоров или пул переднего уровня на каждом центральном сайте. Вы можете настроить приоритет и вес записей SRV, чтобы пользователи, обслуживаемые этим центральным сайтом, могли получать соответствующий директор и пул переднего входа, а не те, которые находятся в других записях SRV.

- Входящие и исходящие вызовы пользователей, расположенных на других сайтах, должны перенаправляться в ТСОП.

В этой статье описывается рекомендуемое решение для защиты устойчивости центрального сайта голосовой связи.

### <a name="architecture-and-topology"></a>Архитектура и топология

Планирование устойчивости голосовой почты на центральном сайте требует базового понимания центральной роли, которую играет регистратор Skype для бизнеса Server для включения отключаемой голосовой функции. Регистратор Skype для бизнеса Server — это служба, которая обеспечивает регистрацию и проверку подлинности клиентов, а также предоставляет службы маршрутизации. Он выполняется на всех серверах Standard Edition, серверах переднего сервера, директорах или устройствах для survivable Branch Appliance. Пул регистратора состоит из служб регистратора, работающих в пуле переднего входа и находящихся на том же сайте. Клиент Skype для бизнеса обнаруживает пул переднего клиента с помощью следующего механизма обнаружения:

1. DNS-запись SRV

2. Веб-служба автооружия

3. Параметр DHCP 120

После подключения клиента Skype для бизнеса к пулу переднего сервера он направляется под балансой нагрузки на один из серверов переднего сервера в пуле. Этот сервер переднего сервера, в свою очередь, перенаправляет клиента на предпочтительный регистратор в пуле.

Каждый пользователь, включенный Корпоративная голосовая связь, назначен определенному пулу регистратора, который становится основным пулом регистратора этого пользователя. На конкретном сайте сотни или тысячи пользователей обычно совместно используют один основной пул регистратора. Учитывая потребление ресурсов центрального сайта всеми пользователями сайтов филиалов, функции присутствия, конференц-связи или отработки отказа которых зависят от центрального сайта, мы рекомендуем рассматривать каждого пользователя сайта филиала, как если бы он был зарегистрирован на центральном сайте. В настоящее время не существует ограничений на количество пользователей сайта филиала, включая пользователей, зарегистрированных на устройстве для связи в филиалах.

Для обеспечения устойчивости голосовой связи в случае сбоя центрального сайта основной пул регистратора должен иметь один назначенный резервный пул регистратора, расположенный на другом сайте. Резервную копию можно настроить с помощью параметров устойчивости построитель топологий. Если существует устойчивая связь WAN между этими двумя сайтами, то пользователи, чей основной пул регистратора стал недоступен, автоматически перенаправляются в резервный пул регистратора.

Следующие действия описывают процесс обнаружения и регистрации клиента.

1. Клиент обнаруживает Skype для бизнеса Server с помощью записей DNS SRV. В Skype для бизнеса Server записи DNS SRV можно настроить так, чтобы они возвращали несколько FQDN запросу DNS SRV. Например, если предприятие Contoso имеет три центральных сайта (в Северной Америке, в Европе и в Азиатско-Тихоокеанском регионе), и пул Директор существует на каждом центральном сайте, то записи DNS SRV могут указывать на полное доменное имя пула Директор в каждом из этих трех расположений. Пока доступен пул директоров в одном из местоположений, клиент может подключиться к первому прыжку Skype для бизнеса Server.

    > [!NOTE]
    > Использование пула директоров является необязательным. Вместо этого можно использовать пул переднего входа.

2. Пул директоров сообщает клиенту Skype для бизнеса о основном пуле регистратора и резервном пуле регистратора пользователя.

3. Клиент Skype для бизнеса сначала пытается подключиться к основному пулу регистратора пользователя. Если основной пул регистратора доступен, то регистратор принимает регистрацию. Если основной пул регистратора недоступен, клиент Skype для бизнеса пытается подключиться к резервному пулу регистратора. Если резервный пул регистратора доступен и определил, что основной пул регистратора пользователя недоступен (обнаруживая отсутствие пульса в течение указанного интервала от сбойа), резервный пул регистратора принимает регистрацию пользователя. После того как резервный регистратор обнаружит, что основной регистратор снова доступен, резервный пул регистратора перенаправит клиенты от сбойной передачи в основной пул.

### <a name="requirements-and-recommendations"></a>Требования и рекомендации

Следующие требования и рекомендации для реализации устойчивости голосовой связи на центральном сайте подходят для большинства организаций.

- Сайты, на которых размещаются основной и резервный пулы регистратора, должны быть связаны устойчивой связью WAN.

- На каждом центральном сайте должен быть пул регистратора, состоящий из одного или нескольких регистраторов.

- Каждый пул регистратора должен иметь балансировку нагрузки с помощью балансировки нагрузки на DNS, аппаратной балансировки нагрузки или и того, и другого. Подробные сведения о планировании конфигурации балансировки нагрузки см. в описании требований балансировки нагрузки [для Skype для бизнеса.](../../plan-your-deployment/network-requirements/load-balancing.md)

- Каждый пользователь должен быть назначен основному пулу регистратора с помощью либо с помощью команды **Set-CsUser,** либо с помощью панели управления Skype для бизнеса Server.

- Основной пул регистратора должен иметь один резервный пул регистратора, расположенный на другом центральном сайте.

- Основной пул регистратора должен быть настроен для отработки отказа в резервный пул регистратора. По умолчанию задана отработка отказа основного пула регистратора в резервный пул через 300 секунд. Этот интервал можно изменить с помощью построщика топологий Skype для бизнеса Server.

- Настройте маршрут от сбойной передачи. При настройке маршрута укажите шлюз, расположенный на другом сайте, чем шлюз, указанный в основном маршруте.

- Если центральный сайт содержал основной сервер управления и сайт, скорее всего, не будет работать в течение длительного периода времени, потребуется переустановить средства управления на резервном сайте; в противном случае вы не сможете изменить параметры управления.

### <a name="survivable-voice-features"></a>Функции обеспечения голосовой связи

Если предыдущие требования и рекомендации реализованы, то резервным пулом регистратора будут предоставляться следующие функции голосовой связи.

- Исходящие вызовы ТСОП.

- Входящие вызовы ТСОП, если поставщик услуг телефонной связи поддерживает возможность отработки отказа на резервный сайт.

- Корпоративные вызовы между пользователями как одного сайта, так и разных сайтов.

- Базовая обработка вызова, включая удержание, возобновление и переключение вызовов.

- Двусторонний обмен мгновенными сообщениями и совместное использование аудио и видео пользователями на одном сайте.

- Услуги переадресации вызовов, одновременных звонков в нескольких конечных точках, делегирования вызовов и групповых звонков, но только если обе стороны делегирования вызова или все члены группы настроены на одном и том же сайте.

- Продолжение функционирования существующих телефонов и клиентов.

- Регистрация вызовов (CDR).

- Проверка подлинности и авторизация.

Когда основной центральный сайт выходит из строя, могут также работать следующие функции голосовой связи (в зависимости от того, как они были настроены).

- Размещение и извлечение голосовых сообщений.

    Если необходимо, чтобы при выходе из строя основного центрального сайта оставалась доступной единая система обмена сообщениями Exchange, то необходимо выполнить одно из следующих действий.

  - Изменить запись DNS SRV таким образом, чтобы серверы единой системы обмена сообщениями Exchange на центральном сайте указывали на резервные серверы единой системы обмена сообщениями Exchange на другом сайте.

  - Настройте для каждой пользовательской телефонной плана exchange um, чтобы включить серверы exchange UM как на центральном сайте, так и на резервном сайте, но обозначите резервные серверы этой системы как отключенные. Если основной сайт становится недоступным, администратор Exchange должен пометить серверы exchange UM на сайте резервного копирования как включенные.

    Если ни одно из предыдущих решений не возможно, то в случае недоступности центрального сайта будет недоступен обмен данными Exchange.

- Конференц-связь всех типов.

    Пользователь, который в результате отработки отказа оказался на резервном сайте, может присоединяться к конференции, созданной или размещенной тем организатором, чей пул доступен, но не может создавать или размещать конференцию в своем основном пуле, который стал недоступен. Аналогично, другие пользователи не могут присоединяться к конференциям, которые находятся в основном пуле затронутых пользователей.

Следующие функции голосовой связи не будут работать в случае выхода из строя основного центрального сайта.

- Автосекретарь конференции.

- Функция присутствия и маршрутизация на основе DND.

- Обновление параметров переадресации вызовов.

- Служба "Группа ответа" и парковка вызовов.

- Подготовка новых телефонов и клиентов.

- Веб-поиск в адресной книге

## <a name="branch-site-resiliency"></a>Устойчивость сайта филиала

Если вы хотите обеспечить устойчивость сайта филиала, то есть службу Корпоративная голосовая связь высокой доступности, вы можете сделать это тремя вариантами:

- Устройство для обеспечения связи в филиалах

- сервер для обеспечения связи в филиалах

- Полное развертывание Skype для бизнеса Server на сайте филиала

Это руководство поможет вам определить, какое решение обеспечения устойчивости лучше всего подходит для вашей организации, и в зависимости от принятого решения выбрать вариант ТСОП-связи, подходящий для использования. Руководство также поможет вам лучше подготовиться к развертыванию выбранного решения с учетом предварительных требований и других факторов, которые необходимо принять во внимание при планировании.

### <a name="branch-site-resiliency-features"></a>Функции устойчивости сайта филиала

Если обеспечивается устойчивость сайта филиала, при сбойе подключения WAN сайта филиала к центральному сайту или если центральный сайт недостижим, по-прежнему должны быть доступны следующие функции голосовой связи:

- Входящие и исходящие вызовы по телефонной сети общего пользования (ТСОП).

- Корпоративные вызовы между пользователями как одного сайта, так и разных сайтов.

- Базовая обработка вызова, включая удержание, возобновление и переключение вызовов.

- Двусторонний обмен мгновенными сообщениями.

- Переадправление вызовов, одновременные вызовы конечных точек, делегирование вызовов и службы вызовов группы, но только в том случае, если делегатор и делегат (например, руководитель и администратор руководителя) или все участники группы настроены на одном сайте

- Записи регистрации вызовов (CDR).

- Конференц-связь с телефонным подключением из ТСОП с автосекретарем конференц-связи.

- Возможности голосовой почты, если настроены параметры маршрутизации голосовой почты.

- Проверка подлинности и авторизация пользователей.

Следующие функции будут доступны только в том случае, если решение для обеспечения устойчивости является полным развертыванием Skype для бизнеса Server на сайте филиала:

- Обмен мгновенными сообщениями, веб-конференции, а также аудио- и видеоконференции.

- Маршрутизация на основании сведений о присутствии и статуса "не беспокоить" (когда вызовы не осуществляются для расширений со статусом "не беспокоить").

- Обновление параметров переадресации вызовов.

- Приложение группы ответа и приложение парковки вызовов

- Подготовка новых телефонов и клиентов, но только в том случае, если доменные службы Active Directory присутствуют на сайте филиала.

- Enhanced 9-1-1 (E9-1-1).

    Если развернут E9-1-1, а магистраль SIP на центральном сайте недоступна, так как связь WAN не работает, устройство для связи в филиалах будет перенаправживать вызовы E9-1-1 на локальный шлюз филиала. Чтобы включить эту функцию, политики голосовой связи пользователей сайта филиала должны маршрутизовыть вызовы на локальный шлюз в случае сбоя WAN.

> [!NOTE]
> SBA (survivable branch office) не поддерживается для XMPP. Пользователи, которые были в конфигурациях SBA, не смогут отправлять IMs или видеть присутствие с контактами XMPP.

### <a name="branch-site-resiliency-solutions"></a>Решения для обеспечения устойчивости сайта филиала

Существуют очевидные преимущества обеспечения устойчивости сайта филиала для организации. В частности, если вы теряете подключение к центральному сайту, пользователи сайта филиала по-прежнему будут иметь Корпоративная голосовая связь и голосовую почту (при настройке параметров перенастройки голосовой почты). Однако для сайтов с менее чем 25 пользователями решение для обеспечения устойчивости может не обеспечить достаточной прибыли от инвестиций.

Доступно три варианта обеспечения устойчивости сайта филиала. Воспользуйтесь следующей таблицей, чтобы определить оптимальный для себя вариант.

#### <a name="survivable-branch-appliance-details"></a>Описание механизма обеспечения связи в филиалах

После восстановления подключения по глобальной сети пользователям сайта будут снова предоставлены все функциональные возможности. Для отсбора до устройства для survivable Branch Appliance и восстановления службы не требуется присутствие ИТ-администратора.

Skype для бизнеса Server поддерживает до двух устройств для обеспечения survivable Branch Appliance на сайте филиала.

### <a name="branch-site-resiliency-requirements"></a>Требования к устойчивости сайта филиала

В этом разделе приведены инструкции по подготовке пользователей к использованию устойчивости сайта филиала и обеспечения связи для голосовой почты, а также приведены требования к оборудованию и программному обеспечению.

#### <a name="preparing-branch-users-for-branch-site-resiliency"></a>Подготовка пользователей филиала для устойчивости сайта филиала

Подготовьте пользователей к устойчивости на сайте филиала, задав их пул регистратора в качестве устройства для обеспечения безопасности в филиалах (SBA) или сервера survivable Branch Server.

#### <a name="registrar-assignments-for-branch-users"></a>Назначения Регистратора для пользователей филиала

Независимо от того, какое решение для обеспечения устойчивости сайта филиала вы выберете, каждому пользователю необходимо назначить основного регистратора. Пользователи сайта филиала всегда должны регистрироваться в регистраторе на сайте филиала независимо от того, находится ли этот регистратор на устройстве для survivable Branch Appliance, survivable Branch Server или отдельном сервере Skype для бизнеса Server Standard или Enterprise Edition. Чтобы клиент мог обнаружить пул Регистратора, требуется DNS-запись SRV. Если устройство для связи в филиалах становится недоступным, клиенты сайта филиала автоматически обнаруживают резервный регистратор.

Если на сайте филиала нет DNS-сервера, существует два альтернативных способа настройки обнаружения устройства или сервера для survivable Branch Server:

- Настройте параметр DHCP 120 на DHCP-сервере сайта филиала так, чтобы он укаживал полное доменное имя устройства или сервера для связи в филиалах.

- Настройте устройство или сервер для survivable Branch Server для ответа на запросы DHCP 120.

#### <a name="voice-routing-for-branch-users"></a>Маршрутизация голосовых вызовов для пользователей филиала

Рекомендуется создать отдельную политику голосовой связи по протоколу VoIP для пользователей на сайте филиала. Эта политика должна включать основной маршрут, использующий устройство для связи в филиалах или шлюз сервера филиалов, а также один или несколько маршрутов резервного копирования, которые используют магистраль со шлюзом телефонной сети общего пользования (PSTN) на центральном сайте. Если основной маршрут недоступен, используется маршрут резервного копирования, использующий один или несколько шлюзов центрального сайта. Таким образом, независимо от того, где зарегистрирован пользователь — в регистраторе сайта филиала или в резервном пуле регистратора на центральном сайте — политика VoIP пользователя всегда действует. Это важный аспект для сценариев от сбойов. Например, если необходимо переименовать устройство для survivable Branch Appliance или перенастроить устройство для связи в филиалах для подключения к резервному пулу регистратора на центральном сайте, необходимо на время переместить пользователей сайта филиала на центральный сайт. (Подробные сведения о переименовании или перенастройки устройства для survivable Branch Appliance см. в приложении Б. Управление устройством для [survivable Branch Appliance](https://technet.microsoft.com/library/2ec9d505-6d39-491c-9524-8cf36866b855.aspx) в документации по развертыванию.) Если у этих пользователей нет политик VoIP или телефонных планов на уровне пользователя, то при перемещении пользователей на другой сайт политики VoIP и наборные номера уровня сайта центрального сайта применяются к пользователям по умолчанию, а не к политикам VoIP и наборам на уровне сайта филиала. В этом сценарии, если политики VoIP на уровне сайта и телефонные планы уровня сайта, используемые резервным пулом регистратора, не могут также применяться к пользователям сайта филиала, их вызовы не будут действовать. Например, если пользователи с сайта филиала, расположенного в Японии, перемещаются на центральный сайт в Редмонде, то телефонные планы с правилами нормализации, которые предукрасили +1425 для всех 7-значных вызовов, вряд ли будет правильно переводить вызовы для этих пользователей.

> [!IMPORTANT]
> При создании резервного маршрута для филиала мы рекомендуем добавить в пользовательскую политику филиала два режима работы с ТСОП и назначить каждому режиму отдельные маршруты. Первый (основной) маршрут направляет вызовы на шлюз, связанный с устройством для связи в филиалах (SBA) или сервером филиала; Второй маршрут (или резервное копирование) направляет вызовы шлюзу на центральном сайте. Перед использованием второго режима работы с ТСОП устройство для обеспечения связи в филиалах или сервер филиала попытаются использовать все маршруты, назначенные первому режиму работы с ТСОП.

Чтобы убедиться, что входящие вызовы пользователей сайта филиала будут добираться до этих пользователей, когда шлюз филиала или компонент Windows сайта устройства для обеспечения связи в филиалах недоступен (например, если устройство или шлюз филиала были недоступны для обслуживания), создайте маршрут отработки сбой на шлюзе (или работайте с поставщиком прямого входящих наборов (DID) для перенаправления входящих вызовов в резервный пул регистратора на центральном сайте. Звонки будут перенаправляться пользователям филиала через глобальную сеть. Убедитесь, что маршрут преобразует номера в соответствии со шлюзом STN или другими форматами принятых телефонных номеров магистрали. Дополнительные сведения о создании маршрута отработки отказа см. в разделе [Configuring a Failover Route](https://technet.microsoft.com/library/76e48df4-3b78-4fb7-b1f7-c1e604b81bad.aspx). Кроме того, для применения правил нормализации к входящим звонкам создайте абонентские группы на уровне службы для магистрали, связанной со шлюзом на сайте филиала. Если на сайте филиала имеется два устройства для survivable Branch Appliances, можно создать для обоих них план на уровне сайта, если для каждого из них не требуется отдельная план уровня обслуживания.

> [!NOTE]
> Для учета ресурсов центрального сайта, потребляемых пользователями сайта филиала, которые используют центральный сайт для получения сведений о присутствии, конференц-связи или отработки отказа, мы рекомендуем учитывать каждого пользователя сайта филиала как пользователя, зарегистрированного на центральном сайте. В настоящее время не существует ограничений на количество пользователей сайта филиала, включая пользователей, зарегистрированных на устройстве для связи в филиалах.

Кроме того, мы рекомендуем создать абонентскую группу и политику голосовой связи на уровне пользователя и затем назначить их пользователям сайта филиала. For details, [see Create or modify a dial plan in Skype for Business Server](../../deploy/deploy-enterprise-voice/dial-plans.md) and Create the [VoIP Routing Policy for Branch Users](https://technet.microsoft.com/library/10deca9f-f870-4a42-b25d-e4fc53108658.aspx) in the Deployment documentation.

#### <a name="routing-extension-numbers"></a>Маршрутизация добавочных номеров

При подготовке телефонных планов и политик голосовой связи для пользователей сайта филиала обязательно включите правила нормализации и правила трансляции, которые соответствуют строкам и формату номеров, используемым в атрибуте msRTCSIP-line (или Line URI), чтобы вызовы Skype для бизнеса, включенные между пользователями сайта филиала и пользователями центрального сайта, перенапрались правильно, особенно если вызовы необходимо перенаправлять через ТСТС, так как связь WAN недоступна. В отличие от обычных номеров для номеров с добавочными номерами существуют особые требования.

Правила нормализации и преобразования, которые соответствуют Line URI, содержащим добавочный номер (отдельный номер или в дополнении к полному номеру телефона E.164), имеют дополнительные требования. В этом разделе описываются примеры маршрутизации звонков для Line URI с добавочными номерами.

Если в вашей организации не настроен прямой набор внутренних номеров для отдельных пользователей и в качестве URI линии для всех пользователей указан только добавочный номер, внутренние пользователи могут звонить друг другу, используя только добавочный номер. Однако вам потребуется настроить правила нормализации, которые можно применить при звонках пользователя сайта филиала пользователю центрального сайта и которые соответствуют добавочным номерам.

Если канал глобальной сети между сайтом филиала и центральным сайтом доступен, то для звонков пользователей сайта филиала пользователям центрального сайта не требуется правило нормализации для преобразования номеров, поскольку звонки не маршрутизируются через ТСОП. Пример:

|**Название правила**|**Описание**|**Шаблон номера**|**Преобразование**|**Пример**|
|:-----|:-----|:-----|:-----|:-----|
|5digitExtensions  <br/> |Не преобразует номера, состоящие из 5 знаков  <br/> |^(\d {5} )$  <br/> |$1  <br/> |10001 не преобразуется  <br/> |

Вам также необходимо адаптировать добавочные номера для случаев, когда канал глобальной сети между сайтом филиала и центральным сайтом недоступен, а звонок из сайта филиала необходимо перенаправить через ТСОП. Во время отключения WAN-сети, если пользователь сайта филиала вызывает пользователя центрального сайта только путем набора добавок пользователя центрального сайта, необходимо правило трансляции исходящие вызовы, которое добавляет полный номер телефона пользователя центрального сайта. Если URI линии пользователя содержит полный телефонный номер организации и уникальный добавок вместо полного уникального для пользователя номера телефона, то необходимо правило трансляции исходящие звонки, которое добавляет полный номер телефона организации. Пример:

|**Описание**|**Шаблон совпадения**|**Преобразование**|**Пример**|
|:-----|:-----|:-----|:-----|
|Преобразует 5-значные номера в номер телефона пользователя и его расширение  <br/> |^(\d {5} )$  <br/> |+14255550123;ext=$1  <br/> |10001 преобразуется в +14255550123;доб.=10001  <br/> |
|Преобразует 5-значные номера в номер телефона организации и расширение пользователя  <br/> |^(\d {5} )$  <br/> |+14255550100;ext=$1  <br/> |10001 преобразуется в +14255550100;доб.=10001  <br/> |

В этом сценарии, если узел магистрали, который обрабатывает изменение маршрута в ТСОП, не поддерживает добавочные номера, то правило преобразования исходящих звонков также должно удалять добавочные номера. Пример:

|**Описание**|**Шаблон совпадения**|**Преобразование**|**Пример**|
|:-----|:-----|:-----|:-----|
|Удаляет добавочный номер из номеров телефонов, содержащих добавочные номера  <br/> |^\+\*(\d); ext=(\d \* )$  <br/> |+$1  <br/> |+14255550123;доб.=10001 преобразуется в +14255550123  <br/> |

Независимо от того, доступна ли WAN-связь, если в организации не настроены номера DID для отдельных пользователей, а URI линии для пользователя содержит номер телефона вашей организации и уникальный его номер, необходимо настроить номер телефона вашей организации с номером line URI с номером, доступным для магистрали или шлюза STN на сайте филиала. Кроме того, необходимо настроить URI линии телефонного номера организации, чтобы включить собственный уникальный расширение для маршрутизации вызовов на этот номер.

#### <a name="preparing-for-voice-mail-survivability"></a>Подготовка к обеспечению связи для голосовой почты

Единая система обмена сообщениями Exchange обычно устанавливается только на центральном сайте, а не на сайтах филиалов. Даже если канал глобальной сети между сайтом филиала и центральным сайтом недоступен, у вызывающего абонента должна быть возможность отправки сообщения голосовой почты. В результате настройка URI линии для телефонного номера автосекретарь exchange, который предоставляет голосовую почту пользователям сайта филиала, требует особого внимания в дополнение к политике голосовой связи, телефонной линии и правилам нормализации, применимым к этому номеру голосовой почты.

Устройства для обеспечения доступности филиалов (SBAs) и серверы обеспечения надежности филиалов обеспечивают поддержку голосовой почты для пользователей филиалов во время отключения WAN. В частности, если вы используете устройство для связи в филиалах или сервер для связи в филиалах и WAN-сеть становится недоступной, SBA или сервер для связи в филиалах перенаотответ неотвеченные вызовы через ТСТС в exchange UM на центральном сайте. С помощью SBA или сервера для связи в филиалах пользователи также могут получать сообщения голосовой почты через STN во время отключения WAN. Наконец, во время отключения WAN устройство для survivable Branch Appliance или Survivable Branch Server очереди уведомлений о пропущенных вызовах, а затем загружает их на сервер exchange UM при восстановлении WAN. Чтобы обеспечить устойчивость перенастройки голосовой почты, обязательно добавьте запись для FQDN пула центрального сайта и запись для FQDN сервера edge Server в файл hosts на сервере survivable Branch Server. В противном случае при отсутствии DNS-сервера на сайте филиала разрешение DNS-имен не будет работать.

При настройке обеспечения связи для голосовой почты для пользователей сайта филиала необходимо учитывать следующие рекомендации.

- Администратору Microsoft Exchange следует настроить службу автосекретарь Exchange для прием только сообщений. Эта конфигурация отключает все остальные функциональные возможности, например передачу пользователю, передачу оператору, и позволяет автосекретарю только принимать сообщения. В качестве альтернативного варианта администратор Exchange может использовать общий автосекретарь или настроенный секретарь для маршрутизации звонков оператору.

- Администратор Skype для бизнеса Server должен взять номер телефона AA  и использовать этот номер в качестве номера автоотправления для обмена электронной почтой в параметрах перенаправления голосовой почты для устройства или сервера филиала для связи в филиалах.

- Администратор Skype для бизнеса Server должен получить номер телефона для абонентского доступа к um exchange и использовать его в качестве номера абонентского доступа в параметрах перенаправления голосовой почты для устройства или сервера survivable Branch Server. 

- Администратору Skype для бизнеса Server следует настроить exchange UM таким образом, чтобы со всеми пользователями филиала, которым требуется доступ к голосовой почте во время отключения сети WAN, была связана только одна.

- Если связь WAN недоступна, вызовы пользователей сайта филиала могут перенаноситься в почтовый ящик единой системы обмена сообщениями Exchange, но только в том случае, если политика голосовой связи, примененная к вызову, указывает уникальный номер телефона голосовой почты, не включающий дополнительный номер.

#### <a name="hardware-and-software-requirements-for-branch-site-resiliency"></a>Требования к оборудованию и программному обеспечению для устойчивости сайта филиала

Требования к оборудованию и программному обеспечению зависят от решения, используемого для обеспечения устойчивости.

#### <a name="requirements-for-survivable-branch-appliances"></a>Требования к устройствам для обеспечения связи в филиалах

Необходимое оборудование и программное обеспечение встроены в устройство для обеспечения безопасности в филиалах. Однако мы также рекомендуем каждому сайту филиала развернуть DHCP-сервер для получения IP-адресов клиентов; в противном случае по истечении срока аренды DHCP у клиентов не будет IP-подключения.

Если корпоративные DNS-серверы расположены только на центральных сайтах, пользователи сайта филиала не смогут подключиться к ним во время отключения WAN- и, следовательно, обнаружение Skype для бизнеса Server, использующее запись SRV службы (SRV) DNS, не удастся. Для маршрутизации в условиях отсутствия доступа к глобальной сети требуется кэшировать DNS-записи на сайте филиала. Если маршрутизатор филиала поддерживает кэширование DNS, включите его. В противном случае вы можете развернуть DNS-сервер в филиале. Это может быть автономный сервер или версия устройства для обеспечения связи в филиалах, которая поддерживает возможности DNS. Для подробных сведений обратитесь к поставщику устройств для обеспечения связи в филиалах.

> [!NOTE]
> Контроллер домена на сайте филиала не требуется. Устройство для связи в филиалах аутентификация клиентов с помощью специального сертификата, который он отправляет клиенту в ответ на запрос сертификата клиента при его регистрации.

Клиенты Skype для бизнеса могут обнаружить Skype для бизнеса Server с помощью параметра DHCP 120 (параметр регистратора SIP). Этот запрос можно настроить следующими способами:

- Настройте DHCP-сервер на сайте филиала для ответа на запросы DHCP 120, которые возвращают FQDN регистратора на устройстве или сервере для survivable Branch Server.

- Включит DHCP Skype для бизнеса Server. Если этот параметр включен, регистратор Skype для бизнеса Server отвечает на запросы DHCP option 120. Обратите внимание, что Регистратор не отвечает на все DHCP-запросы, кроме "DHCP Options 120".

Кроме того, для крупных сайтов филиалов, имеющих несколько подсетей, нужно включить агенты DHCP-ретрансляции для переадресации запросов "DHCP Option 120" на DHCP-сервер (конфигурация 1) или Регистратор (конфигурация 2).

Наконец, пользователи сайта филиала должны быть настроены для Корпоративная голосовая связь и иметь соответствующую конечную точку объединенных коммуникаций.

### <a name="example-configuring-a-failover-route"></a>Пример: настройка маршрута от сбойов

 В следующем примере показано, как администратор может определить маршрут отработки отказа на тот случай, если шлюз Dallas-GW1 отключается для проведения технического обслуживания или недоступен по иным причинам. В таблице ниже показаны изменения, которые нужно внести в конфигурацию.

|Локальная политика в Редмонде  <br/> |RedmondLocal  <br/> |
|Политика звонков в Далласе  <br/> |DallasUsers  <br/> GlobalPSTNHopoff  <br/> |

**Таблица 2. Маршруты**
