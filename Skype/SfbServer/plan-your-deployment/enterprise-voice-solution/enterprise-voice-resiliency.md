# <a name="plan-for-enterprise-voice-resiliency-in-skype-for-business-server"></a>Планирование устойчивости Корпоративная голосовая связь в Skype для бизнеса Server

Узнайте, как поддерживать отказоустойчивость голосовой Skype для бизнеса Server Корпоративная голосовая связь, как на центральных сайтах, так и на сайтах филиалов. Параметры веб-сайтов филиала включают развертывание устройств для выживших филиалов или серверов филиалов.

Отказоустойчивость голосовой связи относится к способности пользователей продолжать выполнять и принимать вызовы, если центральный сайт, на котором размещены Skype для бизнеса Server, становится недоступным, будь то сбои в широкой сети (WAN) или по другой причине. При сбое центрального узла служба Enterprise Voice должна без прерываний перейти на резервный узел. При сбое глобальной сети вызовы узла филиала должны перенаправляться на локальный шлюз ТСОП. В данном разделе описывается планирование устойчивости голосовой связи при сбое центрального узла или глобальной сети.

## <a name="central-site-resiliency"></a>Устойчивость центрального сайта

Все чаще у предприятий появляется много сайтов, распределенных по всему миру. Обслуживание экстренных служб, доступ к службе поддержки и возможность выполнять критически важные бизнес-задачи в случае, если центральный сайт не может работать, имеет важное значение для любого решения Корпоративная голосовая связь устойчивости. Когда центральный сайт выходит из строя, должны выполняться следующие условия.

- Должна обеспечиваться отработка отказа голосовой связи.

- Пользователи, которые обычно регистрируются в пуле front End на центральном сайте, должны иметь возможность зарегистрироваться в альтернативном пуле переднего конца. Это можно сделать путем создания нескольких записей SRV DNS, каждая из которых решается в пул Директор или пул переднего конца на каждом из центральных сайтов. Вы можете настроить приоритет и вес записей SRV, чтобы пользователи, обслуживаемые центральным сайтом, могли получать соответствующий пул Director и Front End, опережая их в других записях SRV.

- Входящие и исходящие вызовы пользователей, расположенных на других сайтах, должны перенаправляться в ТСОП.

В этой статье описывается рекомендуемое решение для защиты устойчивости центрального сайта голосовой связи.

### <a name="architecture-and-topology"></a>Архитектура и топология

Планирование устойчивости голосовой функции на центральном сайте требует базового понимания центральной роли, которую играет регистратор Skype для бизнеса Server для включения отказоустойчивости голосовой функции. Регистратор Skype для бизнеса Server — это служба, которая позволяет регистрацию и проверку подлинности клиентов и предоставляет службы маршрутизации. Он выполняется на всех выпуск Standard сервере, переднем сервере, директоре или сохранившихся устройствах филиала. Пул регистраторов состоит из служб регистраторов, работающих в пуле front End и проживающих на одном сайте. Клиент Skype для бизнеса обнаруживает пул переднего конца с помощью следующего механизма обнаружения:

1. DNS-запись SRV

2. Веб-служба автооткрытия

3. Параметр DHCP 120

После подключения Skype для бизнеса к пулу переднего конца он направляется балансировкой нагрузки на один из серверов переднего конца в пуле. Этот передний конечный сервер, в свою очередь, перенаправляет клиента в предпочтительный регистратор в пуле.

Каждый пользователь, включенный для Корпоративная голосовая связь, назначен определенному пулу регистраторов, который становится основным пулом регистратора этого пользователя. На конкретном сайте сотни или тысячи пользователей обычно совместно используют один основной пул регистратора. Учитывая потребление ресурсов центрального сайта всеми пользователями сайтов филиалов, функции присутствия, конференц-связи или отработки отказа которых зависят от центрального сайта, мы рекомендуем рассматривать каждого пользователя сайта филиала, как если бы он был зарегистрирован на центральном сайте. В настоящее время не существует ограничений на количество пользователей веб-сайтов филиалов, включая пользователей, зарегистрированных с помощью устройства для ветвей.

Для обеспечения устойчивости голосовой связи в случае сбоя центрального сайта основной пул регистратора должен иметь один назначенный резервный пул регистратора, расположенный на другом сайте. Резервное копирование можно настроить с помощью параметров устойчивости Topology Builder. Если существует устойчивая связь WAN между этими двумя сайтами, то пользователи, чей основной пул регистратора стал недоступен, автоматически перенаправляются в резервный пул регистратора.

Следующие действия описывают процесс обнаружения и регистрации клиента.

1. Клиент обнаруживает Skype для бизнеса Server записи SRV DNS. В Skype для бизнеса Server записи SRV DNS можно настроить, чтобы вернуть несколько FQDN в запрос DNS SRV. Например, если предприятие Contoso имеет три центральных сайта (в Северной Америке, в Европе и в Азиатско-Тихоокеанском регионе), и пул Директор существует на каждом центральном сайте, то записи DNS SRV могут указывать на полное доменное имя пула Директор в каждом из этих трех расположений. До тех пор, пока пул Director в одном из местоположений доступен, клиент может подключиться к первому Skype для бизнеса Server.

    > [!NOTE]
    > Использование пула Director является необязательным. Вместо этого можно использовать пул переднего конца.

2. Пул Директор информирует клиента Skype для бизнеса о основном пуле регистратора пользователя и пуле резервного регистратора.

3. Сначала Skype для бизнеса клиент пытается подключиться к основному пулу регистраторов пользователя. Если основной пул регистратора доступен, то регистратор принимает регистрацию. Если основной пул регистратора недоступен, Skype для бизнеса пытается подключиться к пулу резервного регистратора. Если пул резервного регистратора доступен и определил, что основной пул регистратора пользователя недоступен (обнаружив отсутствие пульса для указанного интервала сбой), пул регистратора резервного копирования принимает регистрацию пользователя. После того как регистратор резервного копирования обнаружит, что основной регистратор снова доступен, пул резервного регистратора перенаправит клиентов с выходом из строя в основной пул.

### <a name="requirements-and-recommendations"></a>Требования и рекомендации

Следующие требования и рекомендации для реализации устойчивости голосовой связи на центральном сайте подходят для большинства организаций.

- Сайты, на которых размещаются основной и резервный пулы регистратора, должны быть связаны устойчивой связью WAN.

- На каждом центральном сайте должен быть пул регистратора, состоящий из одного или нескольких регистраторов.

- Каждый пул регистратора должен быть сбалансирован с помощью балансировки нагрузки DNS, балансировки аппаратной нагрузки или обоих. Подробные сведения о планировании конфигурации балансировки нагрузки см. в описании требований к балансировки нагрузки для [Skype для бизнеса.](../../plan-your-deployment/network-requirements/load-balancing.md)

- Каждый пользователь должен быть назначен основному пулу регистраторов с помощью Skype для бизнеса Server команды **set-CsUser** или Skype для бизнеса Server панели управления.

- Основной пул регистратора должен иметь один резервный пул регистратора, расположенный на другом центральном сайте.

- Основной пул регистратора должен быть настроен для отработки отказа в резервный пул регистратора. По умолчанию задана отработка отказа основного пула регистратора в резервный пул через 300 секунд. Этот интервал можно изменить с помощью Skype для бизнеса Server Topology Builder.

- Настройка маршрута сбойной передачи. При настройке маршрута укажите шлюз, расположенный на другом сайте, чем шлюз, указанный в основном маршруте.

- Если центральный сайт содержал основной сервер управления, а сайт, скорее всего, не будет работать в течение длительного периода времени, вам потребуется переустановить средства управления на резервном сайте; в противном случае вы не сможете изменить параметры управления.

### <a name="survivable-voice-features"></a>Функции обеспечения голосовой связи

Если предыдущие требования и рекомендации реализованы, то резервным пулом регистратора будут предоставляться следующие функции голосовой связи.

- Исходящие вызовы ТСОП.

- Входящие вызовы ТСОП, если поставщик услуг телефонной связи поддерживает возможность отработки отказа на резервный сайт.

- Корпоративные вызовы между пользователями как одного сайта, так и разных сайтов.

- Базовая обработка вызова, включая удержание, возобновление и переключение вызовов.

- Двусторонний обмен мгновенными сообщениями и совместное использование аудио и видео пользователями на одном сайте.

- Услуги переадресации вызовов, одновременных звонков в нескольких конечных точках, делегирования вызовов и групповых звонков, но только если обе стороны делегирования вызова или все члены группы настроены на одном и том же сайте.

- Продолжение функционирования существующих телефонов и клиентов.

- Регистрация вызовов (CDR).

- Проверка подлинности и авторизация.

Когда основной центральный сайт выходит из строя, могут также работать следующие функции голосовой связи (в зависимости от того, как они были настроены).

- Размещение и извлечение голосовых сообщений.

    Если необходимо, чтобы при выходе из строя основного центрального сайта оставалась доступной единая система обмена сообщениями Exchange, то необходимо выполнить одно из следующих действий.

  - Изменить запись DNS SRV таким образом, чтобы серверы единой системы обмена сообщениями Exchange на центральном сайте указывали на резервные серверы единой системы обмена сообщениями Exchange на другом сайте.

  - Настройте телефонную Exchange каждого пользователя, чтобы включить Exchange серверы um как на центральном сайте, так и на сайте резервного копирования, но назначите резервные Exchange серверы um как отключенные. Если основной сайт становится недоступен, Exchange администратору необходимо отметить Exchange серверов системы электронной почты на сайте резервного копирования в качестве включенного.

    Если ни одно из предыдущих решений не возможно, то Exchange не будет доступен в том случае, если центральный сайт станет недоступен.

- Конференц-связь всех типов.

    Пользователь, который в результате отработки отказа оказался на резервном сайте, может присоединяться к конференции, созданной или размещенной тем организатором, чей пул доступен, но не может создавать или размещать конференцию в своем основном пуле, который стал недоступен. Кроме того, другие пользователи не могут присоединяться к конференциям, которые находятся в основном пуле пострадавшего пользователя.

Следующие функции голосовой связи не будут работать в случае выхода из строя основного центрального сайта.

- Автосекретарь конференции.

- Функция присутствия и маршрутизация на основе DND.

- Обновление параметров переадресации вызовов.

- Служба "Группа ответа" и парковка вызовов.

- Подготовка новых телефонов и клиентов.

- Веб-поиск в адресной книге

## <a name="branch-site-resiliency"></a>Устойчивость сайта филиала

Если вы хотите обеспечить устойчивость филиала-сайта, то есть службу высокой доступности Корпоративная голосовая связь, у вас есть три варианта для этого:

- Устройство для обеспечения связи в филиалах

- сервер для обеспечения связи в филиалах

- Полное развертывание Skype для бизнеса Server на сайте филиала

Это руководство поможет вам определить, какое решение обеспечения устойчивости лучше всего подходит для вашей организации, и в зависимости от принятого решения выбрать вариант ТСОП-связи, подходящий для использования. Руководство также поможет вам лучше подготовиться к развертыванию выбранного решения с учетом предварительных требований и других факторов, которые необходимо принять во внимание при планировании.

### <a name="branch-site-resiliency-features"></a>Функции устойчивости веб-сайтов филиала

Если вы обеспечиваете устойчивость веб-сайта филиала, если подключение веб-узла филиала к центральному сайту не удается или если центральный сайт недостижим, необходимо продолжать предоставлять следующие функции голосовой связи:

- Входящие и исходящие вызовы по телефонной сети общего пользования (ТСОП).

- Корпоративные вызовы между пользователями как одного сайта, так и разных сайтов.

- Базовая обработка вызова, включая удержание, возобновление и переключение вызовов.

- Двусторонний обмен мгновенными сообщениями.

- Переадправление вызовов, одновременный звон конечных точек, делегирование вызовов и службы вызова группы, но только в том случае, если делегатор и делегат (например, менеджер и администратор диспетчера) или все члены группы настроены на одном сайте.

- Записи регистрации вызовов (CDR).

- Конференц-связь с телефонным подключением из ТСОП с автосекретарем конференц-связи.

- Возможности голосовой почты, если настроены параметры маршрутизации голосовой почты.

- Проверка подлинности и авторизация пользователей.

Следующие функции будут доступны только в том случае, если решение по отказоустойчивости является полномасштабным Skype для бизнеса Server развертывания на сайте филиала:

- Обмен мгновенными сообщениями, веб-конференции, а также аудио- и видеоконференции.

- Маршрутизация на основании сведений о присутствии и статуса "не беспокоить" (когда вызовы не осуществляются для расширений со статусом "не беспокоить").

- Обновление параметров переадресации вызовов.

- Приложение Группы отклика и приложение Call Park

- Подготовка новых телефонов и клиентов, но только в том случае, если службы домена Active Directory присутствуют на сайте филиала.

- Enhanced 9-1-1 (E9-1-1).

    Если развертывается E9-1-1, а магистраль SIP на центральном сайте недоступна из-за того, что связь WAN не работает, то устройство для выживших филиалов будет перенаправживать вызовы E9-1-1 на шлюз локального филиала. Чтобы включить эту функцию, политики голосовой связи пользователей филиала должны маршрутить вызовы на локальный шлюз в случае сбоя WAN.

> [!NOTE]
> SBA (сохранившийся филиал) не поддерживается для XMPP. Пользователи, управляемые в конфигурациях SBA, не смогут отправлять IMs или видеть Присутствие с контактами XMPP.

### <a name="branch-site-resiliency-solutions"></a>Решения для устойчивости веб-сайтов филиала

Есть очевидные преимущества для обеспечения устойчивости филиала-сайта для вашей организации. В частности, если вы потеряете подключение к центральному сайту, пользователи веб-сайтов филиала будут по-прежнему иметь Корпоративная голосовая связь службу и голосовую почту (если настроить параметры перенастройки голосовой почты). Однако для сайтов с менее чем 25 пользователями решение по устойчивости может не обеспечить достаточной отдачи от инвестиций.

Доступно три варианта обеспечения устойчивости сайта филиала. Воспользуйтесь следующей таблицей, чтобы определить оптимальный для себя вариант.

#### <a name="survivable-branch-appliance-details"></a>Описание механизма обеспечения связи в филиалах

Эти пользователи имеют доступ ко всем возможностям единой системы обмена данными за одним исключением — входящие вызовы на сайт филиала будут переадресовываться на голосовую почту. После восстановления подключения по глобальной сети пользователям сайта будут снова предоставлены все функциональные возможности. Ни сбой в устройстве для выживших филиалов, ни восстановление службы не требуют присутствия ИТ-администратора.

Skype для бизнеса Server поддерживает до двух сохранившихся устройств филиала на сайте филиала.

В этом разделе приведены инструкции по подготовке пользователей к использованию устойчивости сайта филиала и обеспечения связи для голосовой почты, а также приведены требования к оборудованию и программному обеспечению.

#### <a name="preparing-branch-users-for-branch-site-resiliency"></a>Подготовка пользователей филиала для устойчивости сайта филиала

Подготовка пользователей к устойчивости на веб-сайте филиала, задав пул регистратора в качестве выживаемого устройства филиала (SBA) или выживаемого сервера филиала.

#### <a name="registrar-assignments-for-branch-users"></a>Назначения Регистратора для пользователей филиала

Независимо от того, какое решение по устойчивости филиала-сайта вы выбираете, каждому пользователю необходимо назначить основного регистратора. Пользователи веб-сайтов филиала всегда должны регистрироваться в регистраторе на сайте филиала независимо от того, находится ли этот регистратор в устройстве для выживших филиалов, сервере выживших филиалов или отдельном сервере Skype для бизнеса Server Standard или выпуск Enterprise сервере. Чтобы клиент мог обнаружить пул Регистратора, требуется DNS-запись SRV. Если устройство для выживших филиалов становится недоступным, клиенты веб-сайтов филиалов автоматически обнаруживают резервный регистратор.

Если на веб-сайте филиала нет DNS-сервера, существует два альтернативных способа настройки обнаружения устройства для выживших филиалов или выживаемого сервера филиала:

- Настройка параметра DHCP 120 на сервере динамической конфигурации узла филиала (DHCP) для указать полностью квалифицированное доменное имя (FQDN) устройства для выживших филиалов или выживаемого сервера филиала.

- Настройте уцелевшие ветвные устройства или серверы филиалов, чтобы отвечать на запросы DHCP 120.

#### <a name="voice-routing-for-branch-users"></a>Маршрутизация голосовых вызовов для пользователей филиала

Рекомендуется создать отдельную политику голосовой связи на уровне пользователя для пользователей на веб-сайте филиала. Эта политика должна включать основной маршрут, использующий уцелевшие шлюзы устройства филиала или сервер филиала, а также один или несколько маршрутов резервного копирования, использующих магистраль с шлюзом открытой телефонной сети (PSTN) на центральном сайте. Если основной маршрут недоступен, вместо него используется маршрут резервного копирования, использующий один или несколько центральных шлюзов сайта. Таким образом, независимо от того, где зарегистрирован пользователь— на регистраторе веб-сайта филиала или пуле резервного регистратора на центральном сайте — политика voIP пользователя всегда действует. Это важное значение для сценариев сбой. Например, если необходимо переименовать устройство для выживших ветвей или перенастроить устройство для выживших филиалов, чтобы подключиться к резервному пулу регистраторов на центральном сайте, необходимо переместить пользователей веб-сайтов филиала на центральный сайт на время. (Подробные сведения о переименовании или перенастройки устройства для выживших филиалов см. в приложении [B. Управление](/previous-versions/office/lync-server-2013/lync-server-2013-appendix-b-managing-a-survivable-branch-appliance) устройством для сохранившихся филиалов в документации развертывания.) Если у этих пользователей нет политик VoIP уровня пользователя или телефонных планов на уровне пользователей, при перемещении пользователей на другой сайт политики VoIP на уровне сайта и планы набора номера на уровне сайта центрального сайта применяются к пользователям по умолчанию, а не к политикам VoIP-сайтов на уровне филиала и планам набора номеров. В этом сценарии, если политики voIP на уровне сайта и планы набора на уровне сайта, используемые пулом резервного регистратора, также не будут применяться к пользователям веб-сайтов филиала, их вызовы не будут работать. Например, если пользователи с сайта филиала, расположенного в Японии, перемещаются на центральный сайт в Редмонде, то телефонный план с правилами нормализации, предопластив +1425 для всех 7-значных вызовов, вряд ли будет надлежащим образом переводить вызовы для этих пользователей.

> [!IMPORTANT]
> При создании резервного маршрута для филиала мы рекомендуем добавить в пользовательскую политику филиала два режима работы с ТСОП и назначить каждому режиму отдельные маршруты. Первый или основной маршрут будет направлять вызовы на шлюз, связанный с устройством для выживших филиалов (SBA) или сервером филиала; второй или резервный маршрут будет направлять вызовы на шлюз на центральном сайте. Перед использованием второго режима работы с ТСОП устройство для обеспечения связи в филиалах или сервер филиала попытаются использовать все маршруты, назначенные первому режиму работы с ТСОП.

Чтобы обеспечить недоступность входящих вызовов для пользователей веб-сайтов филиала, когда шлюз филиала или компонент Windows сайта для устройства для входящих филиалов недоступны (что произойдет, например, в случае, если для обслуживания был недоступен аппарат филиала или шлюз филиала), создайте маршрут отработки сбой на шлюзе (или работайте с поставщиком прямых входящих диалоговых вызовов (DID) для перенаправления входящих вызовов в пул регистраторов резервного копирования в центре. на сайте. Звонки будут перенаправляться пользователям филиала через глобальную сеть. Убедитесь, что маршрут переводит номера в соответствии с шлюзом PSTN или принятыми форматами номеров телефонов другого узла. Дополнительные сведения о создании маршрута отработки отказа см. в разделе [Configuring a Failover Route](/previous-versions/office/lync-server-2013/lync-server-2013-configuring-a-failover-route). Кроме того, для применения правил нормализации к входящим звонкам создайте абонентские группы на уровне службы для магистрали, связанной со шлюзом на сайте филиала. Если на веб-сайте филиала есть два устройства для ветвей, можно создать телефонную планку на уровне сайта для обоих, если для каждого из них не потребуется отдельный план уровня обслуживания.

> [!NOTE]
> Для учета ресурсов центрального сайта, потребляемых пользователями сайта филиала, которые используют центральный сайт для получения сведений о присутствии, конференц-связи или отработки отказа, мы рекомендуем учитывать каждого пользователя сайта филиала как пользователя, зарегистрированного на центральном сайте. В настоящее время не существует ограничений на количество пользователей веб-сайтов филиалов, включая пользователей, зарегистрированных с помощью устройства для ветвей.

Кроме того, мы рекомендуем создать абонентскую группу и политику голосовой связи на уровне пользователя и затем назначить их пользователям сайта филиала. Дополнительные сведения см. в документе [Create or modify a dial plan in Skype для бизнеса Server](../../deploy/deploy-enterprise-voice/dial-plans.md) and Create the [VoIP Routing Policy for Branch Users](/previous-versions/office/lync-server-2013/lync-server-2013-create-the-voip-routing-policy-for-branch-users) in the Deployment documentation.

#### <a name="routing-extension-numbers"></a>Маршрутизация добавочных номеров

При подготовке планов и голосовых политик для пользователей веб-сайтов филиала обязательно включите правила нормализации и правила перевода, которые соответствуют строкам и формату номеров, используемым в атрибуте msRTCSIP-line (или Line URI), чтобы Skype для бизнеса вызовы, включенные между пользователями веб-сайтов филиала и центральными пользователями сайта, были правильно перенаправлены, особенно если вызовы должны перенаправляться через PSTN, так как ссылка WAN недоступна. В отличие от обычных номеров для номеров с добавочными номерами существуют особые требования.

Правила нормализации и преобразования, которые соответствуют Line URI, содержащим добавочный номер (отдельный номер или в дополнении к полному номеру телефона E.164), имеют дополнительные требования. В этом разделе описываются примеры маршрутизации звонков для Line URI с добавочными номерами.

Если в вашей организации не настроен прямой набор внутренних номеров для отдельных пользователей и в качестве URI линии для всех пользователей указан только добавочный номер, внутренние пользователи могут звонить друг другу, используя только добавочный номер. Однако вам потребуется настроить правила нормализации, которые можно применить при звонках пользователя сайта филиала пользователю центрального сайта и которые соответствуют добавочным номерам.

Если канал глобальной сети между сайтом филиала и центральным сайтом доступен, то для звонков пользователей сайта филиала пользователям центрального сайта не требуется правило нормализации для преобразования номеров, поскольку звонки не маршрутизируются через ТСОП. Пример:

|**Название правила**|**Описание**|**Шаблон номера**|**Преобразование**|**Пример**|
|:-----|:-----|:-----|:-----|:-----|
|5digitExtensions  <br/> |Не преобразует номера, состоящие из 5 знаков  <br/> |^(\d{5})$  <br/> |$1  <br/> |10001 не преобразуется  <br/> |

Вам также необходимо адаптировать добавочные номера для случаев, когда канал глобальной сети между сайтом филиала и центральным сайтом недоступен, а звонок из сайта филиала необходимо перенаправить через ТСОП. Во время отключения WAN, если пользователь сайта филиала вызывает центрального пользователя сайта только путем набора расширения центрального пользователя сайта, необходимо иметь правило исходящие переводы, которое добавляет полный телефонный номер центрального пользователя сайта. Если в строке пользователя URI содержится полный телефонный номер организации и уникальный номер расширения вместо полного номера телефона, уникального для пользователя, необходимо иметь правило исходящие переводы, которое добавляет полный номер телефона организации. Например:

|**Описание**|**Шаблон совпадения**|**Преобразование**|**Пример**|
|:-----|:-----|:-----|:-----|
|Перевод 5-значных номеров на телефонный номер и расширение пользователя  <br/> |^(\d{5})$  <br/> |+14255550123;ext=$1  <br/> |10001 преобразуется в +14255550123;доб.=10001  <br/> |
|Перевод 5-значных номеров на телефонный номер организации и расширение пользователя  <br/> |^(\d{5})$  <br/> |+14255550100;ext=$1  <br/> |10001 преобразуется в +14255550100;доб.=10001  <br/> |

В этом сценарии, если узел магистрали, который обрабатывает изменение маршрута в ТСОП, не поддерживает добавочные номера, то правило преобразования исходящих звонков также должно удалять добавочные номера. Пример:

|**Описание**|**Шаблон совпадения**|**Преобразование**|**Пример**|
|:-----|:-----|:-----|:-----|
|Удаляет добавочный номер из номеров телефонов, содержащих добавочные номера  <br/> |^\+\*(\d); ext=(\d \* )$  <br/> |+$1  <br/> |+14255550123;доб.=10001 преобразуется в +14255550123  <br/> |

Доступна ли ссылка WAN или нет, если в вашей организации не настроены номера DID для отдельных пользователей, а строка URI для пользователя содержит телефонный номер организации и уникальный номер расширения пользователя, необходимо настроить номер телефона вашей организации с номером строки URI с номером, доступным для узла магистрали или шлюза PSTN на сайте филиала. Кроме того, необходимо настроить URI телефонной строки вашей организации, чтобы включить свое уникальное расширение для звонков, которые будут маршрутироваться на этот номер.

#### <a name="preparing-for-voice-mail-survivability"></a>Подготовка к обеспечению связи для голосовой почты

Exchange Единая система обмена сообщениями обычно устанавливается только на центральном сайте, а не на сайтах филиалов. Даже если канал глобальной сети между сайтом филиала и центральным сайтом недоступен, у вызывающего абонента должна быть возможность отправки сообщения голосовой почты. В результате настройка URI линии для номера Exchange um автосекретарь, который предоставляет голосовую почту для пользователей веб-сайта филиала, требует особых соображений, в дополнение к политике голосовой связи, плану набора номера и правилам нормализации, применимым к этому номеру голосовой почты.

Оставшиеся в живых устройства филиала (SBAs) и Выжившие серверы филиалов обеспечивают выживаемость голосовой почты для пользователей филиалов во время отключения WAN. В частности, если вы используете уцелевшие устройства филиала или выживший сервер филиала и WAN становится недоступным, SBA или survivable Branch Server перенастроит безответные вызовы через PSTN, чтобы Exchange um на центральном сайте. С помощью SBA или выживаемого сервера филиала пользователи также могут получать сообщения голосовой почты через PSTN во время отключения WAN. Наконец, во время отключения WAN в очередях из пропущенных уведомлений о пропущенных вызовах в очередях из оставшихся в живых филиалов или серверах филиалов выгружаются на сервер электронной почты Exchange при восстановлении WAN. Чтобы убедиться, что перенастройка голосовой почты устойчива, обязательно добавьте запись для FQDN центрального пула сайтов и запись для FQDN edge Server в файл хостов на сервере выживших филиалов. В противном случае при отсутствии DNS-сервера на сайте филиала разрешение DNS-имен не будет работать.

При настройке обеспечения связи для голосовой почты для пользователей сайта филиала необходимо учитывать следующие рекомендации.

- Администратор microsoft Exchange должен настроить Exchange службу автосекретарь (AA) только для приемки сообщений. Эта конфигурация отключает все остальные функциональные возможности, например передачу пользователю, передачу оператору, и позволяет автосекретарю только принимать сообщения. В качестве альтернативного варианта администратор Exchange может использовать общий автосекретарь или настроенный секретарь для маршрутизации звонков оператору.

- Администратору Skype для бизнеса Server взять номер телефона AA и использовать этот номер телефона в качестве автопособника exchange **um** в параметрах перенаправления голосовой почты для устройства для выживших филиалов или сервера филиала.

- Администратор Skype для бизнеса Server должен получить номер Exchange абонента абонента um и использовать  этот номер в качестве номера доступа абонента в параметрах перенаправления голосовой почты для выживаемого устройства филиала или выживаемого сервера филиала.

- Администратор Skype для бизнеса Server настроить Exchange, чтобы во время отключения WAN со всеми пользователями филиалов, которым необходим доступ к голосовой почте, был связан только один набор номеров.

- Если ссылка WAN недоступна, вызовы пользователям веб-сайта филиала могут быть перенастроены в почтовый ящик Exchange единой системы обмена сообщениями, но только в том случае, если политика голосовой связи, применяемая к вызову, указывает уникальный номер телефона голосовой почты и не включает номер расширения.

#### <a name="hardware-and-software-requirements-for-branch-site-resiliency"></a>Требования к оборудованию и программному обеспечению для устойчивости сайта филиала

Требования к оборудованию и программному обеспечению зависят от решения, используемого для обеспечения устойчивости.

#### <a name="requirements-for-survivable-branch-appliances"></a>Требования к устройствам для обеспечения связи в филиалах

Необходимое оборудование и программное обеспечение встроены в устройство для выживших филиалов. Однако мы также рекомендуем каждому веб-сайту филиала развернуть сервер DHCP для получения IP-адресов клиента; в противном случае, когда срок аренды DHCP истекает, у клиентов не будет ip-подключения.

Если корпоративные DNS-серверы расположены только на центральных сайтах, пользователи веб-сайтов филиалов не смогут подключиться к ним во время отключения WAN, и поэтому Skype для бизнеса Server обнаружения, использующее запись ресурсов DNS SRV (служба SRV) не удастся. Для маршрутизации в условиях отсутствия доступа к глобальной сети требуется кэшировать DNS-записи на сайте филиала. Если маршрутизатор филиала поддерживает кэширование DNS, включите его. В противном случае вы можете развернуть DNS-сервер в филиале. Это может быть автономный сервер или версия устройства для выживших филиалов, поддерживаюшего возможности DNS. Для более подробной информации обратитесь к поставщику бытовых устройств для ветвей филиала.

> [!NOTE]
> Контроллер домена на сайте филиала не требуется. Уцелевшая ветвная техника аутентификация клиентов с помощью специального сертификата, который он отправляет клиенту в ответ на запрос сертификата клиента при его въехании.

Skype для бизнеса клиенты могут обнаружить Skype для бизнеса Server с помощью DHCP Option 120 (параметр регистратора SIP). Этот запрос можно настроить следующими способами:

- Настройте сервер DHCP на сайте филиала, чтобы отвечать на запросы DHCP 120, которые возвращают FQDN регистратора на сохранившихся устройствах филиала или в выживаемом сервере филиала.

- Включи Skype для бизнеса Server DHCP. Когда это включено, Skype для бизнеса Server регистратор отвечает на запросы DHCP Option 120. Обратите внимание, что Регистратор не отвечает на все DHCP-запросы, кроме "DHCP Options 120".

Кроме того, для крупных сайтов филиалов, имеющих несколько подсетей, нужно включить агенты DHCP-ретрансляции для переадресации запросов "DHCP Option 120" на DHCP-сервер (конфигурация 1) или Регистратор (конфигурация 2).

Наконец, пользователи веб-сайтов филиала должны быть настроены для Корпоративная голосовая связь и с соответствующей конечной точкой единой связи.
