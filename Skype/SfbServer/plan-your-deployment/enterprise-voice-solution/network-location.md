# <a name="define-the-network-elements-used-to-determine-location-in-skype-for-business-server"></a>Определение элементов сети, используемых для определения расположения в Skype для бизнеса Server
 
Решения, необходимые для планирования сетевых компонентов, которые будут использовать для связи звонющих с расположениями для развертывания E9-1-1 в Skype для бизнеса Server Корпоративная голосовая связь.
  
Если вы настраивает инфраструктуру Skype для бизнеса Server для поддержки автоматического определения местоположения клиента, сначала необходимо решить, какие элементы сети будут использовать для связи звонющих с расположениями. В Skype для бизнеса Server можно связать следующие элементы сети уровня 2 и 3 с расположениями:
  
- адреса BSSID точек беспроводного доступа (WAP) (уровень 2);
    
- порт коммутатора LLDP (уровень 2);
    
- идентификаторы шасси коммутаторов LLDP (уровень 2);
    
- IP-подсети (уровень 3);
    
- MAC-адреса клиентов (уровень 2).
    
Элементы сети перечислены в порядке приоритета. Если клиент может быть расположен с помощью более чем одного элемента сети, Skype для бизнеса Server использует порядок приоритета для определения используемого механизма. 
  
В следующих подразделах приводятся более подробные сведения об использовании каждого элемента сети.
  
> [!IMPORTANT]
> При использовании сетевых элементов для связи вызывающих с местоположениями крайне важно поддерживать базу данных службы сведений о местонахождении в соответствии с данными о местоположении. Так, если вы добавляете или изменяете элемент сети, например WAP, вам нужно удалить старую запись и добавить новую в базу данных местоположений. 
  
## <a name="wireless-access-point"></a>Точка беспроводного доступа

Когда клиент подключается к сети по беспроводной сети, то для определения его местоположения в запросе местоположения используется адрес BSSID WAP. Если клиент находится в роуминге, указанная WAP может не быть ближайшей, и можно даже получить WAP, расположенную на другом этаже здания. Чтобы указать, что расположение приблизительное, можно заранее указать значение расположения с помощью дескриптора **[Near]** или **[Closeto].**
  
Этот метод определения местоположения предполагает, что адреса BSSID всех точек WAP являются статическими. Однако если поставщик WAP использует динамически присвоенные BSSID, BSSID, полученный из WAP, может измениться (это может произойти после изменения конфигурации WAP), а беспроводные клиенты могут остаться в ситуации, когда они не получают расположение. Чтобы предотвратить эту возможность, необходимо заполнить базу данных службы сведений о местонахождении с помощью ERLs для всех возможных адресов BSSID, используемых каждой WAP. 
  
## <a name="lldp-ports-and-switches"></a>Порты и коммутаторы LLDP

Управляемые коммутаторы Ethernet, поддерживающие протокол LLDP-MED, могут сообщить свои идентификационные данные и сведения о портах клиентам, совместимым с LLDP-MED. Это позволяет затем выполнять запросы к базе данных местоположений для определения местоположения устройства. Вы можете связать сведения о местоположении для экстренных служб с идентификатором шасси коммутатора или сопоставить их также на более низком уровне — уровне портов.
  
> [!NOTE]
> Skype для бизнеса Server поддерживает использование LLDP-MED для определения расположения только устройств Lync Phone Edition и клиентов Skype для бизнеса, работающих на Windows 8. Если вам нужно использовать данные 2-го уровня коммутатора для определения расположения других клиентов Skype для бизнеса Server на основе проводного ПК, необходимо использовать метод MAC-адреса клиента. 
  
## <a name="subnet"></a>Subnet

IP-подсети 3-го уровня предоставляют механизм, поддерживаемый всеми клиентами Skype для бизнеса Server, который можно использовать для автоматического определения местоположения клиента. Использование IP-подсетей является самым простым способом определения местоположения в плане настройки клиентов проводного доступа и управления ими. Однако прежде чем выбрать его, ответьте на следующие вопросы, чтобы определить, будет ли он достаточно точным в вашей ситуации.
  
- Охватывают ли какие-либо клиентские подсети несколько этажей?
    
- Охватывают ли какие-либо подсети несколько зданий?
    
- Какую площадь покрывает каждая клиентская подсеть?
    
Если подсеть покрывает слишком большую площадь, может потребоваться использовать другой способ для определения местоположения клиентов. Однако, если это целесообразно, мы рекомендуем клиентам реорганизовать структуру IP-подсетей в соответствии с требованиями к точности определения местоположений ERL вместо того, чтобы реализовывать сторонние решения на основе SNMP, связанные с дополнительными затратами и повышением сложности.
  
## <a name="client-mac-address"></a>MAC-адрес клиента

Чтобы найти звоняшего с помощью MAC-адреса клиентского компьютера, необходимы управляемые коммутаторы Ethernet, а также развернуть стороннее решение SNMP, которое может обнаруживать MAC-адреса клиентов Skype для бизнеса, подключенных к этим коммутаторам или через них. Решение SNMP постоянно оповешает управляемые коммутаторы, чтобы получить текущие сопоставления MAC-адресов конечных точек, подключенных к каждому порту, и получает соответствующие ИД порта. Во время запроса клиента Skype для бизнеса к службе сведений о расположении служба сведений о расположении запрашивает стороне приложение с помощью MAC-адреса клиента, а затем возвращает все совпадающие IP-адреса коммутатора и ИД порта. Служба сведений о расположении использует эти сведения для запроса опубликованного проволокового карты уровня 2 на соответствие записи и возвращает расположение клиенту. При использовании этого параметра убедитесь, что идентификаторы портов коммутатора согласованы между приложением SNMP и опубликованными записями базы данных расположений.
  
> [!NOTE]
> Некоторые сторонние решения SNMP могут поддерживать неугодные коммутаторы доступа; Если коммутатор, который службирует клиент Skype для бизнеса, неуменяем, но имеет связь с управляемым коммутатором распространения, управляемый коммутатор может сообщить приложению SNMP MAC-адреса клиентов, подключенных к коммутатору доступа. Эти сведения позволяют службе сведений о расположении определить расположение пользователя. Однако можно назначить только один ERL всем портам в неугодном коммутаторе, поэтому специфичность расположения доступна только на уровне корпусов коммутатора доступа, а не на уровне порта.
