# <a name="manage-locations-for-sip-trunk-service-providers-in-skype-for-business-server"></a>Управление расположениями для поставщиков услуг магистрали SIP в Skype для бизнеса Server

Решения, необходимые для планирования базы данных сведений о расположении или аналогичной внешней базы данных для развертывания E9-1-1 с использованием поставщиков магистрали SIP, в Skype для бизнеса Server Корпоративная голосовая связь.

Чтобы настроить Skype для бизнеса Server для автоматического поиска клиентов в сети, необходимо либо заполнить базу данных службы сведений о местоположении сетевой картой и опубликовать расположения, либо связаться с внешней базой данных, которая уже содержит правильные сопоставления. В рамках этого процесса необходимо проверить адреса города в расположениях у поставщика услуг E9-1-1. Подробные сведения [см. в документации](https://technet.microsoft.com/library/8544be31-6958-47ef-b926-fdc80d56191c.aspx) по развертыванию "Настройка базы данных местоположений".

Добавьте в базу данных службы Location Information местоположение для аварийного реагирования (Emergency Response Location — ERL), которое состоит из почтового адреса и точного местоположения в здании. Поле "Расположение **службы** сведений о расположении", которое является определенным расположением в здании, имеет максимальную длину 20 символов (включая пробелы). Учитывая это ограничение, в поле рекомендуется включить следующие данные:

- Понятное имя, идентифицирующее расположение звонящего в аварийную службу, которое позволит агенту аварийной службы быстрее находить указанное расположение в момент прибытия по гражданскому адресу. Имя расположения может включать номер здания, номер этажа, обозначение корпусов, номер квартиры и т. д. Избегайте употребления сокращенный названий, известных только сотрудникам, что может привести к направлению аварийной службы по неправильному адресу.

- Идентификатор расположения, который помогает пользователям легко увидеть, что их клиент Skype для бизнеса выбрал правильное расположение. Клиент Skype для бизнеса автоматически совмещет  и  отобразит обнаруженные поля "Расположение" и "Город" в своем заголке. К каждому идентификатору расположения (например, "1-й этаж") лучше добавить адрес <street number> здания. Без почтового адреса общий идентификатор расположения, такой как "1-ый этаж", может применяться к любому зданию в городе.

- Если расположение приблизительное, так как оно определяется точкой беспроводного доступа, можно добавить слово **[Near]** (например, "Near 1st Floor 1234").

> [!NOTE]
> Расположения, добавленные в центральную базу данных расположений, недоступны клиенту, пока не будут опубликованы с помощью команды командной оболочки Skype для бизнеса Server и реплицированы в локальные хранилища пула. Дополнительные сведения см. в разделе [Publishing the Location Database](https://technet.microsoft.com/library/dd032b5b-df0e-4017-ac46-e17570c1ab1e.aspx) документации по развертыванию.

В следующих разделах рассматриваются аспекты, которые следует учитывать при заполнении и обслуживании базы данных расположения.

## <a name="populating-the-location-database"></a>Заполнение базы данных расположений

Ниже приведены вопросы, которые помогут определить порядок заполнения базы данных.

 **Какой процесс будет использоваться для заполнения базы данных расположений?**

Где хранятся данные и какие действия необходимо предпринять, чтобы преобразовать данные в формат, требуемый базой данных местоположений? Будут ли местоположения добавляться по-отдельности или массово с использованием CSV-файла?

 **Используется ли сторонняя база данных, которая уже содержит сопоставление местоположений?**

С помощью дополнительного параметра службы сведений о местоположении для подключения к сторонней базе данных можно сгруппить расположения и управлять ими с помощью автономной платформы. Преимущество этого подхода заключается в том, что дополнительно к связыванию местоположений с идентификаторами сети можно связать местоположения с пользователями. Это означает, что служба сведений о расположении может возвращать несколько адресов из службы дополнительных сведений о расположении клиенту Skype для бизнеса. Пользователь затем может выбрать наиболее подходящее местоположение.

Для интеграции со службой сведений о расположении сторонная база данных должна следовать схеме запроса или ответа расположения Lync Server. Подробные сведения см. [в "[MS-E911WS]: веб-служба для спецификации протокола поддержки E911".](https://go.microsoft.com/fwlink/p/?linkid=213819) Дополнительные сведения о развертывании службы дополнительных сведений о расположении см. в документации по развертыванию" "Настройка дополнительной службы сведений о местоположении [в Skype](../../deploy/deploy-enterprise-voice/secondary-location-information-service.md) для бизнеса Server".

Дополнительные сведения о заполнении базы данных местоположений см. в разделе [Configure the Location Database](https://technet.microsoft.com/library/8544be31-6958-47ef-b926-fdc80d56191c.aspx) документации по развертыванию.

## <a name="maintaining-the-location-database"></a>Обслуживание базы данных местоположений

После заполнения базы данных необходимо разработать стратегию обновления базы данных по мере внесения изменений в конфигурацию сети. Ниже приведены вопросы, которые помогут определить порядок обслуживания базы данных местоположений.

 **Как будет обновляться база данных расположений?**

Есть несколько сценариев, которые требуют обновления базы данных расположений, включая добавление протоколов WAP, изменение разводки в офисе (в связи с разными назначениями коммутаторов) и расширение подсети. Будет ли выполняться прямое обновление каждого отдельного расположения или массовое обновление всех расположений с помощью CSV-файла?

 **Будет ли использоваться для сопоставления MAC-адресов клиентов Lync с портами и идентификаторами коммутаторов приложение SNMP?**

При использовании приложения SNMP потребуется разработать ручную процедуру, которая позволит поддерживать согласованность данных о портах и корпусах коммутаторов между приложением SNMP и базой данных местоположений. Если приложение SNMP возвращает IP-адрес или ИД порта корпусов, не включенных в базу данных, служба сведений о расположении не сможет вернуть расположение клиенту.
