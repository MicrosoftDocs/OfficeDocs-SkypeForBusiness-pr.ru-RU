# <a name="plan-for-video-interop-server-in-skype-for-business-server"></a>Планирование видео-interop Server в Skype для бизнеса Server
 
**Сводка:** Просмотрите эту тему при планировании интеграции Skype для бизнеса Server с сторонними устройствами телеконференции.
  
Skype для бизнеса Server теперь позволяет интегрироваться с определенными сторонними решениями VTC (Video Teleconferencing System). Новая роль сервера, которая позволяет использовать это видеоконференциалы, — это video Interop Server (VIS), который в настоящее время реализуется в качестве роли автономных серверов, доступных только для локальной установки. VIS выступает в качестве посредника между стороной системы телеконференции и Skype для бизнеса Server развертывания. Для этого выпуска VIS ориентирована на взаимооперабельность с видеосистемами Cisco/Tandberg. Просмотрите эту статью, чтобы определить, следует ли использовать эту функцию в Skype для бизнеса Server установке.
  
## <a name="device-interoperability"></a>Интероперабельность устройств

Взаимодействие тестируется и поддерживается с помощью VTCs Cisco, зарегистрированных в Cisco Unified Communications Manager (CallManager или CUCM) версии 10.5 и магистралях SIP TCP, установленных между CUCM и VIS.
  
Поддерживаемые в настоящее время VTCs:
  
- Cisco C40
    
- Cisco C60
    
- Cisco C90
    
- Cisco MX200
    
- Cisco MX300
    
- Cisco DX80
    
- Cisco EX60
    
- Cisco EX90
    
- Cisco SX20
    
> [!NOTE]
>  Выпуск программного обеспечения Cisco TC7.0.0 или выше необходим для этих систем для интеграции с Skype для бизнеса Server, чтобы работать как ожидалось.
  
## <a name="sip-trunks"></a>Магистрали SIP

Функции сервера межоперации видеосвязи в режиме магистрали SIP, где VTCs продолжают регистрироваться в существующей инфраструктуре Cisco , например, Cisco Call Manager (CUCM). Магистраль SIP видео определяется между CUCM и VIS, чтобы между двумя системами можно было перенаправлищать вызовы. Поддерживаются только вызовы по магистрали SIP из VTC в VIS. Таким образом, VTCs могут звонить на конференцию Skype для бизнеса (набрав номер телефона, связанный с автоматическим помощником вызова), но не может быть перетаскивание и выброшено в конференцию.
  
![Схема VIS в SfB](../media/87753af5-b1d9-4107-9216-fde45a1af197.png)
  
## <a name="features"></a>Функции

Эта роль сервера обеспечивает:
  
- Преобразование форматов H.264, используемых сторонними видеосистемами, и Skype для бизнеса Server развертывания.
    
- Преобразование одного видеопотока в заданном разрешении из ВТС в несколько потоков simulcast различных разрешений для использования в Skype для бизнеса Server развертывании. Эти потоки можно отправить в AVMCU, а затем Skype для бизнеса Server конечные точки и другие видеосистемы, запросив различные разрешения. Это преобразование также используется, когда сторонние видеосистемы участвуют в Skype для бизнеса конференции A/V. После того, как предел транскодинга будет достигнут на определенном сервере VIS, любые следующие запросы на различные разрешения будут получать только поток с наименьшим разрешением. 
    
- Поддержка магистрали SIP видео между шлюзом CUCM и сервером Skype для бизнеса Server межопера; VTCs продолжают регистрироваться в шлюзе Cisco и инициировать вызовы Skype для бизнеса развертывания через шлюз. Вызовы перенаправлищаются из шлюза в Skype для бизнеса сервер видео-интероп-сервера через магистраль SIP видео.
    
- Поддержка пользователя в конференц-зале с поддерживаемой видеосистемой для набора номера из этой системы, чтобы присоединиться к открытой или закрытой конференции. Этот вызов будет проходить магистраль SIP видео.
    
- Поддержка пользователя в конференц-зале с поддерживаемой видеосистемой для вызова Skype для бизнеса клиента. Вызов будет проходить магистраль SIP.
    
- Поддержка управления промежуточным вызовом со стороны Skype для бизнеса Server или из поддерживаемой системы VTC для точековых и многооконченных вызовов, включая звук без звука, паузу или возобновление видео, видео блокировки и вызов удержания и удержания.
    
## <a name="known-limitations"></a>Известные ограничения

Эта роль сервера имеет следующие ограничения:
  
- Новые вызовы Skype для бизнеса развертывания в VTCs над магистралью SIP видео не поддерживаются. . Это означает, что только новые вызовы из VTCs в развертывание Skype для бизнеса поддерживаются через магистраль SIP видео. Присутствие для поддерживаемой видеосистемы не будет доступно через магистраль SIP видео для VIS. 
    
- Только автономный пул VIS будет поддерживаться для режима магистрали SIP видео.
    
-  TLS + SRTP или TCP + RTP будут поддерживаться для связи между VTC и VIS над магистралью SIP видео.
    
- Общий доступ к приложениям не поддерживается. Пользователю Skype для бизнеса конференц-зала необходимо присоединиться к Skype для бизнеса конференции (например, с помощью ноутбука) и отобразить экраны общего доступа к приложениям на одном из бесплатных мониторов в конференц-зале, не связанных с ВТС.
    
- Возможность для ВТС присоединиться к федератовской встрече с помощью VIS не поддерживается.
    
- Возможность для ВТС присоединиться к собранию через VIS не поддерживается.
    
- Звонки из ВТС в ТСТС через VIS не поддерживаются.
    
- Звонки из ТСТС в VTC через VIS не поддерживаются.
    
## <a name="resiliency-mechanisms"></a>Механизмы устойчивости
<a name="resiliency"> </a>

VIS поддерживает входящие вызовы из CUCM, которые осуществляются через магистраль SIP видео. Возможно потерять подключение либо вверх по течению, либо вниз по течению, поэтому для надежной устойчивости рассмотрите обе возможности:
  
1. **Сбой пула VIS** Если основной пул VIS, на который указывает шлюз видео, не работает, восстановление возможно, если шлюз видео определил магистрали до двух (или более) пулов VIS. Если шлюз видео определяет, что он не может звонить в основной пул VIS, он просто передает вызовы во вторичный пул VIS.
    
     ![Схема сбой пула VIS](../media/390d93c3-e132-4bbd-8d5a-c70ead9cdfad.png)
  
    У определенного пула VIS могут быть магистрали для нескольких шлюзов, но обычно у определенного шлюза не могут быть магистрали для нескольких пулов VIS, поэтому для поддержки этого сбойного решения необходимо сделать трюк: определение 2 FDQNs в DNS, которые решаются на один и тот же IP-адрес видео шлюза. Представлять каждый FQDN в качестве отдельного шлюза видео в документе топологии, где у каждого шлюза видео есть магистраль для другого пула VIS, и теперь возможно восстановление. (Если используется TLS, несколько имен должны быть в san сертификата шлюза видео.)
    
    > [!NOTE]
    > VIS разрешает только входящие вызовы из шлюзов, настроенных в документе топологии. 
  
2. **Сбой в передней части** Если пул VIS получает вызов от CUCM, но не может достичь основного регистратора следующего прыжка или пула переднего конца, вызовы будут переназевку в резервный пул переднего конца.
    
     ![Схема сбой переднего конца](../media/6ddc08ec-4708-4c23-9e77-0f88899a2a96.png)
  
    VIS будет отслеживать состояние основного пула переднего конца и резервного пула переднего конца (параметр находится в параметре резервного копирования для службы регистратора в документе топологии). Он отправляет опросы Параметры один раз в минуту в оба пула, и при пяти последовательных сбоях VIS предполагает, что определенный пул переднего конце вниз. Если основной пул переднего конца помечен как вниз и имеется доступное настроенное резервное копирование, VIS отправляет новые вызовы из шлюза в резервный пул переднего конца. После того как основной пул переднего конца возвращается, VIS возобновит использование основного пула переднего конца для новых вызовов.
    
    ViS также реализует 10-секундный отсев для вызовов из магистрали SIP видео. Если для вызова из магистрали SIP видео использовался основной пул переднего прыжка следующего прыжка, а основной пул фронт-хопа не отвечал некоторым SIP-сообщением (в том числе 100 Trying) на приглашение, отправленное ему в течение этого значения времени, необходимо попробовать резервное копирование следующего прокси-сервера для вызова при настройке. 
    
    > [!NOTE]
    > Если сначала была опробована резервная копия следующего прыжка, основная не будет опробована далее. 
  
    Администратор также может использовать команду Windows PowerShell, чтобы заставить VIS использовать резервный пул переднего интерфейса, например, когда необходимо выполнять техническое обслуживание в основном пуле переднего конца.
    
## <a name="co-existence-of-voice-and-video-trunks-to-the-same-gateway-peer"></a>Совместное существование магистрали голосовой и видеосвязи в одном шлюзовом одноранговом шлюзе
<a name="resiliency"> </a>

Skype для бизнеса Server поддерживает, что магистрали SIP голосовых и видеосвязей используют один и тот же одноранговой шлюз. Таким образом, то же развертывание CUCM может иметь магистрали голосовых SIP для сервера-посредника и магистрали SIP видео в VIS.
  
- Шлюз PSTN необходимо определить с определенным FQDN в документе топологии для магистральных магистрали SIP голосовой связи.
    
- Одноранговой службой шлюза PSTN будет сервер-посредник.
    
- При необходимости можно определить несколько магистральных голосовых магистрали, охватывающих от шлюза PSTN до нескольких пулов серверов-посредников.
    
- В документе топологии для магистрали SIP видео с тем же FQDN, что и для шлюза PSTN, необходимо определить шлюз видео.
    
- Одноранговой шлюз будет VIS.
    
- Один магистраль видео можно определить от шлюза видео до определенного пула VIS.
    
- CUCM необходимо настроить для правильного маршрута вызовов через магистраль голосовой почты и магистраль видео. Например, при наборе из ВТС можно использовать специальную префиксу набора; CUCM может связать этот префикс набора с вызовами в VIS, и соответствующие правила перевода лидируют этот префикс из SIP Invite в VIS.
    
## <a name="co-existence-of-vis-in-the-skype-for-business-release-with-previous-releases-of-lync"></a>Совместное существование VIS в выпуске Skype для бизнеса с предыдущими выпусками Lync
<a name="resiliency"> </a>

VIS можно развернуть только в рамках Skype для бизнеса развертывания. Он может работать с конференциями и клиентами Lync 2013, которые являются частью существующего развертывания; в этих случаях пул VIS должен быть частью развертывания Skype для бизнеса, которое включает пул Registrar/FE, который является следующим переходом для пула VIS.
  
VIS не поддерживает транскодинг между RTV и H.264. Во время конференции между клиентами до Lync 2013 и участниками VTC не существует видеосвязи.
  
Наличие клиентов до Lync 2013 в конференции приведет к отправке мобильных клиентов с помощью RTV, в результате чего VTCs не получают видео, когда мобильный клиент становится доминирующим динамиком.
  
Для правильной работы Lync 2013 с VIS, которая является частью развертывания Skype для бизнеса, Lync 2013 необходимо применить соответствующий ТС, который обновляет клиент Lync 2013, CAA и AVMCU для работы с VIS.
  
Проверена и поддерживается интероперабельность VIS с Lync 2013 и Skype для бизнеса настольных клиентов.
  
Интероперабельность VIS с нестандартными устройствами (Android, Ipad, Iphone, Windows Phone, LMX и т.д.) Skype для бизнеса клиенты, доступные в магазине приложений на момент выпуска VIS, были протестированы и поддерживаются.
  
## <a name="recovery-from-packet-loss-via-fec"></a>Восстановление после потери пакета с помощью FEC
<a name="resiliency"> </a>

FEC можно использовать для оказания помощи в восстановлении после потери пакета. Если включено, на 50% больше пропускной способности видео будет использоваться в направлении VIS к VTC.
  
## <a name="vis-sizing-and-transcoding-costs"></a>VIS Sizing and Transcoding Costs
<a name="resiliency"> </a>

Для передачи одного видеопотока из VTC Cisco в несколько потоков simulcast используется емкость ЦП. Примерно 16 VTCs могут иметь свои видеотранскодированные (при условии, что видеопоток 720p из каждого VTC транскодирован в 3 отдельных потоков simulcast на 720p, 360p и 180p) в одном VIS, запущенном на эквиваленте рекомендуемой платформы FE Lync 2013. Если транскодинг отключен, это сэкономит на ЦП VIS. Однако видеосъемка, запрашиваемая VIS из VTC, будет самым низким общим разрешением, удовлетворяющих все приемники на Skype для бизнеса стороне. Обратите внимание, что даже при выключенном транскодировании может быть активирована транскодированная Skype для бизнеса клиенты запрашивают определенные низкие разрешения, которые не могут отправить VTCs.
  
## <a name="call-distribution-from-the-video-gateway-to-vis"></a>Распределение вызовов из видео шлюза в VIS
<a name="resiliency"> </a>

Распределение осуществляется с помощью одного из механизмов распределения CUCM:
  
- Динамически с помощью DNS.
    
- На стороне CUCM можно определить отдельные магистрали, где каждый магистраль завершается на другом сервере в пуле VIS. CUCM будет маршрутить вызовы по различным магистральм.
    
## <a name="no-hybrid-interoperability"></a>Нет гибридной интероперабельности
<a name="resiliency"> </a>

Поддержка присоединения VTCs к онлайн-собраниям с помощью локального VIS не является частью Skype для бизнеса.
  
## <a name="no-federation-support"></a>Нет поддержки федерации
<a name="resiliency"> </a>

Поддержка участия VTCs в федерадных собраниях с помощью VIS не является частью Skype для бизнеса.
