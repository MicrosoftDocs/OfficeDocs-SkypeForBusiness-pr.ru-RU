# <a name="plan-for-video-interop-server-in-skype-for-business-server"></a>Планирование сервера видеосвязи в Skype для бизнеса Server
 
**Сводка:** Просмотрите этот раздел при планировании интеграции Skype для бизнеса Server со сторонними устройствами телеконференции.
  
Skype для бизнеса Server теперь позволяет интегрироваться с определенными сторонними решениями VTC (система видеоконференции). Новая роль сервера, которая обеспечивает такое межсерверное видеоконференцию, — это сервер видеосвязи (VIS), который в настоящее время реализован в качестве роли автономных серверов, доступной только для локальной установки. ViS действует как посредник между стороной системой телеконференции и развертыванием Skype для бизнеса Server. В этом выпуске VIS ориентирован на возможности обеспечения связи с видеосистемами Cisco/Tandberg. Просмотрите эту статью, чтобы определить, следует ли использовать эту функцию в установке Skype для бизнеса Server.
  
## <a name="device-interoperability"></a>Device interoperability

Взаимодействие тестируется и поддерживается при регистрации cisco VTCs в Cisco Unified Communications Manager (CallManager или CUCM) версии 10.5 и магистралях TCP SIP, установленных между CUCM и VIS.
  
В настоящее время поддерживаются такие компьютеры:
  
- Cisco C40
    
- Cisco C60
    
- Cisco C90
    
- Cisco MX200
    
- Cisco MX300
    
- Cisco DX80
    
- Cisco EX60
    
- Cisco EX90
    
- Cisco SX20
    
> [!NOTE]
>  Для работы интеграции со Skype для бизнеса Server в этих системах требуется выпуск программного обеспечения Cisco TC7.0.0 или более высокой версии.
  
## <a name="sip-trunks"></a>Магистрали SIP

Сервер видеосвязи работает в режиме магистрали SIP, где VTCs продолжают регистрироваться в существующей инфраструктуре Cisco, например Cisco Call Manager (CUCM). Магистраль SIP видео определяется между CUCM и VIS, чтобы вызовы можно было перенапраздить между двумя системами. Поддерживаются только вызовы через магистраль SIP из VTC в VIS. Таким образом, VTCs могут звонить в конференцию Skype для бизнеса (набирая номер телефона, связанный с автоматическим помощником по вызову), но не могут быть перетаскиванием и перетаскиванием в конференцию.

## <a name="features"></a>Возможности

Эта роль сервера предоставляет:
  
- Преобразование форматов H.264, используемых сторонними видеосистемами, и развертыванием Skype для бизнеса Server.
    
- Преобразование одного видеопотока в заданном разрешении из VTC в несколько однонамерных потоков различных разрешений для использования в развертывании Skype для бизнеса Server. Эти потоки можно отправить в AVMCU, а затем в конечные точки Skype для бизнеса Server и другие видеосистемы, запросив различные разрешения. Это преобразование также используется, когда сторонная видеосистема участвует в конференции Skype для бизнеса A/V. После того как на определенном сервере VIS достигнут предел перекодинга, все следующие запросы для разных разрешений будут получать поток только с самым низким разрешением. 
    
- Поддержка видеоМАПК между шлюзом CUCM и сервером видеосвязи Skype для бизнеса Server; VTCs continue to register with the Cisco gateway, and initiate calls to the Skype for Business deployment through the gateway. Вызовы перенапращаются со шлюза на сервер видеосвязи Skype для бизнеса через магистраль видеосвязи SIP.
    
- Поддержка того, чтобы пользователь в конференц-зале с поддерживаемой видеосистемой набирал номер из этой системы, чтобы присоединиться к открытой или закрытой конференции. Этот вызов проходит магистраль SIP видео.
    
- Поддержка вызова клиента Skype для бизнеса пользователем в конференц-зале с поддерживаемой видеосистемой. Вызов проходит магистраль SIP.
    
- Поддержка управления вызовами в середине звонка со стороны Skype для бизнеса Server или из поддерживаемой системы VTC для вызовов точки-точки и многоточки, включая звук, приостановку/возобновление видео, блокировку видео и вызов удержания и удержания.
    
## <a name="known-limitations"></a>Известные ограничения

Эта роль сервера имеет следующие ограничения:
  
- Новые вызовы из развертывания Skype для бизнеса на VTCs через видеомаголь SIP не поддерживаются. . Это означает, что только новые вызовы из VTCs в развертывание Skype для бизнеса поддерживаются через видеоМАПК. Присутствие для поддерживаемой видеосистемы не будет доступно через магистраль SIP видео для VIS. 
    
- Только автономный пул VIS будет поддерживаться для режима видеоМайловой магистрали SIP.
    
-  TLS + SRTP или TCP + RTP будут поддерживаться для связи между VTC и VIS через магистраль SIP видео.
    
- Общий доступ к приложениям не поддерживается. Пользователь Skype для бизнеса в конференц-зале должен присоединиться к конференции Skype для бизнеса (например, с помощью ноутбука) и отобразить экраны общего доступа к приложениям на одном из бесплатных мониторов в конференц-зале, не связанном с VTC.
    
- Возможность для VTC присоединяться к федератированному собранию через VIS не поддерживается.
    
- Возможность для VTC присоединяться к собранию по сети через VIS не поддерживается.
    
- Вызовы из VTC в ТСП через VIS не поддерживаются.
    
- Вызовы из ТСТС в VTC через VIS не поддерживаются.
    
## <a name="resiliency-mechanisms"></a>Механизмы устойчивости
<a name="resiliency"> </a>

ViS поддерживает входящие вызовы из CUCM, которые осуществляются через видеомаголь SIP. Подключение можно потерять либо вверх, либо вниз, поэтому для надежной устойчивости рассмотрите обе возможности:
  
1. **Отбой пула VIS** Если основной пул VIS, на который указывает видео шлюз, не работает, восстановление возможно, если видео шлюз определил магистрали для двух (или более) пулов VIS. Если видео шлюз определяет, что он не может звонить основному пулу VIS, он просто перенананошет вызовы на дополнительный пул VIS.
    
     ![Схема от сбойного пула VIS](../media/390d93c3-e132-4bbd-8d5a-c70ead9cdfad.png)
  
    Определенный пул VIS может иметь магистрали для нескольких шлюзов, но обычно определенный шлюз не может иметь магистрали к нескольким пулам VIS, поэтому для поддержки этой отладки необходимо решить проблему: определите 2 FDQNs в DNS, которые будут разрешать один и тот же IP-адрес видео шлюза. Представляете каждое FQDN как отдельный видео шлюз в документе топологии, где каждый видео шлюз имеет магистраль для другого пула VIS, и теперь возможно восстановление. (Если используется TLS, несколько имен должны быть в san сертификата видео шлюза.)
    
    > [!NOTE]
    > ViS разрешает входящие вызовы только от шлюзов, настроенных в документе топологии. 
  
2. **Отбой на переднем конце** Если пул VIS получает вызов от CUCM, но не может связаться с основным регистратором следующего прыжка или пулом переднего планирования, вызовы перена маршрутизются в резервный пул переднего планирования.
    
     ![Схема от сбойного перенаправки на переднем конце](../media/6ddc08ec-4708-4c23-9e77-0f88899a2a96.png)
  
    ViS будет отслеживать состояние основного пула переднего план и его резервного пула переднего копирования (параметр находится в параметре резервного копирования для службы регистратора в документе топологии). Она отправляет опросы options один раз в минуту в оба пула, и при пяти последовательных сбоях VIS предполагает, что определенный пул переднего конца неабит. Если основной пул переднего звена помечен как неавратный и имеется доступная настроенная резервная копия, VIS отправляет новые вызовы из шлюза в резервный пул переднего план. После того как основной пул переднего звена вернется, VIS возобновит использование основного пула переднего входа для новых вызовов.
    
    ViS также реализует 10-секундный timer для вызовов из видеомаголи SIP. Если основной пул переднего сервера следующего прыжка использовался для вызова из видеомагисковой магистрали SIP, а основной пул переднего сервера следующего прыжка не ответил каким-либо SIP-сообщением (включая 100 Попыток) на приглашение, отправленное ему в рамках этого значения времени, при настройке следует попробовать резервный прокси-сервер следующего прыжка для вызова. 
    
    > [!NOTE]
    > Если следующий переход резервного копирования был перепробовано первым, основной этап не будет перенамеряться далее. 
  
    Администратор также может использовать команду Windows PowerShell для принудительного использования viS резервного пула переднего интерфейса, например при выполнении обслуживания основного интерфейсного пула.
    
## <a name="co-existence-of-voice-and-video-trunks-to-the-same-gateway-peer"></a>Совместное существование голосовых и видеоманкетов в одноранговом шлюзе
<a name="resiliency"> </a>

Skype для бизнеса Server поддерживает использование одного одноранговых шлюзов голосовой и видеомагольниками SIP. Таким образом, в том же развертывании CUCM могут быть голосовые магистрали SIP для сервера-посредника и видеоМАПК для VIS.
  
- Шлюз STN необходимо определить с определенным FQDN в документе топологии для магистрали SIP голосовой связи.
    
- Одноранговая служба шлюза STN будет сервером-посредником.
    
- При необходимости можно определить несколько голосовых магистрали от шлюза STN до нескольких пулов серверов-посредников.
    
- Видео шлюз необходимо определить в документе топологии для видеомайловых магистрали SIP с таким же FQDN, что и для шлюза STN.
    
- Одноранговой возможностью видео шлюза будет VIS.
    
- Одну видеомаголь можно определить из видео шлюза в определенный пул VIS.
    
- CUCM необходимо настроить для правильной маршрутки вызовов по голосовой магистрали и видеомайл. Например, при наборе номера из VTC можно использовать специальный префикс набора номера; CUCM может связать этот префикс набора номера с вызовами VIS, и соответствующие правила трансляции будут отбирать этот префикс из приглашения SIP в VIS.
    
## <a name="co-existence-of-vis-in-the-skype-for-business-release-with-previous-releases-of-lync"></a>Сосуществование VIS в выпуске Skype для бизнеса с предыдущими выпусками Lync
<a name="resiliency"> </a>

ViS можно развернуть только в рамках развертывания Skype для бизнеса. Он может работать с конференциями и клиентами Lync 2013, которые являются частью существующего развертывания; В таких случаях пул VIS должен быть частью развертывания Skype для бизнеса, которое включает пул Регистратора или FE, который является следующим прыжком для пула VIS.
  
VIS не поддерживает перекодвание между RTV и H.264. Видеосвязь между клиентами до Lync 2013 и участниками VTC в конференции не существует.
  
Если клиенты до Lync 2013 будут принимать участие в конференции, мобильные клиенты будут отправлять сообщения с помощью RTV, в результате чего VTCs не будут получать видео, когда мобильный клиент станет основным динамиком.
  
Для правильной работы Lync 2013 с VIS, который входит в состав развертывания Skype для бизнеса, Lync 2013 необходимо применить соответствующий cu, который обновляет клиент Lync 2013, CAA и AVMCU для работы с VIS.
  
Было протестировано и поддерживается межсервиативность VIS с классическими клиентами Lync 2013 и Skype для бизнеса.
  
Interoperability of VIS with non-desktop (Android, Ipad, Iphone, Windows Phone, LMX, etc.) Клиенты Skype для бизнеса, доступные в применимом Магазине приложений на момент выпуска VIS, протестировали и поддерживаются.
  
## <a name="recovery-from-packet-loss-via-fec"></a>Восстановление после потери пакетов через FEC
<a name="resiliency"> </a>

FEC может быть включен для помощи в восстановлении после потери пакетов. Если этот режим включен, на 50 % больше пропускной способности видео будет использоваться в направлении VIS в VTC.
  
## <a name="vis-sizing-and-transcoding-costs"></a>Размер и перекодирующая стоимость VIS
<a name="resiliency"> </a>

Перекодвание одного видеопотока из Cisco VTC в несколько одновместных потоков использует емкость ЦП. Приблизительно 16 видеоконферентов могут иметь перекодированную видеопоток (предполагается, что видеопоток 720p из каждой VTC перекодирован в 3 отдельных одновластиковых потока при 720p, 360p и 180p) в одном viS, запущенном на эквиваленте рекомендуемой платформе FE Lync 2013. Если перекодвание отключено, это сохранится на ЦП VIS. Однако видеосъемка, запрашиваемая VIS из VTC, будет самым низким общим разрешением для удовлетворения всех приемников на стороне Skype для бизнеса. Обратите внимание, что даже при отключенной перекодировании перекодировать можно, когда клиенты Skype для бизнеса запрашивают определенные разрешения с низким разрешением, которые не могут отправлять компьютеры.
  
## <a name="call-distribution-from-the-video-gateway-to-vis"></a>Распределение вызовов из видео шлюза в VIS
<a name="resiliency"> </a>

Распространение осуществляется с помощью одного из механизмов распространения CUCM:
  
- Динамическое использование DNS.
    
- На стороне CUCM можно определить отдельные магистрали, где каждая магистраль завершается на разных серверах в пуле VIS. CUCM будет маршрутизовы вызовов через различные магистрали.
    
## <a name="no-hybrid-interoperability"></a>Отсутствие гибридного interoperability
<a name="resiliency"> </a>

Поддержка присоединений КТЗ к собраниям по сети через локальное viS не входит в состав Skype для бизнеса.
  
## <a name="no-federation-support"></a>Поддержка федерации не поддерживается
<a name="resiliency"> </a>

Поддержка присоединений КТЗ к федератным собраниям через VIS не входит в состав Skype для бизнеса.
