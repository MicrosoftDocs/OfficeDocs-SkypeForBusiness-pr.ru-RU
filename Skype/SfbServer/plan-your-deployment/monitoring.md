# <a name="plan-for-monitoring-in-skype-for-business-server"></a>Планирование мониторинга в Skype для бизнеса Server

**Сводка:** Просмотрите эту тему при планировании службы мониторинга в Skype для бизнеса Server.

Служба мониторинга в Skype для бизнеса Server предоставляет администраторам возможность собирать данные об использовании и качестве для сеансов связи, которые проходят в их организации, что позволяет им определять тенденции и проблемы. Постоянный мониторинг развертывания позволяет на ранних стадиях ловить проблемы и поддерживать удовлетворенные пользователи организации.

Мониторинг в Skype для бизнеса Server не требует отдельной роли сервера (как это было в предыдущих версиях Lync); вместо этого служба мониторинга встроена в каждый сервер переднего конца. Мониторинг не включен по умолчанию в Skype для бизнеса Server. В этой статье вы сможете определить, следует ли включить мониторинг во время или после начальной конфигурации Skype для бизнеса Server и какие SQL ресурсы, необходимые для поддержки действий мониторинга. Если вы не уверены в том, что именно контролируется или не контролируется, и как мониторинг может быть полезен, перейдите к [Basics about Monitoring](monitoring.md#Basics). Чтобы начать процесс планирования, перейдите к [определению ваших требований к мониторингу.](monitoring.md#requirements) Дополнительные сведения о требованиях SQL мониторинга перейдите к SQL [для мониторинга.](monitoring.md#topologies)

## <a name="basics-about-monitoring"></a>Основы мониторинга
<a name="Basics"> </a>

Сеанс — это общий термин для подключения пользователя к:

- Conference

- Средство conferencing, например Audio/Video или Application Sharing

- Другой пользователь через одноранговое общение, например обмен мгновенными сообщениями или аудиовызов

> [!NOTE]
> Skype для бизнеса Server отслеживает сведения о каждом сеансе: кто звонил; какие конечные точки использовались в сеансе; как долго длился сеанс; то, как воспринималось качество сеанса; и так далее. Skype для бизнеса Server не регистрирует и не сохраняет сам вызов. Это включает сеансы обмена мгновенными сообщениями: хотя Skype для бизнеса Server записи сведений о сеансах обмена мгновенными сообщениями, она не ведет записи каждого мгновенного сообщения, отправленного во время сеанса.

Основные сведения о детализации вызовов, собранные Skype для бизнеса Server для каждого сеанса, можно использовать для:

- **Анализ возврата инвестиций (ROI).** Администраторы могут сравнить данные об использовании с аналогичными данными, собранными для их предыдущей системы телефонии, чтобы показать экономию затрат и помочь оправдать развертывание Skype для бизнеса Server.

- **Инвентаризация устройств**. Сведения об управлении активами помогают администраторам выявлять старые устройства, которые еще используются, которые необходимо заменить, и выявлять дорогостоящие устройства, которые не используются или не используются.

- **Служба технической поддержки**. Устранение неполадок помогает инженерам определить причину сбоя вызова пользователя без необходимости собирать журналы сторон сервера или клиента. Эти сведения могут быть легко доступны и понятны сотрудникам службы поддержки, которые не имеют глубоких технических знаний о клиенте Skype для бизнеса и Skype для бизнеса Server.

- **Устранение системных неполадок.**. Администраторы могут определять серьезные неполадки, которые приведут к невозможности выполнения базовых задач пользователями, например присоединяться к конференции, выполнять вызов или отправлять мгновенное сообщение.

Мониторинг также предоставляет механизм, позволяющий конечным точкам SIP (например, Skype для бизнеса) предоставлять сведения о устранении неполадок, к которые администратор в противном случае не мог бы получить доступ:

- **Показатели мультимедиа, влияющие на качество**. Эти показатели занимаются фактической передачей самого вызова; они предоставляют своего рода travelogue как путешествие по вызову по сети. В этих метриках (включая потери пакетов, испуг и время в пути) содержатся сведения о том, что произошло с вызовом с того времени, когда он оставил конечную точку одного человека до времени его прибытия в конечную точку другого человека.

- **Ошибки, о которых было сообщено пользователю**. Эти показатели включают уведомления низкого качества, которые Skype для бизнеса для конечных пользователей в тех случаях, когда они слишком далеки от микрофона, слишком мягко говоря, имеют плохое сетевое подключение или испытывают плохое качество, так как другая программа на компьютере потребляет доступные ресурсы.

- **Сведения о среде**. Сюда входят показатели, которые описывают факторы качества связи, такие как используемый тип микрофона и динамиков, подключается ли пользователь через VPN, используется ли беспроводное подключение.

В конце каждого вызова конечные точки, совместимые с SIP, передают эти сведения на сервер переднего конца, облегчив вызов. Нет необходимости предпринимать какие-либо действия для передачи этих сведений с конечных точек, так как подобное поведение характерно для протокола SIP. Однако если необходимо собрать и сохранить эти данные, необходимо установить и включить возможность мониторинга. В этом случае сведения о вызовах будет собираться агентами, работающими на интерфейсном сервере, и заноситься в две базы данных SQL Server. Служба мониторинга (в виде "агентов единого сбора данных") находится на всех серверах переднего конца.

## <a name="define-your-requirements-for-monitoring"></a>Определение требований к мониторингу
<a name="requirements"> </a>

Перед установкой и настройкой мониторинга с помощью Skype для бизнеса Server необходимо решить несколько ключевых Skype для бизнеса Server.

 **Когда вы хотите установить систему мониторинга?** Мониторинг можно установить и настроить одновременно с установкой и настройкой Skype для бизнеса Server; Мастер Skype для бизнеса Server развертывания предоставит вам возможность связать пулы переднего конца с базой данных мониторинга во время установки. Кроме того, после установки Skype для бизнеса Server можно установить мониторинг; Это можно сделать с помощью Topology Builder, чтобы связать пулы и серверы переднего конца с базой данных мониторинга, а затем опубликовать пересмотренную топологию.

Помните, что SQL Server необходимо установить и настроить до развертывания и настройки мониторинга. Однако необходимо развернуть только SQL Server себя; базы данных мониторинга будут созданы для вас при публикации топологии Skype для бизнеса Server.

 **Какие данные необходимо отслеживать?** Skype для бизнеса Server позволяет отслеживать два общих типа данных: данные записи фиксации вызовов (CDR) и данные качества работы (QoE). Запись детализации вызовов позволяет отслеживать использование Skype для бизнеса Server, таких как телефонные вызовы Voice over IP (VoIP). обмен мгновенными сообщениями (IM); перенос файлов; аудио- и видеоконференция; и сеансы совместного использования приложений. Эти сведения помогают узнать, какие Skype для бизнеса Server используются (а какие нет), а также предоставляют сведения о том, когда эти функции используются. Качество данных experience позволяет вести учет качества звуковых и видеозвонков, сделанных в организации, включая такие вещи, как количество потерянных сетевых пакетов, фоновый шум и количество "испуг" (различия в задержке пакетов).

Если вы решите включить мониторинг в Skype для бизнеса Server вы можете включить как мониторинг CDR, так и мониторинг QoE, или вы можете включить один тип мониторинга, оставив при этом другой тип отключен. Например, предположим, что пользователи используют только систему обмена мгновенными сообщениями и функции передачи файлов, и не осуществляют аудио- и видеовызовы. В этом случае нет особого смысла в активации мониторинга QoE. Кроме того, Skype для бизнеса Server позволяет легко включить и отключить мониторинг после развертывания мониторинга. Например, вы можете выбрать развертывание системы мониторинга, но изначально оставить функцию мониторинга QoE отключенной. Если у пользователей начинают возникать проблемы, связанные с аудио- и видеовызовами, вы можете в последствии включить функцию мониторинга QoE и использовать эти данные для устранения неполадок и решения этих проблем.

Нет особых преимуществ (или недостатков) для установки мониторинга одновременно с установкой Skype для бизнеса Server и установки мониторинга после Skype для бизнеса Server установлен. Следует иметь в виду, что перед установкой мониторинга необходимо выбрать компьютер, на который будет установлен хранилище мониторинга заднего приложения, а поддерживаемая версия SQL Server должна быть установлена и настроена на этом компьютере, прежде чем этот компьютер можно использовать для мониторинга. Если вы уже SQL Server на компьютере и компьютер готов к использованию, вы можете установить мониторинг одновременно с установкой Skype для бизнеса Server. Если у вас нет готового заднего компьютера, вы можете приступить к установке Skype для бизнеса Server самостоятельно, а затем установить мониторинг всякий раз, когда дополнительный компьютер готов к использованию.

 **Сколько серверных баз данных мониторинга требуется?** По оценкам, база данных для мониторинга и архива может поддерживать 240 000 Skype для бизнеса Server пользователей. Кроме того, единая база данных мониторинга может использоваться несколькими пулами переднего плана; если в организации имеются три пула переднего плана, можно связать все эти три пула с одним и тем же серверным хранилищем.

Для многих организаций емкость баз данных не будет решающим фактором при определении количества необходимых баз данных мониторинга backend. Более важным фактором может быть скорость сети. Предположим, у вас имеются три пула переднего плана, но один из этих пулов базируется на сетевом подключении с более низкой скоростью. В этом случае может потребоваться использование двух баз данных мониторинга: одна база данных предназначена для обслуживания двух пулов с высокоскоростным сетевым подключением, а отдельная база данных — для обслуживания пула с низкоскоростным сетевым подключением.

Следует также учитывать, что Skype для бизнеса Server поддерживает использование зеркальных баз данных. «Зеркалирование баз данных» предоставляет возможность одновременно поддерживать две копии базы данных, причем эти базы данных размещаются на разных серверах. При каждой записи сведений в основную базу данных эти же самые данные также записываются в зеркальную базу данных. Если базовая база данных должна быть отключена или иным образом станет недоступной, можно "перенаправить" в зеркальную базу данных с помощью простой команды Skype для бизнеса Server PowerShell. Пример:

```PowerShell
Invoke-CsDatabaseFailover -PoolFqdn atl-cs-001.litwareinc.com -DatabaseType "Monitoring" -NewPrincipal "Mirror"
```

Это имеет большое значение при планировании, так как для использования зеркалирования потребуется дублирование необходимого количества баз данных: в дополнение к каждой основной базе данных вам потребуется вторая база данных, выступающая в качестве зеркала.

 **Нужны ли Skype для бизнеса Server сайты собственные настраиваемые конфигурации мониторинга?** При установке Skype для бизнеса Server также устанавливаются глобальные коллекции параметров конфигурации CDR и QoE; эти глобальные коллекции дают возможность применять одинаковые параметры CDR и QoE для всей организации. Во многих случаях этого достаточно: как правило, необходимо, чтобы, например, мониторинг CDR, был включен для всех пользователей.

Однако в некоторых случаях требуется применение различных настроек для разных сайтов. Например, предположим, вы хотите использовать функции мониторинга CDR и QoE на сайте Redmond, но использовать только мониторинг CDR на сайте Dublin. Аналогичным образом, предположим, что вы хотите хранить данные мониторинга в течение 60 дней на сайте Redmond, однако эти же данные требуется хранить на сайте Dublin в течение всего лишь 30 дней. Skype для бизнеса Server позволяет создавать отдельные коллекции параметров конфигурации CDR и QoE в области сайта; что позволяет по-разному управлять каждым сайтом. (Это подразумевает автономное включение и отключение мониторинга, а также настройку параметров управления, включая длительность хранения данных.)

Обратите внимание, что вы можете принять это решение как перед развертыванием функции мониторинга, так и после ее развертывания. Например, вы можете развернуть систему мониторинга, а затем осуществлять управление всей организацией с помощью глобальных параметров. Если вы впоследствии измените свое мнение, вы сможете создать отдельный набор настроек, например, для сайта Redmond, а затем использовать эти настройки для управления мониторингом сайта Redmond. (Настройки, применяемые на уровне сайта, всегда имеют приоритет над настройками, применяемыми на глобальном уровне.) Если еще раз измените свое решение, можно просто удалить параметры конфигурации, применяемые к сайту Redmond. При удалении набора настроек сайта к этому сайту автоматически применяется глобальный набор параметров.

## <a name="sql-requirements-for-monitoring"></a>SQL для мониторинга
<a name="topologies"> </a>

Унифицированные агенты по сбору данных автоматически устанавливаются и активируются на каждом переднем сервере при включении мониторинга. Поддерживаемые версии SQL Server и другие сведения см. в материале [Server requirements for Skype для бизнеса Server 2015 г.](requirements-for-your-environment/server-requirements.md)

Данные наблюдения могут совместно использовать экземпляр SQL Server с другими типами данных. Обычно база данных регистрации вызовов (LcsCdr) и база данных качества взаимодействия (QoEMetrics) совместно используют один экземпляр SQL; также распространены случаи, когда две базы данных мониторинга находятся в одном экземпляре SQL с базой данных архивации (LcsLog). Единственное реальное требование к экземплярам SQL Server заключается в том, что каждый экземпляр SQL Server должен иметь следующие ограничения.

- Один экземпляр базы данных Skype для бизнеса Server 2015 г. (Как правило, не рекомендуется, чтобы база данных мониторинга размещалась в том же экземпляре SQL или даже на том же компьютере, что и внутренняя база данных. Хотя технически это возможно, существует риск того, что база данных мониторинга займет пространство диска, необходимое для внутренней базы данных.)

- Один экземпляр базы данных регистрации вызовов.

- Один экземпляр базы данных качества взаимодействия.

- Один экземпляр базы данных архивации.

Другими словами, нельзя иметь два экземпляра базы данных LcsCdr в одном экземпляре SQL Server. Если требуется несколько экземпляров базы данных LcsCdr, необходимо настроить несколько экземпляров SQL Server.
