# <a name="plan-for-monitoring-in-skype-for-business-server"></a>Планирование мониторинга в Skype для бизнеса Server

**Сводка:** Просмотрите этот раздел при планировании службы мониторинга в Skype для бизнеса Server.

Служба мониторинга в Skype для бизнеса Server позволяет администраторам собирать данные об использовании и качестве для сеансов связи, которые проходят в организации, что позволяет им выявлять тенденции и проблемы. Постоянный мониторинг развертывания позволяет захватить проблемы на ранних стадиях и обеспечить пользователей вашей организации.

Для мониторинга в Skype для бизнеса Server не требуется отдельная роль сервера (как в предыдущих версиях Lync); вместо этого служба мониторинга встроена в каждый сервер переднего сервера. Мониторинг не включен по умолчанию в Skype для бизнеса Server. Эта статья поможет вам определить, следует ли включить мониторинг во время или после начальной настройки Skype для бизнеса Server, а также какие SQL ресурсы, необходимые для поддержки действий мониторинга. Если вы не знаете точно, что именно отслеживается или не отслеживается, и как мониторинг может быть полезен, перейдите к основам [мониторинга.](monitoring.md#Basics) Чтобы начать процесс планирования, перейдите к [процедуре "Определение требований для мониторинга".](monitoring.md#requirements) Дополнительные сведения о требованиях SQL мониторинга можно найти в SQL [мониторинга.](monitoring.md#topologies)

## <a name="basics-about-monitoring"></a>Основные принципы мониторинга
<a name="Basics"> </a>

Сеанс — это универсальный термин для подключения пользователя к:

- Conference

- Средство для аудио- и видеоконференций или общий доступ к приложениям

- Другой пользователь через одноранговое общение, например обмен мгновенными сообщениями или аудиовызов

> [!NOTE]
> Skype для бизнеса Server отслеживает сведения о каждом сеансе: кто кому позвонил; какие конечные точки использовались в сеансе; как долго был последний сеанс; что было воспринимано качеством сеанса; и так далее. Skype для бизнеса Server не записирует и не сохраняет сам вызов. В том числе сеансы обмена мгновенными сообщениями: хотя Skype для бизнеса Server записи сведений о сеансах обмена мгновенными сообщениями, он не ведет запись каждого мгновенного сообщения, отправленного во время сеанса.

Основные сведения о вызове, собираемой Skype для бизнеса Server для каждого сеанса, можно использовать для:

- **Анализ рентабельности инвестиций.** Администраторы могут сравнить данные об использовании с аналогичными данными, собранными для их предыдущей телефонной системы, чтобы показать экономию затрат и обосновать развертывание Skype для бизнеса Server.

- **Инвентаризация устройств**. Сведения об управлении активами помогают администраторам определить старые устройства, которые по-прежнему используются, которые необходимо заменить, и определить дорогостоящие устройства, которые не используются или не используются.

- **Help Desk**. Устранение неполадок с данными помогает инженерам службы поддержки определить причину сбоя вызова пользователя, не собирая журналы на стороне сервера или клиента. Эти сведения могут быть легко доступны и понятны сотрудникам службы поддержки, которые не имеют глубоких технических знаний о клиенте Skype для бизнеса и Skype для бизнеса Server.

- **Устранение системных неполадок.**. Администраторы могут определять серьезные неполадки, которые приведут к невозможности выполнения базовых задач пользователями, например присоединяться к конференции, выполнять вызов или отправлять мгновенное сообщение.

Мониторинг также предоставляет механизм, позволяющий конечным точкам SIP (например, Skype для бизнеса) предоставлять сведения об устранении неполадок, к которые администратору в противном случае не был бы доступ:

- **Показатели мультимедиа, влияющие на качество**. Эти показатели занимаются фактической передачей самого вызова; они предоставляют своего рода туристический маршрут по мере перемещения вызовов по сети. Эти метрики (в том числе потеря пакетов, дрожания и время кругового пути) предоставляют сведения о том, что произошло с вызовом с того времени, когда он оставил конечную точку одного человека до того, как он прибыл в конечную точку другого.

- **Ошибки, о которых было сообщено пользователю**. К этим метрикам относятся уведомления о плохом качестве, которые Skype для бизнеса представляет конечным пользователям в случаях, когда они слишком далеко от микрофона, говорят слишком тихо, имеют плохое сетевое подключение или испытывают неудовлетворительные качества, так как другая программа на компьютере потребляет доступные ресурсы.

- **Сведения о среде**. Сюда входят показатели, которые описывают факторы качества связи, такие как используемый тип микрофона и динамиков, подключается ли пользователь через VPN, используется ли беспроводное подключение.

В конце каждого вызова конечные точки, совместимые с SIP, передают эти сведения на сервер переднего сервера, который облегчил вызов. Нет необходимости предпринимать какие-либо действия для передачи этих сведений с конечных точек, так как подобное поведение характерно для протокола SIP. Однако если необходимо собрать и сохранить эти данные, необходимо установить и включить возможность мониторинга. В этом случае сведения о вызовах будет собираться агентами, работающими на интерфейсном сервере, и заноситься в две базы данных SQL Server. Служба мониторинга (в виде "агентов единого сбора данных") размещена на всех серверах переднего сервера.

## <a name="define-your-requirements-for-monitoring"></a>Определение требований к мониторингу
<a name="requirements"> </a>

Перед установкой и настройкой мониторинга в Skype для бизнеса Server необходимо решить ряд ключевых проблем.

 **Когда вы хотите установить систему мониторинга?** Мониторинг можно установить и настроить одновременно с установкой и настройкой Skype для бизнеса Server; Мастер развертывания Skype для бизнеса Server предоставит вам возможность связать пулы переднего сервера с базой данных мониторинга во время установки. Кроме того, вы можете установить мониторинг после установки самого Skype для бизнеса Server; Это можно сделать с помощью построитель топологий, чтобы связать пулы и серверы переднего сервера с базой данных мониторинга, а затем опубликовать измененную топологию.

Помните, что SQL Server необходимо установить и настроить до развертывания и настройки мониторинга. Однако необходимо развернуть только сам SQL Server; базы данных мониторинга будут созданы для вас при публикации топологии Skype для бизнеса Server.

 **Какой тип данных вы хотите отслеживать?** Skype для бизнеса Server позволяет отслеживать два общих типа данных: данные регистрации вызовов (CDR) и данные качества обслуживания (QoE). Регистрация вызовов позволяет отслеживать использование функций Skype для бизнеса Server, таких как телефонные звонки по voIP; обмен мгновенными сообщениями; передача файлов; аудио- и видеоконференцию; и сеансы общего доступа к приложениям. Эти сведения помогают узнать, какие функции Skype для бизнеса Server используются (а какие нет), а также предоставляют сведения о том, когда эти функции используются. Данные о качестве работы позволяют вести запись о качестве аудио- и видеозвонков, сделанных в организации, включая количество потерянных сетевых пакетов, фоновый шум и количество "дрожания" (различия в задержке пакетов).

Если вы хотите включить мониторинг в Skype для бизнеса Server, вы можете включить мониторинг CDR и QoE или включить один тип мониторинга, оставив другой отключенным. Например, предположим, что пользователи используют только систему обмена мгновенными сообщениями и функции передачи файлов, и не осуществляют аудио- и видеовызовы. В этом случае нет особого смысла в активации мониторинга QoE. Аналогично, Skype для бизнеса Server упрощает и отключает мониторинг после развертывания мониторинга. Например, вы можете выбрать развертывание системы мониторинга, но изначально оставить функцию мониторинга QoE отключенной. Если у пользователей начинают возникать проблемы, связанные с аудио- и видеовызовами, вы можете в последствии включить функцию мониторинга QoE и использовать эти данные для устранения неполадок и решения этих проблем.

Нет особых преимуществ (или недостатков) при установке мониторинга одновременно с установкой Skype для бизнеса Server и установкой мониторинга после установки Skype для бизнеса Server. Следует помнить, что перед установкой мониторинга необходимо выбрать компьютер, на который будет храниться это хранилище мониторинга, а поддерживаемая версия SQL Server должна быть установлена и настроена на этом компьютере, прежде чем этот компьютер можно будет использовать для мониторинга. Если вы уже установили SQL Server на компьютере и этот компьютер готов к использованию, вы можете установить мониторинг одновременно с установкой Skype для бизнеса Server. Если сервер еще не готов, можно перейти к установке Skype для бизнеса Server самостоятельно, а затем установить мониторинг, когда сервер готов к использованию.

 **Сколько серверных баз данных мониторинга требуется?** Было оценено, что совпавая база данных для мониторинга и архива может поддерживать 240 000 пользователей Skype для бизнеса Server. Кроме того, единая база данных мониторинга может использоваться несколькими пулами переднего плана; если в организации имеются три пула переднего плана, можно связать все эти три пула с одним и тем же серверным хранилищем.

Для многих организаций емкость базы данных не является определяющим фактором при определении количества необходимых баз данных мониторинга. Более важным фактором может быть скорость сети. Предположим, у вас имеются три пула переднего плана, но один из этих пулов базируется на сетевом подключении с более низкой скоростью. В этом случае может потребоваться использование двух баз данных мониторинга: одна база данных предназначена для обслуживания двух пулов с высокоскоростным сетевым подключением, а отдельная база данных — для обслуживания пула с низкоскоростным сетевым подключением.

Кроме того, следует учесть, что Skype для бизнеса Server поддерживает использование зеркальных баз данных. «Зеркалирование баз данных» предоставляет возможность одновременно поддерживать две копии базы данных, причем эти базы данных размещаются на разных серверах. При каждой записи сведений в основную базу данных эти же самые данные также записываются в зеркальную базу данных. If the primary database should fail or otherwise become unavailable, you can "fail over" to the mirror database by using a simple Skype for Business Server PowerShell command. Пример:

```PowerShell
Invoke-CsDatabaseFailover -PoolFqdn atl-cs-001.litwareinc.com -DatabaseType "Monitoring" -NewPrincipal "Mirror"
```

Это имеет большое значение при планировании, так как для использования зеркалирования потребуется дублирование необходимого количества баз данных: в дополнение к каждой основной базе данных вам потребуется вторая база данных, выступающая в качестве зеркала.

 **Требуются ли для сайтов Skype для бизнеса Server собственные настраиваемые конфигурации мониторинга?** При установке Skype для бизнеса Server также устанавливаются глобальные коллекции параметров конфигурации CDR и QoE; эти глобальные коллекции дают возможность применять одинаковые параметры CDR и QoE для всей организации. Во многих случаях этого достаточно: как правило, необходимо, чтобы, например, мониторинг CDR, был включен для всех пользователей.

Однако в некоторых случаях требуется применение различных настроек для разных сайтов. Например, предположим, вы хотите использовать функции мониторинга CDR и QoE на сайте Redmond, но использовать только мониторинг CDR на сайте Dublin. Аналогичным образом, предположим, что вы хотите хранить данные мониторинга в течение 60 дней на сайте Redmond, однако эти же данные требуется хранить на сайте Dublin в течение всего лишь 30 дней. Skype для бизнеса Server позволяет создавать отдельные коллекции параметров конфигурации CDR и QoE на уровне сайта; это позволяет по-разному управлять каждым сайтом. (Это подразумевает автономное включение и отключение мониторинга, а также настройку параметров управления, включая длительность хранения данных.)

Обратите внимание, что вы можете принять это решение как перед развертыванием функции мониторинга, так и после ее развертывания. Например, вы можете развернуть систему мониторинга, а затем осуществлять управление всей организацией с помощью глобальных параметров. Если вы впоследствии измените свое мнение, вы сможете создать отдельный набор настроек, например, для сайта Redmond, а затем использовать эти настройки для управления мониторингом сайта Redmond. (Настройки, применяемые на уровне сайта, всегда имеют приоритет над настройками, применяемыми на глобальном уровне.) Если еще раз измените свое решение, можно просто удалить параметры конфигурации, применяемые к сайту Redmond. При удалении набора настроек сайта к этому сайту автоматически применяется глобальный набор параметров.

## <a name="sql-requirements-for-monitoring"></a>SQL мониторинга
<a name="topologies"> </a>

Унифицированные агенты сбора данных автоматически устанавливаются и активируются на каждом сервере переднего сервера при включаем мониторинге. Поддерживаемые версии SQL Server и другие сведения см. в требованиях к серверу [для Skype для бизнеса Server 2015](requirements-for-your-environment/server-requirements.md)

Данные наблюдения могут совместно использовать экземпляр SQL Server с другими типами данных. Обычно база данных регистрации вызовов (LcsCdr) и база данных качества взаимодействия (QoEMetrics) совместно используют один экземпляр SQL; также распространены случаи, когда две базы данных мониторинга находятся в одном экземпляре SQL с базой данных архивации (LcsLog). Единственное реальное требование к экземплярам SQL Server заключается в том, что каждый экземпляр SQL Server должен иметь следующие ограничения.

- Один экземпляр серверной базы данных Skype для бизнеса Server 2015. (Как правило, не рекомендуется, чтобы база данных мониторинга размещалась в том же экземпляре SQL или даже на том же компьютере, что и внутренняя база данных. Хотя технически это возможно, существует риск того, что база данных мониторинга займет пространство диска, необходимое для внутренней базы данных.)

- Один экземпляр базы данных регистрации вызовов.

- Один экземпляр базы данных качества взаимодействия.

- Один экземпляр базы данных архивации.

Другими словами, нельзя иметь два экземпляра базы данных LcsCdr в одном экземпляре SQL Server. Если требуется несколько экземпляров базы данных LcsCdr, необходимо настроить несколько экземпляров SQL Server.
