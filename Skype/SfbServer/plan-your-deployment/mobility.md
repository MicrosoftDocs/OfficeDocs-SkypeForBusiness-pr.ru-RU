# <a name="plan-for-mobility-for-skype-for-business-server"></a>Планирование мобильности для Skype для бизнес-сервера
 
Планирование реализации мобильности для Skype для бизнес-сервера.
  
С помощью Skype для бизнеса Server можно развернуть функцию Mobility для предоставления функций Skype для бизнес-сервера на мобильных устройствах. В этой статье представлены сведения о функции Mobility, а также помогают планировать развертывание.
  
Функция Mobility для Skype для бизнеса Server может поддерживать мобильных клиентов для Skype для бизнеса, а также клиентов Lync, возвращаясь к 2010 году. После развертывания пользователи могут подключиться к развертыванию Skype для бизнеса Server с помощью поддерживаемых мобильных устройств iOS, Android и Windows Phone, чтобы воспользоваться несколькими различными функциями, в том числе Корпоративная голосовая связь функциями. Мы включили неполный список ниже, и вы также можете проверить сравнение функций клиента Desktop для [Skype для](clients-and-devices/desktop-feature-comparison.md) бизнеса, чтобы получить дополнительные сведения:
  
- Отправка и получение сообщений
    
- Просмотр присутствия
    
- Просмотр контактов
    
- Щелкните, чтобы присоединиться к конференции
    
- Вызов с помощью работы
    
- Достижение одного номера
    
- Голосовая почта
    
- Уведомление о пропущенных вызовах
    
- Передача голоса по IP (VoIP)
    
- Видео участника (H.264)
    
- Просмотр контента собраний (PowerPoint и desktop/application sharing)
    
Все это выполняется через веб-API единой связи или UCWA. UCWA впервые была представлена в Lync Server 2013 и по-прежнему используется для Skype для бизнеса Server. Существует дополнительная функциональность для общения с клиентами Lync 2010, а это служба мобильности (MCX). Это бесплатные службы, позволяющие клиентам Lync Server 2010 и 2013, а также клиентам Skype для бизнеса успешно получать доступ к развертываниям Skype для бизнеса Server.
  
> [!NOTE]
> Поддержка mcX (Mobility Service) для устаревших мобильных клиентов больше недоступна в Skype для бизнеса Server 2019. Все текущие мобильные клиенты Skype для бизнеса уже используют веб-API единой связи (UCWA) для поддержки обмена мгновенными сообщениями, присутствия и контактов. Пользователям с устаревшими клиентами, использующими MCX, потребуется обновиться до текущего клиента.
  
Важно отметить, что, хотя все эти функции доступны после реализации Mobility, на некоторых устройствах они могут работать немного по-другому. У нас есть веб-сайт, который обсуждает, какие функции работают на каких устройствах, в сопоставлении функций [мобильных](clients-and-devices/mobile-feature-comparison.md)клиентов для Skype для бизнеса . У нас также есть некоторые отличные сведения об устройстве и ОС [в Plan для клиентов и устройств.](clients-and-devices/clients-and-devices.md)
  
Мобильность использует функцию автооткрытия, которая позволяет клиентам автоматически находить веб-службы Skype для бизнеса Server без необходимости ввода url-адресов (им даже не нужно их знать). Если требуется устранить некоторые неполадки, ручная запись URL-адресов по-прежнему поддерживается.
  
Push-уведомления также поддерживаются, если приложение Skype для бизнеса не работает в фоновом режиме (или для мобильных устройств, которые не поддерживают приложения, работающие в фоновом режиме). На мобильное устройство отправляется push-уведомление о событии, которое происходит, когда устройство или приложение неактивно. Хорошим примером является отсутствие сообщения чата, когда телефон не активен, что приведет к отправлению push-уведомления (это представлено в качестве тоста или уведомления, например, когда приложение работает в фоновом режиме). С помощью push-уведомлений пользователи не пропускают вызовы мгновенных сообщений и голосовых вызовов.

## <a name="mobility-components"></a>Компоненты мобильности
<a name="MobilityComponents"> </a>

Существует четыре службы, которые включают мобильность для Skype для бизнес-сервера:
  
- **Веб-API Unified Communications (UCWA)**
    
    Предоставляет службы для связи в режиме реального времени с мобильными и веб-клиентами для Skype для бизнеса Server. При развертывании Skype для бизнеса server создается виртуальный каталог UCWA во внутренних и внешних веб-службах. Виртуальный компонент в этом виртуальном каталоге, который принимает вызовы от клиентов с поддержкой UCWA. Клиентские приложения взаимодействуют с интерфейсом переноса репрезентаального состояния (REST) для:
    
  - presence
    
  - contacts
    
  - мгновенные сообщения (IM)
    
  - VoIP
    
  - видеоконференция
    
  - совместная работа
    
    UCWA использует канал на основе P-GET для отправки событий, таких как входящий вызов, входящий чат или сообщение в клиентские приложения.
    
- **Служба мобильности (MCX)**
    
    Поддерживает функции Skype для бизнес-сервера, такие как чат, присутствие и контакты на мобильных устройствах. Служба Mobility устанавливается на каждом переднем сервере в каждом пуле, предназначенном для поддержки функций Skype для бизнеса Server на мобильных устройствах. При установке Skype для бизнеса Server 2015 создается новый виртуальный каталог (Mcx) как на внутренних, так и внешних веб-сайтах на серверах переднего входа.
    
    > [!NOTE]
    > Поддержка mcX (Mobility Service) для устаревших мобильных клиентов больше недоступна в Skype для бизнеса Server 2019. Все текущие мобильные клиенты Skype для бизнеса уже используют веб-API единой связи (UCWA) для поддержки обмена мгновенными сообщениями, присутствия и контактов. Пользователям с устаревшими клиентами, использующими MCX, потребуется обновиться до текущего клиента.
  
- **Служба автообнаружения**
    
    Определяет расположение пользователя и позволяет мобильным устройствам и другим клиентам Skype для бизнеса находить ресурсы (например, внутренние и внешние URL-адреса для веб-служб Skype для бизнеса Server, URL-адрес Mcx или URL-адрес UCWA) независимо от расположения сети. Автоматическое обнаружение использует жесткие имена хостов (lyncdiscoverinternal для пользователей внутри сети, lyncdiscover для пользователей за пределами сети) и SIP-домен пользователя. Он поддерживает клиентские подключения, которые используют HTTP или HTTPS. 
    
    Служба автооткрытия установлена на каждом переднем сервере и на каждом директоре каждого пула, предназначенного для поддержки функций Skype для бизнеса Server на мобильных устройствах. При установке службы создается новый виртуальный каталог (Autodiscover) как на внутренних, так и на внешних веб-сайтах на серверах и директорах.
    
- **Служба push-уведомлений**
    
    Облачная служба, расположенная в центре обработки данных Skype для бизнеса Online. На телефонах, на которые не работает клиент Skype для бизнеса в фоновом режиме, при новом событии на мобильное устройство отправляется уведомление о пропущенных событиях (называемое push-уведомлением). Служба Mobility отправляет уведомление службе push-уведомлений (MPNS), которая затем отправляет его на мобильное устройство. Затем пользователь может откликнуться на уведомление на мобильном устройстве, чтобы активировать приложение. Для этой функции необходим edge Server.

У нас есть следующие поддерживаемые приложения Skype для бизнес-сервера для планирования топологии:

Служба Mobility поддерживается на интерфейсных серверах при подстройке с ролью Сервера-посредника с двумя сетевыми интерфейсами, но для настройки этих интерфейсов необходимо предпринять соответствующие действия. Необходимо назначить IP-адреса определенному интерфейсу, который будет взаимодействовать в качестве сервера-посредника, и сетевому IP-интерфейсу, который будет взаимодействовать в качестве переднего end Server. Это можно сделать в Topology Builder, выбрав правильный IP-адрес для каждой службы, а не используя по умолчанию выбор всех настроенных **IP-адресов.**
  
## <a name="technical-requirements"></a>Технические требования
<a name="TechRequirements"> </a>

Важно планировать различные сценарии мобильных приложений, с которыми могут столкнуться пользователи мобильных устройств. Например, кто-то может начать использовать мобильное приложение вне работы, подключившись через сеть 3G, а затем переключиться на корпоративную сеть Wi-Fi по достижении работы. При выходе из здания они могут перейти на 4G. Планирование теперь позволит поддерживать эти сетевые переходы и гарантировать согласованное пользовательский интерфейс.
  
### <a name="dns-configuration"></a>Конфигурация DNS

Службы мобильности Mcx и UCWA используют DNS таким же образом. С помощью автоматического обнаружения мобильные устройства используют DNS для поиска ресурсов. Во время проверки DNS попытка подключения к FQDN, связанная с внутренней записью DNS (lyncdiscoverinternal.[ внутреннее доменное имя]). Если внутренняя запись DNS не может использоваться для подключения, будет предпринята попытка второго подключения, на этот раз к внешней записи DNS (lyncdiscover.[ sipdomain]). Так почему же два? Мобильное устройство, внутреннее в сети, сможет использовать внутренний URL-адрес автооткрытия. Внешние мобильные устройства будут использовать внешний URL-адрес автооткрытия. В любом случае служба автооткрытия возвращает все URL-адреса веб-службы для домашнего пула пользователя, который включает службу мобильности (Mcx и UCWA).
  
Ожидается, что внешние запросы автообнаружаемых будут проходить через обратный прокси,настроенный для Skype для бизнеса Server. Однако внутренний URL-адрес службы мобильности и внешний URL-адрес службы мобильности связаны с внешнимИ веб-службами FQDN. Поэтому независимо от того, является ли мобильное устройство внутренним или внешним в сети, устройство всегда подключается к службе мобильности Skype для бизнеса Server внешне, через обратный прокси.
  
> [!NOTE]
> Как мы только что отметили, весь трафик службы мобильности (внутренний и внешний) будет проходить через обратный прокси. Но иногда возникает проблема, когда внутренний трафик выходит через интерфейс, только чтобы затем попытаться вернуться в том же интерфейсе. Это может привести к нарушению правил безопасности подмены (формально это называется подменой пакетов TCP). Необходимо разрешить функции **"Закрепление волос"** для функции Mobility.
  
> [!NOTE]
> Если вы готовы сделать это, вы также можете использовать обратный прокси, который отделен от брандмауэра (в целях безопасности предотвращение подмены всегда должно применяться на брандмауэре). Таким образом, шпилька может происходить во внешнем интерфейсе обратного прокси-сервера, а не во внешнем интерфейсе брандмауэра. Это позволяет правильно обнаруживать подмену на брандмауэре, а затем расслабить правило в обратном прокси-сервере, а также получить функциональность Mobility. 
  
> [!NOTE]
> Если вы идете по этому маршруту, обязательно используйте записи DNS host или CNAME для определения обратного прокси-сервера для поведения шпильки (а не брандмауэра), если это возможно. 
  
Существует несколько вещей, которые необходимо настроить для поддержки пользователей внутри и за пределами корпоративной сети.
  
Это правила для внутренних и внешних веб-FQDNs:
  
- Новые записи DNS CNAME или A (хост, если IPv6, AAAA) для автоматического обнаружения.
    
- Новое правило брандмауэра, если вы хотите поддерживать push-уведомления через Wi-Fi сети.
    
- Обязывкать альтернативные имена в внутренних сертификатах сервера и обратных прокси-сертификатах для автоматического обнаружения.
    
- Сбалансированная конфигурация загрузки оборудования переднего сервера изменяет сродство источника.
    
Это требования топологии, необходимые для поддержки службы мобильности и автооткрытия:
  
- Внутренний веб-FQDN пула переднего конца должен быть отличается от внешнего веб-FQDN переднего пула.
    
- Внутренний веб-FQDN должен разрешать и быть доступным только внутри корпоративной сети.
    
- Внешний веб-FQDN должен разрешать доступ только к Интернету и быть доступным.
    
- Для пользователя в корпоративной сети URL-адрес службы мобильности должен быть адресован внешнему веб-серверу. Это требование относится к службе Mobility и применяется только к этому URL-адресу.
    
- Для пользователя, не влияемого на корпоративную сеть, запрос должен переходить на внешний веб-сайт FQDN интерфейсного пула или директора.
    
Если вы поддерживаете автоматическое обнаружение, необходимо сделать следующие записи DNS для каждого домена SIP:
  
- Внутренняя запись DNS для поддержки мобильных пользователей, которые подключаются из сети организации.
    
- Внешняя или публичная запись DNS для поддержки мобильных пользователей, подключенных из Интернета.
    
Внутренний URL-адрес автоматического обнаружения не должен быть адресован извне внутренней сети. Внешний URL-адрес автоматического обнаружения не должен быть адресован из сети. Но если это невозможно для внешнего URL-адреса, функциональность вашего мобильного клиента, вероятно, не будет затронута, так как внутренний URL-адрес всегда будет сначала опробовано.
  
### <a name="port-and-firewall-requirements"></a>Требования к порту и брандмауэру

Большую часть этого мы охватывали в другой документации, но специально для Mobility вы хотите, чтобы в корпоративной сети Wi-Fi были открыты следующие порты, если у вас есть какие-либо пользователи, которые имеются в устройстве для выживших филиалов (SBA):
  
- UcwaSipExternalListeningPort требует 5088.
    
- UcwaSipPrimaryListeningPort требует 5089.
    
### <a name="certificate-requirements"></a>Требования к сертификатам

Если вы используете автоматическое открытие для мобильных клиентов Skype для бизнеса, вам потребуется изменить списки SAN (альтернативное имя субъекта) в сертификатах для поддержки безопасных подключений от мобильных клиентов. Если у вас уже есть сертификаты на месте, вам необходимо запросить и назначить новые сертификаты с описанными здесь записями SAN. Это необходимо сделать для каждого переднего сервера и директора (если в вашей среде), который запускает службу автооткрытия. Мы также рекомендуем изменить списки SAN в обратных прокси-сертификатах, добавив записи SAN для каждого домена SIP в вашей организации.
  
Это должен быть простой процесс, если вы запрашиваете новые сертификаты от внутреннего ca (сертификата), но общедоступные сертификаты являются более сложными, и потенциально намного дороже повторно запрашивать, не говоря уже о том, что может быть дорогостоящим добавить много доменов SIP в новый общедоступный сертификат. В этой ситуации существует подход, который поддерживается, но **не рекомендуется**. Вы можете настроить обратный прокси-сервер, чтобы сделать первоначальный запрос на службу автообнаружания над портом 80, который будет использовать HTTP, а не порт 443, который является HTTPS (и 443 является конфигурацией по умолчанию). Этот входящий запрос будет перенаправлен в порт 8080 в пуле передней части или в каталоге. При этом вам не потребуется вносить изменения в сертификаты, так как этот трафик не использует HTTPS для запросов. Но опять же, мы не рекомендуем это, хотя это будет работать для вас.
  
### <a name="windows-and-iis-requirements"></a>Требования к Windows и IIS

Вы должны иметь поддерживаемую версию Windows Server для среды Skype для бизнеса Server. В результате для ваших потребностей в мобильности также должен быть IIS 8 или IIS 8.5. Необходимо внести некоторые изменения в параметры ASP.NET по умолчанию, но установщик службы мобильности сделает это автоматически.
  
### <a name="hlb-requirements"></a>Требования HLB

Если вы используете топологию для Skype для бизнес-сервера, которая включает HLB для переднего пула (которая будет любой топологией, которая включает несколько серверов переднего входа), для источника необходимо настроить внешние виртуальные ИП (VIP-персоны) веб-служб для трафика веб-служб. Сродство источников помогает обеспечить, чтобы несколько подключений от одного клиента отправляли на один сервер для поддержания состояния сеанса.
  
Если вы планируете поддерживать мобильные клиенты Skype для бизнеса только в вашей внутренней Wi-Fi сети, необходимо настроить внутренние VIP-персоны веб-служб для источника, как описано для внешних VIP-персон веб-служб. В этой ситуации следует использовать source_addr (или TCP) для внутренних VIP-персон веб-служб в HLB.
  
Подробные сведения об этом можно просмотреть в документации [Skype для бизнеса](network-requirements/load-balancing.md) по балансировки нагрузки.
  
### <a name="reverse-proxy-requirements"></a>Требования обратного прокси-сервера

Чтобы поддерживать автоматическое обнаружение для мобильных клиентов Skype для бизнеса, необходимо обновить текущее правило публикации следующим образом:
  
- Если вы решили обновить списки SAN в обратных прокси-сертификатах и используете HTTPS для первоначального запроса на службу автообнаружение, необходимо обновить правило веб-публикации для lyncdiscover. \<sipdomain\> . Это обычно сочетается с опубликованным rul для внешнего URL-адреса веб-служб в пуле переднего конца.
    
- Если вы решили использовать HTTP для первоначального запроса службы автообнаружение, чтобы избежать необходимости обновлять список SAN для обратных прокси-сертификатов (которые мы не рекомендуем), необходимо создать новое правило веб-публикации для порта HTTP/TCP 80, если его еще нет. Если это правило существует, обнови его, чтобы включить lyncdiscover.\<sipdomain\> запись.
    
## <a name="defining-your-mobility-needs"></a>Определение потребностей в мобильности
<a name="MobilityNeeds"> </a>

Теперь, когда мы рассмотрели топологии, компоненты и технические требования, давайте рассмотрим, что может понадобиться вашей организации с точки зрения реализации Mobility.
  
### <a name="do-you-want-to-use-automatic-discovery-for-skype-for-business-mobile-clients"></a>Вы хотите использовать автоматическое открытие для мобильных клиентов Skype для бизнеса?

Мы настоятельно рекомендуем использовать автоматическое обнаружение. Для этого потребуется создание новых внутренних и внешних записей DNS, как описано в разделе Технические требования выше. При автоматическом обнаружении клиенты Skype для бизнеса могут автоматически находить веб-службы Skype для бизнес-сервера из любого расположения без необходимости вручную вписать URL-адрес.
  
При необходимости можно использовать параметры вручную. Эти URL-адреса должны быть введены пользователями на своих мобильных устройствах:
  
- **https:// \<ExtPoolFQDN\> /Autodiscover/autodiscoverservice.svc/Root** для внешнего доступа.
    
- **https:// \<IntPoolFQDN\> /Autodiscover/autodiscoverservice.svc/Root** для внутреннего доступа.
    
Опять же, мы рекомендуем использовать автоматическое обнаружение. Для устранения неполадок могут быть полезны параметры вручную.
  
### <a name="are-you-going-to-support-push-notifications"></a>Собираетесь ли вы поддерживать push-уведомления?

Push-уведомления используются для мобильных приложений, которые поддерживают эту функцию, чтобы уведомить пользователя о событиях, пока приложение не активен. Вашему edge Server необходимо будет иметь отношение федерации к облачной службе push-уведомлений Skype для бизнес-сервера, которая находится в центрах обработки данных Skype для бизнеса Online. Для запуска push-уведомлений необходимо запустить комлет.
  
> [!NOTE]
> Если у вас есть все еще использующие клиенты Lync Server 2010, им потребуется открытый исходящие TCP-порт 5223 в корпоративной сети WiFi.

### <a name="should-users-who-arent-enabled-for-enterprise-voice-be-able-to-use-click-to-join-to-join-conferences"></a>Должны ли пользователи, не включенные для Корпоративная голосовая связь, использовать кнопку Нажмите кнопку Присоединиться, чтобы присоединиться к конференциям?

Чтобы пользователи могли иметь доступ к функциям Mobility и Call via Work, их необходимо включить для Корпоративная голосовая связь. Но даже если они не включены, они по-прежнему могут присоединяться к конференциям, щелкнув ссылку на мобильном устройстве, но только если им назначена соответствующая политика голосовой связи. Вы можете либо:
  
- назначить указанным пользователям определенную политику голосовой почты или,
    
- убедитесь, что глобальная политика или политика на уровне сайта существует и применяется к ним.
    
В любом случае политика голосовой связи, которую вы назначаете, должна иметь записи и маршруты использования общедоступных переключеных телефонных сетей (PSTN), которые определят, где пользователи смогут звонить на конференции.
  
> [!NOTE]
> Мобильным пользователям, которые хотят использовать кнопку Нажмите кнопку Присоединиться, требуется политика голосовой связи, а также соответствующие записи об использовании PSTN и маршруты голосовой связи, так как при нажатии на эту ссылку на мобильных устройствах будет результатом исходящие вызовы из Skype для бизнеса Server.
