# <a name="plan-for-mobility-for-skype-for-business-server"></a>Планирование мобильных устройств в Skype для бизнеса Server
 
Составьте план реализации мобильных устройств для Skype для бизнеса Server.
  
С помощью Skype для бизнеса Server вы можете развернуть функцию Mobility Service, чтобы обеспечить поддержку функций Skype для бизнеса Server на мобильных устройствах. В этой статье приводятся сведения о функции мобильных устройств, которые помогут вам спланировать развертывание.
  
Функция мобильной связи в Skype для бизнеса Server поддерживает мобильные клиенты Skype для бизнеса, а также клиенты Lync, переходящие обратно в 2010. После развертывания вы сможете подключиться к развертыванию сервера Skype для бизнеса с помощью поддерживаемых мобильных устройств iOS, Android и Windows Phone, чтобы воспользоваться различными функциями, в том числе корпоративными голосовыми функциями. Мы включили частичный список ниже, и вы также можете проверить [Сравнение функций клиента для настольных систем для Skype для бизнеса](clients-and-devices/desktop-feature-comparison.md) , чтобы получить дополнительные сведения.
  
- Отправка и прием сообщений
    
- Просмотр сведений о присутствии
    
- Просмотр контактов
    
- Присоединение к конференции по щелчку
    
- Звонок с рабочего телефона
    
- Доступ по одному номеру
    
- Голосовая почта
    
- Уведомление о пропущенных звонках
    
- Протокол VoIP
    
- Видео участника (H.264)
    
- Просмотр содержимого собрания (PowerPoint и общий доступ к рабочему столу и приложениям)
    
Все эти функции обеспечиваются с помощью веб-API объединенных коммуникаций — UCWA. УКВА впервые появился в Lync Server 2013, и он по-прежнему используется в Skype для бизнеса Server. У клиентов Lync 2010 есть дополнительные функции, и эта служба мобильной связи (МККС). Это бесплатные службы, позволяющие клиентам Lync Server 2010 и 2013, а также клиентам Skype для бизнеса доступ к развертываниям Skype для бизнеса Server успешно.
  
> [!NOTE]
> Поддержка МККС (служба Mobility Service) для устаревших мобильных клиентов больше не доступна в Skype для бизнеса Server 2019. На всех текущих мобильных клиентах Skype для бизнеса уже используется веб-API единой системы обмена сообщениями (УКВА) для поддержки мгновенных сообщений, присутствия и контактов. Пользователи с устаревшими клиентами, использующими МККС, должны обновиться до текущего клиента.
  
Важно отметить, что несмотря на то, что все эти возможности доступны после реализации мобильных устройств, на некоторых устройствах они могут работать немного по-другому. У нас есть веб-сайт, показывающий, какие функции работают на устройствах: [Сравнение возможностей мобильных клиентов для Skype для бизнеса](clients-and-devices/mobile-feature-comparison.md). У нас также есть несколько удобных данных об устройствах и ОС в [плане для клиентов и устройств](clients-and-devices/clients-and-devices.md).
  
Мобильная связь использует функцию автообнаружения, которая позволяет клиентам автоматически находить веб-службы Skype для бизнеса Server без необходимости указывать их в любых URL-адресах (они даже не знают). Тем не менее ввод URL-адресов вручную поддерживается для поиска и устранения неполадок.
  
Push-уведомления также поддерживаются, если приложение Skype для бизнеса не работает в фоновом режиме (или для мобильных устройств, которые не поддерживают приложения, работающие в фоновом режиме). Push-уведомление передается на мобильное устройство, сообщая о событии, произошедшем, когда устройство или приложение было неактивно. Например, если вы пропустили мгновенное сообщение, когда телефон был отключен, отправляется push-уведомление (оно отображается как всплывающее уведомление при выполнении приложения в фоновом режиме). Благодаря push-уведомлениям пользователи не пропустят важные мгновенные сообщения или голосовые звонки.

## <a name="mobility-components"></a>Составляющие компонента Mobility
<a name="MobilityComponents"> </a>

В Skype для бизнеса Server есть четыре службы, которые входят в состав мобильных устройств:
  
- **Веб-API объединенных коммуникаций (UCWA)**
    
    Предоставляет услуги для общения в реальном времени с мобильными и веб-клиентами для Skype для бизнеса Server. После развертывания Skype для бизнеса Server УКВА виртуальный каталог, созданный во внутренней и внешней веб-службах. Виртуальный компонент в этом виртуальном каталоге принимает вызовы от клиентов, поддерживающих UCWA. Клиентские приложения обмениваются данными по интерфейсу REST, передавая:
    
  - сведения о присутствии,
    
  - контакты,
    
  - мгновенные сообщения,
    
  - VoIP
    
  - видеоконференции,
    
  - данные для совместной работы.
    
    UCWA использует канал на основе протокола P-GET для отправки событий, таких как входящие звонки, входящие мгновенные сообщения и сообщения для клиентского приложения.
    
- **Служба Mobility Service (MCX)**
    
    Поддержка функций Skype для бизнеса Server (например, мгновенных сообщений, присутствия и контактов) на мобильных устройствах. Служба Mobility Service устанавливается на каждый сервер переднего плана в каждом пуле, предназначенный для поддержки функций Skype для бизнеса Server на мобильных устройствах. При установке Skype для бизнеса Server 2015 новый виртуальный каталог (МККС) создается как на внутреннем, так и на внешнем веб-сайтах на серверах переднего плана.
    
    > [!NOTE]
    > Поддержка МККС (служба Mobility Service) для устаревших мобильных клиентов больше не доступна в Skype для бизнеса Server 2019. На всех текущих мобильных клиентах Skype для бизнеса уже используется веб-API единой системы обмена сообщениями (УКВА) для поддержки мгновенных сообщений, присутствия и контактов. Пользователи с устаревшими клиентами, использующими МККС, должны обновиться до текущего клиента.
  
- **Служба автообнаружения**
    
    Определяет расположение пользователя и позволяет мобильным устройствам и другим клиентам Skype для бизнеса находить ресурсы (такие как внутренние и внешние URL-адреса МККС или URL-адрес УКВА) независимо от расположения в сети. Automatic discovery uses hardcoded host names (lyncdiscoverinternal for users inside the network, lyncdiscover for users outside the network), and the SIP domain of the user. It supports client connections that use either HTTP or HTTPS. 
    
    Служба автообнаружения устанавливается на каждый сервер переднего плана и каждый из режиссеров в каждом пуле, предназначенный для поддержки функций Skype для бизнеса Server на мобильных устройствах. После установки службы новый виртуальный каталог (автообнаружения) создается как на внутреннем, так и на внешнем веб-сайтах на серверах и режиссерах переднего плана.
    
- **Служба push-уведомлений**
    
    Облачная служба, расположенная в центре обработки данных Skype для бизнеса Online. На телефонах, не имеющих клиента Skype для бизнеса в фоновом режиме, при появлении нового события уведомление о пропущенном событии (называемом push-уведомлением) отправляется на мобильное устройство. Служба Mobility Service отправляет уведомление в службу push-уведомлений (MPNS), которая затем отправляет ее на мобильное устройство. После этого пользователь может ответить на уведомление на мобильном устройстве, чтобы активировать приложение. Для этой функции требуется пограничный сервер.
    
## <a name="supported-topologies"></a>Поддерживаемые топологии
<a name="SupportedTopos"> </a>

Для планирования топологии у нас есть следующие Поддерживаемые серверные приложения Skype для бизнеса.
  
- Выпуск Standard Edition для мобильных устройств
    
- Мобильность Enterprise Edition
    
Вы сможете использовать эту функцию с пограничными серверами Skype для бизнеса Server или Lync Server 2013 Edge.
  
Служба Mobility Service поддерживается на серверах переднего плана, размещенных с помощью роли сервера обновлений, с двумя сетевыми интерфейсами, но для настройки этих интерфейсов необходимо выполнить соответствующие действия. Необходимо назначить IP-адреса определенному интерфейсу, который будет взаимодействовать как сервер-посредник, а также сетевой интерфейс IP, который будет взаимодействовать как сервер переднего плана. Это можно сделать в построителе топологии путем выбора IP-адреса для каждой службы вместо использования параметров по умолчанию для **всех настроенных IP-адресов** .
  
## <a name="technical-requirements"></a>Технические требования
<a name="TechRequirements"> </a>

Очень важно учесть различные сценарии использования мобильных приложений, с которыми могут столкнуться пользователи мобильных устройств. Например, сотрудник может начать использовать мобильное приложение за пределами организации, подключаясь по сети 3G, а затем переключиться на корпоративную сеть Wi-Fi, когда доберется до работы. При выходе из здания они могут переключиться на 4G. Заблаговременное планирование позволит обеспечить поддержку такой смены сетей и гарантировать согласованное взаимодействие с пользователями.
  
### <a name="dns-configuration"></a>Конфигурация DNS

Службы Mobility Services МККС и УКВА используют DNS таким же образом. При включенной функции автообнаружения мобильные устройства используют DNS для нахождения ресурсов. При поиске DNS выполняется попытка подключения к полному доменному имени, связанному с внутренней записью DNS (lyncdiscoverinternal.[внутреннее_имя_домена]). Если внутреннюю запись DNS нельзя использовать для подключения, выполняется вторая попытка подключения, на этот раз к внешней записи DNS (lyncdiscover.[домен_sip]). Зачем нужны два подключения? Мобильное устройство, находящееся внутри сети, может использовать внутренний URL-адрес автообнаружения. Внешние мобильные устройства используют внешний URL-адрес автообнаружения. В обоих случаях служба автообнаружения будет возвращать URL-адреса для домашнего пула пользователя, который включает службу Mobility Service (МККС и УКВА).
  
Ожидается, что внешние запросы автообнаружения будут проходить через обратный прокси-сервер, настроенный для Skype для бизнеса Server. Однако URL-адрес службы внутренних мобильных устройств и внешний URL-адрес службы Mobility Service связаны с полным доменным именем внешних веб-служб. Таким образом, независимо от того, является ли мобильное устройство внутренним или внешним по отношению к вашей сети, устройство всегда подключается к службе Mobility Service в Skype для бизнеса Server с помощью обратного прокси-сервера.
  
> [!NOTE]
> Как мы только что отмечали, весь трафик службы Mobility Service (внутренний и внешний) проходит через обратный прокси-сервер. Однако иногда возникает проблема, когда внутренний трафик выходит за пределы сети по интерфейсу, после чего сразу пытается вернуться обратно через тот же интерфейс. Это может нарушать правила безопасности, связанные со спуфингом (иначе "спуфинг пакетов TCP"). Для того чтобы использовать функцию мобильной связи, вам нужно разрешить **закрепление** .
  
> [!NOTE]
> Если вы готовы сделать это, вы также можете использовать обратный прокси-сервер, который отделен от вашего брандмауэра (в целях обеспечения безопасности необходимо, чтобы защита от подмены всегда была принудительно установлена на брандмауэре). Таким образом, разворот может происходить на внешнем интерфейсе обратного прокси-сервера, а не на внешнем интерфейсе брандмауэра. Это позволит вам правильно подменить подделку в брандмауэре, пока вы разрешаете правило на обратном прокси-сервере, и вы получаете функцию мобильности. 
  
> [!NOTE]
> Если вы найдете этот маршрут, убедитесь, что вы используете DNS Host или CNAME-записи, чтобы определить обратный прокси для поведения разворот (не для брандмауэра), если это возможно. 
  
Вам потребуется настроить некоторые компоненты для поддержки пользователей в корпоративной среде и за ее пределами.
  
Ниже приведены правила для внутренних и внешних полных доменных имен:
  
- новые записи CNAME или A (узел, если используется IPv6, AAAA) для автоматического обнаружения;
    
- новое правило брандмауэра, если необходимо обеспечить поддержку push-уведомлений в сети Wi-Fi;
    
- Дополнительные имена субъектов во внутренних сертификатах сервера и обратные сертификаты прокси-сервера для автоматического обнаружения.
    
- Конфигурация балансировки нагрузки для серверного оборудования переднего плана изменяет сходство источников.
    
Ниже указаны требования к топологии, необходимые для поддержки службы Mobility Service и службы автообнаружения.
  
- Полное доменное имя внутренней веб-сайта пула переднего плана должно отличаться от полного доменного имени внешнего веб-сайта пула.
    
- Внутреннее полное доменное имя должно разрешаться только внутри корпоративной сети и должно быть доступно только из этой сети.
    
- Внешнее полное доменное имя должно разрешаться только в Интернете и должно быть доступно только из Интернета.
    
- Для пользователей в корпоративной сети URL-адрес службы Mobility Service должен быть адресован внешнему доменному имени сайта. Это требование относится к службе Mobility Service и применяется только к этому URL-адресу.
    
- Для пользователя, находящегося за пределами корпоративной сети, запрос должен перейти на внешнее полное доменное имя пула или режиссера переднего плана.
    
Если поддерживается автоматическое обнаружение, нужно создать следующие записи DNS для каждого домена SIP:
  
- внутренняя запись DNS для поддержки мобильных пользователей, которые подключаются из сети организации;
    
- внешняя (или общедоступная) запись DNS для поддержки мобильных пользователей, которые подключаются из Интернета.
    
Внутренний URL-адрес автоматического обнаружения не должен быть адресуемым извне внутренней сети. Внешний URL-адрес автоматического обнаружения не должен быть адресуемым внутри сети. Однако если не удается выполнить это требование для внешнего URL-адреса, функциональность мобильного клиента не должна снижаться, поскольку сначала всегда выполняется попытка использования внутреннего URL-адреса.

- для UcwaSipPrimaryListeningPort требуется порт 5089.
    
### <a name="certificate-requirements"></a>Требования к сертификатам

Если вы используете автоматическое обнаружение для мобильных клиентов Skype для бизнеса, вам потребуется изменить списки SAN (дополнительные имена субъектов) в сертификатах, чтобы обеспечить надежную связь с мобильными клиентами. If you already have certificates in-place, you'll need to request and assign new certificates with the SAN entries described here. Это необходимо сделать для каждого сервера переднего плана и режиссера (если он находится в вашей среде), на котором запускается служба автообнаружения. Кроме того, мы рекомендуем вам изменить списки SAN на обратных сертификатах прокси-сервера, добавив записи SAN для каждого домена SIP в Организации.
  
Это должен быть прямолинейный процесс, если вы запрашиваете новые сертификаты из внутреннего центра сертификации (центр сертификации), но открытые сертификаты более сложны и, возможно, больше дорогостоящих для повторного запроса, но не стоит заказывать, что может привести к дорогостоящему добавлению большого количества доменов SIP в новый общедоступный сертификат. В этой ситуации существует подход, который поддерживается, но **не рекомендуется**. Вы можете настроить обратный прокси так, чтобы он был первоначально запросом службы автообнаружения по порту 80, который будет использовать HTTP, а не порт 443, который является конфигурацией по умолчанию HTTPS (и 443). Входящие запросы будут перенаправляться на порт 8080 в пуле или в директории переднего плана. Для этого вам не потребуется вносить изменения в сертификат, так как этот трафик не использует HTTPS для запросов. Но опять же, мы не рекомендуем использовать этот параметр, хотя он будет работать.
  
### <a name="windows-and-iis-requirements"></a>Требования для Windows и служб IIS

У вас должна быть поддерживаемая версия Windows Server для серверной среды Skype для бизнеса. Это позволит вам использовать службы IIS 8 или IIS 8.5 для удовлетворения потребностей в мобильности. Вам потребуется внести некоторые изменения в параметры ASP.NET по умолчанию, но программа установки службы Mobility Service сделает это автоматически.
  
### <a name="hlb-requirements"></a>Требования для аппаратной балансировки нагрузки

Если вы используете топологию для сервера Skype для бизнеса, который включает в себя ХЛБ для своего пула переднего плана (это может быть любая топология, включающая более одного сервера переднего плана), необходимо настроить внешние IP-адреса для внешних виртуальных служб (VIP) для трафика веб-служб. для исходного кода. Source affinity helps to ensure that multiple connections from a single client are sent to the same server to maintain session state.
  
Если вы планируете поддерживать мобильные клиенты Skype для бизнеса только в рамках внутренней сети Wi-Fi, необходимо настроить для него внутренний IP-адрес внутренней службы, как описано в VIP внешних веб-служб. В этой ситуации следует использовать сходство source_addr (или TCP) для VIP внутренних веб-служб в ХЛБ.
  
Дополнительные сведения об этих компонентах см. в документации [Load balancing requirements for Skype for Business](network-requirements/load-balancing.md).
  
### <a name="reverse-proxy-requirements"></a>Требования к обратному прокси-серверу

Чтобы поддерживать автоматическое обнаружение для мобильных клиентов Skype для бизнеса, необходимо обновить текущее правило публикации следующим образом:
  
- Если вы решите обновить списки SAN на обратных сертификатах прокси-сервера, и вы используете HTTPS для первоначального запроса на обслуживание автообнаружения, вам нужно обновить правило веб-публикации для lyncdiscover. \<сипдомаин\>. Обычно этот параметр сочетается с Рул публикации для URL-адреса внешней веб-службы в пуле переднего плана.
    
- Если вы решили использовать HTTP для первоначального запроса на обслуживание автообнаружения, чтобы не допустить необходимости обновлять список SAN для сертификатов прокси-сервера, которые не рекомендуются, вам потребуется создать новое правило веб-публикации для порта HTTP/TCP 80, если оно еще не используется. Если это правило существует, обновите его, чтобы включить lyncdiscover. \<сипдомаин\> запись.
    
## <a name="defining-your-mobility-needs"></a>Определение требований к мобильной связи
<a name="MobilityNeeds"> </a>

Теперь, когда мы проверили топологию, компоненты и технические требования, давайте взглянем на то, что в вашей организации может потребоваться в рамках реализации мобильных устройств.
  
### <a name="do-you-want-to-use-automatic-discovery-for-skype-for-business-mobile-clients"></a>Требуется ли использовать автоматическое обнаружение для мобильных клиентов Skype для бизнеса?

Мы настоятельно рекомендуем использовать автоматическое обнаружение. Для этого потребуется создать новые внутренние и внешние записи DNS, как указано в разделе "Технические требования" выше. С помощью автоматического обнаружения клиенты Skype для бизнеса могут автоматически находить веб-службы Skype для бизнеса Server из любого расположения, не требуя ввода URL-адреса вручную.
  
При желании можно использовать настройку вручную. Пользователям потребуется ввести на мобильных устройствах следующие URL-адреса:
  
- **https://\<екстпулфкдн\>/аутодисковер/аутодисковерсервице.СВК/Рут** для внешнего доступа.
    
- **https://\<интпулфкдн\>/аутодисковер/аутодисковерсервице.СВК/Рут** для внутреннего доступа.
    
Тем не менее, рекомендуется использовать автоматическое обнаружение. Ввод адресов вручную может потребоваться в целях поиска и устранения неполадок.
  
### <a name="are-you-going-to-support-push-notifications"></a>Требуется ли поддержка push-уведомлений?

Push-уведомления используются для мобильных приложений, поддерживающих эту функцию, чтобы уведомлять пользователей о событиях, произошедших, когда приложение было неактивно. У вашего пограничного сервера должно быть отношение Федерации с вашей облачной службой push-уведомлений Skype для бизнеса Server, которая находится в центре обработки данных Skype для бизнеса Online. Для включения push-уведомлений потребуется выполнитьзапустить командлет.
  
> [!NOTE]
> Если вы все еще используете Пользователи Lync Server 2010, им потребуется TCP-порт 5223 Open Outbound для вашей корпоративной сети WiFi.

### <a name="should-users-who-arent-enabled-for-enterprise-voice-be-able-to-use-click-to-join-to-join-conferences"></a>Требуется ли предоставить пользователям, для которых не включена поддержка корпоративной голосовой связи, возможность присоединения к конференциям щелчком мыши?

Чтобы пользователи могли получать доступ к функциям мобильных устройств и звонить через работу, они должны быть активированы для корпоративной голосовой связи. Тем не менее, даже если такая поддержка не включена, они все равно могут присоединяться к конференциям, щелкнув ссылку на мобильном устройстве, но только при назначении им соответствующей политики голосовой связи. Вы можете:
  
- назначить таким пользователям определенную политику голосовой связи или
    
- убедиться, что существует применимая к ним глобальная политика или политика уровня сайта.
    
В любом случае назначаемая политика голосовой связи должна иметь режимы работы с ТСОП и маршруты, которые определяют области, к которым пользователи могут подключаться по телефонной линии для присоединения к конференции.
  
> [!NOTE]
> Мобильные пользователи, которым нужно присоединиться, требуют политики голосовой связи, а также связанные записи использования PSTN и голосовые маршруты, так как при щелчке по этой ссылке на мобильных устройствах будет получен исходящий вызов из Skype для бизнеса Server.
