<span data-ttu-id="22c33-106">Надежные пользователя — это один, учетные данные прошедшие проверку подлинности доверенного сервера в Скайп для Business Server 2015.</span><span class="sxs-lookup"><span data-stu-id="22c33-106">A trusted user is one whose credentials have been authenticated by a trusted server in Skype for Business Server 2015.</span></span> <span data-ttu-id="22c33-107">Этот сервер — это обычно сервера Standard Edition, сервер переднего плана Enterprise Edition или директора.</span><span class="sxs-lookup"><span data-stu-id="22c33-107">This server is usually a Standard Edition server, Enterprise Edition Front End Server, or Director.</span></span> <span data-ttu-id="22c33-108">Скайп для Business Server использует доменных служб Active Directory как репозиторий серверной single, доверенных учетных данных пользователя.</span><span class="sxs-lookup"><span data-stu-id="22c33-108">Skype for Business Server relies on Active Directory Domain Services as the single, trusted back-end repository of user credentials.</span></span>

- <span data-ttu-id="22c33-111">**Протокол MIT Kerberos версии 5 безопасности** для внутренних пользователей с использованием учетных данных Active Directory.</span><span class="sxs-lookup"><span data-stu-id="22c33-111">**MIT Kerberos version 5 security protocol** for internal users with Active Directory credentials.</span></span> <span data-ttu-id="22c33-112">Kerberos требуется возможность подключения клиента для доменов Active Directory, поэтому он не может использоваться для проверки подлинности клиентов за пределами корпоративного брандмауэра.</span><span class="sxs-lookup"><span data-stu-id="22c33-112">Kerberos requires client connectivity to Active Directory Domain Services, which is why it cannot be used for authenticating clients outside the corporate firewall.</span></span>
    
- <span data-ttu-id="22c33-113">**Протокол NTLM** для пользователей с использованием учетных данных Active Directory, подключающихся из конечную точку за пределами корпоративного брандмауэра.</span><span class="sxs-lookup"><span data-stu-id="22c33-113">**NTLM protocol** for users with Active Directory credentials who are connecting from an endpoint outside the corporate firewall.</span></span> <span data-ttu-id="22c33-114">Пограничная служба доступа передает запросы на вход директор, если этот параметр указан, или для проверки подлинности на сервере переднего плана.</span><span class="sxs-lookup"><span data-stu-id="22c33-114">The Access Edge service passes logon requests to a Director, if present, or a Front End Server for authentication.</span></span> <span data-ttu-id="22c33-115">Пограничная служба доступа самого выполняет без проверки подлинности.</span><span class="sxs-lookup"><span data-stu-id="22c33-115">The Access Edge service itself performs no authentication.</span></span>
    
    > [!NOTE]
    > <span data-ttu-id="22c33-116">Протокол NTLM обеспечивает более слабую защиту от атак в сравнении с Kerberos, поэтому некоторые организации сводят использование NTLM к минимуму.</span><span class="sxs-lookup"><span data-stu-id="22c33-116">NTLM protocol offers weaker attack protection than Kerberos, so some organizations minimize usage of NTLM.</span></span> <span data-ttu-id="22c33-117">В результате запрещен доступ к Скайп для Business Server 2015 для внутренних или клиентов, подключенных через подключение к виртуальной частной сети или DirectAccess.</span><span class="sxs-lookup"><span data-stu-id="22c33-117">As a result, access to Skype for Business Server 2015 might be restricted to internal or clients connected through a VPN or DirectAccess connection.</span></span> 
  
- <span data-ttu-id="22c33-p107">**Дайджест-проверка** для так называемых анонимных пользователей. Анонимные пользователи — это внешние пользователи, которые не имеют распознанных учетных данных Active Directory, но были приглашены в локальную конференцию и обладают действительным ключом конференции. Digest authentication is not used for other client interactions.</span></span>
    
<span data-ttu-id="22c33-121">Скайп для проверки подлинности Business Server 2015 состоит из двух этапов:</span><span class="sxs-lookup"><span data-stu-id="22c33-121">Skype for Business Server 2015 authentication consists of two phases:</span></span>
  
1. <span data-ttu-id="22c33-122">Между клиентом и сервером устанавливается сопоставление безопасности.</span><span class="sxs-lookup"><span data-stu-id="22c33-122">A security association is established between the client and the server.</span></span>
    
2. <span data-ttu-id="22c33-p108">Клиент и сервер используют существующее сопоставление безопасности для подписи сообщений, которые они отправляют, и для проверки получения сообщений. <span data-ttu-id="22c33-p109">Доверие пользователя прикрепляется к каждому сообщению, которое исходит от пользователя, но не к удостоверению пользователя. Если учетные данные пользователя действительны, сообщение принимается не только первым получившим его сервером, но и всеми другими серверами в доверенном облаке серверов.</span><span class="sxs-lookup"><span data-stu-id="22c33-p109">User trust is attached to each message that originates from a user, not to the user identity itself. <span data-ttu-id="22c33-128">Пользователи с действительными выданными федеративным партнером учетными данными являются доверенными, но к ним могут применяться другие ограничения привилегий, которые полностью доступны внутренним пользователям.</span><span class="sxs-lookup"><span data-stu-id="22c33-128">Users with valid credentials issued by a federated partner are trusted but optionally prevented by additional constraints from enjoying the full range of privileges accorded to internal users.</span></span>
  
<span data-ttu-id="22c33-129">Протоколы ICE и TURN также используют требование Digest, описанное в документе IETF TURN RFC.</span><span class="sxs-lookup"><span data-stu-id="22c33-129">The ICE and TURN protocols also use the Digest challenge as described in the IETF TURN RFC.</span></span>
