- <span data-ttu-id="fcd1f-113">**Протокол NTLM** для пользователей с учетными данными Active Directory, которые подключаются из конечной точки за пределами корпоративного брандмауэра.</span><span class="sxs-lookup"><span data-stu-id="fcd1f-113">**NTLM protocol** for users with Active Directory credentials who are connecting from an endpoint outside the corporate firewall.</span></span> <span data-ttu-id="fcd1f-114">Служба пограничного доступа передает запросы на вход в директории, если она есть, или на сервер переднего плана для проверки подлинности.</span><span class="sxs-lookup"><span data-stu-id="fcd1f-114">The Access Edge service passes logon requests to a Director, if present, or a Front End Server for authentication.</span></span> <span data-ttu-id="fcd1f-115">Сама служба доступа EDGE не выполняет проверку подлинности.</span><span class="sxs-lookup"><span data-stu-id="fcd1f-115">The Access Edge service itself performs no authentication.</span></span>
    
    > [!NOTE]
    > <span data-ttu-id="fcd1f-116">Протокол NTLM обеспечивает более слабую защиту от атак в сравнении с Kerberos, поэтому некоторые организации сводят использование NTLM к минимуму.</span><span class="sxs-lookup"><span data-stu-id="fcd1f-116">NTLM protocol offers weaker attack protection than Kerberos, so some organizations minimize usage of NTLM.</span></span> <span data-ttu-id="fcd1f-117">В результате доступ к Skype для бизнеса Server может быть ограничен внутренними или клиентами, подключенными через VPN или подключение DirectAccess.</span><span class="sxs-lookup"><span data-stu-id="fcd1f-117">As a result, access to Skype for Business Server might be restricted to internal or clients connected through a VPN or DirectAccess connection.</span></span>

<span data-ttu-id="fcd1f-121">Проверка подлинности в Skype для бизнеса Server состоит из двух этапов:</span><span class="sxs-lookup"><span data-stu-id="fcd1f-121">Skype for Business Server authentication consists of two phases:</span></span>

2. <span data-ttu-id="fcd1f-p108">Клиент и сервер используют существующее сопоставление безопасности для подписи сообщений, которые они отправляют, и для проверки получения сообщений. <span data-ttu-id="fcd1f-p109">Доверие пользователя прикрепляется к каждому сообщению, которое исходит от пользователя, но не к удостоверению пользователя. <span data-ttu-id="fcd1f-128">Пользователи с действительными выданными федеративным партнером учетными данными являются доверенными, но к ним могут применяться другие ограничения привилегий, которые полностью доступны внутренним пользователям.</span><span class="sxs-lookup"><span data-stu-id="fcd1f-128">Users with valid credentials issued by a federated partner are trusted but optionally prevented by additional constraints from enjoying the full range of privileges accorded to internal users.</span></span>
