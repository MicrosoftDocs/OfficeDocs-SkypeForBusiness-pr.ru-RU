# <a name="planning-to-turn-off-legacy-authentication-methods-internally-and-externally-to-your-network"></a>Планирование отключения устаревших методов проверки подлинности для внутренней и внешней сети.

> [!NOTE]
> Если вы собираетесь ознакомиться с этой статьей, вы уже знаете о поддерживаемых современных топологиях проверки подлинности, ADAL и о современной конфигурации проверки подлинности, но в случае если вы не можете найти нужные статьи, вот что нужно для начала: 
>  + [https://docs.microsoft.com/skypeforbusiness/plan-your-deployment/modern-authentication/topologies-supported](https://docs.microsoft.com/skypeforbusiness/plan-your-deployment/modern-authentication/topologies-supported)
>  + [https://docs.microsoft.com/skypeforbusiness/manage/authentication/use-adal](https://docs.microsoft.com/skypeforbusiness/manage/authentication/use-adal)
  
При использовании современной проверки подлинности не разрешается использовать более безопасные методы проверки подлинности, например двухфакторная проверка подлинности или проверка подлинности на основе сертификатов, она может выполнить авторизацию пользователя, не требуя имени пользователя и пароля. Это очень удобно.

В этой статье рассказывается о том, как использовать в своей сети старые методы для проверки подлинности, внешних, внутренних и и того, и других, с целью предотвращения атак типа "отказ на обслуживание" (DOS) на серверах Skype для бизнеса. Например, одним из способов предотвращения атак в DOS является отключение встроенной проверки подлинности Windows (включая NTLM и Kerberos). Отключение NTLM во внешней сети и полагаться на проверку подлинности на основе сертификатов помогает защитить пароли от уязвимости. Это связано с тем, что NTLM использует учетные данные пароля для проверки подлинности пользователей, но проверка подлинности на основе сертификатов — не поддерживается современной проверкой почтовых служб. Это означает, что один идеальный вариант для уменьшения числа атак через DOS — блокировка NTLM внешней стороны и использование только проверки подлинности на основе сертификатов.

Все правильно, приступим!

## <a name="what-would-you-be-changing"></a>Что вы хотите изменить? 

Эти командлеты работают как для SIP, так и для доступа к веб-службам. Несмотря на то, что эти два канала используют различные методы доступа, выполнение цветового диапазона из NTLM и Kerberos для анонимного доступа, все стандартные методы, используемые в Skype для бизнеса, были учтены.

## <a name="how-to-get-the-get--and-set-csauthconfig-cmdlets"></a>Получение командлетов Get-и Set-Ксаусконфиг

Эти командлеты будут установлены только в накопительном обновлении 2018 за июнь (6.0.9319.534) для Microsoft Skype для бизнеса Server 2015, а затем есть несколько топологий, которые можно выгрузить для вашего сервера Skype для бизнеса.

## <a name="topologies"></a>Топологий

Важно помнить, что это поддерживаемые топологии, задействованные в этом сценарии! Если вам нужно перейти в службу поддержки для блокирования метода, например, необходимо иметь конфигурацию из указанных ниже типов. 

> [!IMPORTANT]
> В таблице и описаниях ниже *современная проверка* подлинности сокращена __,__ а *Встроенная проверка подлинности Windows* — сокращена, чем __Win__. Напоминание о том, что встроенная проверка подлинности Windows состоит из двух методов: NTLM и проверки подлинности Kerberos. Для правильного чтения таблицы вам потребуется ознакомиться с этой таблицей.


|       |Внешний  |Внутренний  |Параметр  |
|---------|:---------|:---------|---------|
|__Введите 1__   |  MA + Win       | MA + Win         |  алловаллекстерналляндинтерналли       |
|__Введите 2__   |  СИЛА       | MA + Win         | блокквиндовсаусекстерналли        |
|__Тип 3__   |  СИЛА       | СИЛА        | блокквиндовсаусекстерналляндинтерналли        |
|__Введите 4__   |  СИЛА       | Клавиш        | блокквиндовсаусекстерналляндмодернаусинтерналли    |
|__Введите 5__   |  MA + Win       | Клавиш        | блоккмодернаусинтерналли         |

__Введите 1 Описание:__ Этот сценарий используется по умолчанию, если включена функция MA для сервера __Skype для бизнеса__ Server. Другими словами, это *отправная точка* при настройке MA.

__Введите 2 Описание:__ Эта топология блокирует *внешний*доступ к NTLM, но позволяет использовать протоколы NTLM или Kerberos (для клиентов, не поддерживающих ADAL) для *внутренних целей*. Если ваши клиенты поддерживают ADAL, они будут использовать для внутренних целей.

__Введите 3 Описание:__ Для этой топологии требуется мА для всех пользователей. Все клиенты, поддерживающие ADAL, будут работать в этой топологии, а пароли не будут использоваться, например, если вы отключите использование паролей с помощью проверки подлинности на основе сертификатов.

__Введите 4 Описание:__ Эта топология блокирует *Внешние* и мА для внутреннего использования NTLM. Это позволяет *всем клиентам* использовать устаревшие методы проверки подлинности для *внутренних целей* (даже для клиентов с поддержкой ADAL).

__Тип 5 Описание:__ *внешне*ваши современные клиенты ADAL будут использовать MA и любые клиенты, не поддерживающие ADAL, будут использовать устаревшие методы проверки подлинности. Но ** *все клиенты* будут использовать устаревший способ проверки подлинности (в том числе все клиенты, поддерживающие ADAL).

Вы совсем не сможете отслеживать цель защиты паролей в доступных параметрах. Имейте в виду, что идеальная ситуация — это использование MA External (например, путем настройки проверки подлинности на основе сертификатов), чтобы избежать атак DOS. Если вы хотите использовать его для внутренних клиентов, вы также сможете проверить свою сеть в соответствии с атаками Skype для бизнеса Server в будущем.

## <a name="why-to-use-set-csauthconfig-at-the-global-level"></a>Зачем использовать Set-Ксаусконфиг на глобальном уровне

Настройка `Set-CsAuthConfig` эффектов командлетов на ролях регистратора и веб-служб.

Этот командлет предназначен для работы на глобальном уровне сервера Skype для бизнеса. Оно *может* быть запущено на уровне группы, но *не рекомендуется* , поскольку оно повышает сложность установки. Выполнив эти команды на уровне группы, если в вашем пуле нет всех включенных ролей (например, у него нет веб-служб), параметры будут установлены только для роли регистратора. В этом случае веб-службы будут переноситься с параметрами глобального уровня, что может привести к путанице (особенно если это делается непреднамеренно).

Если клиент использует параметры регистратора из одного пула и параметры веб-служб из другого пула и параметры проверки подлинности находятся в несоответствующем состоянии, клиенты йоус не смогут войти в систему.

Кроме того, если в пуле есть только одна роль: 
* Set-будут установлены только те параметры, которые соответствуют существующей роли. Специальное предупреждение не будет выдано, так как некоторые параметры *не* были заданы. 
* Get — возвращают параметр, соответствующий существующей роли, и глобальные параметры для нее не существует.
* Если для пула не указана ни одна роль, как Set, так и Get — будет возвращено сообщение об ошибке.
* Если в пуле есть обе роли, но политики не определены на уровне пула, функция Get-возвратит сообщение об ошибке.

Это может полагаться на то, что нужно сделать, чтобы получить эти значения, а также на снимок экрана или записать начальное состояние перед внесением изменений. Вы также можете сохранить журнал изменений в OneNote.

> [!NOTE]
> 
> Примечание. После настройки Ксаусконфиг необходимо запустить Enable-Кскомпутер на каждом компьютере, чтобы параметры вступили в силу.

> [!IMPORTANT]
> Если вы используете Lync Web Access (лва) и хотите использовать для внешнего доступа доступ к данным на основе форм (FBA), перенастройте лва таким образом, чтобы клиенты могли получать к ним доступ с помощью анонимного доступа для поддержки этих сценариев. Кроме того, если вы используете ПИН-код телефонного подключения, FBA будет заблокирован только для внешних пользователей. Если вам нужно изменить ПИН-код, для внутренних целей они должны будут входить в свою учетную запись.

> [!NOTE]
> 
> Если вы используете параметр Блокквиндовсаусекстерналли для использования внешней блокировки NTLM, имейте в виду, что это также блокирует внутренние протоколы NTLM для канала SIP. Тем не менее, пользователи Skype для бизнеса и Lync более чем 2010 смогут выполнять вход, так как они будут использовать NTLM через HTTP для входа, внутренне и затем получать сертификат для входа в систему SIP. Тем не менее, в этом случае клиентам более ранних версий, чем 2010, будет невозможно выполнить вход в систему, и вам может потребоваться обновить эти приложения, чтобы пользователи могли возобновлять безопасность функций.
