# <a name="edge-server-system-requirements-in-skype-for-business-server"></a>Требования к системе пограничных серверов в Skype для бизнеса Server
 
**Сводка:** Сведения о требованиях к системе для пограничного сервера в Skype для бизнеса Server.
  
Когда речь идет о развертывании пограничного сервера Skype для бизнеса Server, необходимо выполнить эти действия для сервера или серверов, которые находятся в самой среде, а также для планирования структуры среды. Для получения дополнительных сведений о топологии, DNS, сертификатах и других проблемах с инфраструктурой ознакомьтесь с документацией по требованиям к охране окружающей среды.
  
## <a name="components"></a>Компоненты

При обсуждении среды пограничного сервера мы создаем ссылки на компоненты, которые в большей части разворачиваются в сети периметра (например, в Рабочей группе или домене, который находится за пределами вашей структуры доменов Skype для бизнеса Server).
  
Учитывая это, необходимо учитывать компоненты, которые необходимо учитывать при успешном развертывании пограничного сервера.
  
- [Пограничные серверы](system-requirements.md#EdgeServers)
    
- [Обратные прокси-серверы](system-requirements.md#ReverseProxies)
    
- [Брандмауэров](system-requirements.md#Firewalls)
    
- [Директора](system-requirements.md#Directors) (это необязательно, и если они включены, они будут расположены во внутренней сети)
    
- Подсистемы балансировки [нагрузки](system-requirements.md#LoadBalancers) (можно использовать балансировку нагрузки на DNS или аппаратный балансировщик нагрузки (HLB), но для одного пограничного сервера это не требуется).
    
Ниже приведены подробные сведения по каждому из них.
  
### <a name="edge-servers"></a>Пограничные серверы
<a name="EdgeServers"> </a>

Это серверы Skype для бизнеса, развернутые в среде периметра. Их роль состоит в том, чтобы отправлять и принимать сетевой трафик внешним пользователям для служб, предоставляемых внутренним развертыванием Skype для бизнеса Server. Для успешного выполнения этой задачи каждый пограничный сервер выполняет следующие действия.
  
- **Пограничная служба доступа**: предоставляет одну надежную точку подключения для исходящего и входящего трафика протокола SIP.
    
- **Пограничная служба веб-конференций**: позволяет внешним пользователям присоединяться к собраниям, размещенным во внутренней среде Skype для бизнеса Server.
    
- **Пограничная служба аудио-** и видеоданных, совместного использования приложений и передачи файлов внешним пользователям.
    
- **Прокси-служба XMPP**: принимает и отправляет сообщения по протоколу расширения и из настроенных ФЕДЕРАТИВНЫХ партнеров XMPP.
    
Авторизованные внешние пользователи могут использовать пограничные серверы для подключения к внутреннему развертыванию Skype для бизнеса Server, но в противном случае они не предоставляют другим пользователям доступ к внутренней сети.
  
> [!NOTE]
> Пограничные серверы развертываются для предоставления подключений для включенных клиентов Skype для бизнеса и других пограничных серверов (в сценариях Федерации). Вы не можете подключаться к другим типам клиентов и серверов конечной точки. Сервер шлюза XMPP может разрешать подключения с настроенными партнерами XMPP. Но опять же, это только те типы клиентов и Федерации, которые будут работать. 

> [!NOTE]
> Шлюзы и прокси-серверы XMPP доступны в Skype для бизнеса Server 2015, но больше не поддерживаются в Skype для бизнеса Server 2019. Дополнительные сведения приведены в статье [Миграция Федерации XMPP](../../../SfBServer2019/migration/migrating-xmpp-federation.md) .
  
### <a name="reverse-proxies"></a>Обратные прокси-серверы
<a name="ReverseProxies"> </a>

Сервер обратного прокси (RP) не имеет роли Skype для бизнеса Server, но является важнейшим компонентом развертывания пограничного сервера. Обратный прокси-сервер позволяет внешним пользователям выполнять следующие действия:
  
- подключение к собраниям и конференциям с телефонным подключением с помощью простых URL-адресов.
    
- скачивание контента собраний.
    
- Разверните узел группы рассылки.
    
- получение пользовательских сертификатов для проверки подлинности на основе сертификатов клиентов
    
- Загрузка файлов с сервера адресной книги или отправка запросов в службу веб-запросов к адресной книге.
    
- получение обновлений для клиентского программного обеспечения и программного обеспечения для устройств.
    
И для мобильных устройств:
  
- Она позволяет автоматически обнаруживать серверы переднего плана с мобильными службами.
    
- Он позволяет отправлять push-уведомления от Microsoft 365 или Office 365 на мобильные устройства.
    
Наши текущие рекомендации по обратному прокси-серверу можно найти на странице [инфраструктура телефонии для Skype для бизнеса](https://docs.microsoft.com/SkypeForBusiness/certification/infra-gateways) . Поэтому обратный прокси-сервер:
  
- для подключения к опубликованным внешним веб-службам следует использовать протокол TLS, который появился в вашей среде с помощью общедоступных сертификатов.
    
  - директор или пул директоров;
    
  - Сервер переднего плана или интерфейсный пул
    
- необходимо иметь возможность публиковать внутренние веб-сайты с помощью сертификатов для шифрования или при необходимости публиковать их по незашифрованным средствам.
    
- возможность публикации внутреннего размещенного веб-сайта с помощью полного доменного имени (FQDN).
    
- должна иметь возможность публиковать все содержимое размещенного веб-сайта. По умолчанию можно использовать **/\\** директиву *, которая распознается большинством веб-серверов в качестве значения "опубликовать все содержимое на веб-сервере". Кроме того, можно изменить директиву (например, * */Увка/ \\ * * *), что означает "опубликовать весь контент в виртуальном каталоге Ucwa".
    
- для клиентов, запрашивающих контент с опубликованного веб-сайта, требуются TLS-подключения.
    
- принимает сертификаты с записями альтернативного имени субъекта (SAN).
    
- должна иметь возможность разрешить привязку сертификата к прослушивателю или интерфейсу, через который будет разрешаться полное доменное имя внешних веб-служб. Конфигурации прослушивателя предпочтительны для интерфейсов. Для одного интерфейса можно настроить множество прослушивателей.
    
- необходимо разрешить настройку обработки заголовков узлов. Часто исходный заголовок узла, отправленный запрашивающим клиентом, необходимо передавать прозрачным, а не изменять обратном прокси-сервером.
    
- Разрешите мосту трафика TLS с одного порта, определенного извне (например, TCP 443), на другой определенный порт (например, TCP 4443). Обратный прокси-сервер может расшифровать полученный пакет и затем повторно зашифровать пакет при отправке.
    
- Разрешите мосту незашифрованного TCP-трафика с одного порта (например, TCP 80) на другой (например, TCP 8080).
    
- необходимо разрешить настройку, или принять, проверку подлинности NTLM, без проверки подлинности и сквозную проверку подлинности.
    
Если обратный прокси-сервер может удовлетворить все требования, указанные в этом списке, следует иметь в виду, что мы рекомендуем использовать приведенные выше ссылки.
  
### <a name="firewalls"></a>Брандмауэров
<a name="Firewalls"> </a>

Необходимо разместить пограничный сервер, наблюдая за внешним брандмауэром, но рекомендуем использовать два брандмауэра, один внешний и один внутренний между пограничной средой и внутренней средой. Все наши документы в наших сценариях будут иметь два брандмауэра. Рекомендуется использовать два брандмауэра, так как они обеспечивают гарантированную маршрутизацию из одной сети в другую и удваивает защиту брандмауэра для внутренней сети.
  
### <a name="directors"></a>Директор
<a name="Directors"> </a>

Это необязательная роль. Это может быть один сервер или пул серверов, на которых работает роль директора. Это роль, которую можно найти во внутренней среде Skype для бизнеса Server.
  
Директор — это внутренний сервер следующего прыжка, который получает входящий трафик SIP от пограничных серверов, предназначенных для внутренних серверов Skype для бизнеса Server. Он предварительно проверяет подлинность входящих запросов и перенаправляет их в домашний пул или на сервер пользователя. Такая предварительная проверка подлинности позволяет удалять неопознанные запросы учетных записей пользователей.
  
Зачем это важно? Для директоров важно обеспечить защиту серверов Standard Edition и серверов переднего плана или интерфейсных пулов от вредоносных данных, таких как атаки типа "отказ в обслуживании" (DoS). Если сеть передается с недопустимым внешним трафиком, трафик будет остановлен в директоре.
  
### <a name="load-balancers"></a>Подсистемы балансировки нагрузки
<a name="LoadBalancers"> </a>

Масштабируемая консолидированная пограничная топология Skype для бизнеса Server оптимизирована для балансировки нагрузки на DNS для новых развертываний, поэтому рекомендуется использовать эту службу. Если вам нужна высокая доступность, рекомендуется использовать аппаратный балансировщик нагрузки для одной конкретной ситуации:
  
- Единая система обмена сообщениями Exchange для удаленных пользователей с использованием единой системы обмена сообщениями Exchange **до** Exchange 2013.
    
> [!IMPORTANT]
> Важно отметить, что вы не можете смешивать системы балансировки нагрузки. В среде Skype для бизнеса Server все интерфейсы должны использовать либо DNS, либо HLB. 
  
> [!NOTE]
> В Skype для бизнеса Server не поддерживается преобразование сетевых адресов для прямого возврата сервера (DSR). 
  
#### <a name="hardware-load-balancer-requirements-for-edge-servers-edge-servers-running-the-av-edge-service"></a>требования к аппаратному подсистеме балансировки нагрузки для пограничных серверов, на которых работает пограничная служба аудио-и видеоданных

Для любого пограничного сервера, на котором работает пограничная служба аудио-и видеоданных, необходимы следующие требования:
  
- Отключите протокол TCP Nagle как для внутренних, так и для внешних портов 443 (Nagle — это процесс объединения нескольких небольших пакетов в один более крупный пакет для более эффективной передачи).
    
- Отключите протокол TCP Nagle для диапазона внешних портов 50000-59999.
    
- Не используйте NAT на внутреннем или внешнем брандмауэре.
    
- Внутренний интерфейс пограничного сервера должен находиться в той же сети, что и внешний интерфейс пограничного сервера, а маршрутизация между ними должна быть отключена.
    
- Внешний интерфейс любого пограничного сервера, работающего под управлением пограничной службы аудио-и видеоданных, должен использовать общедоступные IP-адреса без NAT или преобразования портов на любом из внешних IP-адресов пограничных серверов.
    
#### <a name="hlb-requirements"></a>Требования к HLB

В Skype для бизнеса Server нет большого количества требований к сходству на основе файлов cookie. Поэтому нет необходимости использовать сохраняемость на основе файлов cookie, **Если** (и это для Skype для бизнеса Server 2015), которые будут иметь серверы переднего плана Lync Server 2010 или интерфейсные пулы в среде Skype для бизнеса Server. Для них потребуется сходство на основе файлов cookie в методе конфигурации, рекомендуемом для Lync Server 2010.
  
> [!NOTE]
> Если вы решили включить сходство на основе файлов cookie для HLB, это не будет проблемой, даже если в вашей среде это не требуется. 
  
Если в вашей среде **не** требуется сходство на основе файлов cookie:
  
- В правиле публикации обратного прокси-сервера для порта 443 задайте для **заголовка сервера пересылки** **значение true**. Это обеспечит перенаправление исходного URL-адреса.
    
Для развертываний **, для которых требуется** сходство на основе файлов cookie:
  
- В правиле публикации обратного прокси-сервера для порта 443 задайте для **заголовка сервера пересылки** **значение true**. Это обеспечит перенаправление исходного URL-адреса.
    
- Файл cookie аппаратного балансировщика нагрузки **не должен** быть помечен как httpOnly.
    
- Файл cookie аппаратного балансировщика нагрузки **не должен** иметь срок действия.
    
- Файл cookie аппаратного балансировщика нагрузки **должен** называться **MS-WSMan** (это значение, ожидаемое веб-службами и не может быть изменено).
    
- Файл cookie аппаратного балансировщика нагрузки **должен** быть задан во всех HTTP-ответах, для которых в входящем HTTP-запросе отсутствует файл cookie, независимо от того, получил ли предыдущий HTTP-ответ на это же TCP-соединение, файл cookie. Если аппаратная подсистема балансировки нагрузки оптимизирует файл cookie только один раз для каждого подключения по протоколу TCP, эту оптимизацию использовать **не следует** .
    
> [!NOTE]
> Как правило, для конфигураций HLB используется время жизни сеанса связи с исходным и 20 минутами, что подходит для Skype для бизнеса Server и клиентов, так как состояние сеанса поддерживается при использовании клиента и/или при взаимодействии приложения. 
  
Если вы развертываете мобильные устройства, ваш HLB должен иметь возможность балансировки нагрузки отдельных запросов в сеансе TCP (в действительности необходимо обеспечить балансировку нагрузки для отдельного запроса, основанного на целевом IP-адресе).
  
> [!IMPORTANT]
> F5 портами имеет функцию с именем OneConnect. Это гарантирует, что для каждого запроса в TCP-подключении устанавливается отдельное балансировка нагрузки. Если вы развертываете мобильные устройства, убедитесь, что поставщик HLB поддерживает те же функции. Последним мобильным приложениям iOS требуется TLS версии 1,2. Если вам нужно больше узнать, F5 предоставляет конкретные параметры для этого. 
  
Ниже приведены требования к HLB для директоров (необязательно) и (обязательный) интерфейсных веб-служб пула.
  
- Для IP-адресов внутренних веб-служб установите Source_addr сохраняемости (внутренний порт 80, 443) в HLB. Для Skype для бизнеса Server Source_addr сохраняемость означает, что несколько подключений, поступающих из одного IP-адреса, всегда отправляются на один сервер, чтобы поддерживать состояние сеанса.
    
- Используйте время ожидания TCP в течение 1800 секунд.
    
- На брандмауэре между обратным прокси-сервером и HLB пула следующего прыжка создайте правило, разрешающее HTTPS: трафик через порт 4443, из обратного прокси-сервера в HLB. HLB необходимо настроить на прослушивание портов 80, 443 и 4443.

#### <a name="port-monitoring-for-hlbs"></a>Мониторинг порта для портами

Необходимо реализовать мониторинг портов на HLB для мониторинга следующего для внешнего интерфейса HLB:
