# <a name="edge-server-system-requirements-in-skype-for-business-server"></a>Требования к системе Edge Server в Skype для бизнеса Server
 
**Сводка:** Узнайте о системных требованиях к edge Server в Skype для бизнеса Server.
  
Когда речь заходит о развертывании Skype для бизнеса Server Edge Server, это то, что необходимо сделать для сервера или серверов, которые находятся в самой среде, а также планирование структуры среды. Дополнительные сведения о топологии, DNS, сертификатах и других проблемах инфраструктуры ознакомьтесь с документацией по экологическим требованиям.
  
## <a name="components"></a>Компоненты

При обсуждении среды Edge Server мы ссылаемся на компоненты, которые по большей части развернуты в сети периметра (то есть в группе или домене, который находится вне структуры Skype для бизнеса Server домена).
  
Имейте это в виду, это компоненты, которые необходимо иметь в виду для успешного развертывания edge:
  
- [Edge Servers](system-requirements.md#EdgeServers)
    
- [Обратные прокси-серверы](system-requirements.md#ReverseProxies)
    
- [Брандмауэры](system-requirements.md#Firewalls)
    
- [Директора](system-requirements.md#Directors) (они необязательны, и если они включены, они будут расположены во внутренней сети)
    
- [Балансировки](system-requirements.md#LoadBalancers) нагрузки (вы можете иметь балансировку нагрузки DNS или балансировку нагрузки оборудования (HLB), но для одного edge Server это не требуется)
    
Ниже приведены более подробные данные по каждому из этих ниже.
  
### <a name="edge-servers"></a>Edge Servers
<a name="EdgeServers"> </a>

Это серверы Skype для бизнеса, развернутые в среде периметра. Их роль заключается в отправке и приеме сетевого трафика внешним пользователям для служб, предлагаемых внутренними Skype для бизнеса Server развертывания. Чтобы сделать это успешно, каждый edge Server запускается:
  
- **Служба Access Edge**: предоставляет единую доверяемую точку подключения для исходящие и входящие протоколы инициации сеансов (SIP).
    
- **Служба Web Conferencing Edge**: позволяет внешним пользователям присоединяться к собраниям, которые находятся в вашей внутренней Skype для бизнеса Server среде.
    
- **Служба A/V Edge**: делает доступ к аудио, видео, совместному доступу к приложениям и передаче файлов внешним пользователям.
    
- **Прокси-служба XMPP**: принимает и отправляет extensible сообщения с протоколом обмена сообщениями и протоколом присутствия (XMPP) в настроенные партнеры XMPP Federated и от них.
    
Авторизованные внешние пользователи могут использовать ваши edge Servers для подключения к вашему внутреннему развертыванию Skype для бизнеса Server, но в противном случае они не предоставляют другого доступа к вашей внутренней сети для всех.
  
> [!NOTE]
> Edge Servers развернуты для обеспечения подключений для Skype для бизнеса и других edge Servers (в сценариях федерации). Вы не можете подключиться к другим типам конечных точеных клиентов или серверов. Сервер шлюза XMPP может разрешать подключения с настроенными партнерами XMPP. Но опять же, это единственные клиенты и типы федераций, которые будут работать. 

> [!NOTE]
> Шлюзы и прокси XMPP доступны в Skype для бизнеса Server 2015 г., но больше не поддерживаются Skype для бизнеса Server 2019 г. Дополнительные сведения см. в дополнительных сведениях о переносе федерации [XMPP](../../../SfBServer2019/migration/migrating-xmpp-federation.md) .
  
### <a name="reverse-proxies"></a>Обратные прокси-серверы
<a name="ReverseProxies"> </a>

Сервер обратного прокси-сервера (RP) не имеет Skype для бизнеса Server роли, но является важным компонентом развертывания Edge Server. Обратный прокси-сервер позволяет внешним пользователям:
  
- подключение к собраниям или конференц-связи с помощью простых URL-адресов.
    
- загрузка контента собрания.
    
- расширение групп рассылки.
    
- получить пользовательские сертификаты для проверки подлинности на основе клиентского сертификата
    
- скачивание файлов с сервера адресной книги или отправка запросов в службу веб-запросов адресной книги.
    
- получение обновлений клиентского программного обеспечения и программного обеспечения устройств.
    
А для мобильных устройств:
  
- это позволяет им автоматически обнаруживая серверы переднего конечного сервера, предлагающие службы мобильности.
    
- он включает push-уведомления от Microsoft 365 или Office 365 на мобильные устройства.
    
Наши рекомендации обратного прокси-сервера можно найти на странице Инфраструктура телефонии [для](../../../SfbPartnerCertification/certification/infra-gateways.md) Skype для бизнеса. Таким образом, обратный прокси:
  
- должны быть в состоянии использовать безопасность транспортного слоя (TLS), который введен в среду с помощью общедоступных сертификатов для подключения к опубликованным внешним веб-службам:
    
  - директор или пул директоров;
    
  - Front End Server или пул переднего конца
    
- необходимо иметь возможность публиковать внутренние веб-сайты с помощью сертификатов для шифрования или публиковать их незашифрованными средствами, если это необходимо.
    
- должны иметь возможность публиковать внутренний веб-сайт внешне с помощью полностью квалифицированного доменного имени (FQDN).
    
- необходимо иметь возможность публиковать все содержимое размещенного веб-сайта. По умолчанию можно использовать **/\\** директиву _, которая распознается большинством веб-серверов, чтобы означать "Публикация всего контента на веб-сервере". Вы также можете изменить директиву, например _*/Uwca/\\***, что означает "Публикация всего контента в виртуальном каталоге Ucwa".
    
- должны требовать подключения TLS к клиентам, которые запрашивают контент на опубликованном веб-сайте.
    
- должен принимать сертификаты с альтернативными именами субъекта (SAN).
    
- необходимо разрешить привязку сертификата к слушателю или интерфейсу, с помощью которого будет решаться FQDN внешних веб-служб. Конфигурации слушателей предпочтительнее интерфейсов. Многие слушатели могут быть настроены на одном интерфейсе.
    
- должна разрешить конфигурацию обработки загона хостов. Часто исходный загон хост, отправленный запрашивающий клиент, должен передаваться прозрачно, а не изменяться обратным прокси-сервером.
    
- следует разрешить перенос трафика TLS из одного внешне определенного порта (например, TCP 443) в другой определенный порт (например, TCP 4443). Обратный прокси-сервер может расшифровать пакет в квитанции, а затем повторно расшифровать пакет при отправке.
    
- должно разрешить перенапряжение незашифрованного трафика TCP из одного порта (например, TCP 80) в другой (например, TCP 8080).
    
- необходимо разрешить настройку или принятие проверки подлинности NTLM, отсутствие проверки подлинности и проходной проверки подлинности.
    
Если обратный прокси-сервер может решить все потребности в этом списке, вы должны быть хороши, но имейте в виду наши рекомендации по ссылке, предоставленной выше.
  
### <a name="firewalls"></a>Брандмауэры
<a name="Firewalls"> </a>

Необходимо поместить развертывание Edge за внешний брандмауэр, но мы рекомендуем иметь два брандмауэра, один внешний и внутренний между средой Edge и внутренней средой. Вся документация в наших сценариях будет иметь два брандмауэра. Мы рекомендуем два брандмауэра, так как они обеспечивают строгую маршрутику от одного края сети к другому и удваив защиту брандмауэра для внутренней сети.
  
### <a name="directors"></a>Директора
<a name="Directors"> </a>

Это необязательная роль. Это может быть один сервер или пул серверов с ролью Директора. Это роль, найденная во внутренней Skype для бизнеса Server среде.
  
Директор — это внутренний следующий хоп-сервер, который получает входящий трафик SIP с edge Servers, предназначенный для Skype для бизнеса Server внутренних серверов. Он предавентицирует входящие запросы и перенаправляет их в домашний пул или сервер пользователя. Эта предварительная запись позволяет отбросить неопознанный запрос на учетную запись пользователя.
  
Почему это важно? Важной функцией для директора является защита выпуск Standard серверов и серверов переднего или переднего конечного трафика от вредоносных атак, таких как атаки с отказом в обслуживании (DoS). Если сеть затоплена недействительным внешним трафиком, трафик останавливается в Director.
  
### <a name="load-balancers"></a>Подсистемы балансировки нагрузки
<a name="LoadBalancers"> </a>

Масштабированная Skype для бизнеса Server topology Edge оптимизирована для балансировки нагрузки DNS для новых развертывания, и мы рекомендуем это. При необходимости высокой доступности рекомендуется использовать балансировку нагрузки оборудования для одной конкретной ситуации:
  
- Exchange для удаленных пользователей с **Exchange до Exchange** 2013 г.
    
> [!IMPORTANT]
> Важно отметить, что нельзя смешивать балансиры нагрузки. В вашей Skype для бизнеса Server все интерфейсы должны использовать DNS или HLB. 
  
> [!NOTE]
> Nat direct server return (DSR) не поддерживается для Skype для бизнеса Server. 
  
#### <a name="hardware-load-balancer-requirements-for-edge-servers-edge-servers-running-the-av-edge-service"></a>требования к балансире нагрузки оборудования для edge Servers Edge Servers, работающих в службе A/V Edge

Для любого edge Server, на который работает служба A/V Edge, необходимо:
  
- Отключение подковывания TCP для внутренних и внешних портов 443 (nagling — это процесс объединения нескольких небольших пакетов в один, более крупный пакет для более эффективной передачи).
    
- Отключите поднагружение TCP для внешнего диапазона порта 50000 - 59999.
    
- Не используйте NAT на внутренних или внешних брандмауэрах.
    
- Внутренний интерфейс Edge должен быть в другой сети, чем внешний интерфейс Edge Server, и маршрутивка между ними должна быть отключена.
    
- Внешний интерфейс любого edge Server под управлением службы A/V Edge должен использовать общедоступные маршрутируемые IP-адреса, а на внешних IP-адресах Edge нет перевода nat или порта.
    
#### <a name="hlb-requirements"></a>Требования HLB

Skype для бизнеса Server не имеет много требований сродства на основе файлов cookie. Таким образом, вам не нужно использовать сохранение на основе файлов cookie, если  (а это не Skype для бизнеса Server 2015 г.) в вашей среде Skype для бизнеса Server серверов Lync Server 2010. Им потребуется сродство на основе файлов cookie в методе конфигурации, рекомендуемом для Lync Server 2010.
  
> [!NOTE]
> Если вы решите включить сродство на основе файлов cookie для HLB, это не будет проблемой, даже если среда не нуждается в этом. 
  
Если среде **не требуется** сродство на основе файлов cookie:
  
- В обратном правиле публикации прокси-серверов для порта 443 установите заглавную хозяйку **Forward** для **True**. Это гарантирует, что исходный URL-адрес будет переназначен.
    
Для развертывания, **которые требуют** сродства на основе файлов cookie:
  
- В обратном правиле публикации прокси-серверов для порта 443 установите заглавную хозяйку **Forward** для **True**. Это гарантирует, что исходный URL-адрес будет переназначен.
    
- Маркер балансиров аппаратной нагрузки **не должен быть** отмечен httpOnly.
    
- Cookie-файл балансиров аппаратной нагрузки **не должен** иметь срока действия.
    
- Файл cookie балансиров аппаратной нагрузки должен называться **MS-WSMAN** (это значение, которое ожидают веб-службы, и его нельзя изменить).
    
- Файл cookie балансиста нагрузки оборудования должен быть установлен в каждом ответе HTTP, для которого входящий http-запрос не имел cookie, независимо от того, получил ли предыдущий ответ HTTP на том же соединении TCP. Если балансировка нагрузки оборудования оптимизирует вставку cookie только один раз на подключение TCP, эту оптимизацию **нельзя использовать** .
    
> [!NOTE]
> Для конфигураций HLB характерно использование срока службы сеанса TCP с исходным кодом и продолжительностью 20 минут, что отлично для Skype для бизнеса Server и его клиентов, так как состояние сеанса поддерживается с помощью клиентского использования и/или взаимодействия с приложениями. 
  
При развертывании мобильных устройств ваш HLB должен иметь возможность загружать баланс отдельных запросов в сеансе TCP (по сути, необходимо иметь возможность загружать баланс отдельного запроса на основе целевого IP-адреса).
  
> [!IMPORTANT]
> HLB F5 имеют функцию OneConnect. Это гарантирует, что каждый запрос в соединении TCP будет индивидуально сбалансированным. При развертывании мобильных устройств убедитесь, что поставщик HLB поддерживает те же функции. Для последних мобильных приложений iOS требуется TLS-версия 1.2. Если вам нужно узнать больше, F5 предоставляет определенные параметры для этого. 
  
Вот требования HLB к веб-службам (необязательным) директору и (обязательно) веб-службам пула переднего конца:
  
- Для внутренних VIP-пользователей веб-служб установите Source_addr (внутренний порт 80, 443) в HLB. Для Skype для бизнеса Server, Source_addr сохраняемость означает, что несколько подключений с одного IP-адреса всегда отправляются на один сервер для поддержания состояния сеанса.
    
- Использование простоя TCP в течение 1800 секунд.
    
- На брандмауэре между обратным прокси-сервером и HLB следующего пула хопа создайте правило, позволяя https: трафик в порту 4443, из обратного прокси-сервера в HLB. HLB необходимо настроить для прослушивания в портах 80, 443 и 4443.

#### <a name="port-monitoring-for-hlbs"></a>Мониторинг порта для HLBs

Вы определяете мониторинг порта на балансиры нагрузки оборудования, чтобы определить, когда конкретные службы недоступны из-за сбоя оборудования или связи. Например, если служба переднего сервера (RTCSRV) прекращает работу из-за сбоя переднего сервера или пула переднего конца, мониторинг HLB также должен прекратить прием трафика в веб-службах. Чтобы отслеживать внешний интерфейс HLB, необходимо реализовать мониторинг порта в HLB:
  
|**Виртуальный IP-адрес/порт**|**Порт узла**|**Компьютер/монитор узла**|**Профиль сохраняемости**|**Примечания**|
|:-----|:-----|:-----|:-----|:-----|
|\<pool\>web_mco_443_vs  <br/> 443  <br/> |4443  <br/> |Сервер переднего плана  <br/> 5061  <br/> |Нет  <br/> |HTTPS  <br/> |
|\<pool\>web_mco_80_vs  <br/> 80  <br/> |8080  <br/> |Сервер переднего плана  <br/> 5061  <br/> |Нет  <br/> |HTTP  <br/> |
