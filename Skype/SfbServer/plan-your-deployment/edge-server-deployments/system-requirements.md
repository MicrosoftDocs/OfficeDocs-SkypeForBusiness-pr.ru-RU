# <a name="edge-server-system-requirements-in-skype-for-business-server"></a>Требования к системе для сервера Edge Server в Skype для бизнеса Server
 
**Сводка:** Узнайте о системных требованиях для edge Server в Skype для бизнеса Server.
  
Что касается развертывания skype для бизнеса Server Edge Server, это то, что необходимо сделать для серверов или серверов, которые находятся в самой среде, а также планирование структуры среды. Дополнительные сведения о топологии, DNS, сертификатах и других проблемах инфраструктуры можно узнать в документации по требованиям к среде.
  
## <a name="components"></a>Компоненты

При обсуждении среды edge Server мы ссылаемся на компоненты, которые, по большей части, развернуты в сети периметра (то есть находятся в группе или домене, который находится за пределами структуры домена Skype для бизнеса Server).
  
Имейте в виду, что для успешного развертывания edge необходимо иметь в виду указанные компоненты.
  
- [Edge Servers](system-requirements.md#EdgeServers)
    
- [Обратные прокси-серверы](system-requirements.md#ReverseProxies)
    
- [Брандмауэры](system-requirements.md#Firewalls)
    
- [Директоры](system-requirements.md#Directors) (они необязательны, и если они включены, они будут расположены во внутренней сети)
    
- [Балансировщиков нагрузки](system-requirements.md#LoadBalancers) (можно использовать балансировку нагрузки на DNS или аппаратную балансировку нагрузки( HLB), но для одного edge-сервера это не требуется)
    
Ниже приведены более подробные данные о каждом из этих ок.
  
### <a name="edge-servers"></a>Edge Servers
<a name="EdgeServers"> </a>

Это серверы Skype для бизнеса, развернутые в среде периметра. Их роль заключается в отправке и приеме сетевого трафика внешним пользователям для служб, предлагаемых внутренним развертыванием Skype для бизнеса Server. Для этого выполняется каждый из них:
  
- **Пограничная служба доступа**: предоставляет единую доверенного точку подключения для исходящие и входящие SIP-трафика.
    
- **По краям** веб-службы: позволяет внешним пользователям присоединяться к собраниям, которые находятся во внутренней среде Skype для бизнеса Server.
    
- **Поменая служба** аудио- и видеосвязи: делает аудио- и видеосвязь, общий доступ к приложениям и передачу файлов внешним пользователям.
    
- **Служба прокси-сервера XMPP:** принимает и отправляет сообщения XMPP настроенным федератным партнерам XMPP и от них.
    
Авторизованные внешние пользователи могут использовать ваши серверы для подключения к внутреннему развертыванию Skype для бизнеса Server, но в противном случае они не предоставляют другого доступа к внутренней сети.
  
> [!NOTE]
> Edge Servers are deployed to provide connections for enabled Skype for Business clients and other Edge Servers (in federation scenarios). Нельзя подключаться к другим типам клиентов или серверов конечных точек. Сервер шлюза XMPP может разрешать подключения с настроенными партнерами XMPP. Но опять же, это единственные типы клиентов и федерации, которые будут работать. 

> [!NOTE]
> Шлюзы и прокси-серверы XMPP доступны в Skype для бизнеса Server 2015, но больше не поддерживаются в Skype для бизнеса Server 2019. Дополнительные сведения см. в переносе [федерации XMPP.](../../../SfBServer2019/migration/migrating-xmpp-federation.md)
  
### <a name="reverse-proxies"></a>Обратные прокси-серверы
<a name="ReverseProxies"> </a>

Сервер обратного прокси-сервера (RP) не имеет роли Skype для бизнеса Server, но является важным компонентом развертывания edge Server. Обратный прокси-сервер позволяет внешним пользователям:
  
- подключаться к собраниям или конференциям с подключением по телефонной связи с помощью простых URL-адресов.
    
- скачать содержимое собрания.
    
- развернуть группы рассылки.
    
- получить пользовательские сертификаты для проверки подлинности на основе сертификатов клиентов
    
- скачивание файлов с сервера адресной книги или отправка запросов в службу веб-запросов адресной книги.
    
- получение обновлений программного обеспечения клиента и устройства.
    
А для мобильных устройств:
  
- это позволяет им автоматически обнаруживть серверы переднего сервера, предоставляющие мобильные службы.
    
- Он включает push-уведомления из Microsoft 365 или Office 365 на мобильных устройствах.
    
Текущие рекомендации по обратному прокси-серверу можно найти на странице ["Инфраструктура телефонии для Skype для бизнеса".](https://docs.microsoft.com/SkypeForBusiness/certification/infra-gateways) Поэтому обратный прокси-сервер:
  
- должна иметь возможность использовать TLS, введенный в среду через общедоступные сертификаты, для подключения к опубликованным внешним веб-службам:
    
  - директор или пул директоров;
    
  - Сервер переднего сервера или пул переднего сервера
    
- должна иметь возможность публиковать внутренние веб-сайты с помощью сертификатов для шифрования или при необходимости публиковать их с помощью незашифрованного средства.
    
- должна иметь возможность публиковать веб-сайт, который размещен внутри организации, с помощью полного доменного имени.
    
- должна иметь возможность публиковать все содержимое размещенного веб-сайта. По умолчанию можно использовать директиву _, которая распознается большинством веб-серверов, чтобы означать "Опубликовать весь контент на **/\\** веб-сервере". Вы также можете изменить директиву, например _*/Uwca/, \\* *_ что означает "Опубликовать все содержимое в виртуальном каталоге Ucwa".
    
- должны требовать подключения TLS к клиентам, запрашивая содержимое с опубликованного веб-сайта.
    
- должен принимать сертификаты с записями альтернативного имени субъекта (SAN).
    
- должна иметь возможность разрешить привязку сертификата к прослушивательу или интерфейсу, через который будет разрешаться FQDN внешних веб-служб. Конфигурации прослушиватель предпочтительнее интерфейсов. В одном интерфейсе можно настроить несколько прослушивателей.
    
- должно разрешать настройку обработки загона хоста. Часто исходный загон, отправленный запрашивающий клиент, должен передаваться прозрачно, а не изменяться обратным прокси-сервером.
    
- следует разрешить перенаружение трафика TLS из одного внешне определенного порта (например, TCP 443) в другой определенный порт (например, TCP 4443). Обратный прокси-сервер может расшифровать пакет при получении, а затем повторно расшифровать пакет при отправке.
    
- следует разрешить перенагрузку незашифрованного трафика TCP из одного порта (например, TCP 80) в другой (например, TCP 8080).
    
- необходимо разрешить настройку или принятие проверки подлинности NTLM, отсутствие проверки подлинности и сквозную проверку подлинности.
    
Если обратный прокси-сервер может решить все потребности в этом списке, вам следует перейти, но помните о наших рекомендациях по ссылке, предоставленной выше.
  
### <a name="firewalls"></a>Брандмауэры
<a name="Firewalls"> </a>

Необходимо поместить развертывание edge за внешним брандмауэром, но мы рекомендуем использовать два брандмауэра: один внешний и внутренний между средой edge и внутренней средой. Вся документация по нашим сценариям будет иметь два брандмауэра. Рекомендуется использовать два брандмауэра, так как они обеспечивают строгую маршрутику от одного края сети к другому и удваив защиту брандмауэра для внутренней сети.
  
### <a name="directors"></a>Директоры
<a name="Directors"> </a>

Это необязательная роль. Это может быть один сервер или пул серверов с ролью Директора. It's a role found on the internal Skype for Business Server environment.
  
Директор — это внутренний сервер следующего перехода, который получает входящий трафик SIP от серверов, предназначенных для внутренних серверов Skype для бизнеса Server. Он предварительно обнародовывает входящие запросы и перенаправляет их в домашний пул или сервер пользователя. Эта предварительная предварительная запись позволяет отсылать неопознанный запрос учетной записи пользователя.
  
Почему это имеет значение? Важной функцией директора является защита серверов Standard Edition и серверов переднего сервера или пулов переднего сервера от вредоносного трафика, такого как атаки типа "отказ в обслуживании" (DoS). Если в вашей сети много недопустимого внешнего трафика, трафик останавливается на директоре.
  
### <a name="load-balancers"></a>Подсистемы балансировки нагрузки
<a name="LoadBalancers"> </a>

Масштабируемая консолидированная топология Skype для бизнеса Server оптимизирована для балансировки нагрузки на DNS для новых развертывания, и мы рекомендуем это сделать. Если вам требуется высокая доступность, мы рекомендуем использовать аппаратный балансировчик нагрузки для одной конкретной ситуации:
  
- Exchange UM для удаленных пользователей, использующих exchange UM _ *до** Exchange 2013.
    
> [!IMPORTANT]
> Важно отметить, что вы не можете смешивать подстановки нагрузки. В среде Skype для бизнеса Server все интерфейсы должны использовать DNS или HLB. 
  
> [!NOTE]
> Direct server return (DSR) NAT isn't supported for Skype for Business Server. 
  
#### <a name="hardware-load-balancer-requirements-for-edge-servers-edge-servers-running-the-av-edge-service"></a>требования к аппаратным средствам балансировки нагрузки для edge-серверов, на серверах с запущенной службой A/V Edge

Для любого сервера, на который запущена поверная служба A/V, необходимо:
  
- Отключите nagl-nagling TCP для внутреннего и внешнего портов 443 (nagling — это процесс объединения нескольких небольших пакетов в один пакет большего размера для более эффективной передачи).
    
- Отключите nagling TCP для диапазона внешних портов от 50000 до 59999.
    
- Не используйте NAT на внутренних или внешних брандмауэрах.
    
- Внутренний интерфейс edge должен быть в сети, отличаемой от внешнего интерфейса внешнего сервера, а маршруты между ними должны быть отключены.
    
- Внешний интерфейс любого сервера, на который запущена по edge-служба аудио- и аудиоконвейей, должен использовать общедоступные IP-адреса с маршрутизируемыми ip-адресами без трансляции NAT или перевода портов ни на один из внешних IP-адресов.
    
#### <a name="hlb-requirements"></a>Требования к HLB

В Skype для бизнеса Server не так много требований сходства на основе файлов cookie. Поэтому вам не нужно использовать сохраняемость на основе файлов cookie, если **(а** это Skype для бизнеса Server 2015) в среде Skype для бизнеса Server не будут иметься серверы переднего или переднего сервера Lync Server 2010. Им потребуется сходство на основе файлов cookie в методе конфигурации, рекомендованном для Lync Server 2010.
  
> [!NOTE]
> Если вы решите включить сходство на основе файлов cookie для вашей HLB, это не будет проблемой, даже если она не нужна вашей среде. 
  
Если вашей **среде не требуется** сходство на основе файлов cookie:
  
- В правиле публикации обратного прокси-сервера для порта 443 установите для загона сервера **forward** (Переад.  Это обеспечит переад вперед исходный URL-адрес.
    
Для развертыванием, **для которые требуются** сходства на основе файлов cookie, сделайте вот что:
  
- В правиле публикации обратного прокси-сервера для порта 443 установите для загона сервера **forward** (Переад.  Это обеспечит переад вперед исходный URL-адрес.
    
- Файл cookie аппаратного балансира нагрузки **не должен** помечаться как httpOnly.
    
- Файл cookie аппаратного балансировки **нагрузки не должен** иметь срок действия.
    
- Файл cookie аппаратного  балансира нагрузки должен иметь имя **MS-WSMAN** (это значение, которое ожидают веб-службы, и его нельзя изменить).
    
- Файл cookie аппаратного  балансира нагрузки должен быть установлен в каждом HTTP-ответе, для которого входящий HTTP-запрос не имел файла cookie, независимо от того, получил ли предыдущий HTTP-ответ на том же TCP-соединении файл cookie. Если аппаратный балансировчик нагрузки оптимизирует вставку файлов cookie только один раз для каждого подключения TCP, эту оптимизацию **использовать** не следует.
    
> [!NOTE]
> Конфигурации HLB обычно используют сходство источников и время жизни сеанса TCP 20 минут, что отлично для Skype для бизнеса Server и его клиентов, так как состояние сеанса поддерживается за счет использования клиента и/или взаимодействия с приложением. 
  
При развертывании мобильных устройств аппаратная балансовая нагрузка должна быть способна балансировать нагрузку отдельных запросов в рамках сеанса TCP (фактически необходимо иметь возможность балансиировать нагрузку отдельного запроса на основе целевого IP-адреса).
  
> [!IMPORTANT]
> F5 HLB имеют функцию OneConnect. Это гарантирует, что каждый запрос в соединении TCP имеет индивидуальную балансную нагрузку. Если вы развертываете мобильные устройства, убедитесь, что ваш поставщик аппаратной аппаратной поддержки поддерживает те же функции. Для последних мобильных приложений iOS требуется TLS версии 1.2. Если вам нужно узнать больше, F5 предоставляет специальные параметры для этого. 
  
Вот требования к HLB для (необязательно) директора и (обязательно) веб-служб пула переднего конца:
  
- Для внутренних VIP веб-служб установите Source_addr сохраняемость (внутренний порт 80, 443) на вашем HLB. Для Skype для бизнеса Server Source_addr сохраняемость означает, что несколько подключений с одного IP-адреса всегда отправляются на один сервер для поддержания состояния сеанса.
    
- Используйте время ожидания простоя TCP 1800 секунд.
    
- На брандмауэре между обратным прокси-сервером и HLB пула следующего прыжка создайте правило для https: трафик через порт 4443 от обратного прокси-сервера до HLB. Устройство HLB должно быть настроено для прослушивания портов 80, 443 и 4443.

#### <a name="port-monitoring-for-hlbs"></a>Мониторинг портов для HLB

Вы определяете мониторинг портов на аппаратных балансиаторах нагрузки, чтобы определить, когда конкретные службы больше недоступны из-за сбоя оборудования или связи. Например, если служба сервера переднего сервера (RTCSRV) останавливается из-за сбоя сервера переднего сервера или пула переднего сервера, мониторинг HLB также должен прекратить получение трафика веб-служб. Следует реализовать мониторинг портов на устройстве HLB, чтобы отслеживать следующие данные для внешнего интерфейса HLB:
  
|**Виртуальный IP-адрес/порт**|**Порт узла**|**Компьютер/монитор узла**|**Профиль сохраняемости**|**Примечания**|
|:-----|:-----|:-----|:-----|:-----|
|\<pool\>web_mco_443_vs  <br/> 443  <br/> |4443  <br/> |Сервер переднего плана  <br/> 5061  <br/> |Нет  <br/> |HTTPS  <br/> |
|\<pool\>web_mco_80_vs  <br/> 80  <br/> |8080  <br/> |Сервер переднего плана  <br/> 5061  <br/> |Нет  <br/> |HTTP  <br/> |
