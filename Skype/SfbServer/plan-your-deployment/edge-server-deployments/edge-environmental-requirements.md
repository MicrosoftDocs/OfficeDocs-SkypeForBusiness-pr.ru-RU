## <a name="topology-planning"></a>Планирование топологии
<a name="TopoPlan"> </a>

Skype для бизнеса Server Edge Server topologies are able to use:
  
- Общедоступные IP-адреса с маршрутией.
    
- Частные IP-адреса, которые не  могут быть маршрутизируемыми, если используется симметричный перевод сетевых адресов (NAT).
    
> [!TIP]
> Ваш сервер можно настроить на использование одного IP-адреса с отдельными портами для каждой службы или использовать отдельные IP-адреса для каждой службы, но использовать один и тот же порт по умолчанию (который по умолчанию будет TCP 443). Дополнительные сведения см. в разделе "Требования к IP-адресам" ниже. 
  
При выборе частных IP-адресов, которые не могут быть маршрутизируемыми, с NAT, помните об этих моментах:
  
- Во всех трех внешних интерфейсах  необходимо использовать частные IP-адреса с возможностью маршрутной маршрутки.
    
- Вам необходимо настроить **симметричное** nat для входящих и исходящая трафика. Симметричный NAT — единственный поддерживаемый NAT, который можно использовать с skype для бизнеса Server Edge Server.
    
- Настройте NAT так, чтобы он не менял входящие исходные адреса. Чтобы найти оптимальный путь к мультимедиа, постоящая служба A/V Edge должна иметь возможность получать входящий исходный адрес.
    
- Ваши серверы должны иметь возможность взаимодействовать друг с другом с общедоступных IP-адресов A/V Edge. Брандмауэр должен разрешить этот трафик.
    
- NAT можно **использовать только** для масштабированных консолидированных серверов, если используется балансировка нагрузки на DNS. При использовании аппаратной балансировки нагрузки (HLB) необходимо использовать общедоступные IP-адреса с маршрутизативом **без** NAT.
    
У вас не будет проблем с использованием интерфейсов access, веб-аудио- и видеоконференций за маршрутизатором или брандмауэром, которые выполняют симметричную трансляцию NAT для одно- и масштабированных консолидированных серверных тоологий (если аппаратная балансировка нагрузки не используется).

### <a name="ip-address-requirements"></a>Требования к IP-адресам

На фундаментальном уровне ТРИ службы нуждаются в IP-адресах; Побереговая служба доступа, поехавщик веб-служб и постояла служба видео- и видеоконференций. Вы можете использовать либо три IP-адреса, по одному для каждой службы, либо использовать один и выбрать для каждой [](edge-environmental-requirements.md#PortFirewallPlan) службы отдельный порт (дополнительные сведения об этом см. в разделе планирования портов и брандмауэра). Для единой консолидированной среды edge это в значительной степени.
  
> [!NOTE]
> Как было отмечено выше, можно выбрать один IP-адрес для всех трех служб и запустить их на разных портах. Но это не рекомендуется. Если ваши клиенты не могут получить доступ к альтернативным портам, которые вы будете использовать в этом сценарии, они также не смогут получить доступ ко всем функциональным возможностям среды Edge. 
  
С масштабированными консолидированными топологиями это может быть немного сложнее, поэтому рассмотрим некоторые таблицы с требованиями к IP-адресам, уделив внимание, что основными точками принятия решений при выборе топологии являются высокая доступность и балансировка нагрузки. Высокие потребности в доступности могут повлиять на выбор балансировки нагрузки (мы подробнее поговорим об этом после таблиц).

Рассмотрим некоторые дополнительные моменты, которые необходимо думать при планировании.
  
- **Высокая доступность:** если в развертывании требуется высокая доступность, следует развернуть по крайней мере два сервера в пуле. Стоит отметить, что один пул будет поддерживать до 12 edge-серверов (хотя построитель топологий позволит добавить до 20, это не тестироваться и не поддерживаться, поэтому мы рекомендуем не делать этого). Если вам требуется более 12 серверов, необходимо создать дополнительные пулы для них.
    
- **Аппаратная балансировка нагрузки:** для большинства сценариев рекомендуется балансировка нагрузки на DNS. Аппаратная балансировка нагрузки, конечно, также поддерживается, но в частности, она необходима для одного сценария по балансировку нагрузки на DNS:
    
  - Внешний доступ к единой системы обмена сообщениями Exchange 2007 или Exchange 2010 (без sp).
    
- **Балансировка нагрузки на DNS:** для um, Exchange 2010 SP1 и более новых могут поддерживаться балансировка нагрузки на DNS. Обратите внимание, что если требуется балансировка нагрузки на DNS для более ранней версии Exchange, он будет работать, но весь трафик для этого будет перенаправляться на первый сервер в пуле, и если он недо доступен, этот трафик в дальнейшем не будет работать.
    
    Балансировка нагрузки на DNS также рекомендуется при федерации с компаниями, использующими:
- Skype для бизнеса Server 2015:
    - Lync Server 2010
    - Lync Server 2013
    - Microsoft 365 или Office 365
- Skype для бизнеса Server 2019:
    - Lync Server 2013
    - Skype для бизнеса Server 2015
    - Microsoft 365 или Office 365

## <a name="certificate-planning"></a>Планирование сертификатов
<a name="CertPlan"> </a>

Skype для бизнеса Server использует сертификаты для безопасной зашифрованной связи между серверами и между серверами и клиентом. Как ожидается, в сертификатах должны быть записи DNS для серверов, совпадают с любым именем субъекта (SN) и альтернативным именем субъекта (SAN) в сертификатах. Это займет много времени на этапе планирования, чтобы убедиться, что у вас есть правильные FQDN, зарегистрированные в DNS для записей SN и SAN для сертификатов.
  
Мы обсудим требования к внешним и внутренним сертификатам отдельно, а затем посмотрим на таблицу с требованиями для обоих этих сертификатов.
  
### <a name="external-certificates"></a>Внешние сертификаты

Как минимум, сертификат, присвоенный интерфейсам внешнего сервера, должен быть предоставлен общедоступным ЦС. Мы не можем рекомендовать вам конкретный ЦС, но у нас есть список ЦС, партнеров по сертификатам [unified Communications,](/SkypeForBusiness/certification/services-ssl) на которые можно посмотреть, чтобы узнать, указан ли ваш предпочтительный ЦС.
  
Когда вам потребуется отправить запрос на этот общедоступный сертификат в ЦС и как это сделать? Это можно сделать несколькими способами:
  
- Можно перейти к установке Skype для бизнеса Server, а затем к развертыванию edge Server. Мастер развертывания Skype для бизнеса Server должен создать запрос на сертификат, который затем можно отправить выбранному ЦС.
    
- Вы также можете использовать Windows PowerShell для создания этого запроса, если это более влияло на бизнес-потребности или стратегию развертывания.
    
- Наконец, в вашем ЦС может быть собственный процесс отправки, который также может включать Windows PowerShell или другой метод. В этом случае вам потребуется полагаться на их документацию в дополнение к сведениям, предоставленным здесь для справки.
    
После того как вы получите сертификат, вам потребуется перейти к работе и назначить его этим службам в Skype для бизнеса Server:
  
- Интерфейс службы по краям доступа
    
- Интерфейс службы edge веб-аудиоконференций
    
- Служба проверки подлинности аудио- и видео (не путайте это со службой аудио- и видеоконференций, так как она не использует сертификат для шифрования аудио- и видеопотоков)
    
> [!IMPORTANT]
> Все серверы (если они относятся к одному пулу серверов) должны иметь один и тот же сертификат с одним и тем же закрытым ключом для службы проверки подлинности при ретрансляции мультимедиа. 
  
### <a name="internal-certificates"></a>Внутренние сертификаты

Для внутреннего интерфейса сервера можно использовать общедоступный сертификат из общего ЦС или сертификат, выданный внутренним ЦС организации. Что нужно помнить о внутреннем сертификате, так это то, что он использует запись SN, а записи SAN не используются, поэтому вам не нужно беспокоиться о SAN во внутреннем сертификате.
