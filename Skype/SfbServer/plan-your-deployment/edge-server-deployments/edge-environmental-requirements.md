## <a name="topology-planning"></a>Планирование топологии
<a name="TopoPlan"> </a>

Skype для бизнеса Server Edge Server могут использовать:
  
- Общедоступные IP-адреса routable.
    
- Если используется симметричный перевод сетевых адресов (NAT).
    
> [!TIP]
> Edge Server можно настроить для использования одного IP-адреса с отдельными портами для каждой службы или использовать отдельные IP-адреса для каждой службы, но использовать один и тот же порт по умолчанию (который по умолчанию будет TCP 443). Дополнительные сведения см. в разделе Требования к IP-адресу. 
  
Если вы выбираете не маршрутируемые частные IP-адреса с помощью NAT, запомните эти моменты:
  
- Во всех трех внешних интерфейсах необходимо использовать маршрутируемые  частные IP-адреса.
    
- Необходимо настроить симметричный NAT **для** входящих и исходяющих трафика. Симметричный NAT — это единственный поддерживаемый NAT, который можно использовать с Skype для бизнеса Server Edge Server.
    
- Настройте nat, чтобы не изменять входящие исходные адреса. Чтобы найти оптимальный способ мультимедиа, служба A/V Edge должна иметь возможность получать входящий исходный адрес.
    
- Ваши edge Servers должны иметь возможность общаться друг с другом с общедоступных IP-адресов A/V Edge. Брандмауэр должен разрешить этот трафик.
    
- Nat можно **использовать только** для масштабированных консолидированных edge Servers, если используется балансировка нагрузки DNS. При использовании балансировки аппаратной нагрузки (HLB) необходимо использовать общедоступные маршрутные IP-адреса **без** NAT.
    
У вас не будет проблем с интерфейсами Access, Web conferencing и A/V Edge за маршрутизатором или брандмауэром, которые выполняют симметричный NAT для одно- и масштабированной консолидированной топологий Edge Server (пока не используется балансировка нагрузки оборудования).

### <a name="ip-address-requirements"></a>Требования к IP-адресу

На фундаментальном уровне три службы нуждаются в IP-адресах; Служба Access Edge, служба edge веб-конференций и служба A/V Edge. У вас есть возможность использовать три IP-адреса, по одному для каждой из служб, или вы можете использовать один и выбрать, чтобы поместить каждую службу в другой порт (вы можете проверить раздел планирования порта и брандмауэра, чтобы получить дополнительные сведения о некоторых из них).[](edge-environmental-requirements.md#PortFirewallPlan) Для единой консолидированной среды Edge это в значительной степени.
  
> [!NOTE]
> Как уже отмечалось выше, вы можете выбрать один IP-адрес для всех трех служб и запустить их в разных портах. Но чтобы быть понятным, мы не рекомендуем это. Если клиенты не могут получить доступ к альтернативным портам, которые будут использовать в этом сценарии, они также не смогут получить доступ к полному функционалу среды Edge. 
  
Это может быть немного сложнее с масштабированными консолидированными топологиями, поэтому давайте рассмотрим некоторые таблицы, в которые выложены требования к IP-адресу, с тем чтобы иметь в виду, что основными точками принятия решений для выбора топологии являются высокая доступность и балансировка нагрузки. Высокие потребности в доступности могут повлиять на выбор балансировки нагрузки (об этом мы поговорим больше после таблиц).

Давайте рассмотрим некоторые дополнительные моменты, которые необходимо обдумать при планировании.
  
- **Высокая доступность**. Если требуется высокая доступность в развертывании, необходимо развернуть в пуле не менее двух серверов edge. Стоит отметить, что один пул Edge будет поддерживать до 12 edge Servers (хотя Topology Builder позволит добавить до 20, это не проверено и не поддерживается, поэтому мы рекомендуем не делать этого). Если требуется более 12 edge Servers, необходимо создать дополнительные пулы Edge для них.
    
- **Балансировка нагрузки оборудования**. Для большинства сценариев рекомендуется балансировка нагрузки DNS. Конечно, поддерживается также балансировка нагрузки оборудования, но, в частности, она необходима для одного сценария балансировки нагрузки DNS:
    
  - Внешний доступ к Exchange 2007 или Exchange 2010 (без sp) Единой системы обмена сообщениями (ЕДИНОЙ Системы обмена сообщениями).
    
- **Балансировка** нагрузки DNS. Для Exchange 2010 sp1 и более новые могут поддерживаться балансировка нагрузки DNS. Обратите внимание, что если для более ранней версии Exchange требуется балансировка нагрузки DNS, он будет работать, но весь трафик для этого будет переходить на первый сервер в пуле, и если он недодоступн, то трафик будет сбой.
    
    Балансировка нагрузки DNS также рекомендуется, если вы федерансируете с компаниями, использующими:
- Skype для бизнеса Server 2015 г.:
    - Lync Server 2010
    - Lync Server 2013
    - Microsoft 365 или Office 365
- Skype для бизнеса Server 2019 г.:
    - Lync Server 2013
    - Skype для бизнеса Server 2015
    - Microsoft 365 или Office 365

## <a name="certificate-planning"></a>Планирование сертификатов
<a name="CertPlan"> </a>

Skype для бизнеса Server использует сертификаты для безопасных зашифрованных сообщений как между серверами, так и с сервера на клиент. Как и ожидается, для сертификатов необходимо, чтобы записи DNS для серверов совпадали с любым именем субъекта (SN) и альтернативным именем субъекта (SAN) в сертификатах. На этапе планирования это займет много времени, чтобы убедиться, что вы имеете право на регистрацию FQDN в DNS для записей SN и SAN для сертификатов.
  
Отдельно мы обсудим потребности внешнего и внутреннего сертификата, а затем посмотрим на таблицу, в результате чего будут предоставляться требования к обоим сертификатам.
  
### <a name="external-certificates"></a>Внешние сертификаты

Как минимум сертификат, присвоенный внешним интерфейсам Edge Server, должен быть предоставлен государственным органом сертификации (CA). Мы не можем рекомендовать вам определенный ЦС, но у нас есть список партнеров сертификата единой [связи, которые](../../../SfbPartnerCertification/certification/services-ssl.md) можно посмотреть, чтобы узнать, указан ли ваш предпочтительный ЦС.
  
Когда вам потребуется отправить запрос в ЦС для этого общего сертификата, и как это сделать? Существует несколько способов добиться этого:
  
- Вы можете пройти через установку Skype для бизнеса Server, а затем развертывание Edge Server. Мастер Skype для бизнеса Server развертывания будет иметь шаг для создания запроса сертификата, который вы можете отправить в выбранный вами ЦС.
    
- Вы также можете использовать Windows PowerShell для создания этого запроса, если это более вовсю со своими бизнес-потребностями или стратегией развертывания.
    
- Наконец, ваш ЦС может иметь свой собственный процесс отправки, который также может включать Windows PowerShell или другой метод. В этом случае вам потребуется опираться на документацию, а также сведения, предоставленные здесь для вашей справки.
    
После того как сертификат будет выдан, необходимо назначить его этим службам в Skype для бизнеса Server:
  
- Интерфейс службы Access Edge
    
- Интерфейс службы Web Conferencing Edge
    
- Служба проверки подлинности аудио- и видео (не путайте это со службой A/V Edge, так как для шифрования потоков аудио и видео не используется сертификат)
    
> [!IMPORTANT]
> Все edge Servers (если они принадлежат к одному пулу edge Servers) должны иметь точно такой же сертификат с одним и тем же закрытым ключом для службы проверки подлинности в ретрансляторе мультимедиа. 
  
### <a name="internal-certificates"></a>Внутренние сертификаты

Для внутреннего интерфейса Edge Server можно использовать общедоступный сертификат из общедоступных ЦС или сертификат, выданный из внутреннего ЦС организации. Что нужно помнить о внутреннем сертификате, так это то, что он использует запись SN, а записей SAN нет, поэтому вам не нужно беспокоиться о SAN на внутренней сертификации вообще.
