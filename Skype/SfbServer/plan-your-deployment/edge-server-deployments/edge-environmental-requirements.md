## <a name="topology-planning"></a>Планирование топологии
<a name="TopoPlan"> </a>

Топологии пограничного сервера Skype для бизнеса Server могут использовать следующие компоненты:
  
- Общедоступные IP-адреса с маршрутизацией.
    
- Частные IP-адреса без поддержки маршрутизации, если используется **симметричное** преобразование сетевых адресов (NAT).
    
> [!TIP]
> Пограничный сервер можно настроить на использование одного IP-адреса с отдельными портами для каждой службы или для каждой службы может использовать разные IP-адреса, но используйте один и тот же порт по умолчанию (по умолчанию он равен TCP 443). В разделе Требования к IP-адресу представлены дополнительные сведения. 
  
Если вы выбрали частные IP-адреса без поддержки маршрутизации с помощью NAT, помните следующие моменты:
  
- Необходимо использовать частные IP-адреса с маршрутизацией на **всех трех** внешних интерфейсах.
    
- Для входящего и исходящего трафика необходимо настроить **симметричное** преобразование сетевых адресов (NAT). Симметричное преобразование сетевых адресов — это единственная поддерживаемая поддержка NAT, которую можно использовать с пограничным сервером Skype для бизнеса Server.
    
- Настройка NAT для изменения входящих исходных адресов. Пограничная служба аудио-и видеоданных должна иметь возможность получить входящий исходный адрес, чтобы найти оптимальный путь к носителю.
    
- Пограничные серверы должны иметь возможность общаться друг с другом от общедоступных IP-адресов пограничного сервера аудио-и видеоданных. Брандмауэр должен разрешить этот трафик.
    
- Преобразование сетевых адресов (NAT) можно использовать **только** для масштабируемых консолидированных пограничных серверов при использовании балансировки нагрузки на DNS. При использовании аппаратной балансировки нагрузки (HLB) необходимо использовать общедоступные IP-адреса **без** NAT.
    
У вас нет проблем со доступом, веб-конференциями и пограничными интерфейсами аудио-и видеоконференций на маршрутизаторе или брандмауэре, выполняющем симметричное преобразование сетевых адресов для одной и масштабируемой топологии пограничных серверов (при условии, что не используется аппаратная балансировка нагрузки).

### <a name="ip-address-requirements"></a>Требования к IP-адресам

На базовом уровне трем службам требуются IP-адреса; Пограничная служба доступа, пограничная служба веб-конференций и пограничная служба аудио-и видеоконференций. Вы можете использовать три IP-адреса, по одному для каждой службы, или использовать одну из них, чтобы разместить каждую службу на другом порте (вы можете извлечь раздел [планирование портов и брандмауэров](edge-environmental-requirements.md#PortFirewallPlan) , чтобы получить дополнительные сведения об этих случаях). Для одной консолидированной пограничной среды это практически все.
  
> [!NOTE]
> Как отмечалось выше, можно выбрать один IP-адрес для всех трех служб и запускать их на разных портах. Но это не рекомендуется делать. Если клиентам не удается получить доступ к альтернативным портам, которые вы будете использовать в этом сценарии, они не смогут получить доступ ко всем функциональным возможностям пограничной среды. 
  
Это может быть немного сложнее с помощью масштабируемых консолидированных топологий, поэтому рассмотрим некоторые таблицы, в которых размещаются требования к IP-адресам, учитывая, что основные точки принятия решений по выбору топологии обеспечивают высокую доступность и балансировку нагрузки. Требования к высокой доступности могут повлиять на выбор балансировки нагрузки (это будет обсуждаться позже таблиц).

Давайте взглянем на некоторые дополнительные моменты, которые необходимо учесть при планировании.
  
- **Высокая доступность**: Если в развертывании требуется высокая доступность, необходимо развернуть по крайней мере два пограничных сервера в пуле. Следует отметить, что один пограничный пул будет поддерживать до 12 пограничных серверов (хотя построитель топологий позволит вам добавить до 20, который не тестировался или не поддерживается, поэтому мы рекомендуем не делать этого). Если требуется более 12 пограничных серверов, необходимо создать для них дополнительные пулы пограничных серверов.
    
- **Аппаратная балансировка нагрузки**: Мы рекомендуем использовать балансировку нагрузки DNS для большинства сценариев. Аппаратная балансировка нагрузки также поддерживается, но это необходимо для одного сценария по сравнению с балансировкой нагрузки DNS.
    
  - Внешний доступ к Exchange 2007 или Exchange 2010 (без SP) единой системы обмена сообщениями (UM).
    
- **Балансировка нагрузки на DNS**: для единой системы обмена сообщениями Служба балансировки нагрузки на DNS может поддерживать Exchange 2010 с пакетом обновления 1 (SP1) и новее. Обратите внимание, что если вам нужно перейти с балансировкой нагрузки на DNS для более ранней версии Exchange, она будет работать, но весь трафик для этого будет передаваться на первый сервер в пуле, и если он недоступен, то этот трафик по-то пойдет в дальнейшем.
    
    Балансировка нагрузки на DNS также рекомендуется, если вы добавляете Федерацию с компаниями, используя:
- Skype для бизнеса Server 2015:
    - Lync Server 2010
    - Lync Server 2013
    - Microsoft 365 или Office 365
- Skype для бизнеса Server 2019:
    - Lync Server 2013
    - Skype для бизнеса Server 2015
    - Microsoft 365 или Office 365

## <a name="certificate-planning"></a>Планирование сертификатов
<a name="CertPlan"> </a>

Skype для бизнеса Server использует сертификаты для безопасной и зашифрованной связи между серверами и от сервера к клиенту. Как вы ожидаете, ваши сертификаты должны иметь DNS-записи для ваших серверов, которые будут сопоставляться с любым именем субъекта (SN) и альтернативным именем субъекта (SAN) для сертификатов. После этого на этапе планирования необходимо убедиться, что в DNS зарегистрированы полные доменные имена, зарегистрированные в DNS для записей SN и SAN для сертификатов.
  
Внешние и внутренние сертификаты будут обсуждаться по отдельности, а затем взгляните на таблицу, предоставляющую требования для обеих задач.
  
### <a name="external-certificates"></a>Внешние сертификаты

Как минимум сертификат, назначенный внешним интерфейсам пограничного сервера, должен быть предоставлен общедоступным центром сертификации (CA). Мы не можем рекомендовать конкретному центру сертификации, но у нас есть список центров сертификации, [партнеров по сертификатам Объединенных коммуникаций](/SkypeForBusiness/certification/services-ssl) , которые можно просмотреть, чтобы узнать, есть ли в списке предпочитаемый центр сертификации.
  
Когда потребуется отправить запрос в ЦС для этого общедоступного сертификата и как это сделать? Это можно сделать несколькими способами:
  
- Вы можете перейти к установке Skype для бизнеса Server, а затем к развертыванию пограничного сервера. Мастер развертывания Skype для бизнеса Server будет выполнять действия по созданию запроса на сертификат, который затем можно отправить в выбранный центр сертификации.
    
- Вы также можете использовать команды Windows PowerShell для создания этого запроса, если это не так, как в случае с бизнес-потребностями или стратегией развертывания.
    
- Наконец, у вашего центра сертификации может быть собственный процесс отправки, который также может включать Windows PowerShell или другой метод. В этом случае вам потребуется полагаться на документацию, а также информацию, приведенную здесь, для справки.
    
После получения сертификата необходимо присвоить его следующим службам в Skype для бизнеса Server:
  
- Интерфейс пограничной службы доступа
    
- Интерфейс пограничной службы веб-конференций
    
- Служба проверки подлинности аудио-и видеоданных (это не следует путать со службой аудио-и видеоданных, так как она не использует сертификат для шифрования аудио-и видеопотоков)
    
> [!IMPORTANT]
> Все пограничные серверы (если они принадлежат к одному и тому же пулу пограничных серверов) должны иметь один и тот же сертификат с одним и тем же закрытым ключом для службы проверки подлинности ретрансляции мультимедиа. 
  
### <a name="internal-certificates"></a>Внутренние сертификаты

Для внутреннего интерфейса пограничного сервера можно использовать общедоступный сертификат из общедоступного центра сертификации или сертификат, выданный внутренним ЦС организации. Помните о том, что внутренний сертификат использует запись SN, а не записи сети хранения данных, поэтому вам не придется беспокоиться о сети SAN на внутреннем сертификате.
