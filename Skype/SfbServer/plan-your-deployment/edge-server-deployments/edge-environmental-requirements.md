Множество задач планирования и подготовки должны выполняться за пределами среды пограничного сервера в Skype для бизнеса Server. In this article, we'll review what preparations need to be made in the organizational environment, as per our list below:

## <a name="topology-planning"></a>Планирование топологии
<a name="TopoPlan"> </a>

Топологии серверного пограничного сервера в Skype для бизнеса Server можно использовать:
  
- общедоступные IP-адреса с поддержкой маршрутизации;
    
- частные IP-адреса, не поддерживающий маршрутизацию, если используется **симметричное** преобразование сетевых адресов (NAT).
    
> [!TIP]
> На пограничном сервере можно настроить использование одного IP-адреса с разными портами для каждой службы или использовать отдельные IP-адреса для каждой службы, но использовать один и тот же порт по умолчанию (который по умолчанию будет TCP 443). We have more information in IP Address requirements section, below. 
  
Если вы решите использовать частные IP-адреса без поддержки маршрутизации с NAT, обратите внимание на следующие требования.
  
- Необходимо использовать частные IP-адреса с поддержкой маршрутизации на **всех трех** внешних интерфейсах.
    
- You need to configure **symmetric** NAT for incoming and outgoing traffic. Симметричная NAT – это единственная поддерживаемая NAT, которую можно использовать в Microsoft Business Server с сервером Edge.
    
- Configure your NAT to not change incoming source addresses. Чтобы найти оптимальный путь к носителю, служба Edge/V должна иметь возможность получить входящий адрес источника.
    
- Пограничные серверы должны иметь возможность общаться друг с другом через общедоступные IP-адреса Edge. Your firewall needs to allow this traffic.
    
- Преобразование сетевых адресов (NAT) можно использовать **только** для масштабируемых консолидированных пограничных серверов при использовании балансировки нагрузки DNS. If you use hardware load balancing (HLB), you need to use publicly routable IP addresses **without** NAT.
    
У вас нет проблем, связанных с доступом к веб-конференциям, Интернету и интерфейсом Edge за маршрутизатором или межсетевым экраном, выполняющим симметричное преобразование сетевых адресов (до тех пор, пока не используется аппаратная балансировка нагрузки).

### <a name="ip-address-requirements"></a>Требования к IP-адресам

На базовом уровне трем службам требуются IP-адреса; Служба пограничного доступа, служба Edge для веб-конференций и служба Edge/V. Можно использовать либо три IP-адреса (по одному для каждой политики), либо использовать один IP-адрес с возможностью размещения всех служб в разных портах (подробнее см. раздел [Планирование порта и брандмауэра](edge-environmental-requirements.md#PortFirewallPlan)). Вот и все, что нужно учесть при использовании единой консолидированной среды пограничного сервера.
  
> [!NOTE]
> Как указано выше, можно выбрать один IP-адрес для всех трех служб и запустить их на разных портах. Однако следует отметить, что такой подход не рекомендуется. Если ваши клиенты не смогут получить доступ к альтернативным портам, используемым в этом сценарии, они не смогут получить доступ ко всем возможностям среды пограничного сервера. 
  
Использование масштабируемых консолидированных топологий представляет собой более сложный процесс. Требования к высокой доступности могут влиять на решение в отношении балансировки нагрузки (к этому вопросу мы вернемся после рассмотрения таблиц).

Рассмотрим несколько вопросов, которые необходимо учесть при планировании.
  
- **Высокая доступность**: Если в развертывании требуется высокая доступность, вы должны развернуть по крайней мере два пограничных сервера в пуле. Стоит отметить, что один пул пограничного сервера поддерживает до 12 пограничных серверов (хотя в построителе топологии вы можете добавить до 20, что не тестировалась или не поддерживается, поэтому мы рекомендуем вам не делать этого). Если вам требуется более 12 пограничных серверов, вы должны создать для них дополнительные пулы пограничных кромок.
    
- **Балансировка нагрузки оборудования**: для большинства сценариев рекомендуется использовать балансировку нагрузки DNS. Балансировка нагрузки аппаратных средств также поддерживается, но, несомненно, она необходима для одного сценария в рамках балансировки нагрузки DNS.
    
  - Внешний доступ к Exchange 2007 или Exchange 2010 (без пакета обновления) UM.
    
- **Балансировка нагрузки DNS**: в среде UM Exchange 2010 SP1 и более поздние версии могут поддерживаться службой балансировки нагрузки DNS. Обратите внимание, что если вам нужно обратиться в службу балансировки нагрузки DNS для более ранней версии Exchange, она будет работать, но весь трафик для этого сервера пойдет на первый сервер в пуле, и, если он недоступен, трафик в дальнейшем будет невозможен.
    
    Балансировка нагрузки DNS также рекомендуется, если вы собираетесь использовать следующие компании:
- Skype для бизнеса Server 2015:
    - Lync Server 2010
    - Lync Server 2013
    - Microsoft Office 365
- Skype для бизнеса Server 2019:
    - Lync Server 2013
    - Skype для бизнеса Server 2015
    - Microsoft Office 365.

## <a name="certificate-planning"></a>Планирование сертификации
<a name="CertPlan"> </a>

Skype для бизнеса Server использует сертификаты для безопасной и зашифрованной связи между сервером и клиентом. Как и можно было ожидать, для сертификатов понадобятся DNS-записи, чтобы серверы подходили любому имени субъекта (SN) и альтернативному имени субъекта (SAN) в сертификатах. Это требует принятия определенных мер на стадии планирования, для регистрации подходящих полных доменных имен в DNS для записей SN и SAN ваших сертификатов.
  
Мы рассмотрим требования к внешним и внутренним сертификатам по отдельности, а затем ознакомимся с таблицей, в которой объединены и те, и другие требования.
  
### <a name="external-certificates"></a>Внешние сертификаты

Как минимум сертификат, назначенный внешним интерфейсам пограничного сервера, должен быть предоставлен общедоступным центром сертификации (ЦС). Мы не можем рекомендовать вам определенный центр сертификации, но у нас есть список центров сертификации, [партнеров по сертификатам единой системы обмена сообщениями](/SkypeForBusiness/certification/services-ssl) , на котором вы можете узнать, есть ли в списке предпочтительный центр сертификации.
  
Когда и как отравлять запрос на получение общедоступного сертификата в центр сертификации? Есть несколько способов.
  
- Вы можете пройти установку Skype для бизнеса Server и развертывание пограничного сервера. Мастер развертывания Skype для бизнеса Server поможет создать запрос на сертификат, который затем можно отправить на выбранный центр сертификации.
    
- Вы также можете использовать команды Windows PowerShell для создания этого запроса, если это более подробно с учетом потребностей бизнеса или стратегии развертывания.
    
- Наконец, ваш центр сертификации может иметь собственный процесс отправки, который также может включать в себя Windows PowerShell или другой метод. В этом случае вы должны будете обратиться к документации центра сертификации в дополнение к представленной здесь информации.
    
После того как вы получите сертификат, вам нужно будет присвоить его этим службам в Skype для бизнеса Server.
  
- Интерфейс службы Edge Access
    
- Интерфейс службы Edge для веб-конференций
    
- Служба проверки подлинности аудио-и видеосвязи (не путайте со службой Edge/V, так как она не использует сертификат для шифрования звуковых и видеопотоков)
    
> [!IMPORTANT]
> Все пограничные серверы (если они принадлежат одному и тому же пулу пограничных серверов) должны иметь одинаковый сертификат с одним и тем же закрытым ключом для службы проверки подлинности Media Relay. 
  
### <a name="internal-certificates"></a>Внутренние сертификаты

Для интерфейса внутреннего сервера Edge вы можете использовать общедоступный сертификат из общедоступного центра сертификации или сертификат, выданный внутренним центром сертификации организации. The thing to remember about the internal certificate is that it uses an SN entry, and no SAN entries, so you don't have to worry about SAN on the internal cert at all.
