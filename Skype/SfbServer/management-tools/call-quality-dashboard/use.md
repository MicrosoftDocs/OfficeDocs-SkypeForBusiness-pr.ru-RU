# <a name="use-call-quality-dashboard-for-skype-for-business-server"></a>Использование панели мониторинга качества звонков для Skype для бизнеса Server

**Сводка:** Узнайте, как использовать панель мониторинга качества звонков. Панель мониторинга качества звонков — это средство для Skype для бизнеса Server.

Панель мониторинга качества звонков (CQD) позволяет ИТ-специалистам использовать объединенные данные для выявления проблем, возникающих при создании проблем качества мультимедиа, сравнивая статистические показатели для групп пользователей для выявления тенденций и закономерностей. CQD не фокусируется на решении проблем, связанных с вызовами, но при выявлении проблем и решений, которые применяются к нескольким пользователям.

## <a name="call-quality-dashboard-user-guide"></a>Руководство пользователя по панели мониторинга качества звонков

CQD — это веб-портал для быстрого создания и систематизации отчетов на основе данных качества взаимодействия (QoE). CQD развертывает куб SSAS для объединения данных в базе данных метрик QoE и позволяет администраторам создавать и изменять отчеты и проводить расследования в режиме реального времени. Предусмотрена возможность прямого подключения Excel к кубу; при этом портал оптимизирован под различные рабочие процессы, предусматривающие обработку данных о качестве взаимодействия. Данные включают в себя следующее:

- Кэшированные данные отчета для быстрого доступа
- Глубокие ссылки на страницы отчета для обмена информацией и публикации
- Упрощенное редактирование и создание отчетов, а так же редактируемые метаданные для описаний отчетов.

Кроме того, CQD предоставляет веб-интерфейсы API, которые предоставляют пользователям программный доступ к данным куба для использования на настраиваемых панелях мониторинга.

### <a name="feature-overview"></a>Общие сведения о компоненте

Когда вы посещаете панель мониторинга качества звонков, вы увидите следующий экран:

![Использование CQD](../../media/1e061858-db6f-452b-9ae4-eab507220371.png)

1. В области "Сводка" можно найти контекст для "наборы отчетов" (справа).
2. Нажмите кнопку "Изменить" в сводке Панерепорт, чтобы настроить свойства уровня (включая высоту оси Y).
3. Навигатор помогает определить текущее расположение в иерархии наборов отчетов.
4. Отчеты с подчиненными отчетами отображаются с помощью синей ссылки. Щелкните ссылку, чтобы перейти к подчиненным отчетам.

Для отображения подробных значений наведите указатель мыши на линейчатые диаграммы и линии тренда. В отчете, для которого установлен фокус, отображается меню "действия": "Изменить", "Копировать", "Удалить" и "Загрузить".

### <a name="default-reports"></a>Стандартные отчеты

При первом доступе к порталу панели мониторинга качества звонков автоматически создается набор отчетов по умолчанию. Эти отчеты также называют системными. Вы можете свободно изменять и удалять эти отчеты, а также расширять их, создавая новые отчеты на основе одного и того же элемента.

На верхнем уровне в отчете "месячный тренд для звуковых потоков" показана месячная тенденция для всех звуковых потоков. Наведите указатель мыши на отрезки линейчатой диаграммы, чтобы отобразить более подробное представление данных, представленных линейчатой диаграммой. Щелкните заголовок отчета о месячной тенденции, чтобы перейти к отчету "управляемые и неуправляемые звуковые потоки", в котором отчеты распределены между управляемыми и неуправляемыми вызовами. Управляемые звонки посылаются по проводным соединениям из зоны, защищенной корпоративным брандмауэром. Неуправляемые звонки включают звонки, совершенные за пределами корпоративного брандмауэра, и все звонки, совершенные по Wi-Fi.

Другой отчет верхнего уровня называется "Гистограмма" Оценка качества связи, отправленного пользователем ". Рейтинг качества звонка — это количество номеров, предоставленных пользователями Skype для бизнеса в конце звонка для обозначения качества звонка. Номера рейтинга в диапазоне от 1 до 5, 1 — наихудшее, а 5 — лучший. На гистограмме отображается количество звонков голосовой связи, получивших данную оценку в течение одного месяца.

Щелкните заголовок любого из отчетов, чтобы перейти к отчетам с дополнительными фильтрами для данных. В системных отчетах каждый дочерний отчет содержит подмножество данных, представленных в родительском отчете. Модель решения проблем проста: исследование того, какой подчиненный отчет предложит данные или тенденции, и постепенно ограничивает пространство, в котором она связана. Возможность создания вложенных отчетов позволяет исследовать свое предположение относительно причины определенных тенденций данных.

## <a name="cqd-streams"></a>Потоки панели мониторинга качества звонков (CQD)

Поток CQD считается хорошим, низким или неклассифицированным. В настоящее время CQM 1.5 использует следующее определение CQD:

- Слабый поток — это любое сочетание слабых показателей звонка, находящихся за пределами порогового значения.
- Если один поток в вызове слабый, оба потока вызова помечаются как неудовлетворительные. В конференциях каждый участник подсчитывается как уникальный звонок и предоставляется отдельно от других.
- Неклассифицированные потоки — это потоки без метрик качества (то есть искусственные транзакции или короткие звонки).
- Действенные потоки = немобильные клиенты
- Классификатор не может быть изменен

**Определение или классификатор неудовлетворительного звонка**

|Показатель|Пороговое значение|
|:-----|:-----|
|деградатионавг |Больше 1 (–1 оценка MOS сети) |
|RoundTrip |Больше 500  |
|PacketLossRate |Больше 0,1 (10%) |
|JitterInterArrival |Больше 30  |
|ратиоконцеаледсамплесавг |Больше 0,07 |
|||

Определение JPDR = определение неудовлетворительного звонка минус RatioConcealedSamplesAvg 

## <a name="where-is-callercallee"></a>Где поля "Вызывающий абонент" и "Вызываемый абонент"?

CQD не использует поля вызывающего/вызываемого абонента, вместо этого использует "первый" и "второй", так как между вызывающим и вызываемым объектами есть промежуточные этапы.

 **First (первый** ) Всегда будет конечной точкой сервера (например, AV MCU или сервер-посредник), если сервер участвует в потоке.

 **Второй** всегда будет конечной точкой клиента, если только не используется поток типа "сервер — сервер". 

**Пример классификации "Первый" и "Второй"**

|Конечная точка 1 UAType |Конечная точка 2 UUAType |Первый|Второй|
|:-----|:-----|:-----|:-----|
|2 (AVMCU)  |4 (Skype для бизнеса)  |Конечная точка 1 |Конечная точка 2 |
|2 (AVMCU)  |1 (mMediationServer)  |Конечная точка 2 |Конечная точка 1 |
|4 (Skype для бизнеса) |4 (Skype для бизнеса)  |Вызывающий абонент в MediaLine  |Вызываемый абонент в MMediaLine  |
|||||

Если обе конечные точки имеют один и тот же тип, CQD создает сначала запись вызывающего абонента, а также вызываемый объект Second. Дополнительные сведения об именах конечных точек можно найти в [блоге](https://blogs.technet.com/b/jenstr/archive/2015/05/22/call-quality-dashboard-tips-and-tricks.aspx).

## <a name="accounting-for-vpn"></a>Учет VPN

Если вы уверены, что VPN-решение правильно установило флажок VPN, все готово. В противном случае используйте один из следующих методов:

- Создайте тип сети с именем VPN (предпочтительный вариант), затем свяжите подсети VPN с этим новым типом сети VPN.
- Создайте здание с именем VPN, затем свяжите подсети VPN с этим зданием. 

## <a name="query-fundamentals"></a>Запрос основных сведений

Правильно составленный запрос содержит все три этих параметра: 

- Measurement;
- Dimension;
- Filter.

Пример правильно составленного запроса: "Show me Poor Streams [Measurement] by Subnet [Dimension] for Building 6 [Filter]" ("Показать неудовлетворительные потоки [замер] по подсети [измерение] для здания 6 [фильтр]").

## <a name="what-does-union-do"></a>Что делает функция UNION?

Union позволяет фильтровать условия с помощью оператора AND. Существуют сценарии, с помощью которых можно объединять несколько условий фильтра для достижения результата, похожего на операцию или.

Пример. чтобы получить все потоки из здания, UNION предоставляет отдельное представление объединенного набора данных. Для использования UNION вставьте общий текст в поле UNION для двух условий фильтрации, которые нужно объединить.

## <a name="operational-processes"></a>Рабочие процессы

Сначала изучите управляемые потоки и исправьте их. В этой области качество должно полностью контролироваться вами и, следовательно, не должно быть проблем с его улучшением.

### <a name="managed-streams"></a>Управляемые потоки 

Выполняйте обзор и исправление управляемых потоков в следующем порядке. 

1. Сервер — сервер 
2.  Сервер — проводная связь — внутренняя связь
3. Проводная связь — проводная связь — внутренняя связь 

### <a name="unmanaged-streams"></a>Неуправляемые потоки

Ведите проверку и исправление неуправляемых потоков в следующем порядке. 

1. Сервер — беспроводная связь — внутренняя связь
2. Проводная связь — проводная связь — внешняя связь
3. Проводная связь — беспроводная связь — внешняя связь
4. Проводная связь — внешняя связь — прямая связь
5. Проводная связь — внешняя связь — ретрансляция
6. Прочие неуправляемые потоки
