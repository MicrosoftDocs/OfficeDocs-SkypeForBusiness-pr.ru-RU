# <a name="use-call-quality-dashboard-for-skype-for-business-server"></a>Используйте панель мониторинга качества вызовов для Skype для бизнеса Server

**Сводка:** Узнайте, как использовать панель мониторинга качества вызовов. Панель мониторинга качества вызовов — это средство для Skype для бизнеса Server.

Панель мониторинга качества вызовов (CQD) позволяет ИТ-профессионалам использовать агрегированные данные для выявления проблем, связанных с созданием проблем качества мультимедиа, сравнивая статистику для групп пользователей для определения тенденций и шаблонов. CQD ориентирован не на решение отдельных проблем вызовов, а на выявление проблем и решений, которые применяются ко многим пользователям.

## <a name="call-quality-dashboard-user-guide"></a>Руководство пользователя панели мониторинга качества вызовов

CQD — это веб-портал для быстрого создания и организации отчетов, основанных на данных Качества работы (QoE). CQD развертывает куб SSAS для агрегировать данные в базе данных метрики QoE и позволяет администраторам создавать и изменять отчеты или делать исследования в режиме реального времени. Хотя можно использовать Excel непосредственно для подключения к кубу, портал оптимизирован для нескольких процессов, связанных с данными QoE. Данные включают в себя:

- Кэшные данные отчетов для быстрого доступа
- Глубокие ссылки на страницы отчетов для обмена информацией и публикации
- Оптимизация редактирования и создания отчетов, а также редактируемые метаданные для описаний отчетов.

Кроме того, CQD предоставляет веб-API, которые дают пользователям программный доступ к данным куба для использования в настраиваемой панели мониторинга.

### <a name="default-reports"></a>Отчеты по умолчанию

При первом доступе к порталу мониторинга качества вызовов автоматически создается набор отчетов по умолчанию. Эти отчеты иногда называются системные отчеты. Вы можете свободно изменять или удалять эти отчеты или расширять их, создавая новые отчеты о семье и ребенке.

На верхнем уровне в отчете "Аудио Потоки ежемесячный тренд" показана ежемесячная тенденция для всех аудиопотоков. Перемещение мыши по барам в панели диаграммы, чтобы показать более подробное представление данных, представленных на диаграмме панели. Щелкните заголовок отчета Потоки тренда аудиозаписи, чтобы перейти к отчету "Управляемый и неуправимый звук Потоки", в котором отчеты разделены между управляемыми и неугодными вызовами. Управляемые вызовы — это вызовы, сделанные из корпоративного брандмауэра через проводные подключения. Неугодные вызовы включают вызовы, сделанные из-за корпоративного брандмауэра, и все вызовы, сделанные через Wi-Fi.

Другой отчет верхнего уровня называется "Гистограмма оценки качества вызовов, о чем сообщается пользователем". Оценки качества вызовов — это номера, Skype для бизнеса пользователями в конце звонка, чтобы указать качество вызова. Число рейтингов варьируется от 1 до 5, 1 — худшее, а 5 — наилучшее. В гистограмме показано количество звуковых вызовов, которые имели указанную оценку за один месяц.

Щелкните название любого из отчетов, чтобы перейти в отчеты с большим количеством фильтров данных. В отчетах системы каждый отчет о ребенке отображает подмножество данных, доступных в родительском отчете. Модель решения проблем проста: изучите, какие данные или тенденции указывают на то, что проблема ограничена, и постепенно сузить пространство проблем. Возможность создания подэкспортов позволяет исследовать собственные догадки о причинах определенных тенденций данных.

## <a name="cqd-streams"></a>CQD Потоки

Поток CQD считается хорошим, плохим или неклассифицированным. CQM 1.5 теперь использует следующее определение CQD:

- Плохой поток — это любое сочетание плохих показателей вызовов за пороговое значение.
- Если один поток в вызове является плохим, оба потока вызова помечены плохими. В конференциях каждый участник считается уникальным вызовом и сообщается независимо от всех остальных.
- Unclassified streams are streams without quality metrics (that is, Synthetic Transactions or short calls).
- Допустимые Потоки = немобильные клиенты
- Классификатор нельзя изменить

**Плохое определение вызовов/классификатор**

|Метрика|Порог|
|:-----|:-----|
|DegradationAvg |Больше 1.0 (-1 moS сети) |
|RoundTrip |Более 500 |
|PacketLossRate |Больше 0,1 (10%) |
|JitterInterArrival |Больше 30 |
|RatioConcealedSamplesAvg |Больше 0,07 |


Определение JPDR = Плохое определение вызовов минус RatioConcealedSamplesAvg

## <a name="where-is-callercallee"></a>Где caller/Callee?

CQD не использует поля Caller/Callee, а использует "Первый" и "Второй", так как между вызываемой и вызываемой.

 **Во-первых,** Всегда будет конечной точкой Сервера (например, AV MCU или Mediation Server), если сервер участвует в потоке.

 **Во-вторых,** Всегда будет конечной точкой клиента, если это не поток Server-Server.

**Пример первой и второй классификации**

|Endpoint 1 UAType|Endpoint 2 UUAType|Первый|Секунды|
|:-----|:-----|:-----|:-----|
|2 (AVMCU) |4 (Skype для бизнеса) |Конечная точка 1 |Конечная точка 2 |
|2 (AVMCU) |1 (mMediationServer) |Конечная точка 2 |Конечная точка 1 |
|4 (Skype для бизнеса) |4 (Skype для бизнеса) |Вызыватель в MediaLine |Callee в MMediaLine |


Если обе конечные точки одного типа, CQD делает запись Caller First и Callee Second. Дополнительные сведения о именах конечной точки см. [в этом блоге](/archive/blogs/jenstr/call-quality-dashboard-tips-and-tricks).

## <a name="accounting-for-vpn"></a>Учет VPN

Если vpn-решение, как известно, точно настроено VPN-флаг, вы все настроены. В противном случае используйте один из следующих методов:

- Создайте тип сети под названием VPN (предпочтительный), а затем соповедите VPN-подсети с этим новым VPN NetworkType.
- Создайте здание под названием VPN, а затем ассоциайте VPN-подсети с этим зданием.

## <a name="query-fundamentals"></a>Основы запроса

Хорошо сформированный запрос содержит все три этих параметра:

- Измерение
- Dimension
- Filter

Примером хорошо сформированного запроса может быть "Показать мне плохое Потоки [Измерение] по Subnet [Dimension] для построения 6 [Filter]".

## <a name="what-does-union-do"></a>Что делает UNION?

Union позволяет фильтровать условия с оператором AND. Существуют сценарии, в которых можно объединить несколько условий Filter для достижения результата, аналогичного операции OR.

Пример. Чтобы получить все потоки из здания, UNION предоставляет отчетливое представление объединенного набора данных. Чтобы использовать UNION, вставьте общий текст в поле UNION на двух условиях фильтра, которые необходимо использовать ДЛЯ UNION.
