# <a name="use-call-quality-dashboard-for-skype-for-business-server"></a>Использование панели мониторинга качества звонков для Скайп для Business Server
 
**Сводка:** Сведения о том, как использовать панель мониторинга качества звонков. Панель мониторинга качества звонков — это средство для Скайп для Business Server.
  
Панель мониторинга качества звонков (CQD) позволяет ИТ-специалистам использовать сводные данные, чтобы выяснить, на что необходимо обратить внимание в рабочей среде, где есть проблемы с качеством медиаданных. Эта панель позволяет ИТ-специалистам сравнивать статистические данные различных групп пользователей и выявлять тенденции и закономерности. Это средство призвано устранять не отдельные проблемы со звонками, а выявлять проблемы и находить решения, которые будут применяться для большого числа пользователей в этой рабочей среде.
  
## <a name="call-quality-dashboard-user-guide"></a>Руководство пользователя по панели мониторинга качества звонков

Панель мониторинга качества звонков (CQD) представляет собой веб-портал для быстрого формирования и систематизации отчетов на основе данных о качестве взаимодействия (QoE). В панели мониторинга качества звонков развернут куб SSAS, обеспечивающий сводное представление информации из базы данных показателей качества взаимодействия. Это позволяет пользователям создавать отчеты, изменять их и проводить исследования в режиме реального времени. Предусмотрена возможность прямого подключения Excel к кубу; при этом портал оптимизирован под различные рабочие процессы, предусматривающие обработку данных о качестве взаимодействия. К ним относится кэширование данных отчетов для быстрого доступа, прямые ссылки на страницы отчетов для обмена информацией и ее публикации, упрощенное редактирование и создание отчетов, а также редактируемые метаданные для описания отчетов. Панель мониторинга качества звонков также предоставляет доступ к веб-интерфейсам API, обеспечивающий возможность программного обращения к данным куба для их применения в пользовательских панелях мониторинга.  
  
### <a name="feature-overview"></a>Общие сведения о компоненте

При обращении пользователей к панели мониторинга качества вызовов отображается следующая информация.
  
![Использование CQD](../../media/1e061858-db6f-452b-9ae4-eab507220371.png)
  
1. «Сводка области», где можно найти контекст для «Отчет о набор» (справа). 
    
2. Отчет о Set уровня свойств (включая высота оси y) можно задать, щелкнув «Изменить» в области Сводка.
    
3. Строка навигатора позволяет пользователям определить текущее расположение в иерархии набора отчетов.  
    
4. Отчеты с подчиненными отчетами представлены синей ссылкой. Щелкнув ссылку, можно перейти к дочерним отчетам для детализации.  
    
При наведении указателя мыши на линейчатые диаграммы и графики тенденций отображаются подробные значения. Отчет, который имеет фокус будет показывать в меню Действие: «Edit», «Копировать», «Удалить» и «Загрузить». 
  
### <a name="default-reports"></a>Стандартные отчеты

При первом доступе пользователя к порталу панели мониторинга качества звонков автоматически создается стандартный набор отчетов. Эти отчеты также называют системными. Пользователи могут изменять и удалять их по своему усмотрению. Обычно пользователи расширяют их, создавая новые отчеты того же или более низкого уровня.  
  
На верхнем уровне отчета «Звуковых потоков ежемесячный тенденций» показывает ежемесячный тренда для всех аудиопотоков. При наведении указателя мыши на прямоугольники линейчатой диаграммы отображается более подробное представление данных диаграммы. Щелкнув заголовок отчета по тенденциям ежемесячный потоков аудио будут перемещаться в отчет «Управляемые vs неуправляемые потоки звук», где разделение отчетов между управляемые и неуправляемые вызовы. Управляемые звонки посылаются по проводным соединениям из зоны, защищенной корпоративным брандмауэром. К неуправляемым вызовам относятся звонки, источник которых находится за пределами корпоративного брандмауэра, а также все звонки по сетям Wi-Fi.
  
Отчете верхнего уровня называется «Пользователя, обнаруженных вызов качества оценки гистограмма.» Оценки качества звонков — это числовые значения, выставляемые пользователями Skype для бизнеса после завершения звонка и характеризующие его качество. Допускаются оценки от 1 (худшая) до 5 (лучшая). На гистограмме отображается количество звонков голосовой связи, получивших данную оценку в течение одного месяца. 
  
В общем случае, щелкнув заголовок любого отчета, можно перейти к отчетам с дополнительными фильтрами данных. В системных отчетах каждый дочерний отчет содержит подмножество данных, представленных в родительском отчете. Это в принципе позволяет реализовать простую модель решения задач: сузить пространство задачи путем определения подчиненного отчета, которым ограничены связанные с задачей данные или тенденции. Благодаря возможности создавать новые подчиненные отчеты пользователи могут проверять собственные гипотезы о происхождении конкретных тенденций изменения данных.
  
### <a name="creating-and-editing-reports"></a>Создание и редактирование отчетов

Если в меню действие отчета, нажав кнопку «Изменить», пользователи увидят редактор отчетов. В основе каждого отчета лежит запрос, обращенный к кубу. В отчете наглядно представлены данные, возвращенные в результате запроса. Редактор запросов представляет собой пользовательский интерфейс для редактирования таких запросов, а также для отображения параметров отчета. При открытии редактора запросов отображается следующая информация.
  
![Использование CQD](../../media/e8969625-e6f9-4d67-873f-93e78dd12b35.png)
  
1. На левой панели выбраны измерения, показатели и фильтры. При наведении курсора на одну из существующих значений будет отображаться кнопка «x», которая позволяет значение требуется удалить. Нажав кнопку «плюс» рядом с заголовком, откроется диалоговое окно для добавления нового измерения, измерения или фильтра. 
    
2. Вверху отображаются команды для настройки диаграммы.
    
3. В редакторе запросов возможен предварительный просмотр отчета.  
    
4. Внизу расположено редактируемое поле, позволяющее создать подробное описание отчета.

При наведении указателя мыши на линейчатую диаграмму и спарклайны отображаются значения по отдельным месяцам.  
  
![Использование CQD](../../media/fe6b18d7-b8cf-472a-9c93-0f7703f5a700.png)
  
В порядке для линейчатых диаграмм и спарклайны отображаться флажок «Показать спарклайны» в верхней части редактора отчета необходимо вернуть. Это выберите параметр тенденции и вниз месяц быть последнего измерения, которая также может быть выполнено, нажав кнопку на месяц и вверх и вниз стрелки, чтобы перенести StartDate.Month вверх или вниз. 
  
### <a name="settings"></a>Параметры

Меню параметров в правом верхнем углу панели мониторинга содержит ссылки на полезные страницы, содержащие, например, сведения о работоспособности системы или о программе.
  
![Использование CQD](../../media/0e9ae123-e231-4fea-94e1-5788e8f3e1d3.png)
  
Ли показывать описания и метки времени — до отдельных пользователей, и эти параметры только влияет на версию отдельного пользователя панели мониторинга, не изменяя изменять отчет или какие другие пользователи видеть. При очистке кэша из куба повторно загружаются данные по всем запросам, а при восстановлении значений по умолчанию удаляются все отчеты, созданные или измененные пользователем, и восстанавливается набор системных отчетов, который отображался при первом входе пользователя в систему.
  
Ссылка на пользовательские панели мониторинга позволяет перейти к странице, где пользователи могут просматривать список других пользователей панели мониторинга качества звонков и их отчеты. Для предоставления доступа к набору отчетов просто скопируйте ссылку в строке URL-адреса и передайте ее другому пользователю панели мониторинга качества звонков. Этой ссылки будет совпадать с одной другие пользователи будут видеть в странице ссылку пользователи панели мониторинга в поле имя пользователя.

## <a name="cqd-streams"></a>Потоки панели мониторинга качества звонков (CQD)

Поток CQD может быть удовлетворительным, неудовлетворительным или не иметь классификации. В настоящее время CQM 1.5 использует следующее определение CQD:  
  
- Неудовлетворительный поток — это любое сочетание неудовлетворительных показателей звонка, превышающих пороговое значение.
    
- Когда один поток в ходе вызова очень медленно, обоих потоков звонок, отмеченного низкого качества. В конференциях каждый участник подсчитывается как уникальный звонка и отображается на независимо от всех других пользователей.
    
- Потоки без классификации — это потоки, не имеющие показателей качества (например, искусственные транзакции или короткие звонки).
    
- Действенные потоки = немобильные клиенты
    
- Классификатор не может быть изменен
    
**Определение или классификатор неудовлетворительного звонка**

|**Показатель**|**Пороговое значение**|
|:-----|:-----|
|DDegradationAvg  <br/> |Больше 1 (–1 оценка MOS сети)  <br/> |
|RoundTrip  <br/> |Больше 500   <br/> |
|PacketLossRate  <br/> |Больше 0,1 (10 %)   <br/> |
|JitterInterArrival  <br/> |Больше 30   <br/> |
|RRatioConcealedSamplesAvg  <br/> |Больше 0,07   <br/> |
   
Определение JPDR = определение неудовлетворительного звонка минус RatioConcealedSamplesAvg  
  
## <a name="where-is-callercallee"></a>Где поля "Вызывающий абонент" и "Вызываемый абонент"?

CQD не использует поля вызывающего и вызываемого абонента. Они были переименованной "Первый" и "Секунды" из-за промежуточных этапов между вызывающего и вызываемого абонента.
  
 **Первый** всегда будет конечной точкой сервера (например, AV MCU; сервер-посредник), если сервер задействован в потоке.
  
 **Второй** всегда будет конечной точкой клиента, если только не используется поток типа "сервер — сервер". 
  
**Пример классификации "Первый" и "Второй"**

|**Конечная точка 1 UAType **|**Конечная точка 2 UUAType **|**Первый**|**Второй**|
|:-----|:-----|:-----|:-----|
|2 (AVMCU)   <br/> |4 (Skype для бизнеса)   <br/> |Конечная точка 1  <br/> |Конечная точка 2  <br/> |
|2 (AVMCU)   <br/> |1 (mMediationServer)   <br/> |Конечная точка 2  <br/> |Конечная точка 1  <br/> |
|4 (Skype для бизнеса)  <br/> |4 (Skype для бизнеса)   <br/> |Вызывающий абонент в MediaLine   <br/> |Вызываемый абонент в MMediaLine  <br/> |

## <a name="query-fundamentals"></a>Запрос основных сведений

Правильно составленный запрос содержит все три этих параметра:  
  
- Measurement;
    
- Dimension;
    
- Filter.
    
Пример правильно составленного запроса: "Show me Poor Streams [Measurement] by Subnet [Dimension] for Building 6 [Filter]" ("Показать неудовлетворительные потоки [замер] по подсети [измерение] для здания 6 [фильтр]").
  
## <a name="what-does-union-do"></a>Что делает функция UNION?

Функция UNION (объединение) позволяет фильтровать условия, используя оператор AND. Есть ситуации, когда нужно объединить несколько условий фильтрации, чтобы получить результат, схожий по действию с оператором OR.
  
Пример: если вам нужно получить все потоки из здания, функция UNION позволит получить четкое видение объединенного набора данных. Для использования UNION вставьте общий текст в поле UNION для двух условий фильтрации, которые нужно объединить.

## <a name="operational-processes"></a>Рабочие процессы

Начните с обзора и исправления управляемых потоков. В этой области качество должно полностью контролироваться вами и, следовательно, не должно быть проблем с его улучшением.  
  
### <a name="managed-streams"></a>Управляемые потоки 

Выполняйте обзор и исправление управляемых потоков в следующем порядке.  
  
1. Сервер — сервер  
    
2.   Сервер — проводная связь — внутренняя связь
    
3. Проводная связь — проводная связь — внутренняя связь  
    
### <a name="unmanaged-streams"></a>Неуправляемые потоки

Ведите проверку и исправление неуправляемых потоков в следующем порядке.  
  
1. Сервер — беспроводная связь — внутренняя связь
    
2. Проводная связь — проводная связь — внешняя связь
    
3. Проводная связь — беспроводная связь — внешняя связь
    
4. Проводная связь — внешняя связь — прямая связь
    
5. Проводная связь — внешняя связь — ретрансляция
    
6. Прочие неуправляемые потоки
