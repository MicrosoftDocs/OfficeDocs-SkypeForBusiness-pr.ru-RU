# <a name="troubleshoot-statistics-manager-for-skype-for-business-server-2015"></a>Устранение проблем диспетчера статистики в Skype для бизнеса Server 2015
 
**Сводка:** Прочтите этот раздел для устранения неполадок развертывания из диспетчера статистики для Скайп Business Server 2015.
  
В этом разделе описывается развертывание диспетчера статистики Устранение неполадок, связанных с описанием событий, которые могут отображаться в журнале событий приложений и соответствующие действия, выполняемые для устранения события. This topic contains the following sections:
  
- [События агента](troubleshoot.md#BKMK_Agent)
    
- [События прослушивателя](troubleshoot.md#BKMK_Listener)
    
- [Проблемы с веб-сайтом](troubleshoot.md#BKMK_Website)
    
## <a name="agent-events"></a>События агента
<a name="BKMK_Agent"> </a>

- **1000** — не удается настроить ограничитель процессора (объект задания) — причина неизвестна
    
- **1001** — ограничивать процесс не разрешено на процесс (возможно, помещенный в объект задания)
    
    Агент выполняется внутри объекта задания Windows, чтобы автоматически ограничить занимаемый объем памяти. Если агент не запускается, а эти записи о событиях присутствуют в журнале событий, объект задания не может быть создан на сервере. Чтобы обойти эту проблему, может снять верхнее ограничение для памяти, изменив значение в файле конфигурации.
    
  ```
  C:\Program Files\Skype for Business Server StatsMan Agent\PerfAgent.exe.config
  ```

    Поиск «MaxProcessMemoryMB» и измените значение «0», как показано:
    
  ```
  <setting name="MaxProcessMemoryMB" serializeAs="String"> <value>300</value> </setting>
  ```

    > [!NOTE]
    > При этом изменения, агент будут по-прежнему обычно используют \< 100 МБ памяти, однако он не будет принудительно более 300 МБ как используется по умолчанию. После внесения изменения рекомендуется тщательно следить за использованием памяти, чтобы агент не потреблял слишком большой объем памяти на хост-компьютере. 
  
- **2000** — сбой инициализации клиента
    
- **2001**— не удается установить подключение к службе на исходном IP-адресе
    
    Если агенту не удается подключиться к компьютеру с прослушивателем, проверьте следующее:
    
1. Убедитесь, что служба прослушивателя запущена на компьютере с прослушивателем. Если нет, убедитесь, что служба Redis работает на этом сервере, затем перезапустите службу прослушивателя.
    
    Проверьте журнал событий диспетчера статистики на компьютере прослушиватель, убедитесь, что отсутствуют проблемы с самой службы диспетчера статистики прослушивателя.
    
2. С помощью средства проверки подключений, например, Telnet, проверьте подключение от компьютера с агентом к прослушивателю на нужном порте.
    
    If not, make sure the incoming firewall rule is enabled on the Listener computer for the network type that the Listener computer is connected to (private/public/domain). Если прослушиватель компьютер не присоединен к домену, сети могут отображаться как public, и в этом случае не будет применяться правила брандмауэра, установленные с помощью диспетчера Статистика по умолчанию.
    
- **4000** — ошибка загрузки информации о сервере из прослушивателя (причина неизвестна)
    
  - **4001** — сервер не найден в топологии прослушивателя
    
    Эта ошибка возникает, если сервер успешно подключается к прослушиватель, но сервер не был добавлен в топологию в кэше прослушивателя. Возможны следующие варианты решения:
    
  - 	Убедитесь, что вы выполнили инструкции по импорту топологии. См. статью [Импорт топологии](deploy.md#BKMK_ImportTopology).   
    
  - Если агент находится на сервере, который не указан в топологии (например, узлы в кластере SQL AlwaysOn), необходимо добавить агент вручную, следуя инструкциям в статье [Импорт топологии](deploy.md#BKMK_ImportTopology).
    
  - **4002** — недопустимый пароль прослушивателя
    
    Зашифрованный пароль, который агент пытается использовать, не соответствует паролю для службы в самом прослушивателе. Удалите агент и снова установите его, используя правильный пароль для службы.
    
  - **4003** — несоответствие Thumbprint сертификата
    
    Thumbprint сертификата, назначенный агенту при установке, не соответствует Thumbprint в сертификате, который в настоящее время использует прослушиватель, поэтому в подключении будет отказано. Удалите агент и снова установите его, используя правильный Thumbprint сертификата.
    
  - **4004** — недопустимый ответ или HttpStatusCode
    
    Прослушиватель не отвечает ожидаемым состоянием.   
    
  - Если подключение осуществляется через прокси-сервер, проверьте конфигурацию прокси.
    
  - В журнале компьютера прослушиватель StatsMan проблемы с конфигурацией.
    
  - **4005** — не удалось отменить последовательность XML
    
    Информация о сервере на сервере прослушивателя повреждена или имеется несоответствие версий между компьютерами агента и прослушивателя. Убедитесь, что версии совпадают, и поищите проблемы в журнале событий прослушивателя.
    
## <a name="listener-events"></a>События прослушивателя
<a name="BKMK_Listener"> </a>

- **10000** — сбой запуска, причина неизвестна (не удается исправить, служба останавливается или завершает работу со сбоем)
    
  - **10001** — проблема конфигурации
    
    Обычно эти ошибки возникают, когда файл [каталог_установки_прослушивателя]\PerfAgentListener.exe.config был изменен вручную, и приложению не удается его прочитать.
    
  - **10002** — ошибка инициализации HTTP-протокола прослушивателя
    
    Это событие обычно записывается, если список управления доступом для URL-адресов неправильно настроен при установке или если сертификат SSL недействителен. Убедитесь, что сертификат в конфигурации действителен. Если это так, переустановите прослушиватель, следуя инструкциям в статье [Развертывание диспетчера статистики](deploy.md#BKMK_Deploy).
    
  - **10003** — сбой Redis
    
  - **10004** — сбой инфраструктуры кэширования
    
  - **10007** — параметры (которые хранятся в Redis)
    
    Прослушивателю не удалось связаться с Redis или извлечь сформированные данные из кэша и не удалось запуститься. Убедитесь, что служба Redis запущена и правильно настроена на сервере.
    
  - **10005** — извлечение/синтаксический анализ информации о сервере
    
    Информация о топологии в кэше Redis недействительна. Сначала попробуйте перезапустить Redis и прослушиватель. Если ошибка останется, воссоздайте данные топологии, следуя инструкциям в статье [Импорт топологии](deploy.md#BKMK_ImportTopology).
    
- **10100** — сбой связи Redis
    
  - **10101** — продолжительный сбой связи Redis (каждые 60 секунд)
    
  - **30100** — устранение сбоя связи Redis
    
    Эти события будут записаны, если прослушивателю не удается подключиться к Redis. Убедитесь, что служба Redis запущена и подключение между службой и прослушивателем доступно.
    
- **10200** — сбой записи Redis
    
  - **10201** — продолжительный сбой записи Redis (каждые 60 секунд)
    
  - **30100** — устранение сбоя записи Redis
    
    Эти события будут записаны, если прослушивателю не удается выполнить запись в кэш Redis. Убедитесь, что служба Redis запущена и подключение между службой и прослушивателем доступно.
    
- **30000** — успешный запуск
    
    Это событие записывается каждый раз при запуске прослушивателя.
    
- **22000** — инициализации из статистики агента Manager успешно завершен.
    
- **23000** — успешная инициализация EventLogQueryManager (первичная или после сбоя)
    
- **24000** — успешная инициализация информации о сервере (первичная или после сбоя)
    
- **25000** — прослушиватель снова в сети после ошибки публикации (или выполнена первая успешная публикация)
    
- **5000** — автономный запуск прослушивателя для публикации данных
    
- **5001** — прослушиватель работает в автономном режиме в течение длительного времени
    
    Эти события полезны для мониторинга, оповещения и устранения проблем.
    
## <a name="website-issues"></a>Проблемы с веб-сайтом
<a name="BKMK_Website"> </a>

- Входа повторяющихся предупреждений в Chrome - речь шла об ошибках, проблема устранена в версии 1.1. Убедитесь, что после обновления до последней версии диспетчера статистики Если вы видите запросы повторяющиеся для входа в браузере хрома. To verify the version of the website you are running:
    
  - 	В Проводнике файлов откройте (каталог по умолчанию)
    
  - Щелкните правой кнопкой мыши файл StatsManHubWebSite.dll и просмотрите его свойства.
    
  - Если компьютер не найден в представлении ландшафта KHI или в представлении сведений о счетчике, убедитесь, что он входит в состав сайта и пула. Если нет, компьютер не будет отображаться в этих представлениях. Дополнительные сведения об определении сайта и пула для сервера в топологии см. в статье [Импорт топологии](deploy.md#BKMK_ImportTopology).
    
  - Версия продукта будет показана в сведениях описания.
