# <a name="exchange-unified-messaging-online-migration-support"></a>Поддержка сетевой единой системы обмена сообщениями Exchange

> [!IMPORTANT]
> **Поддержка службы единой системы обмена сообщениями в Exchange Online по состоянию на 28 февраля 2020 г. по тихоокеанскому времени. Все учетные записи голосовой почты были перенесены в облачную службу голосовой почты корпорацией Майкрософт. Оставшийся трафик автосекретаря не будет отслеживаться и может быть прерван в любое время.**

В связи с [объявлением](https://blogs.technet.microsoft.com/exchange/2019/02/08/retiring-unified-messaging-in-exchange-online/) 8 февраля 2019 г. корпорация Майкрософт перестает использовать службу Exchange Unified Messaging Online (ExchUMO) до 28 февраля 2020 г. В этой статье приведено сводная информация о том, что затронутые клиенты должны знать и продумать планирование непрерывности бизнеса.

ExchUMO развертывается клиентами для служб голосовой почты, автосекретаря, очереди звонков и интеграции факсов. Корпорация Майкрософт планирует помочь клиентам перейти на службы телефонной системы, которые уже поддерживают тысячи клиентов в Skype для бизнеса Online и Microsoft Teams.

Голосовая почта в основном выполняет миграцию под руководством корпорации Майкрософт; для подмножества клиентов может потребоваться вовлечение в администратор и /или инвестиции. Автосекретарь переносится под управлением администратора; Существующие деревья автосекретаря ExchUMO потребуется создать заново в облачной службе автосекретаря. Клиенты, использующие какие-либо функции ExchUMO со сторонними УАТС, не будут перенесены в облачные службы Skype, так как они не поддерживают сторонние Системы УАТС. В этом блоге мы объявили о поддержке сторонних [разработчиков,](https://techcommunity.microsoft.com/t5/Exchange-Team-Blog/New-date-for-discontinuation-of-support-for-Session-Border/ba-p/607853)и клиенты, участвующих в этой модели развертывания, могут перенести своих пользователей на одну из платформ или служб объединенных коммуникаций Майкрософт или приобрести стороннее решение голосовой почты и/или автосекретаря для этих пользователей. Интеграция с факсами в облачных службах не поддерживается; клиентам придется перейти на стороннее решение.

## <a name="who-is-affected"></a>Кто затронут?

На клиентов, которые используют службу единой системы обмена сообщениями Exchange Online, это оказывают влияние на пользователей, которые используют какие-либо из следующих возможностей:

- Служба голосовой почты
- Служба автосекретаря
- Служба очереди звонков
- Интеграция факсов

> [!Note]
> На клиентов, которые используют любой локальный сервер Exchange Server с единой системой обмена сообщениями, это не оказывает влияние.

Узнайте подробнее о влиянии на работу пользователей и его [влиянии на работу пользователей.](#user-experience-impact)

## <a name="voicemail-migration-guidelines"></a>Рекомендации по миграции голосовой почты

### <a name="get-informed"></a>Получение информированного

Ознакомьтесь с объявлением в [блоге](https://blogs.technet.microsoft.com/exchange/2019/02/08/retiring-unified-messaging-in-exchange-online/) и данной статьей, чтобы ознакомиться с планом гладкой миграции для пользователей. Дополнительные [сведения о возможностях голосовой почты телефонной системы](https://support.office.com/article/check-skype-for-business-voicemail-and-options-2deea7f8-831f-4e85-a0d4-b34da55945a8) см. в статье "Проверка голосовой почты Skype для бизнеса" и "Варианты".  

### <a name="establish-a-skype-for-business-hybrid-topology"></a>Создание гибридной топологии Skype для бизнеса

Если гибридная топология Skype для бизнеса не установлена, необходимо выполнить миграцию пользователей голосовой почты. Дополнительные [сведения см. в разделе "Настройка гибридной среды Skype](../../SfbHybrid/hybrid/configure-federation-with-skype-for-business-online.md) для бизнеса".

> [!Note]
> Переносить пользователей по сети для переноса службы голосовой почты не требуется. Однако для локальных пользователей с использованием службы голосовой почты телефонной системы необходимо создать гибридную топологию.

### <a name="plan-your-auto-attendant-migration"></a>Планирование переноса автосекретаря

Администраторы могут в любое время начать перенос своих автосекретарей из ExchUMO в облачный автосекретарь. Дополнительные [сведения см. в статье "Настройка облачного](https://docs.microsoft.com/microsoftteams/create-a-phone-system-auto-attendant) автосекретаря".

Корпорация Майкрософт продолжает использовать дополнительные возможности автосекретаря, которые могут потребоваться клиентам для миграции. Администраторам следует сортировать набор функций и соответствующим образом перенести экземпляры автосекретаря. Сравнение функций см. в описании [матрицы служб ExchUMO и Azure.](#exchumo-and-azure-cloud-based-services-feature-matrix)

### <a name="plan-for-your-voicemail-post-migration-validation-and-testing"></a>Планирование проверки и тестирования голосовой почты после миграции

Перенос голосовой почты выполняется на основе корпорации Майкрософт. Администраторы могут не выполнять никаких действий, учитывая, что топология является обязательной гибридной топологией. Корпорация Майкрософт выполняет необходимые проверки и тестирование, чтобы убедиться в том, что миграция голосовой почты не нарушена. Администраторы могут проводить тестирование и проверку со своих систем. Сведения о [рекомендуемом плане тестирования и проверке после переноса для администраторов рекомендуемого](#suggested-test-plan-and-post-migration-validation-for-admins) плана тестирования.

> [!Note]
> Lync Server 2010 не поддерживается. Если вы используете развертывание сервера 2010, следует запланировать обновление сервера или попробовать перенести пользователей на Microsoft Teams или Skype для бизнеса Online.  

### <a name="monitor-the-admin-notification-center"></a>Мониторинг Центра уведомлений администратора

Обратите внимание, что в Центре уведомлений администратора можно усилить дополнительные сведения и временную шкалу, касающуюся миграции пользователей. Уведомления отправляются не реже за 30 дней до вашего периода миграции.

> [!Note]
> Если вы получили уведомление с временной шкалой миграции и хотите отложить миграцию, чтобы оказать важную для бизнеса, это можно сделать путем обращения в службу поддержки Майкрософт. Миграция нельзя отложить до 28 февраля 2020 г. По клиентам с дополнительными вопросами обратитесь в службу технической поддержки по работе с клиентами или службе поддержки Майкрософт. Клиенты, уже использующих Microsoft 365 или Office 365, могут отправить обращение в службу поддержки через Центр администрирования Microsoft 365.

### <a name="consider-opting-in-for-a-planned-migration"></a>Рассмотрите возможность принятия решения для запланированной миграции

Вы можете принять увас поддержку запланированного переноса службы голосовой почты в CVM. Перед согласием на ее использование просмотрите сведения, представленные в этой статье, особенно в следующих разделах:

- Шаги миграции (этот раздел)
- Матрица функций облачных служб ExchUMO и Azure
- Влияние на работу пользователей

При выборе управляемой миграции в Центре сообщений портала администрирования Microsoft 365 не будет выдано уведомление за 30 дней.

Чтобы согласиться на запланированную миграцию, отправьте запрос на получение электронной почты с [адреса cvm@microsoft.com электронной почты](mailto:cvm@microsoft.com) со следующими сведениями:

- Предпочитаемая дата (вторник): этапы миграции выполняются каждый вторник. Выберите дату для вторника, не выступившего в 03.02.2019.
 
- Идентификатор клиента: 32 символа в формате 0046728c-688a-4472-a38f-098fec60ac6x. Идентификатор клиента можно найти на портале администрирования Microsoft 365 в Azure AD или с помощью следующего командлета PowerShell: `Get-CsTenant | Select ObjectId`

После успешного переноса клиента вы получите по электронной почте подтверждение.

## <a name="auto-attendant-migration-guidelines"></a>Рекомендации по миграции автосекретарей

Администраторы организаций Microsoft 365 и Office 365 должны повторно создать автосекретари единой системы обмена сообщениями Exchange в службе автосекретарей Microsoft Cloud автосекретаря и переключить на них локальные номера до прекращения поддержки службы UMO для Exchange 28 февраля 2020 г. Это рекомендуемое руководство для успешной миграции и проверки новых автосекретарей в облаке. При наличии большого числа автосекретарей можно использовать скрипты миграции автосекретаря единой системы обмена сообщениями [Exchange](https://github.com/NathanJBennett/ExUMAAMigrationToCloudAA) для переноса в облачную систему автосекретаря, чтобы упростить массовую миграцию автосекретарей.

### <a name="auto-attendant-setup"></a>Настройка автосекретаря

Настоятельно рекомендуем начать настройку новых автосекретарей раньше во избежание последних минут и ознакомиться с функциями и опытом работы службы автосекретаря в облаке. Для автосекретарей, которым требуется один или несколько функций неразрывов, то функции автосекретарей можно создавать и тестировать, когда функции разрывов в подготовке к развертыванию. Дополнительные сведения о функциях gap см. в [приложении.](#appendix)

1. Командлеты Exchange UMO для экспорта конфигурации существующих автосекретарей с помощью [командлета Get-UMAutoAttendant.](https://docs.microsoft.com/powershell/module/exchange/unified-messaging/get-umautoattendant)  
2. С [помощью командлета Export-UMprompt в](https://docs.microsoft.com/powershell/module/exchange/unified-messaging/export-umprompt) Exchange Online PowerShell экспортируйте файлы приветствия (если они используются) и преобразуйте их в формат MP3.
3. Следуйте инструкциям в разделе ["Планирование облачных автосекретарей"](../../SfbHybrid/hybrid/plan-cloud-auto-attendant.md) и ["Настройка облачного](https://docs.microsoft.com/microsoftteams/create-a-phone-system-auto-attendant) автосекретаря" для создания автосекретарей с помощью Центра администрирования Microsoft Teams или PowerShell.
4. Проверьте приветствия, если меню изменены.
5. Настройте переводы в группы ответа, используя обходное решение "Переключение звонка автосекретаря на ТСОП" в разделе ["Известные проблемы"](#known-issues) этой статьи.  
6. Проверьте новые автосекретари, названия им внутренне или назначив тестовый номер телефона.  

### <a name="cutover"></a>Прямая

1. Переключение телефонных номеров с автосекретарей единой системы обмена сообщениями Exchange на новые автосекретари.
2. Перемещение URI SIP из контактного объекта в учетную запись ресурса.
3. Протестируйте и проверьте свои автосекретари с помощью недавно назначенных телефонных номеров.

### <a name="suggested-test-plan-and-post-migration-validation-for-admins"></a>Рекомендуемый план тестирования и проверка после миграции для администраторов

Чтобы убедиться в том, что пользователи перенесены в облачную голосовую почту, оставьте сообщение голосовой почты пользователю и проверьте текст сообщения в Outlook. Сообщения голосовой почты в облаке имеют колонтитул, который читает следующие элементы:

«Благодарим вас за использование транскрибировки! Если выше не отображается транскрибирование, качество звука было недостаточно понятно".

При тестировании функции голосовой почты после миграции пользователей следует рассмотреть следующие сценарии:

- проверять доступ к голосовой почте на всех типах конечных точек в организации, таких как приложения и IP-телефоны;
- Проверьте с демонстрацией пользователей, что настроенные персонализированное приветствие воспроизводится для абонентов.
- Если в вашей организации необходимо законодательно или соблюдение нормативных требований для отключения транскрибировки для пользователей, убедитесь, что она отключена после миграции. Дополнительные сведения см. в статье ["Настройка облачной голосовой почты".](/microsoftteams/set-up-phone-system-voicemail)
- Если вы ранее настраивали политики и правила для виртуальных машин Exchange, убедитесь, что они эффективны.
- Ознакомьтесь с командлетами PowerShell облачной голосовой почты для изменения параметров пользователей.

### <a name="known-issues"></a>Известные проблемы

#### <a name="greeting-inconsistencies"></a>Несоответствий приветствиях

Те, когда служба не будет полностью удалена, даже после перенесения всех пользователей на облачную голосовую почту можно будет работать и до тех пор, пока служба не будет полностью удалена. Во избежание несогласованной работы и различий пользователя отключите абонентский доступ с момента изменения приветствий после миграции. Для этого удалите контакт ный абонентский указываемый контакт, `Get-CsExUmContact | ?{$_.IsSubscriberAccess -eq $true} | Remove-CsExUmContact` использующий.

#### <a name="auto-attendant-call-transfer-to-pstn"></a>Переключение вызова автосекретаря на ТСОП

Для передачи звонка автосекретаря на внешний номер телефона ТСОП через Skype для бизнеса Server или службу группы ответа (RGS) в Skype для бизнеса Server создайте нового локального пользователя, для которых задан алгоритм переадресации вызовов на номер телефона ТСОП или номер телефона RGS. Пользователю должна быть правильно включена и настроена служба "Корпоративная голосовая связь" и ему назначена политика голосовой связи.

#### <a name="shared-mailbox-is-still-accessible"></a>Общий почтовый ящик по-прежнему доступен

Общий почтовый ящик, настроенный с помощью единой системы обмена сообщениями Exchange Online, по-прежнему будет получать сообщения после перехода в CVM и станут доступными для пользователей через Outlook. Однако доступ к измененному сообщению с приветствием от этого почтового ящика будет недоступен после переноса в CVM. Клиенты с общими почтовыми ящиками, которые используются для перехвата абонентов автосекретаря, должны использовать автосекретари и общие почтовые ящики очередей после выпуска (ETA октября 2019 г.).
  
#### <a name="username-is-not-using-skype-for-business-banner-displays"></a>Отображение баннера "Имя пользователя не использует Skype для бизнеса"

Служба CVM основана на инфраструктуре Microsoft Teams, а звонки из клиента Skype для бизнеса могут привести к отображению баннера информации в клиенте с получаемыми свойствами: "Имя пользователя не использует Skype для бизнеса. Чтобы усилить расширенные возможности, перейдите в Teams или запустите собрание Skype".
Обязательно обновите клиент Skype для бизнеса пользователей с помощью последнего обновления клиента "колеба", чтобы этот баннер не появлялся.
  
#### <a name="set-up-voice-mail-takes-you-to-owa"></a>"Настройка голосовой почты" позволяет перенести в OWA

Если **щелкнуть "Настройка голосовой почты"** у клиента, пользователи Skype для бизнеса Server 2015 или 2013 будут перенесены на страницу портала Office Web Access (OWA) после перехода на CVM. Все параметры удалены со вкладки "Voicemail" в Outlook Web App, а в баннере отображается ссылка перенаправления, чтобы перенаправлять пользователей на портал настройки пользователей CVM.

#### <a name="changing-greeting-remotely"></a>Удаленное изменение приветствия

Абонентский доступ по ТСОП не поддерживается в CVM. Для пользователей, которым нужно удаленно изменить приветствие, в службу IVR голосовой почты для мобильных клиентов был добавлен пункт меню "Изменение приветствия". Для вызова этой службы пользователи могут нажать и удерживая клавишу 1 на панели номера для мобильного клиента.
