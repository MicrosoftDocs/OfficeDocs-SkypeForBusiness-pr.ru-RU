# <a name="exchange-unified-messaging-online-migration-support"></a>Поддержка сетевой единой системы обмена сообщениями Exchange

> [!IMPORTANT]
> **В службе единой системы обмена сообщениями в Exchange Online не поддерживается 28 февраля, 2020, 5 по тихоокеанскому времени. Все учетные записи голосовой почты были перенесены в облачную службу голосовой почты Майкрософт. Весь оставшийся трафик автосекретаря не будет отслеживаться и может быть нарушен в любой момент.**

В ссылке на [объявление](https://blogs.technet.microsoft.com/exchange/2019/02/08/retiring-unified-messaging-in-exchange-online/) 8 февраля 2019 Корпорация Майкрософт отпустить службу единой системы обмена сообщениями Exchange Online (ексчумо) по 28 февраля 2020. В этой статье представлены сведения о том, что затронутые клиенты должны знать и как спланировать непрерывность бизнеса.

Ексчумо развертывается клиентами для голосовой почты, автосекретаря, очереди вызовов и служб интеграции с факсом. Корпорация Майкрософт планирует помочь клиентам выполнить миграцию на службы телефонной системы, которые уже поддерживают тысячи клиентов в Skype для бизнеса Online и Microsoft Teams.

В основном Голосовая почта — это миграция на основе Майкрософт; для подмножества клиентов может потребоваться вовлечение в группу и/или инвестиция. Автосекретарь — это миграция, управляемая администратором; Вам потребуется повторно создать существующие деревья автосекретаря Ексчумо в облачной службе облачного секретаря. Клиенты, которые используют любую из функций Ексчумо со сторонней УАТС, не будут перенесены в облачные службы Skype, так как они не поддерживают сторонние системы УАТС. В [этом блоге](https://techcommunity.microsoft.com/t5/Exchange-Team-Blog/New-date-for-discontinuation-of-support-for-Session-Border/ba-p/607853)был объявлен план выбытия от сторонних производителей, и клиенты в этой модели развертывания могут переносить пользователей на одну из платформ или служб единой системы обмена сообщениями корпорации Майкрософт, а также получать решение голосовой почты и/или автосекретаря стороннего производителя для этих пользователей. Интеграция факса не поддерживается в облачных службах; пользователям потребуется выполнить миграцию на стороннее решение.

## <a name="who-is-affected"></a>Кто затронул?

Для клиентов, использующих любую из следующих функций из веб-службы единой системы обмена сообщениями Exchange, влияют:

- Служба голосовой почты
- Служба автосекретаря
- Служба очереди звонков
- Интеграция факса

> [!Note]
> Клиенты, которые используют любой локальный сервер Exchange с единой системой обмена сообщениями, не затрагиваются.

Узнайте больше о работе пользователей и администраторов, влияющих на [взаимодействие с пользователем](#user-experience-impact).

## <a name="voicemail-migration-guidelines"></a>Рекомендации по переносу голосовой почты

### <a name="get-informed"></a>Получение уведомления

Ознакомьтесь с [объявлением блога](https://blogs.technet.microsoft.com/exchange/2019/02/08/retiring-unified-messaging-in-exchange-online/) и этой статьей, чтобы спланировать гладкую миграцию для пользователей. Дополнительные сведения о возможностях голосовой почты в телефонной системе можно найти в статье [Проверка голосовой почты и параметров Skype для бизнеса](https://support.office.com/article/check-skype-for-business-voicemail-and-options-2deea7f8-831f-4e85-a0d4-b34da55945a8) .  

### <a name="establish-a-skype-for-business-hybrid-topology"></a>Создание гибридной топологии Skype для бизнеса

Если вы не установили гибридную топологию Skype для бизнеса, вам нужно сделать это, чтобы обеспечить гладкую миграцию пользователей голосовой почты. Дополнительные сведения можно найти в статье [Настройка гибридной среды Skype для бизнеса](../../SfbHybrid/hybrid/configure-federation-with-skype-for-business-online.md) .

> [!Note]
> Для переноса службы голосовой почты пользователи не обязаны переносить свои пользователи в оперативный режим. Однако для локальных пользователей, которые используют службу голосовой почты телефонной системы, необходимо установить гибридную топологию.

### <a name="plan-your-auto-attendant-migration"></a>Планирование миграции автосекретаря

Администраторы могут начать перенос автоматических ассистентов из Ексчумо в облачный автосекретарь в любое время. Более подробную информацию можно узнать [в статье Настройка облачного автосекретаря](https://docs.microsoft.com/microsoftteams/create-a-phone-system-auto-attendant) .

Корпорация Майкрософт продолжает предоставлять дополнительные возможности для автосекретаря, которые могут быть необходимы клиентам для миграции. Администраторы должны оценить набор компонентов и соответствующим образом перенести их экземпляры. Сведения об сравнении со списками функций можно найти в статье [ексчумо and the облачные службы на основе Azure](#exchumo-and-azure-cloud-based-services-feature-matrix).

### <a name="plan-for-your-voicemail-post-migration-validation-and-testing"></a>Планирование проверки голосовой почты после миграции и тестирования

Миграция голосовой почты осуществляется на основе Майкрософт. Администраторам не требуется выполнять какие-либо действия с учетом того, что установлена предварительная Гибридная топология. Корпорация Майкрософт выполняет необходимую проверку и тестирование, чтобы убедиться, что миграция голосовой почты пользователей не нарушена. Администраторам рекомендуется выполнять тестирование и проверку на их стороне. Рекомендуемый план тестирования представлен в статье [предложенный план тестирования и проверка после миграции для администраторов](#suggested-test-plan-and-post-migration-validation-for-admins) .

> [!Note]
> Lync Server 2010 не поддерживается. Если вы используете развертывание сервера 2010, то вам следует спланировать обновление сервера или рассмотреть возможность переноса пользователей в Microsoft Teams или Skype для бизнеса Online.  

### <a name="monitor-the-admin-notification-center"></a>Мониторинг центра уведомлений администратора

Обратите внимание на уведомление в центре уведомлений администрирования с дополнительными сведениями и временной шкалой, касающейся миграции пользователей. Уведомления отправляются по крайней мере за 30 дней до своего периода миграции.

> [!Note]
> Если вы получили уведомление с временной шкалой миграции пользователей и хотите отложить миграцию по критической причине, можно обратиться в службу поддержки Майкрософт. Вы не можете отложить миграцию за пределами даты выбытия 28 февраля 2020 г. Для клиентов, у которых есть дополнительные вопросы, обратитесь в службу поддержки вашей учетной записи или в службу поддержки Майкрософт. Клиенты, которые уже используют Microsoft 365 или Office 365, могут отдать заявку на поддержку через центр администрирования Microsoft 365.

### <a name="consider-opting-in-for-a-planned-migration"></a>Рассмотрите вопрос о запланированной миграции

Вы можете присоединиться к запланированной миграции службы голосовой почты в КВМ. Прежде чем приступить к отрецензированию, ознакомьтесь со сведениями в этой статье, особенно в следующих разделах:

- Этапы миграции (этот раздел)
- Ексчумо и на облачных службах Azure
- Влияние на взаимодействие с пользователем

При выборе управляемой миграции в центре сообщений портала администрирования Microsoft 365 не отображается уведомление о доходе до миграции в течение 30 дней.

Чтобы принять участие в запланированной миграции, отправьте запрос электронной почты от своего адреса электронной почты администратора в [CVM@microsoft.com](mailto:cvm@microsoft.com) со следующими сведениями:

- Предпочтительная дата (вторникам): этапы миграции выполняются каждый вторник. Выберите дату на вторник, который не превышает 12/3/2019.
 
- Идентификатор клиента: 32 символов в этом формате: 0046728c. 688A. 4472 – a38f – 098fec60ac6x. Идентификатор клиента можно найти на портале администрирования Microsoft 365 в Azure AD или с помощью следующего командлета PowerShell: `Get-CsTenant | Select ObjectId`

После успешной миграции клиента вы получите подтверждение по электронной почте.

## <a name="auto-attendant-migration-guidelines"></a>Рекомендации по миграции автосекретаря

Microsoft 365 и Office 365 организации необходимы для повторного создания интерактивных автосекретарей единой системы обмена сообщениями Exchange в службе автосекретаря Microsoft Cloud, а также для переключения их локальных телефонных номеров до даты выбытия службы УМО Exchange 28 февраля 2020. Это рекомендуемая рекомендация для успешного переноса и тестирования новых автоматических ассистентов. Если вы используете большое количество автосекретарей, вы можете использовать [автосекретарь единой системы обмена сообщениями Exchange для сценариев миграции облачного автоматического секретаря](https://github.com/NathanJBennett/ExUMAAMigrationToCloudAA) , чтобы упростить массовый перенос автосекретарей.

### <a name="auto-attendant-setup"></a>Настройка автосекретаря

Мы настоятельно рекомендуем начать настройку новых автосекретарей раньше, чтобы избежать проблем с последними минутами, а также ознакомиться с функциональными возможностями и опытом службы автосекретаря облачной службы. Для автосекретарей, требующих одну или несколько неразрывов, можно создать и протестировать автоматические ассистенты, когда доступны функции зазоров для подготовки к развертыванию. Дополнительные сведения о неразрывных функциях можно найти в [приложении](#appendix).

1. Используйте командлеты Exchange УМО для экспорта конфигурации существующих автосекретарей с помощью командлета [Get – UMAutoAttendant используется](https://docs.microsoft.com/powershell/module/exchange/unified-messaging/get-umautoattendant).  
2. С помощью командлета [Export-UMprompt](https://docs.microsoft.com/powershell/module/exchange/unified-messaging/export-umprompt) в Exchange Online PowerShell экспортируйте файлы мультимедиа приветствия (если они используются) и преобразуйте их в формат MP3.
3. Следуйте инструкциям в статье [Plan Cloud Attendant Autos](../../SfbHybrid/hybrid/plan-cloud-auto-attendant.md) и [Настройка облачного автосекретаря](https://docs.microsoft.com/microsoftteams/create-a-phone-system-auto-attendant) для создания автоматических ассистентов с помощью центра администрирования Microsoft Teams или PowerShell.
4. Просматривайте свои приветствия, если изменились параметры меню.
5. Настройте передачу в группы ответа с помощью решения "передача автосекретаря абоненту PSTN" в разделе [Известные проблемы](#known-issues) этой статьи.  
6. Протестируйте новые автоматические ассистенты, вызывая их внутренним образом или назначайте тестовый номер телефона.  

### <a name="cutover"></a>Прямая

1. Переключить номера телефонов с автосекретарей Exchange УМО на новые автосекретарей.
2. Переместите URI SIP из объекта Contact в учетную запись ресурса.
3. Протестируйте и проверяйте автоматические ассистенты, используя новые назначенные номера телефонов.

### <a name="suggested-test-plan-and-post-migration-validation-for-admins"></a>Предлагаемый план тестирования и проверка после миграции для администраторов

Чтобы проверить, перенесены ли ваши пользователи в облачную голосовую почту, оставьте голосовую почту для пользователя и проверьте текст сообщения в Outlook. В облачных сообщениях голосовой почты есть нижний колонтитул, который считывает:

"Благодарим за использование транскрипций! Если вы не видите запись, описанную в этой статье, это связано с тем, что для транскрибировала не было достаточно ясно качества звука. "

При тестировании функций голосовой почты после переноса пользователей необходимо учесть следующие сценарии:

- Проверка доступа к голосовой почте для всех типов конечных точек в Организации, таких как приложения и IP-телефоны.
- Проверка с использованием образцов пользователей, которые воспроизводят настроенные приветствия для абонентов.
- Если у вашей организации есть требования к юридическим требованиям или соответствия требованиям для отключения записи для пользователей, убедитесь, что она отключена после миграции. Более подробную информацию можно узнать в статье [Настройка облачной голосовой почты](/microsoftteams/set-up-phone-system-voicemail).
- Если вы уже настроили политики и правила виртуальной машины Exchange, убедитесь, что они эффективны.
- Ознакомьтесь с командлетами PowerShell облачной службы голосовой почты для изменения параметров пользователя.

### <a name="known-issues"></a>Известные проблемы

#### <a name="greeting-inconsistencies"></a>Несоответствия приветствия

Абонентский доступ может продолжать работать для клиента до тех пор, пока служба не будет полностью прекращена, даже после того, как все ваши пользователи будут перенесены в облачную голосовую почту. Чтобы избежать путаницы и несогласованности взаимодействия с пользователем, отключите абонентский доступ, так как после миграции приветствие будет изменено. Для этого удалите контакт EXUM для каждой линии абонентского доступа с помощью `Get-CsExUmContact | ?{$_.IsSubscriberAccess -eq $true} | Remove-CsExUmContact` .

#### <a name="auto-attendant-call-transfer-to-pstn"></a>Передача звонка абоненту PSTN

Чтобы передать вызов автосекретаря на внешний телефонный номер PSTN через Skype для бизнеса Server или службу группы ответа в Skype для бизнеса Server, создайте нового локального пользователя с переадресацией звонков на номер телефона PSTN или номер телефона для RGS. Пользователь должен быть включен и правильно настроен для корпоративной голосовой связи и иметь назначенную политику голосовой связи.

#### <a name="shared-mailbox-is-still-accessible"></a>Общий почтовый ящик по-прежнему доступен

Общий почтовый ящик, настроенный с использованием единой системы обмена сообщениями Exchange Online, продолжает получать сообщения после миграции в КВМ и станет доступным для пользователей через Outlook. Тем не менее, доступ к изменению сообщений приветствия этих почтовых ящиков будет недоступен после переноса в КВМ. Клиенты с общими почтовыми ящиками, которые используются для захвата абонентов автосекретаря, должны использовать все возможности автосекретарей и очередей звонков, которые один раз выпустили (эта дата октября 2019).
  
#### <a name="username-is-not-using-skype-for-business-banner-displays"></a>Отображается баннер "имя пользователя не используется в Skype для бизнеса"

Служба КВМ основана на инфраструктуре Microsoft Teams, а вызовы из клиента Skype для бизнеса могут привести к отображению баннера информации на клиенте, который читает: "username не использует Skype для бизнеса. Для более подробного взаимодействия переключитесь на Teams или начните собрание Skype. "
Обязательно обновите клиент Skype для бизнеса пользователей до последней версии обновления клиента C2R, чтобы отключить отображение этого баннера.
  
#### <a name="set-up-voice-mail-takes-you-to-owa"></a>"Настройка голосовой почты" позволяет переходить на OWA

Если вы нажимаете кнопку **Настройка голосовой почты** для клиента, пользователи Skype для бизнеса Server 2015/2013 смогут перейти на страницу портала Office Web Access (OWA) после миграции в КВМ. Все параметры удалены с вкладки Голосовая почта в OWA, а баннер будет отображаться с ссылкой перенаправление для получения пользователей на портал параметров пользователя КВМ.

#### <a name="changing-greeting-remotely"></a>Удаленное изменение приветствия

Абонентский доступ через PSTN не поддерживается в КВМ. Для пользователей, которым требуется удаленное изменение приветствия, в службу голосовой почты IVR для мобильных клиентов добавлен параметр меню "изменить приветствие". Пользователи могут вызывать эту службу, нажав и удерживая клавишу "1" в наборе для подключения к мобильному клиенту.
