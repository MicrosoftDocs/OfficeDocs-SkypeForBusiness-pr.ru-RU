# <a name="contoso-case-study-phone-system-for-a-multi-national-corporation"></a>Пример использования Contoso: телефонная система для мульти-национальной корпорации

В зависимости от географического расположения и других факторов у компании Contoso были офисы, использующие следующие решения телефонии:

- Тип сайта A: Skype для бизнеса Корпоративная голосовая связь

- Тип сайта B: традиционные устаревшие системы телефонии

- Тип сайта C: сочетание Skype для бизнеса Корпоративная голосовая связь и традиционных устаревших телефонных систем


Чтобы реализовать системное решение Телефон (Майкрософт) для всей организации, компании Contoso&mdash;&mdash;пришлось определить для каждого типа сайта, какой из следующих вариантов будет использоваться с телефонная система для подключения к телефонной сети общего пользования (ТСОП):

- телефонная система с тарифным планом 

- телефонная система с собственным оператором ТСОП через прямую маршрутизацию 

- Сочетание телефонная система с тарифным планом и телефонная система с собственным оператором ТСОП через прямую маршрутизацию
 
Чтобы определить правильное решение для своей организации, компания Contoso использовала функцию планирования решения Teams [голосовой](/microsoftteams/cloud-voice-landing-page) связи и сеанса Ignite 2019 [в](https://techcommunity.microsoft.com/t5/microsoft-teams-blog/what-s-new-for-calling-in-microsoft-teams-ignite-2019-edition/ba-p/974935) Microsoft Teams.  

## <a name="site-type-a-skype-for-business-enterprise-voice"></a>Тип сайта A: Skype для бизнеса Корпоративная голосовая связь 

Contoso Skype для бизнеса Корпоративная голосовая связь в качестве концентратора и периферийной зоны. В центральном расположении шлюз ТСОП в регионе, который предоставил подключение к ТСОП для Skype для бизнеса Корпоративная голосовая связь пользователей в стране. Часто эти вспомогательные офисы не имеют собственного исходящего трафика Из Интернета. Номера этих пользователей, размещенных в магистрали SIP, подключаемой к существующему SBC. 

Чтобы определить, сертифицирован ли уже развернутый SBC для прямой маршрутизации и обхода сервера-посредника, компания Contoso проверила список пограничных контроллеров сеансов, сертифицированных для прямой [маршрутизации](direct-routing-border-controllers.md).  

Пользовательские привычек набора номера были для набора пользователя в устаревшей телефонной системе с помощью расширения, даже если у пользователя есть клиент Skype для бизнеса, доступный для однорангового звука. 

Решение компании Contoso основано на следующих вопросах:

- Q. Нужно ли сохранять функции, предоставляемые локальным развертыванием?<br>
  A. Нет 

- Q. Нужно ли взаимодействовать со сторонними системами УАТС и другим телефонным оборудованием?<br>
  A. Нет 

- Q. Нужно ли сохранить текущего стороннего оператора?<br> A. Да (регулируемые страны) и Нет 

- Q. Нужно ли развертывать рентабельность инвестиций на контроллерах SBC?<br> A. Да и нет  

- Q. Доступны ли в этом регионе планы звонков по ТСОП Майкрософт?<br> A. Да и нет 

На основе ответов на свои вопросы компания Contoso решила:

- Переместите пользователей, расположенных в регионе, в котором планы звонков по ТСОП доступны для телефонная система с тарифными планами. 

- Переместите пользователей, которые не находятся в регионе, где доступны планы звонков по ТСОП, пользователей, расположенных на сайте, где рентабельность инвестиций в SBC еще не выполнены, а пользователей, которые находятся в стране, в которой есть правила телефонии, телефонная система с прямой маршрутизаторией. 

На следующей схеме показано Skype для бизнеса Корпоративная голосовая связь развертывания и способ переноса этого развертывания в планы звонков Майкрософт и прямую маршрутизацию:

## <a name="site-type-b-traditional-legacy-telephony-systems"></a>Тип сайта B: традиционные устаревшие системы телефонии

У компании Contoso было много офисов, в которых использовались устаревшие системы телефонии. Существует подмножество пользователей с номером телефона E1.64, а у других было только расширение. Эти номера находились в магистрали TDM для шлюза ТСОП. Набор номера внутри сайта был настроен с помощью кода сайта перед расширением, чтобы определить, куда направить вызов. Привычки пользователей к набору номера были для набора по расширению.   

Решение компании Contoso основано на следующих вопросах:

- Q. Нужно ли сохранять функции, предоставляемые локальным развертыванием?<br>
  A. Нет 

- Q. Нужно ли взаимодействовать со сторонними системами УАТС и другим телефонным оборудованием?<br> A. Да

- Q. Нужно ли сохранить текущего стороннего оператора?<br> A. Нет 

- Q. Доступен ли план звонков ТСОП Майкрософт в нашем регионе?<br> A. Да и нет 

На основе ответов на свои вопросы компания Contoso решила: 

- Переместите пользователей, расположенных в регионе, в котором планы звонков по ТСОП доступны для телефонная система с тарифными планами. 

- Переместите пользователей, которые не находятся в регионе, где планы звонков по ТСОП доступны для телефонная система прямой маршрутизации. 

- Поддерживать подключение по ТСОП к критически важным для бизнеса аналогам устройств.

На следующих схемах показано исходное устаревшее развертывание системы с удаленными сайтами и переход на развертывание прямой маршрутизации с оптимизацией локальных носителей:

## <a name="site-type-c-combination-of-skype-for-business-enterprise-voice-and-traditional-legacy-telephony-systems"></a>Тип сайта C: сочетание Skype для бизнеса Корпоративная голосовая связь и традиционных устаревших телефонных систем

Contoso Skype для бизнеса Корпоративная голосовая связь номера пользователей находятся в магистрали SIP для SBC от оператора. Номера традиционных телефонных систем, размещенных в магистрали TDM для шлюза ТСОП.   

Решение компании Contoso основано на следующих вопросах:

- Q. Нужно ли сохранять функции, предоставляемые локальным развертыванием?<br>
  A. Нет 

- Q. Нужно ли взаимодействовать со сторонними системами УАТС и другим телефонным оборудованием?<br> A. Нет 

- Q. Нужно ли сохранить текущего стороннего оператора?<br> A. Нет 

- Q. Нужно ли развертывать рентабельность инвестиций на контроллерах SBC?<br> A. Да и нет  

- Q. Доступен ли план звонков по ТСОП корпорации Майкрософт в этом регионе?<br> A. Нет 

Основываясь на ответах на свои вопросы, компания Contoso решила следующее: 

- Для устаревших пользователей телефонии, которым будет разрешена прямая маршрутизация, компания Contoso перенаправлена с магистрали TDM в магистраль SIP для SBC, так как SBC сертифицирован для прямой маршрутизации. 

- Для поддержки подмножества пользователей, переехав на телефонная система и разрешив продолжение маршрутизации по устаревшей системе, устаревшая телефонная система была настроена в качестве следующего прыжка в SBC.   

- Кроме того, чтобы порекомендовать пользователю изменить поведение и удалить зависимость от набора расширений между сайтами и внутри сайта, компания Contoso предоставила руководство по использованию Teams для всех внутренних вызовов.

## <a name="configuration"></a>Конфигурация

### <a name="calling-plans-sites"></a>Сайты планов звонков

Чтобы получить лицензии и назначить телефонные номера пользователям, компания Contoso выполнила действия, описанные в разделе ["Настройка планов звонков"](set-up-calling-plans.md). 

Из-за количества пользователей, которым необходимо назначить номера телефонов, компания Contoso решила назначить номера телефонов с помощью PowerShell. Чтобы узнать, как назначать числа с помощью PowerShell&mdash;&mdash;в дополнение к другим параметрам, компания Contoso [использовала Teams PowerShell](teams-powershell-overview.md).  

### <a name="direct-routing-sites"></a>Сайты прямой маршрутизации

Чтобы подключить локальную инфраструктуру телефонии Contoso к Microsoft Teams, администратор компании Contoso выполнил действия, описанные в разделе "Настройка прямой [](direct-routing-configure.md) маршрутизации", и просмотрел видеонастройку прямой маршрутизации в Microsoft Teams.[](https://www.youtube.com/watch?v=1ASftX_Msb8&index=10&list=PLaSOUojkSiGnKuE30ckcjnDVkMNqDv0Vl)  Компания Contoso также ссылается на документацию по развертыванию прямой маршрутизации от сертифицированного поставщика SBC. 

После настройки прямой маршрутизации между SBC и Телефон (Майкрософт) System компании Contoso необходимо было протестировать конфигурацию. Для этого администраторы Contoso использовали клиент тестатора SIP, который был рассмотрен в разделе "Обновления для сеанса прямой маршрутизации на [Ignite 2019"](https://techcommunity.microsoft.com/t5/microsoft-teams-events-blog/ignite-live-blog-session-vce20-updates-for-direct-routing/ba-p/1025138). Клиентский сценарий тестатора SIP и документация были загружены из скрипта PowerShell для тестирования подключений пограничного контроллера сеанса прямой маршрутизации.   


### <a name="local-media-optimization"></a>Оптимизация локальных носителей

Компания Contoso узнала о возможности использовать оптимизацию локальных носителей в разных регионах по всему миру. Поддерживаемые сценарии для Contoso описаны в статье "Оптимизация локальных носителей [для прямой маршрутизации"](direct-routing-media-optimization.md). Настройка оптимизации локального носителя была завершена с помощью рекомендаций поставщика SBC и Корпорации Майкрософт. Действия по настройке для оптимизации локальных носителей включают в себя: 

- Настройка пользователей и сайтов SBC 

- Настройте SBC в соответствии со спецификацией поставщика SBC. 

- Добавление внешних доверенных IP-адресов на каждый сайт, используемый для оптимизации локального носителя    

- Определение топологии сети 

- Определение топологии виртуальной сети 

- Определение режима: Всегда обходить или только для локальных пользователей 

## <a name="networking-considerations"></a>Рекомендации по работе с сетями

У компании Contoso было несколько пользователей, которым необходимо было работать удаленно в течение продолжительного периода времени после включения телефонная система. Пользователи использовали VPN для доступа к определенным бизнес-приложениям. Во время работы с VPN телефонная система качество звонков у пользователей было ниже. 

Чтобы устранить проблему с качеством, компания Contoso реализовала раздельное туннелирование VPN, которое Office 365 трафик через Интернет, пока подключение к внутренним приложениям оставалось на VPN. Чтобы реализовать туннелирование с разделением VPN, компания Contoso выполнила инструкции по реализации туннелирования с разделением [VPN для](/office365/enterprise/office-365-vpn-implement-split-tunnel) Office 365.
