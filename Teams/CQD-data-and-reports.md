# <a name="data-and-reports-in-call-quality-dashboard-cqd"></a>Данные и отчеты на панели мониторинга качества звонков (CQD)

Панель мониторинга качества звонков (Майкрософт) (CQD) использует канал данных в ближайшем режиме реального времени (NRT). Записи звонков доступны в CQD в течение 30 минут после окончания звонка. Записи звонков из конвейера NRT доступны только в течение нескольких месяцев, прежде чем они будут удалены из набора данных. 


> [!NOTE]
> Расширенная CQD (Новая версия в ноябре 2019) объединяет данные из устаревшего CQD конвейера (который сделал записи о звонках примерно 24 часа) с NRT данными из расширенного конвейера CQD. Запросы на более ранних и сложных порталах для данных из архивного периода выдают одинаковые результаты. Запросы на портале для данных NRT и NRT данных и EUII периодов будут разными.

## <a name="import-the-cqd-report-templates"></a>Импорт шаблонов отчетов CQD

Скачайте [два CQDных шаблонных отчета](https://aka.ms/qertemplates) (все сети и управляемые сети), которые помогут вам быстро приступить к работе с CQD. Шаблон "все сети", оптимизированный для работы с файлом данных здания, можно использовать при сборе и загрузке данных в CQD, как описано в следующем разделе.

**Импорт шаблонов (. CQDX) в CQD**

1. В CQD выберите **подробные отчеты** в меню в верхней части страницы.

2. На левой панели нажмите кнопку **Импорт**. Перейдите к первому шаблону CQDX и нажмите кнопку **Открыть**.

3. После отправки шаблона во всплывающем окне будет отображено сообщение "Импорт отчета успешно завершен". 

4. Повторите действия 2 и 3 для второго шаблона CQD.

## <a name="euii-data"></a>Данные EUII

В целях соответствия требованиям конечные пользователи (EUII) данные (также известные как конфиденциальные сведения или PII) хранятся только в течение 30 дней. Так как данные NRT пересекается с 30-дневной меткой, поля, содержащие EUII, очищаются, что приводит к EUII бесплатных NRT данных. Поля, содержащие данные EUII:

- Полный IP-адрес
- MAC-адрес
- Идентификатор базового Service Set (BSSID)
- Универсальный код ресурса (SIP) для протокола запуска сеанса (только для Skype для бизнеса)
- Имя участника-пользователя (UPN)
- Имя конечной точки компьютера
- Отзывы пользователей
- Идентификатор объекта (ИД объекта службы каталогов Active Directory для пользователя конечной точки)

### <a name="admin-roles-with-and-without-euii-access"></a>Роли администратора с EUII Access и без него

Эти [RBAC](https://docs.microsoft.com/azure/role-based-access-control/overview) роли RBAC **DO** имеют доступ EUII.
- Глобальный администратор
- Администратор службы Teams
- Администратор связи Teams
- Инженер поддержки по коммуникациям Teams
- Глобальный читатель
- Администратор Skype для бизнеса

**У этих РОЛЕЙ RBAC** нет доступа EUII:
- Средство чтения отчетов
- Специалист по поддержке взаимодействия в Teams

### <a name="server-client-tab-and-client-client-tabs"></a>Вкладка "сервер-клиент" и "клиент-клиент"

Эти две вкладки содержат сведения о потоках, которые выполнялись в сценариях конечной точки. Вкладка "сервер-клиент" содержит четыре свертываемых раздела, которые представляют собой четыре сценария, в которых потоки мультимедиа будут перетекать.
  
- Проводные внутренние
- Проводные внешние
- Wi-Fi Inside
- Wi-Fi за пределами

Аналогичным образом на вкладке Клиент-клиент выводятся пять свертываемых разделов.

- Проводная внутренняя — проводные внутренние
- Проводные внутри — проводные внешние
- Проводные внешние — проводная внешняя
- Проводная внутренняя — WiFi Inside
- Проводная внутренняя — WiFi снаружи

#### <a name="inside-versus-outside"></a>Внутри и снаружи

CQD классифицирует поток как *внутри* , так и *за пределами* , используя построение данных, если оно существует. Конечные точки каждого потока связаны с адресом подсети. Если подсеть находится в списке подсетей, помеченных InsideCorp в загруженных сведениях о сборке, они рассматриваются *внутри*. Если сведения о сборках еще не были отправлены, в рамках теста всегда классифицируются потоки *за пределами*. 

В сценарии "сервер-клиент" внутри теста есть только конечная точка клиента. Поскольку серверы всегда находятся за пределами точки зрения пользователя, это не учитывается при тестировании.
  
#### <a name="wired-versus-wifi"></a>Проводное и WiFi

Как видно из имен, критерии классификации основываются на типе клиентских подключений. Сервер всегда является проводным и не включается в вычисления. В данном потоке, если одна из двух конечных точек подключена к сети WiFi, CQD классифицирует ее как Wi-Fi.
> [!NOTE]
> Поток, если одна из двух конечных точек подключена к сети Wi-Fi, она классифицируется как Wi-Fi в CQD.

Если отчеты по умолчанию CQD не соответствуют вашим потребностям, воспользуйтесь этими инструкциями, чтобы создать настраиваемый отчет. Или (за январь 2020) [Используйте Power BI для отчетов CQD ](cqd-power-bi-query-templates.md).

Из раскрывающегося списка отчетов в верхней части экрана, отображаемого на \( странице " **Сводные отчеты** ", \) выберите **подробные отчеты** и нажмите кнопку **создать**. Нажмите кнопку **изменить** в отчете, чтобы открыть редактор запросов. В основе каждого отчета лежит запрос, обращенный к кубу. В отчете наглядно представлены данные, возвращенные в результате запроса. Редактор запросов поможет вам изменить эти запросы и параметры отображения отчета.
> [!IMPORTANT]
> Диапазон сети может использоваться для представления supernet (сочетание нескольких подсетей с одним префиксом маршрутизации). Для всех новых выгрузок построения будут проверяться все перекрывающиеся диапазоны. Если вы ранее загрузили файл здания, вы должны скачать текущий файл и повторно загрузить его, чтобы определить перекрытие и устранить проблему перед повторной отправкой. Перекрытие в ранее загруженных файлах может привести к неправильному сопоставлению подсетей для зданий в отчетах. Некоторые реализации виртуальных частных сетей не сообщат сведения о подсети. При добавлении подсети VPN в файл построения вместо одной записи для подсети рекомендуется добавлять отдельные записи для каждого адреса в подсети VPN в виде отдельной сети 32-бит. Каждая строка может иметь одинаковые стандартные метаданные. Например, вместо одной строки для 172.16.18.0/24 необходимо иметь 256 строк с одной строкой для каждого адреса между 172.16.18.0/32 и 172.16.18.255/32 включительно.
>
> VPN-столбец является необязательным и по умолчанию будет равен 0.  Если для столбца VPN задано значение 1, подсеть, представленная этой строкой, будет полностью развернута для соответствия всем IP-адресам в подсети.  Используй это экономно и только для подсетей VPN, так как полное расширение этих подсетей может отрицательно повлиять на время запроса запросов, связанных с созданием данных.

Наведите указатель на пункт линейчатые диаграммы и линии тренда в отчете для отображения подробных значений. В отчете, в котором установлен фокус, отображается меню "действия": **Редактирование**, **клонирование**, **Удаление**, **Загрузка**и **Экспорт дерева отчета**.



## <a name="query-filters"></a>Фильтры запросов

Фильтры запросов реализуются с помощью редактора запросов в CQD. Эти фильтры используются для уменьшения количества записей, возвращаемых функцией CQD, таким образом уменьшая общий размер отчета и значения его запроса. Это особенно полезно для фильтрации неуправляемых сетей. Фильтры, указанные в приведенной ниже таблице, используют регулярные выражения (RegEx).


| Filter.         | Описание          | Пример фильтра запроса CQD      |
|----------------|----------------------|-------------------------------|

Для ручного отбора пустых значений используйте пустое выражение и установите для фильтра значение равно или не равно, в зависимости от потребностей.      | Второе название ( \<\> \^ \\ с)\*\$                       |
| Исключение распространенных подсетей | При отсутствии действительного файла здания для разделения управляемого кода из неуправляемых сетей в отчеты будут включены домашние сети. Эти домашние подсети находятся за пределами области элемента управления и могут быть быстро исключены из отчета. Наиболее распространенными подсетями, определенными в данном руководстве, являются 10.0.0.0, 192.168.1.0 и 192.168.0.0. | Вторая подсеть \<\> 10.0.0.0 \| 192.168.0.0, \| 192.168.1.0 |

### <a name="url-filters"></a>Фильтры URL-адресов

CQD поддерживает добавление фильтров к URL-адресу. Это упрощает предоставление общего доступа к CQD запросу и его закладку. В URL-адресе можно задать параметры, такие как месяц с определением тенденции, идентификатор клиента или язык. Вы также можете добавить фильтры уровня продукта или панели мониторинга к URL-адресу.
Исключение федеративных данных из отчетов CQD полезно, если вы исправление управляемые строения или сети, где Федеративные конечные точки могут повлиять на отчеты.

Чтобы добавить фильтр, добавьте к концу URL-адреса следующее:

```
/filter/[AllStreams].[Second Tenant Id]\|[YOUR TENANT ID HERE]
```

Образом  

```https://cqd.teams.microsoft.com/cqd/#/1234567/2018-08/filter/[AllStreams].[Second Tenant Id]|[TENANTID]```

Чтобы добавить фильтр уровня панели мониторинга к URL-адресу, этот фильтр должен существовать в CQD как фильтр на уровне продукта или панели мониторинга. Добавьте эти фильтры к URL-адресу после месяца изменения и до параметров URL-адреса.

```filter/DATA_MODEL_NAME|VALUE```

Например, чтобы применить значение фильтра продукта Microsoft Teams, необходимо добавить следующее:

```filter/[AllStreams].[Is%20Teams]|[True]```

Весь URL-адрес выглядит примерно так:

```https://cqd.teams.microsoft.com/spd/#/Dashboard/2624085/2018-9/filter/[AllStreams].[Is%20Teams]|[True]```

Чтобы применить фильтры URL-адресов с несколькими вариантами выбора, разделите каждое значение на символ канала (|). Например:

```filter/[AllStreams].[Media%20Type]|[Video]|[Audio]|[VBSS]```

Если вы указали неправильное имя или значение, фильтр URL-адреса не будет применен.


Вы можете использовать фильтр URL-адресов, чтобы отфильтровать все отчеты для определенного измерения. Наиболее распространенные фильтры URL-адресов используются для фильтрации отчетов, чтобы исключить телеметрии федеративных участников, или сосредоточиться только на Teams или Skype для бизнеса Online. Исключение федеративных данных из отчетов CQD полезно, если вы исправление управляемые строения или сети, где Федеративные конечные точки могут повлиять на отчеты.

Для ручного отбора пустых значений используйте пустое выражение и установите для фильтра значение равно или не равно, в зависимости от потребностей.      | Второе название ( \<\> \^ \\ с)\*\$                       |

## <a name="comparing-teams-and-skype-for-business-cqd-data"></a>Сравнение данных групп и Skype для бизнеса CQD

Даже в последней версии CQD (cqd.teams.microsoft.com) вы увидите различия данных между Teams и Skype для бизнеса. Некоторые причины:
- Различия в механизмах обеспечения производительности и надежности
  - В Teams предусмотрено автоматическое повторное подключение и быстрое перемещение. Skype для бизнеса не работает.
  - Управление динамическим пропускной способностью осуществляется в Teams. Skype для бизнеса не работает.
- Различия в [диапазонах IP-адресов](Office-365-URLs-IP-address-ranges.md) между группами и Skype для бизнеса. Диапазоны IP-адресов групп новее, что может привести к проблемам с подключением в брандмауэре.
