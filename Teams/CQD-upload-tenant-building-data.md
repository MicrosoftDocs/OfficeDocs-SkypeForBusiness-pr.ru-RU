# <a name="upload-tenant-and-building-data-in-call-quality-dashboard-cqd"></a>Upload и здания на панели мониторинга качества звонка


Чтобы использовать все возможное, мы рекомендуем добавить данные клиента и здания. Существует два типа файлов данных клиента: ["Здание"](#upload-building-data-file) и ["Конечная точка".](#endpoint-data-file)

Пример шаблона данных клиента можно [скачать](https://github.com/MicrosoftDocs/OfficeDocs-SkypeForBusiness/blob/live/Teams/downloads/locations-template.zip?raw=true)здесь. Справку по сопоставлению здания можно найти в документе Создание карты здания [для CQD.](CQD-building-mapping.md)

На панели мониторинга CQD Summary Reports (Сводные отчеты CQD) выберите пункт **Tenant Data Upload (Tenant Data Upload)** в меню CQD **Параметры** (значок шестеренки в верхней части панели мониторинга CQD). Здесь администраторы могут загружать сведения о здании и конечной точке своей организации, такие как сопоставление IP-адресов и географических данных, сопоставление каждой беспроводной точки доступа и ее mac-адреса и т. д.

1. Откройте CQD (в центре администрирования Teams или в ), затем выберите значок шестеренки в правом верхнем углу и выберите данные клиента Upload на странице [https://cqd.teams.microsoft.com](https://cqd.teams.microsoft.com) **Сводные**  отчеты.

   ![Снимок экрана: диалоговое окно, которое появляется во время отправки данных.](media/qerguide-image-tenantdataupload.png)
    
2. Кроме того, если вы впервые посещаете CQD, вам будет предложено добавить данные о здании. Вы можете выбрать **Upload,** чтобы быстро перейти на страницу **Данные Upload** клиента.

   ![Снимок экрана: баннер, на который пользователь может отправить данные о здании.](media/qerguide-image-buildingdatauploadbanner.png)

3. На странице **Tenant Data Upload** , чтобы выбрать файл данных, выберите обзор. 

4. После выбора файла данных укажите дату **начала** и при желании укажите дату окончания.

5. После выбора **даты начала** выберите **Upload,** чтобы отправить файл в CQD. <br><br>Перед отправкой файл проверяется. В случае с ошибки отображается сообщение об ошибке с запросом на исправление файла. На рисунке ниже показана ошибка, которая появляется, если в файле данных неверное количество столбцов.

   ![Пример диалогового окна с сообщением об ошибке отправки данных о здании.](media/qerguide-image-buildingdatauploaderror.png)
 
6. Если во время проверки не возникают ошибки, отправка файла будет успешной. После этого вы увидите загруженный  файл данных в таблице Мои загрузки, в которой внизу этой страницы показан полный список всех загруженных файлов для текущего клиента.

> [!NOTE]
> Обработка файла здания может занять до четырех часов. <br><br> Если вы уже добавили файл здания и вам нужно добавить в него подсети, которые могли быть пропущены или исключены, измените исходный файл, добавив новые подсети, удалите текущий файл и повторно загрузите измененный файл. В CQD может быть только один активный файл данных о здании. 


## <a name="upload-building-data-file"></a>Upload создания файла данных

Первым типом файла данных клиента в CQD является **файл данных "Здание".** Столбец Подсеть расширяется путем расширения столбца Network+NetworkRange, а затем соединения столбца Subnet со столбцом First Subnet или Second Subnet записи вызова, чтобы отфиксировать сведения о здании, городе, стране или регионе. Формат отложенного файла данных должен соответствовать следующим условиям для проверки перед отправкой:
  
- Файл должен быть TSV-файлом (столбцы разделены tab) или файлом .csv (столбцы разделены запятой).

- Файл данных не содержит строку заглавной таблицы. Ожидается, что первая строка файла данных будет иметь реальные данные, а не подписи заглавных кегов, такие как "Сеть".

- В файле могут быть только типы данных String, Integer или Boolean. Для типа данных Integer значение должно быть числом. Boolean values must be either 0 or 1.

- Если в столбце используется тип данных String, поле данных может быть пустым, но по-прежнему должно быть разделено табуской или запятой. Пустое поле данных просто назначает пустое значение строки.

- Для каждого файла данных клиента существует ограничение в 1000 000 строк.

- Для каждой строки должно быть по 15 столбцов, каждый столбец должен иметь соответствующий тип данных, а столбцы должны быть указаны в таблице ниже (с запятой или табуля стороной).

  **Формат файла создания данных**
  
  | Имя столбца        | Тип данных | Пример                   | Рекомендации              |
  |--------------------|-----------|---------------------------|-----------------------|
  | NetworkIP          | String    | 192.168.1.0               | Обязательно              |
  | NetworkName        | String    | США/Сиэтл/СИЭТЛ-СИ-1 | Обязательно<sup>1</sup>  |
  | NetworkRange       | Число    | 26                        | Обязательно              |
  | BuildingName       | String    | СИЭТЛ-СИ-1             | Обязательно<sup>1</sup>  |
  | OwnershipType      | String    | Contoso                   | Необязательно               |
  | BuildingType       | String    | Завершение работы ИТ-20            | Необязательно               |
  | BuildingOfficeType | String    | Инженерных               | Необязательно               |
  | City               | String    | Сиэтл                   | Рекомендуется           |
  | ZipCode            | String    | 98001                     | Рекомендуется           |
  | Страны            | String    | НАМ                        | Рекомендуется           |
  | State              | String    | Красноярский край                        | Рекомендуется           |
  | Region             | String    | MSUS                      | Рекомендуется           |
  | InsideCorp<sup>2</sup>         | Bool      | 1             | Обязательно              |
  | ExpressRoute<sup>3</sup>       | Bool      | 0             | Обязательно              |
  | VPN;                | Bool      | 0                         | Необязательно               |

  <sup>1</sup> Хотя это не обязательно, шаблоны настроены для отображения названия здания и сети.

  <sup>2</sup> Этот параметр может использоваться для того, чтобы определить, находится ли подсеть в корпоративной сети. Вы можете настроить использование для других целей.

  <sup>3</sup> Этот параметр может использоваться для того, чтобы определить, используется ли в сети Azure ExpressRoute. Вы можете настроить использование для других целей.  

  **Пример строки:**

  `192.168.1.0,USA/Seattle/SEATTLE-SEA-1,26,SEATTLE-SEA-1,Contoso,IT Termination,Engineering,Seattle,98001,US,WA,MSUS,1,0,0`

> [!IMPORTANT]
> Диапазон сети можно использовать для представления суперсети (сочетания нескольких подсетей с одним префиксом маршрутизации). Все добавленные в новое здание будут проверяться на наличие перекрывающихся диапазонов. Если вы ранее уже загрузили файл здания, скачайте текущий файл и повторно загрузите его, чтобы выявить перекрытия и устранить проблему перед повторной отправкой. Любое перекрытие ранее добавленных файлов может привести к неправильному сопоставлению подсетей со зданиями в отчетах. Некоторые реализация VPN не сообщают точные сведения о подсети. 
>
> Столбец VPN необязателен и по умолчанию имеет значение 0. Если для столбца VPN за установлено значение 1, подсеть, представленная этой строкой, будет полностью расширита в соответствие со всеми IP-адресами в подсети. Используйте этот способ не только для подсетей VPN, так как полное расширение этих подсетей отрицательно скажется на времени запроса для запросов, связанных со созданием данных. Если расширение подсети приводит к превышению ограничения строки расширения в 1000 000, файл здания не будет принят.


### <a name="supernetting"></a>Суперсети

Для определения каждой подсети можно использовать надсети, часто называемые Inter-Domain без классной маршрутизации (CIDR). *Суперсеть —* это сочетание нескольких подсетей с одним префиксом маршрутизации. Вместо добавления записи для каждой подсети можно использовать адрес с надсетью. Надсети поддерживаются, но использовать их не рекомендуется.

Например, маркетинговое здание Contoso состоит из подсетей ниже:

-   10.1.0.0/24 — первый этаж
-   10.1.1.0/24 , второй этаж
-   10.1.2.0/24, третий этаж
-   10.1.3.0/24 — четвертый этаж

Вместо добавления записи для каждой подсети можно использовать суперсетный адрес (в данном примере — 10.1.0.0/22).

-   Network = 10.1.0.0
-   Диапазон сети = 22

Прежде чем внедрять надсети, учитывайте некоторые моменты.

-   Суперсети можно использовать только в сопоставлении подсети с 8-битной и 28-битной маской.

-   Для работы надсети требуется меньше времени, но это стоит того, чтобы снизить насыщенность данных. Предположим, что возникла проблема с качеством подсети 10.1.2.0. Если вы реализовали суперсети, вы не знаете, где находится подсеть и какой тип сети она (например, лабораторный). Если вы определили все подсети для здания и добавили сведения о расположении этажа, вы сможете увидеть это различие.

-   Важно убедиться, что адрес с надсетью правильный и не позволяет найти ненужные подсетей.

-   В 192.168.0.0 часто можно найти данные 192.168.0.0. Для многих организаций это означает, что пользователь находится дома. Для других людей это схема IP-адресов для офисов. Если в вашей организации есть офисы, использующие эту конфигурацию, не включите ее в файл здания, так как трудно различить домашние и внутренние сети с помощью общих [подсетей.](quality-of-experience-review-guide.md#common-subnets) 

> [!IMPORTANT]
> Диапазон сети можно использовать для представления суперсети. Все новые файлы данных здания будут проверяться на наличие перекрывающихся диапазонов. Если вы уже загрузили файл здания, скачайте текущий файл и загрузите его еще раз, чтобы выявить перекрытия и устранить проблему. Любое перекрытие ранее добавленных файлов может привести к неправильному сопоставлению подсетей со зданиями в отчетах.

### <a name="vpn"></a>VPN;

Данные качества качества обслуживания (QoE), отправляемые клиентами в Microsoft 365 или Office 365 , из которых были источником данных CQD, включают VPN-флаг. В CQD этот параметр будет видеться в качестве первого VPN и второго VPN-измерений. Однако этот флажок зависит от отчетности поставщиков VPN Windows том, что зарегистрированный адаптер сети VPN является адаптером удаленного доступа. Не все поставщики VPN правильно регистрируют адаптеры удаленного доступа. Из-за этого, возможно, вы не сможете использовать встроенные фильтры ЗАПРОСОВ VPN. Используйте столбец VPN, рассмотренный выше, для точной пометки и определения подсетей VPN. Кроме того, мы ехимем пометить VPN-сети для удобной идентификации в отчетах. Ниже приведены два примера того, как пометить подсети VPN.

- Определите **имя сети,** введите в этом поле "VPN" для подсетей VPN.

  ![Снимок экрана отчета QCD, на котором показывая VPN с использованием имени сети.](media/qerguide-image-vpnnetworkname.png)

- Определите **имя здания,** введите в этом поле "VPN" для подсетей VPN.

  ![Снимок экрана отчета QCD, на котором показывая VPN с названием здания.](media/qerguide-image-vpnbuildingname.png)

> [!NOTE]
> Как известно, VPN-подключения неправильно интерентизируют тип сетевого подключения как проводные при беспроводном подмосковии. При оценке качества по VPN-подключениям нельзя предположить, что тип подключения точно определен.

## <a name="endpoint-data-file"></a>Файл данных конечной точки

Другой тип файла данных клиента CQD — это файл данных **конечной** точки. Значения столбцов используются в столбце Имя первой клиентской конечной точки или Имя конечной точки второго клиента для показа сведений о конечной точке (Make, Model) или Type (Тип). Формат отложенного файла данных должен соответствовать следующим условиям для проверки перед отправкой:

- Файл должен быть TSV-файлом (столбцы разделены tab) или файлом .csv (столбцы разделены запятой).

- Содержимое файла данных не включает в себя заглавные таблицы. Ожидается, что первая строка файла данных будет иметь реальные данные, а не заглавную метку, например "Имя Конечной точки".

- Во всех семи столбцах используется только тип данных String. Максимальная допустимая длина составляет 64 знака.

- Поле данных может быть пустым, но по-прежнему должно быть разделено табулами или запятыми. Пустое поле данных просто назначает пустое значение строки.

- Имя конечной точки должно быть уникальным, в противном случае отправка завершится сбой. Если в одной или двух строках используется одно и то же имя конечной точки, конфликт приведет к неправильному объединению.

- Метки EndpointLabel1, EndpointLabel2 и EndpointLabel3 можно настраивать. Они могут быть пустыми строками или значениями, такими как "Ноутбук, назначенный ИТ-отделом 2018" или "Тег актива 5678".

- Для каждой строки должно быть семь столбцов, а столбцы должны быть в следующем порядке:

  **Порядок полей:**

  EndpointName, EndpointMake, EndpointModel, EndpointType, EndpointLabel1, EndpointLabel2, EndpointLabel3

  **Пример строки:**

  `1409W3534, Fabrikam, Model 123, Laptop, IT designated 2018 Laptop, Asset Tag 5678, Purchase 2018`

## <a name="update-a-building-file"></a>Обновление файла здания

При сборе сведений о здании и подсети администраторы со временем часто загружают файл здания в несколько итерации, добавляя новые подсети и сведения о здании по мере их получения. В этом случае вам потребуется повторно отправить файл здания. Этот процесс похож на первую отправку, как описано в предыдущем разделе, с несколькими исключениями, описанными в следующем разделе.

> [!Important]
> Одновременно может быть активен только один файл. Несколько файлов здания не являются накопительными.

## <a name="add-net-new-subnets"></a>Добавление новых подсетей

Иногда необходимо добавить в CQD новые сетевые подсети, которые изначально не были частью топологии сети. Чтобы добавить новые подсети, на странице Tenant **Data Upload** CQD сделайте следующее:

1.  Если у вас еще нет нужной копии, скачайте исходный файл.

1.  Удалите текущий файл в CQD.

1.  Отредактируем исходный файл здания и устраив дату окончания, которая произойдет по крайней мере за день до приобретения новых сетей.

1.  Применим новые подсети к исходному файлу здания.

1.  Upload измененный файл здания и установите дату начала на один день после окончания предыдущего файла.

## <a name="add-missing-subnets"></a>Добавление отсутствующих подсетей

После добавления сведений о здании для управляемых сетей каждая из управляемых сетей должна иметь связь между зданиями. Однако это не всегда так. как правило, пропускается несколько подсетей. Чтобы найти эти отсутствующие сети, просмотрите отчет о отсутствующих **подсетях** на странице **"Отчеты** о качестве работы" в CQD. В результате будут представлены все подсетей с 10 или более аудиопотоками, которые не определены в файле данных здания и помечены как внешние. Убедитесь, что в этом списке нет управляемых сетей. Если подсети отсутствуют, обновите исходный файл данных здания и повторно загрузите его в CQD с помощью следующей процедуры:

1. Перейдите на **страницу "Данные Upload** клиента" в CQD.

1. Если у вас еще нет нужной копии, скачайте исходный файл.

1. Удалите текущий файл в CQD.

1. Применив новые подсети к исходному файлу.

1. Upload файла здания. Не забудьте установить дату начала как минимум на восемь месяцев раньше, чтобы CQD обработать исторические данные.


> [!IMPORTANT]
> Вам потребуется добавить свой ИД клиента в  отчет в качестве фильтра запросов для второго клиента, чтобы отфильтровать отчет, чтобы просмотреть только данные клиента организации. В противном случае в отчете покажутся федераированные подсети.

> [!NOTE] 
> Не забудьте настроить фильтр отчета "Месяц год" с текущим месяцем. Выберите **Изменить** и настройте фильтр отчета **"Месяц год",** чтобы сохранить новый месяц по умолчанию.
