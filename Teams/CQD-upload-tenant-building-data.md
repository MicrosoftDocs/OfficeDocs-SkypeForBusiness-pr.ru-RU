# <a name="upload-tenant-and-building-data-in-call-quality-dashboard-cqd"></a>Отправка данных о клиенте и здании на панели мониторинга качества звонка (CQD)


Чтобы использовать панель мониторинга качества звонка (CQD), мы рекомендуем добавить данные клиента и здания. Существует два типа файлов данных клиента: ["Здание"](#upload-building-data-file) и "Конечная [точка".](#endpoint-data-file)

Вы можете скачать образец шаблона данных [клиента здесь.](https://github.com/MicrosoftDocs/OfficeDocs-SkypeForBusiness/blob/live/Teams/downloads/locations-template.zip?raw=true) Справку по сопоставлению здания можно найти в документе [Create a building map for CQD (Создание карты здания для CQD).](CQD-building-mapping.md)

На панели мониторинга CQD Summary  Reports (Сводные отчеты CQD) выберите "Отправка данных клиента" в меню "Параметры CQD" (значок шестеренки в верхней части панели мониторинга CQD).  Здесь администраторы могут загружать сведения о здании и конечной точке своей организации, такие как сопоставление IP-адресов и географических данных, сопоставление каждой беспроводной точки доступа и ее mac-адреса и т. д.

1. Откройте CQD (из Центра администрирования Teams или по крайней мере), затем выберите значок шестеренки в правом верхнем углу и выберите "Отправить данные клиента" на странице [https://cqd.teams.microsoft.com](https://cqd.teams.microsoft.com) **"Сводные** отчеты". 

   ![Снимок экрана: диалоговое окно, которое появляется во время отправки данных](media/qerguide-image-tenantdataupload.png)
    
2. Кроме того, если вы впервые посещаете CQD, вам будет предложено отправить данные о здании. Чтобы быстро перейти **на** страницу "Отправка данных клиента", выберите "Отправить **сейчас".**

   ![Снимок экрана: баннер, на который пользователь может отправить данные о здании](media/qerguide-image-buildingdatauploadbanner.png)

3. На странице **"Отправка данных клиента"** выберите **"Обзор",** чтобы выбрать файл данных.

4. Выбрав файл данных, укажите дату **начала** и при желании укажите дату окончания.

5. После выбора **даты начала выберите** **"Отправить",** чтобы отправить файл в CQD. <br><br>Перед отправкой файл проверяется. Если проверка не будет правильной, отобразиться сообщение об ошибке с запросом на исправление файла. На рисунке ниже показана ошибка, которая появляется, если в файле данных неверное количество столбцов.

   ![Пример диалоговых окна с сообщением об ошибке отправки данных о здании](media/qerguide-image-buildingdatauploaderror.png)
 
6. Если во время проверки не произойдет ошибок, отправка файла будет успешной. После этого вы увидите загруженный  файл данных в таблице "Мои загрузки", в которой в нижней части этой страницы показан полный список всех загруженных файлов текущего клиента.

> [!NOTE]
> Обработка файла здания может занять до четырех часов. <br><br> Если вы уже загрузили файл здания и хотите добавить в него пропущенные или исключенные подсети, измените исходный файл, добавив новые подсети, удалите текущий файл и повторно добавьте измененный файл. В CQD может быть только один активный файл данных о здании. 


## <a name="upload-building-data-file"></a>Отправка файла данных о здании

Первым типом файла данных клиента в CQD является файл данных **о** здании. Столбец "Подсеть" строится путем расширения столбца Network+NetworkRange, а затем присоединения столбца "Подсеть" к столбце "Первая подсеть" или "Вторая подсеть" записи для показа сведений о здании, городе, стране или регионе записи звонка. Формат отложенного файла данных должен соответствовать следующим условиям для проверки перед отправкой:
  
- Файл должен быть либо в TSV-файле (столбцы разделены с помощью tab), либо в CSV-файле (столбцы разделены запятой).

- Файл данных не содержит строку заглавной строки таблицы. Ожидается, что первая строка файла данных будет иметь реальные данные, а не подписи заглавных строк, такие как "Сеть".

- В файле могут быть только типы данных "Строка", "Integer" или "Boolean". Для типа данных "Integer" значение должно быть числом. Boolean values must be either 0 or 1.

- Если в столбце используется тип данных "Строка", поле данных может быть пустым, но все равно должно быть разделено табулом или запятой. Пустому полю данных просто назначается пустое строкное значение.

- Для каждой строки должно быть 15 столбцов, у каждого столбца должен быть подходящий тип данных, а столбцы должны следовать в порядке, запятом или табуляце:

  **Формат файла данных "Построение данных"**
  
  | Имя столбца        | Тип данных | Пример                   | Рекомендации              |
  |--------------------|-----------|---------------------------|-----------------------|
  | NetworkIP          | String    | 192.168.1.0               | Обязательный              |
  | NetworkName        | String    | USA/Seattle/SEATTLE-SEA-1 | Требуется<sup>1</sup>  |
  | NetworkRange       | Число    | 26                        | Обязательный              |
  | BuildingName       | String    | SEATTLE-SEA-1             | Требуется<sup>1</sup>  |
  | OwnershipType      | String    | Contoso                   | Необязательно               |
  | BuildingType       | String    | IT Termination            | Необязательно               |
  | BuildingOfficeType | String    | Проектирование               | Необязательно               |
  | City               | String    | Сиэтл                   | Рекомендуется           |
  | ZipCode            | String    | 98001                     | Рекомендуется           |
  | Страна            | String    | США                        | Рекомендуется           |
  | State              | String    | Красноярский край                        | Рекомендуется           |
  | Region             | String    | MSUS                      | Рекомендуется           |
  | InsideCorp<sup>2</sup>         | Bool      | 1             | Обязательный              |
  | ExpressRoute<sup>3</sup>       | Bool      | 0             | Обязательный              |
  | VPN;                | Bool      | 0                         | Необязательно               |

  <sup>1.</sup> Хотя это не обязательно, шаблоны настроены для отображения названия "Здание" и "Сеть".

  <sup>2</sup> Этот параметр может использоваться для того, чтобы определить, находится ли подсеть внутри корпоративной сети. Вы можете настроить использование для других целей.

  <sup>3</sup> Этот параметр может использоваться для того, чтобы определить, использует ли сеть Azure ExpressRoute. Вы можете настроить использование для других целей.  

  **Образец строки:**

  `192.168.1.0,USA/Seattle/SEATTLE-SEA-1,26,SEATTLE-SEA-1,Contoso,IT Termination,Engineering,Seattle,98001,US,WA,MSUS,1,0,0`

> [!IMPORTANT]
> Диапазон сети можно использовать для представления суперсети (сочетания нескольких подсетей с одним префиксом маршрутизации). Все новые загружающиеся строения будут проверяться на наличие перекрывающихся диапазонов. Если вы ранее загрузили файл здания, скачайте текущий файл и повторно загрузите его, чтобы определить перекрытия и устранить проблему перед повторной отправкой. Любое перекрытие в ранее добавленных файлах может привести к неправильному сопоставлению подсетей со строениями в отчетах. В некоторых реализациях VPN данные подсети не точно сообщаются. 
>
> Столбец VPN необязателен и по умолчанию имеет значение 0. Если значение столбца VPN — 1, подсеть, представленная этой строкой, будет полностью расширина в соответствие со всеми IP-адресами в подсети.  Используйте этот способ только для подсетей VPN, так как полное расширение этих подсетей отрицательно скажется на времени запросов, связанных со созданием данных.


### <a name="supernetting"></a>Суперсети

Для определения подсети можно использовать надсети ( обычно Inter-Domain Classless Inter-Domain Routing ,CIDR). *Суперсеть* — это сочетание нескольких подсетей с одним префиксом маршрутизации. Вместо добавления записи для каждой подсети можно использовать адрес с надсетью. Суперсети поддерживаются, но мы не рекомендуем использовать их.

Например, маркетинговое здание Contoso состоит из подсетей ниже:

-   10.1.0.0/24 — первый этаж
-   10.1.1.0/24 — второй этаж
-   10.1.2.0/24 — третий этаж
-   10.1.3.0/24 — четвертый этаж

Вместо добавления записи для каждой подсети можно использовать адрес с надсетью — в данном примере — 10.1.0.0/22.

-   Network = 10.1.0.0
-   Диапазон сети = 22

Перед внедрением надсети учитывайте несколько моментов.

-   Суперсети можно использовать только в сопоставлении подсети с 8-битной и 28-битной маску.

-   Для работы надсети требуется меньше времени, но это может стоить уменьшения степени разности данных. Предположим, что возникла проблема с качеством подсети 10.1.2.0. Если вы реализовали суперсети, то не знаете, где находится подсеть и к каков типу сети (например, в лабораторной). Если вы определили все подсети для здания и добавленные сведения о расположении этажа, вы сможете различать эти подсети.

-   Важно убедиться, что суперсетный адрес правильный и не позволяет найти ненужные подсетей.

-   Довольно часто используется для поиска данных 192.168.0.0. Во многих организациях это означает, что пользователь дома. Для других людей это схема IP-адресов для спутниковых офисов. Если в вашей организации есть офисы, использующие эту конфигурацию, не включите ее в файл здания, так как трудно отличить домашние и внутренние сети от общих [подсетей.](quality-of-experience-review-guide.md#common-subnets) 

> [!IMPORTANT]
> Диапазон сети можно использовать для представления суперсети. Все новые файлы данных о строениях будут проверяться на наличие перекрывающихся диапазонов. Если вы ранее уже загрузили файл здания, скачайте текущий файл и загрузите его еще раз, чтобы определить перекрытия и устранить проблему. Любое перекрытие в ранее загруженных файлах может привести к неправильному сопоставлению подсетей со строениями в отчетах.

### <a name="vpn"></a>VPN;

К качеству работы данных о качестве обслуживания, отправляемых клиентами в Microsoft 365 или Office 365, источником данных CQD является VPN-флаг. В CQD этот параметр будет видеться как первый VPN и второй VPN-размер. Однако этот флажок зависит от отчетности поставщиков VPN для Windows о том, что зарегистрированный адаптер сети VPN является адаптером удаленного доступа. Не все поставщики VPN регистрируют адаптеры удаленного доступа надлежащим образом. В связи с этим встроенные фильтры VPN-запросов могут быть неав то, что вам нужно. Используйте рассмотренный выше столбец VPN для точной пометки и определения подсетей VPN. Кроме того, мы емкие подписи VPN-сетей для удобной идентификации в отчетах. Ниже приведены два примера метки подсетей VPN:

- Чтобы определить **сетевое имя,** введите в этом поле "VPN" для подсетей VPN.

  ![Снимок экрана отчета о QCD, показывающий VPN с использованием имени сети](media/qerguide-image-vpnnetworkname.png)

- Чтобы определить **название здания,** введите в этом поле "VPN" для подсетей VPN.

  ![Снимок экрана: отчет о QCD, показывающий VPN с названием здания](media/qerguide-image-vpnbuildingname.png)

> [!NOTE]
> Известно, что VPN-подключения неправильно интерлицируют тип сетевого подключения как проводное, когда подключение является беспроводным. При оценке качества по VPN-подключениям нельзя предположить, что тип подключения был точно определен.

## <a name="endpoint-data-file"></a>Файл данных конечной точки

Другим типом файла данных клиента CQD является файл данных **конечной** точки. Значения столбцов используются в столбце "Имя первой клиентской конечной точки" или "Имя второй конечной точки клиента" для показа сведений о конечных точках "Make", "Model" или "Type" (Модель) конечной точки. Формат отложенного файла данных должен соответствовать следующим условиям для проверки перед отправкой:

- Это должен быть TSV-файл (столбцы разделены с помощью tab) или CSV-файл (столбцы разделены запятой).

- Содержимое файла данных не включает в себя заглавные таблицы. Ожидается, что первая строка файла данных будет иметь реальные данные, а не заглавную метку, например "ИмяКонечная точка".

- Во всех семи столбцах используется только тип данных "Строка". Максимальная длина составляет 64 знака.

- Поле данных может быть пустым, но все равно должно быть разделено табулом или запятой. Пустому полю данных просто назначается пустое строкное значение.

- Имя конечной точки должно быть уникальным, в противном случае отправка завершится сбой. Если в одной или двух строках с одинаковым названием конечной точки есть повторяющиеся строки, конфликт приведет к неправильному объединению.

- Конечные точкиLabel1, EndpointLabel2 и EndpointLabel3 — это настраиваемые метки. Они могут быть пустыми строками или значениями, такими как "Ноутбук, назначенный ИТ-отделом 2018" или "Тег актива 5678".

- Для каждой строки должно быть семь столбцов, а столбцы должны следовать в следующем порядке:

  **Порядок полей:**

  EndpointName, EndpointMake, EndpointModel, EndpointType, EndpointLabel1, EndpointLabel2, EndpointLabel3

  **Образец строки:**

  `1409W3534, Fabrikam, Model 123, Laptop, IT designated 2018 Laptop, Asset Tag 5678, Purchase 2018`

## <a name="update-a-building-file"></a>Обновление файла здания

При сборе сведений о здании и подсети администраторы со временем часто загружают файл здания в несколько итерации, добавляя новые подсети и сведения о здании по мере их получения. В этом случае вам потребуется повторно отправить файл здания. Этот процесс похож на начальную отправку, как описано в предыдущем разделе, за исключением нескольких исключений, описанных в следующем разделе.

> [!Important]
> Одновременно может быть активен только один файл здания. Несколько файлов здания не являются накопительными.

## <a name="add-net-new-subnets"></a>Добавление новых подсетей

Иногда необходимо добавить в CQD net новые подсети, которые изначально не были частью топологии сети. Чтобы добавить новые подсети, на  странице "Отправка данных клиента" в CQD сделайте следующее:

1.  Скачайте исходный файл, если у вас еще нет его.

1.  Удалите текущий файл в CQD.

1.  Изменение исходного файла здания и предоставление даты окончания по крайней мере за день до приобретения новых сетей.

1.  В исходный файл здания можно ввести новые подсети.

1.  Загрузите измененный файл и установите дату начала на один день после того, как предыдущий файл оканчивается.

## <a name="add-missing-subnets"></a>Добавление отсутствующих подсетей

После добавления сведений о здании для управляемых сетей каждая из управляемых сетей должна иметь связь здания. Однако это не всегда так: Обычно пропускается несколько подсетей. Чтобы найти отсутствующие сети, просмотрите отчет **об** отсутствующих подсетях на странице **"Отчеты** о качестве работы" в CQD. В результате будут представлены все подсетей с 10 или более аудиопотоками, которые не определены в файле данных здания и помечены как внешние. Убедитесь, что в этом списке нет управляемых сетей. Если подсети отсутствуют, обновите исходный файл данных о здании и повторно загрузите его в CQD с помощью следующей процедуры:

1. Перейдите на **страницу отправки данных** клиента в CQD.

1. Скачайте исходный файл, если у вас еще нет его.

1. Удалите текущий файл в CQD.

1. В исходный файл можно ввести новые подсети.

1. Загрузите файл здания. Обязательно задайте дату начала как минимум на восемь месяцев раньше, чем за восемь месяцев до начала обработки исторических данных.


> [!IMPORTANT]
> Вам потребуется добавить свой ИД клиента в  отчет в качестве фильтра запроса для второго клиента, чтобы отфильтровать отчет, чтобы просмотреть только данные клиента вашей организации. В противном случае в отчете будут отированы федераированные подсети.

> [!NOTE] 
> Не забудьте настроить фильтр отчета за месяц с текущим месяцем. Выберите **"Изменить"** и настройте фильтр отчета **"Месяц год",** чтобы сохранить новый месяц по умолчанию.
