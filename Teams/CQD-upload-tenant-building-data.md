# <a name="upload-tenant-and-building-data-in-call-quality-dashboard-cqd"></a>Отправка данных о клиенте и сборке на панели мониторинга качества звонков (CQD)


Чтобы максимально эффективно использовать панель мониторинга качества звонков (CQD), рекомендуется отправить клиент и создать данные. Существует два типа файлов данных клиента: ["Сборка](#upload-building-data-file) " и " [Конечная точка"](#endpoint-data-file).

Пример шаблона данных клиента можно скачать [здесь](https://github.com/MicrosoftDocs/OfficeDocs-SkypeForBusiness/blob/live/Teams/downloads/locations-template.zip?raw=true). Сведения о создании сопоставления см. в статье ["Создание карты здания для CQD"](CQD-building-mapping.md).

На панели мониторинга сводных отчетов CQD выберите "Отправка данных клиента" в меню "Параметры CQD **"** (значок шестеренки в верхней части CQD). Здесь администраторы могут отправлять сведения о зданиях и конечных точках своей организации, такие как сопоставление IP-адресов и географической информации, сопоставление каждой точки беспроводного доступа и ЕЕ MAC-адреса и т. д.

1. Откройте CQD (в Центре администрирования Teams [https://cqd.teams.microsoft.com](https://cqd.teams.microsoft.com)или по адресу), а затем щелкните значок шестеренки в правом верхнем  углу и выберите команду "Отправить данные клиента" на странице "Сводные **отчеты**".

   ![Снимок экрана: диалоговое окно, которое появляется во время отправки данных.](media/qerguide-image-tenantdataupload.png)
    
2. Кроме того, если вы впервые посещаете CQD, вам будет предложено отправить данные о сборке. Чтобы **быстро перейти на** страницу отправки данных клиента, нажмите кнопку " **Отправить сейчас"** .

   ![Снимок экрана: баннер с уведомлением пользователя о передаче данных сборки.](media/qerguide-image-buildingdatauploadbanner.png)

3. На странице **"Отправка данных клиента** " нажмите **кнопку "Обзор** ", чтобы выбрать файл данных.

4. После выбора файла данных укажите дату **начала** и при необходимости укажите дату окончания.

5. После выбора **даты начала выберите** **"Отправить** ", чтобы отправить файл в CQD. <br><br>Перед отправкой файла он проверяется. В случае сбоя проверки отображается сообщение об ошибке с запросом на исправление файла. На следующем рисунке показана ошибка, когда число столбцов в файле данных неверно.

   ![Пример диалогового окна с сообщением об ошибке отправки данных здания.](media/qerguide-image-buildingdatauploaderror.png)
 
6. Если во время проверки не возникает ошибок, отправка файла будет выполнена успешно. После этого вы увидите отправленный файл данных в таблице "Мои отправки", где в нижней части этой страницы отображается полный список всех отправленных файлов для текущего клиента.

> [!NOTE]
> Обработка файла сборки может занять до четырех часов. <br><br> Если вы уже отправили файл сборки и вам нужно добавить подсети, которые могли быть пропущены или исключены, измените исходный файл, добавив новые подсети, удалите текущий файл и повторно отправьте измененный файл. В CQD может быть только один активный файл данных сборки. 


## <a name="upload-building-data-file"></a>Отправка файла данных сборки

Первым типом файла данных клиента в CQD является файл **данных "Сборка** ". Столбец подсети создается путем расширения столбца Network+NetworkRange, а затем присоединения столбца подсети к столбцу "Первая подсеть" или "Вторая подсеть" записи вызова для отображения сведений о здании, городе, стране или регионе. Формат файла данных, который вы отправляете, должен соответствовать следующим критериям, чтобы пройти проверку перед отправкой:
  
- Файл должен быть TSV-файлом (столбцы разделены табулятором) или файлом .csv (столбцы разделяются запятыми).

- Файл данных не содержит строку заголовка таблицы. Первая строка файла данных должна быть реальными данными, а не метами заголовков, такими как "Сеть".

- В файле могут использоваться только типы данных String, Integer или Boolean. Для целочисленного типа данных значение должно быть числовым значением. Логические значения должны быть либо 0, либо 1.

- Если столбец использует тип данных String, поле данных может быть пустым, но по-прежнему должно быть разделено вкладкой или запятой. Пустое поле данных просто присваивает пустое строковое значение.

- Существует ограничение в 1 000 000 расширенных строк на файл данных клиента.

- Для каждой строки должно быть 15 столбцов, каждый столбец должен иметь соответствующий тип данных, а столбцы должны находиться в порядке, указанном в следующей таблице (с разделителями-запятыми или вкладками):

  **Создание формата файла данных**
  
  | Имя столбца        | Тип данных | Пример                   | Рекомендации              |
  |--------------------|-----------|---------------------------|-----------------------|
  | NetworkIP          | String    | 192.168.1.0               | Обязательно              |
  | NetworkName        | String    | США/Seattle/SEATTLE-SEA-1 | Обязательный<sup>параметр 1</sup>  |
  | NetworkRange       | Число    | 26                        | Обязательно              |
  | BuildingName       | String    | СИЭТЛ-СИ-1             | Обязательный<sup>параметр 1</sup>  |
  | Тип владения      | String    | Contoso                   | Необязательно               |
  | BuildingType       | String    | Завершение ИТ-процессов            | Необязательно               |
  | BuildingOfficeType | String    | Инженерных               | Необязательно               |
  | City               | String    | Сиэтл                   | Рекомендуется           |
  | ZipCode            | String    | 98001                     | Рекомендуется           |
  | Страны            | String    | НАМ                        | Рекомендуется           |
  | State              | String    | Красноярский край                        | Рекомендуется           |
  | Region             | String    | MSUS                      | Рекомендуется           |
  | InsideCorp<sup>2</sup>         | Bool      | 1             | Обязательно              |
  | ExpressRoute<sup>3</sup>       | Bool      | 0             | Обязательно              |
  | VPN;                | Bool      | 0                         | Необязательно               |

  <sup>1</sup> . Несмотря на то, что CQD не требуется, шаблоны настроены для отображения имени здания и сети.

  <sup>2</sup> Этот параметр может использоваться для отражения того, находится ли подсеть в корпоративной сети. Вы можете настроить использование для других целей.

  <sup>3</sup> Этот параметр можно использовать для отражения того, использует ли сеть Azure ExpressRoute. Вы можете настроить использование для других целей.  

  **Пример строки:**

  `192.168.1.0,USA/Seattle/SEATTLE-SEA-1,26,SEATTLE-SEA-1,Contoso,IT Termination,Engineering,Seattle,98001,US,WA,MSUS,1,0,0`

> [!IMPORTANT]
> Диапазон сети можно использовать для представления суперсети (сочетание нескольких подсетей с одним префиксом маршрутизации). Все новые сборки будут проверяться на наличие перекрывающихся диапазонов. Если вы ранее отправили файл сборки, скачайте текущий файл и повторно отправьте его, чтобы выявить перекрытия и устранить проблему перед повторной отправкой. Любое перекрытие ранее отправленных файлов может привести к неправильному сопоставлению подсетей со зданиями в отчетах. Некоторые реализации VPN не содержат точный отчет о подсети. 
>
> Столбец VPN является необязательным и по умолчанию имеет значение 0. Если для столбца VPN задано значение 1, подсеть, представленная этой строкой, будет полностью развернута в соответствии со всеми IP-адресами в подсети. Используйте эту функцию с осторожностью и только для подсетей VPN, так как полное расширение этих подсетей отрицательно повлияет на время выполнения запросов, связанных со сборкой данных. Если расширение подсети приведет к превышению ограничения строки расширения в 1 000 000, файл сборки не будет принят.


### <a name="supernetting"></a>Суперсети

Вместо определения каждой подсети можно использовать суперсети, обычно называемые бесклассовой Inter-Domain маршрутизации (CIDR). *Суперсеть* — это сочетание нескольких подсетей с одним префиксом маршрутизации. Вместо добавления записи для каждой подсети можно использовать адрес суперсети. Суперсети поддерживаются, но мы не рекомендуем использовать их.

Например, маркетинговое здание Contoso состоит из следующих подсетей:

-   10.1.0.0/24 — первый этаж
-   10.1.1.0/24 — второй этаж
-   10.1.2.0/24 — третий этаж
-   10.1.3.0/24 — четвертый этаж

Вместо добавления записи для каждой подсети можно использовать адрес суперсети ( в этом примере — 10.1.0.0/22).

-   Сеть = 10.1.0.0
-   Диапазон сети = 22

Перед реализацией суперсети необходимо учесть несколько моментов.

-   Суперсети можно использовать только в сопоставлении подсети с 8-разрядной и 28-разрядной маской.

-   Суперсети заранее занимает меньше времени, но это происходит за счет снижения полноты данных. Предположим, что проблема с качеством связана с подсетью 10.1.2.0. Если вы реализовали суперсети, вы не узнаете, где в здании находится подсеть или какой тип сети она (например, лаборатория). Если бы вы определили все подсети для здания и отправили сведения о расположении этажа, вы сможете увидеть это различие.

-   Важно убедиться, что адрес суперсети правильный и не перехватываю нежелательные подсети.

-   Довольно часто в данных можно найти 192.168.0.0. Для многих организаций это означает, что пользователь находится дома. Для других это схема IP-адресов для вспомогательного офиса. Если в вашей организации есть офисы, использующие эту конфигурацию, не включайте ее в файл сборки, так как трудно различать домашнюю и внутреннюю сети с помощью общих [подсетей](quality-of-experience-review-guide.md#common-subnets). 

> [!IMPORTANT]
> Диапазон сети можно использовать для представления суперсети. Все новые отправки файлов данных сборки будут проверяться на наличие перекрывающихся диапазонов. Если вы ранее отправили файл сборки, скачайте текущий файл и отправьте его еще раз, чтобы выявить перекрытия и устранить проблему. Любое перекрытие ранее отправленных файлов может привести к неправильному сопоставлению подсетей со зданиями в отчетах.

### <a name="vpn"></a>VPN;

Данные качества взаимодействия (QoE), отправляемые клиентами в Microsoft 365 или Office 365 ( то есть источник данных CQD), включают флаг VPN. CQD будет видеть это как первое и второе измерения VPN. Однако этот флаг зависит от отчетности поставщиков VPN в Windows о том, что зарегистрированный сетевой адаптер VPN является адаптером удаленного доступа. Не все поставщики VPN правильно регистрируют адаптеры удаленного доступа. Из-за этого вы не сможете использовать встроенные фильтры ЗАПРОСОВ VPN. Используйте столбец VPN, описанный выше, чтобы точно пометить и определить подсети VPN. Кроме того, рекомендуется пометить vpn-сети для упрощения идентификации в отчетах. Ниже приведены два примера того, как пометить подсети VPN.

- Определите **сетевое имя** , введя в этом поле "VPN" для подсетей VPN.

  ![Снимок экрана отчета QCD, показывающий VPN с использованием сетевого имени.](media/qerguide-image-vpnnetworkname.png)

- Определите **имя здания,** введя в этом поле "VPN" для подсетей VPN.

  ![Снимок экрана отчета QCD, показывающий VPN с именем здания.](media/qerguide-image-vpnbuildingname.png)

> [!NOTE]
> Известно, что VPN-подключения неправильно называют тип сетевого подключения проводным, если базовое подключение является беспроводным. При просмотре качества VPN-подключений нельзя предположить, что тип подключения был точно определен.

## <a name="endpoint-data-file"></a>Файл данных конечной точки

Другим типом файла данных клиента CQD является файл данных **конечной** точки. Значения столбца используются в столбце "Имя первой конечной точки клиента" или "Имя второй конечной точки клиента" записи вызова для отображения сведений о конечных точках "Make", "Model" или "Type". Формат файла данных, который вы отправляете, должен соответствовать следующим критериям, чтобы пройти проверку перед отправкой:

- Файл должен быть TSV-файлом (столбцы разделены табулятором) или файлом .csv (столбцы разделяются запятыми).

- Содержимое файла данных не включает заголовки таблицы. Первая строка файла данных должна быть реальными данными, а не меткой заголовка, например EndpointName.

- Все семь столбцов используют только строковый тип данных. Максимальная допустимая длина составляет 64 символа.

- Записи чувствительны к регистру; EndpointName **ABC123** будет считаться уникальным из EndpointName **abc123**.

- Поле данных может быть пустым, но по-прежнему должно быть разделено вкладкой или запятой. Пустое поле данных просто присваивает пустое строковое значение.

- EndpointName должен быть уникальным, в противном случае отправка завершится ошибкой. Если имеется повторяющаяся строка или две строки, использующие одно и то же имя конечной точки, конфликт приведет к неправильному соединению.

- EndpointLabel1, EndpointLabel2 и EndpointLabel3 — это настраиваемые метки. Они могут быть пустыми строками или значениями, такими как "Ноутбук назначен ИТ-отделом 2018" или "Тег ресурса 5678".

- Для каждой строки должно быть семь столбцов, а столбцы должны быть в следующем порядке:

  **Порядок полей:**

  EndpointName, EndpointMake, EndpointModel, EndpointType, EndpointLabel1, EndpointLabel2, EndpointLabel3

  **Пример строки:**

  `1409W3534, Fabrikam, Model 123, Laptop, IT designated 2018 Laptop, Asset Tag 5678, Purchase 2018`

## <a name="update-a-building-file"></a>Обновление файла сборки

При сборе сведений о сборке и подсети администраторы часто отправляют файл сборки в нескольких итерациях с течением времени, добавляя новые подсети и сведения о зданиях по мере их доступности. В этом случае необходимо повторно отправить файл сборки. Этот процесс аналогично начальной отправке, как описано в предыдущем разделе, с несколькими исключениями, как указано в следующем разделе.

> [!Important]
> Одновременно может быть активен только один файл сборки. Несколько файлов сборки не являются накопительными.

## <a name="add-net-new-subnets"></a>Добавление новых подсетей

Иногда необходимо добавить в CQD новые сетевые подсети, которые изначально не были частью топологии сети. Чтобы добавить новые подсети, выполните следующие действия на странице  отправки данных клиента в CQD:

1.  Скачайте исходный файл, если у вас еще нет актуальной копии.

1.  Удалите текущий файл в CQD.

1.  Измените исходный файл сборки и укажите дату окончания, которая происходит по крайней мере за один день до получения новых подсетей.

1.  Добавьте новые подсети в исходный файл сборки.

1.  Отправьте измененный файл сборки и задайте дату начала на один день после окончания предыдущего файла сборки.

## <a name="add-missing-subnets"></a>Добавление недостающих подсетей

После отправки сведений о сборке для управляемых сетей каждая управляемая сеть должна иметь связь со сборкой. Однако это не всегда так. Как правило, пропускаются несколько подсетей. Чтобы найти эти отсутствующие сети, просмотрите  отчет о отсутствующей подсети **на странице** отчетов о качестве взаимодействия в CQD. Здесь представлены все подсети с 10 или более аудиопотоками, которые не определены в файле данных сборки и помечены как внешние. Убедитесь, что в этом списке нет управляемых сетей. Если подсети отсутствуют, выполните следующую процедуру, чтобы обновить исходный файл данных сборки и повторно отправить его в CQD.

1. Перейдите на **страницу отправки данных клиента** в CQD.

1. Скачайте исходный файл, если у вас еще нет актуальной копии.

1. Удалите текущий файл в CQD.

1. Добавьте новые подсети в исходный файл.

1. Отправьте файл сборки. Не забудьте задать дату начала по крайней мере на восемь месяцев раньше, чтобы CQD обрабатывал исторические данные.


> [!IMPORTANT]
> Вам потребуется добавить идентификатор клиента в качестве фильтра запросов для второго идентификатора клиента  в этот отчет, чтобы отфильтровать отчет, чтобы просмотреть только данные клиента вашей организации. В противном случае в отчете будут отображаться федеративные подсети.

> [!NOTE] 
> Не забудьте настроить фильтр отчета "Месяц года" по текущему месяцу. Выберите **"Изменить**" и измените фильтр отчета **"** Месяц года", чтобы сохранить новый месяц по умолчанию.
