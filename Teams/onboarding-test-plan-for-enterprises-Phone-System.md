<a name="define-and-document-your-phone-system-with-calling-plans-in-teams-test-plan-for-enterprises"></a>Определите и задокументируйте телефонной системой с вызова планы в план тестирования группами для предприятий 
============================================================================================

После определенного и описаны телефонной системой с планами вызов успеха бизнес группами и планы технического внедрения как часть этапа Предвидения, следующим шагом проверил выполнены ожиданиям и требования к организации посредством компонент, функциональные возможности и удобство использования. Следует выполнять это действие проверки перед развертыванием пилотных или окончательного развертывания в производственной среде.

Вы можете использовать возможности бизнес-план успеха, которую вы задали на этапе Предвидения, который будет служить основой для определения действий, ожидания, функции/тестовых случаев и общей области для оценки на этапе тестирования.

<a name="identify-testing-support-stakeholders"></a>Определение заинтересованных лиц тестирования поддержки
-------------------------------------

При подготовке к оценка телефонной системы с помощью вызова планы функции создания тестирования браузерах заинтересованных лиц для определения ролей, необходимых для поддержки этап тестирования.

> [!TIP]
> Как заполнить группы тестирования поддержки заинтересованные лица, можно увидеть, что некоторые роли являются такие же, как определяемую на этапе Предвидения, но с описания ролей наклонными к тестированию. Может потребоваться определение дополнительных ролей, в зависимости от уникальных требований тестирования сценариев.

<a name="define-phone-system-with-calling-plans-feature-requirements"></a>Определение телефонной системой требованиям вызов планы компонента 
------------------------------------------------------------

Как часть определения телефонной системой требованиям вызов планы компонентов для пользователей в области действия в разделе [Телефонной системы с помощью вызова планы](calling-plan-landing-page.md).
Затем Оцените последняя версия плана общедоступных групп для определения.

-   Какие телефонной системы с использованием функций вызова планы вы развертываете для пользователей в области действия.

-   Ожидается пользователя телефонной системой требованиям вызов планы функциональные возможности, учитывая текущий Скайп альбомная развертывания Business, Exchange и SharePoint.

-   Можно убедиться, что телефонной системы с использованием функций вызова планы описано в последние общедоступных схема, удовлетворяют ли пользователь, функциональные возможности и требования области в временная шкала развертывания

> [!TIP]
> Последняя версия плана группами для идентификации телефонной системы с использованием вызова планы функций в области действия для развертывания, можно найти в <https://aka.ms/O365Roadmap>.

Теперь, когда система телефона с функциями и вызов планы пользователя были определены, далее критерии оценки будет качества взаимодействия с группами. Для получения дополнительных сведений о качества взаимодействия и параметры конфигурации, доступные видеть [группами Майкрософт и Скайп для обеспечения взаимодействия бизнеса](https://docs.microsoft.com/microsoftteams/teams-and-skypeforbusiness-coexistence-and-interoperability).

<a name="define-and-document-your-phone-system-with-calling-plans-test-plan"></a>Определите и задокументируйте телефонной системой с помощью вызова планы плана тестирования
------------------------------------------------------------------

После определения телефонной системой с функциями вызов планы в область следующим шагом является создание плана тестирования. На высоком уровне план тестирования включает в себя общий стратегия тестирования и методики, которые будут использоваться для поддержки проверки компонента в процессе тестирования.

На высоком уровне необходимо включить в план тестирования.

-   **Тестирование области:** Приведены данные об областях фокус (задачах, сценарии и цели) для оценки как часть этапа тестирования

-   **Тестовых случаев:** Набор тестовых случаев для проверки для телефонной системы с помощью вызова планы функций в области видимости

-   **Ресурсов тестирования:** Таблица ресурсов, необходимых для поддержки по тестированию с окружающей среды, технологии и персонал точки зрения

-   **Тестирования расписания (временная шкала):** Представляет при тестировании начнется, как долго оно скорее всего последнего и когда ожидать этап тестирования, чтобы закончить

-   **Отключение отчетов и исправлению:** Рекомендации по как должны быть выявлены проблемы, связанные с тестирования, отслеживания и рассмотрены

-   **Отключение рассмотрения и эскалации:** Структура для как и когда следует инициировать эскалации ошибки

-   **Тестирование выход и приостановка критерии:** Руководство по структуре критерии для любого из достижения подписи, чтобы выйти из этап тестирования или приостановка тестирования, пока не будут разрешены приоритетов дефекты

-   **Тестирование конечные результаты:** Сводка из которых результаты будут разработки и поддержке приемки подписи и выходе из процесс тестирования

> [!TIP]
>   Методология тестирования уже может существовать в вашей организации, но ниже инструкции отражает рекомендации, которые могут быть включены и выполнять отдельно для тестирования функции группами в вашей среде.

В следующих разделах, вы найдете дополнительные предписанный рекомендации, которые будут полезны в конкретных решений, шаблоны и разделы, которые необходимо учесть после завершения плана тестирования.

На высоком уровне это обычно состоит из:

-   **Задачи:** Высокоуровневые действия для завершения

-   **Вехи:** Высокоуровневая задача или ход выполнения, который был выполнен

-   **Владелец:** Назначенный ресурс, который несет ответственность за обеспечение на время, в соответствии с датой завершения выполнения действия, вехи или задач

-   **Оценки:** Количество часов, назначенных ресурсов чтобы ожидаемые его потребуется для обеспечения на время выполнения действия, вехи или задач

### <a name="define-and-document-test-defect-report-criteria"></a>Определите и задокументируйте критерии отчета дефектов тестирования

При выполнении тестовых случаев в рамках данного этапа или поток проблемы могут возникнуть в которых не будет результат тестового случая ожидаемого.

При возникновении этих типов результатов, они будут заноситься в дефектов отчетов и исправлению план, который перерастает в указанном интерес для проверки, отчетов и разрешение ошибки.

Как правило план отчетов и исправлению ошибок включает в себя следующее:

-   **Открепить ID:** Номер, назначенный этой проблеме

-   **Идентификатор тестового случая на которые оказывает**: номер, назначенный для этого тестового случая, при вычислении во время, обнаруженных ошибок

-   **Открепить Описание:** Сводка о проблеме

-   **Среды и действия для воспроизведения:** Сводка по тестирования Настройка среды, а также точное действий, необходимых для воспроизвести суть проблемы

-   **Уровень серьезности ошибки**: влияние проблемы, включающих от предотвращения требуется утверждение для его принят с минимальным риском тестового случая. Может включать несколько примеров:

-   **Low:** Проблема мало влияет и не предотвратить обеспечения допустимости, если более поздней версии для устранения проблемы тестового случая.

-   **Среднего:** Проблема имеет значительно сократить объем вложений, но не запретить обеспечения допустимости, если проблема не разрешается до завершения этапа тестирования тестового случая.

-   **Высокой:** -проблему имеет важные воздействия на тестового случая. Проблема должна быть решена, прежде чем можно будет принято тестового случая.

-   **Состояние:** Проблемы, разрешенные, — это после открытия или находится в отношении которых ведется расследование

-   **Кем отправлено:** Инженер-испытатель решения проблемы

-   **Владелец назначения:** Ресурс, назначенный от тестирования команды, который несет ответственность за устранение проблемы

-   **Вспомогательных сведений:** Сюда может входить —, но не ограничиваясь — журналов на стороне клиента, снимки экрана или видео о проблеме.

Как инженеры-испытатели выполняют тестовые случаи, описанные в план тестирования, они должны обязательно выполнения плана отчетов и исправлению ошибок для всех ошибок, возникающих.
В этом примере потенциальном влиянии, которое может снизить или приостановить процесс тестирования оценки.

-   Проверьте подключение к сети и производительности через моделирование трафика

-   Туннелирование Split проверки

<a name="execute-subscriber-number-port-validation"></a>Выполнить проверку номера порта подписчика
-----------------------------------------

Если вам потребуется преобразование чисел в ходе предоставления входящих и исходящих вызовов возможности телефонной системы с помощью служб вызов планы, поддерживаемые только группами, следует выполнить частичное порт для службы номера. Эта информация поможет проверка ожиданиям, требования и сравнительно временной шкалы, по окончании Подготовка развертывания системы телефона с помощью вызова планы служб в рабочей среде.

Для выполнения частичное порт номер абонента от поставщика услуг текущего PSTN группам, познакомьтесь с помощью шагов, описанных ниже.

#### <a name="step-1"></a>Шаг 1

Укажите номер теста, который вы хотите портов в Office 365 как пользователь может быть назначен номер (номер абонентского) для обслуживания через телефонную систему со службой вызов планы в группах.

> [!IMPORTANT]
>При планировании номеров переноса теста, убедитесь, что просмотрите последние рекомендации для номеров переноса запросов в [Часто задаваемые вопросы по номеру порта](https://docs.microsoft.com/SkypeForBusiness/what-are-calling-plans-in-office-365/transferring-phone-numbers-common-questions).

#### <a name="step-2"></a>Шаг 2

Определите и Задокументируйте все сведения учетной записи (включая имя, используемое для конкретной учетной записи) для текущего поставщика номер теста, который вы будете перенос.
Как правило, можно найти сведения, необходимые в последнюю Билл или создания счетов-фактур от текущего поставщика услуг.

> [!TIP]
>   Можно перенести или перемещение телефонных номеров в рамках всех поддерживаемых стран или регионов; Тем не менее способ отправки запроса заказа порт может отличаться в зависимости от страны или региона, где телефонных номеров, поступают из. Текущий список поддерживаемых стран или регионов в разделе [странах и область доступности для конференц-связи аудио и вызов планов](country-and-region-availability-for-audio-conferencing-and-calling-plans/country-and-region-availability-for-audio-conferencing-and-calling-plans.md).
> 
>   Для получения дополнительных сведений о передаче телефонных номеров телефонной системы с помощью вызова планы — а также потенциальные ограничения — [передачи телефонных номеров в Office 365](https://docs.microsoft.com/SkypeForBusiness/what-are-calling-plans-in-office-365/transfer-phone-numbers-to-office-365) и [бесплатно набору номера, ограничения в Office 365](https://docs.microsoft.com/SkypeForBusiness/what-are-calling-plans-in-office-365/toll-free-dialing-limitations-and-restrictions.).

#### <a name="step-3"></a>Шаг 3

Загрузите и Создание письма из авторизации (ЗАГРУ) для вашей страны или региона, основанный на «служба номер» как тип номера переноса.

>[!NOTE]
>   Так как форматы ЗАГРУ можно зависят от страны, региона или тип номера (, географическое сравнение географическое или номер пользователя или службы или телефоны), убедитесь, что используется правильный шаблон ЗАГРУ для соответствующего типа определенного сценария. [Загрузите буква авторизации (ЗАГРУ)](https://docs.microsoft.com/SkypeForBusiness/what-are-calling-plans-in-office-365/download-a-letter-of-authorization-loa) более подробные сведения о выборе ЗАГРУ или непосредственно перейдите к [странице загрузки](https://www.microsoft.com/download/details.aspx?id=49167).

#### <a name="step-4"></a>Шаг 4

На портале администрирования клиента перейдите на вкладку **поддержки** и создать и выдать запрос службы. Присоединение завершенный файл ЗАГРУ планирование номер телефона, связанного с текущей поставщиком услуг для миграции. Чтобы выбрать наиболее подходящий для вашего клиента размер метод запроса службы, обратитесь к разделу [вручную отправить запрос настраиваемые службы](https://docs.microsoft.com/SkypeForBusiness/what-are-calling-plans-in-office-365/manually-submit-a-custom-service-request).

#### <a name="step-5"></a>Шаг 5

После подтверждения номер как были перенесены через как новый номер абонента в Office 365 присвоить номер телефонной системой со службой вызов планы, перейдя на портал администрирования клиента \> **Скайп по центру администрирования Business** \> ** Голосовой связи**. На вкладке **Номера телефонов** назначьте новый номер службы телефонной системой со службой вызов планы.

> [!TIP]
>   Хотя эту задачу назначает новый номер службы телефонной системы с помощью вызова планы, во время написания этой статьи, администрирования для этой задачи выполняется с использованием Скайп по центру администрирования бизнеса.
