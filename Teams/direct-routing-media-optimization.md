# <a name="local-media-optimization-for-direct-routing"></a>Оптимизация локальных файлов мультимедиа для прямой маршрутизации

Коммутируемая телефонная сеть с открытым коммутируемым подключением (PSTN) считается важным для бизнеса приложением с высокими приоритетами для качества голосовой связи. Прямая маршрутизация позволяет управлять потоком трафика мультимедиа в соответствии с множеством сетевых топологий и настройками телефонной связи для различных предприятий в любой стране мира. 

Локальная Оптимизация файлов мультимедиа для прямой маршрутизации позволяет управлять качеством голоса, выполнив указанные ниже действия.

-   Управление обменом трафиком мультимедиа между клиентами Teams и контроллерами границ (SBCs) для сеансов работы с клиентом.
-   Хранение локальных файлов мультимедиа в рамках корпоративной сети.
-   Разрешать потоки мультимедиа между клиентами Teams и SBCs, даже если они находятся за корпоративными брандмауэрами с частными адресными адресами, а не видны в Microsoft напрямую.

Локальная Оптимизация файлов мультимедиа поддерживает два сценария.

- Централизованное управление всеми локальными магистральами с помощью централизованного SBC, подключенного к магистральной магистрали протокола SIP — предоставление услуг телефонной связи для всех местных филиалов компании.

-   Создание однобайтовой топологии виртуальной сети, где SBCs-связь в локальных офисах филиалов подключается к централизованному прокси-серверу, который является видимым для телефонной системы Microsoft и может взаимодействовать с ним с помощью внешнего IP-адреса. В топологии виртуальных сетей более нижестоящий SBCs-трафик подключается к внутренним IP-адресам и не отображается непосредственно в телефонной системе.

В этой статье описаны функции функций и сценарии клиентов и решения. Подробнее о конфигурации можно найти в разделе [Настройка оптимизации локальных носителей](direct-routing-media-optimization-configure.md). 

  > [!NOTE]
  > Если вы хотите хранить локальную версию мультимедиа в пределах вашей интрасети, рекомендуется использовать локальную оптимизацию мультимедиа. Если у вас уже есть обход мультимедиа и вы используете только общедоступные IP-адреса для SBCs, вам не обязательно перемещаться на локальную оптимизацию мультимедиа. Вы можете продолжать использовать мультимедиа для обхода. Дополнительные сведения можно найти в разделе [Планирование обхода мультимедиа](direct-routing-plan-media-bypass.md).


## <a name="supported-customer-scenarios"></a>Поддерживаемые сценарии для клиентов

В этой статье предполагается, что компания Contoso использует несколько предприятий по всему миру, как описано ниже. (Обратите внимание, что в странах Европы и APAC регионы используются только примеры. У компании может быть несколько разных регионов с похожими потребностями.)
 
- **В Европе** компании Contoso есть офисы в течение примерно 30 стран. У каждого офиса есть собственный частный обмен филиалами (АТС). 

  Компания Contoso предложит вам возможность централизации каналов в одном месте----для всех 30 европейских офисов. Компания Contoso развернула SBC в Амстердам и послужила достаточной пропускной способностью для выполнения звонков через централизованное расположение, подключения центрального канала SIP к централизованному расположению и начала обслуживания всех европейских расположений из Амстердам. 

- **В APAC регионе** компания Contoso имеет несколько офисов в разных странах мира. 

  Во многих странах компания по-прежнему имеет магистральные направления (TDM) в локальных офисах филиалов. Централизованные магистрали TDM не являются вариантами в APAC регионе, поэтому переключение на SIP невозможно. Предположим, что в APAC регионе более 50 офисов филиалов с сотнями шлюзов (SBCs). В этом случае невозможно связать все шлюзы с интерфейсом Direct Routing из-за отсутствия общедоступных и (или) локальных адресов Интернета. Кроме того, в некоторых странах действуют законодательные требования, которые не могут быть выполнены без подключения к локальной сети PSTN.

В соответствии с бизнес-требованиями компания Contoso реализовала два решения с оптимизацией локальных файлов для прямой маршрутизации.

- **В Европе** все магистральные линии являются централизованными и потоками мультимедиа между центральным SBC и пользователями на основе местоположения пользователя. 

  - Если пользователь подключен к локальной подсети корпоративной сети (то есть он является внутренним), поток мультимедиа между внутренним IP-адресом центрального SBC и клиентской версией пользователя Teams. 
  
  - Если пользователь находится за пределами корпоративной сети (например, если пользователь использует открытое беспроводное подключение к Интернету), то пользователь считается внешним. В этом случае мультимедиа перетекает между внешним IP-адресом центрального SBC и клиента Teams.

- **В регионе APAC** централизованный прокси-объект SBC связан с прямой маршрутизацией Майкрософт, который направляет носители между интерфейсом Direct Routing и подчиненной SBCs в локальных отделениях филиалов. 

  Подчиненные SBCs в локальных офисах филиалов не видны непосредственно в APAC, но они сопряжены с помощью командлета Set-CSOnlinePSTNGateway для создания топологии виртуальной сети в телефонной системе Microsoft. По возможности мультимедиа всегда останется на локальном компьютере. Внешние пользователи имеют мультимедиа, находящиеся между клиентом Teams и общедоступным IP-адресом одноименного объекта SBC.


## <a name="central-sbc-with-centralized-trunks"></a>Центральный SBC с централизованными магистральными коммуникационными приложениями

Чтобы создать решение, в котором для всех местных филиалов предоставляются услуги PSTN по единому центральному элементу управления с помощью подключенной централизованной магистрали SIP, в качестве пары "Администратор клиента Contoso" используется один SBC (centralsbc.contoso.com) для службы. SBC имеет централизованно подключенную магистраль SIP. 

- Когда пользователь входит в внутреннюю сеть компании, SBC обеспечивает внутренний IP-адрес SBC для мультимедиа.

- Если пользователь находится за пределами корпоративной сети, SBC предоставляет внешний (общедоступный) IP-адрес SBC.

## <a name="local-media-optimization-modes"></a>Локальные режимы оптимизации файлов мультимедиа

Локальная Оптимизация файлов мультимедиа поддерживает два режима:

- **Режим 1: всегда обходить**. В этом случае, если пользователь является внутренним, он будет перетекать на внутренний IP-адрес локального нижестоящих SBC независимо от фактического расположения внутреннего пользователя. Например, в том же офисе филиала, где расположен один и тот же однонаправленный SBC, или в другом филиале.

- **Режим 2: только для локальных пользователей**. В этом режиме мультимедиа помещается непосредственно на внутренний IP-адрес локального нижестоящих SBC только при создании внутренним пользователем в том же офисе филиала, что и нижестоящий одноэлементный SBC. 

Чтобы отличать режимы оптимизации локальных файлов, администратору клиента необходимо задать для параметра-BypassMode значение "Always" или "OnlyForLocalUsers" для каждого SBC с помощью командлета Set-CSonlinePSTNGateway. Дополнительные сведения можно найти в разделе [Настройка оптимизации локальных носителей](direct-routing-media-optimization-configure.md).  

### <a name="mode-1-always-bypass"></a>Режим 1: всегда обходить

Если у вас есть хорошее соединение между филиалами, рекомендуемый режим всегда обходится.

Например, предположим, что у компании есть централизованная магистральная линия SIP в Амстердам, которая обслуживает 30 стран и имеет хорошую связь между всеми 30 сайтами и локальными пользователями. Кроме того, существует ветвь в Германии, где развернут локальный SBC.

SBC в Германии можно настроить в режиме "всегда минуя". Пользователи, независимо от их местоположения, будут подключаться к SBC прямо по внутреннему IP-адресу SBC (например, из Франции в Германии) (для получения справки ознакомьтесь со схемой ниже).

### <a name="mode-2-only-for-local-users"></a>Режим 2: только для локальных пользователей.

Если между локальными филиалами есть плохие соединения, но при этом связь между локальным офисом филиалов и региональным Office является надежной, рекомендуемый режим — "только для локальных пользователей".

Например, предположим, что в APAC регионе компания Contoso имеет несколько офисов в разных странах мира. Для многих стран переключение на SIP невозможно, так как компания по-прежнему имеет TDMную магистраль в нескольких местных офисах филиалов. Централизованные магистрали TDM не являются вариантами в APAC регионе. Кроме того, у вас более 50 филиалов в области APAC с сотнями шлюзов (SBCs). 

Чтобы создать решение, в котором службы PSTN предоставляются во всех местных офисах филиалов в регионе APAC, если централизованная Маршрутизация магистральных каналов TDM не является одним из них, в качестве прокси-объекта SBC для прямой маршрутизации используется пара региональных SBC в Сингапур. Прямое соединение между локальными офисами филиалов не имеет ничего хорошего, но есть связь между всеми локальными филиалами и региональными SBC в Сингапур. Для региональных SBC, администратор выбрал режим "всегда обходить" и для локальной однобайтовой записи администратор выбрал режим "только для локальных пользователей".
