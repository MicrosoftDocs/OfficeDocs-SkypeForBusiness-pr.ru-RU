## <a name="functional-and-technical-requirements"></a>Функциональные и технические требования  

### <a name="authentication"></a>Проверка подлинности  

Авторизации уровня приложения *без поддержки для авторизации уровня пользователя* — наиболее часто поддерживаемые способ выполнения преобразования данных и предоставляют доступ к подключения к данным EHR через FHIR, несмотря на то, что система EHR может внедрить уровня авторизации пользователей . Службы взаимодействия (партнера) получает с повышенными правами доступа к данным EHR и они предоставляют доступ к тому же данных соответствующие ресурсы FHIR контекст отсутствует авторизации, переданных потребитель службы взаимодействия (пострадавших приложение) интеграция с взаимодействия Служба или платформы. Приложение пострадавших не сможет для применения параметров уровня авторизации пользователя, но поддержки проверки подлинности приложения между приложением пациентов и службы взаимодействия партнера.

1. Служба-партнер позволяет приложению пострадавших следует создать учетную запись с партнером, что позволяет пострадавших приложения для создания и собственные client_id и client_secret, управляемые через портал регистрации проверки подлинности на сервере проверки подлинности партнеров.
2. Служба партнеров несет ответственность за системы проверки подлинности и авторизации, который принимает и проверяет (проходит проверку подлинности) клиент учетные данные, предоставляемые и дает обратно маркер доступа с подсказкой клиента в области действия, как описано ниже.

### <a name="routing-mapping-aad-tenant-to-the-provider-endpoint"></a>Маршрутизация: Сопоставление AAD клиента к конечной точке поставщика

Приложение пострадавших подключается к службе партнера по одной конечной точки. Служба партнера несет ответственность за и поддерживает механизм сопоставления каждого клиента Microsoft (идентификатор клиента AAD) соответствующих здравоохранения поставщика (FHIR server), работа службы партнеров с.

Сопоставление AAD клиента с конечной точкой поставщика использует AAD клиента идентификатор (GUID). Приложение пострадавших передает идентификатор клиента в области действия, при запросе маркер доступа для каждого запроса. Служба партнера отслеживает сопоставления ИД клиента к конечной точке поставщика и перенаправляет запросы к конечной точке поставщика на основании ИД клиента. Для этого партнера поддерживает конфигурации на их окончания (вручную или через портал как часть входящая поставщика организациям их платформе взаимодействия).

Ниже показан рабочего процесса проверки подлинности и маршрутизации.

![Приложение пострадавших 6](../../media/Patient-app-6.png)

1. Запрос маркера доступа приложения путем отправки:
 
        {   grant_type: client_credentials,
            client_id: xxxxxx, 
            client_secret: xxxxxx,
            scope: {Provider Identifier, Ex: tenant ID}
        }

2. Ответ с маркером приложения:

        {  access_token: {JWT, with scope: tenant ID},
           expires_in: 156678,
           token_type: "Bearer",
        }

3. Запросите защищенные данные с маркером доступа.
4. Сообщение авторизации: выберите соответствующий сервер FHIR для маршрутизации из ИД клиента в области видимости
5. Отправляет данные приложения защищенных с авторизованного сервера FHIR после проверки подлинности с помощью маркера приложения.
