# <a name="upgrade-from-skype-for-business-to-teams-mdash-for-it-administrators"></a>Переход с Skype для бизнеса на Teams &mdash; для ИТ-администраторов

## <a name="overview"></a>Обзор

При переходе с Skype для бизнеса на Teams для некоторых организаций требуется прогрессивный выпуск, который планируется и управляется ИТ-подразделениями. Эта статья предназначена главным образом для ИТ-администраторов в крупных и локальных организациях, но она также может быть применима к некоторым организациям Skype для бизнеса Online.  Перед тем как читать эту статью, ознакомьтесь со статьей начало [работы с обновлением Teams](upgrade-start-here.md) и [средой обновления](upgrade-framework.md).

>[!NOTE]
>В этой статье приводятся условия использования Skype для бизнеса Online, Skype для бизнеса в локальной среде и Skype для бизнеса.  Последний термин относится как к сети, так и к локальным версиям.

Пользователь, который был перенесен в Teams, больше не использует клиент Skype для бизнеса, за тем исключением, что вы присоединяетесь к собранию в Skype для бизнеса.  Все входящие разговоры и звонки, находящиеся в клиенте Teams пользователя, независимо от того, использует ли отправитель Teams или Skype для бизнеса. Все новые собрания, упорядоченные из мигрировавших пользователей, будут планироваться как собрания Teams. Если пользователь попытается использовать клиент Skype для бизнеса, использование чатов и звонков блокируется<sup>1</sup>.  Тем не менее, пользователь может (и должен) по-прежнему использовать клиент Skype для бизнеса для присоединения к собраниям, на которые они приглашены.
 
Администраторы управляют переходами в Teams с использованием [режима "режим](migration-interop-guidance-for-teams-with-skype.md#coexistence-modes)", который является свойством [теамсупградеполици](https://docs.microsoft.com/powershell/module/skype/grant-csteamsupgradepolicy?view=skype-ps). Пользователь, который был перенесен в Teams, как описано выше, находится в режиме "Теамсонли".  Для Организации, которая переносится в Teams, конечная цель состоит в том, чтобы переместить всех пользователей в режим Теамсонли.

Существует два способа переноса существующей организации в Skype для бизнеса (в Интернете или в локальной среде) для Teams.

- Одновременный **метод** (с использованием режима острова): пользователи в существующей организации Skype для бизнеса вводятся в Teams, чтобы они могли одновременно использовать оба клиента во время перехода. В течение этого периода вы можете использовать для них не все возможности Teams. Режим для этой конфигурации называется «острова», и этот режим используется по умолчанию для любой организации, в которой установлен Skype для бизнеса. После того как Организация готова, администратор перемещает пользователей в режим Теамсонли.

- **Управляемый метод** (с использованием одного или нескольких режимов Skype для бизнеса): администратор управляет переходом (от Skype для бизнеса в Teams), а также с помощью функций планирования чата, звонков и собраний для пользователей в своей организации.  Каждая из этих функций доступна либо в Skype для бизнеса, либо в Teams, но не на обоих. Администраторы используют Теамсупградеполици, чтобы управлять тем, когда вы перемещаете эту функцию в Teams для пользователей. Пользователи, которые еще не в режиме Теамсонли, продолжают использовать Skype для бизнеса для общения и звонков, а два набора пользователей могут взаимодействовать с помощью функций взаимодействия. Администраторы могут управлять переходом, изменяя способно переводя пользователей в режим Теамсонли.  

В этой статье объясняется, как выбрать нужный метод для Организации, обменяя оба способа и выполнив для них достоинства и недостатки каждого из них. 

**Заметки:**
<sup>1</sup>старые клиенты Skype для бизнеса, которые поставлялись до 2017, не соблюдают теамсупградеполици. Убедитесь, что вы используете новейшую версию клиента Skype для бизнеса.


## <a name="side-by-side-method-using-islands-mode"></a>Одностороннее метод (с использованием режима острова)

С помощью этого метода пользователи могут использовать как команды, так и клиенты Skype для бизнеса для общения, звонков по VoIP и собраний. Это состояние называется режимом "острова", поскольку трафик для взаимодействия в Skype для бизнеса и Teams остается отдельным (даже для одного пользователя), а два разных клиента не взаимодействуют друг с другом (для пользователей в пределах одной организации). Например, предположим, что пользователь A находится в режиме "острова".

- Связь, которая инициируется клиентом Skype для бизнеса другого пользователя, всегда будет в клиенте "Skype для бизнеса".
- Связь, инициированная клиентом группы другого пользователя, всегда будет находиться в клиенте Teams для пользователя, *Если другой пользователь находится в той же организации*. 
- Связь, инициированная из клиента группы другого пользователя, всегда будет в клиенте "Skype для бизнеса *", если другой пользователь — в федеративной организации*.

Режим "острова" — режим по умолчанию Теамсупградеполици для любой существующей организации, которая еще не Теамсонли. После назначения лицензии на Office 365 по умолчанию назначаются лицензии для Teams и Skype для бизнеса Online. <sup>2</sup> . Если вы не предоставили какие – либо инструкции по изменению конфигурации по умолчанию, возможно, у вас уже есть существенные способы использования Teams в Организации.  Это один из преимуществ параллельного подхода. Она позволяет быстро и ориентированно на конечную пользователей переходить в пределах организации.

Для эффективного использования этого метода требуется, чтобы все пользователи одновременно работали на обоих клиентах. Входящие разговоры и звонки из Организации в режим "острова" могут быть доступны либо в Skype для бизнеса, либо в клиенте Teams, и это не является элементом управления получателя. Это зависит от того, какой клиент использует отправитель для инициирования обмена данными. Если отправитель и получатель находятся в разных организациях, входящие звонки и разговоры в режиме острова всегда помещаются в клиенте Skype для бизнеса.  

Например, если получатель режима «острова» работает в Skype для бизнеса, но не в Teams, а кто-то отправляет сообщение из Teams, получатель в режиме «острова» не будет видеть его (но в конечном итоге получит сообщения электронной почты о том, что они пропустили сообщение в Teams). Аналогичным образом, если пользователь работает с группами, но не Skype для бизнеса, а кто-то поделился этим пользователем в Skype для бизнеса, он не будет видеть этот чат.  Сообщение о том, что оно было пропущено, будет получено по электронной почте. Поведение в каждом из этих случаев похоже на вызов. Если пользователи не работают на обоих клиентах, это может привести к неудовлетворенности.

Когда пользователь A находится в режиме "острова", для пользователей в Teams, а также в Skype для бизнеса независимым является возможность присутствия.

- Другие пользователи, работающие с группами, увидят состояние присутствия на основе действий пользователя а в Teams. 
- Другие пользователи при использовании Skype для бизнеса увидят состояние присутствия на основе действий пользователя а в Skype для бизнеса. 

Это означает, что другие пользователи могут видеть различные состояния присутствия для пользователя а в зависимости от того, какой клиент они используют. Дополнительные сведения можно найти в разделе состояние [присутствия](#presence).

После того как вы будете готовы перейти к обновлению режима Теамсонли, вы можете обновить пользователей по отдельности или обновить весь клиент за один раз с помощью политики на уровне клиента<sup>3</sup>. После того как пользователь будет обновлен до Теамсонли режима, он получит все входящие разговоры и звонки в Teams. 

Однако необновляемые получатели в режиме "острова" могут по-прежнему принимать разговоры и звонки от Теамсонли пользователя в Skype для бизнеса или клиентах Teams.  Это связано с тем, что клиент Teams поддерживает отдельные потоки беседы для обмена сообщениями между группами и группами в Skype для бизнеса, даже для одного и того же пользователя.  (См. [: беседы в группах — взаимодействие и потоки взаимодействия и машинного кода](#teams-conversations---interop-versus-native-threads).)  Например, предположим, что пользователь A использует команды для сообщения Теамсонли User B. Когда пользователь B отправляет ответы на этот чат, Обмен данными осуществляется в клиенте Teams для пользователей. Теперь предполагаю, что пользователь A использует свой клиент Skype для бизнеса в сообщение Теамсонли пользователя B. пользователь B получит собеседника в Teams, но это будет отдельной беседа в клиенте Teams для пользователей по сравнению с другой беседой. Если пользователь B отвечает на этот чат с пользователем A, он будет находиться в клиенте "Skype для бизнеса" пользователя "A".

## <a name="managed-transition-method-using-skype-for-business-modes"></a>Управляемый метод перехода (в режимах Skype для бизнеса)

В некоторых организациях предпочтительнее предоставить конечным пользователям более простой и предсказуемый опыт в переходе между организациями из Skype для бизнеса в Teams. В этой модели ИТ-администраторы используют один из режимов Skype для бизнеса в Теамсупградеполици, чтобы явным образом определять, какие пользователи остаются в Skype для бизнеса перед переходом в режим Теамсонли. После того как они будут готовы к смене выбранных пользователей в режиме Теамсонли, администратор обновляет режим для этих пользователей на Теамсонли.  По мере выполнения развертывания новые пользователи переходят из Skype для бизнеса в режим Теамсонли.  Во время этого перехода:

- Пользователи Skype для бизнеса смогут получать все входящие разговоры и звонки в клиенте Skype для бизнеса независимо от того, откуда они поступили: от других пользователей Teams или Skype для бизнеса. Кроме того, для пользователей Skype для бизнеса возможности звонков и чата в клиенте Teams отключаются, чтобы предотвратить путаницу с конечными пользователями и обеспечить правильную маршрутизацию. 

- Пользователи в режиме Теамсонли получают все входящие разговоры и звонки в своем клиенте Teams, независимо от того, откуда поступило соединение: teams, Skype для бизнеса или любой другой федеративный пользователь. 

В отличие от метода "острова", в методе управляемого перехода пользователи Skype для бизнеса и пользователи Теамсонли могут общаться друг с другом. Связь между пользователями Skype для бизнеса и пользователями Teams называется взаимодействием или взаимодействием. (См. раздел [взаимодействие](#interoperability).) Взаимодействие между абонентами в Skype для бизнеса и другим пользователем в Teams может осуществляться с помощью взаимодействия между пользователями. Кроме того, приглашенные пользователи всегда могут присоединиться к собранию Skype для бизнеса или Teams, однако они должны использовать клиент, соответствующий типу собрания. Дополнительные сведения можно найти в разделе [собрания](#meetings).

Так как пользователи в управляемом переходе обычно не находятся в режиме "острова", присутствие пользователя является единым, независимо от того, какой клиент используется другим пользователем. Если пользователь входит в один из режимов Skype для бизнеса, другие пользователи видят сведения о присутствии на основе активности пользователя в Skype для бизнеса. Точно так же, если пользователь находится в режиме Теамсонли, все другие пользователи видят сведения о присутствии на основе активности этого пользователя в Teams. Подробнее: Просмотр сведений о [присутствии](#presence).

Для Организации, которая еще не начиналась с помощью Teams, администратор должен изменить режим клиента с островов на Сфбвистеамсколлаб. (В организациях, у которых уже есть использование некоторых групп, администратор должен "грандфасер" пользователей, которые уже активны в Teams, чтобы не применять это изменение. Дополнительные сведения можно найти [в разделе Управление обновлением для Организации, в котором уже используется Teams в режиме "острова](#a-managed-upgrade-for-an-organization-that-is-already-using-teams-in-islands-mode)".)

Если режим изменится с островов на Сфбвистеамсколлаб, пользователь, который никогда не использовался в Teams, не получит никаких различий в том, как они используют Skype для бизнеса. Тем не менее, если пользователь начал использовать Teams, они будут доступны только для таких функций, как команды & канал и файлы. Skype, звонки и собрания в Teams будут недоступны, так как администратор (сейчас) заблокировал (-а) в качестве нужного клиента для этих функций.  

Примечание. когда пользователь а изменяются из островов в один из режимов Skype для бизнеса, клиент Teams любого другого пользователя, который взаимодействует с пользователем, должен знать, что пользователь изменил режим, чтобы он мог перенаправлять связь на соответствующий клиент для пользователя A.  Для пользователей, которые уже установили сеансы в Skype для бизнеса с пользователем A, может потребоваться до 36 часов для того, чтобы клиенты Teams могли учитывать изменения в режимах, отличных от островов и до любого из режимов, в которых вы можете работать в любом из них.   Напротив, изменения существующего пользователя в режим Теамсонли будут обнаружены другими клиентами в течение 2 часов.

После того как администраторы будут готовы, они смогут одновременно перемещаться между абонентом, вызовом и планированием собраний для определенного пользователя в Teams, обновляя режим пользователя на Теамсонли.  

Кроме того, администратор может сначала сдвинуть только планирование собраний в Teams, в то время как в Skype для бизнеса не покидая функций чата и звонков, используя режим Сфбвистеамсколлабандмитингс. Этот режим позволяет организациям переходить в Teams для собраний — если пользователи пока еще не готовы к переходу в режим Теамсонли (как правило, это может потребоваться для миграции существующих функций PSTN). Этот переходный сценарий в [первую очередь называется собранием](meetings-first.md).

## <a name="tools-for-managing-the-upgrade"></a>Инструменты для управления обновлением

Для любого из описанных выше методов администратор управляет переходом к Теамсонли с помощью [теамсупградеполици](https://docs.microsoft.com/powershell/module/skype/grant-csteamsupgradepolicy?view=skype-ps), который управляет режимом сосуществования пользователя. Дополнительные сведения о каждом из режимов можно найти в разделе [режимы сосуществования](migration-interop-guidance-for-teams-with-skype.md#coexistence-modes).


, Осуществляет ли администратор управляемый переход с помощью режимов Skype для бизнеса или просто переходите к режиму Теамсонли из стандартной конфигурации, Теамсупградеполици является основным инструментом.  Как и любые другие политики в Teams, Теамсупградеполици можно назначать непосредственно пользователю, и его также можно задать в качестве значения по умолчанию на уровне клиента. Любое назначение пользователю имеет приоритет над параметром клиента по умолчанию.  Управлять ими можно как на консоли администрирования Teams, так и в PowerShell.

Администраторы могут назначать пользователям режим Теамсупградеполици в Skype для бизнеса Online или локальной среде, за исключением того, что режим Теамсонли можно назначить только пользователю, уже подключенному к Skype для бизнеса Online. <sup>4</sup>  

Пользователи с учетными записями Skype для бизнеса, расположенные в локальной среде, [должны быть перемещены в Интернет](https://docs.microsoft.com/SkypeForBusiness/hybrid/move-users-from-on-premises-to-teams) (как в Skype для бизнеса Online, так и прямо в Teams) с помощью функции Move-CsUser в локальном наборе инструментов Skype для бизнеса. Эти пользователи могут быть перемещены в Теамсонли в одном или двух шагах:

-   1 шаг: укажите параметр-Моветотеамс в разделе Move-CsUser. Для этого требуется Skype для бизнеса Server 2019 или Skype для бизнеса Server 2015 с обновления HF1.

-   2 шага: после запуска Move-CsUser, предоставьте пользователю режим Теамсонли, используя Теамсупградеполици.

В отличие от других политик, нельзя создавать новые экземпляры Теамсупградеполици в Office 365. Все существующие экземпляры встроены в службу.  (Обратите внимание, что Mode является свойством в Теамсупградеполици, а не именем экземпляра политики.) В некоторых случаях имя экземпляра политики является таким же, как и в случае с режимом. В частности, чтобы назначить пользователю режим Теамсонли, необходимо предоставить этому пользователю экземпляр "Упградетотеамс" Теамсупградеполици. Чтобы просмотреть список всех экземпляров, вы можете выполнить следующую команду:

```
Get-CsTeamsUpgradePolicy|ft Identity, Mode, NotifySfbUsers
```

Чтобы обновить пользователя Online до режима Теамсонли, назначьте экземпляр "Упградетотеамс": 

```
Grant-CsTeamsUpgradePolicy -PolicyName UpgradeToTeams -Identity $user 
```

Чтобы обновить локальный пользователь Skype для бизнеса до режима Теамсонли, воспользуйтесь функцией Move-CsUser в локальном наборе инструментов.

```
Move-CsUser -identity $user -Target sipfed.online.lync.com -MoveToTeams -credential $cred
```

Чтобы изменить режим для всех пользователей в клиенте, за исключением тех, у которых есть явное разрешение пользователя (приоритет), выполните следующую команду:

```
Grant-CsTeamsUpgradePolicy -PolicyName SfbWithTeamsCollab -Global
```


>[!NOTE]
>Если у вас есть пользователи с локальными учетными записями Skype для бизнеса, вы не должны назначать режим Теамсонли на уровне клиента, если вы явно не назначаете какой-либо другой режим для всех пользователей, использующих локальные учетные записи Skype для бизнеса.

**Примечание:**
<sup>4</sup> это связано с тем, что взаимодействие с пользователями и федерациями Skype для бизнеса возможно только в том случае, если пользователь размещен в Skype для бизнеса Online.

### <a name="using-notifications-in-skype-for-business-clients"></a>Использование уведомлений в клиентах Skype для бизнеса

Администраторы могут предоставить уведомления конечных пользователей в клиенте Skype для бизнеса, чтобы уведомить пользователей о том, что они вскоре будут обновлены до Teams, как показано на следующей схеме. Например, за неделю до того, как администратор планирует обновить группу пользователей до Теамсонли режима, администратор может попытаться включить эти уведомления для этой группы пользователей. Эти уведомления включены с помощью экземпляра Теамсупградеполици с Нотифисфбусерс = true.  Для всех режимов, кроме Теамсонли, существует два экземпляра на каждый режим, соответствующие двум значениям Нотифисфбусерс.  Для всех режимов, кроме Теамсонли, существует два экземпляра на каждый режим, соответствующие двум значениям Нотифисфбусерс. 

![Диаграмма, на которой показаны уведомления](media/teams-upgrade-sfb-with-notifications.png)

Если пользователи находятся в Skype для бизнеса Online, просто назначьте экземпляр политики, имеющий тот же режим, что и у пользователя, но с помощью Нотифисфбусерс = true. 

Если пользователи находятся в локальной среде Skype для бизнеса Server, вам потребуется использовать локальный набор инструментов, и вам понадобится Skype для бизнеса Server 2019 или обновления HF1 для Skype для бизнеса Server 2015. В локальном окне PowerShell создайте новый экземпляр Теамсупградеполици с Нотифисфбусерс = true:

```
New-CsTeamsUpgradePolicy -Identity EnableNotification -NotifySfbUsers $true
```

Затем в том же локальном окне PowerShell Назначьте новую политику для нужных пользователей:

```
Grant-CsTeamsUpgradePolicy -Identity $user -PolicyName EnableNotification
```


### <a name="meeting-migration"></a>Миграция собрания

Когда пользователь переносится в режим Теамсонли, существующие собрания Skype для бизнеса по умолчанию будут преобразованы в Teams. Вы можете дополнительно отключить поведение по умолчанию при назначении пользователю режима Теамсонли. При перемещении пользователей из локальной сети собрания должны быть перенесены в облако для работы с учетной записью пользователя в Интернете, но если не указать параметр-Моветотеамс, собрания будут переноситься в качестве собраний Skype для бизнеса, а не преобразуются в Teams. 

При назначении режима Теамсонли на уровне клиента миграция собрания не запускается ни для каких пользователей. Если вы хотите назначить режим Теамсонли на уровне клиента и перенести собрания, вы можете использовать PowerShell для получения списка пользователей в клиенте (например, с помощью команды Get-CsOnlineUser с необходимыми фильтрами), а затем пройти каждый из этих пользователей для запуска собрания. Миграция с помощью Start-Ксексмитингмигратион. Подробности можно найти в разделе [Использование службы миграции собраний (MMS)](https://docs.microsoft.com/skypeforbusiness/audio-conferencing-in-office-365/setting-up-the-meeting-migration-service-mms).


### <a name="additional-considerations-for-organizations-with-skype-for-business-server-on-premises"></a>Дополнительные рекомендации для организаций с локальными приложениями Skype для бизнеса Server

- Настройка гибридного развертывания Skype для бизнеса является обязательным условием для перехода в режим Теамсонли. Несмотря на то что вы можете использовать Teams в режиме "острова" без гибридного развертывания, переход в режим Теамсонли невозможно, пока пользователь не перейдет из Skype для бизнеса локально в Skype для бизнеса Online (с помощью функции [Move-CsUser](https://docs.microsoft.com/SkypeForBusiness/hybrid/move-users-between-on-premises-and-cloud)). Дополнительные сведения можно найти в разделе [Настройка гибридного подключения](https://docs.microsoft.com/skypeforbusiness/hybrid/configure-hybrid-connectivity).

- Пользователи Teams, у которых есть учетная запись Skype для бизнеса, локально (то есть они еще не были перемещены в облако с помощью функции Move-CsUser), не могут взаимодействовать с пользователями Skype для бизнеса, а также не могут использовать их вместе с внешними пользователями. Эта функция доступна только в том случае, если пользователи перемещаются в облако (как в режиме острова, так и в качестве пользователей Теамсонли). 

- Если у вас есть пользователи с локальными учетными записями Skype для бизнеса, вы не должны назначать режим Теамсонли на уровне клиента, если вы явно не назначаете какой-либо другой режим для всех пользователей, использующих локальные учетные записи Skype для бизнеса. 

- Вы должны убедиться, что ваши пользователи правильно синхронизируются с Azure AD с правильными атрибутами Skype для бизнеса. Эти атрибуты являются всеми префиксами с помощью "msRTCSIP-". Если пользователи не будут правильно синхронизованы с Azure AD, средства управления в Teams не смогут управлять этими пользователями. Дополнительные сведения можно найти в разделе [Настройка Azure AD Connect для Teams и Skype для бизнеса](https://docs.microsoft.com/SkypeForBusiness/hybrid/configure-azure-ad-connect).

- Чтобы создать в гибридной организации нового пользователя Теамсонли или Skype для бизнеса Online, *необходимо сначала включить пользователя в локальной среде Skype для бизнеса Server*, а затем переместить пользователя из локальной сети в облако с помощью функции Move-CsUser.  Предварительное создание пользователя в локальной системе гарантирует, что все остальные пользователи Skype для бизнеса смогут перенаправляться на только что созданному пользователю. После того как все пользователи будут перемещены в сеть, вам больше не нужно, чтобы они были включены в локальную версию.

- Когда пользователь перемещается из локальной сети в облако, собрания, упорядоченные по этому пользователю, переносятся в Skype для бизнеса Online или Teams, в зависимости от того, указан ли параметр-Моветотеамс.

- Если вы хотите, чтобы уведомления отображались в клиенте Skype для бизнеса для локальных пользователей, необходимо использовать Теамсупградеполици в локальном наборе инструментов. Для локальных пользователей применим только параметр Нотифисфбусерс.  Локальные пользователи получают доступ к своим режимам из Интернет-экземпляров Теамсупградеполици. Ознакомьтесь с заметками в [Grant-кстеамсупградеполици](https://docs.microsoft.com/powershell/module/skype/grant-csteamsupgradepolicy?view=skype-ps). 

>[!NOTE]
> Все новые клиенты, созданные после сентября 3, 2019 создаются как клиенты Теамсонли, не имея возможности понизить уровень прав администратора. В организациях, исучетных в локальной версии Skype для бизнеса, которые раньше никогда не получали подписку на Office 365 до сентября 3, 2019 вам потребуется обратиться в службу поддержки Майкрософт, чтобы понизить уровень клиента, после того как они получат подписку с помощью Office 365. 


## <a name="perform-the-upgrade-for-your-organization"></a>Выполнение обновления для своей организации

В этом разделе описаны следующие варианты обновления:

- Параллельное обновление (с использованием режима острова)
- Управляемое обновление для Организации, которое еще не было запущено в Teams
- Управляемое обновление для Организации, уже использующей Teams в режиме "острова"

### <a name="side-by-side-upgrade-using-islands-mode"></a>Параллельное обновление (с использованием режима острова)

Для параметра "параллельное обновление":

- Используйте этот вариант, если вы можете быстро выполнить обновление всей Организации.  Так как есть потенциальная опасность путаницы при работе с обоими клиентами, лучше всего уменьшить этот период времени. Вы должны убедиться, что ваши пользователи знают, что они работают и на обоих клиентах.

- Этот параметр является встроенной моделью и не требует вмешательства администратора, чтобы приступить к работе с группами за исключением назначения лицензии на Office 365. Если у пользователей уже есть Skype для бизнеса Online, возможно, эта модель уже есть.

- Это может стать затруднительным выходить из режима параллельных и переходить в Теамсонли. Так как обновленные пользователи обмениваются данными только с помощью Teams, любые другие пользователи в Организации, взаимодействующие с этим пользователем, должны использовать Teams.  Если у вас есть пользователи, которые не начали работать с группами, они будут открыты для отсутствующих сообщений. Кроме того, пользователи Теамсонли в Skype для бизнеса не увидят пользователей в Интернете. В некоторых организациях для того, чтобы избежать этого, необходимо выполнить обновление на уровне клиента с помощью глобальной политики клиента, но при этом требуется подождать, пока все пользователи не будут готовы к обновлению.


### <a name="a-managed-upgrade-for-an-organization-that-has-not-yet-started-using-teams"></a>Управляемое обновление для Организации, которое еще не было запущено в Teams

Если в вашей организации еще нет активных пользователей в Teams, сначала необходимо настроить политику клиента по умолчанию для Теамсупградеполици на один из режимов Skype для бизнеса, например Сфбвистеамсколлаб.  Пользователи, которые еще не начали работать с командами, не заметят никакого отличия от поведения. Тем не менее, установка этой политики на уровне клиента позволяет начать обновление пользователей до режима Теамсонли и гарантирует, что обновленные пользователи смогут взаимодействовать с необновленными пользователями.  После того как вы определили, что ваши пилотные пользователи смогут обновить их до Теамсонли.  Если они являются локальными, используйте Move-CsUser. Если они подключены к Интернету, просто назначьте им режим Теамсонли с помощью Теамсупградеполици.  По умолчанию в Teams будут перенесены собрания Skype для бизнеса, запланированные этими пользователями.

Ниже приведены основные команды.

1. Установите для уровня клиента по умолчанию режим Сфбвистеамсколлаб, как описано ниже.

   ```
   Grant-CsTeamsUpgradePolicy -PolicyName SfbWithTeamsCollab -Global
   ```

2. Обновите пользователя до Теамсонли следующим образом:

   - Если пользователь уже подключен к сети:

     ```
     Grant-CsTeamsUpgradePolicy -PolicyName UpgradeToTeams -Identity $username 
     ```

   - Если пользователь является локальным:

     ```
     Move-CsUser -identity $user -Target sipfed.online.lync.com -MoveToTeams -credential $cred 
     ```

Notes
 
- Вместо того чтобы задать для клиента политику Сфбвистеамсколлаб, вы можете установить для нее значение Сфбвистеамсколлабандмитингс. Это приведет к тому, что все пользователи смогут планировать все новые собрания в Teams.
- Move-CsUser является командлетом в локальных средствах. Для переключателя Моветотеамс требуется Skype для бизнеса Server 2019 или Skype для бизнеса Server 2015 с обновления HF1. - По умолчанию собрания Skype для бизнеса переносятся в Teams при переходе в режим Теамсонли или при назначении Сфбвистеамсколлабандмитингс режима.

### <a name="a-managed-upgrade-for-an-organization-that-is-already-using-teams-in-islands-mode"></a>Управляемое обновление для Организации, уже использующей Teams в режиме "острова"

Если некоторые пользователи в вашей организации активно используют Teams в режиме острова, вам, скорее всего, не нужно удалять функциональность существующих пользователей. Таким образом, перед изменением политики на уровне клиента требуется дополнительный шаг. Решение — это "грандфасер", которые были активны активными пользователями Teams в режиме "острова", прежде чем приступить к настройке политики на уровне клиента в Сфбвистеамсколлаб.  После того как вы закончите это, вы можете перейти к развертыванию, как описано выше, но у вас есть две группы пользователей, которые переходят в Теамсонли: пользователи, которые были активны в Teams, будут находиться в режиме "острова", а остальные пользователи будут в режиме Сфбвистеамсколлаб. Вы можете последовательно переместить этих пользователей в Теамсонлиный режим.

1. Найдите пользователей, активных в Teams, как описано ниже.

   1. На портале администрирования Office 365 на левой панели навигации перейдите в раздел отчеты и использование. 
   2. В раскрывающемся списке "выберите отчет" выберите Microsoft Teams, а затем действия пользователя. Будет предоставлена экспортируемая таблица пользователей, которые были активны в Teams. 
   3. Нажмите кнопку Экспорт, откройте Excel и выберите пункт фильтр, чтобы отобразить только тех пользователей, которые активны в Teams.

2. Для каждого активного пользователя Teams, найденного на этапе 1, назначьте режим "острова" в удаленной оболочке PowerShell. Это позволяет перейти к следующему этапу и гарантировать, что вы не измените пользовательский интерфейс.  

   ```
   $users=get-content “C:\MyPath\users.txt” 
    foreach ($user in $users){ 
    Grant-CsTeamsUpgradePolicy -identity $user -PolicyName Islands} 
   ```

3. Установите для параметра политики на уровне клиента значение Сфбвистеамсколлаб:

   ```
   Grant-CsTeamsUpgradePolicy -Global -PolicyName SfbWithTeamsCollab 
   ```

4. Переход с выбранных пользователей в режим Теамсонли. Вы можете обновить пользователей в режиме "острова" или в Сфбвистеамсколлаб режиме, но может возникнуть необходимость в определении приоритета обновления пользователей в режиме острова, чтобы свести к минимуму вероятность путаницы, которая может возникнуть, если пользователи находятся в режиме "острова".   

   Для пользователей, размещенных в Skype для бизнеса Online:  

   ```
   Grant-CsTeamsUpgradePolicy -Identity $user -PolicyName UpgradeToTeams 
   ```

   Для пользователей, размещенных в локальной среде Skype для бизнеса Server, выполните указанные ниже действия.  

   ```
   Move-CsUser -Identity $user -Target sipfed.online.lync.com -MoveToTeams -credential $cred 
   ```

## <a name="considerations-for-pstn-calling"></a>Рекомендации по КОММУТИРУЕМой телефонной связи

Если задействована функция КОММУТИРУЕМого звонка, при переходе в Теамсонлиный режим возможны четыре варианта:

- *Пользователь в Skype для бизнеса Online с планом вызова Microsoft*. После обновления у этого пользователя будет по-прежнему содержаться план звонков по Microsoft.

- *Пользователь в Skype для бизнеса Online и локальная функция голосовой связи* через Skype для бизнеса в локальной или облачной версии. Переход пользователя на Teams необходимо координировать с помощью миграции пользователя в направлении прямой маршрутизации, чтобы убедиться в том, что у пользователя Теамсонли есть функция КТСОП.

- *Пользователь в локальной среде Skype для бизнеса с корпоративной голосовой связью, который будет переходить на Интернет и поддерживать локальную сеть PSTN*.  Для миграции этого пользователя в Teams требуется переместить локальную учетную запись пользователя Skype для бизнеса в облако, а также перенести его с помощью миграции пользователей на прямую переадресацию. 

- *Пользователь в локальной среде Skype для бизнеса с корпоративной голосовой*связью, который будет переноситься в Интернет и с помощью плана звонков Microsoft.  Для миграции этого пользователя в Teams требуется переместить локальную учетную запись пользователя Skype для бизнеса в облако, а также указать, что вы можете переместиться с помощью A) на номер телефона этого пользователя в план абонента Майкрософт или B) назначения нового номера абонента из Доступные регионы.

В этой статье приводятся общие сведения только о высоком уровне.  Дополнительные сведения можно найти в разделе планы прямой маршрутизации и [звонков](calling-plan-landing-page.md)на [телефонную систему](direct-routing-landing-page.md) . Кроме того, обратите внимание на то, что использование телефонной системы в Teams поддерживается, только когда пользователь находится в режиме Теамсонли.  Если пользователь находится в режиме "острова", телефонная система поддерживается только в Skype для бизнеса. 

### <a name="from-skype-for-business-online-with-microsoft-calling-plans"></a>Из Skype для бизнеса Online с тарифами Майкрософт 

Это простейший сценарий обновления, включающий голосовую связь. 

1. Убедитесь, что пользователям назначена лицензия на команды Teams. По умолчанию при назначении лицензии на Office 365 Teams включается, но если вы ранее не отключили лицензию Teams, никаких действий не требуется.

2.  Если у пользователей уже есть план звонков Microsoft с номером телефона, необходимо изменить его только в том случае, если вы применяете режим Теамсонли пользователя в Теамсупградеполици.  Перед назначением режима Теамсонли входящие звонки PSTN будут поступать в клиенте Skype для бизнеса пользователя. После перехода на режим Теамсонли входящие звонки PSTN будут помещаются на клиенте Teams пользователя.  

### <a name="from-skype-for-business-online-with-on-premises-voice"></a>Из Skype для бизнеса Online с использованием локального голоса

В этом сценарии пользователь уже входит в Skype для бизнеса Online, но подключение по протоколу PSTN является локальным либо с помощью Skype для бизнеса Server в гибридном режиме, либо из облачного соединителя Edition. Миграция этих пользователей в режим Теамсонли с помощью функции PSTN означает, что она позволяет осуществлять прямую маршрутизацию, в которой КОММУТИРУЕМые магистрали подключаются непосредственно к службе прямой маршрутизации в облаке с помощью локального контроллера (SBC).

Это можно сделать в любое время, но если вы хотите, чтобы пользователи правильно настроили конфигурацию, лучше сделать это перед тем, как пользователь будет обновлен до Теамсонли Mode.

   - Онлиневоицераутингполици (в зависимости от локального Воицераутингполици, если вы переносите пользователей из Skype для бизнеса Online, и в зависимости от того, Воицеполици ли вы миграция пользователей из локальной сети в корпоративную голосовую почту).
   - Онлинепстнусаже объекты (в соответствии с использованием локальной сети PSTN).

### <a name="manage-enterprisevoiceenabled-property-during-migration"></a>Управление свойством Ентерприсевоицеенаблед во время миграции 

У пользователя должно быть Ентерприсевоицеенаблед = true в Azure AD, чтобы пользователь имел возможность использовать функцию КТСОП.  Ентерприсевоицеенаблед ("EV-Enabled") — это свойство (не политика), которое существует как в локальном, так и в облачном каталогах. Значение в облаке — это то, что важно для Teams.  Точная логика использования режима EV определяется значением true зависит от следующего сценария: 

- Если пользователь поддерживает функцию EV в локальной версии Skype для бизнеса и пользователю назначена лицензия на телефонную систему до того, как вы перейдете в облако с помощью функции Move-CsUser, пользователь в сети будет поддерживаться с параметром EV-enabled = true. 

- Если существующему пользователю Теамсонли или Skype для бизнеса Online назначена лицензия на телефонную систему, по умолчанию для EV включена не задано значение "true".  Это происходит также в том случае, если локальный пользователь перемещается в облако до назначения лицензии на телефонную систему. В любом случае администратор должен указать следующий командлет: 

  ```
  Set-CsUser -EnterpriseVoiceEnabled $True 
  ```

### <a name="meetings"></a>Собрания

Независимо от их режима пользователи всегда могут присоединиться к любому собранию, на которое они приглашены, как в Skype для бизнеса, так и в рабочих группах.  Тем не менее пользователи должны присоединиться к собранию с помощью соответствующего клиента, который соответствует типу собрания:

- Если собрание — собрание Teams, все участники (Теамсонли, острова и пользователи Skype для бизнеса) смогут присоединиться к собранию с помощью клиента Teams. Если Teams не установлен, пользователь переходит в Интернет при попытке присоединиться к собранию.

- Если собрание — собрание Skype для бизнеса, все участники (будь то Теамсонли, острова или пользователи Skype для бизнеса) могут присоединиться к собранию с помощью клиента Skype для бизнеса. Если клиент Skype для бизнеса не установлен, пользователь перейдет в Интернет, чтобы присоединиться с помощью приложения "собрания Skype".

При Организации собраний тип собрания, по которому выполняется планирование, основывается на режиме организатора, как показано в приведенной ниже таблице.

| Режим организатора    |      Поведение |
| :------------------ | :---------------- |
| Теамсонли, Сфбвистеамсколлабандмитингс |    Все собрания, запланированные в Teams. Надстройка Skype для бизнеса недоступна в Outlook. | 
| Сфбвистеамсколлаб, Сфбонли   | Все собрания, запланированные в Skype для бизнеса. Надстройка Teams недоступна в Outlook. | 
| Блоки |     Собрания можно запланировать в Skype для бизнеса или в Teams. Обе надстройки доступны в Outlook. | 


### <a name="interoperability"></a>Взаимодействия

В некоторых сценариях Teams поддерживается взаимодействие ("взаимодействие") с помощью Skype для бизнеса. Связь взаимодействия — это чат или звонок между пользователем Skype для бизнеса и пользователем Teams.  Взаимодействие между приложениями возможно только между двумя пользователями; участие в чате и звонке с несколькими абонентами, а также добавление дополнительных пользователей не поддерживается.

Чат взаимодействия или звонок между двумя пользователями создается, когда истинно каждое из следующих условий:

- Один пользователь использует Teams, а другой — Skype для бизнеса.

- Режим получателя первоначальной связи не является островом (в противном случае связь будет налагаться на том же клиенте), если оба пользователя находятся в одной организации. В федеративных сценариях отправляющий пользователь использует команды, а получатель — не в режиме Теамсонли. 

- У пользователя Teams также нет учетной записи Skype для бизнеса, размещенной в локальной среде. 

В связи с взаимодействием взаимодействие — это просто текст. Кроме того, функция обмена файлами и скриншаринг невозможна *в чате*. Тем не менее пользователи в чате могут легко дополнить файл и (или) скриншаринг, создав собрание по запросу в рамках чата, как описано ниже.

- Если пользователь Teams пытается открыть общий доступ к своему экрану, собрание группы по запросу создается автоматически, а ссылка "пригласить на собрание" отправляется на клиент пользователя Skype для бизнеса. После нажатия этой ссылки пользователь Skype для бизнеса будет открывать группы и присоединиться к собранию. Оба пользователя теперь находятся в собрании Teams и могут предоставлять вам необходимые возможности.

- Если пользователь Skype для бизнеса использует клиент из 2018 или более поздней версии и пытается предоставить общий доступ к содержимому, будет автоматически создано собрание в Skype для бизнеса и ссылка "пригласить" на это собрание будет отправлена клиентскому пользователю Teams. После щелчка ссылки пользователь Teams попытается присоединиться к собранию Skype для бизнеса. Если на компьютере Teams установлен клиент Skype для бизнеса, он откроется и пользователю будет предложено выполнить вход (если он еще не вошел в систему).  Если у пользователя Teams не установлен клиент Skype для бизнеса, пользователю будет предложено использовать веб-версию. После того как оба пользователя вошли в систему, они будут собраны в Skype для бизнеса и их можно будет передавать нужным образом.

### <a name="teams-conversations---interop-versus-native-threads"></a>Беседы Teams — взаимодействие и собственные потоки

Поскольку связь по взаимодействию не поддерживает все возможности неуправляемой беседы в Teams, клиент Teams поддерживает отдельные потоки беседы для взаимодействия между группами и группами в Skype для бизнеса. Эти беседы отображаются по-разному в пользовательском интерфейсе: потоки взаимодействия можно отличать от обычных машинных потоков команд, выполнив указанные ниже действия.

- Отсутствие элементов управления для форматированного текста, файла или скриншаринг, невозможности добавить пользователей.
- Изменение значка целевого пользователя с отображением "S" для Skype для бизнеса.

### <a name="presence"></a>Присутствие

Сведения о присутствии для определенного пользователя основываются на действиях пользователя в службе через клиент. Сведения о присутствии затем публикуются для просмотра другими пользователями.  Skype для бизнеса и Teams представляют собой отдельные службы с отдельными клиентами, поэтому каждая служба имеет свое состояние присутствия пользователя.   Кроме того, существует синхронизация служб присутствия в Teams и Skype для бизнеса Online.  Это позволяет одной службе потенциально публиковать сведения о присутствии пользователя из другой службы (при необходимости). 

Поведение публикации при присутствии основано на пользовательском режиме. Существуют три основных случая.

- Если пользователь находится в режиме Теамсонли, все остальные пользователи видят сведения о присутствии в Teams для этого пользователя, независимо от того, какой клиент он использует.

- Если пользователь входит в какой-либо из режимов Skype для бизнеса, все остальные пользователи видят сведения о присутствии в Skype для бизнеса для этого пользователя, независимо от того, какой клиент он использует.

- Если пользователь находится в режиме "острова", присутствие, опубликованное в Skype для бизнеса и в Teams, независимо, поэтому присутствие, показанное для пользователей в пределах одной организации, будет зависеть от клиента другого пользователя. Пользователи в федеративных организациях увидят наличие этого пользователя в зависимости от активности Skype для бизнеса, так как федеративный трафик в режиме острова является пользовательским в Skype для бизнеса.

Например, предположим, что пользователь A находится в режиме "острова". Если пользователь A является активным в Teams, но не выполнил вход в Skype для бизнеса, другие пользователи увидят, что он является активным из своего клиента Teams, но в клиенте Skype для бизнеса они будут видеть пользователя а в автономном режиме. Это является особенностью, так как пользователь а не может связаться с ним, если он не работает в клиенте. 

Дополнительные сведения можно найти в разделе сведения о [присутствии](coexistence-chat-calls-presence.md#presence).

### <a name="federation"></a>Федерация

Для использования Skype для бизнеса в целях интеграции с другими пользователями необходимо, чтобы пользователь Teams был подключен к Интернету в Skype для бизнеса. Теамсупградеполици управляет маршрутизацией входящих федеративных чатов и звонков. Режим федеративного маршрутизации аналогичен сценариям для одного и того же клиента, за исключением случаев, когда он находится в режиме "острова". Если получатели находятся в режиме острова, сделайте следующее:

- Разговоры и звонки, инициированные из групп в Skype для бизнеса, если получатель находится в Федеративной клиенте.
- Разговоры и звонки, инициированные из Teams в Teams, если получатель находится в том же клиенте.
- Разговоры и звонки, инициированные из Skype для бизнеса, всегда помещаются в Skype для бизнеса.

Федеративного чата между пользователем Teams и Skype для бизнеса — это поток взаимодействия, поэтому невозможно сделать так, чтобы текст в формате RTF и общий доступ были недоступны. Пользовательский интерфейс представляет Федеративные разговоры сходным образом с потоками взаимодействия одного и того же клиента, за исключением замечаний, указывающих на то, что пользователь является внешним.

Когда команды впервые вводили Федерацию, в качестве федеративного чата между двумя пользователями Teams существовал также поток взаимодействия, но в будущем будет введена собственная Федерация Teams, обеспечивающая полную функциональность взаимодействия между пользователями в режиме Теамсонли. . 

Дополнительные сведения можно найти в разделе [Федеративные маршруты для новых чатов и звонков](coexistence-chat-calls-presence.md#federated-routing-for-new-chats-or-calls).

### <a name="contacts"></a>Контакты

В Teams и Skype для бизнеса есть отдельные списки контактов. Это означает, что добавление, удаление и изменение контактов, внесенных в одной системе, не синхронизируются с другой системой. Тем не менее, контакты из Skype для бизнеса автоматически копируются в Teams при возникновении одного из двух конкретных событий. 

- Для любого пользователя Skype для бизнеса Online при первом входе в Teams, контакты из Skype для бизнеса будут скопированы в Teams.  Это поведение недоступно для пользователей, у которых есть локальная учетная запись в Skype для бизнеса Server.  

- После того как пользователь будет обновлен до Теамсонли (с помощью назначения Теамсупградеполици или с помощью Move-CsUser-Моветотеамс), при следующем входе пользователя в Teams, существующие контакты в Skype для бизнеса будут объединены с существующими контактами, уже имеющимися в Teams. Это происходит в том случае, если учетная запись пользователя Skype для бизнеса размещена в локальной среде или в сети. 

В обоих случаях передача контактов из Skype для бизнеса в Teams выполняется асинхронно, поэтому она может быть несколько минут, прежде чем контакты отобразятся в Teams. Два приведенных выше события инициируют копию.
