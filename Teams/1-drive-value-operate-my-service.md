# <a name="operate-my-service"></a>Мои службы

В этой статье дается обзор требований для успешной работы облачные службы голосовой связи для вашей организации. С надлежащим образом операционной облачным службам голосовой связи, можно убедитесь, что более высокое качество, надежные возможности предоставляется для вашей организации.

## <a name="introduction-to-the-operations-guide"></a>Знакомство с руководством по работе

Руководство пользователя дается обзор все задачи и действия, необходимые как часть функция управления службой для групп Майкрософт.

Управление службами — это обширная тема, которая охватывает повседневные операции службы Microsoft Teams после ее развертывания и предоставления пользователям. Служба Teams включает в себя Microsoft Office 365 и инфраструктурные компоненты, развертываемые локально (например, сети).

Идея управления службами, скорее всего, не нова для большинства организаций. Может уже был реализован процессов и задач, связанных с существующими службами. С другой стороны, вероятно, можно дополнить текущих процессов, при планировании для управления службой сегодня для поддержки команд в будущем.

Служба управления включает все действия и процессов, участвующих в управлении группами сквозного. Как было указано ранее, некоторые компоненты службы управления — инфраструктуры, которая состоит из самой службы Office 365 — являются ответственность корпорации Майкрософт, в то время как вы клиента, отвечают пользователям управлять различных аспектов командами, сети и предоставлении конечных точек.

Задачи и действий в этом руководстве сгруппированы в восемь категории, как показано на следующем рисунке. Каждый из этих категорий будет развернуто на в следующих разделах.

### <a name="operational-role-mapping"></a>Сопоставление операционных ролей

Планирования, которое вы решили реализовать крупномасштабную для операций на этапе Предвидения крайне важна, так как действия операции начинается при включении первого пилотных пользователей. В этом руководстве приведены действия и задачи, которые должны быть выполнены на основе ежедневно, еженедельно, ежемесячно или по необходимости обслуживать развертывание групп высокое качество. В этом руководстве представлены базы знаний и рекомендации для выполнения этих задач и важных действий.

Один ключевой компонент успешного развертывания состоит в том, что, как и в начале Предвидения этап планирования включает определение пользователей, которые будут отвечать за для выполнения определенных действий. После развертывания применяется какие задачи и действия поняли, нужные для понимания и следуют групп или пользователей, которые нужно назначить их.

Каждой группы, которое указывается необходимо просмотреть и согласовать задачи и обязанности определяемую и запуск подготовки. Сюда может входить обучения и подготовки к, предоставляя обновлений персонал план или проверка того, что все готово для предоставления внешних поставщиков.

Действия и роли, определенные в данном руководстве должен быть допустимым большинству сценариев, но каждое развертывание групп должны быть уникальными; Таким образом в этом руководстве можно использовать в качестве отправной точки для настройки действия и роли по умолчанию в соответствии со своими потребностями.

Убедитесь, что каждый ответственность группы имеет хорошее понимание действия, необходимые для запуска службы. Очень важно, что каждой группы принимает и утверждает их передаче в вашей организации до начала первого пилотного проекта.

После соглашение будет готова, соответствующих групп должно начаться вводят их роли в действие.

### <a name="teams-service-dependencies"></a>Зависимости службы групп

Группами Майкрософт объединяет технологий в Office 365 для обеспечения работы сервера-концентратора. Примеры:

-   Azure Active Directory (Azure AD) предоставляет службы проверки подлинности и авторизации для групп.

-   Exchange Online предоставляет расширенные функции, такие как удержания по юридическим причинам и обнаружение электронных данных.

-   SharePoint Online предоставляет возможность совместного использования файлов в каналы и OneDrive для бизнеса обеспечивает механизм для общего доступа к файлам в частной беседы.

Организации могут также использовать существующие вложения в локальной инфраструктуре. Например существующих учетных записей Active Directory локальной можно использовать для проверки подлинности с помощью Azure AD подключение. Некоторые версии Exchange Server можно использовать вместо Exchange Online.

Эти технологии собираться для предоставления набора расширенными возможностями, совместной работы и intelligent коммуникаций для пользователей. В этом тесного взаимодействия является ключевым преимуществом групп, но также накопителя требования для управления службой для этих технологий.

В этом руководстве рассматривается основных областей для управления службой группы. Скорее всего у вас есть управления планы обслуживания на месте для вспомогательных технологий, работающее с группами. Если не, вам потребуется установить планы соответствующих прав службы управления для этих компонентов технологии (в локальной и online) также. Это будет гарантировать, что пользователи откройте для себя высокое качество и надежные возможности с группами.

Таким образом не менее важно наблюдения за работоспособностью зависимые службы.

## <a name="manage-organizational-change"></a>Управление организационных изменений

Группами Майкрософт — это облачная служба. Появилась возможность предоставления новые возможности и функции быстрого слайдами. Инновации, обеспечивающая предоставляет очевидных преимуществ для организаций, но эти изменения должны быть надлежащим образом в вашей организации, чтобы избежать сопротивление пользователя или эскалации в службу поддержки.

Обновления для групп автоматически создаются для пользователей. Пользователи всегда будут иметь последние клиента и функции, доступные в службе группами. Невозможно управлять развертыванием обновлений группам пользователей, поэтому очень важно для управления изменениями через эффективной связи, обучение и внедрения программы. Если пользователи находятся в сообщении об изменении, обоснованные о преимуществах и возможность использовать новые возможности&mdash;они сможете адаптировать быстрее и Добро пожаловать изменения.

### <a name="monitoring-for-change"></a>Отслеживание изменений

Первым шагом для управления изменениями является мониторинга изменений, запланированных для группы. Мониторинг эти изменения наилучшего источника — это [План Office 365](https://products.office.com/business/office-365-roadmap), которой перечислены функции, которые в настоящее время разработки, предназначенных для развертывания для клиентов, или полностью запуска. Можно выполнить поиск функции, относящиеся к группам с помощью фильтров, или вы можете загрузить плану в файле Excel для анализа. Для каждого компонента схема содержит краткое описание, а также ожидаемые выхода.

В этом [блоге группы Microsoft](https://techcommunity.microsoft.com/t5/Microsoft-Teams-Blog/bg-p/MicrosoftTeamsBlog)могут узнать о оптимальные методы, тенденции и новости об обновлениях продукта группами. Нужны для поиска обновлений основных компонентов группам здесь указано позднее. Можно также подписаться на блог через RSS-канала. Затем можно добавить [RSS-канал,](https://techcommunity.microsoft.com/gxcuf89792/rss/board?board.id=MicrosoftTeamsBlog) непосредственно в канал команд, поэтому все важные новости доставляется непосредственно из групп.

Все функции, которые будут отпущены описаны в [Заметки о выпуске для групп Майкрософт](https://support.office.com/article/Release-notes-for-Microsoft-Teams-d7092a6d-c896-424c-b362-a472d5f105de).
Здесь можно найти список функций, которые были выпущены рабочего стола, веб-и мобильных устройств. Один и тот же набор заметки о выпуске, также доступны на вкладке Заметки о выпуске [Microsoft команды T-программы-робота](t-bot.md).

-   **После основных этапов**: сбор отзывов на протяжении жизненного цикла проекта, можно определить допустимое пользователей пользователя на постоянной основе и внесите необходимые изменения. Эта функция особенно полезна после основных этапов.

-   **Ongoing**: продолжить для измерения пользователей пользователя неопределенное время. ### <a name="nps"></a>СЕРВЕР ПОЛИТИКИ СЕТИ 

NET просто продавать счета (ТРУБЫ) — это показатель постоянных industrywide клиента и рекомендуется придерживаться следующего подхода использовать для оценки пользователей пользователя. Сервер политики сети, можно рассчитать с двумя вопросов: «вероятность вы рекомендовать группами коллегой?», «Почему?» следуют freeform вопрос

Сервер политики сети представляет собой индекс, от – 100 до 100, измеряет желание клиента, рекомендуем продукта или услуги компании. Сервер политики сети основано на анонимный опрос, который отправляется пользователям с помощью электронной почты или других электронных средств. Сервер политики сети измеряет постоянных между поставщиком и потребителем. Он состоит из только один вопрос, который запрашивает у пользователей для оценки их впечатления от 1 до 10 с возможность предоставления дополнительных комментариев. Пользователи затем классифицируются по следующие классы:

-   9 или 10, Promoters: надежных технологий, которые будут повышение уровня службы и сделают развитие другим пользователям.

-   7 или 8, пассивный: выполнены, но unenthusiastic, уязвимы к другой службы или предложения.

-   От 1 до 6, Detractors: Разочарованных клиентов, которые могут повреждения от услуг и снизить роста.

## <a name="assess-and-ensure-quality"></a>Оценка и обеспечения качества 

Все организации должны группы или отдельного быть ответственность за качества. Это самая важная роль в управлении службами. Известный качества роль назначается пользователь или группа, кто отношусь взаимодействия своих пользователей.
Для этой роли нужны навыки по выявлению тенденций в среде, а также стремление сотрудничать с другими командами для принятия корректирующих мер. Оптимальным кандидатом на роль лидера по качеству обычно является владелец службы клиента. В зависимости от размера организации и сложности это может быть любой пользователь или группа с страсть по обеспечению высокого качества пользовательского интерфейса.

Известный качества использует имеющихся средств и задокументировать процессов, таких как вызов качества панели мониторинга (CQD) и качества из взаимодействия руководство по решению, отслеживать взаимодействие с пользователем, определение тенденций качества, а также исправление при необходимости.
Известный качества должны работать с соответствующих групп на диске исправлению действия и отчет о хода выполнения и все открытые проблемы для исполнительного комитета.

[Качество из взаимодействия руководство по решению](https://aka.ms/qerguide) содержит действия, которые оценки и даются рекомендации по исправлению в ключевых областях, которые имеют наибольшее влияние на повышение взаимодействие с пользователем. В центре внимания с помощью CQD Online в качестве основного средства для регистрации и исследовать каждой области с фокусом на аудио для обеспечения максимальной воздействия и внедрения рекомендациями, изложенными в руководство по решению для качества взаимодействия. Любые оптимизации для сети, направленные на улучшение звукового взаимодействия, также вносят вклад в улучшение видеосвязи и совместного доступа к рабочему столу.

Мы настоятельно рекомендуем на раннем этапе используют известный качества. После назначить им следует запустить ознакомиться с содержимым в Quality of качества руководство по решению и связанные обучающие материалы.

## <a name="manage-endpoints"></a>Управление конечными точками

Конечные точки группы Microsoft может быть определен как ПК, Mac, планшетном ПК или мобильных устройств (или другие) устройство с запущенным продуктом клиентских групп. Термин *Конечная точка* не только включает в себя устройство самого, но как пользователь подключается к устройства — например, с помощью устройства встроенных микрофона или динамика, earbuds или оптимизированный гарнитуры. После они развернуты, конечные точки должны ли. Конечные точки команды требуется постоянная внимания и обслуживания. В следующих разделах конкретных необходимо уделить особое внимание.

### <a name="endpoint-requirements"></a>Требования к конечных точек

Одним из основных преимуществ группы — это, что клиент будет храниться в актуальном состоянии автоматически. Клиенты на компьютере и Mac обновляются с помощью фонового процесса, который проверяет наличие новых сборок и скачивает новый клиент при простое приложения. Группы мобильных приложений сохраняются в прежнем текущего через их хранилища соответствующего приложения.

Клиент группы имеет минимальные требования в отношении базовой платформы программного обеспечения. Эти требования может изменяться со временем и поэтому важно отслеживать их для изменения. Например клиента групп имеется версия минимальные операций ввода-вывода. Если клиент использует веб-браузер, браузер должен поддерживаться текущего также. Список поддерживаемых платформ можно найти в [Начало клиентов для групп Майкрософт](get-clients.md).

### <a name="endpoint-firewalls"></a>Брандмауэры конечных точек

Клиентские брандмауэры могут оказывать значительное негативное влияние на взаимодействие с пользователем.
Брандмауэры со стороны клиента могут повлиять на качество звонка и даже не звонка созданию. После настройки соответствующих исключения на брандмауэре клиентского компьютера, они должны быть сохранены в актуальном состоянии на основе данных в [Office 365 URL-адреса и диапазоны IP-адресов](https://aka.ms/o365ips). Независимых поставщиков будут иметь конкретные инструкции для обновления исключения.

### <a name="wi-fi-drivers"></a>Драйверы Wi-Fi

Драйверы Wi-Fi может вызывать трудности. Например драйвер может быть очень агрессивные поведение перемещения между точками доступа, можно вызвать ненужных-точка доступа переключение, приводя к вызовов плохого качества. Неэффективный драйвер Wi-Fi может обнаруживать через из интерфейса анализа качества (см [Quality of качества руководство по решению](https://aka.ms/qerguide) для более подробные сведения). Очень важно для реализации на основе качества процесс, который отслеживает новые драйверы Wi-Fi и гарантирует, что они в случае протестировано до развертывания для заполнения общие пользователя.

### <a name="endpoint-management"></a>Конечная точка управления

Каталог поддерживаемые конечные точки и интерфейса устройства (например, наушники) должно быть доступно и обслуживается. В этом каталоге будет включать список утвержденных устройств, которые были выбраны и проверки в рамках Предвидения и Onboard этапов. Как правило определенных устройствах выбраны для каждого типа пользователя в вашей организации в соответствии с требованиями атрибуты этого пользователя. Все конечные точки имеют жизненного цикла и необходимо управлять поставщика контракты, гарантия замены, рассылки и восстановить политики, связанные с помощью этих устройств.

### <a name="endpoint-troubleshooting"></a>Устранение неполадок в конечной точки

Даже если выполнения предыдущей рекомендации, пользователи в вашей организации по-прежнему столкнуться проблемы, связанные с группами. Несмотря на то, что проблема не может быть с конечной точкой самого, признаки проблемы обычно размещаются через клиента к пользователю. Следующие рекомендации предназначен для предоставления общие действия, которые можно предпринять для устранения проблемы; еще раз быть полное руководство по устранению неполадок. Эти действия представлены в определенном порядке, но они не нужно следовать явным образом и может быть неприменимо, в зависимости от природы проблемы.

1.  **Проверки работоспособности службы:** Проблемы, которые пользователь может возникнуть могут относиться к событие, которое негативно влияет на службу группами или зависимых служб. В качестве первого шага мы рекомендуем, что вы, убедитесь, что отсутствуют проблемы в активной службы. Обратитесь к [проверки работоспособности службы Office 365](https://support.office.com/article/How-to-check-Office-365-service-health-932AD3AD-533C-418A-B938-6E44E8BC33B0).
    Не забудьте проверить состояние зависимые службы (например, Exchange, SharePoint, OneDrive для бизнеса). Мониторинг работоспособности службы является более подробно рассматриваются в в предыдущем разделе, [Наблюдение за работоспособностью системы службы](#monitor-service-health).

2.  **Проверить возможность подключения клиента:** Проблемы с подключением вызвать функциональные возможности или неполадок входа в группах. Мы рекомендуем (особенно для новых сайтов или расположения) проверить возможность подключения к службе. Убедитесь, что следует следующие рекомендации [Office 365 URL-адреса и диапазоны IP-адресов](https://aka.ms/o365ips) для каждого сайта. Можно использовать [Средство оценки производительности сети Microsoft](https://www.microsoft.com/download/details.aspx?id=53885) , чтобы выполнить проверку подключения для проверки, что портов мультимедиа открытых правильно для возможности голосовой связи облака. Подробное описание действий по запуску тесты подключения, предоставляются в руководстве по [подготовке сети](3-envision-evaluate-my-environment.md#network-readiness) .

3.  **Контрольный список известные проблемы:** Обратитесь к [список известных проблем для групп](known-issues.md) , чтобы определить, является ли пользователь был негативное воздействие на один из этих проблем. Следуйте обходной путь, предоставляемые (если есть) для устранения этой проблемы.

4.  **Посетите сообщества группами Майкрософт:** [Сообщество групп Майкрософт](https://techcommunity.microsoft.com/t5/Microsoft-Teams/ct-p/MicrosoftTeams) предлагает выделенного пробелов для групп. Сообщество групп предоставляет список обсуждений, блогов и объявления центральным для группы. Можно задать вопрос или поиск предыдущих обсуждения для решения проблемы.

5.  **Обратиться в службу поддержки Майкрософт:** Можно обратиться в службу технической поддержки Майкрософт проблемы, связанные с группами через Интернет или по телефону. [Контакт поддержка групп Майкрософт](https://support.office.com/article/Contact-support-for-business-products-Admin-Help-32a17ca7-6fa0-4870-8a8d-e25ba4ccfd4b)сведения см.
    Для Premier клиентов, поддержка запросов может быть инициирован следуя инструкциям в [обратиться в службу поддержки для групп Майкрософт (премьер-клиенты)](https://support.microsoft.com/premier/contacts).

### <a name="license-management"></a>Управление лицензиями

Как ваша организация растет или сокращения, важно планирование лицензирования для текущие и будущие потребности. Нет базового лицензии групп, в дополнение к лицензирования для облачных голосовые возможности ([Телефонной системой](https://docs.microsoft.com/SkypeForBusiness/what-is-phone-system-in-office-365/here-s-what-you-get-with-phone-system) и [Конференц-связи звук](https://products.office.com/skype-for-business/audio-conferencing)).

Для групп телефонной системой лицензий требуется [Вызов планирование](https://docs.microsoft.com/skypeforbusiness/what-are-calling-plans-in-office-365/what-are-calling-plans-in-office-365) лицензии. Вызов план лицензирования позволяет выполнение и прием внутренних и/или международных телефонных звонков. Эти планы основаны на использование и иметь minute пулов, связанных с ними. Подготовка [Кредитов Communications](https://docs.microsoft.com/SkypeForBusiness/skype-for-business-and-microsoft-teams-add-on-licensing/what-are-communications-credits) будет убедитесь, что никогда не запускать из обслуживания.

Аудиоконференции обеспечивает tolled конференц-связи и службы конференц-связи для внутренних подключение по телефонной линии. Бесплатный номер конференц-связи или данными по телефону сценариев может привести к несут дополнительная плата для каких [Кредитов Communications](https://docs.microsoft.com/SkypeForBusiness/skype-for-business-and-microsoft-teams-add-on-licensing/what-are-communications-credits) являются обязательными.

Кредитов коммуникации можно дополнить лицензий на вызов планирование и звук конференц-связи. Вызов планирование лицензий и кредитов связи основаны на использование и поэтому необходимо отслеживать и соответствующим образом подготовлены для.

Вы можете использовать возможности [отчет об использовании ТСОП](https://docs.microsoft.com/skypeforbusiness/skype-for-business-online-reporting/pstn-usage-report) для наблюдения за использования вызов планирование минут и кредитов коммуникаций. На основе результатов этого действия, можно настроить на лицензирование соответствующим образом. Ожидается в ближайшее время будет предлагать отчета [minute пулы ТСОП](https://docs.microsoft.com/skypeforbusiness/skype-for-business-online-reporting/pstn-minute-pools-report) для более эффективного помощи выполнение этой операции.

### <a name="telephone-number-management"></a>Управление телефонных номеров

Существует два метода для получения номера в группах: можно перенести телефонных номеров от другого поставщика или вы можете подготовить номеров непосредственно из номеров инвентаризации корпорации Майкрософт. Оба метода описаны в [начало номера телефонов для пользователей](https://docs.microsoft.com/skypeforbusiness/what-are-calling-plans-in-office-365/getting-phone-numbers-for-your-users).

Существует ограничение на номер номера телефонов, которые вы можете подготовить из номеров запасов корпорации Майкрософт. Ограничения зависит от нескольких факторов, описанные в [Сколько номеров телефонов можно получить?](https://docs.microsoft.com/skypeforbusiness/what-are-calling-plans-in-office-365/how-many-phone-numbers-can-you-get).
Ограничения зависят от типа номера — международную без использования службы, платные службы номера и номеров подписчика (пользователь). Каждый имеет свои собственные ограничения и должно осуществляться независимо друг от друга. Если вы близко к ограничение (или Превышен предел), можно применять для шаг значения. Этот процесс описан в статье выше.

Может возникнуть при номер недоступен подготавливается в области, где служба недоступна. Сведения о процессе для запрашивающей номеров содержатся в разделе [Управление номера телефонов для вашей организации](https://docs.microsoft.com/skypeforbusiness/what-are-calling-plans-in-office-365/manage-phone-numbers-for-your-organization/manage-phone-numbers-for-your-organization).

[Качество из взаимодействия руководство по решению](https://aka.ms/qerguide) включают набор действий, которые оценки и даются рекомендации по исправлению в ключевых областях, которые имеют наибольшее влияние на улучшение взаимодействия с пользователем, как показано ниже.

![Ключевые области для проверки во время проверки качества Quality of: аудио-, надежности и результатах опроса пользователя.] (media/plan-my-service-management-image2.png "Ключевые области для проверки во время проверки качества Quality of: аудио-, надежности и результатах опроса пользователя.")

Постоянно вызов и устранению области, описанные в руководстве по, позволит сократить их потенциально может отрицательно сказаться на взаимодействие с пользователем. Большинство встречающихся в развертывании проблем с взаимодействием с пользователем можно разделить на следующие категории:

-   Неполная настройка брандмауэра или прокси-сервера

-   Слабый охват сети Wi-Fi

-   Недостаточная пропускная способность

-   VPN

-   Использование неоптимальных или встроенных звуковых устройств

-   Проблемы с подсетями или сетевыми устройствами

В центре внимания использование Online вызова панели мониторинга качества (CQD) в качестве основного средства для регистрации и исследовать каждой области, описанного с фокусом на аудио для обеспечения максимальной воздействия и внедрения рекомендациями, изложенными в Quality of качества руководство по решению. Любые оптимизации для сети, направленные на улучшение звукового взаимодействия, также вносят вклад в улучшение видеосвязи и совместного доступа к рабочему столу.

Мы настоятельно рекомендуем на раннем этапе используют известный качества. После назначить им следует запустить ознакомиться с содержимым в [Quality of качества руководство по решению](https://aka.ms/qerguide).
