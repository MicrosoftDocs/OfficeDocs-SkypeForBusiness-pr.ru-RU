# <a name="information-barriers-in-microsoft-teams"></a>Информационные барьеры в Microsoft Teams

[Информационные барьеры Microsoft Purview](/microsoft-365/compliance/information-barriers) — это политики, которые администратор может настроить, чтобы запретить отдельным пользователям или группам взаимодействовать друг с другом. Операции ввода-вывода полезны, если, например, один отдел выполняет обработку сведений, которые не должны быть предоставлены другим отделам. Операции ввода-вывода также полезны, если группа должна быть изолирована или не должна взаимодействовать с другими пользователями за пределами этой группы. Общие каналы в Microsoft Teams поддерживаются информационными барьерами. В зависимости от типа общего доступа политики информационных барьеров могут ограничить общий доступ определенными способами. Дополнительные сведения об общих каналах и поведении информационных барьеров см. в разделе ["Информационные барьеры и общие каналы"](information-barriers-shared-channels.md).

В Microsoft Teams информационные барьеры могут определять и предотвращать следующие виды несанкционированной совместной работы:

- Добавление пользователя в команду или канал
- Доступ пользователей к содержимому команды или канала
- Доступ пользователей к чатам 1:1 и групповым чатам
- Доступ пользователей к собраниям
- Предотвращает поиск и обнаружение, пользователи не будут отображаться в средстве выбора людей.

>[!NOTE]
>- Группы информационных барьеров нельзя создавать в разных клиентах.
>- Использование ботов, приложений Azure Active Directory (Azure AD), API для отправки уведомлений веб-канала действий и некоторых API для добавления пользователей не поддерживается в версии 1.
>- Частные каналы соответствуют настроенным политикам информационных барьеров.
>- Сведения о поддержке барьеров для сайтов SharePoint, подключенных к Teams, см. в разделе "Сегменты", связанные [с сайтами Microsoft Teams](/sharepoint/information-barriers#segments-associated-with-microsoft-teams-sites).

## <a name="background"></a>Общие сведения

Основным драйвером для операций ввода-вывода является отрасль финансовых услуг. Центр управления финансовой отраслью ([FINRA]( https://www.finra.org)) проверяет операции ввода-вывода и конфликты интересов в компаниях-членах и предоставляет рекомендации по управлению такими конфликтами (FINRA 2241, уведомление о регулятивных исследованиях по вопросам [15–31](https://www.finra.org/sites/default/files/Regulatory-Notice-15-31_0.pdf).

Однако с момента внедрения баз данных многие другие области нашли их полезными. К другим распространенным сценариям относятся:

- **Образование**: учащиеся в одном учебном заведении не могут найти контактные данные для учащихся других учебных заведений.
- **Юридическая** информация. Сохранение конфиденциальности данных, полученных владельцем одного клиента, и предотвращение доступа к ним со стороной той же компании, которая представляет другого клиента.
- **Для государственных** организаций: доступ к информации и управление ими ограничены в разных отделах и группах.
- **Профессиональные услуги**: группа сотрудников компании может общаться в чате только с клиентом или конкретным клиентом с помощью гостевого доступа во время взаимодействия с клиентом.

Например, Enrico принадлежит к сегменту "Банковский", а Pradeep — к сегменту финансового консультанта. Enrico и Pradeep не могут взаимодействовать друг с другом, так как политика IB организации блокирует обмен данными и совместную работу между этими двумя сегментами. Однако Enrico и Pradeep могут взаимодействовать с Ли в отделе кадров.

![Пример информационных барьеров, препятствующий обмену данными между сегментами.](media/information-barriers-example.png)

## <a name="when-to-use-information-barriers"></a>Когда следует использовать информационные барьеры

Вы можете использовать операции ввода-вывода в следующих ситуациях:

- Команде необходимо запретить обмен данными с определенной командой или предоставление к нее общего доступа.
- Команда не должна обмениваться данными и обмениваться данными с другими пользователями.

Служба оценки политики информационных барьеров определяет, соответствует ли обмен данным политикам IB.

## <a name="managing-information-barrier-policies"></a>Управление политиками информационных барьеров

Управление политиками IB выполняется в Портал соответствия требованиям Microsoft Purview (SCC) с помощью командлетов PowerShell. Дополнительные сведения см. в [разделе "Определение политик информационных барьеров"](/office365/securitycompliance/information-barriers-policies).

>[!IMPORTANT]
>Перед настройкой или определением политик необходимо включить поиск по каталогам с заданной областью в Microsoft Teams. Подождите по крайней мере несколько часов после включения поиска в каталоге с заданной областью, прежде чем настраивать или определять политики информационных барьеров. Дополнительные сведения см. в [разделе "Определение политик информационных барьеров"](/office365/securitycompliance/information-barriers-policies#prerequisites).

## <a name="information-barriers-administrator-role"></a>Роль администратора информационных барьеров

Роль управления соответствием IB отвечает за управление политиками IB. Дополнительные сведения об этой роли см. в [разделе "Разрешения" Портал соответствия требованиям Microsoft Purview](/office365/securitycompliance/permissions-in-the-security-and-compliance-center).

## <a name="information-barrier-triggers"></a>Триггеры информационных барьеров

Политики IB активируются при возникновении следующих событий Teams:

- **Участники добавляются** в команду. Каждый раз, когда вы добавляете пользователя в команду, политика пользователя должна оцениваться с учетом политик IB других участников команды. После успешного добавления пользователя пользователь может выполнять все функции в команде без дополнительных проверок. Если политика пользователя блокирует их добавление в команду, пользователь не будет отображаться в поиске.

    ![Снимок экрана: поиск нового участника для добавления в команду и поиск совпадений.](media/information-barriers-add-members.png)

- **Запрашивается** новый чат: каждый раз, когда пользователь запрашивает новый чат с одним или несколькими другими пользователями, чат оценивается, чтобы убедиться, что он не нарушает политики IB. Если диалог нарушает политику IB, беседа не запускается.

    Ниже приведен пример чата 1:1.

    ![Снимок экрана: заблокированное взаимодействие в чате 1:1.](media/information-barriers-one-one-chat.png)

    Ниже приведен пример группового чата.

    ![Снимок экрана: групповой чат.](media/information-barriers-group-chat.png)

- Пользователю предлагается присоединиться к собранию **. Когда** пользователю предлагается присоединиться к собранию, политика IB, применяемая к пользователю, оценивается с учетом политик IB, применяемых к другим участникам команды. В случае нарушения пользователю не будет разрешено присоединяться к собранию.

    ![Снимок экрана: пользователь, заблокированный на собрании.](media/information-barriers-meeting.png)

- Доступ **к** экрану предоставляется двум или более пользователям. Когда пользователь предоставляет общий доступ к экрану другим пользователям, необходимо оценить общий доступ, чтобы убедиться, что он не нарушает политики IB других пользователей. Если политика IB нарушена, общий доступ к экрану будет запрещен.

    Ниже приведен пример общего доступа к экрану перед использованием политики.

- Пользователь выполняет телефонный звонок в **Teams**: каждый раз, когда пользователь инициирует голосовой звонок (через VOIP) другому пользователю или группе пользователей, звонок оценивается, чтобы убедиться, что он не нарушает политики IB других участников команды. Если имеется нарушение, голосовой звонок блокируется.

- **Гости в Teams**: политики IB также применяются к гостям в Teams. Если гости должны быть доступны для обнаружения в глобальном списке адресов вашей организации, см. раздел "[Управление гостевым доступом" в Группы Microsoft 365](/microsoft-365/admin/create-groups/manage-guest-access-in-groups). После того как гости будут доступны для обнаружения, можно [определить политики IB](/office365/securitycompliance/information-barriers-policies).

## <a name="how-policy-changes-impact-existing-chats"></a>Влияние изменений политики на существующие чаты

Когда администратор политики IB вносит изменения в политику или когда изменение политики активируется из-за изменения профиля пользователя (например, для изменения задания), служба оценки политики информационных барьеров автоматически ищет участников, чтобы убедиться, что их членство в команде не нарушает политики.

Если между пользователями есть чат или другой обмен данными, а также установлена новая политика или существующая политика изменена, служба оценивает существующие сообщения, чтобы убедиться, что обмен данными по-прежнему разрешен. 

- **Чат 1:1**. Если обмен данными между двумя пользователями больше не разрешен (из-за приложения одному или обоим пользователям политики, блокируют обмен данными), дальнейшее взаимодействие блокируется. Существующие беседы в чате становятся доступными только для чтения.

    Ниже приведен пример, показывающий, что чат виден.

    ![Снимок экрана: доступен чат пользователя.](media/ib-before-1-1chat-policy.png)

    Ниже приведен пример отключения чата.

    ![Снимок экрана: отключен чат пользователя.](media/ib-after-1-1chat-policy.png)

- Групповой **чат: если** связь между одним пользователем и группой больше не разрешена (например, из-за того, что пользователь изменил задания), пользователь вместе с другими пользователями, участие которых нарушает политику, может быть удален из группового чата, а дальнейшее взаимодействие с группой будет запрещено. Пользователь по-прежнему может видеть старые беседы, но не сможет просматривать новые беседы с группой или участвовать в них. Если новая или измененная политика, препятствующая обмену данными, применяется к нескольким пользователям, пользователи, на которых распространяется эта политика, могут быть удалены из группового чата. Они по-прежнему могут видеть старые беседы.

  В этом примере Enrico перемещен в другой отдел в организации и удален из группового чата.

  ![Снимок экрана: групповой чат, из которого был удален пользователь.](media/information-barriers-user-changes-job.png)

  Enrico больше не может отправлять сообщения в групповой чат.

  ![Снимок экрана: не удается отправить сообщения в групповой чат, так как пользователь был удален из группы.](media/information-barriers-user-changes-job-2.png)

- **Команда**. Все пользователи, которые были удалены из группы, удаляются из группы и не смогут просматривать существующие или новые беседы или участвовать в них.

## <a name="scenario-a-user-in-an-existing-chat-becomes-blocked"></a>Сценарий: пользователь в существующем чате становится заблокированным

В настоящее время пользователи могут использовать следующие сценарии, если политика IB блокирует другого пользователя:

- **Люди:** пользователь не может видеть заблокированных пользователей на вкладке **Люди**.

- **Люди выбора**: заблокированные пользователи не будут отображаться в средстве выбора людей.

    ![Снимок экрана: Teams оповещение пользователя о том, что политика запрещает отображение сведений другого пользователя.](media/information-barriers-people-picker.png)

- **Вкладка "** Действие". Если пользователь посещает вкладку **"** Действие" заблокированного пользователя, записи не отображаются. (На **вкладке "** Действие" отображаются только записи каналов, и между двумя пользователями не будет общих каналов.)

    Ниже приведен пример заблокированного представления вкладки действия.

    ![Снимок экрана: заблокированная вкладка действия.](media/ib-after-activity-tab-policy.png)

- **Организационные диаграммы**: если пользователь имеет доступ к организационной диаграмме, на которой отображается заблокированный пользователь, заблокированный пользователь не будет отображаться на организационной диаграмме. Вместо этого появится сообщение об ошибке.

- **Люди** карточки: если пользователь участвует в беседе, а пользователь заблокирован позже, другие пользователи будут видеть сообщение об ошибке вместо карточки людей при наведении указателя мыши на имя заблокированного пользователя. Действия, перечисленные на карте (например, звонки и чат), будут недоступны.

- **Предлагаемые контакты**: заблокированные пользователи не отображаются в списке предлагаемых контактов (начальный список контактов, отображаемый для новых пользователей).

- **Контакты чата**: пользователь может видеть заблокированных пользователей в списке контактов чатов, но заблокированные пользователи будут определены. Единственным действием, которое пользователь может выполнить с заблокированным пользователями, является их удаление. Пользователь также может выбрать их, чтобы просмотреть их прошлые беседы.

- **Контакты вызовов**: пользователь может видеть заблокированных пользователей в списке контактов звонков, но заблокированные пользователи будут определены. Единственным действием, которое пользователь может выполнить с пользователями блокировки, является их удаление.

    Ниже приведен пример заблокированного пользователя в списке контактов звонков.

    > [!div class="mx-imgBorder"]
    > ![Снимок экрана: пользовательский чат.](media/ib-before-chat-contacts-policy.png)

    Ниже приведен пример отключения чата для пользователя в списке содержимого вызовов.

    > [!div class="mx-imgBorder"]
    > ![Снимок экрана, показывающий пользователя, заблокированного в чате.](media/ib-after-chat-contacts-policy.png)

- **Миграция из Skype в Teams**. Во время миграции с Skype для бизнеса в Teams все пользователи , даже те пользователи, которые заблокированы политиками IB, будут перенесены в Teams. Затем эти пользователи обрабатываются, как описано выше.

## <a name="information--barrier-modes-and-teams"></a>Режимы информационных барьеров и Teams

Режим информационных барьеров помогает укрепить пользователей, которые могут быть добавлены в команду или удалены из нее. При использовании информационных барьеров в Teams поддерживаются следующие режимы IB:

- **Открытый**. Эта конфигурация является режимом IB по умолчанию для всех существующих групп, подготовленных до включения информационных барьеров. В этом режиме политики IB не применяются.
- **Неявное**. Эта конфигурация является режимом IB по умолчанию, когда команда подготавливается после включения информационных барьеров. Неявный режим позволяет добавить всех совместимых пользователей в группу.
- **Модерация владельца**. Этот режим задается в команде, если вы хотите разрешить совместную работу между несовместимыми пользователями сегментов, которые модерация выполняется владельцем. Владелец команды может добавлять новых участников в соответствии с их политикой IB.

Команды, созданные перед активацией политики информационных барьеров в клиенте, автоматически устанавливаются в режим *открытия* по умолчанию. После активации политик IB в клиенте необходимо обновить режим существующих команд до неявного, чтобы убедиться,  что существующие команды соответствуют требованиям IB.

Используйте [командлет Set-UnifiedGroup](/powershell/module/exchange/set-unifiedgroup) с параметром *InformationBarrierMode* , который соответствует режиму, который вы хотите использовать для сегментов. Допустимый список значений для *параметра InformationBarrierMode* : *Open*, *Implicit* и *Owner Moderated*.

Например, чтобы настроить неявный режим для группы Microsoft 365, используйте следующую команду PowerShell:

```powershell
Set-UnifiedGroup -InformationBarrierMode Implicit
```

Чтобы обновить режим с "Открыть" на "Неявный" для всех существующих команд, используйте этот [сценарий PowerShell](information-barriers-mode-script.md).

Если вы измените конфигурацию открытого режима в существующих группах, подключенных к Teams, в соответствии с требованиями к соответствию требованиям вашей организации, необходимо обновить режимы [IB](/sharepoint/information-barriers#view-and-manage-ib-modes-as-an-administrator-with-sharepoint-powershell) для связанных сайтов SharePoint, подключенных к команде Teams.

## <a name="known-issues"></a>Известные проблемы

- **Пользователи** не могут присоединяться к нерегламентированному собранию. Если политики IB включены, пользователям не разрешается присоединяться к собраниям, если размер участников собрания превышает ограничения на количество [участников собрания.](limits-specifications-teams.md) Основная причина в том, что проверки IB зависят от того, могут ли пользователи быть добавлены в список чатов собраний, и только в том случае, если они могут быть добавлены в список, они могут присоединиться к собранию. Пользователь, присоединяющийся к собранию, добавляет этого пользователя в список; Таким образом, для повторяющихся собраний список может заполняться быстро. Когда количество участников чата [достигнет предельного](limits-specifications-teams.md) числа участников собрания, к собранию нельзя добавить дополнительных пользователей. Если IB включен для организации и список чатов заполнен для собрания, новые пользователи (пользователи, которых еще нет в списке) не смогут присоединиться к собранию. Но если IB не включен для организации и список чатов собраний заполнен, новые пользователи (пользователи, которых еще нет в списке) смогут присоединиться к собранию, хотя они не будут видеть параметр чата на собрании. Краткое решение — удалить неактивных участников из списка чатов собраний, чтобы сделать место для новых пользователей. Однако мы будем увеличивать размер списков чатов собраний позже.
- **Пользователи не могут** присоединяться к собраниям каналов. Если политики IB включены, пользователям не разрешается присоединяться к собраниям каналов, если они не являются участниками команды. Основная причина в том, что проверки IB зависят от того, могут ли пользователи быть добавлены в список чатов собраний, и только в том случае, если они могут быть добавлены в список, они могут присоединиться к собранию. Поток чата в собрании канала доступен только участникам команды или канала, и другие участники не могут видеть поток чата или получать к нему доступ. Если IB включен для организации и участник, не включив команду, пытается присоединиться к собранию канала, этот пользователь не может присоединиться к собранию. Однако если IB не включен* для организации и участник, не включаемый в команду, пытается присоединиться к собранию канала, пользователь может присоединиться к собранию, но не будет видеть параметр чата на собрании.
- **Максимальное число сегментов, разрешенных** в организации: каждая организация может настроить до 100 сегментов при настройке политик IB. Количество политик, которые можно настроить, не ограничивается.
- **Политики IB** не работают для федеративных пользователей. Если разрешить федерацию с внешними организациями, пользователи этих организаций не будут ограничены политиками IB. Если пользователи вашей организации присоединяются к чату или собранию, организованным внешними федеративными пользователями, политики IB также не ограничивают обмен данными между пользователями вашей организации.
