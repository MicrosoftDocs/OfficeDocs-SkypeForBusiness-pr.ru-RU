# <a name="information-barriers-in-microsoft-teams"></a>Информационные барьеры в Microsoft Teams

Барьеры данных (ГЕО) — это политики, которые администратор может настроить для предотвращения взаимодействия пользователей и групп друг с другом. Параметр «геобухгалтерия» полезен, если, например, один отдел обрабатывает данные, которые не должны быть предоставлены другим подразделениям, и они должны быть недоступными для общения с пользователями, находящимися за пределами этой группы.

> [!NOTE]
> - Группы барьера для данных нельзя создавать на разных клиентах.
> - Использование ботов для добавления пользователей не поддерживается в версии 1.
> - Закрытые каналы соответствуют настройкам для заданных политик барьера.
> - Новое: сведения о поддержке барьеров для сайта SharePoint, подключенного к Teams, щелкните [здесь](https://docs.microsoft.com/sharepoint/information-barriers#segments-associated-with-microsoft-teams-sites).

Политики барьера информации также предотвращают подстановки и обнаружение. Если вы попытаетесь общаться с кем-то, к которому вам не нужно взаимодействовать, вы не сможете найти этого пользователя в средстве выбора людей.

## <a name="background"></a>Общие сведения

Основной драйвер для барьеров, поступающих от отрасли финансовых услуг. Бухгалтерские органы для финансовых органов ([FINRA]( http://www.finra.org)) просматривают препятствия и конфликты информации в рамках компаний-специалистов и предоставляют рекомендации по управлению такими конфликтами (FINRA 2241, [Поиск задолженности, соответствие нормативным требованиям 15-31](http://www.finra.org/sites/default/files/Regulatory-Notice-15-31_0.pdf).  

Тем не менее, так как при этом возникают препятствия, многие другие части нашли их полезными. Ниже перечислены наиболее распространенные сценарии.

- Учебный курс: учащиеся в одном учебном заведении не могут найти контактные данные для учащихся других учебных заведений.
- Юридическая информация: обеспечение конфиденциальности данных, полученных юрист одного клиента, от доступа юрист для одной фирмы, представляющей другого клиента.
- Правительственные учреждения: доступ к данным и управление ограничены для разных отделов и групп.
- Профессиональные услуги: группа пользователей в компании может общаться с клиентом или конкретным пользователем через гостевой доступ во время обслуживания клиентов.

Например, Enrico принадлежит сегменту банка и Pradeep принадлежит к сегменту "финансовый консультант". Enrico и Pradeep не могут взаимодействовать друг с другом, так как политика переписки Организации блокирует связь и совместную работу между этими двумя сегментами. Однако Enrico и Pradeep могут общаться с другими сотрудниками отдела кадров.

![Пример, демонстрирующий барьеры информации, препятствующие обмену данными между сегментами](media/information-barriers-example.png)

## <a name="when-to-use-information-barriers"></a>Когда следует использовать барьеры информационных данных

Возможно, вы захотите использовать информационные барьеры в подобных ситуациях:

- Команде необходимо запретить взаимодействие с другими пользователями и совместное использование данных с другими участниками группы.
- Участники команды не должны передавать данные и обмениваться ими с другими пользователями за пределами команды.

Служба оценки политики "барьер данных" определяет, соответствует ли связь политикам барьера информации.

## <a name="managing-information-barrier-policies"></a>Управление политиками барьера данных

Политики управления информационными барьерами управляются в центре соответствия требованиям Microsoft 365 (SCC) с помощью командлетов PowerShell. Дополнительные сведения можно найти в разделе [Определение политик для барьеров информации](https://docs.microsoft.com/office365/securitycompliance/information-barriers-policies).

> [!IMPORTANT]
> Перед настройкой или определением политик **необходимо включить поиск по каталогу в Microsoft Teams**. Подождите не менее нескольких часов после включения поиска в каталог с областью действия, прежде чем настраивать или определять политики для барьеров информации. Дополнительные сведения [о предварительных требованиях для информационных барьеров](https://docs.microsoft.com/office365/securitycompliance/information-barriers-policies#prerequisites).

## <a name="information-barriers-administrator-role"></a>Роль администратора "препятствия данных"

Роль управления соответствием требованиям несет ответственность за управление политиками барьера информации. Дополнительные сведения об этой роли можно найти [в разделе разрешения в центре соответствия требованиям Microsoft 365](https://docs.microsoft.com/office365/securitycompliance/permissions-in-the-security-and-compliance-center).

## <a name="information-barrier-triggers"></a>Триггеры информационного барьера

Политики для работы с информационными барьерами активируются, когда выполняются следующие события teams:

- **Участники добавляются в группу** , каждый раз, когда вы добавляете пользователя в группу, она должна оцениваться с учетом политик барьера информации других участников группы. После того как пользователь будет успешно добавлен, пользователь сможет выполнять все функции в команде без дополнительных проверок. Если пользовательская политика блокирует их добавление в группу, пользователь не будет отображаться в области поиска.

    ![Снимок экрана, на котором показан групповой чат](media/information-barriers-add-members.png)

- **Запрашивается новый чат** — каждый раз, когда вы запрашиваете новый чат между двумя или более пользователями, этот чат оценивается так, чтобы он не наруша ни одной политики на основе данных. Если диалоговое окно нарушает политику информационного барьера, разговор не инициируется.

    Вот пример разговора в 1:1.

     ![Снимок экрана, на котором показано заблокированное общение в чате 1:1](media/information-barriers-one-one-chat.png)

    Ниже приведен пример группового чата.

    ![Снимок экрана, на котором показан групповой чат](media/information-barriers-group-chat.png)

- **Пользователь может присоединиться к собранию** — когда пользователь приглашается присоединиться к собранию, политика пользователя оценивается по политикам других участников группы, и если возникает нарушение, пользователь не может присоединиться к собранию.

    ![Снимок экрана, на котором показано, что пользователь заблокирован из собрания](media/information-barriers-meeting.png)

- **Экран, разделяемый двумя или более пользователями** — при совместном доступе к экрану между двумя и более пользователями необходимо оценивать экранный ресурс, чтобы убедиться, что он не нарушает политики информационных барьеров других пользователей. Если нарушена политика информационных барьеров, демонстрация экрана не разрешается. 
 
    Ниже приведен пример общего экрана, пока не будет применена политика. 

    ![Снимок экрана, на котором показан чат пользователя](media/ib-before-screen-share-policy.png)

    Ниже приведен пример общего доступа к экрану после применения политики. Общий доступ к экрану и значки вызываемого экрана не видны.

    ![Снимок экрана, на котором показан пользовательский символ с заблокированными параметрами](media/ib-after-screen-share-policy.png)

- **Пользователь помещает телефонный звонок (VoIP) в Teams** – любое время, когда пользователь может инициировать голосовой звонок другому пользователю или группе пользователей, этот вызов оценивается, чтобы убедиться, что он не нарушает политики барьера информации других участников группы. Если возникнут нарушения, голосовой звонок блокируется.
- **Гостевые пользователи в Teams** – политики барьера данных также применяются к гостевой машине в Teams. Если гостевые пользователи должны быть обнаружены в глобальном списке адресов своей организации, ознакомьтесь со статьей [Управление гостевым доступом в группах Microsoft 365](https://docs.microsoft.com/microsoft-365/admin/create-groups/manage-guest-access-in-groups). После того как гостевые пользователи будут обнаруживаемыми, вы можете [определить политики барьера информации](https://docs.microsoft.com/office365/securitycompliance/information-barriers-policies).

## <a name="how-policy-changes-impact-existing-chats"></a>Влияние изменений политики на существующие разговоры

Если администратор политики заряда данных вносит изменения в политику, или изменение политики вступает в силу из-за изменения профиля пользователя (например, для изменения задачи или аналогичной причины), Служба оценки политики банка данных автоматически выполняет поиск пользователей, чтобы убедиться, что участники команды не нарушают никакие политики.

Если вы уже используете чат или другую связь между пользователями, а новая политика задана или изменилась существующая политика, служба проверяет существующие коммуникации, чтобы убедиться в том, что связь по-прежнему разрешена. 

- **чат 1:1** – если связь между двумя пользователями больше не разрешена (если политика блокирует связь с одним или несколькими пользователями), дальнейшая связь блокируется, а разговор чата становится доступен только для чтения. 

    Вот пример отображения чата.

    ![Снимок экрана, на котором показан чат пользователя](media/ib-before-1-1chat-policy.png)

    Ниже приведен пример отключения чата.

    ![Снимок экрана, на котором показано участие пользователя, отключен](media/ib-after-1-1chat-policy.png)

- **Групповой чат** : Если связь от одного пользователя к группе больше не разрешена (например, если пользователь изменяет задания), пользователь вместе с другими пользователями, которые нарушают политику, может быть удален из группового чата, а дальнейшая связь с ней не будет разрешена. Пользователь по-прежнему может видеть старые беседы (только для чтения), но не сможет просматривать и принимать участие в новых беседах с этой группой. Если политика новой или измененной политики препятствует подключению к нескольким пользователям, пользователи, которые затрагивают эту политику, могут быть удалены из группового чата. Они по-прежнему могут просматривать старые беседы.

В этом примере Enrico перемещается в другой отдел в Организации и удаляется из группового чата.

  ![Снимок экрана, на котором показан групповой чат](media/information-barriers-user-changes-job.png)

Enrico больше не может отправлять сообщения в чат группового чата.

  ![Снимок экрана, на котором показан групповой чат](media/information-barriers-user-changes-job-2.png)

- **Группа** — все пользователи, которые были удалены из группы, удаляются из нее, и вы не сможете просматривать или участвовать в существующих или новых беседах.

## <a name="scenario-a-user-in-an-existing-chat-becomes-blocked"></a>Сценарий: пользователь в существующем чате становится заблокированным

В настоящее время пользователи сталкиваются со следующими сценариями, если политика блокировки данных блокирует другого пользователя.

- **Вкладка "люди"** : пользователь не видит "Заблокированные пользователи" на вкладке " **люди** ".
- **Средство выбора пользователей** — заблокированные пользователи не будут отображаться в средстве выбора людей.

    ![Снимок экрана, на котором показан групповой чат](media/information-barriers-people-picker.png)
    
- **Вкладка "действия"** — если пользователь посещает вкладку " **действия** " заблокированного пользователя, записи не отображаются. (На вкладке " **действия** " отображаются только учетные записи каналов, и в этом случае не будет обычных каналов между двумя пользователями.)

    Ниже приведен пример заблокированного представления "действия".

    ![Снимок экрана, на котором показана Заблокированная вкладка "действия"](media/ib-after-activity-tab-policy.png)


- **Организационные диаграммы** — если пользователь получает доступ к организационной диаграмме, в которой отображается заблокированный пользователь, он не отображается на организационной диаграмме, а вместо этого появляется сообщение об ошибке.
- **Карточка "люди** " — Если пользователь участвует в беседе, и пользователь в дальнейшем заблокирован, то при наведении указателя мыши на имя заблокированного пользователя другие пользователи увидят сообщение об ошибке, а не на карточке "люди". Действия, указанные на карточке (например, звонки и чат), будут недоступны.
- **Предлагаемые контакты** — заблокированные пользователи не отображаются в списке предлагаемых контактов (список первоначальнох контактов, который отображается для новых пользователей).
- **Контакты чата** — пользователь может видеть заблокированных пользователей в списке контактов для чатов, но заблокированные пользователи будут идентифицированы, а единственное действие, которое пользователь сможет выполнить, — удалить их. Пользователь также может щелкнуть по нему, чтобы просмотреть предыдущую беседу.
- **Звонки контактам** — пользователь может видеть заблокированных пользователей в списке контактов, но заблокированные пользователи будут идентифицированы и единственное действие, которое пользователь сможет выполнить, — удалить их.

    Ниже приведен пример заблокированного пользователя в списке контактов для звонков.

    ![Снимок экрана, демонстрирующий чат пользователя](media/ib-before-chat-contacts-policy.png)

    Ниже приведен пример отключенного чата для пользователя в списке звонки.

    ![Снимок экрана, на котором показано, что пользователь заблокирован из чата](media/ib-after-chat-contacts-policy.png)

- **Переход с Skype на Teams** – при миграции в Skype для бизнеса на Teams в Teams будут перенесены все пользователи, даже те, которые заблокированы политиками барьера, а затем будут обрабатываться так, как описано выше.

## <a name="teams-policies-and-sharepoint-sites"></a>Политики Teams и сайты SharePoint

При создании группы сайт SharePoint подготавливается к работе с файлами и связывается с Microsoft Teams. На этом сайте и файлах SharePoint по умолчанию не соблюдаются политики барьера данных. Чтобы включить политики барьера данных, администратор уже заполнил форму, запросив в SharePoint и OneDrive такие политики на геодоступном месте (ознакомьтесь с разделом " *Предварительные требования* [").](https://docs.microsoft.com/sharepoint/information-barriers#prerequisites) Если в SharePoint и OneDrive включена политика "барьеры данных", то на сайтах SharePoint, подготовленных с помощью Microsoft Teams, будут работать политики перегородки.

**Пример политик пересчета на сайте SharePoint команды** : в банке Contoso "пользователь" Sesha@contosobank.onmicrosoft.com "входит в сегмент банковской организации, а пользователь" Nikita@contosobank.onmicrosoft.com "принадлежит к рекомендациям сегмента. Политика перерезки Организации блокирует связь и совместную работу между этими двумя сегментами.
Когда пользователь Sesha создает группу для сегмента капиталовложений на инвестиции, группа и сайт SharePoint, которые ее зарезервировать, будут доступны только для пользователей, которые являются сегментами для капиталовложений в банк. Пользователь Nikita не может получить доступ к этому сайту, даже если у него есть ссылка на сайт.

Дополнительные сведения приведены в статье [сведения о барьере на информационной](https://docs.microsoft.com/sharepoint/information-barriers#segments-associated-with-microsoft-teams-sites) панели.

## <a name="known-issues"></a>Известные проблемы
- **Пользователи не могут присоединиться** к собраниям: Если включен "политики раскрывающегося списка", пользователи не могут присоединяться к собранию, если их размер не превышает ограничений на количество участников [собрания](limits-specifications-teams.md). Основная причина заключается в том, что при использовании средств с предельными чеками Вы сможете добавлять пользователей в список чата и получать этот сигнал, позволяющий пользователям присоединяться к собраниям. Если присоединиться к собранию, вы добавите этого пользователя в список, следовательно, для повторяющихся собраний Этот список будет заполнен быстрее. После того как вы достигнет ограничений на доступ к [собраниям](limits-specifications-teams.md), вы не сможете добавлять других пользователей в список чата для собрания. Если для этого клиента включена возможность включения подведения для собрания, новые пользователи (не включенные в этот список) не смогут присоединиться к собранию. Но если для клиента не включена функция "геоусловное использование", а список чата собрания заполнен, новые пользователи (не включенные в этот список) могут присоединиться к собранию, но они не будут видеть параметр чата на собрании. Краткосрочное решение состоит в том, чтобы удалить неактивных участников из списка чата собрания, чтобы освободить место для новых пользователей. Тем не менее, мы будем увеличивать размер списка чата для собраний позже.

- **Пользователи не могут присоединиться к собраниям канала** : Если включены политики, пользователи не могут присоединяться к собраниям канала, если они не являются участниками группы. Основная причина заключается в том, что при использовании средств с предельными чеками Вы сможете добавлять пользователей в список чата и получать этот сигнал, позволяющий пользователям присоединяться к собраниям. Поток чата в собрании канала доступен только для участников группы или канала, а не участники не могут просматривать и получать доступ к обсуждению чата. Если для клиента включена возможность включения и попытка присоединиться к собранию по каналу, пользователь не может присоединиться к собранию. Но если для клиента не включена функция "предельный", а участник, который не является членом группы, попытается присоединиться к собранию по каналу, пользователь может присоединиться к собранию, но он не сможет увидеть параметр чата в собрании.
