# <a name="how-exchange-and-microsoft-teams-interact"></a>Взаимодействие Exchange и Microsoft Teams

> [!Tip]
> Посмотрите следующий эпизод, чтобы узнать, как Teams взаимодействует с Azure Active Directory (AAD), группами Microsoft 365, Exchange, SharePoint и OneDrive для бизнеса: [Основы Microsoft Teams](https://aka.ms/teams-foundations)

Для полноценной работы с Teams каждому сотруднику нужно разрешить использовать Exchange Online, SharePoint Online, а также создавать группы Microsoft 365.

Почтовые ящики пользователей Exchange могут размещаться как в сети, так и локально.

Пользователям, размещенным в Exchange Online или Exchange Dedicated vNext, доступны все функции Teams. Они могут создавать команды и каналы и присоединяться к ним, создавать и просматривать собрания, звонить и общаться в чате, изменять пользовательские аватары (если политика почтового ящика Outlook в Интернете разрешает это), а также добавлять и настраивать соединители, вкладки и боты. Более подробный список доступных функций см. в таблице ниже.

Пользователи, размещенные в Exchange Online Dedicated (устаревшая среда), должны быть синхронизированы с Azure Active Directory в Microsoft 365 или Office 365. Они могут создавать команды и каналы, присоединяться к ним, добавлять и настраивать вкладки и боты, а также использовать функции чата и звонков. Однако они не могут изменять аватары, управлять собраниями, получать доступ к контактам Outlook или управлять соединителями.

> [!IMPORTANT]
> Для интеграции с локальной средой настоятельно рекомендуется использоваться полноценное классическое гибридное развертывание Exchange с Exchange Server 2016 или более поздней версией. Поддержка современного гибридного развертывания ограничена сведениями о доступности и, к примеру, не обеспечивает интеграцию календаря из Teams в локальные почтовые ящики. Дополнительные сведения о настройке гибридного развертывания см. в статье [Гибридные развертывания Exchange Server](https://docs.microsoft.com/exchange/exchange-hybrid).

Пользователи с почтовыми ящиками, размещенными в локальной среде, должны быть синхронизированы с Azure Active Directory. Они могут использовать все функции из описанного выше сценария и дополнительно управлять собраниями, если выполняются требования, указанные в разделе [Требования для почтовых ящиков, размещенных в локальной среде](#requirements-to-create-and-view-meetings-for-mailboxes-hosted-on-premises).

## <a name="requirements-to-get-the-most-out-of-microsoft-teams"></a>Требования для эффективного использования Microsoft Teams

Microsoft Teams взаимодействует с несколькими службами Microsoft 365 и Office 365, чтобы предоставлять пользователям обширные возможности. Для поддержки этих возможностей вам требуется включить определенные функции или службы и назначить лицензии.

- Пользователям должна быть назначена лицензия Exchange Online.

- Для хранения и совместного использования файлов в беседах команд требуется SharePoint Online. Microsoft Teams не поддерживает локальную конфигурацию SharePoint.

- Пользователям должна быть назначена лицензия SharePoint Online, если они хотят совместно использовать файлы в чатах. Если пользователям не назначены активные лицензии SharePoint Online, у них нет хранилища OneDrive для бизнеса в Microsoft 365 или Office 365. Общий доступ к файлам продолжит работать в каналах, однако пользователи не могут совместно использовать файлы в чатах без хранилища OneDrive для бизнеса в Microsoft 365 или Office 365.

- Чтобы создавать команды в Microsoft Teams, пользователи должны быть в состоянии создавать группы Microsoft 365.

  > [!IMPORTANT]
  > Если вы перемещаете пользователя в режим **Только Teams**, а затем удаляете клиент Skype для бизнеса, присутствие в Outlook и других приложениях Office может перестать отображаться. Присутствие в Teams отображается нормально. Чтобы устранить эту проблему, щелкните свой аватар в правом верхнем углу окна Microsoft Teams и выберите **Настройки**. На вкладке **Общие** в разделе **Приложение** выберите **Зарегистрировать Teams как приложение чата для Office (требует перезапуска приложений Office)**. Выбрав этот параметр, закройте и вновь откройте все приложения Office, включая Outlook. После открытия Outlook сведения о присутствии будут доступны.

## <a name="requirements-to-create-and-view-meetings-for-mailboxes-hosted-on-premises"></a>Требования к созданию и просмотру собраний для почтовых ящиков, размещенных в локальной среде

Если почтовые ящики размещаются в локальной среде, для создания и просмотра собрания должны выполняться следующие требования:

- Требуется назначить обязательную лицензию Teams для синхронизированного пользователя Azure Active Directory.

- Пользователи должны быть синхронизированы с Azure Active Directory. Сведения об использовании Azure AD Connect для синхронизации с Azure Active Directory см. в разделе [Документация по гибридной идентификации](https://docs.microsoft.com/azure/active-directory/hybrid/).

- Почтовые ящики размещаются в Exchange Server 2016 с накопительным пакетом обновления 3 или более поздней версии.

- Автообнаружение и веб-службы Exchange публикуются во внешней среде.

- Проверка подлинности OAuth предпочтительно настроена путем создания мастером гибридной конфигурации Exchange полной гибридной конфигурации (классической или современной). Если вы не можете использовать мастер гибридной конфигурации, настройте OAuth, как описано в статье [Настройка проверки подлинности OAuth между организациями Exchange и Exchange Online](https://docs.microsoft.com/exchange/configure-oauth-authentication-between-exchange-and-exchange-online-organizations-exchange-2013-help).

  > [!NOTE]
  > Exchange доверяет маркеру OAuth из службы Teams под названием EvoSTS. Этап 1 достаточен, но только для EvoSTS; ACS используется для поиска сведения о доступности в календаре.

- Установлен флажок функции гибридного развертывания Exchange в Azure AD Connect.

- Чтобы поддерживать приложение календаря и надстройку Teams Outlook для Mac, URL-адреса веб-службы Exchange должны быть настроены как SPN в Azure AD клиента для субъекта-службы Exchange. Этот этап выполняется с помощью мастера гибридной конфигурации или [ручных действий для гибридной современной проверки подлинности](https://docs.microsoft.com/microsoft-365/enterprise/configure-exchange-server-for-hybrid-modern-authentication#add-on-premises-web-service-urls-as-spns-in-azure-ad).

Чтобы включить делегирование календаря для этих пользователей:

- Вы также должны выполнить этапы 2–3 из статьи [Настройка интеграции и OAuth между Skype для бизнеса Online и Exchange Server](https://docs.microsoft.com/skypeforbusiness/deploy/integrate-with-exchange-server/oauth-with-online-and-on-premises). Эти этапы предоставят приложению планирования Teams нужные разрешения для подтверждения разрешений представителя.
 
  > [!NOTE]
  > Этап 2 включает назначение роли для ArchiveApplication, что не требуется для делегирования.

- Надстройке планирования Teams для Outlook при планировании собрания от имени другого пользователя требуется Exchange 2013 CU19 или более поздняя версия. Это предназначено для поддержки неавторизованного обнаружения почтового ящика нашей службой, чтобы проверить разрешения представителя на соответствие почтовому ящику делегирующего. Представитель и делегирующий могут располагаться в Exchange 2013 или более поздней версии либо в Exchange Online, но автообнаружение должно разрешаться для Exchange 2013 CU19 или более поздней версии.

## <a name="additional-considerations"></a>Дополнительные сведения

Вот некоторые дополнительные моменты, которые следует учесть при реализации Microsoft Teams в организации.

- Функции обеспечения безопасности и соответствия в Microsoft Teams, такие как обнаружение электронных данных, поиск контента, архивация и удержание по юридическим причинам, лучше всего работают в средах Exchange Online и SharePoint Online. Во время бесед в каналах сообщения передаются в почтовый ящик группы Exchange Online, где они доступны для обнаружения электронных данных. Если пользователи организации имеют доступ к SharePoint Online и OneDrive для бизнеса (с помощью учебной или рабочей учетной записи), эти функции по обеспечению соответствия также доступны для всех файлах в Teams.

- Управляйте конфигурацией политик соответствия требованиям и защищайте ее в Teams и Exchange с помощью условного доступа. Дополнительные сведения см. в разделе [Как работают политики условного доступа для Teams](security-compliance-overview.md#how-conditional-access-policies-work-for-teams).

- Если согласно применяемым в вашей организации требованиям соответствия все обсуждения на собраниях должны быть доступны для обнаружения, следует отключить частные собрания, когда организатор использует почтовый ящик в локальной среде Exchange. Дополнительные сведения см. в разделе [Разрешение планирования частных собраний](https://docs.microsoft.com/microsoftteams/meeting-policies-in-teams#allow-scheduling-private-meetings).

- В гибридном развертывании Exchange содержимое из сообщений чатов доступно для поиска независимо от того, какой почтовый ящик используют участники чата: облачный или локальный. Дополнительные сведения см. в разделе [Поиск облачных почтовых ящиков для локальных пользователей](https://docs.microsoft.com/office365/securitycompliance/search-cloud-based-mailboxes-for-on-premises-users). Сведения о поиске контента в Teams см. в статье [Поиск контента в Центре соответствия требованиям Microsoft 365](https://docs.microsoft.com/Office365/SecurityCompliance/content-search#searching-microsoft-teams-and-office-365-groups).

- Для указания статуса присутствия Microsoft Teams должен проверить, где размещен почтовый ящик — в Exchange Online или в локальной среде. После этого служба определяет, где получить доступ к почтовому ящику. Чтобы разрешить службе Teams проверять расположение почтового ящика с помощью вызова API REST в службу Exchange Online, вы должны развернуть гибридную среду Exchange, запустив мастер гибридной конфигурации Exchange, как описано в статье [Создание гибридного развертывания с помощью мастера гибридной конфигурации](https://docs.microsoft.com/exchange/hybrid-deployment/deploy-hybrid).
