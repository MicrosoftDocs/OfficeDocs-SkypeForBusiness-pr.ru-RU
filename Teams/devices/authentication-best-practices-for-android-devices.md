# <a name="authentication-best-practices-for-teams-shared-device-management-on-android-devices"></a>Рекомендации по проверке подлинности для управления общими устройствами Teams на устройствах Android

Цели устройств, используемых с Teams, делают необходимыми различные стратегии управления устройствами. Например, персональный бизнес-планшет, используемый одним менеджером по продажам, отличается от набора потребностей телефонов, используемых по телефону, которыми поделились многие сотрудники службы поддержки клиентов.

Администраторы безопасности и группы эксплуатации должны планировать устройства, которые могут использоваться в организации. Они должны реализовать *меры безопасности* , наиболее подходящие для каждой цели. Рекомендации, приведенные в этой статье, упрощают некоторые из этих решений.

>[!NOTE]
>Для условного доступа требуется подписка Azure Active Directory (Azure AD) Premium.

>[!NOTE]
>Политики для мобильных устройств Android могут не применяться к устройствам Android в Teams.

## <a name="authentication-recommendations-are-different-for-personal-versus-shared-android-devices"></a>Рекомендации по проверке подлинности отличаются для личных и общих устройств Android

Общие устройства Teams не могут использовать те же требования для регистрации и соответствия требованиям, которые используются на личных устройствах. Применение требований к проверке подлинности личных устройств к общим устройствам приведет к проблемам со входом.

1.  **Устройства выписыются из-за политик паролей.**

Учетные записи, используемые на устройствах Teams, имеют политику истечения срока действия пароля. Учетные записи, используемые с общими устройствами, не имеют определенного пользователя для обновления и восстановления их в рабочее состояние по истечении срока действия паролей. Если вашей организации требуется, чтобы срок действия паролей истекал и время от времени сбрасывались, эти учетные записи перестанут работать на устройствах Teams, пока администратор Teams не сбросит пароль и не выполнит вход.

**Задача**. Когда речь идет о доступе. Teams с устройства, учетная запись пользователя имеет политику истечения срока действия пароля. По истечении срока действия пароля он просто изменяется. Однако учетные записи, используемые на общих устройствах *(* учетных записях ресурсов), могут не быть подключены к одному пользователю, который может изменить пароль по мере необходимости. Это означает, что срок действия пароля может истечь и оставить сотрудников на месте, не зная, как возобновить работу.

Если вашей организации требуется сброс пароля или принудительное применение срока действия пароля, убедитесь, что администратор Teams готов сбросить пароль, чтобы эти общие учетные записи могли снова войти в систему.

2.  **Устройства не могут войти в систему из-за политик условного доступа.**

**Проблема**. Общие устройства не могут соответствовать Azure AD условного доступа для учетных записей пользователей или личных устройств. Если общие устройства группируются с учетными записями пользователей или личными устройствами для политики условного доступа, вход завершится *ошибкой*.

Например, если для доступа к Teams требуется многофакторная проверка подлинности, для завершения этой проверки подлинности требуется ввод кода пользователем. На общих устройствах обычно нет одного пользователя, который может настраивать и выполнять многофакторную проверку подлинности. Кроме того, если учетная запись должна повторно выполнять проверку подлинности каждые X дней, общее устройство не сможет решить проблему без вмешательства пользователя.

Многофакторная проверка подлинности не поддерживается на общих устройствах. Методы, которые следует использовать вместо этого, описаны ниже.

## <a name="best-practices-for-the-deployment-of-shared-android-devices-with-teams"></a>Рекомендации по развертыванию общих устройств Android с помощью Teams

Корпорация Майкрософт рекомендует следующие параметры при развертывании устройств Teams в вашей организации.

### <a name="use-a-resource-account-and-curtail-its-password-expiration"></a>**Использование учетной записи ресурса и прекращение срока действия пароля**

Общие устройства Teams должны использовать [почтовый ящик ресурсов Exchange](/exchange/recipients-in-exchange-online/manage-resource-mailboxes). При создании этих почтовых ящиков учетная запись создается автоматически. Эти учетные записи можно синхронизировать с Azure AD из Active Directory или создать непосредственно в Azure AD. Любые политики срока действия паролей для пользователей также будут применяться к учетным записям, используемым на общих устройствах Teams, поэтому, чтобы избежать нарушений, вызванных политиками истечения срока действия паролей, задайте политику истечения срока действия пароля для общих устройств без истечения срока действия.

Начиная с устройств Teams CY21 [с](https://support.microsoft.com/office/what-s-new-in-microsoft-teams-devices-eabf4d81-acdd-4b23-afa1-9ee47bb7c5e2#ID0EBD=Desk_phones) обновлением 1 (Teams версии 1449/1.0.94.2021022403 для телефонов Teams) и [CY2021 с](https://support.microsoft.com/office/what-s-new-in-microsoft-teams-devices-eabf4d81-acdd-4b23-afa1-9ee47bb7c5e2#ID0EBD=Teams_Rooms_on_Android) обновлением 2 (Teams версии 1449/1.0.96.2021051904 для Комнаты Microsoft Teams на Android), администраторы клиентов могут удаленно входить на устройства Teams. Вместо того чтобы предоставлять доступ к паролям техническим специалистам для настройки устройств, администраторам клиентов следует использовать удаленный вход для выдачи кодов проверки. Вход для этих устройств можно выполнить из Центра администрирования Teams.

Дополнительные сведения см. в разделе ["Удаленная подготовка и вход для устройств Android в Teams"](/MicrosoftTeams/devices/remote-provision-remote-login). 

### <a name="review-these-conditional-access-policies"></a>**Ознакомьтесь с этими политиками условного доступа**

Azure AD условный доступ задает дополнительные требования, которые должны соответствовать устройствам для входа. Для устройств Teams ознакомьтесь с приведенными ниже рекомендациями, чтобы определить, создали ли вы политики, которые позволяют пользователям общих устройств выполнять свою работу.

> [!TIP]
> Общие сведения об условном доступе см. [в статье "Что такое условный доступ"](/azure/active-directory/conditional-access/overview)?

### <a name="do-not-use-multi-factor-authentication-for-shared-devices"></a>Не используйте многофакторную проверку подлинности для общих устройств

Учетные записи общих устройств связаны с комнатой или физическим пространством, а не с учетной записью конечного пользователя. Так как общие устройства не поддерживают многофакторную проверку подлинности, исключите общие устройства из политик многофакторной проверки подлинности.

>[!TIP]
>Используйте [именованное расположение или](/azure/active-directory/conditional-access/location-condition) [требуйте соответствующее устройство](/azure/active-directory/conditional-access/howto-conditional-access-policy-compliant-device) для защиты общих устройств.

### <a name="you-can-use-location-based-access-with-named-locations"></a>Доступ на основе расположения можно использовать с именованными расположениями.

Если общие устройства подготавливаются в четко определенном расположении, которое можно определить с помощью диапазона IP-адресов, можно настроить условный доступ с помощью именованных расположений для этих устройств.[](/azure/active-directory/conditional-access/location-condition) Этот условный параметр позволит этим устройствам получать доступ к корпоративным ресурсам только в том случае, если они находятся в вашей сети.

### <a name="when-and-when-not-to-require-compliant-shared-devices"></a>Когда и когда не требуются соответствующие требованиям общие устройства

>[!NOTE]
>Для соответствия устройств требуется Intune лицензии.

При регистрации общих устройств в Intune вы можете настроить соответствие устройств в качестве элемента управления в условном доступе, чтобы доступ к корпоративным ресурсам могли получить только соответствующие устройства. Устройства Teams можно настроить для политик условного доступа на основе соответствия устройств. Дополнительные сведения см. в [разделе "Условный доступ: требовать соответствующее или гибридное Azure AD присоединенное устройство"](/azure/active-directory/conditional-access/howto-conditional-access-policy-compliant-device).

Сведения о настройке параметров соответствия для устройств с Intune см. в статье "Использование политик соответствия требованиям для настройки правил для устройств, управляемых с [помощью Intune](/intune/protect/device-compliance-get-started)".

>[!NOTE]
> Общие устройства, используемые для *горячего рабочего места* , следует исключить из политик соответствия требованиям. Политики соответствия требованиям запрещают регистрацию устройств в учетной записи пользователя "горячего рабочего места". **Вместо этого используйте именованные расположения для защиты этих устройств**.
> Чтобы повысить безопасность, можно также потребовать [](/azure/active-directory/authentication/tutorial-enable-azure-mfa) многофакторную проверку подлинности для пользователей или учетных записей пользователей службы технической поддержки в дополнение к именованным политикам расположения.

### <a name="exclude-shared-devices-from-sign-in-frequency-conditions"></a>Исключение общих устройств из условий частоты входа

В условном доступе можно настроить [](/azure/active-directory/conditional-access/howto-conditional-access-session-lifetime#user-sign-in-frequency) частоту входа, чтобы требовать от пользователей повторного входа для доступа к ресурсу по истечении указанного периода времени. Если для учетных записей комнат применяется частота входа, общие устройства будут выходить из системы, пока администратор не выполнит вход снова. Корпорация Майкрософт рекомендует исключить общие устройства из политик частоты входа.

### <a name="using-filters-for-devices"></a>Использование фильтров для устройств

[Фильтры для устройств](/azure/active-directory/conditional-access/concept-condition-filters-for-devices) — это функция условного доступа, которая позволяет настраивать более детализированные политики для устройств на основе свойств устройств, доступных в Azure AD. Вы также можете использовать собственные пользовательские значения, заключив атрибуты расширения 1–15 в объекте устройства, а затем используя их.

Используйте фильтры для устройств, чтобы идентифицировать устройства общего содержимого и включить политики в двух ключевых сценариях:

1.  Исключение общих устройств из политик, применяемых к личным устройствам. Например, требование соответствия устройств не  применяется к общим устройствам, используемым для горячего доступа, но применяется  ко всем другим устройствам на основе номера модели.

2.  Применение специальных политик на общих устройствах, *которые не* должны применяться к личным устройствам. Например, требование именованных расположений в качестве политики только для устройств с общей областью на основе атрибута расширения, задаваемого для этих устройств (например, CommonAreaPhone).

>[!NOTE] 
> Некоторые атрибуты, такие как **модель**, **производитель** и **operatingSystemVersion**, можно задать только в том случае, если устройства управляются Intune. Если ваши устройства не управляются Intune, используйте атрибуты расширения.
