# <a name="dimensions-and-measurements-available-in-call-quality-dashboard-cqd"></a>Измерения и измерения, доступные на панели мониторинга качества вызовов (CQD)

Панель мониторинга качества звонков (CQD) для Microsoft Teams и Skype для бизнеса Online позволяет лучше понять качество звонков, выполненных с помощью этих служб. В этом разделе содержатся подробные сведения о измерениях и измерениях, видимых с помощью CQD. Дополнительные сведения о CQD см. в статье "Использование CQD для управления качеством звонков и [собраний в Microsoft Teams"](quality-of-experience-review-guide.md).

## <a name="first-and-second-endpoint-classification"></a>Классификация первых и вторых конечных точек

Многие измерения и измерения в CQD помечены как первое или второе. Ниже описана логика, согласно которой конечная точка, относящаяся к потоку или звонку, считается первой.

- Конечная точка сервера (AV MCU, сервер-посредник и т. д.) считается первой, когда сервер участвует в потоке или вызове.
- Конечная точка клиента считается второй, если поток не находится между двумя конечными точками сервера.
- Если обе конечные точки имеют одинаковый тип, первая и вторая задается на основе внутреннего упорядочения категории агента пользователя, чтобы обеспечить согласованность порядка.

Например, здесь каждая строка представляет пару агентов пользователей, участвующих в потоке:

|Категория агентов пользователя для вызывающего абонента |Категория агентов пользователя для вызываемого абонента |First Endpoint |Second Endpoint|First Is Caller
|:--- |:--- |:--- |:--- |:--- |
|AV-MCU |OC (клиент Skype для бизнеса) |AV-MCU |OC (клиент Skype для бизнеса) |TRUE |
|OC (клиент Skype для бизнеса) |AV-MCU |AV-MCU |OC (клиент Skype для бизнеса) |FALSE |
|OC (клиент Skype для бизнеса) |OC (клиент Skype для бизнеса) |OC (клиент Skype для бизнеса) |OC (клиент Skype для бизнеса) |FALSE |
|AV-MCU |Сервер-посредник |Сервер-посредник |AV-MCU ||
|Сервер-посредник |AV-MCU |Сервер-посредник |AV-MCU |TRUE |
|OC (клиент Skype для бизнеса) |Телефон OC (IP-телефон Skype для бизнеса) |OC (клиент Skype для бизнеса) |Телефон OC (IP-телефон Skype для бизнеса) |TRUE |
|Телефон OC (IP-телефон Skype для бизнеса) |OC (клиент Skype для бизнеса) |OC (клиент Skype для бизнеса) |Телефон OC (IP-телефон Skype для бизнеса) |FALSE |

> [!NOTE]
> Такая классификация не учитывает, какая конечная точка является вызывающим или вызываемым абонентом. Измерение First Is Caller (Первая точка является вызывающим абонентом) позволяет проще определить принадлежность конечной точки к вызывающему или вызываемому абоненту.

## <a name="dimensions"></a>Измерения

Сведения о измерениях частично основаны на данных, отправленных на портал CQD. Многие значения измерения также можно использовать в качестве фильтров.

### <a name="dimension-data-types-and-units"></a>Типы и единицы данных измерения

#### <a name="boolean"></a>Boolean

Логические значения всегда имеют значение True или False. В некоторых случаях значение True также может быть представлено как 1, а false — как 0.

#### <a name="range"></a>Диапазон

Измерения, выраженные в виде диапазона или группы значений, отображаются в следующем формате.

 _\<sort order string\> [\<lower bound inclusive\> - \<upper bound exclusive\>_

Например, измерение Duration (Minutes) (Длительность в минутах) представляет длительность звонка в секундах, значение которой отображается в виде диапазона.

|Duration (Minutes) |Что означает |
|:--- |:--- |
|062: [0-0) |Длительность потока = 0 минут |
|064: [1-2) |1 минута < = длительность потока < 2 минуты |
|065: [2-3) |2 минуты < = длительность потока < 3 минуты |
|066: [3–4) |3 минуты < = длительность потока < 4 минуты |
|  | |

Строка \<sort order string> служит для управления порядком сортировки при отображении данных и может использоваться для фильтрации. Например, выполнив фильтрацию по параметру Duration (Minutes) < 065, мы отобразим потоки с длительностью меньше 2 минут (0 вначале нужен для правильной работы фильтрации). Фактическое значение указанной строки не имеет значения.

> [!NOTE]
> Вы можете заметить диапазоны, которые кажутся недопустимыми для заданного измерения. Например, wifi Signal Strength показывает вызовы в диапазоне 082: [100–110), когда 100 является максимальным возможным значением для wifi Signal Strength. Это связано с тем, как числа назначаются диапазонам в модели данных CQD. Если целое числовое значение равно 99, оно будет подсчитано в диапазоне 081: [90–100). Если это значение равно 100, оно будет подсчитано в диапазоне 082: [100–110). Это не означает, что имеются значения стойки сигнала Wi-Fi, превышающее 100 %.

#### <a name="enumeration-strings"></a>Строки перечисления

Строки, используемые CQD, часто являются производными от файлов данных, и они могут быть практически любым сочетанием символов в пределах допустимой длины. Некоторые измерения выглядят как строки, но поскольку они могут быть только одним из короткого списка предопределенных значений, они являются перечислением, а не строками true. Некоторые строки перечислений также используются в парах.

#### <a name="enumeration-pair"></a>Парное перечисление

Измерения, выраженные в виде парного перечисления, отображаются в следующем формате.

 _\<enumeration value from one end point\> : \<enumeration value from the other endpoint\>_

Порядок значений перечисления является согласованным, но не отражает порядок для первой и второй конечных точек.

Например, Network Connection Detail Pair (Парные сведения о сетевом подключении) показывает значения сведений о сетевом подключении для двух этих конечных точек:

|Network Connection Detail Pair |Что означает |
|:--- |:--- |
|Wired : Wired |Как первая, так и вторая конечные точки использовали проводные подключения Ethernet. |
|Wired : wifi |Первая конечная точка использовала проводное подключение Ethernet, а вторая  подключение Wi-Fi, либо наоборот. |
|: wifi |Первая конечная точка использовала подключение Wi-Fi, а сетевое подключение, использованное второй конечной точкой, неизвестно, либо наоборот. |
| | |

#### <a name="blank-values"></a>Пустые значения

В таблице ниже перечислены возможные причины, по которым измерение может быть пустым. Многие измерения и измерения будут пустыми, если доступное измерение QoE Record Available имеет значение false. Обычно это происходит, когда вызов не был успешно установлен или клиенту не удалось отправить данные телеметрии в службу.

### <a name="notes-on-measurements"></a>Примечания к измерениям

#### <a name="accuracy-limitations"></a>Ограничения точности
Некоторые измерения числа пользователей и вызовов зависят от выполнения отдельной операции подсчета для набора данных для вычисления количества. В настоящее время существует ошибка до 0,2 %, которая возникает при выполнении отдельной операции подсчета в зависимости от количества строк, в которых выполняется операция. Для наиболее точного тома следует использовать меры счетчика потоков, так как они не зависят от этой операции подсчета. Фильтрация для уменьшения объема данных может уменьшить ошибку, но не исключить этот источник ошибки в различных вызовах и количестве пользователей. Дополнительные сведения об этом ограничении см. здесь: [агрегатная функция dcount](/azure/data-explorer/kusto/query/dcount-aggfunction).

## <a name="filters"></a>Фильтры

Многие значения измерений и измерений также можно использовать в качестве фильтров. Вы можете использовать фильтры в запросе, чтобы исключить информацию так же, как вы выбрали бы измерение или измерение, чтобы добавить или включить информацию в запрос.
