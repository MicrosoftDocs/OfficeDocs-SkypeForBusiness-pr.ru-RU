# <a name="manage-emergency-calling"></a>Управление экстренным вызовом

В этой статье описаны основные понятия, необходимые для управления звонками в экстренные службы, в том числе сведения об адресах для экстренного вызова, динамические адреса для экстренного вызова и маршруты экстренных &mdash; вызовов. В этой статье используется следующая терминология:

- **Адрес для экстренного** помощи : физический или улицный адрес места для &mdash; организации.

  Например, адрес  *12345 North Main Street, Redmond, WA 98052* используется для маршрутизации экстренных вызовов в соответствующие диспетчерские органы и для оказания помощи в поиске вызывающего экстренных служб.

- **Place** - Typically a floor, building, wing, or office number. Место связано с адресом для экстренного устранения проблем, чтобы получить более точное расположение в здании. Вы можете иметь неограниченное количество мест, связанных с адресом для экстренного помощи. Например, если в организации несколько зданий, может потребоваться включить сведения о каждом здании и каждом этаже.  

- **Местоположение для экстренного** помощи. Расположение — это городской адрес &mdash; с необязательным местом. Если в вашей компании несколько физических расположений, скорее всего, понадобится несколько местоположений для экстренного экстренного помощи. 

  При создании адреса для экстренного помощи для этого адреса автоматически создается уникальный ИД местоположения.  Например, если вы добавляете этаж в адрес здания, для его сочетания создается ИД &mdash; &mdash; местоположения.  В этом примере будут 2 ИД местоположения: один для городского адреса; один для городского адреса и связанного с ним места.

  Когда вы назначаете пользователю или сайту местоположение для экстренного помощи, он связан с этим пользователем или сайтом.

- **Зарегистрированный адрес** : адрес для экстренного вызова, который назначен каждому пользователю плана звонков; иногда называется статическим адресом для экстренного помощи или адресом записи.  (Зарегистрированные адреса не относятся к пользователям прямой маршрутинга.)

Адреса для экстренного вызова создаются для пользователей плана звонков с помощью Центра администрирования Teams.  

>[!Note]
>Существует ряд отличий в управлении экстренные вызовы в зависимости от того, используете ли вы планы звонков телефонной системы или прямую маршрутизовую маршрутику телефонной системы для подключения к ОКП. Эти факторы описаны в этой статье.

## <a name="emergency-address-validation"></a>Проверка адреса для экстренного устранения

Чтобы назначить адрес для экстренного помощи пользователю или сетевому идентификатору, необходимо убедиться, что адрес для экстренного экстренного устранения помечен как "проверенный".  Проверка адреса гарантирует, что адрес является допустимым и его невозможно изменить после того, как он будет назначен. 

Если вы определяете адрес для экстренного помощи с помощью функции поиска адресной карты в Центре администрирования Teams, адрес автоматически помечется как проверенный. Проверенный адрес для экстренного устранения проблемы изменить нельзя. Поэтому при изменениях формата или представления адреса необходимо создать новый адрес в обновленном формате.


## <a name="emergency-address-geo-codes"></a>Географические коды адресов для экстренного решения

Каждый адрес для экстренного помощи может иметь связанный с ним географический код (широта и долгота). Эти географические коды используются в некоторых странах для маршрутации экстренных вызовов с динамическими расположениями. 

Если вы определяете адрес для экстренного помощи с помощью функции поиска адресной карты в Центре администрирования Teams, географический код автоматически сопоируется с адресом для экстренного обращения. Если адрес определен с помощью PowerShell, можно также связать с адресом географические коды. Однако корпорация Майкрософт рекомендует создавать адреса для экстренного вызова для плана звонков с помощью функции поиска карт в Центре администрирования Teams, которая гарантирует форматирование, проверку адресов и правильное географическое коды.  

>[!Important]
>Чтобы назначить сетевой идентификатор для динамических звонков в экстренные службы, адрес для экстренного вызова должен содержать соответствующий географический код.

Чтобы узнать, доступны ли планы звонков в вашем регионе, см. данные о доступности планов звонков в странах [и регионах.](country-and-region-availability-for-audio-conferencing-and-calling-plans/country-and-region-availability-for-audio-conferencing-and-calling-plans.md)


### <a name="emergency-call-enablement"></a>Включить экстренные вызовы

Каждый пользователь плана звонков автоматически может звонить в экстренные службы и должен иметь зарегистрированный адрес для экстренного вызова, связанный с его номером телефона. 

Когда расположение должно быть связано с телефонным номером, зависит от страны или региона:

- Например, в США и Канаде при назначении пользователю номера требуется местоположение для экстренного помощи.

- В других странах, например в Европе, На Среднем Востоке и в Африка (EMEA), при телефонном номере, который вы получаете от Microsoft 365 или Office 365, или при его переносе от другого поставщика услуг или оператора связи, требуется местоположение для экстренного звонков.

### <a name="dynamic-emergency-calling"></a>Динамические экстренные вызовы

Динамические экстренные вызовы для планов звонков Майкрософт обеспечивают возможность настройки и направления экстренных вызовов в зависимости от текущего расположения клиента Teams. Возможность автоматической маршрутики до соответствующего точки ответа на общедоступный запрос безопасности (PSAP) или уведомления сотрудников службы безопасности зависит от страны использования пользователя Teams.  

Для пользователей плана звонков динамическое расположение для маршрутации экстренных вызовов поддерживается только в США. (Сведения о динамических вызовах экстренной помощи и прямой маршрутке см. в сведениях [о прямой маршрутке.](#considerations-for-direct-routing)

- Если клиент Teams для плана звонков в США динамически получает адрес для экстренного вызова в США, этот адрес используется для маршрутировки в экстренных службах, а не для зарегистрированного адреса, и звонок будет автоматически перенаначен в PAP в области, используемой в адресе.

- Если клиент Teams для пользователя плана звонков для США не получает динамический адрес для экстренного вызова в США, зарегистрированный адрес для экстренного вызова используется для вызова справки и направления вызова. Однако вызов будет по экрану, чтобы определить, требуется ли обновить адрес, прежде чем подключать вызываемую к соответствующей psAP.

В США необходимо настроить городской адрес, который является частью местоположений для экстренного экстренного помощи, которые назначены сетевым идентификаторам, и включить связанные с ним &mdash; географические коды. Дополнительные сведения см. в [планировании и настройке динамических экстренных вызовов.](configure-dynamic-emergency-calling.md)


### <a name="emergency-call-routing"></a>Маршрутия экстренных вызовов

Когда пользователь плана звонков Teams набирает номер для вызова экстренной помощи, то, как звонок перенается в PAP, зависит от следующего:

- Определяет ли адрес для экстренного обращения динамически с помощью клиента Teams.

- Является ли адрес для экстренного помощи зарегистрированным адресом, связанным с номером телефона пользователя.

- Сеть экстренных вызовов в этой стране.

  **В США:**

  - Если клиент Teams расположен в динамическом расположении, определяемом клиентом, экстренные вызовы из этого клиента автоматически перена маршрутируются в PAP, обслуживающий это географическое местонахождение. 

  - Если клиент Teams не расположен в динамическом расположении, определяемом клиентом, в национальном центре обработки вызовов для определения местоположения вызываемого прежде чем перенаводить вызов на телефонную службу, отстоятельную в этом географическом расположении, в центр экстренной помощи.

  - Если звонящая не может обновить свое местоположение для экстренного вызова до центра отбора, звонок будет перенаправлен на PAP, обслуживающий зарегистрированный адрес вызываемого.

  **В Канаде, Ирландии** и Соединенном Королевстве для определения текущего местоположения пользователя перед подключением звонка в соответствующий диспетчерский центр сначала звонков в экстренные службы. 

  **Во Франции, Германии** и Испании экстренные вызовы перена маршрутируются непосредственно в PAP, который обслуживает адрес для экстренного вызова, связанный с номером, независимо от местоположения вызываемого.

  **В Нидерланды экстренные** вызовы перенанаются в ПСПС для кодов местных номеров независимо от местонахождения вызываемого.

  **В Австралии** адреса для экстренного устранения экстренных ситуаций настраиваются и перенанаются партнером-оператором.

  **В Японии** экстренные вызовы не поддерживаются.

## <a name="considerations-for-direct-routing"></a>Факторы, которые следует учитывать при прямой маршрутии

Если планы звонков недоступны в вашем регионе или вы хотите сохранить существующий оператор, рассмотрите возможность прямой [маршрутинга.](direct-routing-landing-page.md) Дополнительные сведения см. в настройках [прямой маршрутинга](direct-routing-configure.md) и управлении политиками [маршрутинга экстренных вызовов.](manage-emergency-call-routing-policies.md)

### <a name="emergency-call-enablement-and-configuration"></a>Включить и конфигурацию вызова экстренной помощи

Вы должны определить политики экстренных вызовов для пользователей прямой маршрутизации с помощью политики маршрутизации экстренных вызовов Teams (TeamsEmergencyCallRoutingPolicy), чтобы определить номера экстренных служб и место их назначения маршрутизации. (Обратите внимание, что зарегистрированные местоположения для экстренного помощи не поддерживаются для пользователей прямой маршрутации.)

Вы можете назначить политику маршрутинга экстренных вызовов учетной записи пользователя teams Direct Routing, сетевому сайту или обоим пользователям. Когда клиент Teams запускает или изменяет сетевое подключение, Teams выполняет подстановку сетевого сайта, на котором находится клиент:

- Если с сайтом связана политика маршрутинга экстренных вызовов, она используется для настройки экстренных вызовов.

- Если с сайтом не связана политика маршрутирования экстренных вызовов или клиент подключен к неопределенным сайтам, для настройки экстренных вызовов используется политика маршрутирования экстренных вызовов, связанная с учетной записью пользователя. 

- Если клиенту Teams не удается получить политику маршрутизов экстренных вызовов, пользователю не включена возможность вызова экстренной помощи.

### <a name="dynamic-emergency-calling"></a>Динамические экстренные вызовы

Клиенты Teams для пользователей прямой маршрутации могут получить динамический адрес для экстренного вызова, который можно использовать для динамического перенаправки звонков в зависимости от местоположения вызываемого пользователя. Дополнительные сведения см. [в сведениях о настройке динамических экстренных вызовов.](configure-dynamic-emergency-calling.md)

### <a name="emergency-call-routing"></a>Маршрутия экстренных вызовов

Политика маршрутации экстренных вызовов ссылается на использование ЗВОНКОВ по сети, которое должно иметь соответствующую конфигурацию прямой маршрутации для правильного направления экстренных вызовов на соответствующие шлюзы ННР. В частности, необходимо убедиться, что для строки экстренного набора есть строка OnlineVoiceRoute. Дополнительные сведения см. [в сведениях о настройке прямой маршрутинга.](direct-routing-configure.md) 

(Примечание. Клиенты Teams предварительно добавили знак "+" перед номерами экстренной помощи точно так же, как клиент Skype для бизнеса, то есть +911. Это поведение будет изменено в ближайшие месяцы, чтобы в Teams экстренные вызовы больше не отправляли "+" перед номером; то есть 911.)

Возможность динамического перенаправки экстренных вызовов для пользователей прямой маршрутки зависит от сети экстренных вызовов в данной стране. Существует два решения:

- Поставщики услуг для экстренных маршрутов (только в США) 
- Приложения для идентификации местоположения для экстренного местоположения (ELIN)

#### <a name="emergency-routing-service-providers"></a>Поставщики услуг для экстренных маршрутов

В США существует множество сертифицированных поставщиков услуг экстренных маршрутов (ERSP), которые могут автоматически перенау числеть экстренные вызовы в зависимости от местоположения вызываемого.

- Если поставщик услуг для экстренных маршрутов интегрирован в прямое развертывание маршрутов, экстренные вызовы с динамически приобретенным местоположением будут автоматически перенаправляться в общедоступный пункт ответа (PSAP), обслуживающий это расположение.

-  При звонках в экстренные службы без динамического расположения сначала по экрану определяется текущее местонахождение пользователя, прежде чем подключаться к соответствующему диспетчеру на основе обновленного расположения.

Дополнительные сведения см. в диспетчерах границ [сеансов, сертифицированных для прямой маршрутинга.](direct-routing-border-controllers.md)


#### <a name="emergency-location-identification-number-elin-applications"></a>Приложения для идентификации местоположения для экстренного местоположения (ELIN)

Пограничные контроллеры сеанса могут включать приложения для идентификации местоположения экстренного местоположения (ELIN). Если приложение ELIN службы SBC интегрировано с развертыванием прямой маршрутации, необходимо настроить адреса для экстренного вызова и связанные с ним телефонные номера в приложении ELIN, а затем отправить записи ELIN в базу данных экстренных вызовов в соответствующей базе данных ННР.  Местоположения для экстренного помощи Teams с идентификатором ELIN должны совпадать с расположениями в приложении ELIN.

Если экстренные вызовы с динамически полученным местоположением перенаются в соответствующую SBC, приложение ELIN:

- Проансирует местоположение вызываемого экстренных служб.
- Соответствует расположению записи ELIN.
- Заменяет номер экстренных вызовов номером телефона ELIN.
- Перенаправит звонок в службу PAP, обслуживаемую в этом расположении, и диспетчеры получат расположение из добавленной записи ELIN.

При обратном звонке на номер экстренных служб приложение ELIN будет заменять номера исходного вызываемого экстренных служб. 

Дополнительные сведения см. в диспетчерах границ [сеансов, сертифицированных для прямой маршрутинга.](direct-routing-border-controllers.md)


## <a name="security-desk-notification"></a>Уведомление службы безопасности

Уведомление службы безопасности доступно как для планов звонков Майкрософт, так и для прямой маршрутики телефонной системы.

Политика для экстренных вызовов в Teams (TeamsEmergencyCallingPolicy) используется для настройки того, кто должен быть уведомлен во время вызова экстренных служб и как они должны быть уведомлены: только чат, конференц-вызов, отключен или конференц-вызов с отключенными, но с возможностью его отключения.  Вы также можете указать внешний номер ННР пользователя или группы, чтобы позвонить и присоединиться к экстренным вызовам. 

Политику экстренных вызовов можно предоставить учетной записи пользователя Teams, назначенной сетевому сайту или обоим пользователям.  Когда клиент Teams запускает или изменяет сетевое подключение, Teams выполняет подстановку сетевого сайта, на котором находится клиент:

- Если политика экстренных вызовов связана с сетевым сайтом, она используется для настройки уведомлений службы безопасности.

- Если с сайтом не связана политика экстренных вызовов или клиент подключен к неопределенным сайтам, для настройки уведомлений службы безопасности используется политика экстренных вызовов, связанная с учетной записью пользователя.  

- Если клиенту Teams не удается получить политику для экстренных вызовов, пользователю не включено уведомление службы безопасности.

Во время звонка в экстренные службы к вызову конференц-зал безопасности, а пользовательский интерфейс пользователя службы безопасности контролируется на основе политики звонков в экстренные службы Teams. Групповой чат начинается с каждым сотрудником службы безопасности, и о расположении вызываемого экстренных вызовов вы можете получить уведомление о важном сообщении.  Если конференц-зал настроен как часть политики, каждый пользователь службы безопасности также будет называться как часть конференции.
