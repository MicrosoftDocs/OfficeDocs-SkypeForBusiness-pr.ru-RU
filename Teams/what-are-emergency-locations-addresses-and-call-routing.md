# <a name="manage-emergency-calling"></a>Управление вызовом экстренной помощи

В этой статье описаны основные понятия, которые необходимо знать для управления вызовом экстренной помощи, а также сведения об экстренных адресах, динамических экстренных адресах и маршрутизации экстренных вызовов. В этой статье используется следующая терминология:

- **Адрес для экстренного** реагирования — административный адрес — адрес физического или почтового помещения для Организации.

  Например, адрес 12345 в *Северной (Москва) улица, Redmond, WA 98052* используется для направления экстренных вызовов в соответствующие ИТ – службы и для того, чтобы помочь вам найти вызывающий объект для экстренного реагирования.

- **Место** (обычно — основание, здание, крыло или номер Office). Место, связанное с адресом для экстренного реагирования, позволяет указать более точное расположение в здании. Адрес для экстренного реагирования не может быть связан с неограниченным числом мест. Например, если в вашей организации есть несколько зданий, вы можете добавить сведения о расположении для каждого здания и для каждого цеха в каждом здании.  

- **Место для экстренного** реагирования — это административный адрес — необязательное место. Если в вашей компании есть более одного физического местоположения, возможно, вам потребуется больше одного местоположения для экстренного реагирования. 

  Когда вы создаете адрес для экстренного реагирования, для этого адреса автоматически создается уникальный идентификатор местоположения.  Если вы добавите место в адрес для экстренного реагирования, например, если вы добавите этаж в адрес здания, — код местоположения создается для комбинации адреса и места для экстренного реагирования.  В этом примере есть два идентификатора расположения: один для административного адреса. One для Объединенного административного адреса и связанного места.

  Назначение местоположения для экстренного реагирования пользователю или сайту — это уникальный идентификатор местоположения, связанный с пользователем или сайтом.

- **Зарегистрированный адрес** — адрес для экстренного реагирования, назначаемый каждому пользователю плана звонков; Иногда ее называют статическим адресом для экстренного реагирования или адресом записи.  (Зарегистрированные адреса не распространяются на прямые пользователи маршрутизации).

Вы создаете адреса для экстренного реагирования для пользователей плана звонков с помощью центра администрирования Teams.  

>[!Note]
>Существуют некоторые отличия в том, как управлять вызовом экстренной помощи в зависимости от того, используете ли вы планы звонков по телефонным системам или прямую маршрутизацию для подключения к сети PSTN. Эти вопросы описаны в этой статье.

## <a name="emergency-address-validation"></a>Проверка адресов для экстренного реагирования

Чтобы назначить адрес для экстренного реагирования пользователю или сетевому идентификатору, необходимо убедиться в том, что адрес для экстренного реагирования помечен как "Проверено".  Проверка адреса гарантирует подлинность адреса и его изменение после назначения. 

Если вы определяете адрес для экстренного реагирования с помощью функции поиска в карте адресов в центре администрирования Teams, адрес автоматически помечается как проверенный. Вы не можете изменить проверяемый адрес для экстренного реагирования. Таким образом, при изменении формата или представления адреса необходимо создать новый адрес с помощью обновленного формата.


## <a name="emergency-address-geo-codes"></a>Коды географических адресов для экстренного реагирования

Каждый адрес для экстренного реагирования может содержать код географической и долготы, связанный с ним. Эти геокоды используются в некоторых странах для упрощения маршрутизации экстренных вызовов с динамическими адресами. 

Если вы определяете адрес для экстренного реагирования с помощью функции поиска в карте адресов в центре администрирования Teams, код Geo автоматически связывается с адресом экстренной помощи. Вы также можете связать географических коды с адресом, если указать адрес с помощью PowerShell. Тем не менее Корпорация Майкрософт рекомендует создавать адреса для экстренного реагирования на план звонков с помощью функции "Поиск карт" в центре администрирования Teams, которая гарантирует, что адреса форматируются, проверены и содержат соответствующие географических коды.  

>[!Important]
>Чтобы назначить место для экстренного сетевого идентификатора для динамического вызова экстренной помощи, адрес для экстренного реагирования должен состоять из соответствующего географического кода.

### <a name="emergency-call-enablement"></a>Возможность вызова экстренной помощи

Каждый пользователь плана звонков автоматически включается для вызова экстренной помощи и должен иметь зарегистрированный адрес для экстренного реагирования, связанный с назначенным номером телефона. 

Если расположение должно быть связано с номером телефона, зависит от страны или региона:

- Например, в США и Канаде требуется указать расположение для экстренного реагирования, когда ему назначается номер.

- Для других стран (например, Европы, Ближнего Востока и Африки) — место для экстренного реагирования требуется при получении номера телефона из Office 365 или при его передаче от другого поставщика услуг или оператора.

### <a name="dynamic-emergency-calling"></a>Динамическая экстренная вызов

Динамическое обращение в службу экстренной помощи в планах звонков в Microsoft обеспечивает возможность настройки и маршрутизации экстренных вызовов в зависимости от текущего расположения клиента Teams. Возможность автоматической маршрутизации к соответствующей точке ответа на общедоступную безопасность (PSAP) или уведомлению сотрудников службы безопасности в зависимости от страны использования пользователя Teams.  

В настоящее время только пользователи плана звонков в США могут использовать динамические места для маршрутизации экстренных вызовов, как описано ниже.

- Если клиент Teams для абонентского плана в США может динамически получить адрес для экстренного реагирования в США, этот адрес используется для маршрутизации в экстренной службе, а не с зарегистрированного адреса, и звонок будет автоматически перенаправлен на PSAP в области "обслуживать".

- Если клиент Teams для абонентского плана в США не может динамически получить адрес для экстренного реагирования в США, для него используется зарегистрированный адрес для экстренной помощи. Тем не менее, во время звонка на экран будет определяться, требуется ли обновленный адрес перед подключением вызывающего абонента к соответствующему PSAP.

В США вы должны настроить административный адрес, который является частью местоположений для экстренного реагирования, назначенных сетевым идентификаторам, и включить в него соответствующие географических коды. Дополнительные сведения можно найти в разделе [планирование и Настройка динамического вызова экстренной](configure-dynamic-emergency-calling.md)помощи.


### <a name="emergency-call-routing"></a>Маршрутизация вызова экстренной помощи

Когда пользователь в плане звонков в Teams набирает номер для экстренного реагирования, способ направления звонка в PSAP зависит от следующих параметров:

- Определяется ли динамический адрес для экстренного реагирования клиентом Teams.

- Является ли адрес для экстренного реагирования зарегистрированным адресом, связанным с номером телефона пользователя.

- Сеть для экстренных вызовов в этой стране.

  **В Соединенных Штатах Америки:**

  - Если клиент Teams находится на динамическом месте, определенном клиентом, это автоматически перенаправляется в PSAP, который обслуживает географическое расположение. 

  - Если клиент Teams не находится в динамическом расположении, определяемом клиентом, для определения местоположения вызывающего абонента перед передачей звонка в PSAP, который обслуживает географическое положение, вызов экстренной помощи от этого клиента помещается в центр звонков.

  - Если вызывающему абоненту не удается обновить расположение для экстренного реагирования до центра обработки, Звонок будет передан в PSAP, который обслуживает зарегистрированный адрес вызывающего абонента.

  **В Канаде, Ирландии и соединении с помощью**экстренной помощи на первом экране, чтобы определить текущее расположение пользователя перед подключением к соответствующему центру отправки. 

  **В Франции, Германии и Испании**обращение в экстренных случаях осуществляется напрямую в PSAP, который обслуживает адрес, связанный с этим номером, независимо от местонахождения вызывающего абонента.

  **В Нидерландов**экстренные вызовы пересылаются непосредственно в PSAP для местного кода города, независимо от местонахождения вызывающего абонента.

  **В Австралии**адреса для экстренного реагирования настраиваются и направляются партнером перевозчиков.

  Вызов экстренной помощи **в Японии**не поддерживается.

## <a name="considerations-for-direct-routing"></a>Рекомендации по прямой маршрутизации

Если планы звонков недоступны в вашем регионе или вы хотите сохранить существующий перевозчик, рассматривайте [прямую переадресацию](direct-routing-landing-page.md). Дополнительные сведения можно найти в разделе [Настройка](direct-routing-configure.md) [политики маршрутизации и управления маршрутизацией вызовов экстренной](manage-emergency-call-routing-policies.md)помощи.

### <a name="emergency-call-enablement-and-configuration"></a>Включение и Настройка вызова экстренной помощи

Вы должны определить политики вызова экстренной помощи для прямого маршрутизации пользователей с помощью TeamsEmergencyCallRoutingPolicy, чтобы определить номера для экстренного реагирования и связанные с ними назначения маршрутизации. (Обратите внимание на то, что зарегистрированные расположения для экстренной помощи не поддерживаются для прямых маршрутов пользователей).

Вы можете назначить TeamsEmergencyCallRoutingPolicy учетной записи пользователя, сетевом сайте или обоими участниками. Когда клиент Teams запускает или изменяет сетевое подключение, Teams выполняет поиск сетевого сайта, на котором расположен клиент, как описано ниже.

- Если объект TeamsEmergencyCallRoutingPolicy связан с сайтом, для настройки вызова экстренной помощи используется политика сайта.

- Если с сайтом не связан ни один TeamsEmergencyCallRoutingPolicy или если клиент подключен на неопределенном сайте, для настройки вызова экстренной помощи используется TeamsEmergencyCallRoutingPolicy, связанный с учетной записью пользователя. 

- Если клиент Teams не может получить TeamsEmergencyCallRoutingPolicy, у пользователя не включена возможность вызова экстренной помощи.

### <a name="dynamic-emergency-calling"></a>Динамическая экстренная вызов

Клиенты Teams для Direct Routing Users могут получить динамический адрес для экстренного реагирования, который можно использовать для динамического направления звонков на основе местоположения вызывающего абонента. Дополнительные сведения можно найти в разделе [Настройка динамического вызова экстренной](configure-dynamic-emergency-calling.md)помощи.

### <a name="emergency-call-routing"></a>Маршрутизация вызова экстренной помощи

TeamsEmergencyCallRoutingPolicy ссылается на использование сетевой PSTN, и для правильного направления экстренных вызовов на соответствующие шлюзы PSTN (-ов) необходимо использовать соответствующую конфигурацию прямого направления. В частности, необходимо убедиться в том, что для строки набора экстренных номеров есть OnlineVoiceRoute. Дополнительные сведения можно найти в разделе [Настройка прямого маршрута](direct-routing-configure.md). 

(Примечание. клиенты групп добавляют знак "+" перед номерами экстренного реагирования так, как это происходит в клиенте Skype для бизнеса (+ 911). Это поведение будет изменено в течение ближайших месяцев, так что вызовы с помощью экстренных вызовов команд больше не будут отправлять "+" перед номером; то есть 911.)

Возможность динамического направления вызова экстренной помощи для пользователей Direct Routing может зависеть от сети вызова экстренной помощи в данной стране. Существует два решения, которые можно найти ниже.

- Поставщики услуг маршрутизации экстренной помощи (только для США) 
- Идентификационный номер расположения для экстренного реагирования (ELIN): Application Gateway

#### <a name="emergency-routing-service-providers"></a>Поставщики услуг маршрутизации для экстренной помощи

В Соединенных Штатах есть множество сертифицированных поставщиков услуг по маршрутизации (ERSPs), которые могут автоматически перенаправлять экстренные вызовы в зависимости от расположения вызывающего абонента.

- Если поставщик услуг маршрутизации экстренной помощи интегрирован в прямое развертывание по маршрутизации, экстренные вызовы с динамически приобретенным расположением будут автоматически перенаправлены в точку ответа для общедоступной защиты (PSAP), обслуживающей это расположение.

-  Вызов экстренной помощи без динамически приобретенного местоположения — первый экран, чтобы определить текущее расположение пользователя, прежде чем присоединиться к соответствующему центру отправки, основываясь на обновленном расположении.

Дополнительные сведения можно найти в разделе [контроллеры границ сеансов, сертифицированные для прямого маршрутизации](direct-routing-border-controllers.md).


#### <a name="emergency-location-identification-number-elin-applications"></a>Приложения идентификационного номера расположения для экстренного реагирования (ELIN)

Контроллеры границ сеансов (SBCs) могут включать приложения с идентификационным номером расположения для экстренного реагирования (ELIN). Если приложение SBC ELIN интегрировано в прямую переадресацию, необходимо настроить адреса экстренного реагирования и связанные номера телефонов в приложении ELIN, а затем добавить записи ELIN в базу данных вызова экстренной помощи в соответствующей КТСОП.  Местоположения для экстренных случаев с идентификаторами ELIN должны соответствовать друг другу в приложении ELIN.

Если вызов экстренной помощи с динамически приобретенным расположением перенаправляется на соответствующий объект SBC, приложение ELIN:

- Анализирует место расположения вызывающего абонента.
- Совпадает с расположением с записью ELIN.
- Подставляет номер вызывающего абонента с номером телефона ELIN.
- Направляет Звонок в PSAP, который обслуживает это расположение, а затем диспетчеры получают расположение из отправленной записи ELIN.

После обратного звонка на номер для экстренного реагирования приложение ELIN выполняет замену обратной нумерации в исходном вызове. 

Дополнительные сведения можно найти в разделе [контроллеры границ сеансов, сертифицированные для прямого маршрутизации](direct-routing-border-controllers.md).


## <a name="security-desk-notification"></a>Уведомление на рабочем месте

Уведомление на рабочем месте доступно как в планах звонков, так и на прямом маршруте телефонной системы.

С помощью TeamsEmergencyCallingPolicy вы можете настроить пользователей, которые должны получать уведомления во время вызова экстренного реагирования, и как они будут уведомлены: только чат, Конференции и отключены, а также Конференции с возможностью включения звука.  Вы также можете указать внешний номер КТСОП для пользователя или группы, чтобы позвонить и присоединиться к экстренному звонку. 

TeamsEmergencyCallingPolicy можно предоставить учетной записи пользователя Teams, назначенному сетевому сайту или обоим тем.  Когда клиент Teams запускает или изменяет сетевое подключение, Teams выполняет поиск сетевого сайта, на котором расположен клиент.

- Если объект TeamsEmergencyCallingPolicy связан с сетевым сайтом, для настройки уведомления на рабочем месте используется политика сайта.

- Если с сайтом не связан ни один TeamsEmergencyCallingPolicy, или если клиент подключен на неопределенном сайте, для настройки уведомления на рабочем месте используется TeamsEmergencyCallingPolicy, связанный с учетной записью пользователя.  

- Если клиент Teams не может получить TeamsEmergencyCallingPolicy, пользователь не включен в уведомление на службу безопасности.

Во время вызова экстренной помощи в звонке встроена безопасность, а опыт пользователя в области безопасности контролируется на основе TeamsEmergencyCallingPolicy. Групповой чат начинается с каждого участника для обеспечения безопасности, а расположение вызывающего объекта для экстренного реагирования предоставляется через важное уведомление о сообщении.  Если параметр конференции настроен как часть политики, каждый пользователь в службе поддержки также может называться как часть конференции.
