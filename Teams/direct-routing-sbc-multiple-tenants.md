# <a name="configure-a-session-border-controller-for-multiple-tenants"></a>Настройка множества клиентов в пограничном контроллере сеансов

Прямая маршрутия поддерживает настройку одного контроллера границы сеанса (SBC) для обслуживания нескольких клиентов.

> [!NOTE]
> Этот сценарий предназначен для партнеров Майкрософт и операторов STN, которые называются операторами далее в этом документе. Оператор продает телефонные услуги, доставленные клиентам Microsoft Teams телефонии. 

Оператор:
- Развертывает SBC и управляет им в своих центрах обработки данных (клиентам не нужно внедрять SBC, и они получают телефонные службы от Teams клиента).
- Подключение SBC к нескольким клиентам.
- Предоставляет клиентам услуги телефонной сети общего перейти на телефонную сеть (STN).
- Управляет окончанием качества звонка.
- Взимается отдельно за службы STN.

Корпорация Майкрософт не управляет операторами. Корпорация Майкрософт Teams Телефон частный филиал Exchange (УАКС) и Teams клиента. Корпорация Майкрософт также удостоверяет телефоны и удостоверяет SBCs, которые можно использовать с Teams телефонная система. Прежде чем выбирать оператора связи, убедитесь, что ваш выбор имеет сертифицированный SBC и может управлять качеством звука до конца.

Ниже следующую техническую реализацию можно найти, чтобы настроить сценарий.

**Только для операторов:**
1. Развернем SBC и настройте его для сценария размещения в соответствии с инструкциями сертифицированных [поставщиков SBC](#deploy-and-configure-the-sbc).
2. Зарегистрируйте базовое доменное имя в клиенте оператора связи и запросируйте сертификат с подданными знаками.
3. Зарегистрируйте поддомен для каждого клиента, который является частью базового домена.

**Оператор с глобальным администратором клиента:**
1. Добавьте имя поддомена в клиент клиента клиента.
2. Активируйте имя поддомена.
3. Настройте перенастройку связи между оператором и клиентом-клиентом и подготовка пользователей.

*Убедитесь, что вы понимаете основы DNS и управление доменным именем в Microsoft 365. [Дополнительные действия см. в Microsoft 365 справки](https://support.office.com/article/Get-help-with-Office-365-domains-28343f3a-dcee-41b6-9b97-5b0f4999b7ef) по доменам.*

## <a name="register-a-base-domain-and-subdomains"></a>Регистрация базового домена и поддоменов

В сценарии размещения необходимо создать:

- Одно базовое доменное имя, которое принадлежит оператору связи.
- Поддомен, который является частью базового доменного имени в каждом клиенте.

В следующем примере:

- Adatum — это оператор, обслуживающий несколько клиентов, предоставляющих услуги Интернета и телефонии.
- Woodgrove Bank, Contoso и Adventure Works — это три клиента, которые Microsoft 365 доменами, но получают телефонные службы от Adatum.

Поддомены  должны соответствовать имени FQDN линии связи, которая будет настроена для клиента, и FQDN в заглавной части контакта при отправке приглашения на Microsoft 365. 

Когда звонок поступает в интерфейс Microsoft 365 прямой маршрутии, интерфейс использует заглавную линию контакта для поиска клиента, в котором нужно найти пользователя. При прямой маршрутике не используется подсмотр номеров телефонов в приглашении, так как у некоторых клиентов могут быть номера, неидавающиеся на нескольких клиентах. Таким образом, имя FQDN в заглавной записи контакта необходимо для определения точного клиента для искомого пользователя по номеру телефона.

*Дополнительные сведения о создании доменных имен в Microsoft 365 организациях см. в Microsoft 365 [доменах](https://support.office.com/article/Get-help-with-Office-365-domains-28343f3a-dcee-41b6-9b97-5b0f4999b7ef).*

![Схема с требованиями к доменам и заглавным данным контакта.](media/direct-routing-1-sbc-requirements.png)

### <a name="create-a-trunk-and-provision-users"></a>Создание сети и подготовка пользователей

В первом выпуске Direct Routing корпорация Майкрософт требовала добавить линию связи в каждый обслуживающий клиент (клиентский) с помощью New-CSOnlinePSTNGateway клиента.

Однако этот метод не является оптимальным по двум причинам:
 
- **Управление накладных расходов**. Например, при разгрузке или разрядке SBC изменяется ряд параметров, например включение или отключение обхода мультимедиа. Для изменения портов требуется изменить параметры в нескольких клиентах (с помощью Set-CSOnlinePSTNGateway), но фактически это один и тот же SBC. 

-  **Обработка накладных расходов**. Сбор и отслеживание данных о состоянии дел на магистрали — параметры SIP, собранные из нескольких логических ленток, которые, на самом деле, являются одним и тем же SBC и одним и тем же физическими данными, замедляют обработку данных маршрутки.
 
На основе этих отзывов корпорация Майкрософт привнося новую логику, чтобы обу числеть связи для клиентов клиента.

Были введены два новых сущности:

- Линия связи, зарегистрированная в клиенте оператора, с помощью команды New-CSOnlinePSTNGateway. Например: 
   
   ```PowerShell
   New-CSOnlinePSTNGateway -FQDN customers.adatum.biz -SIPSignalingport 5068 -ForwardPAI $true
    ```

-  Производная магистраль, которая не требует регистрации. Это просто нужное имя хоста, добавленное из связи оператора связи. Все параметры конфигурации извлекались из оператора связи. Производную черту не требуется создавать в PowerShell, а связь с ней основана на имени FQDN (см. подробные сведения ниже).

**Логика и пример подготовка**

- Операторы связи должны настроить только одну магистраль (в домене оператора связи) и управлять Set-CSOnlinePSTNGateway связи. В примере выше это adatum.biz.

- В клиенте клиента оператор должен добавить производное FQDN связи в политики голосовой маршрутии пользователей. Нет необходимости запускать New-CSOnlinePSTNGateway для магистрали.

- Производная магистраль, как следует из названия, наследует или наследует все параметры конфигурации от оператора связи. 

Примеры:
- Customers.adatum.biz — оператор связи, который необходимо создать в клиенте оператора связи.

- Sbc1.customers.adatum.biz — производная магистраль в клиенте клиента, которую не нужно создавать в PowerShell.  Вы можете просто добавить имя производной линии связи в клиенте клиента в политику маршрутизации голосовой связи, не создавая ее (используйте производное FQDN связи при настройке политики голосовой маршрутики в TAC в Teams-Voice-Direct Routing-Voice Routes field SBCs enrolled).

- Оператору связи потребуется настроить запись DNS, разрешая связи с IP-адресом связи SBC.

- Любые изменения, внесенные в ленту оператора связи (в клиенте оператора связи), автоматически применяются к производным. Например, операторы связи могут изменить порт SIP на связи, и это изменение будет изменению для всех производных операторов связи. Новая логика настройки туловища упрощает управление, так как вам не нужно переходить в каждый клиент и изменять параметр на каждой туловище.

- Параметры отправляются только в FQDN оператора связи. Состояние связи операторов связи применяется ко всем производным операторам связи и используется для принятия решений о маршрутинге. Узнайте больше о [параметрах прямой маршрутии](./direct-routing-monitor-and-troubleshoot.md).

- Оператор может разрядить магистраль, а все производные связи будут разумещаться. 
 
> [!NOTE]
> Правила перевода номеров, применяемые к связи с оператором связи, не применяются к производным операторам связи. Это известная проблема. В качестве альтернативного решения необходимо создать правила перевода номеров для каждого клиента клиента.

**Переход с предыдущей модели на оператор связи**
 
При переходе с текущей реализации модели, которая была организована на новую модель, операторы связи должны будут перенастроить связи для клиентов. Удалить связи из клиентов клиента с помощью Remove-CSOnlinePSTNGateway (оставив ее в клиенте оператора связи)-

Мы настоятельно рекомендуем как можно скорее перейти на новое решение, так как мы усовершенствовам мониторинг и подготовка с использованием оператора связи и модели связи.
 
Дополнительные сведения о настройке отправки имени FQDN поддоменов в заглавной части контакта см. в инструкциях [поставщика SBC](#deploy-and-configure-the-sbc).

## <a name="considerations-for-setting-up-multi-tenant-failover"></a>Факторы, которые следует учитывать при настройке отбойного с несколькими клиентами 

Чтобы настроить отбой для среды с несколькими клиентами, необходимо сделать следующее:

- Для каждого клиента добавьте FQDNs для двух разных SBCs. Например:

   customer1.sbc1.contoso.com <br>
   customer1.sbc2.contoso.com <br>

- В политиках маршрутинга голосовой почты в Интернете укажите оба SBCs. Если один из SBC не справился, политика маршрутинга перенаправит вызовы на второй SBC.
