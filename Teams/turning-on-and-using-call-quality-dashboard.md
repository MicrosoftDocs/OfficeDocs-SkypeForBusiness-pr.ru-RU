# <a name="turn-on-and-use-call-quality-dashboard-for-microsoft-teams-and-skype-for-business-online"></a>Включение и использование панели мониторинга качества звонков для Microsoft Teams и Skype для бизнеса Online

В этой статье объясняется, как настроить организацию Office 365 для отслеживания качества связи с помощью панели мониторинга качества звонков.
  
Панель мониторинга качества звонков (CQD) обеспечивает понимание качества звонков, выполненных в Microsoft Teams и службах Skype для бизнеса Online. В этой статье описаны действия, которые необходимо выполнить, чтобы начать сбор данных, которые можно использовать для устранения проблем с качеством связи.

В настоящее время CQD версии 3 и CQD версии 2 доступны для использования. CQD v3 доступна по адресу <span>https://cqd.teams.microsoft.com</span>. Войдите в систему с помощью учетных данных администратора Microsoft Teams.

## <a name="latest-changes-and-updates"></a>Последние изменения и обновления

CQD версии 3 обеспечивает около CQD панели мониторинга в реальном времени (задержка до 30 минут) и использует сведения конечного пользователя (ЕУИИ), давая администраторам возможность увеличить масштаб до уровня пользователя. Кроме того, можно создавать отчеты об интерактивных операциях для поддержки новых сценариев, например:

- Качество связи по регионам:
  - Дата по регионам
  - агрегировано до часов по регионам
  - определенные расположения
  - Особая подсеть
  - затронутые пользователи

- Вызвать надежность и сбои по регионам:
  - Дата по регионам
  - агрегировано до часов по регионам
  - определенные расположения
  - Особая подсеть
  - затронутые пользователи

- Оцените мой звонок (РМК) по регионам: от месяца по регионам, сгруппированным по определенному адресу для пользователей, которые предоставляют низкие РМК оценки. CQD v3 также включает в себя точные отзывы.
- Служба поддержки: доступна для определенного пользователя на звонках или собраниях P2P, а также для всех участников и для звонков. Помогает выявить возможные проблемы системы на основе расположения сети, устройств или микропрограммы.  
- Клиентские версии: Просмотр сеанса и количество пользователей для каждой версии клиента или Детализация до имен пользователей для каждой версии клиента. Предварительно созданные фильтры для продукта и типа клиента помогают сосредоточиться на версиях конкретных клиентов.
- Конечные точки: отображает конечные точки компьютера, сопоставленные с ПК или Mac. Показывает обобщенное качество с помощью функций "создать" и "модель". Данные сопоставления загружаются аналогично созданию данных.

Кроме того, версия 3 обеспечивает поддержку RBAC, в случае если ЕУИИ Access недоступен.  

Администратор может управлять Skype для бизнеса Server 2019 (а не только Skype для бизнеса Online и Microsoft Teams) через CQD версии 3. Для этого требуется Гибридная реализация и использование соединителя данных звонка. Дополнительные сведения вы видите в [соединителе данные о звонках планов](/SkypeForBusiness/hybrid/plan-call-data-connector) .

Добавлена версия 2 CQD:

- Данные для Microsoft Teams и Skype для бизнеса Online
- Сводные отчеты включают в себя фильтр продукта для выбора всех данных, данных Microsoft Teams или данных Skype для бизнеса Online.
- Обновлена логика классификации качества потока видео и VBSS. Для определений классификаторов можно использовать [классификацию потока на панели мониторинга качества вызова](stream-classification-in-call-quality-dashboard.md) .

В этой статье приведены сведения о том, какие [измерения и меры доступны на панели мониторинга качества звонков](dimensions-and-measures-available-in-call-quality-dashboard.md).

CQD версии 1 в Skype для бизнеса Server 2015 администраторы предоставляют следующие возможности:

## <a name="cqd-near-real-time-nrt-data"></a>CQD (НРТ) данных в режиме реального времени

CQD v3 использует веб-канал данных в ближайшем режиме реального времени. Записи звонков доступны на портале CQD в течение 30 минут после окончания звонка. Записи звонков из конвейера НРТ доступны только в течение нескольких месяцев, прежде чем они будут удалены из набора данных. CQD v3 объединяет данные из текущего конвейера v2 с данными НРТ из конвейера v3. Запросы на порталах v2 и v3 для данных из архивного периода выдают одинаковые результаты. Запросы данных v2 и v3 для данных НРТ и НРТ, а также для данных и многоpiiных периодов будут разными.

### <a name="piieuii-data"></a>Данные PII/ЕУИИ

PII или ЕУИИ данные поступают только из конвейера v3. Из-за того, что в целях соответствия требованиям персональные и ЕУИИ данные хранятся только в течение 30 дней. Поскольку данные НРТ пересекается с 30-дневной меткой, поля PII/ЕУИИ очищаются, что приводит к бесплатной НРТ информации. Поля PII/ЕУИИ:

- Полный IP-адрес
- MAC-адрес
- Идентификатор базового Service Set (BSSID)
- Универсальный код ресурса (SIP) для протокола запуска сеанса (только для Skype для бизнеса)
- Имя участника-пользователя (UPN)
- Имя конечной точки компьютера
- Отзывы пользователей
- Идентификатор объекта (ИД объекта службы каталогов Active Directory для пользователя конечной точки)

### <a name="date-controls"></a>Элементы управления датой

CQD v3 добавляет следующие новые типы чередующихся тенденций.

- 5-дневный период
- 7-дневный период
- 30 дней
- 60 – день
- 90 – день

Параметр даты URL-адреса теперь может допускать поле дня. В отчетах за день недели используются даты, заданные в формате гггг-мм-дд, как и последний день тенденции.  Параметр даты URL-адреса "00" указывает на "сегодня".

|АДРЕСА| Конечная дата тенденции изменения дневного дня|
|:---|:---|
|<span>https://<cqdv3>/СПД/#/дашбоард/<reportid>/2019-02/</span>   |Текущий день — Фев 2019|
|<span>https://<cqdv3>/СПД/#/дашбоард/<reportid>/2019-02-15/</span>|15 февраля 2019 г.|
|<span>https://<cqdv3>/СПД/#/дашбоард/<reportid>/00/</span>        |Текущий день|
|||

По умолчанию текущий день месяца используется в качестве последнего дня тенденции за несколько дней.

### <a name="drill-thru-functionality"></a>Функции детализации

CQD v3 поддерживает использование полей детализации и детализации в отчетах в SPD. Если выбраны эти поля измерения, отчет автоматически открывает другую вкладку отчета и фильтры по выбранному значению. Поля с назначенным фильтром детализации определяются другим значком курсора (указательом) при наведении на них указателя мыши.

Если выбрано поле детализации, панель мониторинга автоматически переходит на новую заданную вкладку и применяет фильтр к выбранному значению. Если на этой вкладке есть собственные поля детализации и выбрано значение, Предыдущая детализация фильтров и новая функция переносятся вперед. Это позволяет создавать отчеты, которые постепенно сужает результирующий набор данных.

Например, в отчете с детализацией качества связи пользователь может щелкнуть дату, которую вы хотели бы сделать сквозной, что приводит к вкладке "расположение".

Вы можете добавить несколько дат на вкладке "расположение", например добавить 2019-09-22 к дате: 2019-09-24: 

![Снимок экрана: Добавление даты к отчету о переходе](media/CQD-add-date.png)

## <a name="activate-microsoft-call-quality-dashboard-cqd-summary-reports"></a>Активация сводных отчетов на панели мониторинга качества вызовов Microsoft (CQD)

Прежде чем приступить к работе с CQD, активируйте его для своей организации Office 365, выполнив указанные ниже действия.

![Значок, показывающий логотип](media/teams-logo-30x30.png) Microsoft Teams в **центре администрирования Microsoft Teams**

1. Войдите в свою организацию Office 365 с помощью учетной записи администратора Microsoft Teams Service, а затем щелкните плитку **Администратор** , чтобы открыть центр администрирования.
2. В левой области в разделе **центры администрирования**выберите **Microsoft Teams** , чтобы открыть центр администрирования Microsoft Teams.
3. В центре администрирования Microsoft Teams выберите **панель мониторинга качества звонков** на левой панели.
4. На странице, которая открывается \(HTTPS://<span>CQD.Teams.Microsoft.com<span/>\), нажмите **Вход** и введите глобальную учетную запись администратора или сведения об учетной записи администратора Microsoft Teams Service.

    ![Снимок экрана: отображение запроса учетных данных](media/ac4c1699-d8c1-4bda-af30-0fec35b5fd22.png)
  
После того как вы выполните вход, после активации CQD начнет собирать и обрабатывать данные.  
> [!NOTE]
> Обработка достаточного количества данных для отображения значимых результатов в отчетах может занять одно или несколько часов.

![Значок логотипа](media/sfb-logo-30x30.png) Skype для бизнеса **с помощью устаревшего портала Skype для бизнеса**

1. Войдите в свою организацию Office 365 с помощью учетной записи администратора, а затем щелкните плитку **Администратор** , чтобы открыть центр администрирования.
2. В левой области в разделе **центры администрирования**выберите **Microsoft Teams** , чтобы открыть центр администрирования Microsoft Teams.
3. В области "центр администрирования Microsoft Teams" выберите " **устаревший портал** " на левой панели, щелкните " **инструменты**", а затем выберите **панель мониторинга качества звонков в Skype для бизнеса Online**.

     ![Снимок экрана: выбор панели мониторинга качества связи](media/6cc7f80f-b8e2-4a9b-aab8-ac871d07a261.png)

4. На открывшейся странице войдите в систему с помощью учетной записи глобального администратора, а затем укажите учетные данные для учетной записи при появлении соответствующего запроса.

После того как вы выполните вход, после активации панель мониторинга качества связи начнет собирать и обрабатывать данные.

Все выпуски CQD обеспечивают высокоуровневый контрольную точку для общих данных о звонке, но способ представления данных в сводных отчетах отличается от подробных отчетов.  
  
Сводные отчеты — это упрощенный режим отчета на странице с вкладками, благодаря чему вы сможете быстро просматривать и анализировать общее состояние качества связи и тенденции.

- **Общее качество связи** — предоставляет сведения обо всех потоках, которые представляют собой статистические и ежедневные тенденции.
  - Серверные и клиентские потоки
  - Клиент-клиентские потоки
  - Разделение потоков клиента и клиента сервера

- **Клиент — клиент** — предоставляет сведения о потоках между двумя конечными точками клиента.

### <a name="overall-call-quality-tab"></a>Вкладка "Общее качество связи"

С помощью данных на этой вкладке можно оценить состояние качества связи и тенденции в зависимости от количества потоков и некачественных процентных значений. Легенда в верхнем правом углу показывает, какие цвета и визуальные элементы представляют эти показатели.
  
![Снимок экрана: отображение вкладки "качество звонка"](media/c8d183b1-6592-49b0-a81d-35cc0568d5f0.png)
  
Потоки делятся на три группы: хорошие, плохие и Неклассифицированные. Также рассчитываются *некачественные* значения "%", которые предоставляют процент *неудовлетворительных* потоков на общее число потоков, классифицированных. Так как *плохое значение% = слабые потоки/(слабые потоки + хорошие потоки) * 100*, *некачественный%* не влияет на наличие нескольких *неклассифицированных* потоков. Чтобы узнать, в каких случаях поток считается неудовлетворительным или качественным, ознакомьтесь со статьей [классификация потоков на панели мониторинга качества звонков](stream-classification-in-call-quality-dashboard.md).
  
Используйте шкалу слева для измерения значений количества потоков.
  
![Снимок экрана: отображение значений количества потоков](media/850bd25d-d9b2-4df4-8ca6-526a528897c2.png)
  
Используйте шкалу справа для измерения слабых% значений.
  
![Снимок экрана: отображение неудовлетворительных значений "%"](media/29795f71-ca96-4763-a76c-b4bb7c0e5828.png)

### <a name="server-client-tab-and-client-client-tabs"></a>Вкладка "сервер-клиент" и "клиент-клиент"

Эти две вкладки содержат сведения о потоках, которые выполнялись в сценариях конечной точки. Вкладка "сервер-клиент" содержит четыре свертываемых раздела, которые представляют собой четыре сценария, в которых потоки мультимедиа будут перетекать.
  
- Проводные внутренние
- Проводные внешние
- Wi-Fi Inside
- Wi-Fi за пределами

Аналогичным образом на вкладке Клиент-клиент выводятся пять свертываемых разделов.

- Проводная внутренняя — проводные внутренние
- Проводные внутри — проводные внешние
- Проводные внешние — проводная внешняя
- Проводная внутренняя — WiFi Inside
- Проводная внутренняя — WiFi снаружи

#### <a name="inside-test"></a>Внутренняя проверка

Во время обработки CQD Серверная часть классифицирует поток как *внутри* , так и *за ее пределами* с помощью сборки данных, если она существует. Конечные точки каждого потока связаны с адресом подсети. Если подсеть находится в списке подсетей, помеченных Инсидекорп в загруженных сведениях о сборке, они рассматриваются *внутри*. Если сведения о сборках еще не были отправлены, в рамках теста всегда классифицируются потоки *за пределами*.  

> [!NOTE]
> В сценарии "сервер-клиент" внутри теста есть только конечная точка клиента. Поскольку серверы всегда находятся за пределами точки зрения пользователя, это не учитывается при тестировании.
  
#### <a name="wired-vs-wifi"></a>Проводная и WiFi

Как видно из имен, критерии классификации основываются на типе клиентских подключений. Опять же, сервер всегда будет проводен и не будет включен в вычисление.
  
> [!NOTE]
> Поток, если одна из двух конечных точек подключена к сети Wi-Fi, она классифицируется как Wi-Fi в CQD.

### <a name="tenant-data-file-format-and-structure"></a>Формат и структура файлов данных клиента

<a name="BKMKTenantDataFile"> </a>

### <a name="building-data-file"></a>Создание файла данных

CQD использует файл данных здания, который помогает получить полезные сведения о звонке. Столбец Subnet является производным, разворачивая столбец Network + Нетворкранже, а затем присоединяется к столбцу подсети в первой подсети или столбце второй подсети, чтобы отобразить сведения о здании, городе, стране или регионе. Формат загружаемого файла данных должен соответствовать следующим критериям для прохождения проверки перед отправкой:

Образец шаблона можно загрузить [здесь](https://github.com/MicrosoftDocs/OfficeDocs-SkypeForBusiness/blob/live/Teams/downloads/locations-template.zip?raw=true)
  
- Файл должен иметь формат TSV (столбцы разделяются ТАБУЛЯЦИей) или CSV-файл (столбцы разделяются запятыми).
- Файл данных не содержит строку заголовка таблицы. Предполагается, что первая строка файла данных является реальными данными, а не метками заголовков, например "сеть".
- Типы данных в файле могут быть только строкой, целым числом или логическим значением. Для целочисленного типа данных значение должно быть числовым значением. Логические значения должны быть как 0, так и 1.
- Если столбец использует строковый тип данных, поле данных может быть пустым, но по-прежнему должно разделяться символом табуляции или запятыми. Пустое поле данных просто присваивает пустую строковое значение.
- Для каждой строки должно быть 14 столбцов, каждый столбец должен иметь соответствующий тип данных, а столбцы должны быть в том порядке, в котором они указаны в приведенной ниже таблице.

||||||||||||||||
|:--- |:--- |:--- |:--- |:--- |:--- |:--- |:--- |:--- |:--- |:--- |:--- |:--- |:---  |:--- |:---|
|**Имя поля столбца**|нетворкип  |нетворкнаме              |NetworkRange|BuildingName  |овнершиптипе| буилдингтипе  |BuildingOfficeType|City   |ZipCode|Страну|State |Region|InsideCorp&dagger;|ExpressRoute&Dagger;|VPN (необязательно)|
|**Тип данных**        | String    | String                  |Число      | String       | String      | String        |String            |String |String |String |String|String|Boolean   |Boolean     |Boolean|
|**Пример значения**    |192.168.1.0|США/Сиэтл/Сиэтл (SEA-1)| 26         | СИЭТЛ — МОРСКОЙ ВОЛНЫ (1 – 7)| Contoso     | Прекращение ИТ|Техника       |Сиэтл|98001  |Американские     |Красноярский край    |мсус  | 1        |до           | до|
|||||||||||||||||

&dagger;Этот параметр можно использовать, чтобы показать, входит ли подсеть в корпоративную сеть. При необходимости вы можете настроить использование других целей.

&Dagger;Этот параметр можно использовать, чтобы показать, использует ли сеть Azure ExpressRoute. При необходимости вы можете настроить использование других целей.  

**Образец строки:**

`192.168.1.0,USA/Seattle/SEATTLE-SEA-1,26,SEATTLE-SEA-1,Contoso,IT Termination,Engineering,Seattle,98001,US,WA,MSUS,1,0,0`

> [!IMPORTANT]
> Диапазон сети может использоваться для представления Супернет (сочетание нескольких подсетей с одним префиксом маршрутизации). Для всех новых выгрузок построения будут проверяться все перекрывающиеся диапазоны. Если вы ранее загрузили файл здания, вы должны скачать текущий файл и повторно загрузить его, чтобы определить перекрытие и устранить проблему перед повторной отправкой. Перекрытие в ранее загруженных файлах может привести к неправильному сопоставлению подсетей для зданий в отчетах. Некоторые реализации виртуальных частных сетей не сообщат сведения о подсети. При добавлении подсети VPN в файл построения вместо одной записи для подсети рекомендуется добавлять отдельные записи для каждого адреса в подсети VPN в виде отдельной сети 32-бит. Каждая строка может иметь одинаковые стандартные метаданные. Например, вместо одной строки для 172.16.18.0/24 необходимо иметь 256 строк с одной строкой для каждого адреса между 172.16.18.0/32 и 172.16.18.255/32 включительно.
>
> VPN-столбец является необязательным и по умолчанию будет равен 0.  Если для столбца VPN задано значение 1, подсеть, представленная этой строкой, будет полностью развернута для соответствия всем IP-адресам в подсети.  Используй это экономно и только для подсетей VPN, так как полное расширение этих подсетей может отрицательно повлиять на время запроса запросов, связанных с созданием данных.

### <a name="endpoint-data-file"></a>Файл данных конечной точки

CQD использует файл данных конечной точки. Значения столбцов используются в первом имени конечной точки клиента записи вызова или в столбце имени конечной точки клиента для отображения сведений о создании конечной точки, модели или типе. Формат загружаемого файла данных должен соответствовать следующим критериям для прохождения проверки перед отправкой:

- Файл должен иметь формат TSV (столбцы разделяются ТАБУЛЯЦИей) или CSV-файл (столбцы разделяются запятыми).
- Содержимое файла данных не содержит заголовков таблиц. Предполагается, что первая строка файла данных является реальными данными, а не меткой заголовка, например "EndpointName".
- Все семь столбцов используют только тип данных String. Максимально допустимая длина составляет 64 символов.
- Поле данных может быть пустым, но по-прежнему должно разделяться символом табуляции или запятыми. Пустое поле данных просто присваивает пустую строковое значение.
- EndpointName должны быть уникальными, в противном случае отправка завершается сбоем. Если в одной строке или двух строках используется один и тот же элемент EndpointName, конфликт может привести к неправильному присоединению.
- EndpointLabel1, EndpointLabel2 и EndpointLabel3 являются настраиваемыми наклейками. Они могут быть пустыми строками или значениями, например "ИТ-отдел обозначены в 2018 ноутбука" или "актива 5678".
- Для каждой строки должен быть семь столбцов, а столбцы должны находиться в указанном ниже порядке.

  **Порядок полей:**

EndpointName, Ендпоинтмодел, Ендпоинттипе, EndpointLabel1, EndpointLabel2, EndpointLabel3

  **Образец строки:**

`1409W3534, Fabrikam Model 123, Laptop, IT designated 2018 Laptop, Asset Tag 5678, Purchase 2018,`

## <a name="frequently-asked-questions"></a>Ответы на часто задаваемые вопросы

### <a name="why-does-my-cqd-v2-report-data-look-different-than-the-cqd-v3-report-data"></a>Почему данные отчета CQD v2 выглядят иначе, чем данные отчета CQD v3? 

Если вы видите различия данных между CQD v2 и v3, убедитесь, что сравнение данных и их проверка производится на уровне "яблоки-яблоки" и узкого уровня, а не на объединенном уровне. Например, если вы отфильтровать оба отчета для МСИТ "сборка 30 ' Wi-Fi для настольных клиентов, то процент низкого качества должен быть одинаковым в пределах от v2 до v3.

CQD v2 и CQD v3 имеют разные общие показатели, так как в CQD v2 есть новые сценарии, не представленные в CQD v3. Суммарные итоги или агрегированные все значения без фильтров должны отличаться.  

Если сценарий использования включает в себя звонки в Skype для бизнеса Server 2019, данные CQD v3 включают звонки Skype Bot (автосекретаря, Кви, интерфейс виртуальных рабочих столов), события Live Events и звонки по протоколу PSTN. CQD v2 не использует эти данные. (Для CQD v3 требуется Skype для бизнеса Server 2019 с настроенным соединителем облачных данных.)

Например, если в сводном отчете CQD v2 отображаются звуковые потоки 200 000 с ошибками 5000, это не будет считаться нечастой записью звуковых потоков 300 000 с 5500 ошибками (это может быть вызвано тем, что в Skype для бизнеса Server 2019 звонки, звонки Кви, звонки по протоколу PSTN и т. д.) в сводном отчете CQD v3.

Для устранения неоднозначности непредвиденных различий рассмотрим более одной разбивки на общие данные. Отфильтруйте данные одним или несколькими из следующих параметров:

- User Agent Category Pair
- Первый продукт
- Второй продукт

### <a name="other-expected-differences-between-cqd-v2-and-cqd-v3"></a>Другие ожидаемые различия между CQD v2 и CQD v3

Существует несколько улучшений качества и надежности групп, но не Skype для бизнеса Online:

- Автоматическое повторное подключение
- Быстрый роуминг
- Улучшенное управление BWми

При сравнении данных для следующих двух служб:

- Выберите сценарий с неограниченным фокусом, например Корпоративная Проводная связь, Настольные компьютеры с Windows или единая область или здание.
- Проверьте диапазоны IP-адресов для команд MR, TR и MP. Диапазоны команд более новые, чем Skype для бизнеса Online, и это может вызвать проблемы с подключением, связанные с брандмауэрами
- Не сравнивать сводку и номера верхнего уровня. Эти сравнения помогут вам сравнить большой объем звонков в Skype для бизнеса Online на корпоративном сетевом подключении с небольшим объемом звонков в сетях на LTE или в частную сеть.
- Будьте осторожны с разницей в местоположении и заполнении: существует множество сравнений, которые могут быть очень полезны.
  - СЕВЕРНАЯ: APAC
  - Москва: Гоа
  - Wired : wifi
  - Корпоративная Сеть: Домашняя сеть
