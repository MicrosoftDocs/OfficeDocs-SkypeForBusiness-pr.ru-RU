# <a name="turning-on-and-using-call-quality-dashboard-for-microsoft-teams-and-skype-for-business-online"></a>Включение и использование панели мониторинга качества звонков для Microsoft Teams и Skype для бизнеса Online

В этой статье объясняется, как настроить организацию Office 365 для отслеживания качества связи с помощью панели мониторинга качества звонков.
  
Панель мониторинга качества звонка (ПМКЗ) для Microsoft Teams и Skype для бизнеса Online позволяет получить представление о качестве звонков, сделанных с помощью команд Microsoft и Skype для бизнес-услуг. В этой статье описаны действия, которые необходимо выполнить для начала сбора данных.
  
  
## <a name="latest-changes-and-updates"></a>Последние изменения и обновления

Последние изменения в CQD описаны ниже.
  
- Включает данные Microsoft Teams в дополнение к данным Skype для бизнеса Online.
    
- Сводные отчеты включают в себя фильтр продукта для выбора всех данных, данных Microsoft Teams или данных Skype для бизнеса Online.

- Логика классификации качества потоков видео и VBSS обновлена. Для последних определений классификаторов ознакомьтесь со сведениями о [классификации потоков на панели мониторинга качества звонков](stream-classification-in-call-quality-dashboard.md) .

В этой статье приведены сведения о том, какие [измерения и меры доступны на панели мониторинга качества звонков](dimensions-and-measures-available-in-call-quality-dashboard.md).
  
> [!NOTE]
> Сведения об обновлениях и изменениях, внесенных в информационную панель, можно найти, щелкнув ссылку в разделе **хорошие новости!** баннер, отображаемый на панели мониторинга.
  
## <a name="activate-microsoft-call-quality-dashboard-cqd-summary-reports"></a>Активация сводных отчетов на панели мониторинга качества вызовов Microsoft (CQD)

Прежде чем приступить к работе с CQD, вам потребуется активировать его для своей организации Office 365.
![Teams-Logo-30x30. png](media/teams-logo-30x30.png) **с помощью центра администрирования Microsoft Teams**
 
1. Войдите в свою организацию Office 365 с помощью учетной записи администратора Microsoft Teams Service, а затем щелкните плитку **Администратор** , чтобы открыть центр администрирования.
    
2. В левой области в разделе **центры администрирования**выберите **Microsoft Teams** , чтобы открыть центр администрирования Microsoft Teams.
    
3. В центре администрирования Microsoft Teams выберите **панель мониторинга качества звонков** на левой панели.
    
  
4. На открывшейся странице войдите в систему с помощью учетной записи глобального администратора или учетной записи администратора Microsoft Teams Service, а затем укажите учетные данные для учетной записи при появлении соответствующего запроса.
    
     ![Вход в CQD](media/ac4c1699-d8c1-4bda-af30-0fec35b5fd22.png)
  
После того как вы выполните вход, после активации CQD начнет собирать и обрабатывать данные.  
> [!NOTE]
> Обработка достаточного количества данных для отображения значимых результатов в отчетах может занять несколько часов. 

![SFB-Logo-30x30. png](media/sfb-logo-30x30.png) **с помощью центра администрирования Skype для бизнеса**
 
1. Войдите в свою организацию Office 365 с помощью учетной записи администратора, а затем щелкните плитку **Администратор** , чтобы открыть центр администрирования.
    
2. В левой области в разделе **центры администрирования**выберите **Skype для бизнеса** , чтобы открыть центр администрирования Skype для бизнеса.
    
3. В центре администрирования Skype для бизнеса выберите **инструменты** на левой панели, а затем выберите **панель мониторинга качества звонков в Skype для бизнеса Online**.
    
     ![Средства Skype для бизнеса](media/6cc7f80f-b8e2-4a9b-aab8-ac871d07a261.png)
  
4. На открывшейся странице войдите в систему с помощью учетной записи глобального администратора, а затем укажите учетные данные для учетной записи при появлении соответствующего запроса.
    
     ![Вход в CQD](media/ac4c1699-d8c1-4bda-af30-0fec35b5fd22.png)
  
После того как вы выполните вход, после активации CQD начнет собирать и обрабатывать данные.

### <a name="out-of-the-box-reports"></a>Готовые отчеты

Оба выпуска CQD предоставляют встроенные показатели качества, позволяя вам присоединиться к метрикам, не создавая никаких новых отчетов. После того как данные обрабатываются в серверной части, вы можете просматривать данные качества связи в отчетах.
  
### <a name="overview-reports"></a>Обзорные отчеты

Оба издания CQD предоставляют высокоуровневый контрольную точку на общую информацию о связи, но способ представления данных в сводных отчетах отличается от подробных отчетов.
  
Сводные отчеты включают упрощенное представление отчета на вкладке, которое позволяет пользователям быстро просматривать и понимать общее состояние качества связи и тенденции.
  
Четыре вкладки включают:
  
- **Общее качество связи** — представлена информация обо всех потоках, которые представляют собой агрегаты потоков сервера и клиента, а также отдельные потоки клиент-сервер и клиент-клиент — в форме ежемесячных и ежедневных тенденций.
    
- **Сервер-клиент** — предоставляет дополнительные сведения о потоках между сервером и конечными точками клиента.
    
- **Клиент-клиент** — предоставляет дополнительные сведения о потоках между двумя конечными точками клиента.
    
- **Соглашение об уровне обслуживания голосовой связи** — сведения о звонках, включенных в соглашение об уровне обслуживания голосовой связи в Skype для бизнеса Online.

#### <a name="inside-test"></a>Внутренняя проверка

Во время обработки CQD Серверная часть классифицирует поток как *внутри* , так и *за ее* пределами с помощью сборки данных, если она существует. Конечные точки каждого потока связаны с адресом подсети. Если подсеть находится в списке подсетей, помеченных Инсидекорп в загруженных сведениях о сборке, они рассматриваются *внутри*. Если сведения о сборках еще не были отправлены, в рамках теста будет всегда выполнена классификация потоков *за*пределами. Обратите внимание на то, что внутри теста для сценария сервера-клиента только считается конечная точка клиента. Поскольку серверы всегда находятся за пределами точки зрения пользователя, это не учитывается при тестировании.
  
#### <a name="wired-vs-wifi"></a>Проводная и WiFi

Как видно из имен, это критерии классификации, основанные на типе клиентских подключений. Опять же, сервер всегда будет проводен и не будет включен в вычисление.
  
> [!NOTE]
> Поток, если одна из двух конечных точек подключена к сети Wi-Fi, она классифицируется как Wi-Fi в CQD.

## <a name="upload-tenant-data-information"></a>Отправка сведений о данных клиента
<a name="BKMKTenantDataInformationUpload"></a>

Информационная панель "Сводные отчеты CQD" включает страницу **отправки данных клиента** , доступную с помощью команды " **отправить данные клиента** " из меню "Параметры" в правом верхнем углу. Эта страница используется для администраторов для передачи собственной информации, такой как сопоставление IP-адреса и географических данных, сопоставление каждого беспроводного ТД и его MAC-адреса, сопоставление конечной точки с конечной точкой (модель/тип) и т. д.
  
![Панель мониторинга CQD](media/839c9ab4-0246-46c9-8402-aafd83a0bc63.png)
  
1. На странице **Отправка данных клиента** в раскрывающемся меню выберите тип файла данных для отправки. Тип данных file указывает на содержимое файла (например, "сборка" — это сопоставление IP-адреса и здания, а также других сведений о географическом формате, "конечная точка" — сопоставление имени конечной точки с конечной точкой создание/модель/тип... сведения). В настоящее время поддерживаются типы данных "сборка" и "конечная точка" для CQD. Teams. Microsoft. com (на этапе предварительной версии и еще не доступно), cqd.lync.com поддерживает только тип данных "здание". В последующих выпусках будут добавлены еще несколько типов данных.
    
2. После выбора типа данных файла нажмите кнопку **Обзор** , чтобы выбрать файл данных.
    
   - Файл данных должен быть файлом в формате TSV (значения, разделенными табуляцией) или CSV-файлом (в виде значений, разделенных запятыми). Если вы используете CSV-файл, то любое поле, содержащее запятую, должно быть заключено в кавычки или удалено с помощью запятой. Например, если названием здания является «Москва», «Москва» в CSV-файле, его следует ввести как «Нью, Нью-Йорк».
    
   - Размер файла данных не должен превышать 50 МБ.

   - Файл, загруженный в cqd.teams.microsoft.com, развернуло ограничение на количество строк 1 000 000 для ускорения производительности запроса. На cqd.lync.com также может быть установлен этот лимит.
    
   - Для каждого файла данных каждый столбец в файле должен соответствовать предопределенному типу данных, который рассматривается далее в этом разделе.
    
3. После выбора файла данных укажите **дату начала** и, при необходимости, **Укажите дату окончания**.
    
4. После выбора **даты начала**нажмите кнопку **Отправить** , чтобы отправить файл на сервер CQD.
    
    Перед передачей файла он сначала проверяется. После проверки оно хранится в большом двоичном объекте Azure. Если проверка не удалась или файл не сохраняется в большом двоичном объекте Azure, выводится сообщение об ошибке, требующее внесения изменений в файл. На приведенном ниже рисунке показана ошибка, возникающая при неправильном числе столбцов в файле данных.
    
     ![Ошибка проверки отправки CQD примера](media/22716a32-3d3d-4870-983c-46089e8b212a.png)
  
5. Если во время проверки появления ошибок не возникло, Отправка файла будет выполнена успешно. Затем вы можете просмотреть загруженный файл данных в таблице **Мои отправленные** данные, в которой показан полный список всех загруженных файлов для текущего клиента в нижней части страницы.
    
    Каждая запись показывает один загруженный файл данных клиента с типом файла, временем последнего обновления, периодом, описанием, значком удаления и значком скачивания. Чтобы удалить файл, щелкните значок корзины в таблице. Чтобы загрузить файл, щелкните значок скачивания в столбце Download ( **скачать** ) таблицы.
    
     ![CQD Моя таблица отправки](media/4168a883-bbea-461a-80b1-42eedf2e7732.png)
  
### <a name="tenant-data-file-format-and-structure"></a>Формат и структура файлов данных клиента
<a name="BKMKTenantDataFile"> </a>

### <a name="building-data-file"></a>Создание файла данных
CQD использует создание файла данных, сначала производный столбец подсети из развернутого столбца Network + Нетворкранже, а затем присоединить столбец подсети к первому столбцу подсети записи вызова/второй подсети, чтобы отобразить здания/город/страна или регион... Информация. Формат загружаемого файла данных должен соответствовать следующим требованиям для прохождения проверки перед передачей.
  
- Файл должен быть либо TSV-файлом, что означает, что в каждой строке столбцы разделяются символом ТАБУЛЯЦИи или CSV-файлом, в котором каждый столбец отделяется запятыми.
    
- Содержимое файла данных не содержит заголовков таблиц. Это означает, что первая строка файла данных должна быть реальными данными, а не заголовками, например "сеть" и т. д.
    
- Для каждого столбца тип данных может быть только строкой, числом или bool. Если это число, оно должно быть числовым значением. Если это значение является логическим, оно должно быть равно 0 или 1.
    
- Для каждого столбца, если тип данных — String, данные могут быть пустыми (но по-прежнему должны разделяться подходящим разделителем (то есть символом табуляции или запятыми). Это просто назначает это поле пустым строковым значением.
    
- Для каждой строки должно быть 14 столбцов, каждый столбец должен иметь следующий тип данных, а столбцы должны быть в том порядке, в котором они указаны в приведенной ниже таблице.
    
|**Имя столбца**|**Тип данных**|**Пример**|
|:-----|:-----|:-----|
|Сеть  <br/> |String  <br/> |192.168.1.0  <br/> |

|NetworkRange  <br/> |Число  <br/> |26  <br/> |

|Буилдингтипе  <br/> |String  <br/> |Прекращение ИТ  <br/> |
|BuildingOfficeType  <br/> |String  <br/> |Техника  <br/> |

> [!IMPORTANT]
> Диапазон сети может использоваться для представления Супернет (сочетание нескольких подсетей с одним префиксом маршрутизации). Для всех новых выгрузок построения будут проверяться все перекрывающиеся диапазоны. Если вы ранее загрузили файл здания, вы должны скачать текущий файл и повторно загрузить его, чтобы определить перекрытие и устранить проблему перед повторной отправкой. Перекрытие в ранее загруженных файлах может привести к неправильному сопоставлению подсетей для зданий в отчетах. Некоторые реализации виртуальных частных сетей не сообщат сведения о подсети. При добавлении подсети VPN в файл построения вместо одной записи для подсети рекомендуется добавлять отдельные записи для каждого адреса в подсети VPN в виде отдельной сети 32-бит. Каждая строка может иметь одинаковые стандартные метаданные. Например, вместо одной строки для 172.16.18.0/24 необходимо иметь 256 строк с одной строкой для каждого адреса между 172.16.18.0/32 и 172.16.18.255/32 включительно. 

### <a name="endpoint-data-file"></a>Файл данных конечной точки
CQD использует файл данных конечной точки, присоединяя его столбец EndpointName к первому доменному имени конечной точки клиента записи или имени конечной точки клиента, чтобы отобразить сведения о создании конечной точки, модели и типа. Формат загружаемого файла данных должен соответствовать следующим требованиям для прохождения проверки перед передачей.

- Файл должен быть либо TSV-файлом, что означает, что в каждой строке столбцы разделяются символом ТАБУЛЯЦИи или CSV-файлом, в котором каждый столбец отделяется запятыми.

- Содержимое файла данных не содержит заголовков таблиц. Это означает, что первая строка файла данных должна быть реальными данными, а не заголовками, например "EndpointName" и т. д.

- Для каждого столбца тип данных может быть только строкой и содержать не более 64 знаков, максимально допустимую длину.

- Для каждого столбца данные могут быть пустыми (но по-прежнему должны разделяться соответствующим разделителем (например, символом табуляции или запятыми). Это просто назначает это поле пустым строковым значением.

- Для каждой строки должно быть 7 столбцов, а столбцы должны быть в том порядке, в котором они указаны в приведенной ниже таблице.

- Если в противном случае перезагрузка не выполнена из-за повторения строки, она должна быть уникальной, так как она вызовет неправильное соединение.

-  EndpointLabel1, EndpointLabel2, EndpointLable3 — это настраиваемые метки, которые могут быть пустыми строками или значениями, например "ИТ-отдел обозначены 2018 ноутбук", "название актива 5678"... документов.

|**Имя столбца**|**Тип данных**|**Пример**|
|:-----|:-----|:-----|
|EndpointName  <br/> |String  <br/> |1409W3534  <br/> |
|Ендпоинтмаке  <br/> |String  <br/> |Fabrikam Inc  <br/> |
|Ендпоинтмодел  <br/> |String  <br/> |Модель Fabrikam 123  <br/> |
|Ендпоинттипе   <br/> |String  <br/> |Компьютера  <br/> |
|EndpointLabel1  <br/> |String  <br/> |ОН обозначен 2018 ноутбуком  <br/> |
|EndpointLabel2  <br/> |String  <br/> |Тег Asset 5678  <br/> |
|EndpointLabel3  <br/> |String  <br/> |Покупка 2018   <br/> |


## <a name="selecting-media-type-in-detailed-reports"></a>Выбор типа мультимедиа в подробных отчетах
<a name="BKMKMediaType"></a>

В подробных отчетах поддерживается просмотр качества и надежности мультимедиа для файлов мультимедиа, а также общего обмена видеофайлами, видеороликами и приложениями на основе видео. Измерения, меры и фильтры, специфичные для одного типа мультимедиа, имеют тип "звук", "видео", "AppSharing" или "VBSS".
