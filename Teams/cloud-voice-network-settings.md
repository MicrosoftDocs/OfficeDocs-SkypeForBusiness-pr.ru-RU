# <a name="network-settings-for-cloud-voice-features-in-microsoft-teams"></a>Сетевые параметры для функций голосовой связи в облаке в Microsoft Teams

Сведения о регионах сети, сетевых сайтах, подсетях сети и о доверенных IP-адресах. Эти термины и понятия используются в нашей голосовой документации по облачным технологиям для [маршрутизации на основе местоположения для прямой маршрутизации](location-based-routing-plan.md) и [динамической службы экстренного вызова](configure-dynamic-emergency-calling.md). Если вы развертываете эти облачные функции в вашей организации, необходимо настроить параметры сети для использования с этими функциями в Microsoft Teams.

В этой статье вы найдете общие сведения о параметрах сети, которые являются общими для маршрутизации на основе местоположения и динамического вызова экстренной помощи. В зависимости от того, какая функция голосовой связи облачных функций и возможностей развертывается, вы можете настроить некоторые или все эти параметры. Инструкции по настройке этих параметров [можно найти в статье Управление топологией сети для облачных функций в Teams](manage-your-network-topology.md).

> [!NOTE]
> Любые особые требования к компонентам для настройки сети описаны в разделе Настройка этой функции.

## <a name="network-region"></a>Сетевой регион

Область сети содержит коллекцию сетевых сайтов. Она соединяет различные части сети в нескольких географических регионах. Например, если ваша организация содержит много сайтов в Индии, вы можете указать "Индия" в качестве сетевого региона. Каждый сетевой сайт должен быть связан с сетевым регионом.

Одни и те же сетевые регионы совместно используются в маршруте на основе местоположения для прямой маршрутизации и расширенных служб экстренной помощи. Если вы уже создали сетевые регионы для одной из функций, вам не нужно создавать новые сетевые регионы для других функций.

## <a name="network-site"></a>Сетевой сайт

Сетевой сайт — это расположение, в котором Организация имеет физическое место проведения, например Office, набор зданий или кампус. Сетевые сайты определяются как набор IP-подсетей. Каждый сетевой сайт должен быть связан с сетевым регионом.

Вы также можете использовать сетевые сайты для включения и настройки вызова экстренной помощи.

## <a name="network-subnet"></a>Сетевая подсеть

Каждая подсеть должна быть связана с определенным сетевым сайтом. Расположение клиента определяется на основе сетевой подсети и связанного сайта сети. Вы можете связать несколько подсетей с одним сетевым сайтом, но вы не можете связать несколько сайтов с одной и той же подсетью.

Сведения о подсети используются для определения сетевого сайта, на котором находится конечная точка при инициировании нового сеанса. Когда место каждого абонента в сеансе известно, функция голосовой связи с облаком может применить эти данные, чтобы определить способ обработки настройки и маршрутизации звонков.

Для каждого сетевого сайта свяжитесь с администратором сети, чтобы определить, какие IP-подсети назначены каждому сетевому сайту. В нашем примере сайту "Нью-Йорк" в регионе "Северная Америка" назначены следующие IP-подсети: 172.29.80.0/23, 157.57.216.0/25, 172.29.91.0/23, 172.29.81.0/24. Если Боб, который обычно работает в Детройт, перейдет в Office 365 для обучения, включит его компьютер и подключится к сети, он получит IP-адрес в одном из четырех диапазонов, которые распределены в Нью-Йорке, например 172.29.80.103.

## <a name="trusted-ip-address"></a>Доверенный IP-адрес

Доверенные IP-адреса — это внешние IP-адреса в сети Организации. Они определяют, входит ли конечная точка пользователя в корпоративную сеть, прежде чем проверять соответствие определенного сайта.

Если внешний IP-адрес пользователя соответствует IP-адресу, указанному в списке доверенный IP-адрес, функция облачного голосовой связи проверяет внутреннюю подсеть, в которой находится конечная точка пользователя. Соответствие можно получить по IP-адресам IPv4 или IPv6 и зависит от формата пакета IP, отправляемого в параметры сети. (Если у общедоступного IP-адреса есть и IPv4, и IPv6, необходимо добавить как доверенные IP-адреса.)

Если внешний IP-адрес пользователя не соответствует IP-адресу, указанному в списке доверенный IP-адрес, конечная точка классифицируется как неизвестная.
