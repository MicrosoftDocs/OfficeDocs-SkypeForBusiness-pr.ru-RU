# <a name="migration-and-interoperability-guidance-for-organizations-using-teams-together-with-skype-for-business"></a>Руководство по миграции и взаимодействия для организаций, с помощью команды Скайп для бизнеса

Взаимодействие и миграция осуществляется с помощью «сосуществования режима» согласно TeamsUpgradePolicy. Выбор режима пользователя управляет маршрутизации входящих звонков и чатов и ли пользователь планирует собраний в группы или Скайп для бизнеса.  В ближайшее время в сочетании с предстоящие TeamsAppPermissionsPolicy режим также преимущественную силу в которой клиентские пользователь может запустить чаты и звонки. 

Как ранее вышеперечисленные 2018 апреля TeamsInteropPolicy — это сейчас не поддерживается. Его функциональные возможности были объединить в TeamsUpgradePolicy и настройка TeamsInteropPolicy больше не требуется. Не применяется только при наличии TeamsUpgradePolicy mode = прежних версий.  Теперь, когда поддержка TeamsUpgradePolicy завершен, *клиентам следует обновить их конфигурации для использования режима, отличного от прежних версий*. Предоставление экземпляры TeamsUpgradePolicy с режимом = прежних версий заблокирована по умолчанию.

## <a name="fundamental-concepts"></a>Основные понятия

1.  *Взаимодействия* : 1-1 обмена данными между Lync/Скайп для корпоративных пользователей и групп пользователей.

2.  *Федерация* : обмена данными между пользователями из различных клиентов.

3.  Все команды, у пользователей есть базового Скайп для бизнеса учетной записи, являющейся «размещенных» либо через Интернет или локальной:
    - Пользователи, уже используется Скайп для бизнеса в Интернет использовать их существующей учетной записи online.
    - Уже используется Скайп для бизнеса/Lync локальных пользователей используйте их существующей локальной учетной записи.
    - Пользователи, для которых не удалось обнаружить существующий Скайп для бизнес-партнер будет иметь Скайп для бизнеса в Интернет учетной записи автоматически подготавливаться при создании группы пользователей. Не Скайп для бизнеса лицензии не требуется.

4.  При наличии локального развертывания либо Скайп для бизнеса или Lync, и эти пользователи должны быть группами пользователей, по крайней мере убедитесь, что подключение Azure AD выполняет синхронизацию msRTCSIP-DeploymentLocator атрибут в AAD, поэтому этой команды/Скайп для бизнеса Online правильно обнаруживает в локальную среду. Кроме того для перемещения всех пользователей в режим только для группы (например, обновление пользователя), *необходимо настроить Скайп для бизнеса гибридный режим*. Для получения дополнительных сведений см [Настройка Azure AD для Скайп для бизнеса и рабочих групп](https://docs.microsoft.com/en-us/SkypeForBusiness/hybrid/configure-azure-ad-connect).

5.  Взаимодействие между рабочими группами и Скайп для бизнес-пользователи — только возможные *Если Интернет-версия расположен пользователь групп в Скайп для бизнеса*. Получатель Скайп для бизнес-пользователь может быть размещен либо локальной (и требуется настройка Скайп для гибридных Business) или в оперативном режиме. Пользователи, которые размещаются в Скайп для бизнеса в локальной можно использовать команды в режиме острова (задать далее в этом документе), но не могут использовать команды для взаимодействия и создать федерацию с других пользователей, использующих Скайп для бизнеса.  

6.  Для обновления пользователя к группам (то есть, предоставить им TeamsUpgradePolicy с режимом = TeamsOnly), пользователь должен быть размещен в Скайп для бизнеса через Интернет. Это необходимо для обеспечения взаимодействия, федерации и полного управления группами пользователей. Чтобы обновить пользователи, размещенные локально, используйте `Move-CsUser` из локальной admin средства для первого move пользователю Скайп для бизнеса в Интернет:

    - Если у вас есть Скайп Business Server 2019 или CU8 для Скайп для Business Server 2015, укажите `-MoveToTeams` переключитесь в `Move-CsUser` для перемещения пользователей группам.
    - В противном случае — после `Move-CsUser` завершается, назначить режим TeamsOnly пользователю с помощью PowerShell или группами центра администрирования. 

7.  Основные политики для управления обновлением и взаимодействия — TeamsUpgradePolicy. TeamsInteropPolicy больше не учитываются при использовании режима TeamsUpgradePolicy = устаревшим и клиентов, использующих режим = прежних версий следует обновить их конфигурации TeamsUpgradePolicy для использования в другой режим.  Предоставление mode = прежних версий заблокирована по умолчанию, хотя администраторы могут переопределить это с помощью `-Force` на данный момент. В конце концов `-Force` коммутатор будет удален и предоставление режим = прежних версий будет невозможно. 

8.  Использование функции групп телефонной системой с группами, пользователи должны быть в режиме TeamsOnly (то есть, расположенным в Скайп для бизнеса в Интернет и обновлены для группы) и должен быть настроен либо для телефонной системой Microsoft [Прямой маршрутизации](https://techcommunity.microsoft.com/t5/Microsoft-Teams-Blog/Direct-Routing-is-now-Generally-Available/ba-p/210359#M1277) (который позволяет использовать телефонной системой с помощью собственных магистралях SIP и SBC) или Office 365 вызов плана.   

9.  Планирование собраний группы с аудиоконференции (по телефону с помощью ТСОП или модем) в настоящее время доступен только для пользователей, которые размещаются в Скайп для бизнеса в Интернет. Для группы пользователей с Скайп локальной учетной записи бизнес поддерживается в TAP.


## <a name="coexistence-modes"></a>Режимы сосуществования

Взаимодействие и миграция управляются зависимости «сосуществования режима» с помощью TeamsUpgradePolicy. Определяет режим пользователя:

- *Маршрутизация входящих*: В какой клиента (группы или Скайп для бизнеса) входящего обсуждений и вызывает Земля? 
- *Планирование на собрания*: какие служба используется для планирования собраний новые и проверка того, что соответствующие надстройки присутствует в Outlook. Обратите внимание на то, что TeamsUpgradePolicy не влияет на присоединения к собранию. Пользователи могут всегда *присоединиться к* любого собрания, будет ли это Скайп для собраний или группами собрания.
- *Взаимодействия с пользователем*: какие функциональные возможности для клиента Business доступна в группы и/или Скайп? Этот метод реализован для режима TeamsOnly. Поддержка для других режимов, зависит от предстоящие TeamsAppPermissionsPolicy. При этом новой политики будет готова, TeamsUpgradePolicy, имеют зависимость ее, чтобы убедиться, что команды надлежащим образом настроена для нужный режим.

## <a name="teamsinteroppolicy-and-legacy-mode-being-retired"></a>TeamsInteropPolicy и режиме прежних версий не поддерживается 

Как было сказано ранее, TeamsInteropPolicy вместо нее TeamsUpgradePolicy. Все компоненты, которые ранее соблюдаться TeamsInteropPolicy были обновлены принять TeamsUpgradePolicy. 

Ранее появился режим «Устаревший» в TeamsUpgradePolicy для облегчения перехода с TeamsInteropPolicy на TeamsUpgradePolicy. В режиме прежних версий компоненты маршрутизации, которые поняты TeamsUpgradePolicy будет вернуться к TeamsInteropPolicy. Маршрутизация полностью поддерживает TeamsUpgradePolicy и больше не осталось требуется использовать режим прежних версий. *Клиенты, которые используют режим прежних версий следует обновить их конфигурации TeamsUpgradePolicy, чтобы воспользоваться одним из других режимов.* 

Пользователи по-прежнему используется режим прежних версий получают уведомление, поддерживаемые только три конкретные экземпляры TeamsInteropPolicy, перечисленных ниже. В обоих случаях значение CallingDefaultClient совпадает со значением ChatDefaultClient и AllowEndUserClientOverride всегда имеет значение false.

## <a name="federation-considerations"></a>Сведения о выполнении федерации

Федерации из рабочих групп на другого пользователя, с помощью Скайп для бизнеса требуется пользователей группам быть Интернет-версия размещен в Скайп для бизнеса. В конечном счете пользователей, размещенных в Скайп для бизнеса в локальной группы смогут федерацию с другими группами пользователей.

TeamsUpgradePolicy управляет маршрутизации для входящих федеративных чаты и звонки. Федеративные поведение маршрутизации — это так же владельцу сценариев, *за исключением в режиме острова*же.  Когда получатели находятся в режиме о-ва: 
- Чаты и вызовам из групп land SfB Если получатель находится в *федеративных клиента*.
- Чаты и вызовам из Земля группами в группах, если получатель является в *же владельцу*.
- Чаты и вызовам из SfB всегда land в SfB.


## <a name="completing-the-transition-to-mode-management"></a>Завершение работы перехода к разделу Управление режим

В этом году, корпорация Майкрософт планирует вводить новый тип политики, TeamsAppPermissionsPolicy, чтобы контролировать, какие части клиента команды включены (например, обмен мгновенными Сообщениями, собрания, беседы, каналы). По мере поступления новой политики для включения и отключения рабочих нагрузок в группах TeamsUpgradePolicy будут обновлены таким образом, когда администратор пытается предоставить пользователю экземпляр TeamsUpgradePolicy, сначала проверяет, убедитесь, что TeamsAppPolicy должным образом настроен нужный режим. В противном случае grant завершится с ошибкой, о том, как другие политики сначала должно быть задано. 

Пока TeamsAppPolicy станет доступным, TeamsUpgradePolicy по сути управляет маршрутизации вызовов и беседы, а также планирования собраний (на которых предоставляется через надстроек Outlook). Так как поведение клиента групп еще не на месте, не все режимы включены в современном портала. С точки зрения маршрутизации режимы SfBOnly, SfBWithTeamsCollab и SfBWithTeamsCollabAndMeetings, совпадают. 

## <a name="action-required-for-organizations-that-were-using-teamsinteroppolicy"></a>Действия, необходимые для организации, которые использовали TeamsInteropPolicy

Все пользователи по-прежнему используется TeamsInteropPolicy: 
1. Убедитесь, что пользователи с TeamsInteropPolicy назначаются только один из этих трех встроенных экземпляры какие CallingDefaultClient = ChatDefaultClient и для каких AllowEndUserClientOverride = false. Эти экземпляры являются:
   </br>
   </br>

   |Identity |AllowEndUserClientOverride |CallingDefaultClient|ChatDefaultClient|
   |---|---|---|---|
   |`DisallowOverrideCallingDefaultChatDefault`|False|По умолчанию|По умолчанию|
   |`DisallowOverrideCallingSfbChatSfb`|False|Sfb|Sfb|
   |`DisallowOverrideCallingTeamsChatTeams`|False|Teams|Teams|
   |||||

    Используйте следующий синтаксис командлета, где $policy — это один из приведенных выше значений удостоверений:

    `Grant-CsTeamsInteropPolicy -PolicyName $policy -Identity $SipAddress`

    **Microsoft запросов, что обновление их политик с 30 июня 2018.** После этого приходится Microsoft удаление других экземпляров TeamsInteropPolicy.</br> 
    ***Организации, которые нечасто обновляют к одной из этих экземплярах со временем будут иметь своим пользователям автоматически обновляется, один из этих экземпляров. Очевидно, что желательно, что клиенты этого, чтобы вы могли выбрать, что лучше всего подходит для пользователей.***

2. Если настройка встроенных глобальную политику, отменить это. Глобальная политика должна иметь следующие значения:
   </br>
   </br>

    |Параметр|Значение|
    |---|---|
    |`AllowEndUserClientOverride`|False|
    |`CallingDefaultClient`|По умолчанию|
    |`ChatDefaultClient`|По умолчанию|
    |||

    Если любое из значений, отличаются от выше, выполните следующие действия, чтобы удалить все настройки, зависящие от клиента.

    `Grant-CsTeamsInteropPolicy -PolicyName $null`




## <a name="action-required-for-organizations-that-are-using-modelegacy"></a>Необходимые действия для организаций, использующих режим = прежних версий

Клиенты, которые используют режим = прежних версий в TeamsUpgradePolicy (экземпляр политики = NoUpgrade или политики экземпляр = NotifyForTeams) необходимо обновить их конфигурации для использования политики с режимом отличный от прежних версий. 

 **Microsoft запросов, что пользователям удалить все использование режима прежних версий с 15 ноября 2018.** Иногда после этого Microsoft будет удаление экземпляров TeamsUpgradePolicy с режимом = прежних версий.</br>

Этому пользователю:</br><ul><li>Можно инициировать чаты и вызовы только от Скайп для бизнеса.<li>Получает все чата/вызов в их Скайп для клиента Business вне зависимости от где инициировал, пока инициатор — это группы пользователя с помощью Скайп для бизнеса, размещенных в локальной. * <li>Можно запланировать только Скайп для собраний бизнеса, но могут присоединиться к Скайп для бизнеса или группами собраний. </br> *Острова использование режима с локальными пользователями не рекомендуется в сочетании с другими пользователями в режиме SfBOnly. |**SfBWithTeamsCollab</br>AndMeetings**|Один пользователь запускает оба Скайп для бизнеса и групп-одновременно.
