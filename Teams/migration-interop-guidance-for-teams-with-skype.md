# <a name="migration-and-interoperability-guidance-for-organizations-using-teams-together-with-skype-for-business"></a>Руководство по миграции и взаимодействия для организаций, с помощью команды Скайп для бизнеса

В апреле 2018 Microsoft уточнен его руководство по миграции групп из Скайп для бизнеса и как эти два клиенты могут одновременно использоваться для данного пользователя. В этот момент мы объявили об изменении запланированных изменений для упрощения управляемости и повышения уровня удовлетворенности конечных пользователей. Затем Microsoft доставке обновление admin взаимодействия с пользователем, схожих с этого плана. Эти изменения отображаются инструкции в этом документе.

Как ранее вышеперечисленные TeamsInteropPolicy будет из эксплуатации (предназначенной для конечных Q3) и консолидированной среды его функциональные возможности в TeamsUpgradePolicy. Взаимодействие и миграция будут управляться в режиме «совместная работа», определяемой на TeamsUpgradePolicy, который теперь доступен. Выбор режима пользователя имеют преимущественную силу маршрутизации входящих вызовов и чатов и в которой клиентские пользователя можно инициировать чаты и звонков или планировать собрания. Во время будет не поддерживается TeamsInteropPolicy, он все еще необходимо задать параллельно с TeamsUpgradePolicy во время phaseout.  

В рамках этой работы недавно были обновлены в «Microsoft групп & Скайп для бизнеса Центр администрирования» (также известной как современные портал) в соответствии с новой модели управления на основании режимы сосуществования. В современных портала Настройка будет TeamsUpgradePolicy теперь автоматически также необходимо задать TeamsInteropPolicy согласованные значения, поэтому TeamsInteropPolicy больше не отображается в интерфейсе пользователя. *Тем не менее с помощью PowerShell "Администраторы" по-прежнему необходимо задать как TeamsUpgradePolicy и TeamsInteropPolicy друг с другом для обеспечения правильной маршрутизации.* После завершения перехода на TeamsUpgradePolicy оно больше не будет необходимо также установить TeamsInteropPolicy.

## <a name="fundamental-concepts"></a>Основные понятия

1.  *Взаимодействия* : 1-1 обмена данными между Lync/Скайп для корпоративных пользователей и групп пользователей.

2.  *Федерация* : обмена данными между пользователями из различных клиентов.

3.  Все команды, у пользователей есть базового Скайп для бизнеса учетной записи, являющейся «размещенных» либо через Интернет или локальной:
    - Пользователи, уже используется Скайп для бизнеса в Интернет использовать их существующей учетной записи online.
    - Уже используется Скайп для бизнеса/Lync локальных пользователей используйте их существующей локальной учетной записи.
    - Пользователи, для которых не удалось обнаружить существующий Скайп для бизнес-партнер будет иметь Скайп для бизнеса в Интернет учетной записи автоматически подготавливаться при создании группы пользователей. Не Скайп для бизнеса лицензии не требуется.

4.  При наличии локального развертывания либо Скайп для бизнеса или Lync, и эти пользователи должны быть группами пользователей, по крайней мере убедитесь, что подключение Azure AD выполняет синхронизацию msRTCSIP-DeploymentLocator атрибут в AAD, поэтому этой команды/Скайп для бизнеса Online правильно обнаруживает в локальную среду. Кроме того для перемещения всех пользователей в режим только для группы (например, обновление пользователя), *необходимо настроить Скайп для бизнеса гибридный режим*.

5.  Взаимодействие между рабочими группами и Скайп для бизнес-пользователи — только возможные *Если Интернет-версия расположен пользователь групп в Скайп для бизнеса*. Скайп для бизнес-пользователь может быть размещен либо локальной (и требуется настройка Скайп для гибридных Business) или в оперативном режиме. Пользователи, которые размещаются в Скайп для бизнеса в локальной можно использовать команды в режиме острова (задать далее в этом документе), но не могут использовать команды для взаимодействия и создать федерацию с других пользователей, использующих Скайп для бизнеса.  

6.  Для обновления пользователя к группам (то есть, предоставить им TeamsUpgradePolicy с режимом = TeamsOnly), пользователь должен быть размещен в Скайп для бизнеса через Интернет. Это необходимо для обеспечения взаимодействия, федерации и полного управления группами пользователей. Чтобы обновить пользователи, размещенные локально, используйте `Move-CsUser` из локальной admin средства для первого move пользователю Скайп для бизнеса в Интернет. Затем предоставить TeamsUpgradePolicy и TeamsInteropPolicy online пользователю или с помощью современных портала можно назначать TeamsOnly режим. Один раз CU8 Скайп для входящего Business Server 2015, клиент может просто воспользоваться нового `-MoveToTeams` переключитесь в `Move-CsUser` который объединяет эти шаги 2 в 1.

7.  Основные политики для управления обновлением и взаимодействия, TeamsUpgradePolicy и TeamsInteropPolicy.  Тем не менее TeamsInteropPolicy находится в процессе не поддерживается и все функциональные возможности будут заменены TeamsUpgradePolicy. До завершения перехода пользователей необходимо задать TeamsUpgradePolicy и TeamsInteropPolicy постоянно (как описано [более поздней версии](#important) в этом документе) для обеспечения нормальной или использования новых современных портала, который выполняется автоматически.

8.  Использование функций группам телефонной системой, пользователи должны быть в режиме TeamsOnly (то есть, расположенным в Скайп для бизнеса в Интернет и обновлены для группы) и должен быть настроен либо для Microsoft телефонной системы прямой маршрутизации (который позволяет использовать телефонной системы с помощью собственных SIP магистральных линий связи и SBC) или Office 365 вызов плана. Прямое маршрутизации является [общедоступным](https://techcommunity.microsoft.com/t5/Microsoft-Teams-Blog/Direct-Routing-is-now-Generally-Available/ba-p/210359#M1277) по состоянию на 28 июня 2018.  

9.  Планирование собраний группы с аудиоконференции (по телефону с помощью ТСОП или модем) в настоящее время доступен только для пользователей, которые размещаются в Скайп для бизнеса в Интернет. Для группы пользователей с Скайп локальной учетной записи бизнес поддерживается в TAP.


## <a name="coexistence-modes"></a>Режимы сосуществования

Чтобы упростить управление и повысить удовлетворенность конечного пользователя, взаимодействия и перехода теперь управляются зависимости «сосуществования режима» с помощью TeamsUpgradePolicy. Определяет режим пользователя:

- *Маршрутизация входящих* : В какой клиента (группы или Скайп для бизнеса) входящего обсуждений и вызывает Земля? Эта функция заменяет, что ранее было обработано с TeamsInteropPolicy. TeamsInteropPolicy будет не поддерживается, после завершения перехода. ***Во время перехода, TeamsUpgradePolicy и TeamsInteropPolicy должна соответствовать согласованных так, как описано в следующем разделе***.
- *Планирование на собрания* : какие служба используется для планирования собраний новые и проверка того, что соответствующие надстройки присутствует в Outlook? Поддержки надстроек Outlook должен быть развернут в течение нескольких недель. Обратите внимание на то, что TeamsUpgradePolicy не влияет на присоединения к собранию. Пользователи могут всегда *присоединиться к* любого собрания, будет ли это Скайп для собраний или группами собрания.
- *Взаимодействия с пользователем* : какие функциональные возможности для клиента Business доступна в группы и/или Скайп? Этот метод реализован для режима TeamsOnly. Поддержка для других режимов, зависит от предстоящие TeamsAppPolicy. При этом новой политики будет готова, TeamsUpgradePolicy, имеют зависимость ее, чтобы убедиться, что команды надлежащим образом настроена для нужный режим.

Ниже перечислены плановых режимах. SfBWithTeamsCollab и SfBWithTeamsCollabAndMeetings позволит смешанного использования обоих клиентов, но не перекрывающиеся функции. О-ва режим позволяет об использовании обоих клиентов, но с перекрывающимися функциональные возможности. Например в режиме острова, пользователь может начать разговор в любом Скайп для бизнеса или группами, но в SfBWithTeamsCollab, они могут только беседа в Скайп для бизнеса. Обратите внимание на то, что не все режимы еще не доступны.

## <a name="teamsinteroppolicy-to-be-retired"></a>TeamsInteropPolicy из эксплуатации 

Как было сказано ранее, TeamsInteropPolicy будет заменена TeamsUpgradePolicy. До завершения этого перехода, поддерживаются только три конкретные экземпляры TeamsInteropPolicy, перечисленных ниже. В обоих случаях значение CallingDefaultClient совпадает со значением ChatDefaultClient и AllowEndUserClientOverride всегда имеет значение false. 
</br>
</br>
**Поддерживаемые экземпляры TeamsInteropPolicy до пенсионного обеспечения**
|Identity |AllowEndUserClientOverride|CallingDefaultClient|ChatDefaultClient|
|---|---|---|---|
|`DisallowOverrideCallingDefaultChatDefault`|False|"Default" (По умолчанию)|"Default" (По умолчанию)|
|`DisallowOverrideCallingSfbChatSfb`|False|Sfb|Sfb|
|`DisallowOverrideCallingTeamsChatTeams`|False|Teams|Teams|
|||||

## <a name="federation-considerations"></a>Сведения о выполнении федерации

Федерации из рабочих групп на другого пользователя, с помощью Скайп для бизнеса требуется пользователей группам быть Интернет-версия размещен в Скайп для бизнеса. В конечном счете пользователей, размещенных в Скайп для бизнеса в локальной группы смогут федерацию с другими группами пользователей.

После включения поддержки федерации TeamsUpgradePolicy (а также TeamsInteropPolicy во время перехода) управляет маршрутизации для входящих федеративных чаты и звонки. Чтобы облегчить других организаций, инициализация федеративного обмена данными с пользователями в вашей организации, рекомендуется выбрать режим, в частности маршрутизирует либо для Скайп для бизнеса или группами, а не о-ва.
</br>
|Для маршрутизации вызовов и беседы, чтобы...|Предоставление экземпляр TeamsUpgradePolicy</br> с помощью одного из следующих режимов|И предоставить ей права этот экземпляр TeamsInteropPolicy|
|---|---|---|
|Skype для бизнеса|SfBOnly, SfBWithTeamsCollab, </br>SfBWithTeamsCollabAndMeetings|`DisallowOverrideCallingSfbChatSfb`|

## <a name="completing-the-transition-to-mode-management"></a>Завершение работы перехода к разделу Управление режим

В этом году, корпорация Майкрософт планирует вводить новый тип политики, TeamsAppPolicy, чтобы контролировать, какие части клиента команды включены (например, обмен мгновенными Сообщениями, собрания, беседы, каналы). По мере поступления новой политики для включения и отключения рабочих нагрузок в группах TeamsUpgradePolicy будут обновлены таким образом, когда администратор пытается предоставить пользователю экземпляр TeamsUpgradePolicy, сначала проверяет, убедитесь, что TeamsAppPolicy должным образом настроен нужный режим. В противном случае grant завершится с ошибкой, о том, как другие политики сначала должно быть задано. 

Пока TeamsAppPolicy станет доступным, TeamsUpgradePolicy по сути управляет маршрутизации вызовов и беседы, а также планирования собраний (на которых предоставляется через надстроек Outlook). Так как поведение клиента групп еще не на месте, не все режимы включены в современном портала. С точки зрения маршрутизации режимы SfBOnly, SfBWithTeamsCollab и SfBWithTeamsCollabAndMeetings, совпадают.

## <a name="summary-of-planned-functionality-changes-to-simplify-coexistence-and-migration"></a>Сводка изменений запланированные функциональные возможности для упрощения совместная работа и миграция

На основе строгого отзывов и других первыми TAP, Microsoft объявили об изменении запланированных изменений в апреле для упрощения управляемости и повышения уровня удовлетворенности конечных пользователей, когда организаций переход с Скайп для бизнеса на группы. Эти изменения обеспечит упрощенный и более предсказуемый конечным пользователем, с помощью модели упрощенный политики для управления обновлением и взаимодействия. Изменений, описанных ниже будет выполнен откат выходной параметр постепенного через следующие несколько месяцев. На высоком уровне существует четыре плановых функциональные изменения, описанные ниже.
</br>
</br>
**Изменение #1: Нет больше перекрывающиеся модальностей при предпочитаемого клиентского установлено**
|||
|---|---|
|**РЕШЕНИЕ**|*Если предпочитаемый клиента задано значение «Команды» или «Sfb»*, заданным модальности (мгновенные сообщения, собрания, планирования, вызов) будет только доступно и поддерживается в клиенте (группы или Скайп для бизнеса), для этого пользователя. (*Предпочтительное клиента* ссылается на параметры DefaultChatClient и DefaultCallingClient в TeamsInteropPolicy). Только в том случае, когда предпочитаемого клиентского задано значение «По умолчанию» (в будущем, называется «О-ва режима») будет заданного модальность быть доступно и поддерживается в обоих этих клиентах. |
|**ЛОГИЧЕСКОЕ ОБОСНОВАНИЕ**|Согласованные свои отзывы и предложения, которые мы получили —, что пользователи будут сбиты с толку для той или иной причине при попытке использования обоих этих клиентах с перекрывающимися модальности. Трудно прогнозирования и понять, почему сообщения и вызовы land в один клиент и другой и сведения о присутствии может быть неточности или ошибки.|
|**СОСТОЯНИЕ**|Это действие поддерживается для режима = TeamsOnly. Поддержка другие режимы будут доступны в следующих нескольких месяцев. |
|||

**Изменение #2: Объединяют в себе предпочитаемого клиентского для системы обмена сообщениями и телефонной**
|||
|---|---|
|**РЕШЕНИЕ**|Корпорация Майкрософт будет объединяют в себе управление предпочтительный клиент для обмена мгновенными Сообщениями и телефонной. В частности DefaultChatClient и DefaultCallingClient должны иметь такое же значение — либо оба по умолчанию (о-ва), группы или оба Скайп для бизнеса.  |
|**ЛОГИЧЕСКОЕ ОБОСНОВАНИЕ**|Если они не выровнен, сведения о присутствии в некоторых случаях будет ввести в заблуждение и некоторая путаница. Кроме того начале звонка из существующего разговора может привести к сбоям так как звонок может оказаться в различные клиентские чем существующие чата. |
|**СОСТОЯНИЕ**|Это изменение используется на уровне политики, с помощью TeamsUpgradePolicy, а также в admin UX (современных портал). Тем не менее переход еще не завершено, поэтому клиентам необходимо также установить TeamsInteropPolicy. Клиенты только необходимо задать одно из трех поддерживаемых экземпляры TeamsInteropPolicy, которые являются: DisallowOverrideCallingTeamsChatTeams, DisallowOverrideCallingDefaultChatDefault, DisallowOverrideCallingSfbChatSfb|
|||

**Изменение #3: Не больше конечного пользователя Выбор предпочитаемого клиентского**
|||
|---|---|
|**РЕШЕНИЕ**|Конечные пользователи больше не будет предоставляться Выбор предпочитаемого клиентского. (AllowEndUserClientOverride в TeamsInteropPolicy должно быть установлено значение false). Предпочитаемый клиента будут основываться на режим, в котором задано администратором.|
|**ЛОГИЧЕСКОЕ ОБОСНОВАНИЕ**|Анализ данных об использовании показывает, что пользователи практически не значение это. Устранение этого варианта упрощает admin/helpdesk сценариев и уменьшает сложность проектирования. И, наконец Если конечный пользователь из которых можно выбрать разные предпочитаемого клиентского что администратор настроил для маршрутизации голосовых вызовов в организации с корпоративной голосовой связи, это приведет к заблуждение работы конечных пользователей, так как она приведет VOIP и Главная в различные вызовы в ТСОП Клиенты.|
|**СОСТОЯНИЕ**|Это изменение используется на уровне политики, с помощью TeamsUpgradePolicy, а также в admin UX (современных портал). Тем не менее переход еще не завершено, поэтому клиенты, которые используют командлеты необходимо также установить TeamsInteropPolicy к одному из трех поддерживаемых экземпляров TeamsInteropPolicy, каждый из которых запретить переопределение.|
|||

**Изменение #4: Управление на основании клиента «режима»**
|||
|---|---|
|**РЕШЕНИЕ**|Microsoft краткие сведения о понятии «режима» для управления поведением клиента организаций, с помощью Скайп для бизнеса начала принять группами. Администраторы могут задать режим на уровне пользователя, а также режим пользователя будет какие функциональные возможности доступны в которой клиентские и где land звонки и беседы. «Администраторы» будет управлять это с помощью оптимизированная TeamsUpgradePolicy, который был обновлен для поддержки режима. В конечном счете TeamsInteropPolicy будет устаревшим и удалена, после обновления всех компонентов для чтения из TeamsUpgradePolicy.|
|**ЛОГИЧЕСКОЕ ОБОСНОВАНИЕ**|Чтобы упростить управление и повысить удовлетворенность конечного пользователя, взаимодействия и перехода будут управляться в режиме «совместная работа» с помощью TeamsUpgradePolicy. Выбор режима пользователя имеют преимущественную силу маршрутизацию входящих звонков и беседы, и в которой клиентские пользователя можно инициировать чаты и звонков или планировать собрания. Консолидация TeamsUpgradePolicy и TeamsInteropPolicy также не позволяет неверные настройки, которые может привести, если эти две политики постоянно не установлены. |
|**СОСТОЯНИЕ**|Программы TAP отобразиться три режима UI администрирования. Как поддержка изменения #1 территории, дополнительные режимы будут доступны. Режим SfBOnly не запрещает в настоящее время пользователи с помощью команды, но он будет в будущем. |
|||
