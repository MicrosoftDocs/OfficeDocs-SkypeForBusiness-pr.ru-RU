# <a name="plan-direct-routing"></a>Планирование прямой маршрутизации

> [!Tip]
> Просмотрите следующий сеанс, чтобы узнать о преимуществах прямой маршрутизации, планировании и [развертывании. Прямая маршрутизация в Microsoft Teams](https://aka.ms/teams-direct-routing)

Прямая маршрутизация позволяет подключить поддерживаемый, предоставленный клиентом пограничный контроллер сеансов (SBC) к телефонной системе. С помощью этой возможности можно настроить подключение локальной телефонной сети общего пользования (ТСОП) к клиенту Microsoft Teams, как показано на следующей схеме: 

![Схема, показывающая конфигурацию локального подключения по ТСОП.](media/PlanDirectRouting1-PSTNwithTeams.png "Настройка локального подключения к ТСОП с клиентом Microsoft Teams")

  > [!NOTE]
  > Skype для бизнеса Online также позволяет связать предоставленный клиентом SBC, но для этого требуется локальное развертывание Skype для бизнеса Server или специальный выпуск Skype для бизнеса, называемый Cloud Connector, между SBC и Microsoft Cloud. Этот сценарий называется гибридной голосовой связью. В отличие от этого, прямая маршрутизация обеспечивает прямое подключение между поддерживаемым SBC и Microsoft Cloud.

> [!Important]
> Cloud Connector Edition будет прекращен 31 июля 2021 г. вместе с Skype для бизнеса Online. После обновления организации до Teams узнайте, как подключить локальную телефонную сеть к Teams с помощью [прямой маршрутизации](direct-routing-landing-page.md). 

С помощью прямой маршрутизации вы можете подключить SBC практически к любой магистрали телефонии или подключиться к оборудованию стороннего ТСОП. Прямая маршрутизация позволяет: 

- Используйте практически любую магистраль ТСОП с телефонной системой. 

- Настройте взаимодействие между оборудованием телефонии, принадлежащим клиенту, например сторонним обменом частной ветвью (УАТС), аналоговыми устройствами и Teams.

Корпорация Майкрософт также предлагает решения голосовой связи в облаке, такие как план звонков. Однако гибридное голосовое решение может оказаться оптимальным для вашей организации, если: 

- План звонков Майкрософт недоступен в вашей стране. 

- Вашей организации требуется подключение к сторонним аналоговым устройствам, центрам обработки вызовов и т. д. 

- У вашей организации есть контракт с оператором ТСОП.

Прямая маршрутизация также поддерживает пользователей, имеющих дополнительную лицензию на план звонков Майкрософт. Дополнительные сведения см. в разделе [Телефонная система и тарифные планы](calling-plan-landing-page.md). 

При использовании прямой маршрутизации, когда пользователи участвуют в запланированной конференции, телефонный номер предоставляется службой аудиоконференций Майкрософт, которая требует надлежащего лицензирования.  При звонке служба аудиоконференций Майкрософт выполняет вызов с помощью возможностей звонков по сети, что требует надлежащего лицензирования. (Обратите внимание, что если у пользователя нет лицензии на аудиоконференции Майкрософт, вызов направляется через прямую маршрутизацию.) Дополнительные сведения см. в разделе [Онлайн-собрания с Teams](https://www.microsoft.com/en-us/microsoft-teams/online-meetings).

## <a name="licensing-and-other-requirements"></a>Лицензирование и другие требования 

Пользователям прямой маршрутизации в Microsoft 365 должны быть назначены следующие лицензии: 

- Телефонная система Майкрософт
- Microsoft Teams + Skype для бизнеса план 2, если он включен в лицензирование
- Аудиоконференции (см. заметки и абзац ниже, чтобы узнать, когда требуется эта лицензия.)

> [!NOTE]
> Skype для бизнеса план не следует удалять из лицензионного соглашения, в котором он включен. 
> 
> [!IMPORTANT]
> Пользователи GCC High и DoD должны отключить любое лицензирование аудиоконференций, включенных в G5, и дождаться включения аудиоконференций, пока прямая маршрутизация не будет полностью настроена. Перед включением лицензий на аудиоконференции пользователи должны иметь настроенные номера телефонных номеров и рабочую панель набора номера. Дополнительные сведения см. в разделе [Аудиоконференции с прямой маршрутизацией для GCC High и DoD](./audio-conferencing-with-direct-routing-for-gcch-and-dod.md) .


> [!IMPORTANT]
>  Если вы хотите добавить внешних участников в запланированные собрания, набрав для них номер номера или указав номер телефона, требуется лицензия на аудиоконференции.
> Для GCC High и DoD не назначайте лицензию на аудиоконференции для пользователей G5. Для пользователей G3 не назначайте лицензию на аудиоконференции, пока не будет полностью настроена прямая маршрутизация и у пользователя есть рабочая панель набора номера.


### <a name="ad-hoc-call-escalation-and-audio-conferencing-license"></a>Лицензия на эскалацию нерегламентированных звонков и аудиоконференций

Пользователь Teams может запустить единый вызов Teams-to-PSTN или Teams-to-Teams и добавить в него участника ТСОП. Этот сценарий называется нерегламентированной конференцией. Путь, который принимает вызов, зависит от того, назначена ли пользователю, который передает вызов, лицензия на аудиоконференции Майкрософт.

- **Если пользователю Teams, который передает вызов, назначена лицензия на аудиоконференции Майкрософт**, эскалация происходит через службу аудиоконференций Майкрософт. Удаленный участник ТСОП, приглашенный на существующий вызов, получает уведомление о входящем вызове и видит номер моста Майкрософт, назначенный пользователю Teams, который инициировал эскалацию.

- **Если пользователю Teams, который передает вызов, не назначена лицензия на аудиоконференции Майкрософт**, эскалация происходит через пограничный контроллер сеансов, подключенный к интерфейсу прямой маршрутизации. Удаленный участник ТСОП, приглашенный на вызов, получает уведомление о входящем вызове и видит номер пользователя Teams, который инициировал эскалацию. Конкретный SBC, используемый для эскалации, определяется политикой маршрутизации пользователя. 

Необходимо убедиться в следующем:
 
- CsOnlineVoiceRoutingPolicy назначается пользователю.

- Разрешение частных звонков включено на уровне клиента для Microsoft Teams.

Прямая маршрутизация также поддерживает пользователей, имеющих лицензию на план звонков Майкрософт. Телефонная система с тарифным планом может маршрутизировать некоторые вызовы с помощью интерфейса прямой маршрутизации. Однако номера телефонов пользователей должны быть приобретены через Интернет или перенесены в корпорацию Майкрософт.  

Сочетание планов звонков и прямой маршрутизации для одного и того же пользователя является необязательным, но может быть полезным. Например, если пользователю назначен план звонков Майкрософт, но он хочет маршрутизировать некоторые вызовы с помощью SBC. Одним из наиболее распространенных сценариев являются вызовы сторонних УАТС.  При использовании УАТС сторонних производителей все звонки, кроме звонков на телефоны, подключенные к этой УАТС, перенаправляются с помощью плана звонков Майкрософт, но звонки на телефоны, подключенные к сторонним УАТС, отправляются в SBC и, следовательно, остаются в корпоративной сети, а не в ТСОП.

Дополнительные сведения о лицензировании телефонной системы см. в разделах [Получение максимальной параметров Office](https://products.office.com/compare-all-microsoft-office-products?tab=2) и [планов](/office365/servicedescriptions/office-365-platform-service-description/office-365-plan-options) и [лицензирование надстроек Microsoft Teams](./teams-add-on-licensing/microsoft-teams-add-on-licensing.md).

## <a name="sbc-domain-names"></a>Доменные имена SBC

Доменное имя SBC должно быть от одного из имен, зарегистрированных в домене клиента. Клиент .onmicrosoft.com нельзя использовать \*для полного доменного имени SBC.

В следующей таблице приведены примеры DNS-имен, зарегистрированных для клиента, можно ли использовать имя в качестве полного доменного имени для SBC, а также примеры допустимых полных доменных имен:

|DNS-имя|Может использоваться для полного доменного имени SBC|Примеры полных доменных имен|
|:--- |:--- |:--- |
contoso.com|Да|**Допустимые имена:**<br/>sbc1.contoso.com<br/>ssbcs15.contoso.com<br/>europe.contoso.com|
|contoso.onmicrosoft.com|Нет|Использование доменов *.onmicrosoft.com не поддерживается для имен SBC.

Предположим, вы хотите использовать новое доменное имя. Например, клиент имеет contoso.com в качестве доменного имени, зарегистрированного в клиенте, и вы хотите использовать sbc1.sip.contoso.com. Прежде чем связать SBC с именем sbc1.sip.contoso.com, необходимо зарегистрировать доменное имя sip.contoso.com в разделе Домены в клиенте. При попытке связать SBC с sbc1.sip.contoso.com перед регистрацией доменного имени вы получите следующую ошибку: "Не удается использовать домен "sbc1.sip.contoso.com", так как он не настроен для этого клиента".

После добавления доменного имени необходимо также создать пользователя с именем участника-пользователя user@sip.contoso.com и назначить лицензию Teams. Полная подготовка доменного имени может занять до 24 часов после его добавления в домены клиента, создания пользователя с новым именем и назначения ему лицензии.

Возможно, у компании может быть несколько sip-адресных пространств в одном клиенте. Например, компания может contoso.com в качестве адресного пространства SIP и fabrikam.com в качестве второго адресного пространства SIP. У некоторых пользователей есть адресные user@contoso.com, а у некоторых — адресные user@fabrikam.com. 

SBC требует только одно полное доменное имя и может обслуживать пользователей из любого адресного пространства в парном клиенте. Например, SBC с именем sbc1.contoso.com может получать и отправлять трафик ТСОП для пользователей с адресами user@contoso.com и user@fabrikam.com при условии, что эти адресные пространства SIP зарегистрированы в одном клиенте.  

 > [!NOTE]
 > Полное доменное имя SBC в Службы коммуникации Azure прямой маршрутизации должно отличаться от полного доменного имени SBC в прямой маршрутизации Teams.
  
## <a name="public-trusted-certificate-for-the-sbc"></a>Общедоступный доверенный сертификат для SBC

Корпорация Майкрософт рекомендует запросить сертификат для SBC, создав запрос на подпись сертификации (CSR). Конкретные инструкции по созданию CSR для SBC см. в инструкциях по подключению или документации, предоставляемых поставщиками SBC.

> [!NOTE]
> Большинству центров сертификации требуется, чтобы размер закрытого ключа был не менее 2048. Помните об этом при создании CSR.

Сертификат должен иметь полное доменное имя SBC в качестве поля общего имени (CN) или альтернативного имени субъекта (SAN).

Кроме того, прямая маршрутизация поддерживает подстановочные знаки в CN и (или) SAN, а подстановочный знак должен соответствовать стандарту [RFC HTTP over TLS](https://tools.ietf.org/html/rfc2818#section-3.1).

Примером может быть использование \*contoso.com, которое будет соответствовать sbc.contoso.com полного доменного имени SBC, но не соответствует sbc.test.contoso.com.

Интерфейс SIP прямой маршрутизации будет доверять только сертификатам, подписанным центрами сертификации (ЦС), которые являются частью программы доверенных корневых сертификатов Майкрософт. Убедитесь, что сертификат SBC подписан ЦС, который входит в программу, и что расширение расширенного использования ключа (EKU) сертификата включает проверку подлинности сервера.
Дополнительные сведения: [Требования к программе — доверенная корневая программа Майкрософт](/security/trusted-root/program-requirements)
  
[Список сертификатов включенного ЦС](https://ccadb-public.secure.force.com/microsoft/IncludedCACertificateReportForMSFT)
  
 Для прямой маршрутизации в Office 365 средах GCCH и DoD сертификат должен быть создан одним из следующих корневых центров сертификации:

- Глобальный корневой ЦС DigiCert
- DigiCert High Assurance EV Корневой ЦС

> [!NOTE]
> Если для подключения Teams в SBC включена поддержка mutual TLS (MTLS), необходимо установить сертификаты Baltimore CyberTrust Root и DigiCert Global Root G2 в доверенном корневом хранилище SBC контекста TEAMS TLS. (Это связано с тем, что сертификаты службы Майкрософт используют один из этих двух корневых сертификатов.) Чтобы скачать эти корневые сертификаты, см[. Office 365 цепочки шифрования](/microsoft-365/compliance/encryption-office-365-certificate-chains). Дополнительные сведения см. в разделе [Изменения сертификата TLS Office](/microsoft-365/compliance/encryption-office-365-tls-certificates-changes).

## <a name="sip-signaling-fqdns"></a>Сигнал sip: полные доменные имена

Прямая маршрутизация предлагается в следующих средах:

- Microsoft 365 или Office 365
- Office 365 GCC
- Office 365 GCC High
- Office 365 DoD

Узнайте больше о [средах Office 365 и государственных организаций США](/office365/servicedescriptions/office-365-platform-service-description/office-365-us-government/office-365-us-government), таких как GCC, GCC High и DoD.

### <a name="microsoft-365-office-365-and-office-365-gcc-environments"></a>Microsoft 365, Office 365 и Office 365 среды GCC

Точками подключения для прямой маршрутизации являются следующие три полных доменных имени:

- **сначала** необходимо попробовать sip.pstnhub.microsoft.com — глобальное полное доменное имя. Когда SBC отправляет запрос на разрешение этого имени, DNS-серверы Microsoft Azure возвращают IP-адрес, указывающий на основной центр обработки данных Azure, назначенный SBC. Назначение основано на метриках производительности центров обработки данных и географической близости к SBC. Возвращенный IP-адрес соответствует основному полному доменному имени.

- **sip2.pstnhub.microsoft.com** — дополнительное полное доменное имя — географически сопоставляется со вторым приоритетным регионом.

- **sip3.pstnhub.microsoft.com** — третичное полное доменное имя — географически сопоставляется с третьим приоритетным регионом.

Размещение этих трех полных доменных имен в порядке требуется для:

- Обеспечение оптимального взаимодействия (менее загруженное и ближайшее к центру обработки данных SBC, назначенному путем запроса первого полного доменного имени).

- Обеспечение отработки отказа при установке подключения из SBC к центру обработки данных, в котором возникает временная проблема. Дополнительные сведения см. в разделе [Механизм отработки отказа](#failover-mechanism-for-sip-signaling) ниже.  

Полные доменные имена (sip.pstnhub.microsoft.com, sip2.pstnhub.microsoft.com и sip3.pstnhub.microsoft.com) будут разрешаться в IP-адреса из следующих подсетей:

- 52.112.0.0/14
- 52.120.0.0/14
  
Необходимо открыть порты для всех этих диапазонов IP-адресов в брандмауэре, чтобы разрешить входящий и исходящий трафик на адреса и из них для сигнализации.

### <a name="office-gcc-dod-environment"></a>Среда Office GCC DoD

Точка подключения для прямой маршрутизации — это следующее полное доменное имя:

**sip.pstnhub.dod.teams.microsoft.us** — глобальное полное доменное имя. Так как среда Office 365 DoD существует только в центрах обработки данных США, вторичные и третичные полные доменные имена отсутствуют.

Полное доменное имя sip.pstnhub.dod.teams.microsoft.us будет разрешено в IP-адрес из следующей подсети:

- 52.127.64.0/21

Необходимо открыть порты для всех этих IP-адресов в брандмауэре, чтобы разрешить входящий и исходящий трафик на адреса и из них для сигнализации.

Точка подключения для прямой маршрутизации — это следующее полное доменное имя:

### <a name="failover-mechanism-for-sip-signaling"></a>Механизм отработки отказа для передачи сигналов SIP

SBC выполняет ЗАПРОС DNS для разрешения sip.pstnhub.microsoft.com. В зависимости от расположения SBC и метрик производительности центра обработки данных выбирается основной центр обработки данных. Если в основном центре обработки данных возникла проблема, SBC попытается выполнить sip2.pstnhub.microsoft.com, которая разрешается во второй назначенный центр обработки данных. В редких случаях, когда центры обработки данных в двух регионах недоступны, SBC повторяет последнее полное доменное имя (sip3.pstnhub.microsoft.com), которое предоставляет третичный IP-адрес центра обработки данных.

## <a name="media-traffic-media-processors-geography"></a>Трафик мультимедиа: география обработчиков мультимедиа

Трафик мультимедиа проходит через компоненты, называемые обработчиками мультимедиа. Обработчики мультимедиа размещаются в том же центре обработки данных, что и прокси-серверы SIP:

- NOAM (центрально-южная часть США, два центра обработки данных в западной части США и восточной части США)
- Европа (южная часть Соединенного Королевства, Центральная Франция, Амстердам и Дублин)
- Азия (сингапурский центр обработки данных)
- Япония (центры обработки данных в восточной и западной странах)
- Австралия (центры обработки данных в восточной и юго-восточной части Австралии)
- LATAM (Южная Бразилия)

## <a name="media-traffic-codecs"></a>Трафик мультимедиа: кодеки

### <a name="leg-between-sbc-and-cloud-media-processor-or-microsoft-teams-client"></a>Этап между SBC и облачным обработчиком мультимедиа или клиентом Microsoft Teams

Применяется как к случаям обхода мультимедиа, так и к случаям без обхода.

Интерфейс прямой маршрутизации между пограничным контроллером сеансов и обработчиком облачных носителей (без обхода посредника) или между клиентом Teams и SBC (если обход мультимедиа включен) может использовать следующие кодеки:

- Обход без посредников (SBC to Cloud Media Processor): SILK, G.711, G.722, G.729
- Обход мультимедиа (клиент SBC to Teams): SILK, G.711, G.722, G.729

Вы можете принудительно использовать конкретный кодек на пограничном контроллере сеансов, исключив нежелательные кодеки из предложения.

### <a name="leg-between-microsoft-teams-client-and-cloud-media-processor"></a>Этап между клиентом Microsoft Teams и обработчиком облачных мультимедиа

Применяется только к варианту обхода, отличному от носителя. При обходе мультимедиа мультимедиа мультимедиа передаются непосредственно между клиентом Teams и SBC.

На этапе между обработчиком облачных мультимедиа и клиентом Microsoft Teams используется silk или G.722. Выбор кодека на этом этапе основан на алгоритмах Майкрософт, которые учитывают несколько параметров.

  > [!NOTE]
  > Изменение пункта назначения мультимедиа не поддерживается. Во время вызова прямой маршрутизации, если SBC отправляет новый IP-адрес мультимедиа в службу прямой маршрутизации Teams, хотя это и согласовано в сигнализации SIP, мультимедиа никогда не отправляется на новый IP-адрес из службы прямой маршрутизации Teams.

## <a name="supported-session-border-controllers-sbcs"></a>Поддерживаемые пограничные контроллеры сеансов (SBC)

Корпорация Майкрософт поддерживает только сертифицированные SBC для связывания с прямой маршрутизацией. Так как Корпоративная голосовая связь критически важны для бизнеса, корпорация Майкрософт выполняет интенсивные тесты с выбранными SBC и работает с поставщиками SBC, чтобы обеспечить совместимость этих двух систем.

Устройства, которые были проверены, перечислены как сертифицированные для прямой маршрутизации Teams. Сертифицированные устройства гарантированно будут работать во всех сценариях.

Дополнительные сведения о поддерживаемых контроллерах SBC см. в разделе [Контроллеры границ сеансов, сертифицированные для прямой маршрутизации](direct-routing-border-controllers.md).

## <a name="support-boundaries"></a>Границы поддержки

Корпорация Майкрософт поддерживает телефонную систему с прямой маршрутизацией только при использовании с сертифицированными устройствами. При возникновении проблем сначала обратитесь в службу поддержки клиентов вашего поставщика SBC. При необходимости поставщик SBC перенаправит проблему в Майкрософт по внутренним каналам. Корпорация Майкрософт оставляет за собой право отклонять запросы, направленные в службу поддержки, при подключении к телефонной системе несертифицированных устройств с прямой маршрутизацией. Если корпорация Майкрософт определяет, что проблема прямой маршрутизации клиента связана с устройством SBC поставщика, клиенту необходимо будет снова обратиться за поддержкой к поставщику SBC.
