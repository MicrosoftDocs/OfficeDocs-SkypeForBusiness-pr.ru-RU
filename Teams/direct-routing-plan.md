# <a name="plan-direct-routing"></a>Планирование прямой маршрутизации

> [!Tip]
> В следующем сеансе вы узнаете о преимуществах прямой маршрутинга, планировании и развертывании: Прямая маршрутия [в Microsoft Teams](https://aka.ms/teams-direct-routing)

Прямая маршрутия Microsoft Phone System позволяет подключить поддерживаемый клиентский пограничный контроллер сеанса (SBC) к телефонной системе Майкрософт.  Например, с помощью этой функции можно настроить подключение локальной телефонной сети общего перейти на телефонную сеть (STN) к клиенту Microsoft Teams, как показано на схеме ниже. 

![Схема, показывающая конфигурацию локального подключения к ОКП](media/PlanDirectRouting1-PSTNwithTeams.png "Настройка подключения к локальной сети STN с клиентом Microsoft Teams")

  > [!NOTE]
  > Skype для бизнеса Online также позволяет связать SBC, предоставленный клиентом, но для этого требуется локальное развертывание Skype для бизнеса Server или специальный выпуск Skype для бизнеса, который называется Cloud Connector, между SBC и Microsoft Cloud. Этот сценарий называется гибридной голосовой связи. С другой стороны, прямая маршрутия обеспечивает прямое подключение между поддерживаемой SBC и Microsoft Cloud.

> [!Important]
> Выпуск Cloud Connector Edition будет отсюду 31 июля 2021 г. вместе со Skype для бизнеса Online. После обновления организации до Teams узнайте, как подключить к Teams свою локальное телефонное подключение с помощью прямой [маршрутизации.](direct-routing-landing-page.md) 

С помощью прямой маршрутировки вы можете подключить свой SBC практически к любой телефонной линии связи или к подключению к стороннему оборудованию ОКП. Прямая маршрутная маршрутия позволяет: 

- С помощью телефонной системы Майкрософт можно использовать практически любую телефонную систему ННР. 
- Настройка взаимодействия между телефонным оборудованием, которое принадлежит клиенту, например сторонним частным филиалом (УАП), аналоговыми устройствами и телефонной системой Майкрософт.

Корпорация Майкрософт также предлагает все облачные решения для голосовых вызовов, такие как план звонков. Однако гибридное голосовое решение может быть наилучшим для вашей организации, если: 

- План звонков Майкрософт не доступен в вашей стране. 
- Вашей организации требуется подключение к сторонним аналоговым устройствам, центрам вызовов и так далее. 
- У вашей организации есть контракт с оператором ННР.

Прямая маршрутия также поддерживает пользователей, у которых есть дополнительная лицензия для плана звонков Майкрософт. Дополнительные сведения см. в [телефонах и планах звонков.](calling-plan-landing-page.md) 

При прямой маршрутизации при участии пользователей в запланированной конференции номер телефонного дозвона предоставляется службой аудиоконференции Майкрософт, для которой требуется соответствующее лицензирование.  При звонке из службы аудиоконференций Майкрософт звонок передается с использованием функций онлайн-звонков, для чего требуется соответствующее лицензирование. (Обратите внимание, что если у пользователя нет лицензии на аудиоконференцию Майкрософт, звонок направляется через прямую маршрутизацию.) Дополнительные сведения см. в [веб-собраниях с помощью Teams.](https://products.office.com/microsoft-teams/online-meeting-solutions)

## <a name="licensing-and-other-requirements"></a>Лицензирование и другие требования 

Пользователи direct Routing должны иметь следующие лицензии в Microsoft 365 или Office 365: 

- Телефонная система Майкрософт. 
- Microsoft Teams + Skype для бизнеса (план 2), если они включены в лицензирование.
- Аудиоконференция Майкрософт (ознакомьтесь с заметками и абзацем ниже, чтобы получить конкретные примеры необходимости лицензии).

> [!NOTE]
> План Skype для бизнеса не следует удалять из любого лицензионного соглашения, в которое оно входит. 
> 
> [!IMPORTANT]
> Пользователи GCC High и DoD должны отключить все лицензии аудиоконференций, включенные в G5, и дождаться, когда будут включены все аудиоконференции, пока не будет полностью настроена прямая маршрутия. Перед включением лицензий на аудиоконференцию пользователям необходимо настроить номера телефонов для телефонного звонка и рабочую панель набора номера. Дополнительные сведения см. в записях аудиоконференций с прямой маршрутизацией [GCC High и DoD.](https://docs.microsoft.com/microsoftteams/audio-conferencing-with-direct-routing-for-gcch-and-dod)


> [!IMPORTANT]
>  В случае, если вы хотите добавить внешних участников к запланированным собраниям, набрав для них номер телефонного доступа или предоставив номер для телефонного доступа, потребуется лицензия на аудиоконференцию.


### <a name="ad-hoc-call-escalation-and-audio-conferencing-license"></a>Лицензия на аудиоконференцию и цен на аудиоконференцию

Пользователь Teams может запустить прикренную к STN или Teams вызов Teams и добавить участника STN в него. Этот сценарий называется конференцией. Путь к звонку зависит от того, назначена ли пользователю, который передает звонок, лицензию на аудиоконференцию Майкрософт.

- Если пользователю Teams, который сообщает о вызове, назначена лицензия на аудиоконференцию Майкрософт, она проходит через службу аудиоконференций Майкрософт. Удаленный участник ДНР, приглашенный к существующему звонку, получает уведомление о входящий звонок и видит номер моста Microsoft, который назначен пользователю Teams, инициировал переход.
- Если пользователю Teams, который передает звонок, не назначена лицензия на аудиоконференцию Майкрософт, она проходит через контроллер границы сеанса, подключенный к интерфейсу прямой маршрутации. Удаленный участник ДНР, приглашенный на вызов, получает уведомление о входящий звонок и видит номер пользователя Teams, который инициировал перенастройка. Конкретный SBC, используемый для повышения цен, определяется политикой маршрутации пользователя. 


Кроме того, необходимо обеспечить следующее:
 
- Пользователю назначена лицензия CsOnlineVoiceRoutingPolicy. 
- Разрешение личных звонков включено на уровне клиента для Microsoft Teams. 

Прямая маршрутия также поддерживает пользователей, у которых есть лицензия на план звонков Майкрософт. Телефонная система Майкрософт с планом звонков может перенаправить некоторые звонки с помощью интерфейса прямой маршрутинга. Однако номера телефонов пользователей должны быть либо приобретены через Интернет, либо переносимы в Корпорацию Майкрософт.  

Смешивать план звонков и прямой маршрутинг для одного пользователя необязательно, но это может быть полезно (например, если пользователю назначен план звонков Майкрософт, но он хочет перенаправить некоторые звонки с помощью SBC). Один из наиболее распространенных сценариев — вызовы сторонних УАКС.  При использовании сторонних УАП все звонки, кроме телефонов, подключенных к этой УАПС, перенаправются с помощью плана звонков Майкрософт, но звонки на телефоны, подключенные к УАО сторонних телефонов, перенаправят на SBC и, следовательно, остаются в корпоративной сети, а не через ПС. 

Дополнительные сведения о лицензировании телефонной системы см. в дополнительных [сведениях](https://products.office.com/compare-all-microsoft-office-products?tab=2) о вариантах Office и [плана.](https://technet.microsoft.com/library/office-365-plan-options.aspx) 

Дополнительные сведения о лицензировании телефонной системы см. в лицензировании надстройки [Microsoft Teams.](https://docs.microsoft.com/microsoftteams/teams-add-on-licensing/microsoft-teams-add-on-licensing)

## <a name="sbc-domain-names"></a>Доменные имена SBC

Доменное имя SBC должно быть в одном из имен, зарегистрированных в доменах клиента. Имя \* FQDN SBC не onmicrosoft.com клиента .onmicrosoft.com.

В следующей таблице показаны примеры имен DNS, зарегистрированных для клиента, их можно ли использовать в качестве FQDN для SBC, и примеры допустимого имени FQDN:

|Имя DNS|Может использоваться для FQDN SBC|Примеры имен FQDN|
|:--- |:--- |:--- |
contoso.com|Да|**Допустимые имена:**<br/>sbc1.contoso.com<br/>ssbcs15.contoso.com<br/>europe.contoso.com|
|contoso.onmicrosoft.com|Нет|Использование доменов *.onmicrosoft.com не поддерживается для имен SBC

Предположим, вы хотите использовать новое доменное имя. Например, ваш клиент contoso.com как доменное имя, зарегистрированное в вашем клиенте, и вы хотите использовать sbc1.sip.contoso.com. Прежде чем вы сможете в качестве связи SBC с именем sbc1.sip.contoso.com, необходимо зарегистрировать доменное имя sip.contoso.com в доменах в вашем клиенте. При попытке установить SBC с sbc1.sip.contoso.com перед регистрацией доменного имени вы получите следующее сообщение об ошибке: "Не удается использовать домен sbc1.sip.contoso.com" так как он не был настроен для этого клиента."
После добавления доменного имени также необходимо создать пользователя с именами user@sip.contoso.com и назначить лицензию на Teams. Полное предоставление доменного имени после его добавления в домены клиента, создания пользователя с новым именем и назначенной лицензии может занять до 24 часов. 

Возможно, у компании может быть несколько адресных пробелов SIP в одном клиенте. Например, у компании может быть contoso.com адресной области SIP, fabrikam.com как второй адресной области SIP. У некоторых пользователей есть адресная user@contoso.com а у некоторых есть адресная user@fabrikam.com. 

Для SBC требуется только одно FQDN, и он может обслуживания пользователей из любого адресного пространства в парном клиенте. Например, SBC с именем sbc1.contoso.com может получать и отправлять трафик ННР для пользователей с адресами user@contoso.com и user@fabrikam.com при регистрации этих SIP-адресов в одном клиенте.  

## <a name="public-trusted-certificate-for-the-sbc"></a>Общедоступный доверенный сертификат для SBC

Корпорация Майкрософт рекомендует запросить сертификат для SBC путем создания запроса на сертификацию (CSR). Инструкции по созданию CSR-запроса для SBC можно найти в инструкциях по межсоединению или документации поставщиков SBC. 

  > [!NOTE]
  > Большинству центр сертификации требуется, чтобы размер закрытого ключа не был меньше 2048. Помните об этом при создании CSR.

В качестве общего имени (CN) или альтернативного имени субъекта (SAN) сертификат должен иметь FQDN SBC. Сертификат должен выдаваться непосредственно в сертификации, а не у промежуточного поставщика.

Кроме того, direct Routing поддерживает поддиаграммные знаки в cn и/или SAN, а поддиапный знак должен соответствовать стандарту [RFC HTTP Over TLS.](https://tools.ietf.org/html/rfc2818#section-3.1) В качестве примера можно использовать contoso.com, которая будет совпадать со sbc.contoso.com ИД SBC, но не со \* sbc.test.contoso.com.

## <a name="sip-signaling-fqdns"></a>Сигнальный SIP: FQDNs 

Прямая маршрутная маршрутия предоставляется в следующих средах:
- Microsoft 365 или Office 365
- Office 365 GCC
- Office 365 GCC High
- Office 365 DoD

Узнайте больше об [средах Office 365](https://docs.microsoft.com/office365/servicedescriptions/office-365-platform-service-description/office-365-us-government/office-365-us-government) и государственных органов США, таких как GCC, GCC High и DoD.

### <a name="microsoft-365-office-365-and-office-365-gcc-environments"></a>Среды Microsoft 365, Office 365 и GCC Office 365

Ниже точки соединения для прямой маршрутии связаны с тремя FQDNs.

- **sip.pstnhub.microsoft.com** — глобальное FQDN — необходимо сначала пройти процесс. Когда SBC отправляет запрос на устранение этого имени, DNS-серверы Microsoft Azure возвращают IP-адрес, указывающий на основной центр обработки данных Azure, который назначен SBC. Назначение основано на метриках производительности центра обработки данных и географической близости к SBC. Возвращаемый IP-адрес соответствует основному FQDN.
- **sip2.pstnhub.microsoft.com** — дополнительное FQDN — географически сопоставить со вторым приоритетом регион.
- **sip3.pstnhub.microsoft.com** — tertiary FQDN — географически сопоставить с третьим приоритетом регион.

Для размещения этих трех FQDNs в нужном порядке:

- Обеспечение оптимальной работы (менее загружена и ближе всего к центру обработки данных SBC, назначенного с помощью запроса первого FQDN).
- Отбой в случае подключения с SBC к центру обработки данных, который имеет временную проблему. Дополнительные сведения см. ниже [в механизме failover.](#failover-mechanism-for-sip-signaling)  

FQDNs — sip.pstnhub.microsoft.com, sip2.pstnhub.microsoft.com и sip3.pstnhub.microsoft.com — будут разрешены на одном из следующих IP-адресов:

- 52.114.148.0
- 52.114.132.46 
- 52.114.75.24 
- 52.114.76.76 
- 52.114.7.24 
- 52.114.14.70
- 52.114.16.74
- 52.114.20.29

Для сигнального трафика необходимо открыть в брандмауэре порты для всех этих IP-адресов, чтобы разрешить входящий и исходяющий трафик, который будет входящие и исходяющие сообщения с адресов для сигнального трафика.  Если брандмауэр поддерживает имена DNS,  доменные sip-all.pstnhub.microsoft.com будут разрешать их все IP-адреса. 

> [!IMPORTANT]
>  В рамках расширения и улучшения прямой маршрутии Teams мы развернули новые экземпляры инфраструктуры прямой маршрутинга в Австралии. Это отражается на двух дополнительных IP-адресах (52.114.16.74 и 52.114.20.29), на которые будут разрешены следующие FQDNs для клиентов из Австралии : sip.pstnhub.microsoft.com, sip2.pstnhub.microsoft.com и sip3.pstnhub.microsoft.com. Вам нужно добавить эти два IP-адреса (52.114.16.74 и 52.114.20.29) в списки управления IP-доступом и открыть порты для всех этих IP-адресов в брандмауэре, чтобы разрешить входящий и исходяющий трафик до адресов для сигнального трафика и с них.

## <a name="media-traffic-media-processors-geography"></a>Трафик мультимедиа: географические процессоры мультимедиа

Потоки медиапотока через компоненты, называемые процессорами мультимедиа. Процессоры мультимедиа размещаются в тех же центрах обработки данных, что и прокси-системы SIP. Кроме того, существуют дополнительные процессоры мультимедиа для оптимизации потока мультимедиа. Например, прокси-сервер SIP сейчас не имеется в Австралии (потоки SIP через Сингапур или Гонконг), но имеется процессор мультимедиа локально в Австралии. Потребность в процессорах мультимедиа на локальном уровне определяется задержкой, которую мы видим, отправив трафик междугороднего трафика, например из Австралии в Сингапур или Гонконг. В примере с трафиком, который идет из Австралии в Гонконг или Сингапур, можно сохранить высокое качество звонка для трафика SIP, а для трафика мультимедиа в режиме реального времени — нет.

Расположение процессоров мультимедиа:

Расположения, в которых были развернуты компоненты прокси-сервера SIP и процессора мультимедиа:
- США (два центра обработки данных в западном и восточном центрах США)
- Европа (центр обработки данных в Ирландии и Ирландии)
- Азия (Сингапур и Гонконг)

Расположения, в которых развернуты только процессоры мультимедиа (потоки SIP через ближайший центр обработки данных, указанный выше):
- Япония (центр обработки данных в восточной и западной части Японии)
- Австралия (восточной и юго-восточной части Австралии)




## <a name="media-traffic-codecs"></a>Трафик мультимедиа: кодеки

### <a name="leg-between-sbc-and-cloud-media-processor-or-microsoft-teams-client"></a>Размыкать между SBC и облачным процессором мультимедиа или клиентом Microsoft Teams.
Применяется как к случаям обхода мультимедиа, так и к досье без обхода.

Интерфейс прямой маршрутики на отгоне между контроллером границы сеанса и облачным процессором мультимедиа (без обхода мультимедиа) или между клиентом Teams и SBC (если включен обход мультимедиа) может использовать следующие кодеки:

- Обход без мультимедиа (SBC в облачный процессор мультимедиа): SILK, G.711, G.722, G.729
- Обход мультимедиа (клиент SBC к Teams): SILK, G.711, G.722, G.729

Вы можете принудительно использовать определенный кодек на граничном контроллере сеанса, исключив из предложения нежелательные кодеки.

### <a name="leg-between-microsoft-teams-client-and-cloud-media-processor"></a>Leg between Microsoft Teams Client and Cloud Media Processor
Применяется только к случаям обхода мультимедиа. При обходе мультимедиа потоки мультимедиа напрямую между клиентом Teams и SBC.

На отгоне между облачным процессором мультимедиа и клиентом Microsoft Teams используется silk или G.722. Выбор кодека для этого отреза основан на алгоритмах Майкрософт, которые принимают во внимание несколько параметров.
