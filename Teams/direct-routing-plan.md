# <a name="plan-direct-routing"></a>Планирование прямой маршрутизации

> [!Tip]
> Просмотрите следующий сеанс, чтобы узнать о преимуществах прямой маршрутизации, планировании и развертывании прямой маршрутизации [в Microsoft Teams](https://aka.ms/teams-direct-routing).

Прямая маршрутизация позволяет подключить поддерживаемый предоставляемый клиентом пограничный контроллер сеансов (SBC) к телефонной системе. С помощью этой возможности можно настроить подключение локальной телефонной сети общего пользования (ТСОП) к клиенту Microsoft Teams, как показано на следующей схеме: 

![Схема, показывающая конфигурацию локального подключения ТСОП.](media/PlanDirectRouting1-PSTNwithTeams.png "Настройка подключения локального ТСОП к клиенту Microsoft Teams")

  > [!NOTE]
  > Skype для бизнеса Online также позволяет связать предоставленный клиентом SBC, но для этого требуется локальное развертывание Skype для бизнеса Server или специальный выпуск Skype для бизнеса, называемый Cloud Connector, между SBC и Microsoft Cloud. Этот сценарий называется гибридным голосом. В отличие от этого прямая маршрутизация обеспечивает прямое подключение между поддерживаемым SBC и Microsoft Cloud.

> [!Important]
> Выпуск Cloud Connector будет снят с учета 31 июля 2021 г. вместе с Skype для бизнеса Online. После обновления организации до Teams узнайте, как подключить локальную телефонную сеть к Teams с помощью [прямой маршрутизации](direct-routing-landing-page.md). 

С помощью прямой маршрутизации можно подключить SBC практически к любой магистрали телефонии или подключиться к стороннему оборудованию ТСОП. Прямая маршрутизация позволяет выполнять следующие действия: 

- Используйте практически любую магистраль ТСОП с телефонной системой. 

- Настройте взаимодействие между оборудованием телефонии, принадлежащим клиенту, таким как сторонний обмен частными филиалами (УАТС), аналоговые устройства и Teams.

Корпорация Майкрософт также предлагает все облачные голосовые решения, такие как тарифный план. Однако гибридное голосовое решение лучше всего подходит для вашей организации, если: 

- План звонков Майкрософт недоступен в вашей стране. 

- Вашей организации требуется подключение к сторонним аналоговим устройствам, центрам обработки вызовов и т. д. 

- У вашей организации есть существующий контракт с оператором ТСОП.

Прямая маршрутизация также поддерживает пользователей, у которых есть дополнительная лицензия для плана звонков Майкрософт. Дополнительные сведения см. в [разделе "Телефонная система и планы звонков"](calling-plan-landing-page.md). 

При прямой маршрутизации, когда пользователи участвуют в запланированной конференции, телефонный номер предоставляется службой аудиоконференций Майкрософт, которая требует надлежащего лицензирования.  При исходящих звонках служба аудиоконференций Майкрософт выполняет вызов с помощью сетевых функций звонков, что требует надлежащего лицензирования. (Обратите внимание, что если у пользователя нет лицензии на аудиоконференции Майкрософт, вызов направляется через прямую маршрутизацию.) Дополнительные сведения см. в [статье "Собрания по сети с помощью Teams"](https://www.microsoft.com/en-us/microsoft-teams/online-meetings).

## <a name="licensing-and-other-requirements"></a>Лицензирование и другие требования 

Пользователям прямой маршрутизации в Microsoft 365 должны быть назначены следующие лицензии: 

- Телефонная система Майкрософт
- Microsoft Teams + Skype для бизнеса план 2, если он включен в лицензирование
- Аудиоконференции Майкрософт (прочитайте заметки и абзац ниже, чтобы узнать, когда требуется эта лицензия).)

> [!NOTE]
> Skype для бизнеса план не следует удалять из лицензионного соглашения, в котором он включен. 
> 
> [!IMPORTANT]
> Пользователи GCC High и DoD должны отключить все лицензии аудиоконференций, включенные в G5, и ожидать включения любой аудиоконференции, пока прямая маршрутизация не будет полностью настроена. Перед включением лицензий на аудиоконференции пользователям необходимо настроить номера телефонов с телефонным подключением и рабочую панель набора номера. Дополнительные сведения см. в статье "Аудиоконференции с прямой маршрутизаций [для GCC High и DoD](./audio-conferencing-with-direct-routing-for-gcch-and-dod.md) ".


> [!IMPORTANT]
>  Если вы хотите добавить внешних участников к запланированным собраниям, выполнив с ними телефонный номер или указав номер, требуется лицензия на аудиоконференции.
> Для GCC High и DoD не назначайте лицензию на аудиоконференции для пользователей G5. Для пользователей G3 не назначайте лицензию на аудиоконференции, пока прямая маршрутизация не будет полностью настроена и у пользователя будет рабочая панель набора номера.


### <a name="ad-hoc-call-escalation-and-audio-conferencing-license"></a>Нерегламентированное укрупнение вызовов и лицензия на аудиоконференции

Пользователь Teams может начать вызов "один к одному" teams-to-PSTN или Teams-to-Teams и добавить в него участника ТСОП. Этот сценарий называется нерегламентированной конференцией. Путь, который выполняется при вызове, зависит от того, назначена ли пользователю, который передает звонок, лицензию на аудиоконференции Майкрософт:

- **Если пользователю Teams,** который передает звонок, назначена лицензия на аудиоконференции Майкрософт, эскалация происходит через службу аудиоконференций Майкрософт. Удаленный участник ТСОП, приглашенный на существующий вызов, получает уведомление о входящем вызове и видит номер моста Майкрософт, назначенного пользователю Teams, который инициировал эскалацию.

- **Если пользователю Teams**, который передает звонок, не назначена лицензия на аудиоконференции Майкрософт, эскалация происходит через пограничный контроллер сеанса, подключенный к интерфейсу прямой маршрутизации. Удаленный участник ТСОП, приглашенный на вызов, получает уведомление о входящем вызове и видит номер пользователя Teams, который инициировал эскалацию. Конкретный SBC, используемый для эскалации, определяется политикой маршрутизации пользователя. 

Необходимо убедиться в следующем:
 
- Пользователю назначается CsOnlineVoiceRoutingPolicy. 

- Разрешение частных звонков включено на уровне клиента для Microsoft Teams. 

Прямая маршрутизация также поддерживает пользователей, у которых есть лицензия на план звонков Майкрософт. Телефонная система с тарифным планом может маршрутизировать некоторые вызовы с помощью интерфейса прямой маршрутизации. Однако номера телефонов пользователей должны быть либо получены в Интернете, либо перенаправлены в корпорацию Майкрософт.  

Смешивание плана звонков и прямого подключения маршрутизации для одного и того же пользователя необязательно, но может оказаться полезным. Например, если пользователю назначен план звонков Майкрософт, но он хочет перенаправить некоторые вызовы с помощью SBC. Один из наиболее распространенных сценариев — вызовы сторонних УАТС.  При использовании сторонних УАТС все вызовы, кроме вызовов телефонов, подключенных к этой УАТС, направляются с помощью плана звонков Майкрософт, но вызовы телефонов, подключенных к сторонним УАТС, направляются в SBC и, следовательно, остаются в корпоративной сети, а не в ТСОП. 

Дополнительные сведения о лицензировании телефонной системы см. в статье "Максимально эффективное использование [Office](https://products.office.com/compare-all-microsoft-office-products?tab=2) " и ["Параметры](/office365/servicedescriptions/office-365-platform-service-description/office-365-plan-options) плана" и "Лицензирование надстроек [Microsoft Teams"](./teams-add-on-licensing/microsoft-teams-add-on-licensing.md).

## <a name="sbc-domain-names"></a>Доменные имена SBC

Доменное имя SBC должно быть из одного из имен, зарегистрированных в доменах клиента. Нельзя использовать \*клиент .onmicrosoft.com для полного доменного имени SBC.

В следующей таблице показаны примеры DNS-имен, зарегистрированных для клиента, возможность использования имени в качестве полного доменного имени для SBC, а также примеры допустимых имен полных доменных имен:

|DNS-имя|Может использоваться для полного доменного имени SBC|Примеры имен полных доменных имен|
|:--- |:--- |:--- |
contoso.com|Да|**Допустимые имена:**<br/>sbc1.contoso.com<br/>ssbcs15.contoso.com<br/>europe.contoso.com|
|contoso.onmicrosoft.com|Нет|Использование доменов *.onmicrosoft.com не поддерживается для имен SBC

Предположим, вы хотите использовать новое доменное имя. Например, ваш клиент contoso.com как доменное имя, зарегистрированное в клиенте, и вы хотите использовать sbc1.sip.contoso.com. Прежде чем связать SBC с именем sbc1.sip.contoso.com, необходимо зарегистрировать доменное имя sip.contoso.com в доменах в клиенте. При попытке связать SBC с sbc1.sip.contoso.com перед регистрацией доменного имени вы получите следующую ошибку: "Не удается использовать домен sbc1.sip.contoso.com, так как он не был настроен для этого клиента".
После добавления доменного имени необходимо также создать пользователя с именем участника-пользователя user@sip.contoso.com и назначить лицензию Teams. Полная подготовка доменного имени после его добавления в домены клиента, создания пользователя с новым именем и назначения пользователю лицензии может занять до 24 часов. 

В организации может быть несколько адресных пространств SIP в одном клиенте. Например, компания может использовать contoso.com адресное пространство SIP и fabrikam.com как второе адресное пространство SIP. У некоторых пользователей есть адрес user@contoso.com, а у некоторых — user@fabrikam.com. 

Для SBC требуется только одно полное доменное имя, и он может обслужить пользователей из любого адресного пространства в парном клиенте. Например, SBC с именем sbc1.contoso.com может получать и отправлять трафик ТСОП для пользователей с адресами user@contoso.com и user@fabrikam.com при условии, что эти адресные пространства SIP зарегистрированы в одном клиенте.  

 > [!NOTE]
 > Полное доменное имя SBC в Службы коммуникации Azure прямой маршрутизации должно отличаться от полного доменного имени SBC в прямой маршрутизации Teams.
  
## <a name="public-trusted-certificate-for-the-sbc"></a>Общедоступный доверенный сертификат для SBC

Корпорация Майкрософт рекомендует запросить сертификат для SBC, создав запрос на подписывание сертификата (CSR). Конкретные инструкции по созданию CSR для SBC см. в инструкциях или документации по подключению, предоставленных поставщиками SBC. 

> [!NOTE]
> Для большинства центров сертификации (ЦС) требуется размер закрытого ключа не менее 2048. Помните об этом при создании CSR.

Сертификат должен иметь полное доменное имя SBC в качестве общего имени (CN) или поля альтернативного имени субъекта (SAN).

Кроме того, прямая маршрутизация поддерживает подстановочный знак в CN и (или) SAN, а подстановочный знак должен соответствовать стандартному [протоколу HTTP RFC по протоколу TLS](https://tools.ietf.org/html/rfc2818#section-3.1). Например, можно использовать \*.contoso.com, который будет соответствовать FQDN SBC sbc.contoso.com, но не будет соответствовать sbc.test.contoso.com.

Интерфейс SIP прямой маршрутизации будет доверять только сертификатам, подписанным центрами сертификации (ЦС), которые являются частью программы доверенных корневых сертификатов Майкрософт. Убедитесь, что сертификат SBC подписан центром сертификации, который является частью программы, и что расширение расширенного использования ключа (EKU) сертификата включает проверку подлинности сервера.
Дополнительные сведения: [Требования к программе — доверенная корневая программа Майкрософт](/security/trusted-root/program-requirements)
  
[Включенный список сертификатов ЦС](https://ccadb-public.secure.force.com/microsoft/IncludedCACertificateReportForMSFT)
  
 Для прямой маршрутизации Office 365 средах GCCH и DoD сертификат должен быть создан одним из следующих корневых центров сертификации:

- DigiCert Global Root CA
- Корневой ЦС DigiCert High Assurance EV

> [!NOTE]
> Если для подключения Teams в SBC включена взаимная поддержка TLS (MTLS), необходимо установить корневой каталог Baltimore CyberTrust и сертификаты DigiCert Global Root G2 в надежном корневом хранилище SBC в контексте TLS Teams. (Это связано с тем, что сертификаты служб Майкрософт используют один из этих двух корневых сертификатов.) Чтобы скачать эти корневые сертификаты, [см. Office 365 цепочки шифрования](/microsoft-365/compliance/encryption-office-365-certificate-chains). Дополнительные сведения см. в статье [об изменениях сертификата Office TLS](/microsoft-365/compliance/encryption-office-365-tls-certificates-changes).

## <a name="sip-signaling-fqdns"></a>СигналЫ SIP: полные доменные имена 

Прямая маршрутизация предлагается в следующих средах:

- Microsoft 365 или Office 365
- Office 365 GCC
- Office 365 GCC High
- Office 365 DoD

Узнайте больше о [Office 365 и](/office365/servicedescriptions/office-365-platform-service-description/office-365-us-government/office-365-us-government) государственных организаций США, таких как GCC, GCC High и DoD.

### <a name="microsoft-365-office-365-and-office-365-gcc-environments"></a>Microsoft 365, Office 365 и Office 365 GCC

Точки подключения для прямой маршрутизации — это три полных доменных имени:

- **sip.pstnhub.microsoft.com** — глобальное полное доменное имя — необходимо сначала выполнить попытки. Когда SBC отправляет запрос на разрешение этого имени, DNS-серверы Microsoft Azure возвращают IP-адрес, указывающий на основной центр обработки данных Azure, назначенный SBC. Назначение основано на метриках производительности центров обработки данных и географической близости к SBC. Возвращенный IP-адрес соответствует основному FQDN.

- **sip2.pstnhub.microsoft.com** — дополнительное полное доменное имя — географически сопоставляется со вторым приоритетным регионом.

- **sip3.pstnhub.microsoft.com** — полное доменное имя — географически сопоставляется с третьим приоритетным регионом.

Размещение этих трех полных доменных имен по порядку требуется для:

- Обеспечивает оптимальное взаимодействие (менее загруженное и ближайшее к центру обработки данных SBC, назначенное путем запроса первого полного доменного имени).

- Укажите отработку отказа при установке подключения из SBC к центру обработки данных, в котором возникла временная проблема. Дополнительные сведения см. в [описании механизма отработки отказа](#failover-mechanism-for-sip-signaling) ниже.  

Полные доменные имена (sip.pstnhub.microsoft.com, sip2.pstnhub.microsoft.com и sip3.pstnhub.microsoft.com) будут разрешены в IP-адреса из следующих подсетей:

- 52.112.0.0/14
- 52.120.0.0/14

Необходимо открыть порты для всех диапазонов IP-адресов в брандмауэре, чтобы разрешить входящий и исходящий трафик с адресов для передачи сигналов.

### <a name="office-gcc-dod-environment"></a>Среда Office GCC DoD

Точка подключения для прямой маршрутизации — это следующее полное доменное имя:

**sip.pstnhub.dod.teams.microsoft.us** — глобальное полное доменное имя. Так как Office 365 DoD существует только в центрах обработки данных США, вторичные и тритериалные полные доменные имена отсутствуют.

Полное доменное имя sip.pstnhub.dod.teams.microsoft.us будет разрешено на IP-адрес из следующей подсети:

- 52.127.64.0/21

Необходимо открыть порты для всех этих IP-адресов в брандмауэре, чтобы разрешить входящий и исходящий трафик с адресов для передачи сигналов.

Точка подключения для прямой маршрутизации — это следующее полное доменное имя:

## <a name="media-traffic-media-processors-geography"></a>Трафик мультимедиа: география обработчиков мультимедиа

Трафик мультимедиа проходит через компоненты, называемые обработчиками мультимедиа. Обработчики мультимедиа размещаются в тех же центрах обработки данных, что и прокси-серверы SIP:

- NOAM (центрально-южная часть США, два в западной части США и восточной части США)
- Европа (южная часть Соединенного Королевства, Центральная Франция, Амстердам и амстердамский центр обработки данных)
- Азия (центр обработки данных Сингапура)
- Япония (восточная часть JP и западная часть центра обработки данных)
- Австралия (восточная часть AU и юго-восточная часть центра обработки данных)
- LATAM (Южная Бразилия)
- Африка (Северная Часть ЮАР)

## <a name="media-traffic-codecs"></a>Трафик мультимедиа: кодеки

### <a name="leg-between-sbc-and-cloud-media-processor-or-microsoft-teams-client"></a>Разбор между SBC и обработчиком облачных носителей или клиентом Microsoft Teams

Применяется как к регистру обхода сервера-посредника, так и к вариантам без обхода.

Интерфейс прямой маршрутизации на участке между пограничным контроллером сеанса и обработчиком облачных носителей (без обхода сервера-посредника) или между клиентом Teams и SBC (если включен обход сервера-посредника) может использовать следующие кодеки:

- Обход без носителя (SBC в обработчик облачных носителей): SILK, G.711, G.722, G.729
- Обход сервера-посредника (клиент SBC в Teams): SILK, G.711, G.722, G.729

Вы можете принудительно использовать определенный кодек в пограничном контроллере сеанса, исключив нежелательные кодеки из предложения.

### <a name="leg-between-microsoft-teams-client-and-cloud-media-processor"></a>Этап между клиентом Microsoft Teams и обработчиком облачных носителей

Применяется только к варианту обхода без носителя. При обходе сервера-посредника мультимедиа передаются непосредственно между клиентом Teams и SBC.

На участке между обработчиком облачных носителей и клиентом Microsoft Teams используется SILK или G.722. Выбор кодека на этом этапе основан на алгоритмах Майкрософт, которые принимают во внимание несколько параметров. 

  > [!NOTE]
  > Перенацель мультимедиа не поддерживается. Во время прямого вызова маршрутизации, если SBC отправляет новый IP-адрес мультимедиа в прямую маршрутизацию Teams, хотя он согласован в сигналах SIP, носитель никогда не отправляется на новый IP-адрес из прямой маршрутизации Teams.

## <a name="supported-session-border-controllers-sbcs"></a>Поддерживаемые пограничные контроллеры сеансов (SBC)

Корпорация Майкрософт поддерживает только сертифицированные пакеты SBC для связывания с прямой маршрутиза службой. Так Корпоративная голосовая связь важно для предприятий, корпорация Майкрософт выполняет интенсивные тесты с выбранными контроллерами SBC и работает с поставщиками SBC, чтобы обеспечить совместимость двух систем. 

Проверенные устройства перечислены как сертифицированные для прямой маршрутизации Teams. Сертифицированные устройства гарантированно будут работать во всех сценариях.
