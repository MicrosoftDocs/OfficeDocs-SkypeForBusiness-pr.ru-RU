# <a name="plan-direct-routing"></a>Планирование прямой маршрутизации

> [!Tip]
> Просмотрите следующий сеанс, чтобы узнать о преимуществах прямой маршрутизации, о том, как ее спланировать и развертывание: [Прямая маршрутизация в Microsoft Teams](https://aka.ms/teams-direct-routing)

Прямая маршрутизация Microsoft Phone System позволяет подключить к телефонной системе Microsoft поддерживаемый клиентский контроллер границ (SBC).  С помощью этой возможности можно настроить локальную связь по протоколу КТСОП с клиентом Microsoft Teams, как показано на следующей схеме: 

![Диаграмма, показывающая конфигурацию локальной сети PSTN] (media/PlanDirectRouting1-PSTNwithTeams.png "Настройка локальной сети PSTN в клиенте Microsoft Teams")

  > [!NOTE]
  > В Skype для бизнеса Online также можно связать пользовательский SBC-объект, но для этого требуется локальное развертывание Skype для бизнеса Server или специальная версия Skype для бизнеса, которая называется облачным соединителем между SBC и Microsoft Cloud. Этот сценарий называется гибридным голосом. Напротив, прямая маршрутизация допускает прямую связь между поддерживаемым SBC и облаком Microsoft. 

С помощью прямой маршрутизации вы можете соединить SBC с любой магистральной или межсоединением телефонной связи с помощью стороннего оборудования, подключенного к общественной коммутируемой телефонной сети. Прямая маршрутизация позволяет выполнять указанные ниже действия. 

- Используйте практически любую магистральную сеть PSTN с помощью телефонной системы Microsoft. 
- Настраивать взаимодействие между устройствами телефонной связи, работающими с клиентами, например УАТС, аналоговые устройства и телефонная система Microsoft.

Корпорация Майкрософт также предложит все возможности голосовой связи по облаку, например план звонков.  Тем не менее, гибридное решение голосовой связи может быть лучше для вашей организации, если: 

- План звонков по Microsoft недоступен в вашей стране. 
- Для организации требуется подключение к сторонним устройствам, центрам обработки звонков и т. д. 
- У вашей организации есть контракт с оператором PSTN.

Кроме того, прямая маршрутизация поддерживает пользователей, у которых есть дополнительная лицензия на план звонков по Microsoft. Дополнительные сведения можно найти в разделе [планы телефонной системы и звонков](calling-plan-landing-page.md). 

При прямой маршрутизации, когда пользователи участвуют в запланированной конференции, номер телефонного подключения предоставляется службой видеоконференций Майкрософт, требующей подходящего лицензирования.  При исходящих звонка Служба голосовой связи Microsoft обработает звонок, используя возможности онлайн-звонков, требующие правильного лицензирования. (Обратите внимание на то, что исходящие звонки не проходят через прямую маршрутизацию.) Дополнительные сведения можно найти [в разделе собрания по сети с помощью Teams](https://products.office.com/microsoft-teams/online-meeting-solutions).

## <a name="licensing-and-other-requirements"></a>Лицензирование и другие требования 

Пользователям прямой маршрутизации должны быть назначены следующие лицензии в Office 365: 

- Телефонная система Майкрософт 
- Microsoft Teams 
- Голосовая конференция Майкрософт 

> [!NOTE]
> При распространении из 1:1 звонка требуется лицензия на голосовую конференцию

> [!IMPORTANT]
>  В случае, если вы хотите добавить внешних участников к запланированным собраниям, либо по исходящим звонкам, либо указав номер телефонного подключения, *требуется*лицензия на голосовую конференцию.

> [!NOTE]
> Лицензия «водоустройства» включает в себя как телефонную систему, так и голосовую конференцию.   

Кроме того, необходимо обеспечить следующее:
 
- Ксонлиневоицераутингполици назначен пользователю. 
- Разрешить частный вызов разрешен на уровне клиента для Microsoft Teams. 

Прямая маршрутизация также поддерживает пользователей, которые имеют лицензию на план звонков Microsoft. Телефонная система Microsoft с тарифным планом может маршрутизировать некоторые звонки с помощью интерфейса Direct Routing. Тем не менее, Номера телефонов пользователей должны быть либо приобретены в Интернете, либо перенесены в Microsoft.  

Смешивание плана звонков и прямой маршрутизации для одного и того же пользователя является необязательным, но может быть полезным, если, например, пользователю назначен план звонков по Microsoft, но требуется маршрутизировать некоторые звонки через SBC. Один из наиболее распространенных сценариев — звонки сторонним АТС.  При использовании сторонней АТС все звонки, кроме звонков на телефоны, подключенные к этой АТС, передаются через план звонков по Microsoft. но звонки на телефоны, подключенные к сторонним АТС, походят на SBC, поэтому не забудьте в корпоративной сети, а не в сетях PSTN. 

Дополнительные сведения о лицензировании телефонной системы можно найти в статьях получение максимально возможного [из Office с помощью параметров плана office 365](https://products.office.com/compare-all-microsoft-office-products?tab=2) и [Office 365](https://technet.microsoft.com/library/office-365-plan-options.aspx). 

Дополнительные сведения о лицензировании телефонной системы можно найти [в разделе Лицензирование надстроек Microsoft Teams](teams-add-on-licensing/microsoft-teams-add-on-licensing.md). 

## <a name="sbc-domain-names"></a>Доменные имена SBC

Доменное имя SBC должно состоять из одного из имен, зарегистрированных в "домены" клиента. Вы не можете использовать клиент *. onmicrosoft.com для полного доменного имени SBC.

В следующей таблице приведены примеры имен DNS, зарегистрированных для клиента, будь то имя, которое может использоваться в качестве полного доменного имени для SBC, и примеры допустимых ДОМЕНных имен:

|**DNS-имя**|**Может использоваться для полного доменного имени SBC**|**Примеры ДОМЕНных имен**|
|:--- |:--- |:--- |
contoso.com|Да|**Допустимые имена:**<br/>sbc1.contoso.com<br/>ssbcs15.contoso.com<br/>europe.contoso.com|
|contoso.onmicrosoft.com|Нет|<br/>Использование доменов *. onmicrosoft.com не поддерживается для имен SBC

Предположим, что вы хотите использовать новое доменное имя. Например, ваш клиент имеет contoso.com в качестве доменного имени, зарегистрированного в клиенте, и вы хотите использовать sbc1.sip.contoso.com. Прежде чем приступить к связыванию SBC с именем sbc1.sip.contoso.com, необходимо зарегистрировать доменное имя sip.contoso.com в поле "домены" в клиенте. Если вы попытаетесь связать объект SBC с sbc1.sip.contoso.com перед регистрацией доменного имени, появится следующее сообщение об ошибке: "не удается использовать домен" sbc1.sip.contoso.com, так как он не настроен для этого клиента ".
После добавления доменного имени вам также потребуется создать пользователя с помощью user@sip.contoso.com UPN и назначить лицензию "Teams". Для полной подготовки домена к домену после его добавления в "домены" вашего клиента может потребоваться до 24 часов. будет создано имя пользователя с новым именем и назначена лицензия пользователю. 

Возможно, у компании есть несколько адресных пространств SIP в одном клиенте. Например, компания может использовать contoso.com в качестве адресного пространства SIP и fabrikam.com в качестве второго адресного пространства SIP. Некоторые пользователи имеют адрес user@contoso.com, а некоторые пользователи имеют адрес user@fabrikam.com. 

Для SBC требуется только одно полное доменное имя, которое может обслуживать пользователей из любого адресного пространства в связанном клиенте. Например, SBC с именем sbc1.contoso.com может получать и отправлять трафик КТСОП для пользователей с адресами user@contoso.com и user@fabrikam.com, пока эти адресные пространства SIP зарегистрированы в одном и том же клиенте.  

## <a name="public-trusted-certificate-for-the-sbc"></a>Общедоступный доверенный сертификат для SBC

  > [!NOTE]
  > Большинству центров сертификации (ЦС) требуется, чтобы размер закрытого ключа не менее 2048. Имейте это в виду при формировании CSR.

## <a name="sip-signaling-fqdns-and-firewall-ports"></a>Сигнализация SIP: полные доменные имена и порты брандмауэра 

Точка соединения для прямой маршрутизации является следующими тремя полными доменными адресами:

- **SIP.pstnhub.Microsoft.com** — глобальное полное доменное имя — необходимо попытаться сначала. Когда SBC отправляет запрос для разрешения этого имени, DNS-серверы Microsoft Azure возвращают IP-адрес, указывающий на основной центр обработки данных Azure, назначенный для SBC. Назначение основывается на метриках производительности центров обработки данных и географических расположений, близких к SBC. Возвращенный IP-адрес соответствует основному доменному имени.
- **sip2.pstnhub.Microsoft.com** — дополнительное полное доменное имя — географически сопоставляется со вторым регионом приоритета.
- **SIP3.pstnhub.Microsoft.com** — ТРЕТИЧНОЕ полное доменное имя — географически сопоставляется с третьим регионом приоритета.

Размещение этих трех полных доменных имен в нужном порядке:

- Обеспечьте оптимальную работу (менее загруженные и близкие к центру обработки данных, назначенному с помощью запроса первого полного доменного имени).
- Отработка отказа при установленном соединении из SBC для центра обработки данных, на котором возникают временные проблемы. Дополнительные сведения можно найти в разделе [механизм](#failover-mechanism-for-sip-signaling) отработки отказа ниже.

### <a name="failover-mechanism-for-sip-signaling"></a>Механизм отработки отказа для сигнализации SIP

SBC делает запрос DNS для разрешения sip.pstnhub.microsoft.com. На основе местоположения SBC и метрик производительности центра обработки данных выбирается основной центр обработки данных. Если в основном центре обработки данных возникает неполадка, то SBC попытается sip2.pstnhub.microsoft.com, который разрешается во второй назначенный центр обработки данных, и в редких случаях, когда центры обработки данных в двух регионах недоступны, SBC повторяет Последнее ДОМЕНное имя ( sip3.pstnhub.microsoft.com), который предоставляет IP-адрес для третьего центра обработки данных.

## <a name="supported-session-border-controllers-sbcs"></a>Поддерживаемые контроллеры границ сеанса (SBCs)

Корпорация Майкрософт поддерживает только сертифицированный SBCs для связи с прямой маршрутизацией. Так как для бизнеса важна Корпоративная голосовая связь, корпорация Майкрософт использует ресурсоемкие тесты с выбранной SBCs и работает с поставщиками SBC, чтобы убедиться, что эти две системы совместимы. 

Проверенные устройства указаны в списке сертифицировано для прямой маршрутизации Teams. Сертифицированные устройства гарантированно работают во всех сценариях.
