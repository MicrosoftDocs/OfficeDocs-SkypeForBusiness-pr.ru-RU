# <a name="plan-direct-routing"></a>Планирование прямой маршрутизации

> [!Tip]
> Просмотрите следующий сеанс, чтобы узнать о преимуществах прямой маршрутизации, о том, как ее спланировать и развертывание: [Прямая маршрутизация в Microsoft Teams](https://aka.ms/teams-direct-routing)

Прямая маршрутизация Microsoft Phone System позволяет подключить к телефонной системе Microsoft поддерживаемый клиентский контроллер границ (SBC).  С помощью этой возможности можно настроить локальную связь по коммутируемой телефонной сети с помощью клиента Microsoft Teams, как показано на рисунке ниже. 

![Диаграмма, показывающая конфигурацию локальной сети PSTN](media/PlanDirectRouting1-PSTNwithTeams.png "Настройка локальной сети PSTN в клиенте Microsoft Teams")

  > [!NOTE]
  > В Skype для бизнеса Online также можно связать пользовательский SBC-объект, но для этого требуется локальное развертывание Skype для бизнеса Server или специальная версия Skype для бизнеса, которая называется облачным соединителем между SBC и Microsoft Cloud. Этот сценарий называется гибридным голосом. Напротив, прямая маршрутизация допускает прямую связь между поддерживаемым SBC и облаком Microsoft.

> [!Important]
> С помощью Cloud Connector Edition будет прекращено 31 июля 2021 г., а также Skype для бизнеса Online. После того как ваша организация будет обновлена до Teams, прочитайте о том, как подключить локальную телефонную сеть к Teams с помощью [прямой маршрутизации](direct-routing-landing-page.md). 

С помощью прямой маршрутизации вы можете соединить SBC с любой магистральной или межсоединением телефонной связи с помощью стороннего оборудования PSTN. Прямая маршрутизация позволяет выполнять указанные ниже действия. 

- Используйте практически любую магистральную сеть PSTN с помощью телефонной системы Microsoft. 
- Настраивать взаимодействие между устройствами телефонной связи, работающими с клиентами, например УАТС, аналоговые устройства и телефонная система Microsoft.

Корпорация Майкрософт также предложит все возможности голосовой связи по облаку, например план звонков. Тем не менее, гибридное решение голосовой связи может быть лучше для вашей организации, если: 

- План звонков по Microsoft недоступен в вашей стране. 
- Для организации требуется подключение к сторонним устройствам, центрам обработки звонков и т. д. 
- У вашей организации есть контракт с оператором PSTN.

Кроме того, прямая маршрутизация поддерживает пользователей, у которых есть дополнительная лицензия на план звонков по Microsoft. Дополнительные сведения можно найти в разделе [планы телефонной системы и звонков](calling-plan-landing-page.md). 

При прямой маршрутизации, когда пользователи участвуют в запланированной конференции, номер телефонного подключения предоставляется службой видеоконференций Майкрософт, требующей подходящего лицензирования.  При исходящих звонка Служба голосовой связи Microsoft обработает звонок, используя возможности онлайн-звонков, требующие правильного лицензирования. (Обратите внимание, что если у пользователя нет лицензии на Конференц-связь Майкрософт, маршруты к ней проходят через прямую маршрутизацию.) Дополнительные сведения можно найти [в разделе собрания по сети с помощью Teams](https://products.office.com/microsoft-teams/online-meeting-solutions).

## <a name="licensing-and-other-requirements"></a>Лицензирование и другие требования 

Пользователям прямой маршрутизации должны быть назначены следующие лицензии в Microsoft 365 или Office 365: 

- Телефонная система Microsoft. 
- Microsoft Teams + Skype для бизнеса (план 2), если он включен в лицензирование.
- Видеоконференции (Майкрософт) ознакомьтесь с заметками и параграфами ниже, в которых приведены примеры того, когда требуется лицензия.

> [!NOTE]
> План Skype для бизнеса не должен быть удален из любого лицензионного соглашения, в котором она включена. 


> [!IMPORTANT]
>  В случае, если вы хотите добавить внешних участников к запланированным собраниям, либо по исходящим звонкам, либо указав номер телефонного подключения, требуется лицензия на голосовую конференцию.


### <a name="ad-hoc-call-escalation-and-audio-conferencing-license"></a>Лицензия на распространение нерегламентированных звонков и голосовую конференцию

Пользователи Teams могут присоединиться к сети PSTN или Teams в Teams и добавить в нее участника PSTN. Этот сценарий называется нерегламентированной Конференцией. Путь, который проходит звонок, зависит от того, назначена ли у пользователя, который эскалируется этот звонок, лицензия Microsoft для конференц-связи.

- Если пользователь Teams, который эскалировать звонок, имеет назначенную лицензию на голосовую конференцию Microsoft, это происходит в службе голосовой связи Microsoft. Удаленный PSTN-участник, который приглашен на существующий звонок, получает уведомление о входящем звонке и видит номер моста Microsoft, назначенного пользователю Teams, который инициировал эскалацию.
- Если пользователь Teams, который эскалировать звонок, не имеет назначенной лицензии Microsoft Audio конференц-связи, эскалация происходит через контроллер границ сеанса, подключенный к интерфейсу Direct Routing. Удаленный PSTN-участник, который приглашен на звонок, получает уведомление о входящем звонке и видит номер пользователя Teams, который инициировал эскалацию. Конкретный SBC, используемый для эскалации, определяется политикой маршрутизации пользователя. 


Кроме того, необходимо обеспечить следующее:
 
- CsOnlineVoiceRoutingPolicy назначен пользователю. 
- Разрешить частный вызов разрешен на уровне клиента для Microsoft Teams. 

Прямая маршрутизация также поддерживает пользователей, которые имеют лицензию на план звонков Microsoft. Телефонная система Microsoft с тарифным планом может маршрутизировать некоторые звонки с помощью интерфейса Direct Routing. Тем не менее, Номера телефонов пользователей должны быть либо приобретены в Интернете, либо перенесены в Microsoft.  

Смешивание плана звонков и прямой маршрутизации для одного и того же пользователя является необязательным, но может быть полезным (например, когда пользователю назначается план звонков по Microsoft, но требуется перенаправление некоторых звонков с помощью SBC). Одним из наиболее распространенных сценариев является вызов сторонней АТС.  При использовании сторонней АТС все звонки за исключением звонков на телефоны, подключенные к этой АТС, передаются по плану звонков по Microsoft, но звонки на телефоны, подключенные к сторонним АТС, походят на SBC и, следовательно, не будут находиться в корпоративной сети, а не в сетях PSTN. 

Дополнительные сведения о лицензировании телефонной системы можно найти в [статьях получение максимально](https://products.office.com/compare-all-microsoft-office-products?tab=2) возможного из Office и [параметров плана](https://technet.microsoft.com/library/office-365-plan-options.aspx). 

Дополнительные сведения о лицензировании телефонной системы можно найти [в разделе Лицензирование надстроек Microsoft Teams](https://docs.microsoft.com/microsoftteams/teams-add-on-licensing/microsoft-teams-add-on-licensing).

## <a name="sbc-domain-names"></a>Доменные имена SBC

Имя домена SBC должно состоять из одного из имен, зарегистрированных в доменах клиента. Вы не можете использовать \* клиент. onmicrosoft.com для полного доменного имени SBC.

В следующей таблице приведены примеры имен DNS, зарегистрированных для клиента, будь то имя, которое может использоваться в качестве полного доменного имени для SBC, и примеры допустимых ДОМЕНных имен:

|DNS-имя|Может использоваться для полного доменного имени SBC|Примеры ДОМЕНных имен|
|:--- |:--- |:--- |
contoso.com|Да|**Допустимые имена:**<br/>sbc1.contoso.com<br/>ssbcs15.contoso.com<br/>europe.contoso.com|
|contoso.onmicrosoft.com|Нет|Использование доменов *. onmicrosoft.com не поддерживается для имен SBC

Предположим, что вы хотите использовать новое доменное имя. Например, ваш клиент имеет contoso.com в качестве доменного имени, зарегистрированного в клиенте, и вы хотите использовать sbc1.sip.contoso.com. Прежде чем приступить к связыванию SBC с именем sbc1.sip.contoso.com, необходимо зарегистрировать доменное имя sip.contoso.com в доменах клиента. Если вы попытаетесь связать объект SBC с sbc1.sip.contoso.com перед регистрацией доменного имени, появится следующее сообщение об ошибке: "не удается использовать домен" sbc1.sip.contoso.com, так как он не настроен для этого клиента ".
После добавления доменного имени вам также потребуется создать пользователя с помощью user@sip.contoso.com UPN и назначить лицензию Teams. Для полной подготовки доменного имени после его добавления в домены может потребоваться до 24 часов. будет создано имя пользователя с новым именем, и ему будет назначена лицензия. 

Возможно, у компании есть несколько адресных пространств SIP в одном клиенте. Например, компания может использовать contoso.com в качестве адресного пространства SIP и fabrikam.com в качестве второго адресного пространства SIP. Некоторые пользователи имеют адрес user@contoso.com, а некоторые пользователи имеют адрес user@fabrikam.com. 

Для SBC требуется только одно полное доменное имя, которое может обслуживать пользователей из любого адресного пространства в связанном клиенте. Например, SBC с именем sbc1.contoso.com может получать и отправлять трафик КТСОП для пользователей с адресами user@contoso.com и user@fabrikam.com, пока эти адресные пространства SIP зарегистрированы в одном и том же клиенте.  

## <a name="public-trusted-certificate-for-the-sbc"></a>Общедоступный доверенный сертификат для SBC

Корпорация Microsoft рекомендует запросить сертификат для SBC, создав запрос на подписывание сертификатов (CSR). Для получения конкретных инструкций по созданию CSR для SBC ознакомьтесь с инструкциями по подключению или документацией, предоставленными поставщиками SBC. 

  > [!NOTE]
  > Большинству центров сертификации (ЦС) требуется, чтобы размер закрытого ключа не менее 2048. Имейте это в виду при формировании CSR.

В поле субъекта сертификата должно быть указано полное доменное имя SBC (CN).

Кроме того, прямая маршрутизация поддерживает подстановочные знаки в сети SAN, а подстановочные знаки должны соответствовать стандартным стандартам [HTTP по](https://tools.ietf.org/html/rfc2818#section-3.1)протоколу TLS. Пример — это использование \* . contoso.com в сети SAN, которое соответствует доменному имени SBC SBC.contoso.com, но не совпадает с SBC.Test.contoso.com.

Сертификат должен быть создан одним из следующих корневых центров сертификации:

Для прямой маршрутизации используются следующие три полных доменных имен:

- **SIP.pstnhub.Microsoft.com** — глобальное полное доменное имя — необходимо попытаться сначала. Когда SBC отправляет запрос для разрешения этого имени, DNS-серверы Microsoft Azure возвращают IP-адрес, указывающий на основной центр обработки данных Azure, назначенный для SBC. Назначение основывается на метриках производительности центров обработки данных и географических расположений, близких к SBC. Возвращенный IP-адрес соответствует основному доменному имени.

Размещение этих трех полных доменных имен в нужном порядке:

- Обеспечьте оптимальную работу (менее загруженные и близкие к центру обработки данных, назначенному с помощью запроса первого полного доменного имени).
- Отработка отказа при установленном соединении из SBC для центра обработки данных, на котором возникают временные проблемы. Дополнительные сведения можно найти в разделе [механизм отработки отказа](#failover-mechanism-for-sip-signaling) ниже.

## <a name="media-traffic-media-processors-geography"></a>Мультимедийный трафик: География процессоров мультимедиа

Трафик мультимедиа проходит через компоненты, называемые процессорами мультимедиа. Процессоры мультимедиа размещаются в одном и том же центре обработки данных, что и прокси-серверы SIP. Кроме того, существуют дополнительные процессоры мультимедиа для оптимизации потоков мультимедиа. Например, у нас нет прокси-сервера SIP прямо сейчас в Австралии (SIP – это поток с помощью Сингапур или Гонконг), но у нас есть процессор мультимедиа локально в Австралии. Потребность в локальных процессорах мультимедиа определяется задержкой, которую мы получаем, отправляя трафик на междугородную линию, например из Австралии в Сингапур или Гонконг. Несмотря на то, что в примере трафика из Австралии в Гонконг или Сингапур приемлемо высокое качество для передачи данных по протоколу SIP, для мультимедийного трафика в реальном времени это не так.

Расположение процессоров мультимедиа:

Места, где развернуты оба компонента SIP и процессоров мультимедиа:
- США (два в США и в центре обработки данных Восточной США)
- Европа (Амстердам и Дублин Datacenter)
- Азия (Сингапур и Гонконг центры обработки данных)

Расположения, в которых развернуты только процессоры мультимедиа (потоки SIP по ближайшему центру обработки данных перечислены выше).
- Япония (JP, Восток и Западная центр обработки данных)
- Австралия (AU и центры обработки данных юго-восток)




## <a name="media-traffic-codecs"></a>Мультимедийный трафик: кодеки

### <a name="leg-between-sbc-and-cloud-media-processor-or-microsoft-teams-client"></a>Соединение между SBC и процессором Microsoft Teams.
Применяется к обоим случаям пропуска и отсутствию обхода.

Интерфейс Direct Routing для маршрутизации между контроллером границ сеанса и процессором облачного мультимедиа (без обобщения), а также между клиентом Teams и SBC (если функция пропуска мультимедиа включена) может использовать следующие кодеки:

- Обойтись без мультимедиа (SBC в обработчик облачных медиа-файлов): SILK, G. 711, G. 722, G. 729
- Обход мультимедиа (SBC с клиентом Teams): SILK, G. 711, G. 722, G. 729

Вы можете принудительно использовать конкретный кодек на контроллере границ сеанса, исключив нежелательные кодеки из предложения.

### <a name="leg-between-microsoft-teams-client-and-cloud-media-processor"></a>Связь между клиентом Microsoft Teams и процессором облачного мультимедиа
Применимо только к случаям пропуска без мультимедиа. При обходе мультимедиа мультимедиа перетекает прямо между клиентом Teams и SBC.

На стороне между облачным процессором мультимедиа и Microsoft Teams Client используется либо SILK, либо G. 722. Выбор кодека в этой области основывается на алгоритмах Microsoft, которые принимают в виду несколько параметров.
