# <a name="plan-direct-routing"></a>Планирование прямой маршрутизации

> [!Tip]
> Просмотрите следующие сеансы, чтобы узнать о прямой маршрутизации, планировании и его развертывании: [прямая маршрутизация в Microsoft Teams](https://aka.ms/teams-direct-routing)

Прямая маршрутизация Microsoft Phone System позволяет подключить поддерживаемый контроллер сеанса с предоставленным заказом (SBC) к телефонной системе Майкрософт.  Например, с помощью этой возможности можно настроить локальный локальный клиент Microsoft Teams, как показано на схеме ниже. 

![Схема с конфигурацией локального подключения через ТСОП](media/PlanDirectRouting1-PSTNwithTeams.png "Настройка локального подключения через ТСОП с клиентом Microsoft Teams")

  > [!NOTE]
  > Skype для бизнеса Online также позволяет связывать клиентские SBC- но для этого требуется локальное развертывание Skype для бизнеса Server или специальную версию Skype для бизнеса , называемый Cloud Connector в между SBC и Microsoft Cloud. Этот сценарий называется гибридной голосовой связью. В строгой маршрутизации прямая маршрутизация позволяет прямое подключение между поддерживаемым SBC и Microsoft Cloud. 

При прямой маршрутизации SBC можно подключить SBC к любому телефонную линию или связаться с сторонним оборудованием ТСОП. Прямая маршрутизация позволяет: 

- Используйте практически любую телефонную тсоп с телефонной системой Майкрософт. 
- Настройте взаимодействие между телегами оборудования, например сторонней частной филиатурной биржа (УАТС), устройстваааа аналогии и телефонной системы Майкрософт.

Майкрософт также предлагает все решения для голосовой связи, например план звонков. Однако рекомендуется использовать гибридное решение для голосовой связи, если: 

- План звонков (Майкрософт) недоступен в вашей стране. 
- Организации требуется подключение к сторонним устройствам аудиоконференций, центрам звонков и т. д. 
- В вашей организации существует контракт с оператором ТСОП.

Прямая маршрутизация также поддерживает пользователей с дополнительной лицензией для плана звонков (Майкрософт). Дополнительные сведения см. в [разделе "Телефонная система" и "Планы звонков".](calling-plan-landing-page.md) 

При прямой маршрутизации пользователи, находящиеся на запланированной конференции, номер для телефонного доступа предоставляется службой аудиоконференций Майкрософт, для которой требуется надлежащее лицензирование.  При наборе номера служба аудиоконференций Майкрософт превращает звонок с помощью возможностей звонков через Интернет, для которого требуется надлежащее лицензирование. (Примечание. Если у пользователя нет лицензии на аудиоконференции (Майкрософт), маршруты звонков выполняются через прямой маршрутизацию.) Дополнительные сведения см. в статье ["Собрания по сети с Teams".](https://products.office.com/microsoft-teams/online-meeting-solutions)

## <a name="licensing-and-other-requirements"></a>Лицензирование и другие требования 

Пользователям прямой маршрутизации должны быть назначены следующие лицензии в Microsoft 365 или Office 365: 

- Microsoft Phone System. 
- Microsoft Teams + Skype для бизнеса (план 2) при включении в лицензирование.
- Аудиоконференции Майкрософт (см. примечания и абзацы ниже).

> [!NOTE]
> План Skype для бизнеса не следует удалять из какого-либо лицензионного соглашения, в котором он входит. 


> [!IMPORTANT]
>  Если вы хотите добавить внешних участников на запланированные собрания, не обращаясь к ним или с помощью номера телефонного подключения, необходима лицензия на аудиоконференцию.


### <a name="ad-hoc-call-escalation-and-audio-conferencing-license"></a>Лицензии на голосовые вызовы и аудиоконференции

Пользователь Teams может запустить личную команду в тсоп или Teams для вызова и добавить в него участника тсоп. Этот сценарий называется самой сотрудничеством. Путь, занимаемый звонком, зависит от того, назначается ли ему лицензия на аудиоконференцию Майкрософт.

- Если у пользователя Teams, который прошел звонок, назначена лицензия на аудиоконференцию Майкрософт, то в процессе происходит проверка через службу аудиоконференций Майкрософт. Удаленный участник ТСОП, приглашенный к существующему звонку, получает уведомление о входящих вызовах и количество моста Microsoft, назначенного пользователю Teams, который начал эксплуатацию.
- Если у пользователя Teams, который проводит звонок, не назначена лицензия на аудиоконференции (Майкрософт) для выполнения этого действия выполняется с помощью контроллера границ, подключенного к интерфейсу Direct Routing. Удаленный участник ТСОП, приглашенный к звонку, получает уведомление о входящих вызовах и количество пользователя Teams, начатого в эксплуатации. Конкретный sBC, используемый для проверки, определяется политикой маршрутизации пользователя. 


Кроме того, необходимо проверить следующее:
 
- Пользователю назначается CsOnlineVoiceRoutingPolicy. 
- Разрешить частные вызовы на уровне клиента Microsoft Teams. 

Прямая маршрутизация также поддерживает пользователей, лицензированные на план звонков Майкрософт. Телефонная система Майкрософт с планом звонков может перенаправлять некоторые звонки с помощью интерфейса маршрутизации. Однако номера телефонов пользователей могут быть либо приобретены в Интернете или в переносимые в Майкрософт.  

Смешанная процедура подключения к плану звонков и direct Routing для одного и того же пользователя необязательно, но может означаться полезным (например, когда ему назначен план звонков Майкрософт, но хочет маршрутизировать некоторые звонки с помощью SBC). Один из наиболее распространенных сценариев звонки состоит из сторонних поставщиков УАТС.  При звонках сторонних УАТС всех звонках, кроме телефонов, подключенных к такой УАТС, звонки с помощью плана звонков Майкрософт на телефоны, подключенные к сторонним поставщикам УАТС, должны проходить через SBC, поэтому не отключайте трафик, а не транспортные сети. 

Дополнительные сведения о лицензировании телефонной системы см. в разделе ["Все возможности Office](https://products.office.com/compare-all-microsoft-office-products?tab=2) и [плана".](https://technet.microsoft.com/library/office-365-plan-options.aspx) 

Дополнительные сведения о лицензировании телефонной системы см. в разделе ["Лицензирование надстроек Microsoft Teams".](https://docs.microsoft.com/microsoftteams/teams-add-on-licensing/microsoft-teams-add-on-licensing)

## <a name="sbc-domain-names"></a>Имена доменов SBC

Доменное имя SBC должно состоять из одного из имен, зарегистрированных в доменах клиента. Для имени \* полного доменного имени SBC нельзя использовать клиент .onmicrosoft.com.

В следующей таблице приведены примеры имен DNS, зарегистрированных для клиента, сведения о его использовании в качестве полного доменного имени для SBC, а также примеры допустимых полных доменных имен:

|имя DNS|Может использоваться для полного полного имени SBC|Примеры имен FQDN|
|:--- |:--- |:--- |
contoso.com|Да|**Допустимые имена:**<br/>sbc1.contoso.com<br/>ssbcs15.contoso.com<br/>europe.contoso.com|
|contoso.onmicrosoft.com|Нет|Домены *.onmicrosoft.com не поддерживаются для имен SBC

Предположим, вы хотите использовать новое доменное имя. Например, у вашего клиента contoso.com как доменное имя, зарегистрированное в вашем клиенте, и вы хотите использовать sbc1.sip.contoso.com. Чтобы связать SBC с именем sbc1.sip.contoso.com необходимо зарегистрировать доменное имя sip.contoso.com доменав в вашем клиенте. Если перед регистрацией доменного имени sbc1.sip.contoso.com перед регистрацией доменного имени, появится следующая ошибка: "Не удается использовать доменsbc1.sip.contoso.com, так как он не настроен для данного клиента".
После добавления доменного имени вам также нужно создать пользователя с именем участника-user@sip.contoso.com и назначить лицензию Teams. Полное доменное имя после его добавления в домены вашего клиента, создается пользователь с новым именем и ему назначена лицензия. 

Возможно, в одном клиенте у компании может быть несколько пробелов SIP. Например, у компании может быть местной contoso.com адреса SIP, а fabrikam.com второе пространство SIP. У некоторых пользователей есть адрес user@contoso.com, а некоторые — user@fabrikam.com. 

SBC требуется только одно полное имя ("Полное имя_клиента") и может обслуживать пользователей службы из любого адресного пространства в парного клиенте. Например, SBC с именем sbc1.contoso.com может получить и отправить трафик ТСОП пользователям с адресами user@contoso.com и user@fabrikam.com, если эти адреса SIP зарегистрированы в одном клиенте.  

## <a name="public-trusted-certificate-for-the-sbc"></a>Открытый доверенный сертификат для SBC

Корпорация Майкрософт рекомендует запрашивать сертификат для SBC, создав запрос подписи сертификации (CSR). Инструкции по созданию CSR для SBC см. в инструкциях по взаимодействию с поставщиками SBC. 

  > [!NOTE]
  > Для большинства центров сертификации требуется размер закрытого ключа не менее 2048. Помните об этом при создании CSR.

Полное доменное имя SBC должно содержать полное доменное имя SBC в качестве общего имени (CN) в поле темы.

Прямая маршрутизация поддерживает подстановочные знаки в САНд-точке, и подстановочные знаки должны соответствовать [стандартному rFC HTTP по протоколу TLS.](https://tools.ietf.org/html/rfc2818#section-3.1) В \* сергеях будет использоваться .contoso.com, который будет соответствовать полному имени SBC по sbc.contoso.com, но не соответствует sbc.test.contoso.com.

### <a name="leg-between-sbc-and-cloud-media-processor-or-microsoft-teams-client"></a>Легкий между облачным процессором Облачного процессора или клиентом Microsoft Teams.
Применяется как к обоузжающим делам, так и к обоузревателю.

Интерфейс Direct Routing в легенде между контроллером облачного процессора облачного процессора (без мультимедиа) или между клиентом Teams и SBC (если включено мультимедиа) могут использовать следующие кодеки:
