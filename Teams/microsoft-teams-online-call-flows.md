# <a name="microsoft-teams-online-call-flows"></a>Команды Microsoft Online вызова денежных средств

## <a name="overview"></a>Обзор
В этом документе описывается, как рабочая нагрузка групп использует денежных средств Office 365 в различных топологиях. Кроме того задает уникальный потоки команд, которые используются для обмена данными мультимедиа для одноранговых. Документ перечисляет эти потоки и описывает их назначение и их происхождения/завершение сетями. Например поток X используется клиентом Office 365 локально для взаимодействия со службой Office 365 в облаке, исходит от сети клиента и прервано конечную точку в облаке Office 365 и поток Y используется клиентом Office 365 локально для взаимодействия с службы в Интернете, с Office 365 зависимость на, исходит от сети клиента и завершается с конечной точки в сети Интернет.

В документе есть три основных раздела. Первая предоставляет базовые сведения, например сетей (, которые может пересекать денежных средств Office 365), тип трафика, подключения к руководство по от сети клиента для конечных точек службы Office 365, взаимодействие с компонентами сторонних производителей и участников, используются только группами для выбора потоков мультимедиа. Второй показано использование этих потоков в различных топологиях. Для каждой топологии его перечисляет все поддерживаемые денежных средств и показано, как использовать эти потоки через несколько вариантов использования. Для каждого варианта использования описывается последовательность и выбор потоков с помощью схемы потока. Третий описывается, как эти потоки, используются при развертывании, экспресс-выпуск маршрут для оптимизации, показано через простая топология.

## <a name="background"></a>Фон
### <a name="network-segments"></a>Сетевые сегменты
**Сеть клиента**: это сегменте сети, который является частью общей сети, управления и управление ими. Этот компонент включает все подключения клиента в офисах клиента проводной или беспроводной связи между зданий для локальных центров обработки данных и подключения к Интернет поставщиков, экспресс-выпуск маршрут или любой другой частной авторами. 

Как правило сеть клиента имеет несколько периметра сети с помощью брандмауэров и прокси-серверы, которой принудительное применение политик безопасности организации и, разрешить только определенные сетевого трафика, который был назначен и настроен (или). Так как клиент управляет этой сети, клиент имеет непосредственно контролировать производительность сети и настоятельно рекомендуется, что полный клиентов сети оценки для проверки производительности обоих сайтах, в сети и от вашей сеть в сеть Office 365. Скайп для бизнеса на локальном развертывании и пограничного контроллера сеансов ТСОП для подключения по локальной сети (то есть прямой маршрутизации) PSTN являются необязательными.

**Интернет**: это сегменте сети, который является частью общей сети, которая будет использоваться пользователями, подключение к Office 365 облака из за пределами сети клиента. Он также используется некоторые трафика из сети клиента Office 365 облако. 

**Просмотренная/гостевой частную сеть**: это сегменте сети за пределами сети клиента, но не в общедоступную сеть Интернет, пользователей и/или их гости могут посетить. К примеру Домашняя страница частной сети или частной сети предприятия, не развертываются группы, которых может расположены пользователей и/или клиентов, которые взаимодействуют с группами служб.

>**Примечание**: подключение к Office 365b, также применимы к этим сетям.

**Office 365 облаке**: это сегменте сети, поддерживающий служб Office 365. Он входит в состав во всем мире края близости к сети клиента в большинстве расположения. Функции, упомянутые в этом документе относятся ретрансляции транспорта, сервер конференций и мультимедиа процессора. 

**Маршрут, экспресс-выпуск (необязательно)**: это сегменте сети, который является частью общей сети, предоставление выделенного закрытого подключения к сети Office 365.

### <a name="types-of-traffic"></a>Типы трафика

**В режиме реального времени мультимедиа**: данные в RTP (в режиме реального времени транспортный протокол) и поддерживает аудио-, видео и рабочих нагрузок совместного использования экрана. В общем случае трафика мультимедиа настоятельно — задержка регистр, поэтому требуется трафик в принимать наиболее коротких путь возможные и использовать UDP и TCP как транспортный протокол, который является наиболее передачи мультимедиа интерактивных режиме реального времени с точки зрения качества. (Примечание: в качестве последнего средства мультимедиа можно использовать TCP/IP и быть переданы в HTTP-протокол, но не рекомендуется из-за последствия качество.) Поток RTP защищен с помощью SRTP, в котором шифруются только полезных данных.

**Передача сигналов**: связи между клиента и сервера или других клиентов, которые используются для управления действия (например, звонок будет запущена) и отправки мгновенных сообщений. Наиболее передачи трафика с помощью интерфейса REST на основе HTTPS, однако в некоторых случаях (например, соединение между Office 365 облако и пограничного контроллера сеансов) использует протокол SIP. Важно понимать, что этот трафик существенно менее зависит от задержки, но может приводить к сбоям службы или вызвать времени ожидания, если задержка между конечными точками превышает несколько секунд. 

### <a name="connectivity-to-office-365"></a>Подключение к Office 365

Службы групп необходимо [подключение к Интернету](https://support.office.com/article/connectivity-to-the-internet-64b420ef-0218-48f6-8a34-74bb27633b10). Группы конечных точек, URL-адреса и IP-адрес диапазоны, перечислены в [Office 365 URL-адреса и диапазоны IP-адресов](https://docs.microsoft.com/office365/enterprise/urls-and-ip-address-ranges). (Примечание: требуется открыть подключение к TCP-порты 80 и 443 и UDP-порты 3478 до 3481.) Кроме того службы группы обладают зависимость Скайп для бизнеса веб-службами, поэтому необходимые для подключения также этой службы в Интернете.

Подключения к потоков мультимедиа группами реализована с помощью стандартных процедур IETF ICE (интерактивная Connectivity Establishment).

### <a name="interoperability-restrictions"></a>Ограничения взаимодействия
**Ретранслирует сторонних производителей мультимедиа**: потока мультимедиа групп (то есть, одной из конечных точек мультимедиа — группам) может пересекать только группы или Скайп для реле собственный мультимедиа Business. Взаимодействие с ретрансляция мультимедиа третья сторона не поддерживается. (Примечание: третья сторона SBC на границе с ТСОП необходимо завершить RTP/RTCP потока, защищенным с помощью SRTP и не пересылать до следующего прыжка.)

**Третья сторона SIP прокси-серверы**: группами сигналы SIP диалоговое окно с третьей стороной SBC и/или шлюз может пересекать группы или Скайп для бизнеса собственный прокси SIP. Взаимодействие с третьей стороной SIP прокси-сервер не поддерживается.

**B2BUA сторонних производителей (то есть, SBC)**: группами потока мультимедиа из/ТСОП разрывается третьей стороной SBC. Тем не менее, взаимодействие с третьей стороне SBC в сети группы (например, третья сторона SBC передает две конечные точки команды/Скайп для бизнеса) не поддерживается.

### <a name="technologies-that-are-strongly-not-recommended-with-microsoft-teams"></a>Технологии, которые не настоятельно рекомендуется с группами Майкрософт

**Сети VPN**: не настоятельно рекомендуется для трафика мультимедиа (то есть, поток 2"). VPN-клиент следует использовать разбиение VPN и маршрута трафика мультимедиа как любой пользователь внешних нет, как указано вhttps://blogs.technet.microsoft.com/nexthop/2011/11/14/enabling-lync-media-to-bypass-a-vpn-tunnel/ 

>**Примечание**: хотя заголовок Lync, его можно применять к группам также.

**Пакет Shapers**: Настоятельно не рекомендуется и может привести к снижению качества значительно любого вида snippers пакетов, проверки сетевых пакетов и пакетов формирователе устройства. 

### <a name="principles"></a>Принципы
Существует четыре общие принципы, которые помогут вам понять потоков обработки вызовов для групп Майкрософт. 
1.  Office 365 облака в той же области, где первый участник присоединился к размещается конференции группами Майкрософт. (Примечание: Если будет исключения к этому правилу в некоторых топологий, то они будут описаны в данном документе, поток соответствующий звонок, как показано.)
2.  Команды мультимедиа конечной точки (MP) в Office 365 в облаке используется на основании требований с мультимедиа и не зависит от типа вызова. (Например, звонок точка-точка может используется конечной точки мультимедиа в облаке, чтобы процесс мультимедиа для транскрибирования и/или запись во время конференции с двумя участниками нельзя использовать любой конечной точки мультимедиа в облаке.) Тем не менее большинство конференции будет использоваться MP для совместное использование и маршрутизации, выделенное размещение конференции. Мультимедиа трафика от клиента MP может перенаправляться непосредственно или используйте ретранслятора транспорта в облаке Office 365, при необходимости из-за ограничения брандмауэра сеть клиента. 
3.  Для вызовов peer-to-peer занять наиболее коротких маршрутов, который доступен, если предполагается, что вызов не требует MP в облаке трафика мультимедиа (см #2 выше). Предпочитаемый маршрута напрямую в удаленный узел (клиент), но если маршрута не поддерживается, один или несколько реле транспорта будет ретрансляция трафика. Рекомендуется, что трафика мультимедиа должна не поворотом серверы как shapers пакетов, VPN-серверы, и др., так как это будет влиять на качество мультимедиа.
4.  Сигналы трафик всегда переходит к серверу ближайшим к пользователю.

#### <a name="use-case-one-to-one"></a>Вариант использования: один к одному
Один к одному вызовов использоваться общая модель, вызывающего будут получать набор кандидатов, состоящий из IP-адреса/портам; локальный, ретрансляции и рефлексивное (общедоступный IP-адрес клиента, видимых ретрансляции). Звонящий отправляет эти кандидатов вызываемый абонент в приглашении на, вызываемый абонент также получает следующий набор кандидатов и отправляет их вызывающего абонента. Проверка возможности подключения STUN сообщений используется для поиска каких вызывающего/называется стороны мультимедиа на работу путей и пути к наиболее рабочей будет установлен. Мультимедиа (то есть RTP/RTCP пакетов, защищенных с помощью SRTP), затем передаются с помощью пары выбранного кандидата. Ретрансляция транспорта будет развернут как часть облака Office 365.

Если локальный IP-адрес/порт кандидатов или рефлексивное кандидатов подключены к, а затем прямого пути между клиентами (или с помощью NAT) будет выбран для мультимедиа. Если клиенты в сети клиента, должен быть установлен прямого пути. Для этого необходимо прямое подключение UDP в сети клиента. Если оба пользователя nomadic облаке клиентами, затем в зависимости преобразования сетевых адресов или брандмауэра, мультимедиа могут использовать прямое подключение.

Если один клиент внутренних клиентов сети, и один клиент external (например, мобильных облачных пользователей), а затем его вряд ли работы прямое подключение между локальной или рефлексивное кандидатов. В этом случае параметр — это один из кандидатов транспорта ретрансляции с любого из клиента (например, внутренних клиентов кандидата ретрансляции полученных при ретрансляции транспорта в облаке Office 365, внешний клиент должен иметь возможность отправлять пакеты STUN/RTP/RTCP Ретрансляция транспорта). Другая возможность — внутренний клиент отправляет ретрансляции кандидата, полученные клиентом мобильных облака. Обратите внимание на то, что несмотря на то, что настоятельно рекомендуется UDP-ПОРТ подключения для мультимедиа, TCP поддерживается.

**Высокоуровневые действия**:
1. Групп пользователей A разрешает URL-адрес доменное имя (DNS) с помощью flow2 следует
2. Группы пользователей A выделяет мультимедиа порт ретрансляции на группы транспорта ретрансляции с помощью поток 4
3. Групп пользователей A отправляет «пригласить» с кандидатов ICE через поток 4 в Office 365
4. OFFICE 365 отправляет уведомление группами пользователей б, поток 4
5. Группы пользователей B выделяет мультимедиа порт ретрансляции на группы транспорта ретрансляции с помощью поток 4
6. Группы пользователей B отправляет «ответ» с кандидатов ICE через поток 4, будет пропущена группы пользователя A через поток 4
7. Групп пользователей A и B группами пользователей вызывать тесты подключения ICE и выбран наиболее доступные пути (см. диаграммы ниже для различных вариантов использования)
8. Группы пользователей отправлять телеметрии в Office 365 с помощью поток 4

**Высокоуровневые действия:**
1. Групп пользователей в сети клиента разрешает URL-адрес доменное имя (DNS) с помощью flow2 следует
2. Групп пользователей в сети клиента выделяет мультимедиа порт ретрансляции на группы транспорта ретрансляции с помощью потока 1
3. Групп пользователей в сети клиента отправляет «пригласить» кандидаты ICE через поток 1 в Office 365
4. OFFICE 365 отправляет уведомления для внешних пользователей группам с помощью поток 3
5. Внешних пользователей группам выделяет мультимедиа порт ретрансляции на группы транспорта ретрансляции с помощью поток 3
6. Внешних пользователей группам отправляет «ответ» с кандидатов ICE через поток 3, будет пропущена группы пользователя A через поток 1
7. Группы пользователей A и B группам пользователей вызов тесты подключения ICE и выбирает потоки 1 и 3, передаются по группам ретрансляции транспорта в облаке Office 365.
8. Группы пользователей отправлять телеметрии в Office 365 с помощью потоков 1 и 3
