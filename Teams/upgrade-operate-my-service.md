# <a name="operate-your-service"></a>Использование службы

![Схема пути обновления с акцентом на этапе эксплуатации](media/upgrade-banner-op-excellence.png "Этапы пути обновления с акцентом на этапе "Рабочие связи"")

Эта статья является частью этапа обновления с Skype для бизнеса до Teams.

В этой статье дается обзор требований для успешной работы Teams в организации после обновления. Правильное обслуживание служб Teams обеспечивает надежную и качественную работу вашей организации.

## <a name="introduction-to-the-operations-guide"></a>Знакомство с руководством по работе

В руководстве по эксплуатации представлен обзор всех задач и действий, необходимых в функции управления службами Для Microsoft Teams.

Управление службами — это обширная тема, которая охватывает повседневные операции службы Microsoft Teams после ее развертывания и предоставления пользователям. Служба Teams включает в себя Microsoft 365 или Office 365 и компоненты инфраструктуры, которые развертываются локально (например, сеть).

Идея управления службами, скорее всего, не нова для большинства организаций. Возможно, вы уже реализовали процессы и задачи, связанные с существующими службами. При этом, возможно, вы сможете дополнить текущие процессы, если запланируете управление службами уже сегодня для поддержки Teams в будущем.

Управление службами включает в себя все действия и процессы, связанные с управлением Teams. Как было отмечено ранее, некоторые компоненты управления службами — инфраструктура Microsoft 365 или сама служба Office 365 — лежит на корпорации Майкрософт, в то время как клиент несет ответственность перед вашими пользователями за управление различными аспектами Teams, сети и конечных точек, которые вы предоставляете.

### <a name="operational-role-mapping"></a>Сопоставление операционных ролей

Планирование операций на этапе "Создание проекта" крайне важно, так как действия по эксплуатации начинаются после включения первых пилотных пользователей. В этом руководстве перечислены действия и задачи, которые необходимо выполнять ежедневно, еженедельно, ежемесячно или по мере необходимости для качественного развертывания Teams. Это руководство содержит знания и рекомендации по выполнению этих критических задач и задач.

Один из важнейших компонентов успешного развертывания — обеспечить, чтобы планирование на ранней стадии "Определение системы" включало определение ответственных лиц за выполнение определенных действий. После того как вы выясните, какие задачи и действия относятся к развертыванию, они должны быть понятны и должны следовать группам или отдельным лицам, которые вы им назначаете.

Каждой определяемой группе необходимо и проанализировать и согласовать выявленные задачи и обязанности, а также приступить к подготовке. Это может быть обучение и подготовка, обновление плана кадров или проверка готовности внешних поставщиков к доставке.

Действия и роли, определенные в этом руководстве, должны быть действительны в большинстве сценариев, но каждое развертывание Teams уникально. Таким образом, это руководство можно использовать в качестве отправной точки для настройки действий и ролей по умолчанию в нужном вам случае.

Убедитесь, что у каждой подотчетной группы есть понятное представление о действиях, необходимых для запуска службы. Очень важно, чтобы каждая команда принимает и отключается от ответственности в вашей организации до первого пилотного проекта.

### <a name="teams-service-dependencies"></a>Зависимости служб Teams

Microsoft Teams объединяет технологии в Microsoft 365 и Office 365, чтобы предоставить центр для командной работы. Примерами могут быть следующие:

- Azure Active Directory (Azure AD) предоставляет службы проверки подлинности и авторизации для Teams.

- Exchange Online предоставляет дополнительные функции, такие как удержание по юридическим и электронным данным.

- SharePoint Online позволяет делиться файлами в каналах, а OneDrive для бизнеса предоставляет механизм для общего доступа к файлам в приватном чате.

Организации также могут использовать существующие вложения в локальной инфраструктуре. Например, существующие локально учетные записи Active Directory можно использовать для проверки подлинности с помощью Azure AD Connect. Некоторые версии Exchange Server можно использовать на месте Exchange Online.

Эти технологии предоставляют пользователям богатый набор средств для совместной работы и интеллектуальных средств связи. Такая тесная интеграция является ключевым преимуществом Teams, но она также является требованием к управлению службами в этих технологиях.

В этом руководстве огося ключевых областей, которые необходимо уделять управлению службой Teams. Скорее всего, у вас есть планы управления обслуживанием для вспомогательных технологий, от которых зависит Teams. Если нет, вам потребуется создать правильные планы управления службами для этих технологических компонентов (как локально, так и в Интернете). Это поможет вашим пользователям работать с Teams как можно более качественно и надежно.

## <a name="monitor-service-health"></a>Отслеживание состояния службы

Важно понимать общее состояние службы Microsoft Teams, чтобы заблаговременно оповещать других людей в организации о любых событиях, влияющих на службу. Как было описано ранее, Teams зависит от других служб Microsoft 365 и Office 365, таких как Azure Active Directory, Exchange Online, SharePoint Online и OneDrive для бизнеса. Поэтому не менее важно отслеживать состояние зависимых служб.

Включите это действие в процесс управления инцидентами, чтобы заблаговременно сообщать пользователям, службе справки и вашим оперативным группам о том, как справиться с перенаправлением инцидентов.

В следующих разделах описаны средства, которые можно использовать для отслеживания инцидентов [обслуживания,](https://technet.microsoft.com/library/office-365-service-health.aspx#Anchor_1) влияющих на службу Teams. В таблицу ниже включены общие сведения о преимуществах каждого инструмента и о том, когда следует использовать каждый из них.

| Средство мониторинга | Преимущества | Когда использовать |
|---|---|---|
| Центр администрирования Microsoft 365 | Доступно на любом устройстве с поддерживаемой браузером. | Используется в том случае, если вам не нужны уведомления в режиме реального времени. |
| Приложение для администрирования Microsoft 365 | Предоставляет push-уведомления на ваше мобильное устройство. | Используется для уведомлений об инцидентах обслуживания, когда вы в путь. |
| Microsoft System Center | Интеграция с Microsoft System Center. | Используется, если вам нужны расширенные возможности мониторинга и поддержка уведомлений. |
| Microsoft 365 Service Communications API | Программный доступ к microsoft 365 или службе Office 365. | Используется, когда вам нужна интеграция со сторонним средством мониторинга или вы хотите создать собственное решение. |

> [!NOTE]
> Только люди, которым назначена  роль **глобального** администратора или администратора службы, могут просматривать состояние службы.

### <a name="monitoring-with-the-microsoft-365-admin-center"></a>Мониторинг в Центре администрирования Microsoft 365

В [Центре администрирования Microsoft 365](https://portal.office.com/) есть панель мониторинга "Состояние службы", на которой помимо зависимых служб можно просмотреть текущее состояние службы Teams. [](https://portal.office.com/adminportal/home#/servicehealth)

### <a name="monitoring-with-the-mobile-app"></a>Мониторинг с помощью мобильного приложения

Приложение "Администратор Microsoft 365" доступно на устройствах Apple iOS, Android и Windows (пк и мобильном устройстве). Приложение поддерживает push-уведомления, которые могут оповещать вас практически сразу после публикации рекомендации. Это поможет вам быть в курсе состояния, состояния и предстоящих изменений в службе. Поддержка уведомлений делает его рекомендуемой средством мониторинга для администраторов. ### <a name="monitoring-with-microsoft-system-center"></a>Мониторинг с помощью Microsoft System Center

Microsoft System Center — это интегрированная платформа управления, которая помогает управлять центрами обработки данных, клиентских устройств и гибридными облачными ИТ-средами. Администраторы Microsoft 365 и Office 365, которые используют System Center, теперь могут импортировать пакет управления, позволяющий просматривать все сообщения службы в Operations Manager в System Center. С помощью этого средства можно получить доступ к статусу служб, на которые вы подписаны, активным и разрешенным инцидентам обслуживания, а также к информационному сообщению Центра сообщений (предстоящие изменения). Дополнительные сведения можно найти в следующей [записи блога.](https://blogs.office.com/2014/07/29/new-office-365-admin-tools/?eu=true)

Если вы работаете с System Center для отслеживания состояния служб Teams (и зависимых служб), вы можете дополнительно настроить пакет управления, чтобы оповещать или уведомлять определенные группы или отдельных пользователей о реагировании на инциденты.
Эти группы могут включать владельцев служб, службы технической поддержки, группы поддержки второго и третьего уровней, а также руководителей инцидентов в вашей организации.

## <a name="manage-organizational-change"></a>Управление изменениями в организации

Microsoft Teams — это облачная служба. Благодаря этому вы быстро предоставляете новые функции и возможности. Постоянное внесение изменений обеспечивает явное преимущество для организаций, но этими изменениями необходимо управлять в организации, чтобы избежать сопротивление пользователей или повышения их цен в вашей справке.

Обновления для Teams автоматически 2010 г. 2016 для ваших пользователей. У ваших пользователей всегда будут последние версии клиента и функции, доступные в службе Teams. Управлять внедрением обновлений Teams для пользователей невозможно, поэтому управление изменениями путем эффективного взаимодействия, обучения и внедрения программ крайне важно. Если пользователи знают об этих изменениях, могут уделить внимание преимуществам и использовать новые возможности, которые они смогут быстрее адаптировать, и поприветствовать &mdash; изменения.

### <a name="monitoring-for-change"></a>Мониторинг изменений

Первым шагом в управлении изменениями является отслеживание изменений, запланированных в Teams. Для мониторинга изменений лучше всего следить за изменениями в [стратегии развития Microsoft 365,](https://www.microsoft.com/microsoft-365/roadmap)в которой перечислены функции, которые в настоящее время находятся в разработке, перечислены для клиентов или полностью запущены. Вы можете искать функции для Teams с помощью предоставленного фильтра или скачать карту развития в файл Excel для дальнейшего анализа. Для каждой функции в кратком описании с предполагаемой датой выпуска приводится краткое описание.

В блоге [Microsoft Teams вы](https://techcommunity.microsoft.com/t5/Microsoft-Teams-Blog/bg-p/MicrosoftTeamsBlog)можете узнать о лучших методиках, тенденциях и новости об обновлениях продуктов Teams. Будьте в курсе всех важных обновлений функций, которые вы можете озвучить здесь. Вы также можете подписаться на блог с помощью RSS-канала. После этого вы можете [добавить RSS-канал](https://techcommunity.microsoft.com/gxcuf89792/rss/board?board.id=MicrosoftTeamsBlog) непосредственно в канал Teams, чтобы все важные новости доставлялись непосредственно в Teams.

Все выпущенные функции описаны в заметках [о выпуске Microsoft Teams.](https://support.office.com/article/Release-notes-for-Microsoft-Teams-d7092a6d-c896-424c-b362-a472d5f105de)
Здесь вы найдете список функций, которые были выпущены для настольных компьютеров, браузера и мобильных устройств. Этот же набор заметок о выпуске также доступен на вкладке **"Что нового"** в [справке.](get-help-in-microsoft-teams.md)

Ознакомьтесь с доступными ресурсами и убедитесь, что вы назначили соответствующих владельцев для отслеживания изменений.

### <a name="planning-for-change"></a>Планирование изменений

Теперь, когда вы знаете о предстоящих изменениях в службе Teams, следующий этап — подготовиться и спланировать их соответствующим образом. Оцените каждое изменение, чтобы определить, какие изменения требуют взаимодействия с пользователями, информационных кампаний, обучения для групп поддержки или пользователей, а также кампаний по оценке и внедрению функций. ## <a name="assess-teams-usage"></a>Оценка использования Teams

После начала начального пилотного проекта крайне важно определить регулярный курс для оценки фактического использования Teams. Это позволяет вашей организации получить представление о том, насколько фактическое использование соответствует использованию, прогнозируемом на этапе "Представление". Хотя в этом разделе основное внимание уделяется использованию Teams, он должен быть частью более широкого усилия по оценке и оценке использования Microsoft 365 или Office 365 в целом.

При частом просмотре использования в начале развертывания вы можете:

- Проверьте, используют ли пользователи Teams.

- Определите потенциальные проблемы с внедрением, прежде чем они будут создавать критические проблемы в организации.

- Понять, есть ли расхождения между требованиями к этапу "Проект" и фактическим использованием.

Если использование не то, что вы ожидаете, это может быть вызвано проблемой развертывания, неправильной выполнением плана принятия или другой проблемой. В зависимости от фактической причины низкого использования администратор службы должен сотрудничать со связанными командами, чтобы устранить барьеры.

Два других отчета являются конкретными для Teams и предоставляют более подробные данные об использовании Teams с точки зрения пользователя и устройства. Оба отчета можно найти здесь:

[Отчет об использовании устройств с Microsoft Teams](https://support.office.com/article/Office-365-Reports-in-the-Admin-Center-Microsoft-Teams-device-usage-917b3e1d-203e-4439-8539-634e80196687)

## <a name="assess-user-sentiment"></a>Оценка тона работы пользователя

Понимание тональности пользователя может действовать в качестве ключевого индикатора для успешного развертывания Teams. Отзывы пользователей могут вносить изменения в вашу организацию; это может включать изменения в ваших планах коммуникаций, учебных программах или способах поддержки пользователей.

Очень важно заранее получить отзыв и продолжить оценку тона работы пользователей на протяжении всего жизненного цикла проекта и после него. Воспользуйтесь следующим руководством, чтобы определить интервал, в который ваша организация будет обращаться к отзыву.

- **Начало проекта.** Оценив тональность пользователей в начале проекта, вы можете получить представление о том, как пользователи подумаете, как они работают в Teams.

- **После важных** вех: сбор отзывов в течение жизненного цикла проекта может постоянно оценивать тональность пользователей и вносить необходимые изменения. Это особенно полезно после важных вех.

- **Заключение проекта.** Оценка тона работы пользователей в конце проекта показывает, насколько хорошо вы сделали и где все еще нужно сделать работу, и позволяет сравнить результаты с результатами предыдущего опроса.

- **Текущая:** продолжайте измерять тональность пользователя в течение неограниченного времени. Это может быть вызвано изменениями в среде организации или изменениями в службе Teams. Изумившись, вы можете через определенные промежутки времени понять, насколько хорошо работают команды управления службами и как организация отвечает на изменения в службе Teams.

Тональность пользователя можно оценить различными способами. К ним относятся опросы электронной почты, собеседования по телефону или собеседования по телефону либо просто создание канала отзывов в Teams или Yammer. Дополнительные сведения см. в методах обратной связи [пользователей в Microsoft Teams.](best-practices-feedback.md)

Для оценки тональности пользователей, называемой net promotor score (NPS), можно также использовать отраслевый подход, который описан в следующем разделе.

### <a name="nps"></a>NPS

Net promoter score (NPS) — это показатель отраслевых стандартов, который используется для оценки тона работы пользователей. Чтобы вычислить NPS, задав два вопроса: "Насколько вероятно, что вы порекомендуете Teams коллеге?", а затем — вопрос с бесплатной формы "Почему?"

NPS — это индекс в диапазоне от –100 до 100, который измеряет готовность клиента порекомендовать продукт или службу компании. NPS — это анонимный опрос, который проводится пользователям по электронной почте или другими электронными способами. NPS измеряет связь между поставщиком и потребителем. Она состоит только из одного вопроса, в котором пользователи могут оценить свои впечатления от 1 до 10 с возможностью предоставления дополнительных комментариев. После этого пользователи классифицируются на основе следующих оценок:

- 9 или 10 являются продвигателями: невероятные энтузиасты, которые будут продвигать вашу службу и задействовать другие.

- 7 или 8 являются пассивными: удовлетворенные, но невезимические, уязвимыми для другой службы или предложения.

- От 1 до 6 — это неравные клиенты: недовольные клиенты, которые могут нарушить службу и неохожим рост числа клиентов.

![Схема, демонстрируемая масштаб NPS](media/operate-my-service-image2.png "На этой схеме показано масштаб NPS. В нем показано, что ранжирование от 0 до 6 является детрактором, 7–8 — пассивным, а 9–10 — продвигателями.")

Хотя базовый номер NPS удобен, при анализе комментариев пользователей вы получите наибольшее значение. Они помогут вам понять, почему пользователь порекомендует Teams другим пользователям. Эти комментарии помогают группам управления проектами и службами понять, какие корректировки необходимы для обеспечения высокого качества обслуживания.

Для предоставления опросов NPS в организации можно использовать любимое средство опроса в Интернете.

## <a name="assess-and-ensure-quality"></a>Оценка и обеспечение качества

Во всех организациях должна быть группа или отдельные лица, которые должны быть подотчетны за качество. Это самая важная роль в управлении службами. Роль "Защитник качества" назначена человеку или группе, которые с энтузиазмом с энтузиазмом относят к опыту своих пользователей.
Для этой роли нужны навыки по выявлению тенденций в среде, а также стремление сотрудничать с другими командами для принятия корректирующих мер. Оптимальным кандидатом на роль лидера по качеству обычно является владелец службы клиента. В зависимости от размера и сложности организации это может быть любой человек или группа, у которых есть желание обеспечить высокое качество работы пользователей.

The quality champion leverages existing tools and documented processs, such as Call Quality Dashboard (CQD) and [Improve and monitor call quality for Teams,](monitor-call-quality-qos.md)to monitor user experience, identify quality trends, and drive remediation where needed.
Защитник качества должен вместе с соответствующими командами управлять действиями по исправлению проблем и сообщать о ходе их устранения в оправляющий комитет по поводу хода выполнения и любых открытых проблем.

[Улучшение и отслеживание качества](monitor-call-quality-qos.md) звонка для Teams включает в себя действия, которые оценивают и предоставляют рекомендации по исправлению в ключевых областях, самое сильное влияние на работу пользователей. В руководстве по проверке качества основное внимание уделяется использованию CQD Online в качестве основного средства для создания отчетов и изучения каждой области с акцентом на звуке для максимального внедрения и его влияния. Любые оптимизации для сети, направленные на улучшение звукового взаимодействия, также вносят вклад в улучшение видеосвязи и совместного доступа к рабочему столу.

Мы настоятельно рекомендуем заранее выдвигать в качестве защитника нужное время. После того как он будет назначен в качестве кандидата, он должен приступить к ознакомлению с содержимым в области "Улучшить качество содержимого" и отслеживать качество звонка для [Teams](monitor-call-quality-qos.md) и связанные учебные материалы.

## <a name="manage-endpoints"></a>Управление конечными точками

Конечные точки Microsoft Teams можно определить как любые ПК, Mac, планшеты, мобильные (или любые другие) устройства с клиентом Teams. Термин  "конечная точка" включает не только само устройство, но и способ подключения пользователя к устройству ( например, с помощью встроенного микрофона, динамика, наушников или оптимизированной гарнитуры). После развертывания конечные точки не должны быть забыты. Конечные точки Teams требуют постоянного обслуживания. В следующих разделах описаны конкретные области, на которые следует сосредоточиться.

### <a name="endpoint-requirements"></a>Требования к конечной точке

Одно из главных преимуществ Teams состоит в том, что клиент автоматически обновляется. Клиенты на компьютере и Mac обновляются с помощью фонового процесса, который проверяет наличие новых сборок и скачивает новый клиент при простое приложения. Мобильные приложения Teams хранятся в соответствующих магазинах приложений.

Клиент Teams имеет минимальные требования с точки зрения платформы программного обеспечения. Со временем эти требования могут измениться, поэтому важно следить за их изменениями. Например, у клиента Teams есть минимальная версия iOS. Если клиент использует веб-браузер, он также должен быть актуальным. Список поддерживаемых платформ можно найти в оке ["Получить клиенты" для Microsoft Teams.](get-clients.md)

### <a name="endpoint-firewalls"></a>Брандмауэры конечных точек

Клиентские брандмауэры могут оказывать значительное негативное влияние на взаимодействие с пользователем.
Брандмауэры на стороне клиента могут влиять на качество звонка и даже предотвращать переупомироание звонка. После настройки соответствующих исключений в брандмауэре клиента их необходимо будет настраивать на основе сведений в URL-адресах и диапазонах IP-адресов [Office 365.](https://aka.ms/o365ips) Для вашего стороного поставщика будут указаны инструкции по обновлению исключений.

### <a name="wi-fi-drivers"></a>Драйверы Wi-Fi

Wi-Fi драйверы могут привести к проблеме. Например, драйвер может использовать очень активную роуминг между точками доступа, что может привести к ненужным переключениям точках доступа, что может привести к плохому качеству связи. При проверке качества Wi-Fi качества обслуживания может быть обнаружен плохо выполняющий драйвер (дополнительные подробности см. в этом видео). [](monitor-call-quality-qos.md) Важно реализовать процесс на основе качества, который отслеживает новые Wi-Fi драйверы и гарантирует их тестирование перед развертыванием для всех пользователей.

### <a name="endpoint-management"></a>Управление конечными точками

Каталог поддерживаемых конечных точек и устройств интерфейса (например, гарнитур) должен быть доступен и поддерживаться. Этот каталог содержит список утвержденных устройств, которые были выбраны и проверены на этапе "Создание проекта" и "Доска". Как правило, для каждого типа пользователей в организации выбраны отдельные устройства, которые соответствуют его атрибутам. У всех конечных точек есть жизненный цикл, и вам необходимо управлять политиками поставщиков, гарантии, замены, распространения и ремонта, связанными с этими устройствами.

### <a name="endpoint-troubleshooting"></a>Устранение неполадок конечной точки

Даже если вы следовали предыдущим указаниям, у пользователей в вашей организации могут по-прежнему возникнуть проблемы с Teams. Хотя проблема может быть не в самой конечной точке, ее признаки обычно выназначаются клиентом пользователю. Ниже указаны общие действия по устранению проблемы. это не полное руководство по устранению неполадок. Действия предоставляются в определенном порядке, но они не должны следовать явным образом и могут быть не применимы в зависимости от характера проблемы.

1. **Проверка состояния службы.** Проблема, которая может возникнуть у пользователя, может быть связана с событием, которое отрицательно влияет на службу Teams или ее зависимые службы. В качестве первого шага рекомендуется подтвердить отсутствие активных проблем со службами. [Проконсультируйтесь с проверкой состояния служб Microsoft 365.](https://docs.microsoft.com/office365/enterprise/view-service-health) Не забудьте проверить состояние зависимых служб (например, Exchange, SharePoint, OneDrive для бизнеса). Мониторинг состояния службы подробно рассмотрен в предыдущем разделе "Отслеживание [состояния службы".](#monitor-service-health)

2. **Проверка подключения клиента.** Проблемы с подключением вызывают проблемы с функциями или входом в Teams. Мы рекомендуем проверять подключение к службе (особенно для новых сайтов или местоположений). Убедитесь, что для каждого сайта должны следовать следующие инструкции по URL-адресам и диапазонам IP-адресов Office [365.](https://aka.ms/o365ips) Средство оценки [сети Майкрософт](https://www.microsoft.com/download/details.aspx?id=53885) можно использовать для проверки подключения, чтобы проверить правильность открытия медиа портов для команд. Подробные инструкции по запуску проверок подключения указаны в руководстве [по проверке готовности](prepare-network.md) сети.

3. **Проверьте список известных проблем:** [Проконсультируйтесь с командой](https://docs.microsoft.com/MicrosoftTeams/troubleshoot/teams) по устранению неполадок, чтобы определить, не повлияла ли на пользователя одна из этих проблем. Чтобы устранить проблему, выполните предоставленное обходное решение (если есть).

4. **Посетите сообщество Microsoft Teams:** В [сообществе Microsoft Teams](https://techcommunity.microsoft.com/t5/Microsoft-Teams/ct-p/MicrosoftTeams) есть выделенные пробелы для Teams. В сообществе Teams есть список обсуждений, записи блога и объявления, центр для teams. Вы можете задать вопрос или найти решения своей проблемы в предыдущих обсуждениях.

5. **Обратитесь в службу поддержки Майкрософт.** Вы можете обратиться в службу поддержки Майкрософт для решения проблем с Teams online или по телефону. Сведения см. в справке для администраторов службы поддержки [продуктов для бизнеса.](https://docs.microsoft.com/microsoft-365/admin/contact-support-for-business-products) Для клиентов Premier запросы на поддержку можно инициировать, следуя указаниям в службе поддержки [Microsoft Teams (для](https://support.microsoft.com/premier/contacts)клиентов Premier).
