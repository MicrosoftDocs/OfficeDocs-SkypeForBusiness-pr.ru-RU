# <a name="stream-classification-in-call-quality-dashboard-cqd"></a>Классификация потоков на панели мониторинга качества звонков (CQD)

Панель мониторинга качества звонка (ПМКЗ) для Microsoft Teams и Skype для бизнеса Online позволяет получить представление о качестве звонков, сделанных с помощью команд Microsoft и Skype для бизнес-услуг. В этом разделе представлена ​​подробная информация о классификации качества потоков мультимедиа. Дополнительные сведения о CQD и его настройке см. в разделе ["Настройка панели мониторинга качества звонков"](turning-on-and-using-call-quality-dashboard.md).

## <a name="classifier-definitions"></a>Определения классификатора

Потоки в CQD классифицируются как _хорошие,_ неудовлетворительные или неклассифицированные на основе значений доступных метрик качества ключа. Метрики и условия, используемые для классификации потока, отображаются в следующих таблицах. Измерения CQD "Poor Due To" можно использовать, чтобы понять, какая метрика отвечает за _неудовлетворительную классификацию_ . Дополнительные сведения об этих измерениях см. в [разделе "Измерения и меры", доступных на панели мониторинга качества вызовов](dimensions-and-measures-available-in-call-quality-dashboard.md).

## <a name="unclassified-streams"></a>Неклассифицированные потоки

В CQD поток помечается как  неклассифицированный при сбое подключения к interactive Connectivity Establishment (ICE) или когда не передаются все метрики, необходимые для вычисления классификации потока.

Чтобы проверить сбои подключения ICE, изучите размеры «First Connectivity Ice» и «Second Connectivity Ice» для значения «FAILED». Если одно из значений указывает на сбой, поток помечается как _неклассифицированный_.

Если ice connectivity succeeded for an _Unclassified_ stream, the stream is likely considered _Unclassified_ because key stream metrics were not reported. Есть несколько причин, по которым эти показатели могут не сообщаться:

- **Отчеты о QoE** не были получены— метрики, используемые для классификации, передаются в отчете качества взаимодействия, отправляемом в конце вызова. Если этот отчет не создается (например, из-за того, что некоторые сторонние конечные точки могут не отправлять QoE) или не могут быть отправлены (например, из-за сбоя сети), CQD не может классифицировать поток.

  > [!TIP]
  > Измерение «QoE Record Available» может использоваться для определения того, был ли получен отчет QoE для потока. Обратите внимание, что это измерение будет иметь значение «True», если отчет QoE был получен от любой конечной точки. Отчет о QoE с обеих конечных точек требуется для наиболее точной отчетности показателей.

- **Короткие вызовы** — короткие вызовы могут не иметь достаточной активности мультимедиа для вычисления ключевых метрик потока. Без этих показателей CQD не может классифицировать поток.

  > [!TIP]
  > Для определения продолжительности потока можно использовать размеры «Длительность (секунды)», «Длительность (минуты)», «Длительность 5 секунд или меньше» и «Длительность 60 секунд или более». Измерение «Средняя продолжительность звонка» также может использоваться для вычисления средней продолжительности для набора потоков.

- **Низкая загрузка пакетов** — как и в сценарии "короткого вызова", для вычисления ключевых метрик потока требуется достаточное использование пакетов. Без этих показателей CQD не может классифицировать поток.
  - Распространенный сценарий низкой загрузки пакетов возникает, когда участник присоединяется к собранию, чтобы прослушивать выступающую, но никогда не говорит (микрофон отключен в большинстве вызовов). В этом случае аудиопоток, входящий в клиент, имеет высокую загрузку пакетов, а исходящий аудиопоток от клиента практически не использует пакеты. Длительность потока может быть часом или дольше, но использование пакетов в потоке от клиента к серверу низко с момента отключения микрофона и приводит к неклассифицированным  результатам потока.

  > [!TIP]
  > Измерение «Утилизация пакета» и измерение «Среднее использование пакета» могут использоваться для определения активности пакета в потоке.
