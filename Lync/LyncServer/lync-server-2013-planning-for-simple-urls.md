# Планирование простых URL-адресов в Lync Server 2013

 

_**Дата изменения раздела:** 2015-12-11_

Простые URL-адреса упрощает присоединение к собраниям для пользователей и доступ к средствам администрирования Lync Server для администраторов.

В Lync Server поддерживаются три простых URL-адреса.

  - URL-адрес **собраний** (meet) используется в качестве основного URL-адреса для всех конференций в узле или организации. Пример простого URL-адреса собраний – https://meet.contoso.com. URL-адрес конкретного собрания может иметь вид https://meet.contoso.com/ *имя\_пользователя* /7322994.
    
    Простой URL-адрес собраний делает ссылки на присоединение к собраниям более понятными и удобными для обмена и распространения.

  - URL-адрес **телефонного подключения** (dialin) обеспечивает доступ к веб-странице "Параметры конференц-связи с телефонным подключением". На этой странице приводятся номера телефонного подключения к конференциям с указанием доступных языков, назначенные данные конференций (для собраний, которые не нужно планировать) и элементы управления DTMF. Она также обеспечивает управление персональными идентификационными номерами (ПИН-кодами) и назначенными данными конференций. Простой URL-адрес телефонного подключения включается во все приглашения на собрания. Таким образом пользователи, которые хотят подключиться к собранию с помощью телефона, могут найти в нем необходимый номер телефона и ПИН-код. Пример простого URL-адреса телефонного подключения – https://dialin.contoso.com.

  - URL-адрес **администратора** (admin) обеспечивает быстрый доступ к панели управления Lync Server. Администратор может открыть панель управления Lync Server с любого компьютера, защищенного брандмауэрами организации, введя простой URL-адрес администратора в браузере. Этот URL-адрес используется внутри организации. Пример простого URL-адреса администратора – https://admin.contoso.com

## Область URL-адресов

Вы можете настроить для простых URL-адресов глобальную область действия или задать разные простые URL-адреса для каждого центрального узла в организации. Если заданы как глобальный простой URL-адрес, так и простой URL-адрес на уровне узла, последний имеет приоритет.

В большинстве случаев мы рекомендуем задавать простые URL-адреса только на глобальном уровне, чтобы простой URL-адрес собрания пользователя не изменялся при перемещении из одного узла в другой. Исключением являются организации, в которых требуются различные номера телефонного подключения для пользователей в разных узлах. Обратите внимание на то, что если вы зададите один простой URL-адрес (например, простой URL-адрес телефонного подключения) в узле как простой URL-адрес на уровне узла, вам также нужно задать другие простые URL-адреса как адреса на уровне узла.

Вы можете задать глобальные простые URL-адреса в топологий. Чтобы задать простой URL-адрес на уровне узла, воспользуйтесь командлетом Set-CsSimpleURLConfiguration.

## Задание имен простых URL-адресов

Существует три рекомендованных способа задания имен простых URL-адресов. Выбираемый способ косвенно зависит от способа настройки записей A DNS и сертификатов для поддержки простых URL-адресов. В любом случае вам нужно настроить один простой URL-адрес собраний для каждого домена SIP в организации.

В любом случае в организации требуется только один простой URL-адрес для телефонного подключения и один URL-адрес администратора вне зависимости от количества доменов SIP.

Подробные сведения о необходимых записях A DNS и сертификатах см. в разделах [Требования DNS для простых URL-адресов в Lync Server 2013](lync-server-2013-dns-requirements-for-simple-urls.md) и [Требования к сертификатам для внутренних серверов в Lync Server 2013](lync-server-2013-certificate-requirements-for-internal-servers.md) в документации по планированию.

В первом случае вам нужно создать имя домена SIP для каждого простого URL-адреса.

В этом случае для каждого простого URL-адреса требуется отдельная запись A DNS, а каждый простой URL-адрес собрания должен быть поименован в сертификатах

## Правила именования и проверки простых URL-адресов

Командлеты топологий и Командная консоль Lync Server обеспечивают несколько правил проверки простых URL-адресов. Вы должны задать простые URL-адреса для собраний и телефонного подключения, однако задавать URL-адрес администратора необязательно. Для каждого домена SIP требуется отдельный простой URL-адрес собраний, однако для всей организации нужен только один простой URL-адрес телефонного подключения и URL-адрес администратора.

Каждый простой URL-адрес в организации должен иметь уникальное имя и не может являться префиксом другого простого URL-адреса (например, нельзя задать адрес lync.contoso.com/Meet в качестве простого URL-адреса собраний, а адрес lync.contoso.com/Meet/Dialin – в качестве простого URL-адреса телефонного подключения). Имена простых URL-адресов не могут содержать полные доменные имена пулов или сведения о портах (например, адрес https://FQDN:88/meet недопустим). Простые URL-адреса должны начинаться с префикса https://.

Простые URL-адреса могут содержать только буквы и цифры (a–z, A–Z, 0–9), а также точки (.). При использовании других символов простые URL-адреса могут работать неправильно.

## Изменение простых URL-адресов после развертывания

Если вы изменяете простой URL-адрес после развертывания, вам следует учесть, как это изменение повлияет на записи DNS и сертификаты для простых URL-адресов. Если изменяется базовый адрес простого URL-адреса, то вам следует изменить соответствующим образом записи DNS и сертификаты. Например, при изменении адреса https://lync.contoso.com/Meet на https://meet.contoso.com базовый URL-адрес меняется с lync.contoso.com на meet.contoso.com, поэтому вам нужно изменить записи DNS и сертификаты так, чтобы в них указывался адрес meet.contoso.com. Если вы изменили простой URL-адрес с https://lync.contoso.com/Meet на https://lync.contoso.com/Meetings, базовый URL-адрес lync.contoso.com остается неизменным, поэтому вносить изменения в DNS или сертификаты не требуется.

Однако при любом изменении простого URL-адреса необходимо выполнить командлет **Enable-CsComputer** для каждой роли-директоре и сервере переднего плана с целью регистрации изменения.
