# <a name="defining-the-network-elements-used-to-determine-location-in-lync-server-2013"></a>Определение сетевых элементов, используемых для определения расположения в Lync Server 2013

</div>

<div id="mainSection">

<div id="mainBody">

<span> </span>

_**Последнее изменение темы:** 2012-10-29_

Если вы настраиваете инфраструктуру Lync Server для поддержки автоматического обнаружения местонахождения клиентов, сначала необходимо решить, какие сетевые элементы будут использоваться для сопоставления абонентов с расположениями. В Lync Server 2013 вы можете сопоставить следующие сетевые элементы уровня 2 и 3 с расположениями:

  - адреса BSSID точек беспроводного доступа (WAP) (уровень 2);

  - порт коммутатора LLDP (уровень 2);

  - идентификаторы шасси коммутаторов LLDP (уровень 2);

  - IP-подсети (уровень 3);

  - MAC-адреса клиентов (уровень 2).

Элементы сети перечислены в порядке приоритета. Если клиент можно найти с помощью нескольких элементов сети, Lync Server использует порядок приоритетов для определения используемого механизма.

В следующих подразделах приводятся более подробные сведения об использовании каждого элемента сети.

<div>


> [!IMPORTANT]  
> При использовании сетевых элементов для сопоставления вызывающих абонентов с расположением местоположениями важно, чтобы база данных службы сведений о расположениях была актуальной. Так, если вы добавляете или изменяете элемент сети, например WAP, вам нужно удалить старую запись и добавить новую в базу данных местоположений.



</div>

<div>

## <a name="wireless-access-point"></a>Точка беспроводного доступа

Когда клиент подключается к сети беспроводным способом, запрос расположения использует адрес BSSID в WAP, чтобы определить его расположение. Если клиент находится в роуминге, указанный WAP может не быть самым ближайшим, а также можно взять WAP, который находится на другом этаже здания. Чтобы указать, что расположение приблизительно указано, можно присвоить параметру location значение NEAR или Close для дескриптора.

Этот метод определения местоположения предполагает, что адреса BSSID всех точек WAP являются статическими. Однако если производитель WAP использует динамически назначаемые адреса BSSID, полученный адрес BSSID может измениться (вследствие изменения конфигурации WAP) и беспроводные клиенты могут не получить местоположение. Чтобы предотвратить эту возможность, необходимо заполнить базу данных службы сведений о местоположении на Erl для всех возможных адресов BSSID, используемых каждым приложением WAP.

</div>

## <a name="lldp-ports-and-switches"></a>Порты и коммутаторы LLDP

Управляемые коммутаторы Ethernet, поддерживающие протокол LLDP-MED, могут сообщить свои идентификационные данные и сведения о портах клиентам, совместимым с LLDP-MED. Это позволяет затем выполнять запросы к базе данных местоположений для определения местоположения устройства. Вы можете связать сведения о местоположении для экстренных служб с идентификатором шасси коммутатора или сопоставить их также на более низком уровне — уровне портов.

<div>


> [!NOTE]  
> Lync Server 2013 поддерживает использование LLDP — MED для определения расположений только для устройств Lync Phone Edition и Lync 2013 под управлением Windows 8. Если необходимо использовать данные уровня 2 уровня коммутатора, чтобы определить расположение других проводных клиентов Lync на основе компьютера, необходимо использовать метод клиентского MAC-адреса.



</div>

## <a name="subnet"></a>Подсеть

IP-подсети уровня 3 предоставляют механизм, поддерживаемый всеми клиентами Lync Server, которые можно использовать для автоматического обнаружения местоположения клиента. Использование IP-подсетей является самым простым способом определения местоположения в плане настройки клиентов проводного доступа и управления ими. Однако прежде чем выбрать его, ответьте на следующие вопросы, чтобы определить, будет ли он достаточно точным в вашей ситуации.

  - Охватывают ли какие-либо клиентские подсети несколько этажей?

  - Охватывают ли какие-либо подсети несколько зданий?

  - Какую площадь покрывает каждая клиентская подсеть?

Если подсеть покрывает слишком большую площадь, может потребоваться использовать другой способ для определения местоположения клиентов. Однако, если это целесообразно, мы рекомендуем клиентам реорганизовать структуру IP-подсетей в соответствии с требованиями к точности определения местоположений ERL вместо того, чтобы реализовывать сторонние решения на основе SNMP, связанные с дополнительными затратами и повышением сложности.

## <a name="client-mac-address"></a>MAC-адрес клиента

Чтобы использовать MAC-адрес клиентского компьютера для поиска вызывающего абонента, необходимы управляемые коммутаторы Ethernet и решение SNMP стороннего производителя, которое может обнаружить MAC-адреса клиентов Lync, подключенных к (или) этим коммутаторам. Решение SNMP постоянно опрашивает управляемые коммутаторы для получения текущих сопоставлений MAC-адресов конечной точки, подключенных к каждому порту, и получения соответствующих идентификаторов портов. Во время запроса клиента Lync в службе сведений о расположении служба сведений о местоположении запрашивает приложение стороннего производителя, используя MAC-адрес клиента, а затем возвращает все подходящие IP-адреса коммутатора и идентификаторы портов. Служба "сведения о местоположении" использует эти сведения для запроса к опубликованному уровню 2 карты географических соответствий для соответствующей записи и возвращает расположение клиенту. Если вы используете этот параметр, убедитесь, что идентификаторы портов коммутатора согласованы между записями SNMP и опубликованными записями базы данных расположения.

<div>


> [!NOTE]  
> Некоторые сторонние решения SNMP могут поддерживать неуправляемые ключи доступа. Если коммутатор, который работает с клиентом Lync, неуправляемый, но имеет привязку к управляемому коммутатору распространения, управляемый коммутатор может отчитываться о нем в SNMP-адреса клиентов, подключенных к параметру доступа. Эта информация позволяет службе сведений о местоположении определить расположение пользователя. Тем не менее, можно назначить только один Пэр всем портам неуправляемого коммутатора, поэтому характер расположения доступен только на уровне корпуса коммутатора доступа, а не на уровне порта.
