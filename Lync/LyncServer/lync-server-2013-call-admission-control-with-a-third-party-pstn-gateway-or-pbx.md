# <a name="call-admission-control-in-lync-server-2013-with-a-third-party-pstn-gateway-or-pbx"></a>Контроль допуска звонков с помощью стороннего шлюза ТСОП или УАТС в Lync Server 2013

</div>

<div id="mainSection">

<div id="mainBody">

<span> </span>

_**Тема последнего изменения:** 2012-10-20_

В этом разделе описываются примеры развертывания контроля допуска звонков (CAC) на канале между интерфейсом шлюза сервера-посредника и сторонним шлюзом ТСОП или УАТС.

<div>

## <a name="case-1-cac-between-the-mediation-server-and-a-pstn-gateway"></a>Случай 1. Контроль допуска звонков между сервером-посредником и шлюзом ТСОП

Контроль допуска звонков можно развернуть на канале глобальной сети, связывающем интерфейс шлюза сервера-посредника и сторонний шлюз ТСОП или УАТС.

**Случай 1. Контроль допуска звонков между сервером-посредником и шлюзом ТСОП**

![Случай 1: CAC между сервером посредников и шлюзом PSTN] (images/Gg398762.4bebf9ee-2732-4ea6-bbe5-0269b2903d8c(OCS.15).jpg "Случай 1: CAC между сервером посредников и шлюзом PSTN")

В этом примере для сервера-посредника и шлюза PSTN применяется CAC. Если пользователь клиента Lync на сетевом сайте 1 размещает КОММУТИРУЕМый Звонок через шлюз PSTN в сетевом сайте 2, мультимедиа проходит через ГЛОБАЛЬную сеть. Поэтому для каждого сеанса ТСОП выполняются две проверки контроля допуска звонков.

  - Между клиентским приложением Lync и сервером обновлений

  - Между сервером исправлений и шлюзом PSTN

Проверки выполняются как для входящих звонков ТСОП, поступающих в клиент на сетевом сайте 1, так и для исходящих звонков ТСОП из клиентского приложения на сетевом сайте 1.

<div>


> [!NOTE]
> Убедитесь в том, что IP-подсеть, к которой относится шлюз ТСОП, настроена и связана с сетевым сайтом 2.<BR>Убедитесь, что IP-подсеть, к которой принадлежат оба интерфейса сервера-посредника, настроена и соответствует сетевому сайту 1.<BR>Дополнительные сведения можно найти <A href="lync-server-2013-associate-a-subnet-with-a-network-site.md">в разделе Связывание подсети с сетевым сайтом в Lync Server 2013</A>.



</div>

</div>

<div>

## <a name="case-2-cac-between-the-mediation-server-and-a-third-party-pbx-with-media-termination-point"></a>Случай 2: CAC между сервером-посредником и сторонней АТС с точкой завершения мультимедиа

Эта конфигурация аналогична случаю 1. В обоих случаях сервер-посредник определяет, какое устройство завершает работу с мультимедиа на противоположном конце ссылки WAN, и IP-адрес шлюза PSTN или УАТС с точкой завершения (MTP) настроен на сервере-посреднике в качестве следующего прыжка.

**Случай 2. Контроль допуска звонков между сервером-посредником и сторонней УАТС с точкой MTP**

![Случай 2: CAC между серверными АТС с помощью MTP] (images/Gg398762.1c0b5263-c053-4cca-842f-85dd670760c8(OCS.15).jpg "Случай 2: CAC между серверными АТС с помощью MTP")

В этом примере для сервера-посредника и УАТС (MTP) применяется CAC. Если пользователь клиента Lync на сетевом сайте 1 помещает PSTN-Звонок через УАТС/MTP, расположенную на сетевом сайте 2, мультимедиа передается через ГЛОБАЛЬную сеть. Поэтому для каждого сеанса ТСОП выполняются две проверки контроля допуска звонков.

  - Между клиентским приложением Lync и сервером обновлений

  - Между сервером-посредником и УАТС/MTP

Проверки выполняются как для входящих звонков ТСОП, поступающих в клиент на сетевом сайте 1, так и для исходящих звонков ТСОП из клиента на сетевом сайте 1.

<div>


> [!NOTE]
> Убедитесь в том, что IP-подсеть, к которой относится точка MTP, настроена и связана с сетевым сайтом 2.<BR>Убедитесь, что IP-подсеть, к которой принадлежат оба интерфейса сервера-посредника, настроена и соответствует сетевому сайту 1.<BR>Дополнительные сведения можно найти <A href="lync-server-2013-associate-a-subnet-with-a-network-site.md">в разделе Связывание подсети с сетевым сайтом в Lync Server 2013</A>.



</div>

</div>

<div>

## <a name="case-3-cac-between-the-mediation-server-and-a-third-party-pbx-without-a-media-termination-point"></a>Случай 3: CAC между сервером-посредником и сторонней АТС без точки завершения мультимедиа

Случай 3 несколько отличается от первых двух. Если в сторонней УАТС нет MTP, то для запроса исходящего сеанса на независимую АТС не удается определить, в каком месте на границах УАТС будет прерываться мультимедиа. В этом случае мультимедиа перетекает прямо между сервером-посредником и сторонним устройством конечной точки.

**Случай 3. Контроль допуска звонков между сервером-посредником и сторонней УАТС без точки MTP**

![Дело 3: CAC между серверными АТС без MTP] (images/Gg398762.f4bcf800-3a68-4037-bb3f-adb2fdf50d32(OCS.15).jpg "Дело 3: CAC между серверными АТС без MTP")

В этом примере, если пользователь клиента Lync на сетевом сайте 1 размещает вызов пользователя через УАТС, сервер-посредник может выполнять проверки CAC только для прокси-сервера (между клиентским приложением Lync и сервером-посредником). Поскольку сервер-посредник не имеет сведений о устройстве конечной точки во время запроса сеанса, перед установкой звонка невозможно выполнить проверки CAC на канале WAN (между сервером-посредником и сторонней конечной точкой). Однако после того, как сеанс будет установлен, сервер исправлений облегчает учет пропускной способности, используемой в канале.

Для вызовов, инициированных из сторонней конечной точки, сведения об этом устройстве будут доступны на момент запроса сеанса, а проверка CAC может выполняться на обеих сторонах сервера-посредника.

<div>


> [!NOTE]
> Убедитесь в том, что IP-подсеть, к которой относятся устройства конечных точек, настроена и связана с сетевым сайтом 2.<BR>Убедитесь, что IP-подсеть, к которой принадлежат оба интерфейса сервера-посредника, настроена и соответствует сетевому сайту 1.<BR>Дополнительные сведения можно найти <A href="lync-server-2013-associate-a-subnet-with-a-network-site.md">в разделе Связывание подсети с сетевым сайтом в Lync Server 2013</A>.



</div>
