Системное администрирование включает повседневные административные задачи, как запланированные, так и по требованию, которые необходимы для обеспечения бесперебойной работы системы. Как правило, для задач по администрированию системы используются написанные процедуры. Эти процедуры гарантируют, что одни и те же стандартные средства и методы используются всеми сотрудниками службы поддержки.

В среде Lync типичные задачи по администрированию системы включают резервное копирование и архивацию пулов, мониторинг журналов, создание пользователей и управление ими, а также обновление антивирусного программного обеспечения.

## <a name="system-troubleshooting"></a>Устранение неполадок системы

Организация должна быть подготовлена к работе с непредвиденными проблемами и должна содержать процедуру по управлению проблемами с момента их устранения. Сведения о том, как сотрудники службы поддержки определили проблемы, которые необходимо записать и использовать в будущем, чтобы избежать ненужной повторяющейся выполненной работы.

## <a name="system-troubleshooting-process"></a>Процесс устранения неполадок системы

На следующей схеме показан процесс устранения неполадок системы и взаимодействия с другими ролями операций.

**Блок-схема устранения неполадок системы**

![Блок-схема устранения неполадок системы](images/Dn720318.869d0b89-6473-4b1f-9d90-59604b4b8e98(OCS.15).jpg "Блок-схема устранения неполадок системы")

  - **Классификация и определение приоритетов**     Эта задача обычно выполняется службой Service Desk. Например, ошибка может быть сгруппирована как ошибка программного обеспечения или аппаратная ошибка. Затем эта ошибка перенаправляется в соответствующую группу поддержки для изучения. Правила определения приоритета проблемы, а также время для ответа и время для разрешения, обычно определяются в соглашении об уровне обслуживания.

  - **Исследование и диагностика**     Соответствующая команда поддержки выполняет диагностику проблемы и предлагает изменения, необходимые для решения этой проблемы. Если решение является простым и не нуждается в контроле изменений, решение можно применить немедленно. Если решение непростое, должно быть вызвано требование на изменение, а в процессе управления изменениями, как правило, используется процедура "Fast-Track". Все внесенные изменения необходимо записать с помощью процесса управления конфигурацией.

  - **Закрыть и записать**     После проверки решения проблемы необходимо закрыть. Если есть уроки, которые следует изучить из этой ситуации, в базе знаний необходимо создать запись.

  - **Анализ и анализ**     тенденций Для определения тенденций проблем следует выполнить периодические обзоры последних проблем. Например, если у пользователей часто возникают проблемы с медленным входом на сайты Lync, это может быть вызвано проблемами с пропускной способностью сети. Устранение проблем с устранением проблем, а также результат, влияющий на доступность системы, и их сравнение с соглашением об уровне обслуживания. Пользователь, взаимодействует с клиентом при возникновении проблем со службой, таких как менеджер по работе с учетными записями, должен сообщить о значительных проблемах.

Средства службы поддержки позволяют сотрудникам записывать, классифицировать и определять приоритет новых проблем. Затем средства предоставляют рабочие процессы для управления запросом на обслуживание проблемы с помощью расследования и диагностики, как правило, с помощью более одной команды поддержки. Средства, которые часто предоставляют отчеты о времени разрешения и прошлых тенденциях, также могут включать базу данных базы знаний, которая может использоваться для поиска в прошлых вопросах.

## <a name="centralized-vs-decentralized-administration"></a>Централизованное и децентрализованное администрирование

Роли и обязанности по выполнению задач системы администрирования зависят от того, соответствует ли Организация централизованной модели, децентрализованной модели или их комбинации.

  - **Централизованная модель**     В централизованной модели одна или несколько административных групп обеспечивают полный контроль над всей средой Lync Server. Эта административная модель напоминает центр обработки данных, в котором все задачи администрирования выполняются одной группой информационных технологий. Роли и ответственность в группе должны определяться в соответствии с опытом и квалификацией.

  - **Децентрализованная модель**     Децентрализованные организации размещаются в нескольких географических расположениях и работают с серверами Lync Server и Teams в различных расположениях. Например, для каждой страны или региона может быть местный персонал администрирования и один или несколько серверов, на которых работает Lync Server 2013. Возможно также, что существует пул серверов, на которых работает Lync Server 2013 и административная группа для Северной Америки и одна для Европы. Иногда администраторы могут быть ответственны только за собственную географическую область и ограничивать разрешения на администрирование ресурсов в других областях.

Lync Server также позволяет делегировать конкретные задачи администрирования определенным пользователям или группам с помощью управления доступом на основе ролей (RBAC). RBAC позволяет администраторам делегировать определенные права пользователей и разрешения другим администраторам для выполнения подмножества возможных административных задач. С помощью RBAC пользователи могут выполнять определенные задачи администрирования, зависящие от ролей RBAC, назначенных пользователю. RBAC предоставляет список командлетов, которые пользователь может запускать на основе ролей RBAC, участником которых является пользователь.
