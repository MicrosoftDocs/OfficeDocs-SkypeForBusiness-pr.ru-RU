## <a name="transferring-a-file-in-an-instant-message-fails-if-the-file-is-open-in-another-application"></a>Передача файла в мгновенном сообщении завершается сбоем, если файл открыт в другом приложении

**Проблемах**

Если вы пытаетесь передать файл, например документ Word, включив его в мгновенное сообщение другому пользователю Lync, передача будет выполнена успешно, но в действительности может возникнуть сбой при передаче файла. В клиенте Lync отображается значок для типа файлов, но этот файл нельзя открыть с помощью предполагаемого получателя. Выводится сообщение об ошибке, в котором сообщается о том, что передача данных завершилась неудачно.

**Смысла**

Чтобы устранить эту ошибку, закройте открытый файл или приложение, которое оно открыто, прежде чем пытаться передать его в мгновенное сообщение.

## <a name="if-lync-server-storage-service-data-replication-fails-administrators-will-need-to-check-performance-counters-for-stale-storage-service-queue-items"></a>Если служба хранилища данных Lync Server не может выполнить репликацию, администраторы должны будут проверять счетчики производительности для устаревших элементов очереди службы хранилища.

**Проблемах**

Служба хранилища Lync Server использует Windows Fabric для репликации. Если данные удаляются на основном сервере переднего плана, но удаление на дополнительном сервере переднего плана завершается сбоем (например, при неожиданном завершении работы или ошибках на сервере переднего плана), данные могут быть оставлены и "потерянные". Потерянные данные могут привести к снижению производительности и потреблению свободного места на диске.

**Смысла**

Чтобы обойти эту ошибку, если в журнале событий\_будут\_регистрироваться\_ошибки, используемые в пространстве\_данных Лисс DB\_(\_ID\_=\_32058) и Лисс DB, используется критическое значение (ID = 32059), администраторы должны проверить Счетчик производительности на сервере переднего плана в разделе **Ls: Лисс-API службы хранилища** с именем **Лисс-Current Number службы хранилища устарело элементы очереди**. Если этот счетчик производительности имеет большое значение (например, более 50000), администратор должен запустить средство Клеануупсторажесервицедата. exe в наборе ресурсов Lync Server 2013, в котором будут удалены все потерянные данные из пула. Подробнее об этом средстве можно найти в документации по набору ресурсов Lync Server 2013.

## <a name="whenever-the-ip-address-configuration-is-changed-for-a-server-or-pool-lync-server-services-need-to-be-restarted"></a>При изменении конфигурации IP-адреса для сервера или пула необходимо перезапустить службы Lync Server.

**Проблемах**

При изменении конфигурации IP-адреса для развертывания Lync Server 2013, например перехода с IPv4 на сдвоенный или из двух стеков в IPv6, не все серверные компоненты изменяют конфигурацию до тех пор, пока не будут перезапускаются эти службы.

**Смысла**

Чтобы обойти эту ошибку, перезапустите службы Lync Server после изменения конфигурации IP-адреса для развертывания. Для этого выполните в командной консоли Lync Server следующие командлеты:

   ```
    Stop-CsWindowsService -graceful
   ```

   ```
    Start-CsWindowsService
   ```

## <a name="the-dial-in-conferencing-synthetic-transaction-cmdlet-is-no-longer-available-in-the-lync-server-2013-management-pack"></a>Командлет искусственной транзакции конференц-связи с телефонным подключением больше не доступен в пакете управления Lync Server 2013

**Проблемах**

Командлет "синтетическая транзакция конференц-связи с телефонным подключением" **— ксдиалинконференЦинг** больше не доступен в пакете управления Lync Server 2013.

**Смысла**

Использование командлета "Виртуальная транзакция конференц-связи с телефонным подключением" **— ксдиалинконференЦинг** поддерживается только внутри предприятия.

Администраторы могут продолжать использовать командлет в командной консоли Lync Server для устранения неполадок. В случае необходимости предприятие также может разработать частный пакет управления, чтобы выполнить командлет внутренне.

## <a name="the-centralized-logging-service-stops-if-network-traffic-is-disrupted-when-log-files-are-being-copied-to-network-share"></a>Служба централизованного ведения журналов останавливается, если сетевой трафик недоступен при копировании файлов журнала в сетевой ресурс

**Проблемах**

Если Служба централизованного ведения журналов настроена на использование сетевого пути (значение параметра Качефиленетворкфолдер командлета **Get-ксклсконфигуратион** — допустимый путь в формате UNC), кэшированные файлы журнала копируются в сетевую папку. Если при копировании файлов происходит перерыв в сетевом трафике, произойдет исключение, которое вызовет остановку службы централизованного протоколирования.

Служба настроена на автоматическую перезагрузку в три значения, поэтому служба будет восстанавливаться из первых трех исключений.

**Смысла**

Для этой проблемы не существует обходного пути. Чтобы определить причину ошибки, проследите за журналом событий для события с кодом 7031 от диспетчера управления службами, которые регистрируются в журнале при внезапном завершении работы службы «Служба централизованного ведения журнала» Lync Server. Если это происходит более трех вхождений, перезапустите службу вручную с помощью командлета **Start-ксвиндовсервице** .

## <a name="storage-service-queue-items-need-to-be-imported-manually"></a>Элементы очереди обслуживания хранилища нужно импортировать вручную

**Проблемах**

В Lync Server 2013 хранятся данные о конференциях и обмен мгновенными сообщениями, в том числе архивированные сообщения и запись сведений о вызове (CDR) для базы данных на каждом сервере переднего плана. Данные хранятся в базе данных во время ее обработки до того момента, когда они будут переданы в целевое место назначения. Для повышения производительности Lync Server 2013 периодически экспортирует элементы очереди из локальной базы данных, которые не обрабатываются в течение длительного периода времени, и сохраняет их в хранилище файлов. Если хранилище файлов недоступно, элементы хранятся на каждом сервере переднего плана. Эта же операция возникает для предотвращения потери данных во время перемещения пула.

В ходе операции экспорта служба хранилища Lync Server регистрирует все этапы в журнале событий с кодами событий 32075 (полная очистка), 32076 (полная очистка завершена), 32082 (полное удаление), 32083 (запись на уровне обслуживания завершено) завершено), 32089 (произошел сброс из-за заполнения базы данных). Эти данные не будут автоматически импортированы в систему и будут обрабатываться и доставляться в конечное расположение (SQL Server или Exchange Server).

**Смысла**

Для импорта данных в систему администраторам потребуется средство Импортсторажесервицедата в наборе ресурсов Lync Server Resource Kit, который добавляет данные обратно в систему для обработки и доставляется в конечную точку.

</div>

## <a name="address-book-web-query-searches-will-fail-if-the-default-value-for-usenormalizationrules-is-changed-to-false"></a>Поиск в веб-книге не выполняется, если значение по умолчанию для Усенормализатионрулес изменено на "ложь"

**Проблемах**

Если значение по умолчанию для Усенормализатионрулес изменено на "false", поиск в адресной книге веб-запроса завершается сбоем. После того как значение по умолчанию будет изменено, пользователи клиента Lync не смогут использовать веб-запрос адресной книги Lync для поиска пользователей.

**Смысла**

Если в качестве значения по умолчанию для Усенормализатионрулес задано значение false, чтобы пользователи могли использовать номера телефонов, определенные в доменных службах Active Directory 2013 без применения правил нормализации, обойти эту ошибку, выполнив указанные ниже действия.

1.  Запустите командную консоль Lync Server Management Shell: нажмите кнопку **Пуск**, выберите **все программы**, а затем — **Microsoft Lync Server 2013**, а затем — **Командная консоль Lync Server Management Shell**.

2.  Выполните одно из следующих действий:
    
      - Если в развертывании есть только серверы Lync Server 2013, выполните следующий командлет на глобальном уровне, чтобы изменить значения Усенормализатионрулес и Игнореженерикрулес на true:
        
            Set-CsAddressBookConfiguration -identity <XdsIdentity> -UseNormalizationRules=$true -IgnoreGenericRules=$true
    
      - Если в развертывании есть сочетание Lync Server 2013 и Lync Server 2010 или Office Communications Server 2007 R2, запустите следующий командлет и назначьте его каждому пулу Lync Server 2013 в топологии.
        
            new-csAddressBookConfiguration -identity <XdsIdentity> -UseNormalizationRules=$true -IgnoreGenericRules=$true

3.  Дождитесь появления репликации CMS на всех пулах.

4.  Измените файл правил нормализации телефона для развертывания, чтобы очистить содержимое. Файл находится в общей папке каждого пула Lync Server 2013. Если файл отсутствует, создайте пустой файл\_под названием "\_\_\_правила нормализации" номера компании ". txt".

5.  Подождите несколько минут, пока все клиентские пулы смогут прочитать новые файлы.

6.  Выполните следующий командлет на каждом пуле Lync Server 2013 в развертывании.
    
        Update-csAddressBook

## <a name="address-book-server-error-event-21054-is-generated-once-daily-for-each-lync-2013-pool"></a>Сообщение об ошибке сервера адресной книги 21054 формируется один раз для каждого пула Lync 2013

**Проблемах**

Lync Server 2013 Server Book создает событие ошибки 21054 один раз в день, когда выполняется ежедневное обслуживание. Это сообщение об ошибке также генерируется каждый раз, когда администратор выполняет командлет **Update-ксаддрессбук** , даже если обновление прошло успешно. Однако это событие ошибки можно спокойно проигнорировать после успешного обновления.

**Смысла**

При возникновении этого события с ошибкой запустите следующий командлет:

    Debug-csAddressBookReplication -Poolfqdn <Pool FQDN for which the event was generated>

Если в командлете сообщается, что нет неиндексированных или отброшенных объектов, событие Error 21054 может быть проигнорировано.

Кроме того, индикатор работоспособности ключа (КХИ) "правильно индексированные пользователи в адресной книге" должен быть отключен в System Center Operations Manager.

## <a name="the-invoke-cspoolfailback-cmdlet-may-fail-during-pool-failback"></a>Во время восстановления пула может произойти сбой командлет Invoke-Кспулфаилбакк

## <a name="data-loss-may-occur-when-you-add-a-front-end-server-to-an-already-established-pool--hybrid-skype-for-business-online"></a>При добавлении сервера переднего плана в уже установленный пул (гибридная среда, Skype для бизнеса Online) может произойти потеря данных

**Проблемах**

Эта проблема может возникать в среде, в которой в пуле есть более одного сервера переднего плана, и вы либо перезагрузите один из серверов переднего плана, либо добавьте новый сервер переднего плана, который ранее не был частью пула.

Пользователи, чьи данные архивируются, могут столкнуться с потерей данных до тех пор, пока не будет установлено стабильное распределение архивации данных для пула. Этот период возможной потери данных ограничен 15 минутами общения между абонентами и 30 минут для конференц-связи.

**Смысла**

Если вы выполняете обслуживание, вместо того, чтобы запускать серверы переднего плана в пуле по одному, вы должны выполнить переключение из пула в другой пул, а затем выполнить на серверах задачи обслуживания. Вы также можете сделать службу недоступной, прежде чем выполнять задачи обслуживания, а затем восстановить доступность по завершении обслуживания.

## <a name="administrators-cannot-get-licensee-count-by-using-the-get-csclientaccesslicense-cmdlet"></a>Администраторам не удается получить подсчеты лицензий с помощью командлета Get-Ксклиентакцесслиценсе

**Проблемах**

Администраторам не удается получить точную информацию об использовании клиентских лицензий с помощью командлета **Get-ксклиентакцесслиценсе** .

**Смысла**

Для проверки типа лицензий сервера можно выполнить командлет **Get-кссервице** , чтобы получить полные доменные имена (фдкнс) всех баз данных. Если полное доменное имя сервера переднего плана совпадает с полным доменным именем серверной базы данных, лицензия является лицензией Standard Edition. В противном случае лицензия является лицензией Enterprise Edition.

## <a name="client-licensee-count-is-not-accurately-reported"></a>Счетчик клиентских лицензий не сообщается точно

**Проблемах**

При определении счетчиков клиентских лицензий могут возникать указанные ниже условия.

1.  **Неверное число лицензий для мобильных пользователей**
    
    Количество лицензий зависит от количества уникальных IP-адресов для пользователей, работающих с устройством. Число лицензий будет ограничено указанными ниже способами.
    
      - Если IP-адрес пользователя изменяется во время сеансов Lync, лицензии будут перечислены. Это может произойти, если пользователь подключается к серверу Lync Server из нескольких местоположений с помощью настольного клиента.
    
      - Лицензии будут подсчитываться, если пользователь подключается к мобильному клиенту, так как не удается определить IP-адрес устройства.

2.  **Лицензии используются дважды для звонков по коммутируемой телефонной сети, а клиент Lync — на линии PSTN и звонки из Lync, переадресованные в PSTN-линии**
    
    В следующих сценариях вместо одной будет учитываться две дополнительные лицензии, так как номер телефона и пользователь Lync будут учитываться для определения количества используемых лицензий. Чтобы получить точные данные о лицензировании, вручную удалите лицензии, созданные с помощью номера телефона.
    
      - Телефонный звонок в Lync по телефонной линии
    
      - Вызов Lync на линии PSTN
    
      - КОММУТИРУЕМый Звонок в Lync, а затем Lync пересылает Звонок на телефон PSTN. Будет учитываться одна из линий PSTN.

3.  **Лицензия не будет учитываться при входе в систему Lync на телефоне**
    
    Если пользователь использует телефон, сертифицированный Lync, если телефон входит в систему и остается подключенным, то есть состояние входа в систему, но при этом номер телефона не будет считаться на использование лицензии, если запрос на лицензии произошел после входа в систему.

4.  **Лицензии, подсчитанные для КОММУТИРУЕМых телефонов, соединяющих конференции**
    
    Когда пользователь присоединяется к Конференции с помощью телефонной сети PSTN, лицензия будет считаться неточной для присоединения к Конференции. Тем не менее, для присоединения к Конференции с помощью телефонной сети PSTN лицензия не требуется.

**Смысла**

1.  **Неверное число лицензий для мобильных пользователей**
    
      - Вы можете вручную идентифицировать IP-адреса, которые принадлежат тому же устройству, а затем удалить один из них в подсчете лицензий.
    
      - Для этой проблемы не существует обходного пути для мобильных устройств, которые подключаются с помощью клиента Lync.

2.  **Лицензии используются дважды для звонков по коммутируемой телефонной линии в клиенте Lync, клиентам Lync для звонков по сети PSTN и звонков в Lync, пересылаемых на линии PSTN**
    
    Вам потребуется вручную указать номер телефона КТСОП и удалить число лицензий, созданное для него.

3.  **Лицензия не будет учитываться для входного телефона Lync**
    
    Вы можете настроить телефон Lync для выхода из системы, а затем снова войти в систему через определенный интервал, например каждые 3 месяца.

    Вы можете вручную идентифицировать телефонный номер PSTN, который используется для присоединения к Конференции, и удалить лицензию, созданную номером телефона.

## <a name="lync-server-2013-fails-to-install-on-ita-windows-server-2012-os-rtm-version"></a>Не удается установить Lync Server 2013 в операционной системе Windows Server 2012 версии RTM

**Проблемах**

Не удается установить Lync Server 2013 на сервер ITA Windows Server 2012 из-за сбоя установки Windows Fabric.

Установка Windows Fabric завершается сбоем, так как при этом трассировка структуры создается с использованием формата времени ЧЧ: мм: СС. Однако в операционной системе Windows Server ITA формат времени — чч. MM.SS.

**Смысла**

Чтобы устранить эту ошибку, обновите системный реестр перед установкой Lync Server 2013. Раздел реестра, который необходимо обновить: HKEY\_пользователи.\\ Международная\\\\стимеформат по\\умолчанию для панели управления. Измените значение Стимеформат на чч: мм: СС с помощью интерфейса командной строки Windows PowerShell следующим образом:

1.  Запустите Windows PowerShell и выполните следующие командлеты:
    
       ```
        New-PSDrive -Name HKU -PSProvider Registry -Root HKEY_USERS
       ```
    
       ```
        $a="HKU:\.Default\Control Panel\International"
       ```

2.  Чтобы просмотреть текущее значение, выполните следующий командлет:
    
        Get-itemproperty $a -Name sTimeFormat
    
    Запишите текущее значение для Стимеформат, чтобы его можно было восстановить после завершения установки.

3.  Чтобы установить новое значение, выполните следующий командлет:
    
        Set-ItemProperty $a -Name sTimeFormat -Value "HH:mm:ss"

4.  После успешной установки Lync Server 2013 восстановите исходное значение для Стимеформат, выполнив следующий командлет:
    
        - Set-Итемпроперти $a-Name Стимеформат-value "<значение, указанное в действии 3. выше> "

**Проблемах**

При создании и редактировании собрания в турецком варианте веб-планировщика Lync параметр "люди, выбранные пользователем" в разделе "кто является выступающим" не поддерживается. Для решения этой проблемы пользователи могут использовать параметр по умолчанию "люди из моей организации" или любой другой вариант, например "только организатор" или "все, включая пользователей за пределами моей компании".
