# <a name="reference-topology-for-lync-server-2013-in-large-organizations-with-multiple-data-centers"></a>Эталонная топология для Lync Server 2013 в крупных организациях с несколькими центрами обработки данных

</div>

<div id="mainSection">

<div id="mainBody">

<span> </span>

_**Последнее изменение темы:** 2012-10-22_

Эталонная топология для крупной организации с поддержкой нескольких центров обработки данных предназначена для организации любого размера с несколькими центральными сайтами. Конкретная топология, показанная на следующей схеме, предназначена для организации с 50000 пользователей — по 20000 пользователей на центральных сайтах A и B и 10000 пользователей на центральном сайте C и на сайтах филиалов. Тип топологии, показанный на этой схеме, подходит для организаций с любым количеством пользователей.

В дополнение к высокой доступности, обеспечиваемым пулами серверов переднего плана, эта топология добавляет поддержку аварийного восстановления. Пулы переднего плана на центральных сайтах A и B объединяются вместе. Если один из этих пулов выйдет из строя, администратор сможет переместить службы для пострадавших пользователей в сопряженный пул на работающем сайте.

Эта топология показана на нескольких схемах, с обзором, за которым следует подробное описание центральных сайтов.

  - **Внешние пулы объединены для обеспечения аварийного восстановления.**    Пулы переднего плана на сайтах а и б связаны друг с другом, чтобы обеспечить поддержку аварийного восстановления. Если происходит сбой пула на одном сайте, администратор может отработка отказа для пользователей с этого сайта в связанный пул переднего плана на другом сайте с минимальным перерывом в обслуживании для пользователей. Каждый из двух интерфейсных пулов содержит шесть серверов, которых достаточно для всех 40000 пользователей обоих пулов в случае отработки отказа. Дополнительные сведения приведены в статье [Планирование обеспечения высокой доступности и аварийного восстановления в Lync Server 2013](lync-server-2013-planning-for-high-availability-and-disaster-recovery.md).

  - **Внутренние серверы зеркально отображаются**   , чтобы обеспечить более высокую доступность для основных пользовательских функций, Организация развернула зеркальную зеркальную копию внутренних серверов для каждого пула переднего плана. Это необязательная топология, и вы можете вместо этого развернуть один сервер внутреннего сервера.

  - **Использование сервера Standard Edition на сайте филиала.**    Эта организация рассматривает сайт C как сайт филиала, так как имеет только 600 сотрудников. Однако эти пользователи проводят в своей среде много аудио- и видеоконференций. Если она была развернута в Lync Server как сайт филиала, мультимедиа для этих конференций будет выполняться в глобальной сети (WAN) на центральном сайте с развернутым сервером переднего плана. Чтобы избежать этой потенциальной нагрузки пропускной способности, на этом сайте установлены две сервера Standard Edition, на которых будут размещаться эти конференции. Так как в этом случае устанавливаются серверы Standard Edition, Lync Server by Definition рассматривает его как центральный сайт, который рассматривается в построителе топологий и средство планирования.
    
    Для повышения производительности достаточно одного сервера Standard Edition, но Организация развернула две и соединить их, чтобы обеспечить высокую доступность на случай, если один сервер отключается.
    
    Хотя сайт В рассматривается как центральный сайт, там не требуется разворачивать пограничные серверы. В данном примере сайт В будет использовать пограничные серверы, развернутые на сайте А.

  - **Мониторинг и архивация**   в этой организации развернута и мониторинг, и архивация. После развертывания мониторинг и архивация выполняются на каждом сервере переднего плана. Базы данных для этих функций могут быть размещены совместно с серверной базой данных или на отдельном сервере. В данной организации эти базы данных размещены на отдельном сервере на центральном сайте B. Базы данных получают данные мониторинга и архивации с серверов переднего плана на всех сайтах.

  - **Параметры развертывания сайта филиала.**    У этой Организации на самом деле более 50 сайтов филиалов, в подробных схемах показаны только три из них. Сайты филиалов 1 и 3 не имеют отказоустойчивой связи WAN с центральным сайтом, поэтому у них есть устройства для обеспечения связи в филиалах, развернутые для предоставления телефонной службы на случай, если канал глобальной сети переключается на центральный сайт. У сайта филиала 2 нет отказоустойчивой глобальной сети, поэтому требуется только шлюз телефонной сети общего пользования (PSTN). Развернутый шлюз PSTN поддерживает обход сервера-посредника, поэтому сервер-посредник не нужен на сайте филиала B. Сведения о том, как установить на сайте филиала, можно узнать в статье [Планирование устойчивости корпоративной голосовой связи в Lync Server 2013](lync-server-2013-planning-for-enterprise-voice-resiliency.md) в документации по планированию.

  - **Магистральная линия SIP и сервер-посредник.**    Обратите внимание на то, что на центральном сайте б сервер-посредник не размещается вместе с серверами переднего плана. Это объясняется тем, что для сайтов, использующих распределение каналов SIP, рекомендуется изолированный сервер-посредник. В большинстве других случаев мы рекомендуем размещать сервер-посредник вместе с серверами переднего плана. Дополнительные сведения о топологиях серверов-посредников приведены в статье [компоненты и топологии для сервера-посредника в Lync server 2013](lync-server-2013-components-and-topologies-for-mediation-server.md) в документации по планированию.

  - **Развернут сохраняемый чат.**    Эта организация развернула серверы, необходимые для включения сохраняемого чата. Было развернуто несколько серверов переднего плана сохраняемого чата, как для обработки нагрузки от множества пользователей в пуле, так и для обеспечения высокого уровня доступности. Также было развернуто соответствие для сохраняемого чата, а также хранилище сохраняемого чата и соответствие сохраняемого чата были размещены на разных серверах. Эти хранилища могли быть размещены вместе, и даже вместе с тыловым сервером, но данная организация выбрала раздельное размещение в целях обеспечения лучшей производительности.

  - **Балансировка нагрузки на DNS.**    Пул переднего плана и пул пограничных серверов. Это устраняет потребность в аппаратных балансировщиках нагрузки для внутреннего интерфейса пограничных серверов, а также значительно уменьшает время, затрачиваемое на настройку и обслуживание аппаратных балансировщиков нагрузки для других пулов, поскольку аппаратные балансировщики нагрузки нужны только для трафика HTTP. Для получения дополнительных сведений о балансировке нагрузки на DNS обратитесь к разделу [Балансировка нагрузки на DNS в Lync Server 2013](lync-server-2013-dns-load-balancing.md) в документации по планированию.

  - **Развертывание единой системы обмена сообщениями Exchange.**   Lync Server работает как с *локальными* развертываниями единой системы обмена сообщениями Exchange, так и с *размещенной* единой системой обмена сообщениями Exchange. Центральный сайт A включает сервер единой системы обмена сообщениями Exchange (единой системы обмена сообщениями Exchange), на котором работает Microsoft Exchange Server, а не Lync Server. Функция единой системы обмена сообщениями Exchange для Lync Server работает в пуле переднего плана.
    
    Центральный сайт Б использует размещенный Exchange, поэтому функциональность сервера единой системы обмена сообщениями Exchange также размещенная.
    
    Подробные сведения о единой системе обмена сообщениями Exchange приведены [в статье Планирование интеграции единой системы обмена сообщениями Exchange в Lync server 2013](lync-server-2013-planning-for-exchange-unified-messaging-integration.md) и [Интеграция единой системы обмена сообщениями Exchange в Lync Server 2013](lync-server-2013-hosted-exchange-unified-messaging-integration.md) в документации по планированию.

  - **Сервер Office Web Apps.**   Организациям, использующим веб-конференции, рекомендуется развернуть сервер или ферму серверов Office Web Apps. Ферму серверов Office Web Apps можно развернуть на одном сайте, обслуживающем трафик со всех сайтов, или на каждом сайте. Сервер Office Web Apps позволяет использовать в веб-конференциях слайды Powerpoint. Дополнительную информацию можно узнать в статье [Настройка интеграции с сервером Office Web Apps и Lync Server 2013](lync-server-2013-enabling-office-web-apps-server-and-lync-server-2013.md).

  - **Можно добавить режиссеров.**   Если для организации требуется повысить уровень защиты от атак типа "отказ в обслуживании", он также может развернуть пул директоров. Директор — это отдельная, необязательная роль сервера в Lync Server, которая не является домашними учетными записями пользователей или предоставляют услуги по присутствию или конференции. Он выступает в качестве внутреннего сервера следующего прыжка, на который пограничный сервер направляет входящий трафик SIP, предназначенный для внутренних серверов. Директор предварительно проверяет подлинность входящих запросов и перенаправляет их в домашний пул или на сервер пользователя. Предварительная проверка подлинности в Директоре позволяет отклонять запросы от неизвестных для развертывания учетных записей пользователей. Директор помогает разделяют серверы переднего плана от вредоносного трафика, например из-за атак типа "отказ в обслуживании" (DoS). Если при такой атаке сеть передается с недопустимым внешним трафиком, трафик завершается в директоре.

  - **Развернут System Center Operations Manager.**   Рекомендуется следить за работоспособностью развертывания Lync Server, чтобы обеспечить доступность служб для конечных пользователей. Вы можете отслеживать Lync с помощью пакета управления System Center Operations Manager для Lync, который доступен для бесплатной загрузки Майкрософт. С помощью пакета управления Lync можно получать уведомления в режиме реального времени при возникновении какой-либо неполадки, составлять отчеты по доступности служб и выполнять другие функции. Это позволит упреждающее реагировать на неполадки в развертывании, прежде чем эти неполадки будут замечены пользователями.
    
    Эта организация развернула сервер System Center Operations Manager на каждом центральном сайте.
