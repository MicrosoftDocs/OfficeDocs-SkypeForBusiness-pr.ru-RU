# <a name="mn-trunk-in-lync-server-2013"></a>Магистраль M:N в Lync Server 2013

<div data-xmlns="http://www.w3.org/1999/xhtml">

_**Последнее изменение темы:** 2012-10-01_

Lync Server 2013 обеспечивает большую гибкость в определении магистрали для маршрутизации вызовов из предыдущих выпусков. Магистраль — это логическая связь между сервером-посредником и номером прослушивающего порта с шлюзом и номером прослушивающего порта. Это подразумевает несколько моментов: сервер-посредник может иметь несколько магистральных линий связи с одним шлюзом; Сервер-посредник может иметь несколько магистральных линий связи между разными шлюзами; напротив, шлюз может иметь несколько магистральных каналов для разных серверов-посредников.

Корневая магистраль по-прежнему должна создаваться при добавлении шлюза в топологию Lync с помощью построителя топологий. Количество шлюзов, которые может обрабатывать данный сервер-посредник, зависит от мощности обработки сервера в пиковые часы занятости. Если вы разворачиваете сервер-посредник на оборудовании, который превышает минимальные требования к оборудованию для Lync Server 2013, как описано в статье [Supported Hardware for Lync server 2013](lync-server-2013-supported-hardware.md) в документации по поддержке, то при расчете количества активных вызовов с автономным сервером-посредником может обрабатываться около 1000 вызовов. При развертывании на оборудовании с соблюдением этих спецификаций ожидается перекодирование сервера-посредника, но по-прежнему выполняется маршрутизация вызовов для нескольких шлюзов, даже если шлюзы не поддерживают обход сервера-посредника.

При определении маршрута вызовов указываются магистральные линии связи, связанные с этим маршрутом, но не указываются серверы-посредники, связанные с этим маршрутом. Вместо этого вы используете построитель топологий для связи магистрали с серверами-посредниками. Другими словами, служба маршрутизации определяет, какая магистральная линия будет использоваться для вызова, а затем сервер-посредник, связанный с этой магистралью, отправляет сигнал для этого вызова.

Сервер-посредник можно развернуть как пул; Этот пул можно разместить в пуле переднего плана, или он может быть развернут как изолированный пул. Если сервер-посредник размещен в пуле переднего плана, размер пула не может превышать 12 (предельный размер пула регистратора). В совокупности эти новые возможности увеличивают надежность и гибкость развертывания серверов-посредников, но для них требуются связанные возможности в следующих одноранговых сущностях:

  - **Шлюз PSTN.** Квалифицированный шлюз Lync Server 2013 должен реализовать балансировку нагрузки на DNS, которая позволяет шлюзу PSTN использовать квалифицированный шлюз для работы с подсистемой балансировки нагрузки для одного пула серверов-посредников и, следовательно, для вызовов балансировки нагрузки в пуле.

  - **Пограничный контроллер сеансов.** Для магистрали SIP одноранговый объект является пограничным контроллером сеансов (SBC) у поставщика услуг Интернет-телефонии. В направлении от пула серверов-посредников к SBC, SBC может принимать подключения от любого сервера-посредника в пуле. В направлении от SBC к пулу трафик можно отправлять на любой сервер-посредник в пуле. Один из способов достижения этого заключается в балансировке нагрузки на DNS, если она поддерживается поставщиком услуг и SBC. Кроме того, можно предоставить поставщику услуг IP-адреса всех серверов-посредников в пуле, а поставщик услуг будет подготавливать их в своих SBC-каналах в качестве отдельной магистральной линии SIP для каждого сервера-посредника. После этого поставщик услуг будет обрабатывать балансировку нагрузки для собственных серверов. Не все поставщики служб или SBCs могут поддерживать эти возможности. Кроме того, поставщик услуг может взимать дополнительную плату за эту возможность. Как правило, каждая магистральная линия SIP на SBC приводит к ежемесячной плате.

  - **IP-УАТС.** В направлении от пула серверов-посредников к оконечному IP-адресу SIP IP-УАТС, IP-УАТС может принимать подключения от любого сервера-посредника в пуле. В направлении от IP-УАТС к пулу трафик можно отправлять на любой сервер-посредник в пуле. Так как большинство IP-PBXs не поддерживают балансировку нагрузки на DNS, рекомендуется определять отдельные прямые подключения SIP из IP-УАТС на каждый сервер-посредник в пуле. IP-АТС затем будет осуществлять собственную балансировку нагрузки, распределяя трафик через группу магистралей. Подразумевается, что для группы магистралей в IP-АТС задан согласованный набор правил маршрутизации. Поддерживает ли отдельная IP-УАТС эту концепцию магистральной группы и то, как она пересекается с собственной избыточностью и архитектурой кластеризации IP-УАТС, необходимо определить, чтобы можно было определить, может ли кластер серверов-посредников правильно взаимодействовать с IP-УАТС.

Пул серверов-посредника должен иметь единообразное представление однорангового шлюза, с которым он взаимодействует. Это означает, что все участники пула получают доступ к одному определению шлюза из хранилища конфигурации и могут с одинаковой вероятностью взаимодействовать с ним при обработке исходящих вызовов. Таким образом, невозможно сегментировать пул, чтобы некоторые серверы-посредники могли общаться только с определенными одноранговыми узлами шлюза для исходящих вызовов. Если требуется такая сегментация, необходимо использовать отдельный пул серверов-посредников. Это может потребоваться, например, если отсутствуют связанные возможности в шлюзах PSTN, SIP-магистралях или IP-АТС для взаимодействия с пулом (как описывалось ранее в этой статье).

Конкретный шлюз PSTN, IP-УАТС или магистральная магистральная сеть SIP могут маршрутизироваться к нескольким серверам-посредникам или магистральным линиям. Количество шлюзов, которыми может управлять конкретный пул серверов-посредников, зависит от количества вызовов, использующих обход сервера-посредника. Если при большом количестве звонков используется обход сервера-посредника, то сервер-посредник в пуле может обрабатывать множество дополнительных вызовов, так как требуется только обработка сигнальных уровней.
